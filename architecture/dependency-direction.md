# 依存の向き（DIP）

- タグ: principle, coupling
- シグナル: 安定して変わりにくいもの（抽象・ドメインの中核）ではなく、変わりやすい具体実装
  （特定のライブラリ・外部 API クライアント・具体型）に上位のロジックが直接依存している。
- なぜ問題か: 変わりやすい側の変更が、変わってほしくないビジネスロジック側に波及する。
  依存性逆転の原則（DIP）の違反。
- 検出方法: 中核ロジックの import 文に、インフラ・外部 SDK・具体実装が直接並んでいないか見る。
- 重要度の目安: imo が基本。ただし小規模リポジトリで抽象層を挟むコストの方が高い場合は
  指摘しない判断もする（[[rule-of-three]] の過剰抽象と表裏の関係。バランスで見る）。
- 例外: 使い捨てスクリプト・薄い glue コード。抽象を挟む価値が出るのは差し替え・テストの
  必要が現実にあるとき。
