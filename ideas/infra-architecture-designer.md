# infra-architecture-designer

> インフラ設計をするエンジニアが、AWS / Google Cloud の構成を一人で抱えずに壁打ちしながら決め、
> 構成図・設計判断の記録・セキュリティ観点のセルフレビューまでを 1 セットで残すためのスキル。

---

## 場面 (When)

- ユーザーが「インフラ構成を一緒に設計して」と言ったとき
- 新規案件・新規サービスの立ち上げで「この要件で AWS の構成図を描いて」と言ったとき
- 方向性だけ決まっていて「GCP でサーバレス構成を検討したい」と言ったとき
- 構成の選択肢に迷っていて「構成案を 2〜3 案比較して」と言ったとき
- 既にある構成図を指して「この構成図のセキュリティレビューをして」と言ったとき
- 提案・見積の前段で「インフラ設計の壁打ちに付き合って」と言ったとき

## 解決する課題 (Problem)

- 現状: 一人でインフラ構成を決めると、案の比較（コスト・運用負荷・スケール性）が頭の中だけで
  終わり、なぜその構成にしたかが残らない。後から「なんで NAT Gateway 2 つなの」「なぜ ECS でなく
  Cloud Run なの」に答えられない。さらにセキュリティ観点の見直しは後回しになりがちで、
  public サブネットの DB・Editor ロール流用のような穴が本番直前や事後に見つかる。
- 痛みの種類: 見落とし（セキュリティの穴・単一障害点）/ 手戻り（設計判断の記録がなく再検討）/
  認知負荷（案の比較を一人で抱える）
- 影響範囲: 個人作業（設計の質）〜プロダクション（セキュリティ事故・コスト暴走）〜
  チーム共有物（依頼者への構成説明資料の下敷き）

---

## 解決策 (Solution)

1. 要件を対話でヒアリングする（2〜3 問ずつ。ワークロード種類 / トラフィック・データ量 /
   可用性 / 予算感 / 運用体制 / 既存資産 / コンプライアンス / IaC の有無）
2. 特性の異なる 2〜3 案をトレードオフ表（月額コスト概算・運用負荷・スケール性）付きで提示し、
   対話で 1 案に収束させる
3. Mermaid のコンポーネント図 + コンポーネント表（リソース種別 / 役割 / 冗長化 / ネットワーク配置）を生成する
4. references のチェックリスト（8 観点）でセキュリティセルフレビューを 1 周回し、
   重大度付きの指摘と「確認した観点」の列挙を出す
5. 決定事項・不採用案とその理由・ASSUMPTION を含む設計記録を対象プロジェクト配下
   `infra-design/<YYYY-MM-DD>-<topic>.md` に保存する（保存前に確認）
6. 完了条件: 構成図・レビュー（指摘 or 確認済み観点の列挙）・設計記録の 3 点が揃っていること

### 入力 (Inputs)

| 名前 | 型 | 必須 | 出どころ | 例 |
|---|---|---|---|---|
| requirements | 対話での回答（ヒアリング 8 項目） | yes（未回答は ASSUMPTION で補完） | ユーザー発話 | 「Web API、月間 10 万 req、一人運用」 |
| cloud_provider | enum (AWS / Google Cloud / 未定) | no (既定: 対話で確定) | ユーザー発話 | 「GCP で」 |
| target_project | string (保存先のリポジトリ / projects/<name>) | yes（保存時） | ユーザー発話 / cwd | `projects/acme-api/` |
| existing_diagram | Mermaid / 構成図テキスト | no（レビューのみ実行する場合） | ユーザー提示 | 既存構成図の貼り付け |

### 出力 (Outputs)

| 名前 | 形式 | どこに置く | 例 |
|---|---|---|---|
| 候補構成の比較表 | markdown 表 | チャット返信 | 3 案 × コスト概算/運用負荷/スケール性 |
| 構成図 + コンポーネント表 | Mermaid + markdown 表 | チャット返信 + 設計記録内 | `graph TB` のコンポーネント図 |
| セキュリティセルフレビュー | markdown（指摘表 + 確認済み観点 + 免責） | チャット返信 + 設計記録内 | High 1 件・Mid 2 件 + 8 観点確認済み |
| 設計記録 | markdown | `<repo または projects/<name>>/infra-design/<YYYY-MM-DD>-<topic>.md` | `2026-07-03-acme-api-infra.md` |

## 想定ユーザー (Who)

- 主: SES / フリーランスで参画中のエンジニア本人。インフラ設計を一人で決める場面が多く、
  壁打ち相手と判断の記録が欲しい
- 副: 構成の説明を受ける依頼者側の技術リーダー（設計記録を説明資料の下敷きにする）

ユーザーが持っている知識: 対象ワークロードの中身と、AWS / GCP の基本的なサービス名。
持っていない知識: 案ごとのトレードオフの整理と、セキュリティ観点の網羅的なチェックリスト。

## 既存ツール・スキルとの差別化 (Differentiation)

| 代替 | 限界 | このスキルが上回る点 |
|---|---|---|
| 素の Claude との雑談で設計 | 図・記録・レビューが残らず、聞き方次第で観点が抜ける | ヒアリング項目・レビュー観点・記録フォーマットが固定されており、毎回同じ水準で回る |
| `edith-tech-selection-research` | 個別技術の選定比較が目的で、構成全体の設計・図・セキュリティレビューはしない | 構成設計に特化。深いミドルウェア比較が要る場面ではそちらへ誘導して結果を入力に使う |
| `friday-design-doc-generator` | 既存コードの現在形をドキュメント化するスキルで、これから作る構成の設計はしない | 構成が存在しない段階の設計・比較・収束を担う |
| draw.io 等の作図ツール | 図は描けるが、案の比較・セキュリティ観点・判断の記録は人力 | 図 + レビュー + 記録を 1 フローで生成する |
| 公式の Well-Architected レビュー | 網羅的だが重く、個人の設計初期に回すには過剰 | 設計初期に 1 周で回る 8 観点に絞り、指摘と確認済みを分けて出す |

## 成功条件 (Success Criteria)

- [ ] 設計セッション後に、構成図・レビュー結果・設計記録の 3 点が揃って残っている
- [ ] セルフレビューの出力に、指摘の有無によらず「確認した観点」8 項目の列挙が毎回含まれている
- [ ] 設計記録に不採用案とその理由が最低 1 件残っており、後から「なぜこの構成か」に答えられる
- [ ] コスト概算に「概算」の明記と幅表示があり、ユーザーが確定額と誤読しない
- [ ] ヒアリングが 1 回のメッセージで 4 問以上を詰め込んでいない（2〜3 問ずつの対話が守られている）

## 失敗モード予測 (Failure Modes)

| 失敗 | 起きる条件 | 回避策 |
|---|---|---|
| コスト概算を確定額と誤読され、見積・予算の根拠に使われる | 単価の記憶が古い / リージョン・割引で変動する項目を断定したとき | 「概算」を毎回明記し、曖昧な単価は幅で示す。正確な見積は公式計算ツール（AWS Pricing Calculator / Google Cloud 料金計算ツール）を案内 |
| セルフレビューを「セキュリティ監査済み」と受け取られる | 指摘ゼロの出力を「安全のお墨付き」と誤読されたとき | 専門監査の代替ではない免責を毎回明記。指摘ゼロでも「確認した観点」の列挙に留め、安全の保証表現を使わない |
| ヒアリング未回答の項目を勝手な前提で埋め、実態と合わない構成になる | ユーザーが答えられない / 対話が途中で切れたとき | 推測で進めてよいが、設計記録に ASSUMPTION として必ず明記し、前提一覧の章にまとめる |
| 質問攻めでヒアリングが苦痛になり対話が途切れる | 8 項目を一度に聞いたとき | 2〜3 問ずつに分け、分かっている情報は聞き直さない |
| 設計の勢いでリソース作成・IaC 生成まで踏み込む | 「ついでに Terraform も」と流れたとき | クラウドへの実操作・リソース作成は範囲外と SKILL.md に明記（Terraform 化は将来の diagram-to-terraform スキルの領分） |
| 設計記録が当リポジトリ側に保存され、プロジェクト固有情報が混在する | 保存先の確認を省いたとき | 保存先は対象プロジェクト配下 `infra-design/` に固定し、保存前にユーザーへ確認する |

## 必要なリソース (Resources)

- 外部 API / CLI: なし（設計・作図・レビューはローカル完結。クラウド API は呼ばない）
- 権限: 対象プロジェクトディレクトリへの書き込み（設計記録の保存時のみ、保存前に確認）
- 参照すべき公開知識: AWS Well-Architected Framework / Google Cloud Architecture Framework の
  セキュリティ・信頼性の考え方（チェックリストの観点の下敷き）、各サービスの料金体系
- ローカル前提: 保存先となる対象プロジェクトのディレクトリ（リポジトリ or `projects/<name>/`）があること

## 実装メモ (Implementation Notes)

- ディレクトリ構成: `.claude/skills/arc-reactor-infra-architecture-designer/`
  - `SKILL.md` … 役割・トリガー・標準フロー（ヒアリング → 候補比較 → 構成図 → セルフレビュー → 記録保存）・品質規約
  - `references/security-review-checklist.md` … セキュリティレビュー 8 観点を AWS / Google Cloud の
    典型リソース名で具体化したチェックリスト + レビュー出力フォーマット
- scripts は作らない: 中核が対話・作図・観点照合という Claude の読解 / 生成作業で、
  決定論的に計算すべき数値（正確な料金等）はスキルの範囲外（公式計算ツールへ案内）のため
- 既存スキルとの依存関係: `edith-tech-selection-research`（ミドルウェア選定の深い比較が要るとき誘導。
  結果を本スキルの入力に使える）。将来の diagram-to-terraform スキルが本スキルの構成図を入力にする想定
- persona: `personas/arc-reactor.md` に従う（応答は日本語、コード・識別子は英語、推測は ASSUMPTION 明記）
