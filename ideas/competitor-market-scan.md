# competitor-market-scan

> フリーランスエンジニアが、参画案件や自社サービスの企画・提案の前に、対象ドメインの競合と市場トレンドを出典付きで 1 本の要約にまとめるためのスキル。

---

## 場面 (When)

- ユーザーが「〇〇領域の競合を調べて」「この市場のトレンドをまとめて」と明示的に依頼したとき
- 新規案件の参画前・提案書作成前に、対象サービスの周辺市場を把握したいと言ったとき
- 自社サービス（またはクライアントのサービス）の類似サービス・代替サービスを列挙してほしいと言ったとき
- 既存の競合調査結果を見せながら「最新情報で更新して」と言ったとき（鮮度の更新）
- 「この機能って他社はどうしてる？」のように、特定の機能軸で競合比較を求めたとき

### 想定トリガー文 (実際のユーザー発話例)

1. 「オンライン議事録サービスの競合を調べて」
2. 「この案件の周辺市場をスキャンして」
3. 「〇〇の類似サービスと料金を比較して」
4. 「最近の生成 AI 議事録界隈のトレンドをまとめて」
5. 「競合マーケットスキャンして。対象は個人向け家計簿アプリ」
6. 「前回の競合調査、情報が古くなってないか最新化して」

---

## 解決する課題 (Problem)

- 現状: 案件・自社サービスの周辺市場を調べるとき、検索 → タブを大量に開く → メモが散らばる → 出典を控え忘れる、を毎回手作業でやっている。数時間かかるうえ、後から「その情報どこ情報？いつ時点？」に答えられない。
- 痛みの種類: 時間（都度数時間）、認知負荷（探索範囲の管理）、見落とし（単一の検索角度に依存した網羅漏れ）、信頼性毀損（出典・鮮度不明の情報を提案に混ぜてしまうリスク）
- 影響範囲: 個人作業（調査そのもの）＋チーム共有物（調査結果を提案・報告に使うため、誤情報は対外的な信用に直結する）

---

## 解決策 (Solution)

1. 対象ドメイン・調査目的・重視する切り口（競合列挙 / 機能 / 価格 / トレンド / ニュース）をヒアリングで確定する
2. 複数の切り口で WebSearch / WebFetch による探索を行う（切り口ごとに検索クエリを変える）
3. 収集した情報を検証する: 出典 URL と情報の日付を必ず控え、一次情報（公式サイト・公式発表）を優先し、日付不明・出典不明の情報は破棄するか不確実と明記する
4. 競合一覧表 + トレンド要約 + 出典リスト（URL・取得日・情報の日付）+ カバーできなかった範囲、の固定フォーマットで Markdown レポートを生成する
5. 情報の日付が古いもの（目安: 12 か月超）には「古い可能性あり」の注記を付けて完了

### 入力 (Inputs)

| 名前 | 型 | 必須 | 出どころ | 例 |
|---|---|---|---|---|
| domain | string | yes | ユーザー発話 | 「オンライン議事録サービス」 |
| purpose | string | yes | ヒアリング | 「新規案件の提案準備」 |
| focus | enum[]（competitors / features / pricing / trends / news） | no（省略時は全部） | ヒアリング | [competitors, pricing] |
| region | string | no（デフォルト: 日本） | ヒアリング | 「日本」「グローバル」 |
| prev_report | file path (markdown) | no | ユーザー指定 | 前回の調査レポート（更新モード時） |

### 出力 (Outputs)

| 名前 | 形式 | どこに置く | 例 |
|---|---|---|---|
| scan report | markdown（固定フォーマット） | `market-scan/<domain-slug>/<YYYY-MM-DD>.md` **(ASSUMPTION: データ出力先はリポジトリ直下 `market-scan/` とした。既存スキルが `worklog-data/` 等リポジトリ直下にデータ置き場を持つ慣習に合わせた)** | 競合一覧表 + トレンド要約 + 出典 + 未カバー範囲 |
| summary | markdown | チャット返信 | レポートの要点（競合数・主要トレンド・注意点）を 10 行以内で提示 |

---

## 想定ユーザー (Who)

- 主: フリーランスエンジニア本人。案件の提案・技術選定・自社サービス企画のために市場感を短時間で掴みたい状況。
- 副: 調査結果の読み手としてのクライアント・チーム（ただし清書・共有ドキュメント化は friday 系スキルの責務）。

ユーザーが持っている知識: 対象ドメインの技術的背景・自分の調査目的。
ユーザーが持っていない知識: そのドメインのプレイヤー一覧・直近の市場動向・各社の価格。

---

## 既存ツール・スキルとの差別化 (Differentiation)

| 代替 | 限界 | このスキルが上回る点 |
|---|---|---|
| 手動の Web 検索 | タブ地獄・出典控え忘れ・網羅性が検索者のクエリ力に依存 | 切り口を固定した多角探索 + 出典/取得日を強制記録 |
| `deep-research` スキル（汎用） | 汎用リサーチで、競合比較表・鮮度注記・未カバー範囲の明示といった市場調査特化の出力構造を持たない | 競合一覧表・価格・トレンドの固定フォーマットと鮮度管理が組み込み |
| 生成 AI に単発で聞く | 学習時点の知識で回答され、出典も鮮度も不明。ハルシネーション混入 | Web 探索ベース + 出典不明情報は破棄/明記するルール |
| ultron-high-dividend-stock-screener | 日本株スクリーニング専用 | 任意ドメインの市場・競合が対象 |

## 成功条件 (Success Criteria)

- [ ] レポート内のすべての事実主張（競合の存在・機能・価格・トレンド）に出典 URL が紐づいており、出典リストに「取得日」と「情報の日付（判明分）」が記録されている
- [ ] 競合が 5 社以上列挙される（5 社未満の場合は「なぜ少ないか（ニッチ市場・探索限界）」が明記されている）
- [ ] 「カバーできなかった範囲」セクションが空でなく、探索しなかった切り口・確認できなかった項目が具体的に書かれている
- [ ] 情報の日付が 12 か月超の項目すべてに「古い可能性あり」注記が付いている
- [ ] 前回レポートを渡して更新を依頼したとき、差分（新規参入・価格改定・撤退）が識別できる

## 失敗モード予測 (Failure Modes)

| 失敗 | 起きる条件 | 回避策 |
|---|---|---|
| 鮮度切れ: 古い記事（数年前のまとめ記事等）を現在の事実として提示する | 検索上位に古い比較記事が来る / 記事に日付がない | 各情報に「情報の日付」を必ず控え、12 か月超は「古い可能性あり」を注記。価格・機能は可能な限り公式サイト（一次情報）で再確認。日付不明は「日付不明」と明記 |
| 網羅漏れ: 有名プレイヤーの見落とし・特定切り口だけの偏り | 単一クエリ・単一言語での探索 / 早期打ち切り | 切り口別（競合列挙・機能・価格・トレンド・ニュース）に別クエリで探索し、「〇〇 alternatives」「〇〇 比較」等の複数パターンを使う。探索しなかった範囲を「カバーできなかった範囲」に必ず書き、暗黙の打ち切りをしない |
| 出典不明情報の混入: モデルの内部知識やまとめ記事の孫引きが事実として混ざる | Web で確認できなかった知識で表の空欄を埋めてしまう | 出典 URL の無い事実主張は書かない（破棄する）。どうしても載せる価値がある場合は「未確認（出典なし）」と明記して事実と区別する |
| ドメイン解釈のズレ: 「議事録サービス」の範囲認識がユーザーとずれ、無関係な競合が並ぶ | ヒアリング不足のまま探索開始 | 探索前に対象ドメインの定義・目的・地域をヒアリングで確定し、競合の包含基準を冒頭に明記する |
| 価格情報の誤り: プラン改定・キャンペーンで実勢と乖離 | 二次情報（比較記事）だけで価格を書く | 価格は原則、公式料金ページを出典にし、取得日を付す。公式で確認できない場合はその旨を明記 |

## 必要なリソース (Resources)

- 外部 API / CLI: WebSearch / WebFetch ツール（Claude Code 標準）
- 権限: Web アクセス
- 参照すべき公開知識: 各社公式サイト・料金ページ・プレスリリース・業界レポート
- ローカル前提: 出力先 `market-scan/` ディレクトリを作成できること

## 実装メモ (Implementation Notes)

- ディレクトリ構成: `.claude/skills/edith-competitor-market-scan/`（SKILL.md + templates/scan-report.md）
- scripts は作らない: 探索・検証・要約はすべてエージェントの対話 + WebSearch/WebFetch で完結し、決定的な処理（ファイル整形等）が薄いため
- テンプレート `templates/scan-report.md` を出力の正とし、見出し構成を固定する（後から差分更新しやすくするため）
- persona `personas/edith.md` の原則（出典明示・一次情報優先・複数切り口・未カバー範囲の明示）をフローと品質セクションに反映する

## 自己進化を入れるか (Self-Evolution Scope)

- [ ] パイプラインあり
- [x] パイプラインなし — 理由: 調査対象ドメインが毎回異なり、実行ログから汎化できる学習項目（検索クエリの型・出力フォーマット）はテンプレートと SKILL.md の探索手順に静的に織り込み済み。むしろ過去ドメイン固有の知見が別ドメインの調査に混入するリスク（プロジェクト固有情報の分離ルールに抵触）があるため、ログ蓄積による自動書き換えは行わない。改善は使用時に手動で SKILL.md へ反映する。
