# learning-roadmap

> フリーランス本人が、身につけたいスキル（技術・資格・語学など）の学習を始めるとき・続けるときに、
> 「目標・現状 → 段階的ロードマップ（週次計画つき）」を作り、定期的な進捗レビューで計画を現実に追従させ続けるためのスキル。

---

## 場面 (When)

- 新しい技術や資格の学習を始めたいが、何から手を付けるべきか分からず着手できていないとき
- 「Rust を 3 ヶ月で業務レベルにしたい」のように目標だけあって計画がないとき
- 過去に立てた学習計画が崩れて放置されており、立て直したいとき
- 週次・月次の振り返りで「学習の進捗どうだっけ」と確認したいとき
- 案件で必要になる技術のキャッチアップ計画を、稼働の合間の可処分時間ベースで組みたいとき

### 想定トリガー文 (実際のユーザー発話例)

1. 「Rust の学習ロードマップを作って」
2. 「AWS SAA を 3 ヶ月で取りたい。学習計画を立てて」
3. 「英語学習の計画、目標から一緒に整理して」
4. 「学習ロードマップの進捗レビューをして」
5. 「先週の学習計画、崩れたから立て直したい」
6. 「このロードマップを今の状況に合わせて更新して」

---

## 解決する課題 (Problem)

- 現状: 学習したいことはあるが、目標が曖昧なまま教材だけ買う / 計画を立てても初週で崩れて放置される / 「どこまでできるようになったか」を測る基準がなく達成感も軌道修正もない。
- 痛みの種類: 手戻り（計画倒れの繰り返し）、認知負荷（次に何をやるか毎回考える）、機会損失（スキル獲得の遅れ = 案件獲得力の低下）。
- 影響範囲: 個人作業（本人のスキル形成・キャリア）。ただし獲得スキルは案件対応力として間接的に仕事へ波及する。

---

## 解決策 (Solution)

1. ヒアリング: 目標（何が・いつまでに・どのレベルで）、現状（既有スキル・過去の学習歴）、**使える時間（週あたりの現実的な学習可能時間）** を対話で確定する。曖昧なら確定するまで質問する。
2. マイルストーン分割: 目標を 3〜6 個程度の段階に分割し、各段階に **観測可能な達成基準（「〜ができる」を検証可能な形で）と成果物（コード・合格証・記事など）** を必ず付ける。
3. 週次計画: 直近のマイルストーンを週単位のタスクに落とす。使える時間を超えない量に抑える。
4. 保存: 固定フォーマットの Markdown として `learning-roadmaps/` **(ASSUMPTION: リポジトリ直下に新設。他スキルのデータ置き場と同じ流儀)** に保存する。
5. 進捗レビュー（2 回目以降）: 既存ロードマップを読み込み、進捗ログに実績を追記し、遅れ・前倒しに応じて週次計画とマイルストーン期間を再計画する。

### 入力 (Inputs)

| 名前 | 型 | 必須 | 出どころ | 例 |
|---|---|---|---|---|
| goal | string | yes | ユーザー発話（ヒアリングで確定） | 「3 ヶ月後に Rust で CLI ツールを自作できる」 |
| current_level | string | yes | ユーザー発話（ヒアリングで確定） | 「他言語経験 5 年、Rust は未経験」 |
| available_hours_per_week | number | yes | ユーザー発話（ヒアリング必須） | 5 |
| deadline | date \| null | no | ユーザー発話 | 2026-10-01 |
| existing_roadmap | markdown file path | レビュー時 yes | `learning-roadmaps/<topic>.md` | `learning-roadmaps/rust.md` |
| progress_report | string | レビュー時 yes | ユーザー発話 | 「週次タスクの 2/3 まで終わった」 |

### 出力 (Outputs)

| 名前 | 形式 | どこに置く | 例 |
|---|---|---|---|
| roadmap | markdown（固定フォーマット: 目標/前提/マイルストーン表/週次計画/進捗ログ） | `learning-roadmaps/<topic>.md` | `learning-roadmaps/rust.md` |
| review_summary | markdown（チャット返信） | チャット | 進捗判定・遅延要因・再計画の差分の要約 |

---

## 想定ユーザー (Who)

- 主: フリーランスエンジニア本人。案件の合間に新技術・資格・語学を計画的に学びたい。
- 副: なし（本人専用）。

ユーザーが持っている知識: ソフトウェア開発の一般知識、自分の稼働状況。
ユーザーが持っていない知識: 学習対象分野の標準的な習得順序、適切なマイルストーンの粒度。

---

## 既存ツール・スキルとの差別化 (Differentiation)

| 代替 | 限界 | このスキルが上回る点 |
|---|---|---|
| 手書きの TODO リスト / メモ | 達成基準がなく「やった気」で終わる。崩れたら放置 | 各段階に観測可能な達成基準と成果物を強制し、レビューで計画を更新し続ける |
| 市販のロードマップ記事（roadmap.sh 等） | 汎用で自分の現状・可処分時間を考慮しない | 現状と週あたりの使える時間をヒアリングして個別化する |
| `karen-problem-essence-organizer` | 課題整理が目的で、期間・週次粒度の計画と進捗管理は対象外 | マイルストーン・週次計画・進捗ログという計画運用そのものを担う |
| `deep-research` | 学習素材の調査はできるが計画・レビューの継続運用はしない | 同一ファイルへの積み上げ更新でライフサイクル全体を管理 |

---

## 成功条件 (Success Criteria)

- [ ] 生成されたロードマップの全マイルストーンに、観測可能な達成基準と成果物が 1 つ以上付いている（欠落 0 件）
- [ ] 週次計画の合計見積時間が、ヒアリングした「使える時間」を超えていない
- [ ] 進捗レビュー実行後、`learning-roadmaps/<topic>.md` の進捗ログに日付つきエントリが追記され、週次計画が実績に合わせて更新されている
- [ ] 初回作成が 1 セッション（ヒアリング込み）で完了し、ユーザーが別ツールで清書し直す必要がない

---

## 失敗モード予測 (Failure Modes)

| 失敗 | 起きる条件 | 回避策 |
|---|---|---|
| 計画倒れ（初週で崩れて放置） | 週次計画が意欲ベースで詰め込みすぎ / レビューの仕組みがない | 使える時間のヒアリングを必須にし、その 8 割程度 **(ASSUMPTION: バッファ 2 割が妥当という一般的な計画法に基づく推定)** を上限に計画。進捗レビューモードで「崩れたら作り直す」を正規フローにする |
| 非現実的な見積（マイルストーン期間が短すぎ/長すぎ） | 学習対象の難易度や本人の前提スキルを考慮せず期間を置く | 現状レベルを必ずヒアリングし、期間は「週次時間 × 週数」から逆算。根拠が推測なら **(ASSUMPTION)** として明記し、レビューで実績から補正する |
| 目標の曖昧さ（「なんとなく上達したい」のまま計画を作る） | ユーザーの初期発話が抽象的なまま生成に進む | 目標が「何が・いつまでに・どのレベルで」の形で言語化されるまで生成に進まない。曖昧な場合は達成基準の例を提示せず、オープン質問で具体化する |
| 達成基準が主観的（「理解する」「慣れる」） | 達成基準を機械的に生成した場合 | 「第三者が合否判定できるか」をチェックし、検証可能な行動（〜を実装できる、模試で〜点）に言い換える |
| レビュー時に既存ファイルを壊す（進捗ログの消失） | 全文再生成で上書きした場合 | 進捗ログは追記のみ。再計画で書き換えるのは週次計画とマイルストーンの期間・状態のみとし、過去ログは保全する |

## 必要なリソース (Resources)

- 外部 API / CLI: なし（必要に応じて学習素材調査に WebSearch を使ってもよいが必須ではない）
- 権限: リポジトリ直下 `learning-roadmaps/` への読み書き
- 参照すべき公開知識: 学習対象分野の標準的な習得順序（一般知識で足りない場合のみ調査）
- ローカル前提: onclimb-industries リポジトリ内で実行

## 実装メモ (Implementation Notes)

- ディレクトリ構成: `.claude/skills/karen-learning-roadmap/`（`SKILL.md` + `templates/roadmap-template.md`）
- scripts は作らない: 生成・更新はすべて Markdown の読み書きで完結し、決定的な処理がないため
- ロードマップの固定フォーマットは `templates/roadmap-template.md` に置き、SKILL.md から参照する
- データ置き場はリポジトリ直下 `learning-roadmaps/` **(ASSUMPTION: worklog-data/ や knowledge-base/ と同じ「スキルのデータ出力はリポジトリ直下」の慣例に従う)**
- 依存する既存スキルなし。`karen-problem-essence-organizer` とは独立（目標が曖昧すぎる場合にそちらを先に使う導線はあり得る）

---

## 自己進化を入れるか (Self-Evolution Scope)

- [ ] パイプラインあり
- [x] パイプラインなし — 理由: このスキルの改善ループは「進捗レビューで計画自体を実績に合わせて更新する」という形でスキルの通常フローに内蔵されており、学習・適応の対象は SKILL.md ではなく各ロードマップファイルである。固定フォーマットと少数のヒアリング項目で構成が安定しているため、ログ蓄積 → SKILL.md 書き換えのパイプラインを持つコスト（logs/・scripts/ の維持）に見合う学習余地が薄い。フォーマット改善の要望が繰り返し出るようなら、その時点で導入を再検討する。
