# api-designer

> 参画中（または自分の）プロジェクトの開発者が、新機能の API を作る前に、
> 既存 API の流儀に揃った設計とスキーマ（OpenAPI / GraphQL SDL）を壁打ちで固めるためのスキル。

---

## 場面 (When)

- 新機能の実装に入る前に「先に API を設計して」と言われたとき
- 既存 API にエンドポイントを追加する際、「既存と揃えて設計して」と言われたとき
- 設計は頭の中にあるが、OpenAPI YAML に落とすのが面倒で「スキーマ書いて」と言われたとき
- 他メンバーが作った設計案やドラフトを指して「これ既存と一貫してるかチェックして」と言われたとき
- 新規プロジェクトで API の規約が何も無い状態から「REST API 設計の壁打ちに付き合って」と言われたとき

### 想定トリガー文 (実際のユーザー発話例)

1. 「この機能の API を設計して」
2. 「新しいエンドポイントを既存 API と揃えて設計して」
3. 「OpenAPI のスキーマを書いて」
4. 「REST API 設計の壁打ちに付き合って」
5. 「この API 設計が既存と一貫してるかチェックして」
6. 「GraphQL のスキーマを設計して」
7. 「注文キャンセルの API、どう切るのがいいか一緒に考えて」

---

## 解決する課題 (Problem)

- 現状: API はエンドポイント単位で場当たり的に増え、「エンドポイントごとに流儀がバラバラ」になる。
  作った人によってページネーションが offset だったり cursor だったり、エラーボディの構造が違ったり、
  複数形・単数形が混ざったりする。既存の流儀を確認するには過去のコントローラとスキーマを
  横断して読む必要があり、確認せずに書くと後で直せない不整合が契約として固定される。
- 痛みの種類: 手戻り（レビューで流儀の指摘 → 設計やり直し）/ 見落とし（不整合が本番契約になる）/
  認知負荷（クライアント側がエンドポイントごとに別の作法を覚える）
- 影響範囲: チーム共有物（API 契約はクライアント・外部利用者を巻き込む）〜プロダクション
  （破壊的変更の見落としは障害になる）

---

## 解決策 (Solution)

1. 既存 API（ルーティング・OpenAPI・コントローラ）から現行の流儀を規約表として抽出する。
   既存が無い新規プロジェクトなら references の標準案を提示して合意を取る
2. 要件（リソースと操作 / 利用者 / 認証・認可の粒度 / 互換性制約）を対話で 2〜3 問ずつヒアリングする
3. エンドポイント一覧表（メソッド / パス / 目的 / 認可 / 主なステータスコード）で設計案を提示し、
   判断が分かれる点は選択肢とトレードオフで確認する
4. OpenAPI 3.x YAML または GraphQL SDL を生成する（リクエスト / レスポンス例・エラー例入り。
   既存スキーマがあれば追記形式）
5. 生成した設計を規約表と突き合わせ、逸脱を「意図的（理由付き）/ 要修正」に分類した
   セルフレビュー表を添付する
6. スキーマと設計記録を対象プロジェクト配下に保存する（保存前に確認）
7. 完了条件: セルフレビュー表に「要修正」が残っていないこと（意図的逸脱には理由が付いていること）

### 入力 (Inputs)

| 名前 | 型 | 必須 | 出どころ | 例 |
|---|---|---|---|---|
| target-repo | path | yes | ユーザー発話 / cwd | `projects/foo-app/` |
| feature | string | yes | ユーザー発話 + ヒアリング | 「注文のキャンセルと返金」 |
| api-style | enum | no | 既存コード / ユーザー発話 | REST（既定）/ GraphQL |
| compat-constraints | string | no | ヒアリング | 「モバイルアプリ v2.x が既存 /orders を利用中」 |

### 出力 (Outputs)

| 名前 | 形式 | どこに置く | 例 |
|---|---|---|---|
| 規約表 | markdown 表 | チャット + 設計記録 | 命名・エラー形式・ページネーション等 + 根拠 file:line |
| エンドポイント一覧 | markdown 表 | チャット + 設計記録 | `POST /orders/{id}/cancel` の行など |
| スキーマ | OpenAPI 3.x YAML / GraphQL SDL | 既存スキーマ or `<repo>/docs/api/` | paths + components 追記 |
| 設計記録 | markdown | `<repo>/docs/api/design-<topic>-<日付>.md` | 判断・不採用案・セルフレビュー表・ASSUMPTION |

## 想定ユーザー (Who)

- 主: SES 参画中 / 個人開発中のエンジニア本人。新機能の API を切る役
- 副: 設計レビューを受ける・する側のチームメンバー（設計記録とセルフレビュー表を読む）

ユーザーは HTTP / REST の基礎と対象プロジェクトのドメインは分かっている。
既存 API 全体の流儀（どこが規約で、どこが偶然か）は把握しきれていない。

---

## 既存ツール・スキルとの差別化 (Differentiation)

| 代替 | 限界 | このスキルが上回る点 |
|---|---|---|
| `friday-design-doc-generator` | 設計書一般。コードの現在形を文書化する方向で、API 契約の新規設計や機械可読スキーマは出さない | API 契約に特化し、OpenAPI / SDL まで生成。既存規約との一貫性チェックが組み込み |
| 素の Claude に「API 設計して」 | 一般的なベストプラクティスで書き、既存 API の流儀を無視しがち。毎回流儀が変わる | 既存規約の抽出 → 準拠 → 逸脱の分類までフローで強制する |
| OpenAPI エディタ / lint (Spectral 等) | 書いた後の形式チェック。設計の壁打ちや既存コードからの規約抽出はしない | 設計段階から規約に揃え、判断の記録まで残す |
| 既存コードのコピペ改変 | 元にしたエンドポイント自体が流儀から外れていると不整合を複製する | 複数箇所から規約を抽出し、割れている場合はユーザーに確認する |

## 成功条件 (Success Criteria)

- [ ] 生成したスキーマが OpenAPI / GraphQL のバリデータを通る
- [ ] セルフレビュー表の全逸脱が「意図的（理由付き）」で、「要修正」ゼロで完了している
- [ ] 設計レビューで「既存と流儀が違う」系の指摘が出ない（出た場合は規約検出の観点漏れとして references に追記）
- [ ] 規約表の各行に根拠 file:line が付いており、ユーザーが即検証できる

---

## 失敗モード予測 (Failure Modes)

| 失敗 | 起きる条件 | 回避策 |
|---|---|---|
| 既存規約の誤検出（偶然の一致を規約と誤認、少数派に揃える） | 既存 API 内で流儀が割れている | 割れている事実を規約表に併記し、どちらに揃えるかユーザーに確認。勝手に多数派へ寄せない |
| 「改善」のつもりで既存規約から黙って逸脱 | 既存の流儀に明らかな難がある場合 | 大原則で禁止。逸脱は必ず理由 + トレードオフ付きの提案とし、採否はユーザー |
| 破壊的変更の見落とし | 既存エンドポイントの変更を含む設計 | 設計案提示の段階で互換性影響（壊れる操作・移行パス）の明記を必須にする |
| スキーマだけ出して実装と乖離 | スキーマを別ファイルに新規作成した場合 | 既存スキーマファイルがあれば追記形式にし、共通 components を再利用する |
| ヒアリングが長引いて壁打ちにならない | 一度に質問を並べすぎる | 2〜3 問ずつに制限。発話から自明な項目は確認だけで済ませる |

## 必要なリソース (Resources)

- 外部 API / CLI: なし（任意で openapi バリデータがあれば検証に使う）
- 権限: 対象リポジトリの読み取り。スキーマ・設計記録の保存時のみ書き込み（保存前に確認）
- 参照すべき公開知識: OpenAPI 3.x 仕様、RFC 9457 (Problem Details)、GraphQL 仕様 / Relay Connection
- ローカル前提: 対象リポジトリが `projects/<name>/` 等ローカルにあること

## 実装メモ (Implementation Notes)

- ディレクトリ構成: `.claude/skills/arc-reactor-api-designer/`
  - `SKILL.md` … 大原則・トリガー・6 段フロー（規約検出 → ヒアリング → 設計案 → スキーマ → セルフレビュー → 保存）・品質規約
  - `references/api-design-checklist.md` … 10 観点（命名 / メソッドとステータスコード / エラー形式 /
    ページネーション / フィルタ・ソート / バージョニング / 冪等性 / 認証・認可 / 後方互換 /
    レートリミット）の検出ポイントと標準案 + GraphQL 読み替え表
- scripts は作らない: 規約抽出は `rg` とコード読解、スキーマ生成は Claude の生成が主で、
  固定スクリプト化する処理が無い
- 既存スキルとの依存関係: なし。`friday-design-doc-generator` とは別系統
  （本スキルは API 契約特化で機械可読スキーマまで出す）
- persona: `personas/arc-reactor.md` に従う（応答は日本語、コード・識別子は英語、推測は ASSUMPTION 明記）

## 自己進化を入れるか (Self-Evolution Scope)

- [ ] パイプラインあり
- [x] パイプラインなし — 理由: 磨かれるべきは設計観点と標準案であり、それは
  `references/api-design-checklist.md` への追記で足りる（レビューで観点漏れが見つかったら
  1 観点ずつ足す運用）。対話スタイルとフロー構造は固定でよく、使用ログには対象プロジェクトの
  API 仕様という固有情報が含まれるため、ログ蓄積は CLAUDE.md の分離ルールと相性が悪い。
