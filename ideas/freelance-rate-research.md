# freelance-rate-research

> SES 参画中のフリーランスエンジニアが、契約更新・単価交渉・新規案件検討の前に、
> 自分の職種・スキル・地域に対応する公開相場レンジと根拠を 1 本のレポートで把握するためのスキル。

---

## 場面 (When)

- 契約更新の面談が近づき、ユーザーが「今の単価が相場と比べてどうか知りたい」と言ったとき
- 新しい案件・エージェントからの提示単価を受け取り、妥当性を確かめたいと言ったとき
- スキルセットが増えた（例: 新しい言語・クラウド資格）タイミングで、市場価値の変化を確認したいとき
- 稼働条件（リモート可否・週稼働日数・地域）を変える検討をしていて、条件別の相場差を知りたいとき
- 単価交渉のための「根拠資料」を準備したいと明示的に依頼されたとき

### 想定トリガー文 (実際のユーザー発話例)

1. 「バックエンドエンジニアのフリーランス単価相場を調べて」
2. 「Go + AWS、経験5年、東京フルリモートだと単価いくらぐらい？」
3. 「今月60万でやってるけど相場と比べてどうか調べて」
4. 「単価交渉の材料になる相場データをまとめて」
5. 「SES の案件相場をスキル別に整理して」
6. 「フリーランスの市場単価をレポートにして」

---

## 解決する課題 (Problem)

- 現状: 単価・案件相場の肌感がなく、契約更新や新規案件の場でエージェント・クライアント側の提示額を
  そのまま受け入れがち。相場を調べようとしても、エージェントサイトを 1 つずつ開いて目視で拾う手作業になり、
  出典・取得日を控えないため後で交渉資料として使えない。
- 痛みの種類: 見落とし（相場より低い単価の継続）/ 時間（複数サイトの手動巡回）/ 認知負荷（根拠の整理）
- 影響範囲: 個人の収入・契約条件（年間で数十万円単位の差になり得る）

---

## 解決策 (Solution)

1. 条件ヒアリング: 職種・主要スキル・経験年数・地域/リモート可否・稼働形態を対話で確定する（不足分のみ質問）
2. 相場収集: WebSearch / WebFetch で複数の公開ソース（フリーランスエージェントの案件検索・単価統計ページ、
   業界調査レポート）から該当条件の単価情報を収集する
3. レンジ算出と根拠整理: ソースごとの下限・中央値相当・上限を突き合わせ、全体レンジと各ソースの根拠
   （URL・取得日・サンプルの性質）を表にする
4. 交渉材料の観点整理: 「相場と現状の位置関係」「単価を押し上げ得る要素 / 押し下げ得る要素」を
   事実ベースで列挙する（交渉戦術の助言はしない）
5. 免責を末尾に必ず付けてレポートを出力し、カバーできなかった範囲を明示して完了

### 入力 (Inputs)

| 名前 | 型 | 必須 | 出どころ | 例 |
|---|---|---|---|---|
| role | string | yes | ユーザー発話 | "バックエンドエンジニア" |
| skills | string[] | yes | ユーザー発話 | ["Go", "AWS", "Kubernetes"] |
| region | string | yes | ユーザー発話（既定: 東京 or フルリモート） | "東京 / フルリモート可" |
| years_of_experience | number | no | ユーザー発話 | 5 |
| work_style | string | no | ユーザー発話（既定: 準委任・週5） | "準委任・週5・リモート" |
| current_rate | number (万円/月) | no | ユーザー発話（比較したい場合のみ） | 60 |

### 出力 (Outputs)

| 名前 | 形式 | どこに置く | 例 |
|---|---|---|---|
| 相場レポート | markdown（テンプレート準拠） | チャット返信（依頼があればファイル保存） | 相場レンジ表＋出典＋取得日＋偏りの注記＋免責 |
| 相場レンジ | 表（ソース別 下限/中央/上限、万円/月） | レポート内 | "レバテック: 60–90 / 中央値目安 75" |
| 交渉材料の観点 | 箇条書き（事実と推測を区別） | レポート内 | "K8s 経験は掲載案件の上位帯に頻出（推測: +5〜10万の要因）" |

---

## 想定ユーザー (Who)

- 主: SES・準委任で参画中のフリーランスエンジニア本人（契約更新・単価交渉・案件乗り換えの検討時）
- 副: これからフリーランス転向を検討している会社員エンジニア

ユーザーが持っている知識: 自分のスキルセット・経験年数・現行単価。
ユーザーが持っていない知識: 各エージェントサイトの相場データの所在、単価表記の揺れ（税・精算幅・商流）の読み方。

---

## 既存ツール・スキルとの差別化 (Differentiation)

| 代替 | 限界 | このスキルが上回る点 |
|---|---|---|
| エージェントサイトを手動巡回 | 出典・取得日を残さない、1 サイトの偏りに気づけない | 複数ソースを突き合わせ、出典・取得日・偏り注記付きで残る |
| `deep-research` スキル | 汎用調査で、単価の正規化（月額換算・条件揃え）や免責の定型がない | 単価ドメインに特化した正規化・出力フォーマット・免責を固定 |
| エージェント担当者に聞く | 担当者側の利害があり中立でない、根拠が口頭 | 公開情報ベースで根拠 URL が残る |

---

## 成功条件 (Success Criteria)

- [ ] 出力レポートに、独立した 3 ソース以上の出典 URL と取得日が記載されている
- [ ] 相場レンジが「万円/月」に正規化された数値レンジ（下限–上限）として提示され、ソース別内訳表がある
- [ ] 出力末尾に免責（相場の保証・交渉助言ではなく公開情報の整理である旨）が毎回含まれている
- [ ] ユーザーが追加で「出典は？」「いつ時点？」と聞き返す必要がない（出典・取得日が本文に埋まっている）
- [ ] カバーできなかった条件（地域・稼働形態など）が「調査範囲外」として明示されている

---

## 失敗モード予測 (Failure Modes)

| 失敗 | 起きる条件 | 回避策 |
|---|---|---|
| 古い相場情報を最新として提示 | 検索結果が数年前の記事・統計に当たる | ページ内の公開日/更新日を必ず確認し、レポートに「情報の時点」を明記。1 年超のデータは「古い可能性あり」と注記し、直近データを優先 |
| サンプル偏りによるレンジの歪み | 特定エージェント 1 社のみ、または高単価案件が目立つ広告的ページに依存 | 独立 3 ソース以上を必須化し、各ソースの性質（エージェント掲載単価か、成約実績か、アンケートか）を注記。1 ソースしか取れない場合はその旨を明示 |
| 単価表記の混同（月額/時給、税込/税抜、精算幅） | ソースごとに表記基準が異なる | 「万円/月・税抜・準委任・週5 相当」に正規化し、換算した場合は換算方法を注記 **(ASSUMPTION: 国内 SES/準委任の慣行では月額・税抜表記が支配的とみなす)** |
| 相場保証・交渉助言と受け取られる | 「あなたの適正単価は X 万円」の断定表現 | 断定表現を禁止し、「掲載レンジ」「候補」「観点」の語彙に統一。免責を出力テンプレートに固定で埋め込む |
| サイトへの過度なアクセス・規約違反 | 案件検索を大量にクロールする | WebSearch 中心に、WebFetch は要点ページに限定（1 ソースあたり数ページ）。robots.txt・利用規約を尊重 |

## 必要なリソース (Resources)

- 外部 API / CLI: WebSearch / WebFetch（追加 API キー不要）
- 権限: なし（公開 Web の閲覧のみ）
- 参照すべき公開知識: フリーランスエージェント各社の単価統計・案件検索ページ、
  フリーランス白書等の業界調査レポート **(ASSUMPTION: レバテックフリーランス・フリーランスHub（フリーランススタート）・Midworks 等の公開単価情報を主要ソース候補とする。実行時に到達可能なものを使う)**
- ローカル前提: なし（リポジトリ外の情報を参照するのみ）

---

## 実装メモ (Implementation Notes)

- ディレクトリ構成: `SKILL.md` + `templates/rate-report.md`（出力雛形）。scripts は不要
  （決定論的に計算すべき数値処理がなく、収集と整理が主のため）
- persona `edith` の原則（出典明示・一次情報優先・事実と推測の区別・カバー範囲の明示）を
  出力フォーマットに構造として埋め込む
- 免責文はテンプレートに固定文言として置き、スキル側で削除・省略を禁止する
- ultron-high-dividend-stock-screener の免責スタイル（毎回明示・冒頭スタンス＋末尾免責）を踏襲

## 自己進化を入れるか (Self-Evolution Scope)

- [ ] パイプラインあり
- [x] パイプラインなし — 理由: 1 回の調査で完結するタスクであり、実行のたびに Web の最新情報を
  取り直すため、過去実行ログから学習して SKILL.md を書き換える価値が薄い。ソース候補の追加・淘汰は
  頻度が低く、ユーザーが SKILL.md のソース一覧を手動更新すれば足りる。相場データ自体をローカルに
  溜め込むと「古い相場情報」の失敗モードをむしろ助長するため、状態を持たない設計とする。
