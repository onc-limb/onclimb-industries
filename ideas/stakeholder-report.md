# stakeholder-report (依頼者向け 日次報告ジェネレータ)

> フリーランスエンジニアが、worklog の「プロジェクト整理情報(project digest)」を入力に、
> **案件ごと・日付ごとの報告記録(Markdown)を積み上げ**、そこから**非エンジニアの依頼者向けに
> 専門用語を排した日次報告スライド(HTML)を1本生成**するための、2段構えのスキル群。
> 「何の課題に・何をして・今どうなっていて・この先どうするか・どんな障害があったか」を、
> 依頼者が30秒で掴める粒度で伝えることを目的にする。

---

## 0. このアイデアの背景 (なぜ必要か)

SES / 受託のフリーランスは、技術的な作業内容とは**別の言語**で依頼者に報告する義務がある。
依頼者(発注元・上長・顧客)は多くが非エンジニアで、知りたいのは実装の詳細ではなく
「**頼んだことが進んでいるか / 困りごとはないか / 次に何が起きるか**」だけ。

一方で worklog が既に生成している project digest は、報告の素材として**ほぼ理想的な構造**を持つ:

| project digest の見出し | 報告で使う観点 |
|---|---|
| 目的・背景 | なぜこの作業に取り組んだか(課題) |
| 作業内容(進捗ステータス付き) | 何をしたか |
| 最終的にどうしたか | 今どうなっているか(結果・現状) |
| つまづき・問題(Problem) | どんな障害があったか |
| 次の予定・次への改善(Next / Try) | この先何をするか |
| 課題・判断待ち | 依頼者に判断を仰ぐ事項 |
| 成果サマリ(上長・顧客向け抜粋) | 既に「報告向け」に圧縮された層 |

つまり worklog は「報告書を書くための整理情報」であって**報告書そのものではない**
([[work-log-system-instructions]] の位置づけ通り)。本スキルはその後段に立ち、
digest を**依頼者言語の報告物**へ変換する最終工程を担う。

### なぜ worklog や problem-essence-organizer で代替できないか

- **worklog**: 技術者自身が後で過程を辿るための網羅的な digest。専門用語をむしろ温存する設計で、
  非エンジニア向けではない。複数日・複数案件を横断した「日次の1枚」も作らない。
- **problem-essence-organizer**: 単発の思考整理(課題発見→手段検討)で、クライアント向け HTML も出すが、
  対象は「**ひとつの論点をMinto流で構造化**」する単発タスク。日々の進捗を時系列で積み上げる
  運用ログ的な用途ではない。([[problem-essence-organizer]] と出力フォーマットは一部似るが運用が別)
- **giziroku**: 会議の文字起こし→議事録。入力が transcript で、進捗報告ではない。([[giziroku]])

→ 「**日々の作業を、案件横断で、非エンジニア向けに、定型フォーマットで報告し続ける**」という
運用は、既存スキルのどれもカバーしていない。

## 1. 全体設計 — 2段階処理

```
[worklog project digest]  ─┐
[前日までのStage1記録]     ─┼─►  Stage 1  ──►  案件×日付の報告記録(Markdown)
[ユーザーへの対話ヒアリング] ─┘     (記録)        records/<project>/<date>.md
                                                      │
                              対象日の全案件分を集約    ▼
                                            Stage 2  ──►  日次報告スライド(HTML)
                                            (清書)        decks/<date>.html
```

- **Stage 1 = 記録**: 案件ごと・対応日ごとに、課題/作業/結果/障害/次の一手を Markdown で残す。
  入力は ①worklog の project digest、②同案件の前日までの Stage1 記録(継続性のため)、
  ③digest に無い情報は対話でヒアリング。**技術者が読める正確な一次記録**(信頼の源泉)。
- **Stage 2 = 清書**: 対象日の Stage1 記録を**案件横断で集約**し、非エンジニア向けに
  専門用語を一般語へ言い換えた **HTML スライド1本**を定型テンプレートで生成。

### ★ 決定: スキルは「2つに分ける」(2026-06-27 ユーザー承認済み)

結論として **2スキルに分割**する。理由:

| 観点 | Stage 1 (記録) | Stage 2 (清書スライド) |
|---|---|---|
| 起動頻度 | 毎営業日(記録は毎日残す) | 報告するタイミングのみ(毎日とは限らない) |
| 主作業 | 事実の正確な抽出 + 対話ヒアリング | 言い換え(脱専門用語) + テンプレ整形 |
| 読み手 | 技術者本人(一次記録) | 非エンジニアの依頼者 |
| 出力 | Markdown(構造化テキスト) | 自己完結HTML(配布物) |
| 主な失敗 | 事実誤り・ヒアリング漏れ | 専門用語の残存・テンプレ崩れ |
| 入力の単位 | 1案件 | 対象日の全案件 |

- 失敗モードと品質基準が**根本的に違う**(片方は事実精度、片方は翻訳精度+体裁)。
  1スキルに混ぜると評価軸が混線し、自己進化のフィードバックも噛み合わない。
- **記録だけ残してスライドは作らない日**が普通にある(社内日や、週次でまとめて報告する運用)。
  分けておけば Stage 1 を独立して毎日回せる。
- この repo は既に「**digest → 後段スキルで報告物**」の二段構え(worklog → knowledge-base /
  giziroku)を採っている。本件もその慣習に沿う。Stage1 が digest 相当、Stage2 が報告物相当。
- 連携は疎結合: Stage2 は Stage1 の出力(決まった見出しの Markdown)だけを入力にする。
  Stage1 を改善しても Stage2 は壊れにくい。

> 補足: 「`今日の報告まとめて`」の一声で **Stage1→Stage2 を連続実行**したい体験も成立させられる。
> 2スキルでも、Stage2 のフローが「未作成の当日 Stage1 を先に作るか確認 → 作る → 集約」と
> 呼び出せばよい。分割は運用上の連続性を損なわない。

スキル名(仮):
- Stage 1: `report-record`(または `client-record` / 「日次案件記録」)
- Stage 2: `report-deck`(または `daily-deck` / 「日次報告スライド」)

## 2. 場面 (When)

両スキルとも **ユーザーが意図的に起動する**(自動起動・常駐しない)。

### Stage 1 (記録) が呼ばれる場面
- その日の作業を終え「**今日の作業を案件ごとに記録して**」と言ったとき
- worklog で digest を生成した直後に「**これを報告用に整理して**」と続けたとき
- 「**A案件の今日の分を記録**」と特定案件を指したとき
- digest に無い背景(依頼者とのやり取り、口頭の判断)を「**補足して記録**」したいとき

### Stage 2 (清書スライド) が呼ばれる場面
- 「**今日の報告スライド作って**」「**依頼者向けの日報スライドにして**」と言ったとき
- 「**6/27 の分を報告資料に**」と日付を指したとき
- 週次で「**今週分まとめて報告スライドに**」(日付範囲指定)したとき

### 想定トリガー文 (実際のユーザー発話例)
1. 「今日の作業を案件ごとに記録して」
2. 「worklog の整理を報告用に落とし込んで」
3. 「noboru-note の今日の分、報告記録に追加して」
4. 「今日の依頼者向けスライド作って」
5. 「6/27 の報告資料を HTML スライドで」
6. 「今週の作業、非エンジニア向けにまとめて」
7. 「報告記録から日報スライド生成して」

> 起動の明示性: description には「worklog の整理情報から**依頼者向け報告**を作るとき」と限定し、
> worklog(作業ログ整理)や giziroku(議事録)と誤起動しないようにする。

## 3. 解決する課題 (Problem)

- **現状**:
  - digest や記憶を頼りに、報告のたびに専門用語を噛み砕いて文章を書き起こす。
    案件が複数並行すると「どの案件が今どこまで来ているか」を毎回頭の中で再構成する。
  - 報告のフォーマットが毎回ブレ、依頼者が読む位置を探す手間が出る。
  - 「障害・遅延・判断待ち」を伝えそびれ、後で大きな手戻りや信頼低下につながる。
  - HTML スライドを手で作ると体裁に時間を溶かす上、装飾過多/AI 臭で逆に安っぽくなる。
- **痛みの種類**: 時間(清書・体裁) / 認知負荷(案件横断の現状把握) / 見落とし(障害・判断待ちの伝え漏れ) / 信頼(専門用語で伝わらない)
- **影響範囲**: クライアント共有物 → 品質要求が高く、誤りや伝わらなさが直接 信頼・継続に響く。

---

## 4. 解決策 (Solution)

### Stage 1 — 案件×日付の報告記録 (Markdown)

1. **対象の確定**: 日付(既定=当日)と案件を受け取る。指定が無ければ、対象日の
   project digest が存在する案件を列挙し、どれを記録するかをユーザーに確認。
2. **入力の収集**:
   - `worklog-data/digests/project/<project>_<date>.md` を読む(主入力)。
   - 同案件の**前日までの Stage1 記録**(`records/<project>/*.md`)を読み、継続中タスクの
     文脈(前日「次にやること」と当日の対応の接続)を引き継ぐ。
3. **不足情報の対話ヒアリング**: digest だけでは埋まらない項目を `AskUserQuestion` で確認。
   例) 依頼者からの口頭フィードバック、当日の対外的なやり取り、digest の `判断待ち=なし` の真偽。
   **digest に無い事実を勝手に創作しない**(空欄は「記録なし」)。
4. **記録生成**: 下記の固定見出しで `records/<project>/<date>.md` を生成/追記。
   この段階は**技術者が読める一次記録**で、用語の言い換えはまだしない(Stage2 の責務)。
5. **完了提示**: 生成パスと、Stage2 へ渡せる状態かどうか(障害・判断待ちの有無)を提示。

#### Stage 1 記録のフォーマット (案: 1案件1日1ファイル)
```markdown
# <案件名> 日次記録 <YYYY-MM-DD>

## 取り組んだ課題 (なぜこの作業をしたか)
- <背景・依頼内容・前日からの継続。digest「目的・背景」+ 前日「次にやること」>

## やったこと
- [完了/進行中/着手のみ] <作業内容。digest「作業内容」を要約>

## 今どうなっているか (結果・現状)
- <digest「最終的にどうしたか」。完了/未完了を明示>

## 障害・つまづき
- <digest「つまづき・問題」。原因と影響。無ければ「なし」>

## この先やること
- <digest「次の予定」。翌営業日以降>

## 依頼者への相談・判断待ち
- <digest「課題・判断待ち」。承認が要る事項。無ければ「なし」>

<!-- meta: source=<digest path>, prev=<前日記録path>, heard=<ヒアリングで補った項目> -->
```

### Stage 2 — 日次報告スライド (HTML)

1. **対象日の確定**: 日付(既定=当日)/日付範囲を受け取る。
2. **集約**: その日の全案件の `records/<project>/<date>.md` を読み込む。
   Stage1 記録が無い案件があれば「先に Stage1 を作るか」をユーザーに確認。
3. **依頼者言語への変換**(★このスキルの核):
   - 専門用語を**一般語へ言い換える**(用語辞書 `glossary.yaml` を参照・追記)。
     例) 「Next.js のビルドを通した」→「画面を表示する仕組みが動く状態にした」、
     「Supabase 認証を流用」→「既存のログインの仕組みをそのまま使える形にした」。
   - 残ってはいけない語(固有技術名・略語)を検出して警告(自己チェック)。
4. **スライド生成**: 固定テンプレート(後述 §5)に流し込み、自己完結 HTML 1ファイルを生成。
5. **完了提示**: 生成パスと、ブラウザで開けば即配布できる旨を提示。スライド枚数・案件数を要約。

#### Stage 2 スライドの論理構成 (案)
1. 表紙 — 日付・報告者名(タイトルのみ、装飾なし)
2. 今日のサマリ — 「今日の要点」を箇条書き3点以内(全案件横断のハイライト)
3. (案件ごとに1枚) — 案件名 / やったこと / 今どうなっているか / この先 / 相談事項
4. 全体の次アクション・ご相談 — 依頼者に判断を仰ぐ事項を集約した最終スライド

### 入力 (Inputs)

| 名前 | 型 | 必須 | 出どころ | 例 |
|---|---|---|---|---|
| date | string(YYYY-MM-DD) | no(既定=当日) | ユーザー発話 | "2026-06-27" |
| project | string | no | ユーザー発話 / digest 列挙 | "noboru-note" |
| project digest | markdown | yes(Stage1) | worklog-data/digests/project/ | digest ファイル |
| 前日までの記録 | markdown | no | records/<project>/ | 継続タスクの文脈 |
| ヒアリング回答 | string | no | AskUserQuestion | 依頼者の口頭FB |

### 出力 (Outputs)

| 名前 | 形式 | どこに置く | 例 |
|---|---|---|---|
| 案件日次記録 | markdown | `<data>/records/<project>/<date>.md` | 1案件1日 |
| 日次報告スライド | 自己完結HTML | `<data>/decks/<date>.html` | 依頼者へ配布 |

> データ置き場(案): worklog の `worklog-data/` に倣い、repo 直下に `report-data/`
> (`records/` `decks/` `glossary.yaml`)。環境変数で上書き可。

## 5. HTML スライド テンプレートの方針 (2026-06-27 ミニマル案で承認済み)

「AI 感・Claude 感を出さない / 余計な装飾なし」を最優先に、下記の**ミニマル単一HTML案**で確定。

### 確定している方針(ユーザー指示)
- **専門用語を使わない**。一般語で解説。残語は自己チェックで検出。
- **毎回同じフォーマット**。テンプレートを1つ固定し、内容だけ差し込む。
- **AI 臭・装飾過多を避ける**。絵文字・グラデーション・過剰なアイコン・キャッチコピー禁止。

### 技術形式の提案
- **単一の自己完結 HTML ファイル**(CSS を `<style>` で内包、外部CDN・JS依存なし)。
  → メール添付・チャット共有でそのまま開ける。オフラインで崩れない。ライブラリ由来の
  「いかにも」感が出ない。reveal.js 等のスライドFWは**使わない**(AI/テンプレ臭・重さの元)。
- ページ送りは最小限の素の JS(←→キー / クリックで `section` を切替)のみ。
- 印刷/PDF 化したいニーズに備え、`@media print` で1スライド1ページ。

### デザインの提案(ミニマル)
- 白背景・黒文字・1アクセント色(濃紺など)1色のみ。影・角丸・グラデなし。
- フォントはOS標準サンセリフ(`system-ui`)。タイトル大・本文中、余白広め。
- 1スライド = 見出し + 箇条書き3〜5行。文字を詰め込まない。
- ステータスは色アイコンでなく**語**で示す(「完了」「進行中」「相談したいこと」)。

### テンプレート構造(差し込みポイント案)
```
[表紙]        {{date}} / {{reporter}}
[サマリ]      今日の要点: {{highlights[]}}
[案件×N]      {{project.name}}
              ・やったこと {{project.did[]}}
              ・今どうなっている {{project.status[]}}
              ・この先 {{project.next[]}}
              ・ご相談 {{project.ask[]}}    (無ければ非表示)
[締め]        ご相談・ご判断いただきたいこと: {{all_asks[]}}
```

## 6. 想定ユーザー (Who)

- **主**: SES / 受託で複数案件を並行するフリーランスエンジニア(= 本人)。worklog 運用済み。
- **副(読み手)**: 依頼者・発注元・上長。多くは非エンジニアで、技術詳細を読む前提がない。

ユーザーが持っている知識: 自分の作業内容・技術。持っていない知識: 依頼者が何を専門用語と
感じるかの線引き(→ glossary で運用しながら学習)。

## 7. 既存ツール・スキルとの差別化 (Differentiation)

| 代替 | 限界 | このスキルが上回る点 |
|---|---|---|
| worklog project digest | 技術者向け一次整理。用語温存・日次横断スライドは作らない | 依頼者言語へ変換し、案件横断の日次報告物にする |
| problem-essence-organizer の HTML | 単発論点の Minto 整理。日々の進捗運用ではない | 案件×日付で継続記録し、定型日報スライドを量産 |
| giziroku | 入力が会議文字起こし。進捗報告ではない | 入力が作業digest。進捗・障害・次手を報告 |
| 手書き / 汎用スライドツール | 体裁に時間、フォーマットがブレる、AI臭・装飾過多になりがち | 固定ミニマルテンプレ・脱専門用語を自動化 |

## 8. 成功条件 (Success Criteria)

- [ ] Stage1: digest 1本 + 前日記録から、固定見出しの記録が生成され、digest に無い事実を
      創作していない(空欄は「記録なし」になっている)。
- [ ] Stage2: 生成 HTML に**技術固有名詞・略語が残っていない**(自己チェックで0件、または明示警告)。
- [ ] Stage2: 出力が**単一HTMLで外部依存ゼロ**、ブラウザで開いて崩れず、印刷で1スライド1ページ。
- [ ] 毎回**同一テンプレート**で出力され、案件数が変わってもレイアウトが破綻しない。
- [ ] 依頼者が「で、結局どうなってるの?」と聞き返さずに済む(障害・現状・次手・相談が揃う)。

## 9. 失敗モード予測 (Failure Modes)

| 失敗 | 起きる条件 | 回避策 |
|---|---|---|
| 専門用語が残る | digest の技術語をそのまま転記 | glossary 参照 + 残語検出の自己チェックを必須化。検出時は言い換え or 警告 |
| digest に無い事実を創作 | 情報不足を埋めようと推測 | 不足は「記録なし」固定 + 対話ヒアリングでのみ補う。創作禁止を明記 |
| 継続タスクの文脈が切れる | 前日記録を読まない | Stage1 で同案件の直近記録を必ず読み、前日「次にやること」と接続 |
| テンプレ崩れ(案件多数/長文) | 案件10件・箇条書き過多 | 1スライド箇条書き上限を設け超過は要約。案件ごと改ページで破綻回避 |
| AI臭・装飾過多 | 凝ったCSS/絵文字/FW採用 | ミニマル固定CSS・外部FW禁止・絵文字禁止をテンプレで強制 |
| 機密の漏えい | digest 内の伏字や実名 | worklog のマスキング済み出力を前提。Stage2 で実名/顧客名を再チェック |
