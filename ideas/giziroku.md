# giziroku (議事録ジェネレータ / transcript → 議事録)

> ビジネスの当事者が、Plaud / Teams 等の音声文字起こしファイルを渡し、
> **対話で参加者と前提を確定したうえで**、そのまま社内共有できる議事録
> (目的・会話の流れ・全体要約・決定事項・TODO) を 1 本生成するためのスキル。
> 対象は会議録音が主だが、AI エージェントとの電話・同僚との雑談も同じ流儀でまとめられる。

---

## 0. このアイデアの背景 (なぜ必要か)

同梱の実サンプル `06-22 週次会議_..._-transcript.txt` を読んで設計上の前提を確定した。

- **話者が匿名**: ラベルは `Speaker 1 / 2 / 3` のみ。誰が何の役割かは文字起こしからは一切わからない。
  → だからユーザーが「開始」を宣言し、**参加者を対話で確定する**工程が必須。
- **音声誤変換が多発**: 「船だね / フラ種」(=船種)、「クラウドフォーメーション」(=CloudFormation)、
  「インファレンス」(=inference)、「BM 脆弱性診断」「SED / SES」など固有語・専門語が壊れている。
  → 誤変換の補正と、補正したことの明示 (推測痕跡) が品質を左右する。
- **構造が埋もれている**: 決定事項・TODO・期日が、冗長で言いさしの多い口語の中に散在する。
  例) 「水曜日にドラフトをお見せする」「先に推論フローだけ直す」「広報系は今日連絡を取る」
  → 時系列の流れと、決定/TODO の抽出を分けて行う必要がある。
- **機密が混じる**: 実名・顧客名・社内固有名が頻出。既存システム同様にマスキング方針が要る
  ([[knowledge-base-idea]] §2、[[work-log-system-instructions]] のマスキング処理と地続き)。

## 1. 場面 (When)

このスキルは **ユーザーが意図的に起動する**（自動起動・常駐はしない）。文字起こしファイルが
手元にあり、それを共有可能な議事録に変換したい、という明確な意思があるときだけ走る。

- ユーザーが「この文字起こしから議事録を作って」と言い、ファイルパスを添えたとき
- Plaud / Teams で書き出した `*-transcript.txt` を作業ディレクトリに置いた直後
- 「さっきのエージェントとの電話、要点だけ議事録にして」と言ったとき
- 「同僚との雑談ログ、決まったことだけ拾ってまとめて」と言ったとき
- 週次会議の録音を「決定事項と TODO を抜き出して共有用にして」と頼まれたとき

### 想定トリガー文 (実際のユーザー発話例)

1. 「この文字起こしから議事録作って」
2. 「`06-22 週次会議...-transcript.txt` を議事録にして」
3. 「Plaud の書き起こし、決定事項と TODO 抜いてまとめて」
4. 「エージェントとの電話の文字起こし、要点まとめて」
5. 「雑談ログだけど、決まったことだけ議事録風に拾って」
6. 「Teams の文字起こし、参加者だけ確認してから議事録化して」

> 起動の明示性: description には「ユーザーが文字起こしファイルを指して議事録化を依頼したとき」と
> 限定して書き、ログ収集・要約系の他スキル ([[work-log-system-instructions]]) と誤起動しないようにする。

## 2. 解決する課題 (Problem)

- **現状**:
  - 文字起こしを手で読み返し、誰の発言かを推測しながら、決定事項と TODO を拾い、清書する。
    1 時間の会議で 30〜60 分かかる。
  - 話者が `Speaker N` のままなので、後から読むと「誰が何を引き受けたか」が辿れない。
  - 専門語の誤変換が残ったまま共有され、読み手に伝わらない/信頼を損なう。
- **痛みの種類**: 時間 (清書コスト) / 認知負荷 (口語からの構造抽出) / 見落とし (TODO・期日の取りこぼし) / 手戻り (誤変換・誤帰属による訂正)
- **影響範囲**: チーム共有物 (社内・顧客への報告物になりうる) → 品質要求が高い。

---

## 3. 解決策 (Solution)

「**対話で前提を固める → 機械的に下処理 → 構造抽出 → 議事録生成**」の 4 段。
最初の対話フェーズを必須にすることで、匿名話者・誤変換・会議種別の不確実性を入口で潰す。

### Phase A. 起動と一次パース (機械処理)
1. ユーザーが起動し、対象ファイルパスを受け取る。
2. 文字起こしを読み込み、話者ラベル (`Speaker N`)・タイムスタンプ・発言量を集計する。
3. **話者分離が無い (ベタ書きの) 文字起こしでも推定する**:
   ラベルが無い場合は、話題転換・敬体/常体・呼びかけ・ターン構造から発言者の切れ目を推定し、
   暫定話者 (`推定話者A/B/...`) を割り当てる。推定であることを明示し、Phase B の確認に回す。
4. 話者ごとに **特徴的な発言の抜粋**（役割を推測できる手がかり）を 1〜2 個ずつ拾う。
   例) Speaker 2 →「BM 脆弱性診断の指摘に対応」「CloudFront の設定を更新」= 実装担当らしい。

### Phase B. 対話で前提確定 (ユーザーとの往復) ★このスキルの核
5. **会議種別**を確認する: 会議 / AI エージェントとの電話 / 雑談 (出力テンプレを切り替える)。
6. **参加者マッピング**を対話で行う。話者ごとに抜粋と発言量を提示し、
   「Speaker 1 はどなたですか？役割は？」と聞く。`AskUserQuestion` で候補（`roster.yaml`
   から推測した候補＋自由入力）を出す。匿名のままで良い話者は「匿名のまま」を許可する。
   話者分離を推定した場合は「この切り分けで合っているか」も併せて確認する。
7. **会議の目的**を確認する。スキル側が文字起こしから推定した目的案を提示し、補正してもらう。
8. **日時・タイトル・公開範囲（マスキング要否）**を確認する。

### Phase C. 下処理 (機械処理 + 推測痕跡)
9. **誤変換の正規化**: 用語集 (`glossary`) と文脈で固有語・専門語を補正する。
   補正箇所は本文に残さず、末尾の「補正メモ」に `元表記 → 補正` で列挙（推測を明示）。
10. **マスキング**: 公開範囲が社外/不明なら、実名・顧客名・固有名を `<REDACTED:種別>` 化。
    既存方針 ([[work-log-system-instructions]] §マスキング) を踏襲。
11. 確定した参加者名で `Speaker N` / `推定話者X` を実名/役割に置換。

### Phase D. 構造抽出と議事録生成
12. 以下の構造を抽出して議事録を組み立てる:
    - **会議の目的** (Phase B で確定したもの)
    - **会話の流れ** (時系列・議題単位のダイジェスト。脱線/雑談は圧縮)
    - **全体要約** (3〜5 行のエグゼクティブサマリ)
    - **決定事項** (合意された結論。決めきれていないものは「保留」として別掲)
    - **TODO / アクションアイテム** (担当者・期日付き。期日不明は `期日: 未定` と明記)
    - **補足** (誤変換補正メモ・不明点・推測箇所)
13. **1 ファイル = 1 議事録**として `giziroku/` (このスキル専用の出力ディレクトリ) に出力する。
    ファイル名は `YYYY-MM-DD_<会議タイトル>_議事録.md`。同一会議の分割録音も 1 議事録に束ねる。
14. **完了条件の判定**: 決定事項と TODO が 0 件のときは「抽出できなかった」ではなく
    「雑談主体で決定/TODO なし」と判断根拠を添えて明示する（空欄で終わらせない）。

### 入力 (Inputs)

| 名前 | 型 | 必須 | 出どころ | 例 |
|---|---|---|---|---|
| transcript_path | string | yes | ユーザー発話 | `"06-22 週次会議...-transcript.txt"` |
| meeting_type | enum(meeting/agent_call/chat) | yes (対話で確定) | 対話 | `meeting` |
| participants | map(Speaker→name/role) | yes (対話で確定) | 対話 | `{Speaker1: "本人/PM"}` |
| purpose | string | no (推定→確認) | 対話 | "週次進捗と役員報告資料の方針すり合わせ" |
| visibility | enum(internal/external/unknown) | yes (対話で確定) | 対話 | `external` → マスキング ON |
| glossary | map | no | 横断共有資産 | `{"船だね":"船種","インファレンス":"inference"}` |

### 出力 (Outputs)

| 名前 | 形式 | どこに置く | 例 |
|---|---|---|---|
| minutes | markdown | `giziroku/` (専用ディレクトリ) + チャット要約 | `giziroku/2026-06-22_週次会議_議事録.md` |
| correction_log | markdown 節 | 議事録末尾「補足」 | `船だね → 船種` |
| open_questions | markdown 節 | 議事録末尾 | 「広報系の打ち合わせ日が未確定」 |

## 4. 出力フォーマット案 (議事録テンプレート)

会議種別で見出しを出し分ける。下は `meeting` の標準形。

```markdown
# <会議タイトル>
- 日時: 2026-06-22
- 種別: 週次会議
- 参加者: 本人(PM) / 大賀(実装) / 南野(広報窓口) / 他1名(匿名)
- 公開範囲: 社外共有可 (マスキング済み)

## 会議の目的
<1〜2 行>

## 全体要約
<3〜5 行のエグゼクティブサマリ>

## 会話の流れ
1. <議題A>: <要点>
2. <議題B>: <要点>
   - 脱線/雑談は「(雑談: 〜) 」と 1 行に圧縮

## 決定事項
- [D1] <決めたこと> (決定者: 本人)

## 保留・継続検討
- [P1] <決めきれていない論点と次の判断材料>

## TODO / アクションアイテム
- [ ] <タスク> — 担当: 大賀 / 期日: 本日中
- [ ] <タスク> — 担当: 本人 / 期日: 水曜 (ドラフト)

## 補足
### 誤変換の補正メモ (推測含む)
- 船だね / フラ種 → 船種
- クラウドフォーメーション → CloudFormation
### 不明点・要確認
- <...>
```

- `agent_call`: 「決定事項」を「エージェントの結論/提案」に、「参加者」を「本人 / AI エージェント」に。
- `chat`: 「会議の目的」を省略可、「決定事項」が薄い前提で「拾えた合意・気づき」中心に。

## 5. 想定ユーザー (Who)

- 主: ビジネスの当事者本人（会議の参加者で、文脈を補える人）。SES 参画中の進捗・報告にも使う。
- 副: 後から議事録だけ読むチームメンバー / 報告先。

ユーザーが持っている知識: 参加者が誰か、会議の背景・固有語の正しい表記。
ユーザーが持っていない/面倒な作業: 口語からの構造抽出・清書・誤変換の網羅的補正。
→ だから「前提はユーザーに対話で出させ、抽出と清書は機械が担う」分担にする。

## 6. 既存ツール・スキルとの差別化 (Differentiation)

| 代替 | 限界 | このスキルが上回る点 |
|---|---|---|
| Plaud / Teams の自動要約 | 話者は匿名のまま、誤変換そのまま、決定/TODO の粒度が浅い | 参加者を対話確定し、誤変換補正＋決定/TODO を担当・期日付きで抽出 |
| `worklog` ([[work-log-system-instructions]]) | Claude Code の操作ログが対象。会議音声は範囲外。要約は「報告書の素材」止まり。**giziroku とは別系統として独立させる** | 音声文字起こしを入力に、そのまま共有できる議事録を完成形で出す |
| 05-19 omni-archive 形式 | ナレッジ抽出向け（タグ付き知見）で、ビジネス議事録の体裁ではない | 目的/流れ/要約/決定/TODO のビジネス定型に整える |
| 素の LLM 要約 | 匿名話者・誤変換・機密をケアしない | 入口の対話で前提を固定し、マスキングと推測痕跡を担保 |

## 7. 成功条件 (Success Criteria)

- [ ] 参加者マッピングが対話で確定し、出力中に `Speaker N` / `推定話者X` が 1 つも残らない（匿名指定を除く）。
- [ ] 話者分離が無い文字起こしでも、暫定話者を推定して Phase B の確認に載せられる。
- [ ] 決定事項・TODO が、サンプル `06-22` で人手抽出した基準リストと 8 割以上一致する。
- [ ] TODO に担当者が付く（不明なら「担当: 不明」と明示、空欄にしない）。
- [ ] 誤変換補正メモに、サンプルの主要誤変換（船種 / CloudFront / inference 等）が載る。
- [ ] 公開範囲 external 指定時、実名・顧客名が本文に素のまま残らない。
- [ ] 出力が `giziroku/` に 1 ファイル 1 議事録で保存される。
- [ ] ユーザーが続けて大きな手直しを指示せずに共有できる割合が体感 8 割。

## 8. 失敗モード予測 (Failure Modes)

| 失敗 | 起きる条件 | 回避策 |
|---|---|---|
| 話者の取り違え | 文字起こしの話者分離が不正確（同一人に複数ラベル等） | Phase B で発言抜粋を見せて確認。「同一人物の統合」操作を許可 |
| 話者分離なしの推定が外れる | ベタ書きでターン境界が曖昧 | 推定話者である旨を明示し、Phase B で切り分けの妥当性をユーザーに確認させる |
| 誤変換を直しすぎて原意を変える | 文脈推定が外れる | 本文は最小補正、補正は末尾メモに `元 → 補正` で残し原文を辿れるように |
| 決定とアイデアの混同 | 言いさし・仮定法が多い口語 | 「合意/決定」は明示的合意のみ。仮のものは「保留・継続検討」に分離 |
| 機密の漏れ | 公開範囲の確認を飛ばす | visibility を必須確認。未確認なら external 扱い（安全側）でマスキング |
| 雑談で決定/TODO ゼロを「失敗」と誤認 | chat 種別 | 0 件は根拠付きで明示、種別に応じて見出しを省く |
| 長大な文字起こしで破綻 | 1 時間超・数万字 | 議題/時間ブロック単位で分割抽出 → 1 議事録にマージ |
| 対話が冗長で離脱 | 質問が多すぎる | 確認は「種別・参加者・公開範囲・目的」の 4 点に絞り、推定値を初期選択に |

## 9. 必要なリソース (Resources)

- 外部 API / CLI: 不要（ローカルファイル読み込みのみ）。`AskUserQuestion` で対話。
- 権限: 対象文字起こしファイルの読み取り、`giziroku/` への書き込み。
- 参照すべき公開知識: 議事録の定型（目的/決定/アクションアイテムの書き分け）。
- ローカル前提: 文字起こしファイルが手元にあること。話者ラベルの有無は問わない（無ければ推定）。
- 既存資産の流用:
  - マスキング規約 → [[work-log-system-instructions]] / [[knowledge-base-idea]] §2。
  - 対話の作法（全肯定しない・選択肢の根拠を出す）→ [[problem-essence-organizer]] のスタイル。

## 10. 実装メモ (Implementation Notes)

- ディレクトリ構成 (確定方針):
  ```
  .claude/skills/giziroku/
    SKILL.md
    templates/
      meeting.md        # 会議用議事録テンプレ
      agent_call.md     # エージェント電話用
      chat.md           # 雑談用
    references/
      dialogue_flow.md  # Phase B の確認 4 点と質問文
      masking.md        # マスキング種別 (worklog の規約を流用、運用は独立)
      extraction.md     # 決定/TODO/保留の判定基準
    config/                  # 自己進化資産の【雛形】(git 追跡・機密なし)
      glossary.example.yaml  # 誤変換辞書の汎用シード
      roster.example.yaml    # 参加者名簿の空テンプレ
    scripts/
      parse_transcript.py   # 話者集計・抜粋・タイムブロック分割・話者なし推定

  giziroku/              # ★このスキル専用のデータ置き場 (git 管理外、worklog-data とは別)
    transcripts/          # 入力 (未処理の文字起こし)
    processed/            # 退避 (議事録化が済んだ原文。削除しない)
    minutes/              # 出力 (YYYY-MM-DD_<会議タイトル>_議事録.md)
    config/               # 自己進化の【実データ】(蓄積先。機密を含むため git 管理外)
      glossary.yaml       # 誤変換 → 正規表記。★プロジェクト横断で共有・蓄積
      roster.yaml         # 過去参加者名簿。★横断共有。Speaker 推定の候補に使う
  ```
- worklog とは**完全に別系統**として実装する（収集・分類・アーカイブのパイプラインを共有しない）。
- 話者集計・タイムブロック分割・話者なし推定は決定的処理として可能な範囲を script 化
  (`parse_transcript.py`)。構造抽出・要約・補正判断・最終的な話者推定は LLM が担当。
- `glossary` / `roster` は **プロジェクト横断で 1 つを共有**し、使うほど精度が上がる資産。
  機密 (固有名) は roster に入りうるため、**実データは `giziroku/config/`（git 管理外）に置き**、
  スキル同梱は `config/*.example.yaml`（汎用シード・機密なし）のみを git 追跡する。
  読み込みは雛形＋実データをマージし、追記（自己進化）は実データ側だけに行う。
- 入力形式の揺れ（Plaud / Teams で話者ラベル・タイムスタンプ書式が違う）を parser で吸収する。

## 11. 自己進化を入れるか (Self-Evolution Scope)

- [x] パイプラインあり

理由: 実行ごとに「誤変換補正 (`glossary`)」と「参加者名簿 (`roster`)」が蓄積し、
回を重ねるほど対話の質問数が減り補正精度が上がる、という明確な学習余地がある。
[[problem-essence-organizer]] / self-evolving-skill-creator のログ→自己改訂パイプラインを踏襲し、
- 確定した参加者マッピングを `roster.yaml` に追記（横断共有）、
- ユーザーが訂正した誤変換を `glossary.yaml` に追記（横断共有）、
- 「どの確認質問が毎回不要だったか」をログから学び、確認 4 点の初期推定を強める。

ただし固定（進化対象外）にすべき芯:
- 「ユーザーが意図的に起動する」起動条件、
- 「対話で参加者・前提を確定してから生成する」4 段フロー、
