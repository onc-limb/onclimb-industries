## 1. 壁打ちの記録

（対話の区切りごとに追記）

### 2026-07-05

- **ユーザー**: ソシム『エンジニアが一生困らない ドキュメント作成の基本』を読んだ。そこから 2 つの分類を得た。
  - 分類軸 1（読まれ方）: **辞書的に読んでもらうもの** / **読み物的に読んでもらうもの**
  - 分類軸 2（型）: **説明型** / **説得型** / **報告型**
- **ユーザー**: これらを元に、ドキュメントを作成する最初の工程として
  「作るドキュメントがどこに分類されるのか」を明確にしたい。

- **ユーザー**: ドキュメントには作成する目的がある。**型（説明/説得/報告）はその目的の種類**を
  表しているので、**型が決まったら「そのドキュメントの最も主張すべきこと」が決まる**。
- **ユーザー**: 辞書的なものは、読み手が「こういう情報があるだろう」と推測して読みに来る。
  読み物的なものは読み手にその前提がない → **ストーリーで流れを作って読ませる**必要がある。
- **ユーザー**: 説得型 × 辞書的が存在しないのは構造的に正しい（辞書的は読み手が必要な情報を
  取得しに来るものだから）。→ **辞書的 / 読み物的の違いは「情報取得の主体がどちらにあるか」**
  （読み手主導か、書き手主導か）ではないか。
- **ユーザー（ルール確定）**: 複数マスにまたがる場合の扱い:
  - **読まれ方（辞書的/読み物的）は、章を明確に分ければ 1 ドキュメント内で混在してよい**。
  - **型（説明/説得/報告）は目的そのものなので、1 ドキュメント内で混ぜない**。

- **ユーザー（型ごとの主張の形）**:
  - 説得型 = 判断・行動を促すこと
  - 説明型 = 手順等の理解と、**それに従った行動をしてもらう**こと
  - 報告型 = 現状を理解して、**読み手の判断の助けになる**こと
  - これは「大きな軸」であって、具体的な主張は作成するドキュメントごとに異なる。
- **ユーザー**: 読まれ方が違うと事前準備は異なる（pull = 疑問の列挙 → 見出し /
  push = 出発点とゴール → ストーリー、の理解で合っている）。
- **ユーザー（混在の配置ルール）**: 辞書的・読み物的を同一ドキュメントに入れる場合は、
  章を分けるだけでなく**サンドイッチにしない**（交互に挟まない）。
  最初の章だけ読み物的にする、最後に読み物部分をまとめる、などの配置にする。
  厳密に分割必須とはしないが、**分割した方が良いと判断したら別ドキュメントに分けることも考慮**する。
- **ユーザー（確定）**: **型が混ざった場合は分割するのが正解**。

- **ユーザー（報告型と説得型の境界）**: 大きな目的が報告で、その結果として次のアクション案を
  出しているなら**報告型のまま**（多くの場合、提案したアクションはそのまま実行される）。
  **説得（提案）型にするのは**、その案に反対された・他の案との比較を指示された等で、
  複数案から「この案がいい」と主張することになったとき。
  - **主目的の付随と判断できる場合は無理に分割しない**。
  - 分割判断の軸 = **「分割したときにストーリーがおかしくなるか、ならないか」**。
- **ユーザー（書籍のドキュメント作成の流れ）**: 「目的を明確にする」→「読み手を正確に定める」→
  「テーマを分解する」→「骨組みを作る」→「文章を書く」。
  この中で、**目的を明確にして読み手が定まったら、文書の型が定まる**のではないか。
- **ユーザー**: 事前準備（工程順序）の確定はもう少し深掘りしてから考えたい。

- **ユーザー（スキルの流れの骨格）**:
  1. どのような目的で、対象読者がどんな人のドキュメントを作成するかを定める。
     **目的は「具体的な読者の行動レベル」まで落とし込まれるまで深掘りする**。
  2. ここが定まったら、**読まれ方と型が決定される**。
  3. 次に、**目的を達成するためのテーマを定め、それを分割する**。
     - 例（日次報告）: 目的 = 作業進捗を伝えて、上層部への連絡やスケジュール判断の助けに
       なること → テーマ = 昨日の作業で**何を重点的に伝えるか**。
     - 例（手順書）: 目的 = 環境セットアップができること → テーマ = 順番にコマンドを
       打ったら（中身は理解せずとも）とりあえず環境ができること。
  4. テーマの中でも**「なぜ・何を・どうやって」の 3 観点**からこのドキュメントの理由を
     説明するのがよい。
- **ユーザー（テーマ分解の定義）**: テーマを分解するとは、**ドキュメントの目的を達成するために
  どのような主張を持ってくるか、その主張のためにはどのような情報・どのような流れが必要か**と
  深掘ること。

- **ユーザー（確定）**: 「なぜ・何を・どうやって」は**テーマ分解の検査枠**として使う
  （3 観点が揃って初めて読者が動ける。分解した主張がカバーしているかをチェックする）。
- **ユーザー（確定）**: **骨組み（見出し構成）の作成は次ステップの個別スキル（Stage 1）に任せる**。
  planner は**テーマ分解まで**を行う。
- **ユーザー（確定）**: 情報源は**テーマ分解の後**に、分解されたそれぞれのテーマに従って
  「どこから情報を持ってくるか」を**対話で埋める**。

## 2. 見えてきた整理（Claude の仮説含む・随時更新）

- **（確定に近い）2 軸の意味**:
  - 軸 1（読まれ方）= **情報取得の主体**。辞書的 = 読み手主導（pull: 疑問を持って引きに来る）/
    読み物的 = 書き手主導（push: 前提のない読み手をストーリーで運ぶ）。
    主に**構成・見出し設計**を決める。
  - 軸 2（型）= **目的の種類**。型が決まると「最も主張すべきこと」が決まる。
- **（帰結）マトリクスは 6 マスではなく実質 5 マス**。説得型 × 辞書的は構造的に存在しない
  （説得は書き手主導の行為なので、読み手主導の pull とは両立しない）。
- **（ルールの帰結・仮説）** 1 ドキュメント 1 型なので、依頼の中に複数の型が混ざっていたら
  **ドキュメントを分割する**のが正しい対応になる。事前準備の工程には
  「型が混ざっていないかの検査と分割の提案」が含まれるのではないか。
- **（仮説）分類は 6 択で直接聞くものではなく、導出するもの**。
  「目的は何か」→ 型が決まる、「読み手はどう読むか（引きに来るか、運ばれるか）」→ 読まれ方が
  決まる。事前準備のヒアリングは分類名を聞くのではなく、目的と読み手を聞いて分類を導出し、
  ユーザーに確認する形が自然ではないか。
- **（仮説）3 つの型は「読後に読者に起きてほしいこと」で統一的に定義できる**:
  - 説明型 = 読者が理解し、**それに従って正しく動ける**（行動の中身はドキュメントが与える）
  - 説得型 = 読者が**書き手の望む特定の判断**をする（結論を書き手が持っている）
  - 報告型 = 読者が**自分の判断を下せる材料**を得る（結論は読者に委ねる）
  - → 説得型と報告型の境界（当初仮説「結論を押すか材料を渡すか」はユーザーの整理で更新）:
    **機械的な判定ではなく主従で見る**。報告の付随としての次アクション案は報告型の範囲。
    反対・複数案比較の文脈で「この案がいい」と主張する段になったら説得型として別に書く。
    検査は「分割したらストーリーがおかしくなるか」で行う。
  - → **（気づき）説得型ドキュメントには生まれ方のパターンがある**: 報告に添えた案が
    反対・比較指示を受ける → 説得型ドキュメントを新たに書く、という**ライフサイクル**。
    型は最初から独立に選ぶだけでなく、前のドキュメントへの反応から次のドキュメントが
    生まれることがある。
- **（工程順序 v3 — ユーザー確認済みの確定版）**
  1. **目的 × 対象読者のヒアリング** — 目的は「読者の行動レベル」
     （読後に読者が具体的に何をする/できるようになるか）まで深掘りして確定する。
  2. **型と読まれ方の導出** — 目的と読者から導出し、ユーザーに確認する
     （分類名を直接 6 択で聞かない）。
  3. **テーマの設定** — 目的を達成するために「何を重点的に伝えるか」を決める。
     目的（読者に起きること）とテーマ（伝えることの焦点）は別物。
  4. **テーマの分解** — 主張の選定 → 主張を支えるのに必要な情報 → 必要な流れ。
     「なぜ・何を・どうやって」の 3 観点で分解の充足を検査する。
     型の混在検査（混ざっていたら分割提案。判定は「分割したらストーリーがおかしくなるか」）も
     ここで行う。
  5. **情報源の対話埋め** — 分解された各テーマについて「どこから情報を持ってくるか」
     （コード・worklog・議事録・既存ドキュメント・ユーザーの頭の中）を対話で埋める。
  6. **brief 化して Stage 1 へ引き渡し** — 骨組み（見出し構成）以降は Stage 1 の領分。
- **（準備と執筆の境界 — 確定）** 書籍の 5 工程のうち「目的」「読み手」「テーマ分解」までが
  planner、「骨組み」「文章」は Stage 1。
- **（仮説）既存 friday スキルのマトリクス上の位置**（ユーザー未確認）:
  - friday-design-doc-generator（設計書 / README）: 説明型 × 辞書的
  - friday-design-doc-generator（ADR）: 報告型寄り × 辞書的（決定の記録。説得の残響もある）
  - friday-proposal-generator（提案書・見積）: 説得型 × 読み物的
  - friday-tech-article-drafter（記事・登壇）: 説明型 × 読み物的
  - friday-daily-report（日次報告）: 報告型 × 読み物的
  - friday-giziroku（議事録）: 報告型 × 辞書的（決定事項・TODO を後から引く）

## 4. スキル設計への示唆（壁打ちの結論 → planner の設計）

- **ヒアリングの中心は「目的 × 読者」の 2 点だけ**。ただし目的は
  「読者の行動レベル」まで深掘りする（「共有したい」で止めない。読後に読者が何をするか）。
  前段の文脈（このドキュメントの前に何があったか — 例: 報告への反対から生まれた説得型）も聞く。
- **分類（型 × 読まれ方）はヒアリングから導出して確認する**。6 択で聞かない。
  - 型（説明/説得/報告）= 目的の種類。決まると「最も主張すべきこと」の形が決まる。
  - 読まれ方（辞書的/読み物的）= 情報取得の主体（読み手主導 pull / 書き手主導 push）。
    決まるとテーマ分解後の流れの作り方が切り替わる。
  - 説得型 × 辞書的は存在しない（実質 5 マス）。
- **planner の作業範囲はテーマ分解まで**（主張・必要情報・流れ + why/what/how 検査 + 混在検査）。
  骨組み（見出し構成）・文章は Stage 1。
- **情報源はテーマ分解の後に、テーマごとに対話で埋める**。
- **ルール**: 1 ドキュメント 1 型（混在は分割。判定は「分割したらストーリーがおかしくなるか」）。
  読まれ方の混在は章分け + サンドイッチ禁止（読み物章は先頭か末尾に寄せる）。
