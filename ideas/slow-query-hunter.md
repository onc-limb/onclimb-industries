# slow-query-hunter

> SES 参画中のエンジニアが、担当プロダクトの「どのクエリが遅いか」を実測 / EXPLAIN の根拠付きで特定し、
> そのまま提案に使える改善案（DDL・コード修正）まで一気に出すためのスキル。
> バックログ #54 query-latency-profiler と #47 slow-query-optimizer を 1 スキルに統合したもの。

---

## 場面 (When)

- ユーザーがスロークエリログや APM のクエリ統計を指して「このログを解析して改善案を出して」と言ったとき
- ログが無い環境で「どのクエリが遅いか実測して」と言ったとき（コードからクエリを抽出して DB で計測）
- 一覧画面が重い等の症状から「N+1 になってる箇所を探して」と言ったとき
- 特定のクエリを貼って「この EXPLAIN を見て直し方を教えて」と言ったとき
- パフォーマンス改善の提案前に「クエリのレイテンシ一覧を作って」と言ったとき

### 想定トリガー文 (実際のユーザー発話例)

1. 「スロークエリを洗い出して」
2. 「このスロークエリログを解析して改善案を出して」
3. 「どのクエリが遅いか実測して」
4. 「N+1 になってる箇所を探して」
5. 「このクエリの EXPLAIN を見て直し方を教えて」
6. 「クエリのレイテンシ一覧を作って」

## 解決する課題 (Problem)

- 現状: スロークエリの解析と改善が後回しになる（#47）。さらにスロークエリログが無い環境では
  そもそもどのクエリが遅いかを把握する手段が無い（#54）。EXPLAIN を読める人が現場に居ないと
  「なんとなくインデックスを足す」対応になり、効かないインデックスが積み上がる。
- 痛みの種類: 時間（手動での EXPLAIN 往復）/ 見落とし（回数の多い軽量クエリ = N+1）/
  手戻り（根拠の無いインデックス追加）
- 影響範囲: プロダクション（クエリ性能はユーザー体感に直結）+ チーム共有物（改善提案の根拠資料）

---

## 解決策 (Solution)

1. 対象リポジトリと DB / ORM を設定ファイルから特定し、ログの有無で入力モードを決める
2. 計測環境を確認する（接続先の表示と確認・データ規模の把握。本番は明示承認必須）
3. 遅いクエリ候補を特定する
   - ログモード: ログを正規化して集計・ランキング → 発行元コード (file:line) を特定
   - コードモード: クエリ発行箇所（ORM / 生 SQL / クエリビルダ）を抽出 → 発行 SQL を組み立て
4. SELECT / EXPLAIN のみで実測・実行計画を取得し、references の基準で解析する
5. 問題を分類（missing index / N+1 / full scan / 不要な列・行 / ロック待ち）し、
   改善案（DDL 案・コード修正案）と期待効果（実測できた場合のみ数値）を付ける
6. スロークエリレポートを生成し、確認のうえ `<repo>/slow-query-report/<YYYY-MM-DD>.md` に保存する

### 入力 (Inputs)

| 名前 | 型 | 必須 | 出どころ | 例 |
|---|---|---|---|---|
| 対象リポジトリ | path | yes | ユーザー発話 / cwd | `projects/foo-app/` |
| スロークエリログ / APM 統計 | file | no（あればログモード） | ユーザー指定 | `slow_query.log` |
| DB 接続先 | 接続情報 | no（無ければ静的解析に縮退） | 設定ファイル + ユーザー確認 | ローカル PostgreSQL |
| スコープ指定 | string | no | ユーザー発話 | 「N+1 だけ」「orders 周りだけ」 |

### 出力 (Outputs)

| 名前 | 形式 | どこに置く | 例 |
|---|---|---|---|
| スロークエリレポート | markdown | `<repo>/slow-query-report/<YYYY-MM-DD>.md`（保存前に確認） | ランキング + 改善案 + 未計測リスト |
| 要約 | チャット返信 | チャット | 要改善クエリ件数と上位の要点 |

## 想定ユーザー (Who)

- 主: SES 参画中のフリーランスエンジニア本人（性能改善の調査・提案を任される立場）
- 副: 改善提案の読み手となる案件先のエンジニア

本人は SQL とインデックスの基礎は分かるが、EXPLAIN の各ノードの読み方・DBMS 固有の癖
（filesort、行数見積もりの乖離、複合インデックスの列順）を毎回調べ直している。
references に判定基準を固定することでこの調べ直しを無くす。

## 既存ツール・スキルとの差別化 (Differentiation)

| 代替 | 限界 | このスキルが上回る点 |
|---|---|---|
| pt-query-digest / pgBadger | ログ集計のみ。発行元コードの特定・改善案までは出ない | file:line の特定から改善 DDL / コード修正案まで一気通貫 |
| APM (Datadog 等) | 導入済み環境でしか使えない。契約が無い案件が多い | ログ無し環境でもコードモードで計測できる |
| LLM に「このクエリ遅い？」と聞く | 見た目で速い遅いを断定しがちで根拠が無い | 実測 / EXPLAIN の証拠が無いものは「未計測」に分離する原則を型に組み込み |
| arc-reactor-tech-debt-auditor | 静的シグナル中心。クエリ性能は対象外 | DB での実測という動的証拠を扱う |
| db-migration-safety-checker（未実装） | マイグレーションの危険検出が領分 | 本スキルは遅いクエリの発見と改善提案。DDL 適用時の安全確認は向こうに接続 |

## 成功条件 (Success Criteria)

- [ ] レポートの計測済みクエリ全件に、実測値または EXPLAIN の抜粋が根拠として付いている
      （根拠なしで「遅い」と書かれた項目がゼロ）
- [ ] ログモードで、上位スロークエリの 8 割以上について発行元 file:line を特定できる
- [ ] 提案した改善（インデックス / eager load）を適用して再実測したとき、改善が確認できる
      （効かない提案が続くようなら references の判定基準を見直す）
- [ ] DB への書き込み操作が一度も発生しない（SELECT / EXPLAIN のみ）

## 失敗モード予測 (Failure Modes)

| 失敗 | 起きる条件 | 回避策 |
|---|---|---|
| 見た目だけで「遅い」と断定 | 接続できる DB が無い・実測を省略した | 大原則「実測なくして断定なし」。証拠が無いものは未計測リストへ分離 |
| 本番 DB に無断接続・負荷 | 設定ファイルの接続先が本番だった | 実行前に接続先を必ず表示して確認。本番は明示承認必須 |
| 書き込みクエリを EXPLAIN ANALYZE で実行してデータ変更 | UPDATE 等に ANALYZE を使った | 書き込み系は計画のみの EXPLAIN。ANALYZE が要る場合はトランザクション + ROLLBACK + 事前確認 |
| 開発 DB の少データで「問題なし」と誤判定 | 行数が本番と乖離 | データ規模を事前把握し「参考値」をレポートに明記 |
| 組み立てた SQL が実際の発行 SQL と違う | ORM の動的クエリ・スコープの見落とし | 組み立ての推測箇所に ASSUMPTION を付け、可能ならクエリログ出力で突き合わせ |
| 接続文字列・実データの機密がレポートに混入 | レポートへの転記 | 接続情報の値は転記しない。SQL パラメータは正規化 / ダミー値にする |

## 必要なリソース (Resources)

- 外部 API / CLI: `psql` / `mysql` クライアント（実測・EXPLAIN 用）、`rg`（クエリ発行箇所の抽出）
- 権限: 対象 DB（ローカル / 開発）への読み取り接続。本番はユーザーの明示承認
- 参照すべき公開知識: PostgreSQL / MySQL 公式ドキュメントの EXPLAIN 仕様
  （要点は references/explain-analysis.md に固定済み）
- ローカル前提: 対象リポジトリが手元にあること。コードモードでは接続可能な DB 環境
  （無ければ静的解析のみに縮退）

## 実装メモ (Implementation Notes)

- ディレクトリ構成: `SKILL.md` + `references/explain-analysis.md`（EXPLAIN の読み方と改善パターン）。
  scripts は初版では持たない（ログのパース・集計はログ形式が案件ごとに違うため、まず手組みで運用し、
  形式が固まったら正規化スクリプトの切り出しを検討する）
- #54（プロファイリング = コードモード）と #47（ログ解析 + 改善提案 = ログモード）を
  入力モードの分岐として 1 スキルに統合。バックログの評価表でも「#47 は単独では作らず #54 に統合」
  「n-plus-one-detector (#24) は #54 の派生として後日検討」とされており、N+1 静的検出も
  コードモードの一部として本スキルが吸収した
- 改善 DDL は提案のみ。マイグレーション化して適用する際の危険検出は
  db-migration-safety-checker（バックログ #50、未実装）に接続する
- レポートは対象プロジェクト配下に閉じる（CLAUDE.md のプロジェクト固有情報分離ルール）
