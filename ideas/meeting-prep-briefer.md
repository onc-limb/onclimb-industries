# meeting-prep-briefer

> フリーランス本人が、クライアント・チームとの打合せの**直前（前日〜当日）**に、
> 議題と手元資料から「論点・自分のスタンス案・想定質問」を整理した事前ブリーフを 10 分で手に入れるためのスキル。
> **friday-giziroku（会議の事後＝議事録）と対になる「会議の事前」スキル**。

---

## 場面 (When)

このスキルが呼ばれる具体的な状況。

- ユーザーが打合せの予定・議題を貼り付けて「明日の打合せの準備をしたい」と言ったとき
- クライアントから届いたアジェンダのメール/メモを指して「これの論点整理して」と言ったとき
- 定例会議の前に「前回の議事録踏まえて、今回の持ち越しと確認事項をまとめて」と言ったとき
- 商談・提案前に「先方に聞かれそうなことと回答案を用意したい」と言ったとき
- 議題が曖昧なまま「来週の MTG、何を決めるべきかから整理したい」と言ったとき
- （非トリガー）会議が終わった後の「議事録にして」→ friday-giziroku。「作業まとめて」→ jarvis-worklog

### 想定トリガー文 (実際のユーザー発話例)

1. 「明日の打合せのブリーフ作って」
2. 「この議題で事前準備したい。論点整理して」
3. 「次の定例の前に、前回議事録から持ち越しと確認事項を洗い出して」
4. 「〇〇さんとの MTG、想定質問と回答案を用意して」
5. 「来週の商談の下調べして。決めたいことも整理したい」
6. 「会議前の頭の整理を手伝って。アジェンダはこれ」

## 解決する課題 (Problem)

このスキルが無い場合に何が起きているか。

- 現状: 打合せ前の論点整理・下調べが不足したまま会議に臨み、その場で考えながら発言している。
  過去の議事録（`giziroku/minutes/`）や作業記録（jarvis 系）に材料はあるのに、会議前に読み返す時間を確保できず活用されない。
- 痛みの種類:
  - **見落とし** — 前回の持ち越し事項・宿題を確認し忘れ、同じ議論を繰り返す
  - **認知負荷** — 会議中に「現状整理」と「意思決定」を同時にやることになり、判断の質が落ちる
  - **手戻り** — スタンスを決めずに臨んだ結果「持ち帰って検討します」が増え、決定が次回送りになる
- 影響範囲: 個人作業（本人の準備）だが、会議の質を通じてクライアントとの合意形成スピードに波及する。

---

## 解決策 (Solution)

スキルが起動してから完了するまでのステップ。SKILL.md の workflow の下書き。

1. **ヒアリング**: 会議の目的・参加者・議題・決めたいことを対話で確定する（不明点は推定案を提示して補正してもらう）。
2. **関連資料の収集**: 過去議事録（`giziroku/minutes/`）・jarvis 系記録（`report-record/`, `knowledge-base/`）・ユーザー指定資料を議題キーワードで検索し、候補一覧をユーザーに提示して取捨選択してもらう。
3. **論点整理**: 議題を「決めたいこと」単位の論点に分解し、論点ごとに現状・選択肢・自分のスタンス案を組み立てる。資料由来の事実と推測を区別する。
4. **ブリーフ生成**: 固定フォーマット（会議の目的 / 決めたいこと / 論点ごとの整理 / 想定質問と回答案 / 前回からの持ち越し / 確認したいこと）でチャットに返す。求められればファイル保存する。
5. **完了条件の判定**: 「決めたいこと」が 1 つ以上言語化され、各論点にスタンス案（または「情報不足で決められない」の明示）が付いていれば完了。

### 入力 (Inputs)

| 名前 | 型 | 必須 | 出どころ | 例 |
|---|---|---|---|---|
| meeting_purpose | string | yes（対話で確定） | ユーザー発話 | 「6月末リリースの可否判断」 |
| agenda | string / markdown | yes（無ければ対話で起こす） | ユーザー発話・貼り付け・メール文面 | 「1. 進捗報告 2. スケジュール見直し」 |
| participants | list[string] | no | ユーザー発話・`giziroku/config/roster.yaml` | 「田中さん(PM)、自分」 |
| meeting_datetime | string (date) | no | ユーザー発話 | 「明日 14:00」 |
| past_minutes | file path(s) | no | `giziroku/minutes/` の検索結果から選択 | `giziroku/minutes/2026-06-25_定例_議事録.md` |
| jarvis_records | file path(s) | no | `report-record/`, `knowledge-base/` の検索結果から選択 | `report-record/<案件>/2026-06-30.md` |
| user_materials | file path(s) / 貼り付けテキスト | no | ユーザー指定 | 提案書ドラフト、先方からの要件メモ |

### 出力 (Outputs)

| 名前 | 形式 | どこに置く | 例 |
|---|---|---|---|
| 事前ブリーフ | markdown | チャット返信（既定） | 固定 6 セクションのブリーフ 50〜150 行 |
| ブリーフファイル | markdown ファイル | `giziroku/briefs/YYYY-MM-DD_<会議タイトル>_ブリーフ.md` **(ASSUMPTION: 保存先は giziroku と対の位置づけを踏まえ `giziroku/briefs/` とした。giziroku/ は git 管理外のため機密面も安全)** | ユーザーが保存を求めたときのみ |
| 参照資料リスト | markdown（ブリーフ末尾） | ブリーフ内セクション | 「参照: 2026-06-25 定例議事録, report-record/...」 |

## 想定ユーザー (Who)

- 主: フリーランスエンジニア本人。クライアント定例・商談・技術打合せに一人で臨む。
- 副: なし（本人専用。共有向け清書は想定しない — 共有資料が要る場面は friday 系の領分）。

ユーザーが既に持っている知識: 案件の技術的文脈・参加者との関係性。
ユーザーが持っていない（忘れている）知識: 前回議事録の決定事項・持ち越し・自分の宿題の詳細。→ ここをスキルが補う。

## 既存ツール・スキルとの差別化 (Differentiation)

| 代替 | 限界 | このスキルが上回る点 |
|---|---|---|
| friday-giziroku | 会議の**事後**に議事録を作るスキル。準備には使えない | 会議の**事前**に論点・スタンス案・想定質問を用意する。giziroku の蓄積（minutes/roster）を入力として活かす |
| karen-problem-essence-organizer | 課題の本質整理が目的で、会議という時限イベントへの準備物（想定質問・持ち越し確認）は出さない | 会議固有の固定フォーマットで「その場で使える」準備物を出す |
| jarvis-record / worklog | 作業の一次記録を作る。会議観点（決めたいこと・論点）への再構成はしない | 記録を「会議で使う形」に再構成する |
| 自力で議事録を読み返す | 時間がかかり、複数資料の横断・論点への再構成は頭の中で行うことになる | 検索・横断・再構成を肩代わりし、10 分でブリーフが出る |

## 成功条件 (Success Criteria)

観測可能な基準。

- [ ] 起動から 10 分以内（対話込み）にブリーフがチャットに出力される
- [ ] ブリーフに「決めたいこと」が 1 件以上、論点ごとの「自分のスタンス案」（または「情報不足」の明示）が全論点分含まれている
- [ ] 前回議事録が存在する会議では、「前回からの持ち越し」セクションに議事録の保留・TODO 由来の項目が漏れなく転記されている（元議事録との突合で欠落ゼロ）
- [ ] ブリーフ内の各記述に事実（資料由来・出典付き）と推測（`(推測)` 表記）の区別が付いている
- [ ] 会議後にユーザーが「準備していなかった論点で詰まった」と振り返るケースが 5 回中 1 回以下

## 失敗モード予測 (Failure Modes)

| 失敗 | 起きる条件 | 回避策 |
|---|---|---|
| **関連資料の見落とし** | 議事録・記録のファイル名/本文と議題の語彙が一致しない（表記ゆれ・別名） | 検索は複数キーワード（案件名・参加者名・技術語）で行い、ヒットした候補一覧を**ユーザーに提示して取捨選択してもらう**。ゼロ件でも「何も見つからなかった」と明示し、ユーザーにパス指定を促す |
| **論点の的外れ** | 議題が曖昧なまま推測で論点を立て、会議の実態とずれる | Phase 1 のヒアリングで「この会議で決めたいことは何か」を必ず確認してから論点分解する。推測で立てた論点には `(推測)` を付け、ブリーフ冒頭で補正を促す |
| **準備過剰** | 全議題を同じ深さで掘り、ブリーフが長大化して直前に読み切れない | 論点数の上限を目安 5 件とし、「決めたいこと」に直結する論点を優先。それ以外は「その他の議題」に 1 行圧縮。ブリーフ全体を 150 行以内に収める |
| **事実と創作の混同** | 資料に無い「先方の意向」等をもっともらしく書いてしまう | 入力資料に無い情報は書かない。推測は `(推測)` と根拠を併記。スタンス「案」であることを明記し断定しない |
| **古い情報の混入** | 数か月前の議事録の決定事項を「現状」として扱う | 参照資料には必ず日付を併記し、直近の記録と矛盾する場合は新しい方を採用して矛盾を注記する |

## 必要なリソース (Resources)

- 外部 API / CLI: なし（ローカルファイル検索のみ。`grep` / `ls`）
- 権限: リポジトリ内ファイルの読み取り。ファイル保存時は `giziroku/briefs/` への書き込み
- 参照すべき公開知識: 会議アジェンダ設計の一般論（決めたいこと駆動）。特別な外部標準は不要
- ローカル前提: onclimb-industries リポジトリ直下で実行。`giziroku/minutes/`・`report-record/`・`knowledge-base/` が存在すれば活用（無くても動く）

## 実装メモ (Implementation Notes)

- ディレクトリ構成: `.claude/skills/karen-meeting-prep-briefer/`（`SKILL.md` + `templates/brief.md`）。
  scripts は不要（検索は grep で足りる。機械処理するほどの定型入力が無い）。
- 既存スキルとの依存関係:
  - friday-giziroku の出力（`giziroku/minutes/`）と参加者名簿（`giziroku/config/roster.yaml`）を**読み取り専用**で参照。書き込みはしない。
  - jarvis 系の出力（`report-record/`, `knowledge-base/`）も読み取り専用で参照。
    **(ASSUMPTION: `worklog-data/` は生ログで量が多くノイズも多いため、既定の検索対象から外し、ユーザー明示指定時のみ読む)**
- templates として残す知識: ブリーフの固定フォーマット雛形（`templates/brief.md`）。
- 分類は karen（本人専用の一時利用・汎用系）。共有ドキュメントを作るわけではないので friday ではない。
  ただし giziroku と運用上ペアなので、将来 friday へ移す判断もあり得る（karen persona の「用途が定まったら移動を検討」に従う）。
