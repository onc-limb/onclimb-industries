# design-doc-generator (コード → 設計書 / README 草案)

> 開発の当事者（フリーランス本人・SES 参画中の技術者）が、実装は進んだのに
> 設計書・README・ADR が整備されないまま放置されているプロジェクトに対して、
> **コードから確認できた事実のみ**を材料に、共有可能なドキュメント草案を生成するためのスキル。
> コードに無い情報は創作せず「TBD」と明示するか、対話でユーザーに確認する。

---

## 場面 (When)

このスキルは **ユーザーが意図的に起動する**（自動起動しない）。対象コードベースが手元にあり、
それを他者に共有できるドキュメントに落としたい、という明確な意思があるときだけ走る。

- ユーザーが「このリポジトリの README 書いて」と言い、`projects/<project-name>/` を指したとき
- 実装がひと区切りつき、「今の構成を設計書に起こして」と依頼されたとき
- 参画先への引き継ぎ・納品を控え、「引き継ぎ用の設計ドキュメントを作って」と言われたとき
- 技術選定の経緯を残したくて「この選定を ADR にまとめて」と言われたとき
- 会話の中で設計方針を議論した直後に「今の話、設計書に反映して」と言われたとき

### 想定トリガー文 (実際のユーザー発話例)

1. 「このプロジェクトの README 作って」
2. 「今のコードベースから設計書の草案を書いて」
3. 「この構成をドキュメントに起こして」
4. 「引き継ぎ用に設計ドキュメントまとめて」
5. 「DB を PostgreSQL にした経緯、ADR にして」
6. 「README が古いから、コードに合わせて書き直して」
7. 「アーキテクチャ図込みで設計書のたたき台を作って」

> 起動の明示性: description には「ユーザーがコードベースやモジュールを指してドキュメント生成を
> 依頼したとき」と限定して書き、worklog / record 系（作業ログの整理）と誤起動しないようにする。

---

## 解決する課題 (Problem)

- 現状:
  - 実装優先で進めるため README・設計書が「後で書く」のまま放置され、
    引き継ぎ・納品・レビュー依頼の直前に慌てて書く。1 本あたり 1〜3 時間かかる。
  - コードと乖離した古いドキュメントが残り、読んだ人を誤誘導する。
  - 素の LLM に書かせると**存在しない機能・設定・エンドポイントを創作**し、
    それを検収・レビューで指摘されて信頼を損なう。
- 痛みの種類: 時間（清書コスト）/ 手戻り（創作・乖離の訂正）/ 見落とし（暗黙の設計判断が消える）/ 認知負荷（何をどこまで書くかの判断）
- 影響範囲: チーム共有物（参画先・依頼者への納品物になりうる）→ 事実性の要求が非常に高い。

---

## 解決策 (Solution)

「**前提の対話確定 → コード調査（事実収集）→ 種別選択 → 草案生成 → TBD 明示とヒアリング**」の 5 段。
中心原則は **創作禁止**: 出力に書いてよいのは (a) コードから確認できた事実、(b) ユーザーが対話で
明言した情報、(c) 「(推測)」タグ付きの明示的な推測、の 3 種のみ。それ以外は「TBD」とする。

1. **対象と読み手の確認**: 対象リポジトリ/モジュールのパスと、読み手
   （チーム内エンジニア / 引き継ぎ先 / 依頼者）を対話で確定する。読み手で用語の粒度を変える。
2. **コード調査（事実収集）**: ディレクトリ構成・エントリポイント・依存関係（manifest 類）・
   設定ファイル・既存ドキュメント・テストを読み、**出典（ファイルパス）付きの事実リスト**を作る。
3. **ドキュメント種別の選択**: README / 設計書 / ADR のどれか（複数可）を、目的から提案し確定する。
4. **草案生成**: 種別ごとの固定構成テンプレートに事実を流し込む。コードで確認できた記述には
   根拠パスを添え、推測には「(推測)」タグを付ける。
5. **不明点の TBD 明示とヒアリング**: コードから読み取れない項目（背景・非機能要件・
   運用手順・採用理由など）は「TBD」と明示し、まとめてユーザーに質問する。回答があれば反映、
   なければ TBD のまま納品する（**空欄を勝手に埋めない**）。

### 入力 (Inputs)

| 名前 | 型 | 必須 | 出どころ | 例 |
|---|---|---|---|---|
| target_path | string (絶対パス) | yes | ユーザー発話 | `projects/foo-api/` |
| doc_type | enum(readme / design-doc / adr) | yes (対話で確定) | 対話 (推奨案を提示) | `design-doc` |
| audience | enum(team-engineer / successor / client) | yes (対話で確定) | 対話 | `successor` |
| scope | string | no | ユーザー発話 | 「認証まわりだけ」 |
| conversation_context | string | no | 直前の会話 (設計議論など) | 「DB は PostgreSQL に決めた」 |
| decision | string | ADR のとき yes | 対話 | 「ORM に Prisma を採用」 |

### 出力 (Outputs)

| 名前 | 形式 | どこに置く | 例 |
|---|---|---|---|
| doc_draft | markdown | 対象プロジェクト配下 (`projects/<project-name>/docs/` または対象リポジトリの慣習に従う場所) + チャットで要点提示 | `projects/foo-api/docs/design-doc.md` |
| fact_sources | markdown 節 | 草案末尾「参照したソース」 | `src/main.py`, `package.json` |
| tbd_list | markdown 節 + チャットの質問 | 草案内の各 TBD + 末尾一覧 | 「デプロイ手順: TBD」 |

> 置き場所は CLAUDE.md の「プロジェクト固有情報の分離ルール」に従い、
> **必ず当該プロジェクトのディレクトリ配下**（`projects/<project-name>/` または対象リポジトリ内）とする。
> 当リポジトリ側にプロジェクト固有の設計書を置かない。

## 想定ユーザー (Who)

- 主: フリーランス本人。自分が実装した（または調査した）コードベースのドキュメントを、
  参画先・依頼者・引き継ぎ先に共有する場面。
- 副: 生成されたドキュメントを読むチームメンバー・引き継ぎ先エンジニア・依頼者。

ユーザーが持っている知識: 設計の背景・採用理由・運用の実態（コードに書かれていない部分）。
ユーザーが持っていない/面倒な作業: コード全体の棚卸し・構成の言語化・体裁の整った清書。
→ だから「コードから読める事実は機械が収集し、コードに無い背景はユーザーに聞く」分担にする。

## 既存ツール・スキルとの差別化 (Differentiation)

| 代替 | 限界 | このスキルが上回る点 |
|---|---|---|
| `/init` (CLAUDE.md 生成) | AI 向けの作業指示書であり、人間の読み手向け設計書・README ではない | 読み手（他者）を確定して共有用ドキュメントに整える |
| `codebase-reader` スキル | 探索結果の構造化レポート止まりで、共有可能なドキュメントの体裁にならない | 調査フェーズの下請けに使いつつ、固定構成の完成形草案まで出す |
| 素の LLM に「README 書いて」 | 存在しない機能・設定を創作する。不明点を勝手に埋める | 事実/推測/TBD の 3 区分を強制し、出典パスを残す |
| `jarvis-record` / `friday-daily-report` | 「作業の記録・報告」が対象で、成果物としての設計書は範囲外 | 時系列でなく、コードの現在形を対象にした構造ドキュメントを作る |
| 手書き | 1 本 1〜3 時間。棚卸しの抜け漏れが出る | 事実収集を自動化し、人は TBD の回答に集中できる |

## 成功条件 (Success Criteria)

- [ ] 草案中の技術的記述（機能・構成・依存・エンドポイント等）がすべて、コード上の根拠
      （ファイルパス）またはユーザーの明言に紐づいており、レビューで「存在しない記述」の指摘が 0 件。
- [ ] コードから確認できなかった項目が、空欄や創作ではなく「TBD」または「(推測)」タグ付きで
      出力に残っている（grep で `TBD` / `(推測)` が検出できる）。
- [ ] 出力ファイルが対象プロジェクト配下（`projects/<project-name>/` 内または対象リポジトリ内）に
      保存され、onclimb-industries 側にプロジェクト固有の設計書が生成されていない。
- [ ] 種別ごとの固定構成（テンプレートの見出し）が毎回同一で、見出しの欠落・順序入れ替えがない。
- [ ] TBD 一覧がチャットで質問として提示され、ユーザーが回答すれば草案に反映される。

## 失敗モード予測 (Failure Modes)

| 失敗 | 起きる条件 | 回避策 |
|---|---|---|
| **存在しない仕様の創作**（最重要） | LLM が「READMEらしさ」を優先して、未実装の機能・設定・手順をそれらしく書く | 事実/推測/TBD の 3 区分を強制。技術的記述には出典パス必須。生成後に自己チェックで各記述の根拠を照合し、根拠なしは TBD 化 |
| 古い・生成物のコードを事実と誤認 | ビルド生成物・vendored code・デッドコードを読む | 調査対象から `dist/` `node_modules/` `vendor/` 等を除外。エントリポイントから辿れる範囲を優先し、疑わしい場合は「(推測)」タグ+確認 |
| TBD だらけで草案の価値がない | コードが小さい/読み手情報がない段階で起動 | 調査後に「事実で埋まる割合」が低ければ、生成前に「先にヒアリングで埋めるか」を提案する |
| コードとの将来の乖離 | 生成後にコードが変わる | 草案冒頭に生成日と対象 commit (取得できる場合) を記録し、スナップショットであることを明示 |
| 巨大リポジトリで調査が破綻 | 数千ファイル規模 | scope を対話で絞る。モジュール単位の分割生成を提案する |
| 機密の混入 | `.env` や設定ファイルに実キーがある | 設定は「キー名と用途」のみ記載し、値は絶対に転記しない（persona の安全性規約） |
| 出力先の誤り | onclimb-industries 直下に書いてしまう | 出力前に対象プロジェクト配下のパスであることを検証。プロジェクト外なら保存先を確認 |

## 必要なリソース (Resources)

- 外部 API / CLI: 不要。ローカルのファイル読み取りと `git log` / `git rev-parse`（対象が git 管理下の場合、生成時点の commit 記録用）のみ。
- 権限: 対象プロジェクト（`projects/<project-name>/` 等）の読み取り、同ディレクトリ配下への書き込み。
- 参照すべき公開知識: README の一般的な構成慣習、ADR (Architecture Decision Record) の
  Michael Nygard 形式（Status / Context / Decision / Consequences）。
- ローカル前提: 対象コードベースが手元（`projects/` 配下または任意のローカルパス）にあること。

## 実装メモ (Implementation Notes)

- ディレクトリ構成:
  ```
  .claude/skills/friday-design-doc-generator/
    SKILL.md
    templates/
      readme.md        # README 雛形
      design-doc.md    # 設計書雛形
      adr.md           # ADR 雛形 (Nygard 形式ベース)
  ```
- scripts は置かない。調査は Read / Glob / Grep / `codebase-reader` サブエージェントで足り、
  決定論的に切り出せる前処理がない **(ASSUMPTION: 運用して定型前処理が見つかれば後から scripts 化する)**。
- 調査フェーズでは既存の `codebase-reader` エージェント/スキルを下請けとして活用してよい
  **(ASSUMPTION: codebase-reader の出力形式はこのスキルの事実リストにそのまま流用できる)**。
- テンプレートの見出しは固定とし、埋められないセクションは削除せず `TBD` を入れる
  （見出しの欠落＝調査漏れか情報不足かが読み手に伝わらなくなるため）。
- persona は `personas/friday.md`（共有ドキュメント系）。読み手が他者である点、
  創作禁止・推測明示・機密除外の規約をそのまま反映する。

## 自己進化を入れるか (Self-Evolution Scope)

- [ ] パイプラインあり
- [x] パイプラインなし — 理由: このスキルの品質は「対象コードの読解精度」と「創作禁止の遵守」で
  決まり、どちらも実行ごとに蓄積して効く共有資産（giziroku の glossary/roster のようなもの）を
  持たない。対象プロジェクトごとに文脈が独立しており、プロジェクト間でログを持ち回ると
  CLAUDE.md の「プロジェクト固有情報の分離ルール」に抵触するリスクもある。
  テンプレート構成の改善は、運用で気づいた時点で templates/ を手で直せば足りる。
