# tech-article-drafter

> フリーランスエンジニアが、日々の作業から得た技術知見を「発信できる実績」に変えるために、
> worklog の tech digest / knowledge-base の vault をネタ元に、機密を匿名化した技術記事ドラフトを生成するスキル。

---

## 場面 (When)

- ユーザーが「ブログ記事のドラフト書いて」と、ネタ元（digest / vault ノート / 技術トピック）を指して言ったとき
- 月末・案件の区切りなどに「今月の作業から記事にできるネタ探して」と発信ネタの棚卸しを頼まれたとき
- vault の特定ノート（例: `knowledge-base/tech/terraform-iac.md`）を開いた流れで「これ Zenn 記事にして」と言われたとき
- 勉強会・LT が近づき「登壇ネタになりそうな話をまとめて」と言われたとき
- 既に書いた記事ドラフトについて「公開前チェックして」「機密残ってないか見て」と確認を頼まれたとき
- **(ASSUMPTION: 自動起動はしない。**発信は公開行為であり機密リスクを伴うため、
  ユーザーが明示的に依頼したときだけ動く設計とする**)**

### 想定トリガー文 (実際のユーザー発話例)

1. 「ブログ記事のドラフト書いて」
2. 「今月の作業から技術記事のネタ探して」
3. 「Terraform のノートを Zenn 記事にして」
4. 「登壇ネタになりそうな話まとめて」
5. 「この記事ドラフト、公開前チェックして」
6. 「先週ハマった件、記事にできる?」
7. 「テック記事のドラフト作って。機密は伏せて」

---

## 解決する課題 (Problem)

- 現状: 案件作業に追われ、ブログ・登壇などの発信が常に後回しになる。worklog / knowledge-base に知見は
  溜まっているのに「公開できる形」への変換コスト（構成を考える・機密を消す・文章化する）が高く、
  実績として外に残らない。特に SES / 受託の知見は顧客名・案件名・社内 URL が混ざるため、
  「うっかり公開して事故る」不安が発信の心理的ブレーキになっている。
- 痛みの種類: 時間（記事1本の起筆に数時間）/ 認知負荷（機密チェックを人力で全文走査）/
  機会損失（発信ゼロ → 営業材料・単価交渉材料が増えない）
- 影響範囲: 個人のキャリア資産（ポートフォリオ・営業材料）。ただし機密漏れが起きた場合の影響は
  **顧客との契約・信頼に及ぶ**ため、安全側の設計が最優先。

---

## 解決策 (Solution)

1. **ネタ選定**: `worklog-data/digests/tech/` の tech digest と `knowledge-base/tech/` の vault ノートを走査し、
   記事にできそうなネタ候補（トラブルシュート・技術選定・設計判断）を出典付きで提示 → ユーザーが選ぶ。
2. **構成の提案と合意**: 選んだネタについて「課題 → 試したこと → 結果 → 学び」の固定骨子で
   見出しレベルの構成案を提示し、ユーザーと合意してから書き始める（勝手に書き上げない）。
3. **ドラフト生成**: 合意した構成に沿って、出典（digest / ノート）の事実だけを使い Markdown ドラフトを生成。
   記録にない箇所は書かない（創作しない）。
4. **機密マスキング**: 顧客名・案件名・社内 URL・API キー・個人名などを公開用に匿名化する
   （置換ルールは references/masking-checklist.md に定義）。何をどう置換したかをマスキングログに残す。
5. **公開前チェックリスト**: 機械チェック（秘密情報パターンの grep）+ チェックリストの全項目確認を行い、
   全て通ったものだけ `status: ready-to-publish` にする。1 つでも未確認なら draft のまま。

### 入力 (Inputs)

| 名前 | 型 | 必須 | 出どころ | 例 |
|---|---|---|---|---|
| ネタ元の指定 | string (トピック名 / ファイルパス / 期間) | no（無ければネタ選定から始める） | ユーザー発話 | 「Terraform の話」「6月の作業から」 |
| tech digest | markdown ファイル群 | yes（vault と少なくとも一方） | `worklog-data/digests/tech/<project>_<date>.md` | `personal-dashboard_2026-06-27.md` |
| vault ノート | markdown ファイル群（Obsidian 形式） | yes（digest と少なくとも一方） | `knowledge-base/tech/<slug>.md` | `terraform-iac.md` |
| 想定媒体 | enum (blog / zenn / qiita / 登壇メモ) | no（既定 blog） | ユーザー発話・対話確認 | 「Zenn に出したい」 |
| 追加 NG ワード | text（1 行 1 語） | no | `articles/_ng-words.txt` **(ASSUMPTION: ユーザーが育てるローカル辞書)** | 顧客名・案件コード |

### 出力 (Outputs)

| 名前 | 形式 | どこに置く | 例 |
|---|---|---|---|
| 記事ドラフト | markdown（frontmatter 付き: title / status / sources / masked） | `articles/<YYYY-MM-DD>-<slug>/draft.md` **(ASSUMPTION)** | `articles/2026-07-02-terraform-portability/draft.md` |
| マスキングログ | markdown（置換前カテゴリ → 置換後の対応表。**置換前の実値は書かない**） | 同ディレクトリ `masking-log.md` | 「顧客名 1 件 → 『ある受託案件』」 |
| チェック結果 | チャット返信 + draft.md frontmatter の `status` 更新 | チャット / draft.md | `status: ready-to-publish` |

## 想定ユーザー (Who)

- 主: フリーランスエンジニア本人（発信で実績・営業材料を作りたいが時間がない）
- 副: なし（生成物の読者はブログ読者・勉強会聴衆だが、スキルの利用者は本人のみ）

持っている知識: 記事の題材となった技術・作業の詳細（本人の実体験）。
持っていない知識: どの知見が「記事として面白いか」の俯瞰、機密チェックの網羅的な観点。

---

## 既存ツール・スキルとの差別化 (Differentiation)

| 代替 | 限界 | このスキルが上回る点 |
|---|---|---|
| `jarvis-knowledge-base` | 蓄積（自分用 vault）が目的で、公開向けの文章・構成にはならない | vault を出典に「公開に耐える記事」へ再構成し、公開専用のマスキング層を追加する |
| `friday-daily-report` | 読み手が依頼者で、日次報告フォーマット固定。不特定多数向けの技術記事は書けない | 読み手が「不特定多数の技術者」で、媒体別の記事構成と公開前チェックを持つ |
| 汎用 LLM に「記事書いて」と頼む | ネタ元との事実対応・機密チェックが担保されず、創作や漏えいのリスク | 出典を digest / vault に限定し、マスキング + チェックリストを必須工程として組み込む |
| 手書き | 1 本数時間かかり後回しになる（現状の課題そのもの） | ネタ選定〜ドラフトまでを短縮し、人間は構成合意と最終チェックに専念できる |

## 成功条件 (Success Criteria)

- [ ] `articles/<YYYY-MM-DD>-<slug>/` に draft.md と masking-log.md の 2 ファイルが生成される
- [ ] 公開前チェックの機械検査（秘密情報パターン + NG ワードの grep）でヒット 0 件になるまで
      `status: ready-to-publish` に遷移しない（frontmatter で観測可能）
- [ ] ドラフトの各主張・数値に出典（digest / ノートのファイル名）が対応づき、出典にない記述が混入しない
      （frontmatter の `sources:` と本文を突き合わせて検証可能）
- [ ] 構成合意後の生成で、ユーザーが節の追加・削除といった構成レベルの手直しをせずに済む
      （語句の微修正のみで公開できる状態）

## 失敗モード予測 (Failure Modes)

| 失敗 | 起きる条件 | 回避策 |
|---|---|---|
| **機密漏れ**（顧客名・案件名・社内 URL・API キー・個人名が記事に残ったまま公開される） | マスキング工程の見落とし / digest 側のマスキング漏れ / 固有名詞の変種（略称・ローマ字表記） | 多層防御: (1) 入力をマスキング済みの digest / vault に限定 (2) 専用マスキングパスで全文置換 (3) 秘密情報パターン + `articles/_ng-words.txt` の機械 grep (4) 公開前チェックリスト全項目の人間確認。**1 つでも未完了なら ready-to-publish にしない** |
| 匿名化しても文脈の組合せで案件・顧客が特定できる（時期 × 業種 × 技術構成のモザイク） | 特徴的な技術構成やニッチな業務ドメインをそのまま書く | チェックリストに「組合せ特定」の観点を独立項目として置き、時期・業種・規模をぼかす／複数案件の知見を一般化して混ぜる |
| 出典にない内容の創作（結果を盛る・検証していない手順を書く） | digest に「記録なし」が多いネタを無理に膨らませる | ネタ選定段階で「記録なし」の多い digest は候補から外す。記録にない箇所は書かず、書く場合は一般論であることを本文中に明示する |
| ネタが薄く記事が浅くなる（手順の羅列だけで学びがない） | 単一 digest だけを出典にする | 同一技術領域の vault ノート（複数 digest の横断集約）を優先的に出典にし、digest 単体の場合は関連ノートを合わせて読む |
| マスキングログ自体が機密の対応表になってしまう | 置換前の実値をログに残す | masking-log.md には「カテゴリ + 件数 + 置換後表現」だけを書き、置換前の実値は一切書かない |

## 必要なリソース (Resources)

- 外部 API / CLI: なし（Read / Write / Grep のみで完結。**(ASSUMPTION: `claude -p` のヘッドレス実行は使わず、
  対話セッション内で生成する。構成合意という対話工程が中心のため)**）
- 権限: リポジトリ内のファイル読み書きのみ
- 参照すべき公開知識: 技術記事の一般的な構成（課題 → 試行 → 結果 → 学び）、各媒体（Zenn / Qiita）の Markdown 方言
- ローカル前提: `worklog-data/digests/tech/`（jarvis-worklog の出力）または `knowledge-base/`
  （jarvis-knowledge-base の出力）が存在すること

## 実装メモ (Implementation Notes)

- ディレクトリ構成:
  - `.claude/skills/friday-tech-article-drafter/SKILL.md` … 本体（フロー・マスキング方針・出力規約）
  - `templates/article-template.md` … 記事雛形（frontmatter + 課題 → 試したこと → 結果 → 学び）
  - `references/masking-checklist.md` … マスキングルール + 公開前チェックリスト（SKILL.md 本文から分離）
  - scripts は置かない（生成は LLM、機械チェックは grep で足りるため）
- 既存スキルとの依存: 入力は `jarvis-worklog`（tech digest）と `jarvis-knowledge-base`（vault）の出力。
  どちらも collect / 生成段階でマスキング済みだが、**それを信頼せず公開用マスキングを独立に実施する**（多層防御）。
- 出力先 `articles/` はリポジトリ直下のデータ置き場（他スキルの `report-deck/` 等と同じ流儀）。
  ドラフト段階では機密が残っている可能性があるため gitignore 対象とする **(ASSUMPTION)**。

## 自己進化を入れるか (Self-Evolution Scope)

- [ ] パイプラインあり
- [x] パイプラインなし — 理由: (1) 記事生成は「構成合意 → 生成 → チェック」の 1 回完結タスクで、
  改善はユーザーの手直し差分から都度反映すれば足り、ログ蓄積による自動書き換えの学習余地が薄い。
  (2) 機密マスキングという安全要件が中心のスキルであり、自己書き換えで SKILL.md やチェックリストが
  「緩む」方向に進化する事故を許容できない。マスキングルールの追加はユーザーが明示的に
  `references/masking-checklist.md` / `articles/_ng-words.txt` を更新する運用とする。
