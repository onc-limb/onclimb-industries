# pr-splitter

> 大きくなりすぎた working diff / ブランチを抱えた開発者が、レビュー依頼の前に
> 変更をレビュー可能な意味単位の PR 列へ安全に分割するためのスキル。

---

## 場面 (When)

- 実装に没頭して working diff が数十ファイル・数千行に膨らんだ後、「この diff、大きすぎるから PR 分割して」と言ったとき
- 1 ブランチにリファクタと機能追加を積んでしまい、「このブランチをレビューしやすい単位に分けて」と言ったとき
- レビュアーから「PR が大きすぎて見られない」と言われ、「PR #123 を分割して」と既存 PR を指したとき
- レビュー依頼の前に「この変更を複数の PR に割って」と依頼したとき
- まだ PR にする前段階で「working diff を意味単位のブランチに分けて」と言ったとき

### 想定トリガー文 (実際のユーザー発話例)

1. 「この diff、大きすぎるから PR 分割して」
2. 「このブランチをレビューしやすい単位に分けて」
3. 「PR #123 が大きすぎるので分割計画を立てて」
4. 「この変更を複数の PR に割って」
5. 「working diff を意味単位のブランチに分けて」
6. 「pr split して」

---

## 解決する課題 (Problem)

- 現状: 大きな PR はレビューが後回しにされるか、流し読みで LGTM される。手作業で分割するには
  `git add -p` や cherry-pick を差分全体と突き合わせながら回す必要があり、1 時間以上かかるうえに
  ハンクの取りこぼしで「分割後の合計が元と一致しない」事故が起きる。怖いので結局分割せずに出す。
- 痛みの種類: 手戻り（レビュー放置・雑レビューによる不具合流出）/ 時間（手作業分割）/
  見落とし（分割時のハンク取りこぼし）/ 認知負荷（どの単位で割るかの判断）
- 影響範囲: チーム共有物（PR・レビュープロセス）。分割ミスはプロダクションの欠落変更に直結する

---

## 解決策 (Solution)

1. 分割対象（working diff / ブランチのコミット列 / 既存 PR）とベースブランチを確定し、
   `git diff` / `git log` で変更全体を読んで、変更の意図をユーザーに確認する
2. 意味単位（リファクタ準備 → 基盤変更 → 機能 → テスト/ドキュメント等）に分け、
   各 PR の目的・含むファイル/ハンク・依存順序・レビュー観点を付けた計画表を提示する。
   絡み合ったハンク（1 ファイルに複数単位が混在）は明示し、割るか妥協するかを確認する
3. ユーザーが計画を承認する（承認まで実施しない。バックアップブランチ名も事前提示）
4. 元の状態をバックアップで無傷に残したまま、新ブランチ列に `git checkout <base> -- <file>` /
   `git apply` で変更を機械的に積み上げ、各ブランチでビルド / テストを回す
5. 全ブランチを重ねた最終状態が元の diff と一致することを `git diff` の差分ゼロで検証する
6. 希望があれば `gh` で依存順に PR を作成する（本文に分割元・依存 PR・レビュー観点を記載）
7. 完了条件: 差分ゼロ検証を通過し、各ブランチが単体で成立していること

### 入力 (Inputs)

| 名前 | 型 | 必須 | 出どころ | 例 |
|---|---|---|---|---|
| target | working diff / branch / PR 番号 | yes | ユーザー発話 / cwd の git 状態 | 「PR #123」「このブランチ」 |
| base_branch | string (ブランチ名) | yes (既定: PR のベース or リポジトリの既定ブランチ) | ユーザー発話 / `gh pr view` | `main` |
| intent_notes | string (変更の意図) | no (不明点はヒアリングで補う) | ユーザーとの対話 | 「リファクタは 2FA 追加の準備」 |
| create_prs | boolean | no (既定: 作成しない) | ユーザー発話 | 「PR 作成までやって」 |

### 出力 (Outputs)

| 名前 | 形式 | どこに置く | 例 |
|---|---|---|---|
| split_plan | markdown の計画表（目的 / ファイル・ハンク / 依存 / レビュー観点） | チャット返信 | 4 PR 構成の計画表 |
| branches | git ブランチ列 + バックアップブランチ | 対象リポジトリのローカル git | `refactor/extract-auth-helper` 等 |
| verification | 差分ゼロ検証の結果（コマンドと出力） | チャット返信 | `git diff --stat` が空であること |
| prs (希望時) | GitHub PR（依存順、分割元・依存・レビュー観点入り） | 対象リポジトリの GitHub | スタック PR 4 本 |

## 想定ユーザー (Who)

- 主: SES / フリーランスで参画中のエンジニア本人。実装を進めた後にレビュー可能な形へ整えたい
- 副: 分割後の PR を受け取るレビュアー（レビュー観点付きの小さい PR を受け取れる）

ユーザーが持っている知識: 変更の意図と対象リポジトリの構成。
持っていない知識: 安全な分割手順（ハンク操作・検証方法）と、どの単位で割るのが妥当かの判断基準。

## 既存ツール・スキルとの差別化 (Differentiation)

| 代替 | 限界 | このスキルが上回る点 |
|---|---|---|
| `git add -p` + 手動ブランチ作成 | ハンク選択が対話式で長丁場。取りこぼしても気づけない | 意味単位の計画を先に合意し、最後に差分ゼロで機械検証する |
| `git rebase -i` でのコミット整理 | コミット単位でしか割れない。1 コミットに混在した変更は分けられない。この環境では対話式 rebase 自体が使えない | ファイル / ハンク単位で再配置でき、非対話コマンドだけで完結する |
| GitHub のスタック PR ツール (graphite 等) | 分割済みであることが前提。何をどう割るかは決めてくれない | 「どう割るか」の計画立案と絡み合いハンクの検出から支援する |
| `arc-reactor-code-review` | レビューをするスキルで、レビュー可能な形に直すことはしない | レビューの前工程（分割）を担い、分割後のレビューは code-review に渡す |

## 成功条件 (Success Criteria)

- [ ] 分割後の全ブランチを重ねた最終状態と元の diff の差分がゼロ（検証コマンドの出力付き）
- [ ] 各 PR / ブランチが単体でビルド・テストを通過する（実行不能な場合はその旨が報告に残る）
- [ ] 元ブランチ / working tree がバックアップとして無傷で残り、復元手順が提示されている
- [ ] 分割計画の PR 単位について、ユーザーが「この割り方はおかしい」と差し戻す回数が 1 回以内
- [ ] 分割後の各 PR がレビュー観点付きで、レビュアーが元 PR より短時間で見られる粒度になっている

## 失敗モード予測 (Failure Modes)

| 失敗 | 起きる条件 | 回避策 |
|---|---|---|
| 分割の過程で変更が欠落・変質し、合計が元と一致しない | ハンクの取りこぼし、パッチ適用時のずれ | 最終状態と元 diff の**差分ゼロ検証を必須工程**にする。不一致なら原因特定まで完了と言わない |
| 元ブランチ / working diff を壊して復元不能になる | バックアップなしで reset / checkout した | 着手前にバックアップブランチ作成（working diff は stash 等で退避）。復元手順を作業前に提示 |
| 中間ブランチが単体で壊れている（ビルド不能・テスト失敗） | 依存順の誤り、生成物と生成元の分離 | 各ブランチでビルド / テストを実行して単体成立を確認。依存順は計画段階で明示して合意する |
| 分割ついでにコードを「改善」して diff が変質する | 気づいたタイポ・バグを直したくなる | 「再配置のみ・1 文字も改善しない」を大原則に固定。気づいた問題は指摘として報告だけする |
| push 済みブランチへの force push で他者の作業を壊す | 既存 PR の元ブランチを書き換えようとした | force push はユーザーの明示承認なしに絶対にしない。新ブランチ列の作成までに留める |
| 絡み合ったハンクの手編集パッチが適用に失敗し、中途半端な状態が残る | 同一ハンク内に複数単位が混在 | 絡み合いは計画段階で明示し、妥協同居を第一候補に提示。行き詰まったら状態を報告して指示を仰ぐ |

## 必要なリソース (Resources)

- 外部 API / CLI: `git`（diff / log / branch / checkout / apply / stash）、
  PR 作成・既存 PR 取得時のみ `gh`
- 権限: 対象リポジトリのローカル git 操作。PR 作成時は GitHub への push / PR 作成権限
- 参照すべき公開知識: スタック PR の運用プラクティス、small CL の考え方（Google eng-practices 等）
- ローカル前提: 対象が git リポジトリで、分割対象の diff / ブランチ / PR がローカルから参照できること

## 実装メモ (Implementation Notes)

- ディレクトリ構成: `.claude/skills/arc-reactor-pr-splitter/`
  - `SKILL.md` … 目的・大原則・トリガー・標準フロー（把握 → 計画 → 承認 → 実施 → 検証 → PR 作成）・品質規約
  - `references/split-criteria.md` … 意味単位の型・依存順序・サイズ目安・絡み合いハンクの扱い・
    分割しないほうがよいケース（SKILL.md の肥大化を避けて分離。失敗パターンはここに追記して育てる）
- scripts は作らない: 分割操作は `git` の非対話コマンドの組み合わせで足り、
  単位の判断は Claude の読解が主。固定スクリプト化するとリポジトリごとの差異に追従できない
- 既存スキルとの依存関係: 分割後の各 PR のレビューは `arc-reactor-code-review` に引き渡す（本スキルはレビューしない）
- persona: `personas/arc-reactor.md` に従う（応答は日本語、コード・ブランチ名は英語、推測は ASSUMPTION 明記）
