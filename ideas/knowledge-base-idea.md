# アイデアドキュメント: ナレッジベース構想

> 作業記録システム（collect → classify → summarize）が生成する
> `project.md` / `tech.md` を活用し、ナレッジベースとして蓄積・派生させる構想のまとめ。
> 確定仕様ではなく、方針と段階計画を記録するためのドキュメント。

---

## 1. 基本方針

Obsidian のタグ・リンクを含む Markdown ファイル群を、GitHub の **private リポジトリ**で管理する
ナレッジベースを中核として作成する。将来的にそこから2つの機能へ派生させる。

1. **公開ポートフォリオのスキルページ**（対外向け・公開情報のみ）
2. **自身のナレッジ検索機能**（自分向け・全件）

---

## 2. 機密情報の切り分け（設計の前提）

### 2-1. ナレッジは本来「転用できる情報」
- `tech.md`（技術整理情報）は、他案件にも再利用できる一般化された知見が主体。
  固有名詞・接続情報・顧客特定情報を除けば、機密はほとんど残らない性質を持つ。
- 一方 `project.md`（プロジェクト整理情報）は、目的・関係者・数値など
  **案件固有の文脈を含むため、機密はこちらに寄る**。

### 2-2. 結論：機密は project 側に寄せ、公開派生は tech 側を一次ソースにする
- 公開・転用に使う情報は `tech.md` 系（ナレッジ）を中心に据える。
- `project.md` 系は機密を含みうる前提で、原則として非公開・自分向けに留める。

### 2-3. private リポジトリの位置づけ（重要な注意）
- private リポジトリは流出耐性を上げる二重の安全網であって、
  「private だから機密を入れてよい」という意味ではない。
  - GitHub に預ける以上リスクはゼロではない。
  - 誤って public 化する事故もありうる。
- したがって基本設計は **「ナレッジは元から機密を含まない（一般化済み）」** を主とし、
  private 管理はあくまで補助的な安全網とする。
- 蓄積段階でマスキング（`<REDACTED:種別>`）を済ませておくことで、公開派生の安全性が一段上がる。

---

## 3. 全体構成

```
[蓄積の中核]
  Obsidian vault（GitHub private リポジトリで管理）
    - project.md / tech.md をプロジェクト・技術領域でタグ付け・リンク
    - Markdown ＋ タグ ＋ リンクで横断検索・関連付け
    - 機密は含めない設計（一般化済みナレッジ＋マスキング適用済み）
        │
        ├─→ [派生1] 公開ポートフォリオ スキルページ
        │       - tech.md 系から公開可能な情報のみを抽出・再構成
        │       - 機密マスキング＋公開用整形を経た抜粋のみを掲載
        │       - スキルの証跡・解決した技術課題の事例として提示
        │
        └─→ [派生2] 自身のナレッジ検索機能
                - vault 全件を対象に高速検索
                - 当面は Obsidian の検索で完結
                - 量が増えたら RAG / 全文検索システムへ拡張
```

---

## 4. 段階計画

最初から全部を作ろうとすると頓挫しやすいため、段階を踏む。

### フェーズ1: ナレッジベースの中核を作る（最優先）
- Obsidian vault を作成し、GitHub private リポジトリで管理。
- `project.md` / `tech.md` を蓄積する運用を確立。
- タグ設計・フォルダ構成・リンク方針を決める
  （例: 技術領域タグ、プロジェクトタグ、課題種別タグ）。
- これだけで「自分の再検索」はすぐ回り始める（活用の主目的）。

### フェーズ2: 可搬性・閲覧性を足す
- vault を Git 管理しているので、他端末からも clone / pull で参照可能。
- 必要ならスマホ閲覧（Obsidian の同期 or Git プラグイン）を追加。

### フェーズ3: 派生機能を個別に生やす
- 中身が十分に貯まってから着手する（薄いうちに作っても効果が出ない）。
- **派生1（公開スキルページ）**:
  - 公開用マスキング＋再構成のステップを挟み、その出力だけをサイトへ。
  - 蓄積場所と公開場所は必ず分ける。
- **派生2（ナレッジ検索）**:
  - Obsidian 検索で不足が出てきたら RAG / 全文検索を検討。
  - Markdown で貯めてあるので、そのまま RAG のインデックス元に転用できる。

---

## 5. 設計上の原則（メモ）

- **蓄積場所と公開場所を分ける。** 公開は必ずマスキング＋再構成を経た抜粋のみ。
- **一次ソースの使い分け。** 公開・転用は tech.md 系、自分向け詳細は project.md 系。
- **Markdown を中核に保つ。** 特定ツールに縛られず、後から RAG・公開・共有のどれにも転用可能。
- **段階的に作る。** まず貯める仕組み → 可搬性 → 派生機能、の順。
- 作業記録システムのマスキング処理（指示書 §2-5）と地続きで設計する。
