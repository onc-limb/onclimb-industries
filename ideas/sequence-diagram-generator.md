# sequence-diagram-generator

> SES 参画中のエンジニアが、「この処理どう流れてる？」と聞かれた・調べたいときに、
> 起点からコードを実追跡して口頭説明・引き継ぎ・調査メモにそのまま貼れる
> Mermaid シーケンス図 1 枚を素早く作るためのスキル。

---

## 場面 (When)

- 同僚や依頼者に「この API、中で何やってるの？」と聞かれ、口頭説明用の図が欲しいとき
- 障害調査で「このエンドポイントがどの外部サービスを叩くか」を経路ごと把握したいとき
- 引き継ぎ資料に処理フローの図を 1 枚入れたいとき
- 初見の処理を読む前に、まず全体の流れを図で掴みたいとき
- レビューや設計議論で「現状の流れ」を共通認識にする図が要るとき
- バッチ・イベントハンドラなど、HTTP 以外の入口から始まる処理の流れを整理したいとき

### 想定トリガー文 (実際のユーザー発話例)

1. 「この処理の流れをシーケンス図にして」
2. 「POST /orders がどう処理されるか図にして」
3. 「このバッチの処理フローを Mermaid で描いて」
4. 「ログインから DB 更新までの流れを追って図にして」
5. 「この関数から先の呼び出しをシーケンス図で見せて」
6. 「処理の流れを引き継ぎ用に図解して」

---

## 解決する課題 (Problem)

- 現状: 処理の流れを説明するたびに、コードを読み直しながら口頭かテキストで説明している。
  図を作るなら手描き（draw.io 等）で、コードとの対応付けが残らず、作った端から陳腐化する。
  LLM に「一般的なフロー」を描かせると、実コードに無い層や呼び出しが混入して信用できない。
- 痛みの種類: 時間（毎回コードを読み直す）/ 認知負荷（口頭説明では境界が伝わらない）/
  見落とし（外部 I/O の見逃し）/ 信頼（創作フローによる誤情報）
- 影響範囲: 個人作業（調査速度）＋チーム共有物（説明・引き継ぎ資料）

---

## 解決策 (Solution)

1. 起点の解決 — 発話からエントリポイントのコード位置を特定（ルーティング定義・
   ハンドラ登録を検索）。曖昧なら候補を提示して確認
2. 追跡 — 呼び出しを実コードで追う（grep と読解）。外部 I/O（DB クエリ・外部 API・
   キュー・キャッシュ）は必ず図中で強調する
3. 粒度の調整 — 既定は層・コンポーネント単位（1 関数 1 矢印で図を潰さない）。
   ユーザー指定で詳細化
4. 図の生成と検証 — Mermaid 構文の検証（レンダリング不能な図を渡さない）＋
   各ステップの根拠 `file:line` 対応表を添付
5. 限界の明示 — 動的ディスパッチ・DI・リフレクション・イベント購読で静的に追えなかった
   箇所は `(要確認)` 注釈で省略・推測と区別
6. 提示と保存 — チャット提示で完結可。保存する場合は対象プロジェクト配下
   `<repo>/docs/flows/` 等（保存前に確認）

### 入力 (Inputs)

| 名前 | 型 | 必須 | 出どころ | 例 |
|---|---|---|---|---|
| entry_point | string | yes | ユーザー発話 | `POST /orders`, `OrderService.create`, `bin/sync-users` |
| repo_path | string (絶対パス) | yes | ユーザー発話 or cwd（`projects/<name>/` が典型） | `projects/foo-api` |
| depth | number/string | no | ユーザー発話 | 「2 ホップ先まで」 |
| granularity | enum: 層単位(既定) \| 関数単位 | no | ユーザー発話 | 「この部分を関数単位で」 |

### 出力 (Outputs)

| 名前 | 形式 | どこに置く | 例 |
|---|---|---|---|
| シーケンス図 | Mermaid `sequenceDiagram`（分岐主役なら `flowchart`） | チャット返信（希望時 `<repo>/docs/flows/` 等） | 参加者 = 層・コンポーネント・外部システム |
| 根拠対応表 | markdown 表 | 図に添付 | ステップ番号 → `file:line` |
| 補足 | 箇条書き | 図に添付 | 分岐条件・エラーパス・トランザクション境界 |

---

## 想定ユーザー (Who)

- 主: SES 参画中のフリーランスエンジニア本人（= このリポジトリのユーザー）
- 副: 図を受け取る同僚・依頼者（図だけ見る読み手）

持っている知識: 一般的な言語・フレームワークの読み方と Mermaid の見方。
持っていない知識: 当該処理が実際にどのコードを通るか・どの外部サービスに触るか。
→ 図は「このリポジトリの実コードの経路」に絞り、一般論の層を描き足さない。

## 既存ツール・スキルとの差別化 (Differentiation)

| 代替 | 限界 | このスキルが上回る点 |
|---|---|---|
| `arc-reactor-codebase-onboarding` | ユースケースのフロー追跡を含むが、資料一式（地図・読む順番）の永続化が目的で重い | 単一フローの図 1 枚を素早く出すことに特化。依頼から図まで 1 往復 |
| `arc-reactor-external-access-mapper` | 外部アクセスの台帳（横断リスト）で、1 本の処理の時系列は分からない | 1 フローの時系列を図にする。台帳が要るなら併用 |
| 手描き（draw.io / PlantUML 手書き） | コードとの対応が残らず陳腐化。作図に時間がかかる | 各矢印に `file:symbol` の根拠付き。再生成が容易 |
| LLM に雑に「フロー図描いて」 | 実コードに無い「一般的なフロー」を創作しがち | 実際に読んだコードだけを図にし、追えない箇所は `(要確認)` で明示 |

## 成功条件 (Success Criteria)

- [ ] 生成された Mermaid が構文エラーなくレンダリングできる（レンダリング不能な図 0 件）
- [ ] 図中の全矢印（`(要確認)` 注釈付きを除く）に根拠 `file:line` の対応があり、
      参照先がすべて実在する
- [ ] フロー中に登場する外部 I/O（DB・外部 API・キュー・キャッシュ）が図中で
      強調表記され、見落としが 0 件
- [ ] 静的に追えなかった箇所が `(要確認)` として確認方法付きで明示され、
      推測が確定経路として描かれていない
- [ ] 対象リポジトリの `git status` が実行前後で不変（保存を依頼された場合を除く）

## 失敗モード予測 (Failure Modes)

| 失敗 | 起きる条件 | 回避策 |
|---|---|---|
| 実コードに無い「一般的なフロー」を描いてしまう | 追跡が途切れた箇所を知識で補完 | 大原則として創作禁止を明文化。追えない箇所は `(要確認)` 注釈＋確認方法を添付 |
| 分岐爆発で図が読めなくなる | 追跡が 3 ホップ以上で分岐が多い処理 | 1 枚に詰めず、フェーズ単位・分岐単位の分割を提案。既定粒度を層単位に固定 |
| Mermaid 構文エラーでレンダリングできない図を渡す | メッセージ中の `:` `{}` 等の特殊文字 | references の構文チェックリストで検証。`mmdc` が使えればパース実行 |
| 起点の取り違え（同名ルート・オーバーロード） | 曖昧な起点指定 | 候補を `file:symbol` 付きで提示してユーザーに確認してから追跡 |
| 外部 I/O の見落とし | ORM・SDK の薄いラッパー越しのアクセス | 追跡中に DB/HTTP/キュー/キャッシュのクライアント呼び出しを意識的に検索。強調記法を references で固定 |

## 必要なリソース (Resources)

- 外部 API / CLI: 不要（`rg`, Read のみ。`mmdc`（mermaid-cli）があれば構文検証に利用、無くても目視チェックリストで代替）
- 権限: 対象リポジトリの読み取り、保存時のみ `<repo>/docs/flows/` 等への書き込み（保存前に確認）
- 参照すべき公開知識: Mermaid sequenceDiagram / flowchart の構文
- ローカル前提: 対象プロジェクトがローカルに clone 済み（`projects/<name>/` 配下が典型）
