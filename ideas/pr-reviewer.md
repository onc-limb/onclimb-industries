# pr-reviewer

> レビュー担当者が、GitHub PR を割り当てられた直後に「観点別・投稿可能なレビューコメントの下書き」を数分で手に入れ、レビューの初動を速く・観点を均一にするためのスキル。

---

## 場面 (When)

- ユーザーが「PR #123 をレビューして」と PR 番号や URL を指してレビューを依頼したとき
- `gh pr list` で自分にレビューが割り当てられた PR を確認した直後の会話で
- ユーザーが「この PR にバグがないか観点別に見て」と観点を指定して依頼したとき
- レビューコメントの下書きを確認したユーザーが「これを GitHub に投稿して」と言ったとき
- 参画先プロジェクトの PR について「うちの規約に照らしてレビューして」と依頼されたとき

### 想定トリガー文 (実際のユーザー発話例)

1. 「PR #123 をレビューして」
2. 「この PR のレビューコメント作って」
3. 「PR を観点別にレビューして」
4. 「この PR、バグと規約違反がないか見て」
5. 「レビューコメントを GitHub に投稿して」
6. 「https://github.com/owner/repo/pull/45 のレビュー下書きちょうだい」

---

## 解決する課題 (Problem)

- 現状: PR レビューの初動が遅い。diff を開いてから「何をどの順で見るか」を毎回組み立て直しており、
  観点（バグ・可読性・規約）がレビュアーの気分と経験に依存して属人的。指摘の粒度・書式もばらつく。
- 痛みの種類: 時間（初動の立ち上がり）/ 認知負荷（観点の組み立て直し）/ 見落とし（観点の抜け）
- 影響範囲: チーム共有物（PR 上のレビューコメントはチーム・依頼者に見える成果物）

---

## 解決策 (Solution)

1. PR 参照（番号 / URL / ブランチ名）を受け取り、`gh` CLI で PR のメタ情報・diff・既存コメントを取得する
2. 対象リポジトリのプロジェクト規約（CLAUDE.md、linter 設定、CONTRIBUTING.md 等）を読み取り、規約観点に反映する
3. `references/review-checklist.md` の固定チェックリスト（バグ / 可読性 / 規約 + 補助観点）に沿って diff をレビューする
4. 観点ラベル・重要度ラベル・`file:line` 付きのレビューコメント下書きをチャットに提示する
5. ユーザーが明示的に承認した場合のみ、`gh` で PR にレビューコメントとして投稿する
6. 完了条件: 全指摘にラベルと位置情報が付き、投稿モードでは PR 上でコメントが確認できること

### 入力 (Inputs)

| 名前 | 型 | 必須 | 出どころ | 例 |
|---|---|---|---|---|
| pr_ref | string (番号 / URL / ブランチ名) | yes | ユーザー発話 | "#123", "https://github.com/o/r/pull/45" |
| repo | string ("owner/repo") | no | cwd の git remote から自動解決 | "onc-limb/onclimb-industries" |
| perspectives | string[] (観点の部分集合) | no | ユーザー発話（省略時は全観点） | ["bug", "convention"] |
| post_mode | enum ("draft" \| "post") | no | ユーザー発話（デフォルト "draft"） | "draft" |

### 出力 (Outputs)

| 名前 | 形式 | どこに置く | 例 |
|---|---|---|---|
| review_report | markdown（サマリ + 観点別指摘一覧） | チャット返信 | `[バグ][must] src/app.py:42 — ...` の一覧 |
| pr_comments | GitHub レビューコメント | `gh` 経由で対象 PR（承認後のみ） | インラインコメント + 総評 |

---

## 想定ユーザー (Who)

- 主: SES / フリーランスとして参画先プロジェクトの PR レビューを担うエンジニア本人
- 副: レビュー観点をチームで揃えたい参画先チーム（投稿されたコメントの読み手）

ユーザーが持っている知識: git / GitHub / `gh` CLI の基本操作、対象プロジェクトのドメイン知識。
ユーザーが持っていない知識: その PR の diff 全体の詳細、観点の網羅的なチェックリスト。

---

## 既存ツール・スキルとの差別化 (Differentiation)

| 代替 | 限界 | このスキルが上回る点 |
|---|---|---|
| `/code-review` | ローカルの作業中 diff（自分の変更）が対象。観点はバグ+品質改善で、観点別の分類・PR への体系的な投稿フローは持たない | GitHub 上の**他人の PR** を対象に、固定チェックリストで観点別（バグ・可読性・規約）に分類し、承認後に PR へ投稿できる |
| `/review` (組み込み) | 汎用の PR レビューで、観点チェックリストの固定化・プロジェクト規約の参照・重要度ラベル体系を持たない | 観点と出力書式を references で固定し、対象リポジトリの規約ファイルを読み込んで指摘に反映する |
| `gh pr diff` を目視 | 観点の組み立てが毎回手動で属人的 | 初動を自動化し、観点の抜けを構造的に防ぐ |
| 参画先の CI / linter | 機械検出可能な規約違反のみ。バグの兆候・可読性は拾えない | 意味レベルの指摘（ロジック誤り、命名、責務分割）を補完する |

棲み分けの要点: **`/code-review` = コミット前の自分のローカル diff の検品、本スキル = GitHub PR（主に他人の変更）への観点別・投稿可能なレビュー**。レビュー対象と成果物の置き場（チャット内 vs PR 上）が異なる。

---

## 成功条件 (Success Criteria)

- [ ] PR 番号の指定から、`gh pr checkout` なしでレビュー報告（下書き）がチャットに返る（目安 5 分以内）
- [ ] 全ての指摘に「観点ラベル + 重要度ラベル（[must]/[imo]/[nit]/[ask]）+ `file:line`」が付いている
- [ ] 投稿モードで承認した場合、生成コメントがそのまま GitHub の PR 上でレビューコメントとして確認できる
- [ ] 承認前に PR への書き込み（コメント投稿・レビュー送信）が一切発生しない（誤投稿ゼロ）

---

## 失敗モード予測 (Failure Modes)

| 失敗 | 起きる条件 | 回避策 |
|---|---|---|
| 巨大 diff でレビューが浅くなる・破綻する | 変更行数が数千行超 / 生成ファイル混在 | ファイル単位に分割してレビューし、lock ファイル等の自動生成物は除外。全量を見きれない場合は重点ファイルを選定し「未レビュー範囲」を明示する |
| 誤指摘（false positive）でレビューの信頼を毀損 | diff 外のコンテキスト不足、規約の思い込み | 確信が持てない指摘は [ask]（質問形式）に落とす。規約は対象リポジトリの CLAUDE.md / linter 設定を根拠として引用し、根拠が無い場合は指摘に ASSUMPTION を明記する |
| 承認前に PR へ投稿してしまう | 投稿系コマンドの先走り実行 | デフォルトは draft。`gh pr review` / `gh api` の投稿系コマンドは、下書き提示 → ユーザーの明示承認の後にのみ実行する |
| 既存レビューコメントと重複した指摘を投稿 | 他レビュアーが先にコメント済み | レビュー前に `gh pr view --comments` で既存コメントを取得し、重複する指摘は下書きから除外 or「既出」と注記する |
| diff 内の機密情報をコメントに転記してしまう | PR にシークレットが誤コミットされている | シークレット様の文字列（トークン・鍵）を検出したら値を伏せて「機密の混入疑い」として [must] 指摘に切り替える |

---

## 必要なリソース (Resources)

- 外部 API / CLI: `gh` CLI（認証済み）、GitHub API（`gh api` 経由）
- 権限: 対象リポジトリの PR 読み取り権限。投稿モードでは PR へのコメント書き込み権限
- 参照すべき公開知識: 一般的なコードレビュー慣行（重要度ラベル [must]/[imo]/[nit]/[ask] 等）
- ローカル前提: git リポジトリ内（remote から repo を解決する場合）。リポジトリ外でも `owner/repo` 明示指定で動作

## 実装メモ (Implementation Notes)

- ディレクトリ構成: `SKILL.md` + `references/review-checklist.md`（観点チェックリストの固定化）
- 既存スキルとの依存関係: なし（単独で完結）。`/code-review` `/review` とは対象・成果物で棲み分け（上記 Differentiation 参照）
- scripts: 作らない。PR 取得・投稿は `gh` CLI がそのまま担い、決定論的な前処理・集計が存在しないため
  **(ASSUMPTION: diff の分割・除外判定は都度の判断で足り、スクリプト固定化するほど定型化していないと推測した。定型化が見えたら scripts に切り出す)**
- references: レビュー観点チェックリスト（主観点: バグ / 可読性 / 規約、補助観点: テスト / セキュリティ）と重要度ラベルの定義
- **(ASSUMPTION: 重要度ラベルは [must]/[imo]/[nit]/[ask] の 4 段階を採用。参画先で別の流儀が指定された事例がまだ無いため、広く使われる慣行から推測した)**
- **(ASSUMPTION: 対象は GitHub のみとし、GitLab / Bitbucket は対象外。現在の作業環境が `gh` CLI 前提であることからの推測)**
- **(ASSUMPTION: デフォルトを「投稿せず下書き提示」とした。レビューコメントはチームに見える成果物であり、無承認投稿の事故コストが下書き確認の手間を上回ると判断した)**
- **(ASSUMPTION: 規約の参照元は対象リポジトリの CLAUDE.md / linter・formatter 設定 / CONTRIBUTING.md の優先順とした。参画先ごとの規約文書の置き場が不定なため、一般的な配置からの推測)**

## 自己進化を入れるか (Self-Evolution Scope)

- [ ] パイプラインあり
- [x] パイプラインなし — 理由: レビュー品質のフィードバック（指摘の的中 / 外れ）は PR 上のスレッドに散在し、
  スキル側で構造化ログとして自動収集する確実な手段がない。誤指摘を「学習」してチェックリストが誤進化する
  リスクの方が大きいため、観点の改善は `references/review-checklist.md` を人が意図的に更新する運用で育てる。
