# tech-debt-auditor

> 参画中（または自分の）プロジェクトの開発者が、リファクタリング計画や依頼者への改善提案の前に、
> リポジトリに溜まった技術負債を「根拠付き・優先度付きのリスト」として可視化するためのスキル。

---

## 場面 (When)

- ユーザーが「このリポジトリの技術負債を洗い出して」と言ったとき
- 参画直後にコードベースを把握した後、「直したほうがいい箇所をリストにして」と言ったとき
- リファクタリングの見積・提案の材料として「負債の一覧と優先度が欲しい」と言ったとき
- スプリント計画・改善デーの前に「負債の棚卸しをして」と言ったとき
- レビューや障害対応の中で「この辺、負債になってそうだから監査して」と特定ディレクトリを指したとき

### 想定トリガー文 (実際のユーザー発話例)

1. 「このリポジトリの技術負債を洗い出して」
2. 「技術負債を監査して優先度を付けて」
3. 「負債の棚卸しをして」
4. 「直すべき箇所を根拠付きでリストにして」
5. 「tech debt audit して」
6. 「`src/legacy` 配下の負債候補を調べて」
7. 「リファクタ提案の材料になる負債リストを作って」

---

## 解決する課題 (Problem)

- 現状: 技術負債は個々人の頭の中や TODO コメントに散在し、一覧化されないまま溜まる。
  棚卸しを手作業でやると grep・git log・依存確認を横断する必要があり、数時間かかる。
  さらに機械的な検出（lint 警告の羅列など）は誤検知が多く、リストが信用されず放置される。
- 痛みの種類: 見落とし（負債が可視化されない）/ 時間（手作業の棚卸し）/ 認知負荷（優先順位が付かない）/
  信頼低下（誤検知だらけのリストは読まれない）
- 影響範囲: 個人作業（リファクタ計画）〜チーム共有物（改善提案・見積の根拠資料）

---

## 解決策 (Solution)

1. 対象リポジトリ（パス）とスコープ（全体 / 特定ディレクトリ）、除外パスをユーザーと確定する
2. 負債カテゴリ別にスキャンする（負債マーカー、肥大ファイル/関数、重複、依存の老朽化、
   テスト欠落、非推奨 API、デッドコード疑い）。grep / git log / ロックファイル等のローカル情報のみ使用
3. 候補ごとに **証拠（ファイル:行、git 履歴、依存メタデータ）を収集**し、
   独立した証拠が 2 つ以上あるものだけ「確定」、1 つ以下は「要確認」に分類する
4. 確定項目を 影響 × 修正コスト のマトリクスで P1〜P4 に優先度付けする
5. Markdown レポート（サマリ → 確定リスト → 要確認リスト → 除外・免責）を生成して提示・保存する
6. 完了条件: 全項目に根拠 `ファイル:行` が付いており、確定/要確認が分離されていること

### 入力 (Inputs)

| 名前 | 型 | 必須 | 出どころ | 例 |
|---|---|---|---|---|
| repo_path | string (絶対パス) | yes | ユーザー発話 / cwd | `projects/acme-api/` |
| scope | string (glob / ディレクトリ) | no (既定: リポジトリ全体) | ユーザー発話 | `src/legacy/**` |
| exclude_paths | string[] | no (既定: vendored / generated / node_modules 等) | ユーザー発話 + 自動判定 | `["dist/", "gen/"]` |
| focus_categories | enum[] (負債カテゴリ名) | no (既定: 全カテゴリ) | ユーザー発話 | `["dependency", "test-gap"]` |

### 出力 (Outputs)

| 名前 | 形式 | どこに置く | 例 |
|---|---|---|---|
| audit_report | markdown | チャット返信 + `<repo_path>/tech-debt-audit/<YYYY-MM-DD>.md` | 確定 12 件・要確認 8 件のレポート |
| debt_items | レポート内の表 (カテゴリ / 根拠 file:line / 証拠数 / 影響 / コスト / 優先度) | audit_report 内 | `P1: src/auth.js:42 非推奨 API + 高churn` |

**(ASSUMPTION: レポートの保存先は対象プロジェクトディレクトリ配下 `tech-debt-audit/` とした。
CLAUDE.md の「プロジェクト固有情報はそのプロジェクト配下に保存する」ルールに従う。
対象リポジトリの git を汚す懸念がある場合は保存前にユーザーに確認する)**

## 想定ユーザー (Who)

- 主: SES / フリーランスで参画中のエンジニア本人。リファクタ計画・改善提案の起点が欲しい
- 副: 提案を受け取る依頼者側の技術リーダー（レポートを共有資料の下敷きにする）

ユーザーが持っている知識: 対象リポジトリの言語・大まかな構成。
持っていない知識: 負債がどこに・どれだけ・どの深刻度で溜まっているかの全体像。

---

## 既存ツール・スキルとの差別化 (Differentiation)

| 代替 | 限界 | このスキルが上回る点 |
|---|---|---|
| linter / 静的解析ツール (ESLint, SonarQube 等) | 警告が大量で優先度がなく、誤検知も混ざる。導入・設定コストもかかる | 複数証拠による確定/要確認の分離と、影響×コストの優先度付けまで行う |
| `grep TODO` の手作業 | マーカー系しか拾えず、履歴・依存・構造の負債を見逃す | 7 カテゴリを横断し、git 履歴・依存メタデータも証拠に使う |
| `codebase-reader` スキル | 構造の把握が目的で、負債の検出・優先度付けはしない | 負債に特化した判定基準と誤検知抑制の設計を持つ |
| `/code-review` | 対象が「現在の diff」であり、リポジトリ全体の蓄積負債は見ない | リポジトリ全体のストック（蓄積）を棚卸しする |

## 成功条件 (Success Criteria)

- [ ] レポートの全項目（確定・要確認とも）に根拠 `ファイル:行`（または依存名+ロックファイル行）が付いている
- [ ] 「確定」に分類された項目のうち、ユーザーが「これは負債ではない」と却下する割合が 2 割以下
- [ ] 確定項目の全件に P1〜P4 の優先度と、影響・修正コストの根拠 1 行が付いている
- [ ] 中規模リポジトリ（〜10 万行）で、スキャン開始からレポート提示まで 10 分以内 **(ASSUMPTION: 実測前の目標値)**
- [ ] レポートがそのままリファクタ提案・issue 起票の下敷きに使える（ユーザーが再整形せずに流用できる）

## 失敗モード予測 (Failure Modes)

| 失敗 | 起きる条件 | 回避策 |
|---|---|---|
| **誤検知**: 意図的な設計（正当な長い関数、意図して残した TODO 等）を負債と断定してリストの信頼を失う | 単一シグナル（マーカー 1 個、行数だけ等）で断定したとき | **独立した証拠 2 つ以上で初めて「確定」**。1 つ以下は必ず「要確認」に落とし、断定表現を使わない。全項目に `ファイル:行` を添付し、ユーザーが即検証できるようにする |
| vendored / 自動生成コードを負債として大量検出しノイズ化 | `node_modules/`, `dist/`, `*_pb.go` 等を除外し忘れたとき | 既知の生成物・ベンダーパスを既定で除外し、除外一覧をレポート末尾に明示。怪しいパスは開始前にユーザーへ確認 |
| 見逃しの誤解: レポートを「負債の全量」と受け取られる | 静的スキャンで検出できない負債（設計の歪み・暗黙知）があるのに免責がないとき | レポートに「本監査は静的シグナルベースであり網羅ではない」旨の免責と、カバーしたカテゴリ一覧を必ず記載 |
| 巨大リポジトリでスキャンが破綻・時間超過 | 数十万行超、モノレポ | 開始前にサイズ概算 (`git ls-files | wc -l` 等) を出し、超過見込みならディレクトリ単位への分割監査を提案 |
| デッドコード判定の外し（動的参照・リフレクションで実際は使用中） | 参照が文字列・DI 設定経由の言語/FW | デッドコード疑いは**単独証拠では確定にしない**カテゴリと定め、常に「要確認」始まりとする |

## 必要なリソース (Resources)

- 外部 API / CLI: `git`, `grep`/`rg`, `wc`, 各言語のロックファイル読解（`package-lock.json`, `go.sum` 等）。
  ネットワークアクセスは不要 **(ASSUMPTION: 依存の最新版照会はオフラインでは不可能なため、
  依存老朽化の判定はロックファイル内バージョンの古さ・EOL 既知情報など Claude の知識の範囲で行い、
  その旨をレポートに明記する)**
- 権限: 対象リポジトリの読み取り。レポート保存時のみ書き込み
- 参照すべき公開知識: 技術負債の分類（Martin Fowler の Technical Debt Quadrant 等）、各言語の非推奨 API 情報
- ローカル前提: 対象が git リポジトリであること（git 履歴を証拠に使うため。非 git でも縮退動作は可能）

## 実装メモ (Implementation Notes)

- ディレクトリ構成: `.claude/skills/arc-reactor-tech-debt-auditor/`
  - `SKILL.md` … 役割・フロー・優先度付け・出力フォーマット・品質規約
  - `references/debt-categories.md` … 負債カテゴリごとの判定基準・証拠の種類・確定条件（SKILL.md 本文の肥大化を避けて分離）
- scripts は作らない: 検出は `git` / `rg` 等の既存 CLI の組み合わせで足り、
  判定・分類は Claude の読解が主のため、固定スクリプト化すると柔軟性を失う
- 既存スキルとの依存関係: なし（`codebase-reader` を事前に使うと精度が上がるが必須ではない）
- persona: `personas/arc-reactor.md` に従う（応答は日本語、コード・識別子は英語、推測は ASSUMPTION 明記）

## 自己進化を入れるか (Self-Evolution Scope)

- [ ] パイプラインあり
- [x] パイプラインなし — 理由: 本スキルの中核は「証拠 2 つ以上で確定 / 1 つ以下は要確認」という
  誤検知抑制の規約であり、これは使用ログで自動書き換えされるべきでない固定ルール。
  また監査対象は毎回異なる他者のリポジトリで、実行ログにプロジェクト固有情報が含まれるため、
  ログを当スキル配下に蓄積すること自体が CLAUDE.md のプロジェクト情報分離ルールと相性が悪い。
  判定基準の改善は `references/debt-categories.md` への手動追記で行う。
