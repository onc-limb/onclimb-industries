# tech-selection-research

> フリーランスエンジニアが、案件・自プロダクトでライブラリ/フレームワーク/技術を選ぶ直前に、
> 「出典付きの比較表 + 推奨・次点・見送り理由」を固定フォーマットで得て、選定判断を記録に残すためのスキル。

---

## 場面 (When)

- 新規機能・新規案件の実装前に、ユーザーが「A と B どっちがいい？」と具体的な技術名を挙げて比較を求めたとき
- 「状態管理ライブラリを選びたい」のようにカテゴリ名で候補洗い出しから頼まれたとき
- 既存の採用技術が deprecated / メンテ停止になり、「代替を探して」と言われたとき
- クライアントへの技術提案の根拠資料として「出典付きで比較して」と言われたとき
- 過去に行った選定の記録を「前に選定した○○の記録を見せて」と参照したいとき

### 想定トリガー文 (実際のユーザー発話例)

1. 「Next.js と Remix を比較して技術選定して」
2. 「この要件に合う ORM を選定調査して」
3. 「状態管理ライブラリのおすすめを出典付きで比較して」
4. 「moment.js の代替を比較表にして推奨を出して」
5. 「技術選定の比較表を作って。推奨と見送り理由もつけて」
6. 「前に選定した DB マイグレーションツールの記録を見せて」

---

## 解決する課題 (Problem)

- 現状: 技術選定のたびに、比較観点の設計・候補の洗い出し・情報収集・表の整形をゼロからやり直している。
  過去の選定理由が会話ログに埋もれ、後日「なぜこれを選んだか」を説明できない。
- 痛みの種類: 時間（毎回 1〜2 時間の調査のやり直し **(ASSUMPTION: 実測ではなく体感値)**）、
  認知負荷（観点の抜け漏れ不安）、手戻り（ライセンスやメンテ状況の見落としによる採用後の差し戻し）。
- 影響範囲: 個人作業（選定調査そのもの）＋チーム共有物（クライアントへの選定根拠説明）。

---

## 解決策 (Solution)

1. テーマ（カテゴリ or 具体候補）を受け取り、要件を `AskUserQuestion` でヒアリングして確定する
   （用途 / 技術的制約 / ライセンス制約 / チームのスキル / must-have と nice-to-have の区別）。
2. WebSearch / WebFetch で候補を洗い出す（指名候補＋"alternatives" 検索で最大 5 件に絞る）。
3. 固定 7 軸（成熟度 / メンテ状況 / ライセンス / 学習コスト / エコシステム / 性能 / 要件適合）で
   各候補を評価する。各評価に出典 URL と取得日を必ず付ける。
4. 比較表 + 「推奨 1 件 / 次点 1 件 / 見送り（理由付き）」の固定フォーマットで出力する。
5. 選定記録を `tech-selection/<YYYY-MM-DD>-<slug>.md` に永続化して完了。

### 入力 (Inputs)

| 名前 | 型 | 必須 | 出どころ | 例 |
|---|---|---|---|---|
| theme | string | yes | ユーザー発話 | "React の状態管理ライブラリ" |
| named_candidates | string[] | no | ユーザー発話 | ["Zustand", "Jotai"] |
| requirements | { must: string[], nice: string[], constraints: string[] } | yes | ヒアリング (`AskUserQuestion`) | { must: ["TypeScript 対応"], constraints: ["MIT 系ライセンスのみ"] } |
| past_records | markdown files | no | `tech-selection/` 配下 | 過去の同カテゴリ選定記録 |

### 出力 (Outputs)

| 名前 | 形式 | どこに置く | 例 |
|---|---|---|---|
| comparison_table | markdown 表（候補 × 固定 7 軸、各セルに出典番号） | チャット返信 + 記録ファイル | 5 候補 × 7 軸の表 |
| recommendation | markdown（推奨 1 / 次点 1 / 見送り＋理由） | チャット返信 + 記録ファイル | "推奨: Zustand — 理由…" |
| sources | markdown リスト（URL + 取得日 + 一次/二次の別） | 記録ファイル末尾 | `[1] https://... (取得: 2026-07-02, 一次)` |
| decision_record | markdown ファイル | `<repo>/tech-selection/<YYYY-MM-DD>-<slug>.md` | `2026-07-02-react-state-management.md` |

---

## 想定ユーザー (Who)

- 主: フリーランスエンジニア本人（実装前の選定調査・クライアントへの根拠説明）
- 副: 選定根拠を受け取るクライアント・チームメンバー（記録を清書して共有する場合）

持っている知識: 技術用語・自分の要件はわかる。持っていない知識: 各候補の最新のメンテ状況・
ライセンス変更・コミュニティ動向（ここを調査で埋める）。

---

## 既存ツール・スキルとの差別化 (Differentiation)

| 代替 | 限界 | このスキルが上回る点 |
|---|---|---|
| 汎用 `deep-research` スキル | 毎回レポート構成・比較観点を設計し直す。要件ヒアリングや結論フォーマットが固定されず、選定記録も残らない | 比較軸を固定 7 軸にテンプレート化、要件ヒアリングを必須ステップ化、「推奨+次点+見送り理由」の固定出力、`tech-selection/` への選定記録の永続化 |
| 素の WebSearch での都度調査 | 観点の抜け漏れ、出典が残らない、毎回ゼロから | 固定軸で網羅、出典 URL+取得日必須、過去記録の再利用 |
| 比較ブログ記事を読む | 鮮度切れ・ポジショントークの混入 | 一次情報（公式 doc / リポジトリ）優先、取得日明示で鮮度を担保 |

deep-research との棲み分けを一言で: **deep-research は「任意テーマの自由形式レポート」、
本スキルは「技術選定に特化した固定軸・固定出力・記録永続化」**。技術選定以外の調査は deep-research に回す。

---

## 成功条件 (Success Criteria)

- [ ] 出力の比較表で「全候補 × 固定 7 軸」が埋まり、各評価セルに出典番号が付き、出典一覧に URL と取得日がある
- [ ] 「推奨 1 件 / 次点 1 件 / 見送り＋理由」が毎回出力され、ユーザーが続けて「で、結局どれ？」と聞き返さない
- [ ] 選定記録ファイルが `tech-selection/` に生成され、後日「前の選定記録を見せて」で参照・再利用できる
- [ ] 出典一覧のうち一次情報（公式ドキュメント / 公式リポジトリ / 公式リリースノート）が過半を占める

## 失敗モード予測 (Failure Modes)

| 失敗 | 起きる条件 | 回避策 |
|---|---|---|
| 出典の鮮度切れ（古い比較記事を根拠に、既に deprecated / メンテ停止のライブラリを推奨） | 二次情報のブログ・まとめ記事に依存したとき | 一次情報（公式 doc / リポジトリの最終リリース日・最終コミット）を必ず WebFetch で確認。全出典に取得日を付け、公開日が 2 年超の二次情報は根拠にしない |
| ハルシネーション（存在しないライブラリ・バージョン・スター数などの数値を捏造） | Web で確認せずモデルの記憶だけで表を埋めたとき | 数値・事実は WebFetch で取得したページの記載のみ採用し、出典 URL を紐付ける。確認できなかったセルは「未確認」と書き、空想で埋めない |
| 要件不在の一般論比較（ユーザーの制約を無視した「世間的な人気」推奨） | ヒアリングを飛ばして即調査に入ったとき | 要件ヒアリングをフローの必須ステップにし、must 要件を「要件適合」軸に直結させる。must 未達の候補は人気に関係なく見送りへ |
| 候補の見落とし（有名どころ 2〜3 件だけで比較して終わる） | 単一の検索クエリしか使わないとき | 「<テーマ> alternatives」「<候補名> vs」等の複数クエリで洗い出し、除外した候補も「見送り」に一行理由で残す。カバーできなかった範囲を明示する |
| 比較表の破綻（候補が多すぎて表が読めない） | カテゴリが広く候補が 8 件以上出たとき | 本表は最大 5 候補に絞り、残りは見送りリストへ理由付きで格下げする |

## 必要なリソース (Resources)

- 外部 API / CLI: WebSearch / WebFetch ツール（外部 API キー不要）
- 権限: Web アクセス、`tech-selection/` 配下への書き込み
- 参照すべき公開知識: 各候補の公式ドキュメント・リポジトリ（GitHub 等）・リリースノート・ライセンスファイル
- ローカル前提: 当リポジトリ内で実行（記録の保存先があるため）

## 実装メモ (Implementation Notes)

- ディレクトリ構成: `SKILL.md` + `references/comparison-axes.md`（固定 7 軸の定義・評価基準）
  + `templates/decision-record.md`（選定記録の雛形）。scripts は不要（Web 調査は組み込みツールで完結）。
- 記録の保存先はリポジトリ直下 `tech-selection/` **(ASSUMPTION: `report-record/` 等の既存データ
  ディレクトリと同じ「リポジトリ直下にスキルごとのデータ置き場」慣習に合わせた)**。
- 既存スキルとの依存関係: なし（単体で完結）。`deep-research` とはトリガーの棲み分けのみ。
- persona は `personas/edith.md`（出典明示・一次情報優先・不確実性明記・カバー範囲の明示）を反映する。

## 自己進化を入れるか (Self-Evolution Scope)

- [ ] パイプラインあり
- [x] パイプラインなし — 理由: 比較軸・出力フォーマット・ヒアリング項目は「毎回同じであること」自体が
  価値（選定記録同士を後日横並びで比較できる）であり、実行ログから自動書き換えするとフォーマットの
  一貫性が壊れる。改善が必要なら `references/comparison-axes.md` を人手で更新する運用で足りる。
  また 1 回の選定は独立したタスクで、実行間で学習させたい状態は「選定記録の蓄積」で既に賄えている。
