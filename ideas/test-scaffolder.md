# test-scaffolder

> 実装を終えたがテストを書く工数が惜しい開発者が、対象コードから「そのプロジェクトの流儀に合った」
> テスト雛形と主要ケース案を数分で得るためのスキル。

---

## 場面 (When)

- ユーザーが実装ファイルを指して「このファイルのテスト書いて」と言ったとき
- 関数・クラス単位で「この関数のテスト雛形作って」と依頼されたとき
- コーディング完了直後に「この変更にテスト足して」と言われたとき（直近 diff が対象）
- レビュー指摘「テストがない」への対応で「ここのユニットテストのたたき台作って」と言われたとき
- 実装前のケース設計として「テストケース案だけ出して」と言われたとき（雛形生成なしのケース列挙のみ）

### 想定トリガー文 (実際のユーザー発話例)

1. 「このファイルのテスト書いて」
2. 「この関数のテスト雛形作って」
3. 「テストケース案を出して」
4. 「この変更にテスト足して」
5. 「ユニットテストのたたき台作って」
6. 「`UserService` のテストが無いのでカバーして」

---

## 解決する課題 (Problem)

- 現状: テスト作成は「FW の流儀を思い出す → 配置・命名を既存に合わせる → ケースを設計する →
  ボイラープレートを書く」の 4 段階があり、実装より腰が重い。結果として後回しになり、
  テストなしのままコミットされる。
- 痛みの種類: 時間（雛形書きに 20〜30 分）+ 認知負荷（言語/FW ごとの流儀の切り替え）+
  見落とし（境界値・異常系のケース漏れ）
- 影響範囲: 個人作業に留まらず、SES 参画先のプロダクションコードの品質（チーム共有物）に波及する。

---

## 解決策 (Solution)

1. 対象コード（ファイル / 関数 / クラス / 直近 diff）を特定する
2. **流儀検出**: プロジェクト内の既存テストと設定ファイルからテスト FW・命名・配置・
   アサーション/モック手法を検出する。既存テストが無ければ `references/` の言語別デフォルト流儀を使う
3. 検出した流儀に沿ってテスト雛形（ファイル）を生成する
4. 主要ケース案（正常系 / 境界値 / 異常系）を表で列挙し、雛形にケースとして反映する
5. テストランナーで生成テストを実行して壊れていないことを確認し、結果を報告する

### 言語/FW ごとの流儀対応の方針（本スキルの中核設計）

候補表のメモ「言語/FW ごとの流儀対応が課題」への回答。**2 段フォールバック**で解決する。

1. **第 1 優先: プロジェクト内の既存テストから流儀を検出する**
   - 設定ファイル（`package.json` / `pyproject.toml` / `go.mod` / `pom.xml` 等）から FW を特定
   - テストファイルの命名パターン（`test_*.py`, `*.test.ts`, `*_test.go`, `*Test.java` 等）で
     既存テストを探索し、対象ファイルに近い順に最大 3 件読んで流儀
     （配置 / 命名 / アサーション / モック / fixture / パラメタライズ）を抽出する
   - この探索は `scripts/detect_test_style.py`（python3 標準ライブラリのみ）で決定論的に行う
2. **第 2 優先: 既存テストが無い場合は `references/` の言語別デフォルト流儀を使う**
   - `references/python.md`（pytest）/ `references/javascript-typescript.md`（vitest/jest）/
     `references/go.md`（標準 testing + table-driven）/ `references/java.md`（JUnit 5）
   - **(ASSUMPTION: デフォルト対応言語は Python / TypeScript・JavaScript / Go / Java の 4 つに絞った。
     直近の案件で遭遇する頻度が高いと推測したため。他言語は公式標準のテスト FW を都度推測し、
     雛形内に `// ASSUMPTION:` コメントで明記する)**
3. 検出結果（採用した FW・配置・参照した既存テスト）は必ずユーザーに提示してから雛形を書く。
   既存テストと references が矛盾するときは**常に既存テストを優先**する（persona の
   「既存コードのスタイルに従う」原則）。

### 入力 (Inputs)

| 名前 | 型 | 必須 | 出どころ | 例 |
|---|---|---|---|---|
| target | string（ファイルパス / 関数名 / クラス名 / "直近の diff"） | yes | ユーザー発話 or 会話文脈 | `src/service/user.py` の `UserService` |
| project_root | path | no（cwd から推定） | 実行ディレクトリ / ユーザー指定 | `projects/foo-app/` |
| scope | enum: `unit` \| `integration` \| `cases-only` | no（デフォルト `unit`） | ユーザー発話 | 「ケース案だけ出して」→ `cases-only` |
| style_override | string（FW 名等） | no | ユーザー発話 | 「jest じゃなくて vitest で」 |

### 出力 (Outputs)

| 名前 | 形式 | どこに置く | 例 |
|---|---|---|---|
| test_scaffold | ソースコードファイル | プロジェクトの既存テスト配置（無ければ言語標準配置を提案して確認） | `tests/service/test_user.py` |
| case_list | markdown 表（分類 / ケース / 入力 / 期待 / 実装状態） | チャット返信 + 雛形内コメント | 正常系 3 / 境界値 2 / 異常系 2 |
| style_report | markdown 箇条書き（検出 FW・参照した既存テスト） | チャット返信 | 「vitest / 併置 / `*.test.ts`」 |
| run_result | テキスト（テストランナー出力の要約） | チャット返信 | `3 passed, 2 skipped` |

## 想定ユーザー (Who)

- 主: SES 参画中のフリーランスエンジニア本人。参画先ごとに言語/FW が変わり、流儀の切り替えコストが高い。
- 副: なし（生成されたテスト自体は参画先チームがレビューする）。

ユーザーが持っている知識: 対象コードの仕様・意図（期待値の最終判断ができる）。
ユーザーが持っていない知識: その言語/FW の最新のテスト流儀の細部（命名・fixture・モック API）。

---

## 既存ツール・スキルとの差別化 (Differentiation)

| 代替 | 限界 | このスキルが上回る点 |
|---|---|---|
| Claude に都度「テスト書いて」と依頼 | 毎回流儀がぶれる。既存テストを見ずに書き、レビューで直される | 流儀検出を必ず先行させ、references で言語別デフォルトも固定 |
| `/verify` スキル | 変更の動作確認が目的で、テスト資産を作らない | 再実行可能なテストコードを成果物として残す |
| IDE のテスト生成機能 | 空のボイラープレートのみで、ケース設計をしない | 正常系/境界値/異常系のケース案を必ず列挙する |
| `pr-reviewer` 候補（未実装） | 指摘するだけで補完しない | 「テストがない」指摘をその場で解消する |

## 成功条件 (Success Criteria)

- [ ] 生成されたテストファイルが**修正なしで**プロジェクトのテストランナーで実行でき、
      skip/todo 以外のケースがパスする
- [ ] 既存テストがあるプロジェクトでは、生成物の命名・配置・アサーションスタイルが
      既存テストと一致している（チーム レビューでの流儀指摘 0 件）
- [ ] ケース案に「正常系 / 境界値 / 異常系」の 3 分類がすべて含まれている
- [ ] ユーザーの手直しが「期待値の具体化（skip 解除）」のみで済み、
      ファイル配置やテスト構造の書き直しが発生しない

## 失敗モード予測 (Failure Modes)

| 失敗 | 起きる条件 | 回避策 |
|---|---|---|
| 流儀の誤検出 | 複数 FW が混在（jest→vitest 移行中等）、古いテストが残っている | 検出結果と参照した既存テストを**書く前に提示**して確認。設定ファイルの hint と食い違えば設定ファイル側（=現行）を優先 |
| 期待値の創作 | 実装から仕様を推測して誤った期待値を断定する | 実装から確信が持てない期待値は skip/todo マーカー + `// ASSUMPTION:` で出力し、実行確認で「落ちるテストを混ぜない」 |
| 既存テストの破壊 | 対象のテストファイルが既に存在する | 上書きせず既存ファイルへの**追記**を基本とし、追記位置と内容を確認してから編集する |
| 雛形の発散 | 対象が巨大ファイル・多責務クラス | 関数/クラス単位に分割し、公開 API のうち重要度上位から着手する旨を提示して合意を取る |
| 環境要因で実行確認が失敗 | 依存未インストール・DB 等の外部依存が必要 | テスト実行失敗を「生成物の欠陥」と「環境要因」に切り分けて報告し、環境要因ならセットアップ手順を提示（勝手にインストールしない） |

## 必要なリソース (Resources)

- 外部 API / CLI: 各言語のテストランナー（`pytest`, `npx vitest` / `npx jest`, `go test`, `mvn test` 等）
- 権限: 対象プロジェクトのファイル読み書き、テストコマンドの実行
- 参照すべき公開知識: 各 FW の公式ドキュメント（references に要点を静的メモ化）
- ローカル前提: 対象プロジェクトが `projects/<project-name>/` 等ローカルに存在し、
  テストが実行可能な状態であること（不能なら実行確認をスキップして明記）

## 実装メモ (Implementation Notes)

- ディレクトリ構成:
  - `SKILL.md` — フロー定義
  - `scripts/detect_test_style.py` — 流儀検出（言語判定 / 設定ファイルの FW hint / 既存テスト探索を JSON で返す。決定論的処理のため script 化）
  - `references/{python,javascript-typescript,go,java}.md` — 言語別デフォルト流儀メモ
- 既存スキルとの依存関係: なし（単体で完結）。`karen-self-evolving-skill-creator` の
  パイプラインは組み込まない（下記）。
- **(ASSUMPTION: 対象は単体テスト（+ 軽量な統合テスト）とし、E2E テストは範囲外とした。
  候補表に粒度の指定が無く、「工数が惜しく後回し」という課題は主にユニットテストで起きると推測)**
- **(ASSUMPTION: 生成テストは参画先プロジェクト（`projects/` 配下等）に書き込むため、
  本リポジトリには成果物を残さない。プロジェクト固有情報の分離ルールに従う)**
- 参画先で新しい言語/FW の流儀を学んだら `references/` に手動で追記して育てる。

## 自己進化を入れるか (Self-Evolution Scope)

- [ ] パイプラインあり
- [x] パイプラインなし — 理由: 本スキルの「学習対象」は言語/FW ごとのテスト流儀だが、
  これは実行ログから自動抽出するより `references/` への**明示的な追記**として育てる方が
  安全で検証しやすい（誤学習した流儀が参画先コードに混入するリスクを避ける）。
  また 1 回の実行結果は対象コード・参画先固有の内容を含むため、ログとして本リポジトリに
  蓄積すること自体がプロジェクト固有情報の分離ルールと相性が悪い。生成物の品質は
  毎回の「実行確認」ステップで観測できるため、ログ駆動の自動書き換えは不要と判断した。
