# db-schema-designer

> 一人でテーブル設計を悩んでいるエンジニアが、設計の壁打ち相手と
> 「ER 図・DDL・設計判断の記録」を一度に得るためのスキル。

---

## 場面 (When)

- ユーザーが新機能の実装前に「テーブル設計を一緒に考えて」と言ったとき
- 既存プロジェクトに機能追加する際「この機能の DB スキーマを設計して」と対象を指したとき
- 設計が頭の中で固まった後「ER 図と DDL を作って」と成果物化を頼んだとき
- 自分で作った設計案を見せて「このテーブル設計をレビューして」と言ったとき
- 「正規化するか迷ってるので壁打ちして」のように特定の論点だけ相談されたとき
- 設計確定後に「マイグレーション案を作って」とファイル案を頼んだとき

### 想定トリガー文 (実際のユーザー発話例)

1. 「テーブル設計を一緒に考えて」
2. 「この機能の DB スキーマを設計して」
3. 「ER 図と DDL を作って」
4. 「このテーブル設計をレビューして」
5. 「正規化するか迷ってるので壁打ちして」
6. 「マイグレーション案を作って」
7. 「ユーザーと注文のテーブル、どう持つのがいいか相談したい」

---

## 解決する課題 (Problem)

- 現状: フリーランス / 少人数の開発では、テーブル設計をレビューしてくれる相手がいない。
  正規化の度合い・主キー戦略・インデックスの判断を一人で決め、根拠が記録に残らないまま
  実装に進む。数ヶ月後に「なぜこの形にしたのか」が誰にも（自分にも）分からなくなる。
  設計の見落とし（NULL の意味の曖昧さ、金額の float、インデックス不足）は
  データが増えてから発覚し、直すコストが跳ね上がる。
- 痛みの種類: 見落とし（レビュー不在）/ 手戻り（後からのスキーマ変更は高コスト）/
  認知負荷（トレードオフを一人で抱える）/ 記録喪失（判断根拠が残らない）
- 影響範囲: 個人作業（設計・実装）〜プロダクション（スキーマは全機能の土台で変更コストが高い）

---

## 解決策 (Solution)

1. 新規設計か既存スキーマへの追加かを確認する。既存なら現行スキーマ（マイグレーション・ER・
   モデル定義）を読み、命名規約・主キー戦略・共通カラム・DB エンジン・マイグレーションツールを
   検出してそれに合わせる
2. 要件を対話で 2〜3 問ずつヒアリングする（エンティティと関係 / データ量見込み /
   主要なアクセスパターン / 整合性要件 / 履歴・監査の要否）。
   インデックス設計はアクセスパターン確定前に決めない
3. 設計案を提示する。判断が分かれるポイント（正規化 vs 非正規化、ポリモーフィック関連の表現等）は
   選択肢とトレードオフを示して確認し、それ以外は案として直接提示する
4. 成果物を生成する: Mermaid `erDiagram` / DB エンジン方言の DDL /
   マイグレーションツールに合わせたファイル案 / 設計判断の記録（採用理由・不採用案・ASSUMPTION）
5. references のチェックリスト（9 観点）でセルフレビューを 1 周回し、懸念を指摘リストとして添付する
6. 設計記録を対象プロジェクト配下 `docs/db-design/` 等に保存する（保存前に確認）。
   マイグレーションファイルの作成はコード変更なのでユーザー確認後
7. 完了条件: ER 図・DDL・判断記録・指摘リストが揃い、全推測に ASSUMPTION が付いていること

### 入力 (Inputs)

| 名前 | 型 | 必須 | 出どころ | 例 |
|---|---|---|---|---|
| repo_path | string (絶対パス) | no (新規設計なら不要) | ユーザー発話 / cwd | `projects/acme-api/` |
| design_target | string (機能・エンティティの説明) | yes | ユーザー発話 | 「サブスクリプション課金のテーブル」 |
| requirements | 対話回答 (関係・量・アクセスパターン・整合性・履歴) | yes | ヒアリング | 「一覧はユーザー別に月で絞る」 |
| existing_design | markdown / DDL | no (レビューだけ頼まれたとき) | ユーザー提示 | 自作の CREATE TABLE 文 |

### 出力 (Outputs)

| 名前 | 形式 | どこに置く | 例 |
|---|---|---|---|
| er_diagram | Mermaid `erDiagram` | 設計記録内 + チャット | users -- orders の 1:N 図 |
| ddl | SQL (プロジェクトの方言) | 設計記録内 | `CREATE TABLE orders (...)` |
| migration_draft | マイグレーションツールの形式 | ユーザー確認後にリポジトリ内 | Prisma migration / Rails migration |
| design_record | markdown (判断記録 + 指摘リスト) | `<repo>/docs/db-design/<YYYY-MM-DD>-<topic>.md`（保存前に確認） | 採用理由・不採用案・ASSUMPTION・懸念一覧 |

## 想定ユーザー (Who)

- 主: SES / フリーランスで開発中のエンジニア本人。設計レビュー相手がおらず、判断の記録も残したい
- 副: 後からプロジェクトに入る人・未来の自分（設計記録の読み手）

ユーザーが持っている知識: 作りたい機能の要件、SQL と ORM の基本。
持っていない知識: 自分の設計の見落とし（NULL の曖昧さ・インデックス不足・型の罠）と、
トレードオフの相場観（連番 vs UUID、論理削除の副作用等）。

---

## 既存ツール・スキルとの差別化 (Differentiation)

| 代替 | 限界 | このスキルが上回る点 |
|---|---|---|
| ORM のスキーマ定義をいきなり書く | 書けるが、判断の根拠が残らずレビューも入らない | ヒアリング → トレードオフ提示 → 判断記録の壁打ちプロセス込み |
| dbdiagram.io 等の作図ツール | 図は描けるが、設計判断の相談相手にはならない | アクセスパターン起点の設計相談と 9 観点セルフレビュー |
| arc-reactor-slow-query-hunter | 遅くなった後の対処。設計時点の予防はしない | 行数見込みとアクセスパターンから設計段階で懸念を指摘 |
| db-migration-safety-checker | 適用時の安全性（ロック等）が対象で、スキーマの形は決めない | スキーマの形とその根拠を決める前工程を担う（既存テーブル変更時はそちらへ接続） |
| friday-design-doc-generator | 既存コードの現在形をドキュメント化する。新規設計の壁打ちはしない | まだ存在しないスキーマを対話で作る |

## 成功条件 (Success Criteria)

- [ ] 成果物 4 点（ER 図・DDL・マイグレーション案・判断記録）が 1 セッションで揃う
- [ ] 判断記録に、判断が分かれた各ポイントの採用理由と不採用案が 1 行以上ずつ書かれている
- [ ] ヒアリングで確認できなかった前提がすべて ASSUMPTION として明記されている
- [ ] 既存スキーマへの追加時、生成された DDL の命名・主キー・共通カラムが現行規約と一致している
- [ ] セルフレビューの指摘リストに、9 観点それぞれの「指摘 or 確認済み」が載っている

## 失敗モード予測 (Failure Modes)

| 失敗 | 起きる条件 | 回避策 |
|---|---|---|
| アクセスパターン不明のままインデックスを「一般的な形」で付けて外す | ヒアリングを省略して成果物を急いだとき | フローで順序を固定: インデックス設計はアクセスパターン確定後。未確定なら「インデックスは未設計」と明記して ASSUMPTION に落とす |
| 既存プロジェクトの規約と食い違うスキーマを提案（複数形 vs 単数形、UUID vs 連番の混在） | 文脈把握（フロー 1）を飛ばして新規設計のつもりで進めたとき | 最初に必ず新規 / 既存を確認し、既存なら現行スキーマを読んでから設計する。規約と異なる提案は理由付きで確認 |
| ヒアリングが尋問化してユーザーが疲れる | 全質問を一度に並べたとき | 2〜3 問ずつに固定し、回答から次の質問を絞る |
| マイグレーション案をそのまま本番適用されて事故る | 生成物が「安全確認済み」に見えたとき | 実適用は範囲外と SKILL.md・成果物の双方に明記。既存テーブル変更時は db-migration-safety-checker でのチェックを毎回提案 |
| 設計記録が対象プロジェクト外（当リポジトリ側）に保存され情報が混在する | 保存先の確認を省いたとき | 保存先は対象プロジェクト配下に固定し、保存前に必ず確認する |

## 必要なリソース (Resources)

- 外部 API / CLI: なし（`rg` / `git` によるローカルのスキーマ・マイグレーション読解のみ。
  ネットワークアクセス不要）
- 権限: 対象リポジトリの読み取り。設計記録・マイグレーションファイルの書き込みは確認後のみ
- 参照すべき公開知識: 正規形（1NF〜BCNF）、各 DB エンジンの型と制約の仕様、
  マイグレーションツールのファイル形式
- ローカル前提: 既存スキーマへの追加の場合、対象リポジトリがローカルにあること
  （新規設計なら前提なし）

## 実装メモ (Implementation Notes)

- ディレクトリ構成: `.claude/skills/arc-reactor-db-schema-designer/`
  - `SKILL.md` … 目的・トリガー・標準フロー（文脈把握 → ヒアリング → 設計案 → 成果物 →
    セルフレビュー → 保存）・品質規約
  - `references/schema-review-checklist.md` … セルフレビュー 9 観点
    （正規化 / 主キー戦略 / インデックスとアクセスパターン / NULL の意味 / 論理削除 /
    時刻とタイムゾーン / 金額の型 / 履歴・監査 / マルチテナント分離）の詳細
- scripts は作らない: スキーマ読解・DDL 生成は Claude の読解と生成が主で、
  DB エンジン・ツールごとの分岐をスクリプトに固定すると柔軟性を失う
- 既存スキルとの依存関係: db-migration-safety-checker へ接続（既存テーブル変更を伴う設計のとき
  チェックを提案）。arc-reactor-slow-query-hunter とは予防 / 対処で補完関係
- persona: `personas/arc-reactor.md` に従う（応答は日本語、コード・識別子は英語、推測は ASSUMPTION 明記）
