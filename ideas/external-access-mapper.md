# external-access-mapper

> SES 参画エンジニアが、参画直後・障害調査・セキュリティレビューの前に、
> 対象コードベースの「外部に繋がっている場所と、それがいつ使われるか」を台帳 1 本で把握するためのスキル。

---

## 場面 (When)

- 新しい案件に参画した直後、「このシステムは何と繋がっているのか」を把握したいとき
- 障害対応で「この外部サービスが落ちたら、うちのどの機能が死ぬか」を逆引きしたいとき
- セキュリティレビューの前に、外部との接点（攻撃面・データ流出経路）を棚卸ししたいとき
- DB 移行・API 乗り換えの前に、影響を受ける呼び出し箇所と経路を洗い出したいとき
- クライアントに「このシステムの外部依存一覧」を求められて、資料の下敷きが要るとき

### 想定トリガー文 (実際のユーザー発話例)

1. 「このリポジトリの外部アクセスを洗い出して」
2. 「外部 API と DB の接続箇所を台帳にして」
3. 「このサービスがどの外部サービスに繋がってるかマップにして」
4. 「障害調査用に DB アクセス箇所と呼び出し元を整理して」
5. 「Stripe を呼んでる場所と経路を全部出して」
6. 「external access map を作って」

---

## 解決する課題 (Problem)

- 現状: 外部依存の把握は `rg` で SDK 名や URL を手で grep して回る場当たり作業になっている。
  接続箇所は見つかっても「どのユースケースから呼ばれるか」までは追わないので、
  障害時に影響範囲を答えられない。ドキュメントがある案件でも大抵古い。
- 痛みの種類: 時間（把握に数時間〜数日）/ 見落とし（監視 SDK や起動時の自動送信など
  「知らない外部通信」）/ 認知負荷（障害時に頭の中だけで経路を逆引きする）
- 影響範囲: 個人作業（参画直後の立ち上がり）に加え、障害対応・セキュリティレビューでは
  プロダクションの判断材料になる

---

## 解決策 (Solution)

1. 対象リポジトリのパスと任意のスコープ（「DB だけ」等）を受け取り、除外パスを確定する
2. references の検出パターン（依存マニフェスト / 設定・環境変数 / コード直接検索の 3 系統）で
   7 カテゴリの外部アクセス箇所を収集する
3. 箇所ごとに接続先・認証方式を特定して台帳化する（値は転記しない。推定は (推測) 明記）
4. エントリポイント（HTTP ルート / ジョブ / CLI / イベントハンドラ）から各アクセス箇所までの
   呼び出し経路を `file:symbol` の連鎖で追跡する。追えない箇所は「経路不明・要確認」
5. 台帳 + 利用場面マップの Markdown レポートを生成し、保存確認のうえ
   `<repo>/external-access-map/<YYYY-MM-DD>.md` に置く

### 入力 (Inputs)

| 名前 | 型 | 必須 | 出どころ | 例 |
|---|---|---|---|---|
| repo_path | string | yes | ユーザー発話 / cwd | `projects/foo-api/` |
| scope | string | no | ユーザー発話 | 「DB だけ」「認証系だけ」「Stripe だけ」 |

### 出力 (Outputs)

| 名前 | 形式 | どこに置く | 例 |
|---|---|---|---|
| 外部アクセス台帳 + 利用場面マップ | markdown | `<repo>/external-access-map/<YYYY-MM-DD>.md`（保存前に確認） | 台帳 18 件 + 経路マップ + 免責 |
| 要約 | チャット返信 | チャット | カテゴリ別件数・経路不明・セキュリティ要報告 |

---

## 想定ユーザー (Who)

- 主: SES 参画中のフリーランスエンジニア本人（参画直後の把握、障害対応、レビュー準備）
- 副: レポートを渡す先のクライアント側エンジニア（外部依存一覧の共有資料の元ネタ）

本人はコードは読めるが、初見のリポジトリの外部境界の全量は知らない。
クライアント側の読み手は自システムの事情は知っているので、レポートは根拠 `file:line` で
即検証できることが重要になる。

---

## 既存ツール・スキルとの差別化 (Differentiation)

| 代替 | 限界 | このスキルが上回る点 |
|---|---|---|
| 手動 `rg` | 知っている SDK 名しか探せず、検索観点が毎回場当たり | 7 カテゴリ × 3 系統の固定検出パターンで網羅的に走査し、パターンを references に蓄積する |
| `arc-reactor-codebase-onboarding` | 全体地図が目的で、外部境界は網羅しない | 外部境界に特化し、認証方式・利用場面まで台帳化する |
| `arc-reactor-tech-debt-auditor` | 負債の列挙が目的で、外部依存は依存の老朽化しか見ない | 良し悪しの判定はせず、接続の事実と経路を全量記録する |
| 依存グラフツール (madge 等) | モジュール間依存しか出ず、外部サービス・認証方式は出ない | 環境変数・設定・IaC まで突き合わせて接続先を特定する |
| SBOM / dependency scan | パッケージ一覧は出るが、実際に呼ぶ箇所と経路は出ない | `file:line` と呼び出し経路まで落とす |

## 成功条件 (Success Criteria)

- [ ] 台帳の全項目に `file:line` が付いており、読み手がその場で検証できる
- [ ] レポートに接続文字列・API キー等の値が 1 件も転記されていない
- [ ] 「〇〇（外部サービス）が落ちたらどの機能に影響するか」に利用場面マップだけで答えられる
- [ ] 経路を追えなかった箇所が推測で埋められず「経路不明・要確認」+ 確認方法で残っている
- [ ] 実案件のリポジトリで、手動 grep で見つけていなかった外部アクセスを 1 件以上拾える

---

## 失敗モード予測 (Failure Modes)

| 失敗 | 起きる条件 | 回避策 |
|---|---|---|
| 機密値の転記 | 接続文字列やキーがハードコードされている | 値は読んでも書かないルールを SKILL.md と references の両方に明記。発見時は場所のみ最優先報告 |
| 経路の創作 | DI・動的ディスパッチで静的に追えないのに経路を推測で埋める | 「経路不明・要確認」への分類を必須化し、確認方法を添えて断定しない |
| 検出漏れを全量と誤読させる | 動的 URL 組み立て・プラグイン機構・実行時設定 | レポートの免責に静的調査の限界を固定文で明記する |
| 大規模リポジトリで走査が破綻 | 10 万行超のモノレポ | 規模を先に概算し、カテゴリ単位・ディレクトリ単位の分割実行を提案する |
| ローカル依存の誤登録 | SQLite・インメモリキャッシュ等を外部扱いする | references の注意点で除外を定義（外部ネットワークを跨ぐものだけ載せる） |

## 必要なリソース (Resources)

- 外部 API / CLI: なし（`rg` / `git` / Read のみ。ネットワークアクセスはしない）
- 権限: 対象リポジトリの読み取り。レポート保存時のみ書き込み（保存前に確認）
- 参照すべき公開知識: 主要言語のクライアントライブラリ名・SDK シンボル・
  フレームワークのルーティング規約（references/access-patterns.md に集約）
- ローカル前提: 対象リポジトリが `projects/<name>/` 等ローカルに clone 済みであること

---

## 実装メモ (Implementation Notes)

- ディレクトリ構成: `SKILL.md` + `references/access-patterns.md` の 2 ファイル。scripts なし
  （検出は rg ベースの探索で、決定論的計算が無いためスクリプト化の必要がない）
- 検出パターン知識（7 カテゴリの検索パターン・エントリポイント検出・注意点）は
  references に切り出し、スキャン前に必読とする
- レポートはプロジェクト固有情報として対象リポジトリ配下に閉じる（CLAUDE.md の分離ルール）
- 棲み分け: codebase-onboarding は全体地図、本スキルは外部境界特化。
  経路追跡部分は将来 sequence-diagram-generator を作る場合に共通化余地あり
