# invoice-builder

> SES 参画中のフリーランスエンジニアが、毎月の締め日後に、稼働時間と単価から検算済みの請求書（適格請求書対応）を数分で作るためのスキル。

---

## 場面 (When)

- 月末〜月初、取引先（SES 商流の契約先）への請求書提出期限が迫り、ユーザーが「今月の請求書を作って」と言ったとき
- 勤怠を締めて確定稼働時間（例: 160.5h）が出た直後に、「6月分は 160.5 時間だった。請求書にして」と稼働時間込みで依頼されたとき
- timesheet-aggregator（または worklog）の稼働サマリを指して「この稼働から請求書と内訳を作って」と言ったとき
- 手作りした請求書やスプレッドシートの数字に不安があり、「この請求書の金額が合ってるか検算して」と言ったとき
- 単価交渉・精算幅の見直し前に、「単価 5,200 円で 170 時間だと請求額いくらになる？」と試算を求められたとき
- 前月の請求書を流用したいとき、「先月の請求書を雛形に今月分を作って」と言ったとき

### 想定トリガー文 (実際のユーザー発話例)

1. 「今月の請求書を作って」
2. 「6月分の稼働 160.5 時間で請求書にして」
3. 「この請求書の金額が合ってるか検算して」
4. 「稼働ログから請求書と内訳を作って」
5. 「単価5,000円で162時間分の請求額を計算して」
6. 「先月の請求書を雛形に今月分を作って」
7. 「超過分込みで6月の請求書お願い」

---

## 解決する課題 (Problem)

- 現状: 毎月、稼働集計 → 精算幅（超過/控除）の計算 → 小計・消費税・源泉徴収の計算 → 請求書フォーマットへの転記を手作業で行っており、30〜60 分かかる。電卓・スプレッドシートでの計算は端数処理（切り捨て位置）や源泉徴収の 100 万円境界でミスが起きやすく、適格請求書の記載要件（登録番号・税率ごとの区分記載）の抜けも起こる。
- 痛みの種類: 時間（毎月の定型作業）＋見落とし（計算ミス・記載要件漏れは取引先への再発行や入金遅延に直結）＋認知負荷（税率・端数・源泉のルールを毎回思い出す）
- 影響範囲: 個人事業の売上請求そのもの（金額を誤ると信用・キャッシュフローに直接影響する）

---

## 解決策 (Solution)

1. 取引先設定（`invoice-data/config/billing.json`: 請求元・登録番号・単価・精算条件）を読む。無ければ対話で作成
2. 対象月・稼働時間・超過/控除・源泉徴収の有無をヒアリングし、ユーザーに復唱して確定
3. Claude が明細 JSON（基本稼働・超過・控除の行）に整形
4. **決定論スクリプト** `bin/calc_invoice.py calc` が小計・税率別消費税・源泉徴収・差引請求額を Decimal で計算し、Fraction の独立パスで全項目を検算（`checks_passed`）
5. 雛形 `templates/invoice.md` に計算結果を**転記のみ**して請求書 Markdown を生成、`invoice-data/invoices/<YYYY-MM>/` に保存（計算入出力 JSON も証跡として保存）
6. 差引請求額・内訳・計算根拠（式）・検算結果・免責をチャットに提示して完了

**核となる設計**: 金額計算は Claude が暗算せず、python スクリプトが計算・検算する。Claude の役割は入力整理・分類・文面のみ。

### 入力 (Inputs)

| 名前 | 型 | 必須 | 出どころ | 例 |
|---|---|---|---|---|
| target_month | string (YYYY-MM) | yes | ユーザー発話 | "2026-06" |
| client | string (client id) | yes | 発話 or billing.json | "client-a" |
| worked_hours | number (小数可) | yes | 発話 / timesheet-aggregator のサマリ / 稼働ログ | 160.5 |
| unit_price | integer (円) | yes | billing.json（発話で上書き可） | 5000 |
| settlement | object（精算幅・超過/控除単価） | no | billing.json | {lower: 140, upper: 180, ...} |
| withholding | boolean | yes（既定 false、必ず確認） | billing.json / 発話 | false |
| line_items | array（上記から Claude が生成する明細） | yes | Claude が整形 | [{description, quantity, unit_price, tax_rate}] |
| expected | object（検算モードのみ） | no | 既存請求書の数値 | {subtotal: 802500, ...} |

### 出力 (Outputs)

| 名前 | 形式 | どこに置く | 例 |
|---|---|---|---|
| 請求書 | markdown（適格請求書の記載要件 6 点＋源泉徴収内訳） | `invoice-data/invoices/<YYYY-MM>/<client>-<YYYYMM>.md` | 明細表＋内訳表＋発行者情報 |
| 計算証跡 | json（calc への入力＋出力＋checks） | 同ディレクトリ `<client>-<YYYYMM>.calc.json` | {"subtotal": 802500, "checks_passed": true, ...} |
| 要約 | markdown（チャット返信） | チャット | 差引請求額・計算式・検算結果・免責 |
| 検算レポート（verify 時） | json → チャット要約 | チャット | mismatches: [{field, expected, computed}] |

## 想定ユーザー (Who)

- 主: SES 参画中のフリーランスエンジニア本人（毎月 1〜2 通の請求書を発行。インボイス登録済み **(ASSUMPTION: 未登録の場合も登録番号欄を「（未登録）」として出力できる設計にした)**）
- 副: 将来的に複数取引先（副業案件・スポット受託）を持った場合の本人

持っている知識: 自分の単価・精算条件・締め支払いサイト。持っていない知識: 源泉徴収の境界計算（100 万円超 20.42%）や適格請求書の端数処理ルール（税率ごとに 1 回）を毎回は覚えていない。

## 既存ツール・スキルとの差別化 (Differentiation)

| 代替 | 限界 | このスキルが上回る点 |
|---|---|---|
| スプレッドシートの自作雛形 | 端数処理・源泉境界の式ミスに気づけない。記載要件の変化に追従しない | 検算 2 系統パス＋記載要件を雛形で担保 |
| クラウド請求書サービス（freee, misoca 等） | 月額費用。稼働ログ→明細への変換は結局手作業 | 会話から明細化まで一気通貫。ローカル完結で機密が外に出ない |
| Claude に直接「計算して」と頼む | LLM の暗算は桁・端数を誤り得る。再現性がない | 金額は決定論スクリプトのみが計算し、独立再計算で検算 |
| ultron-timesheet-aggregator | 稼働の集計まで（請求書・金額計算はスコープ外） | 稼働サマリを入力として受け、請求書まで仕上げる後段工程 |

## 成功条件 (Success Criteria)

- [ ] 「今月の請求書を作って」から請求書 Markdown 保存まで、ユーザーの応答待ちを除き 5 分以内に完了する
- [ ] 生成された全請求書で `calc_invoice.py` の `checks_passed: true` が成立し、`.calc.json` 証跡が必ず残っている（金額の手計算ゼロ）
- [ ] 検算モードで、意図的に 1 円ずらした請求書を渡すと `mismatches` が該当項目を正しく指摘する
- [ ] 出力請求書が適格請求書の記載要件 6 点（登録番号・取引年月日・取引内容・税率別合計額と適用税率・税率別消費税額・交付先名称）をすべて満たしている
- [ ] 3 ヶ月連続で運用して、取引先からの金額・記載不備による差し戻しが 0 件

## 失敗モード予測 (Failure Modes)

| 失敗 | 起きる条件 | 回避策 |
|---|---|---|
| 金額の計算ミス（LLM の暗算・端数誤り） | Claude が JSON 整形時や転記時に自分で計算してしまう | 金額は `calc_invoice.py` の出力のみ転記可とルール化。スクリプト自体も Decimal 計算＋Fraction 独立再計算の 2 系統検算で `checks_passed` を出し、false なら exit 非 0 で止まる |
| 税区分・税率の誤り（8%/10% 混同、明細ごと端数処理で税額がズレる） | 明細に軽減税率対象が混ざる／雛形を手で埋める | 税率は明細行ごとに明示し、消費税は**税率ごとに 1 回だけ**端数処理する仕様をスクリプトに固定。税率別内訳表を雛形に組み込み目視確認可能にする |
| 源泉徴収の要否・境界の誤り（対象外契約に源泉を付ける／100 万円超で 10.21% のまま） | 契約内容の思い込み。境界計算の失念 | 要否は既定 OFF＋毎回ユーザーに確認（推測で確定しない）。境界計算（102,100 円＋超過分 20.42%）はスクリプトが自動で分岐し、式を `formulas` で提示 |
| 稼働時間の取り違え（前月分・別案件の時間で請求） | 稼働ログの集計を無確認で採用 | 集計値は必ず「合計 ○○h（内訳）で請求してよいか」と復唱確認してから明細化する |
| 取引先実名・口座情報の漏出 | 請求書データを git 管理下やスキルディレクトリに置く | データは git 管理外の `invoice-data/` のみ（`.gitignore` 登録）。スキルディレクトリの example には実名を書かない |
| インボイス記載要件の漏れ（登録番号なし等） | 雛形を使わずフリーフォーマットで出力 | 記載要件 6 点を織り込んだ `templates/invoice.md` を必須経由にする。未登録なら「（未登録）」明記＋注意喚起 |

## 必要なリソース (Resources)

- 外部 API / CLI: なし（`python3` 標準ライブラリのみ。オフライン完結）
- 権限: リポジトリ直下 `invoice-data/` への読み書き
- 参照すべき公開知識: 適格請求書等保存方式の記載事項（国税庁）、報酬・料金等の源泉徴収税額の計算（10.21% / 100 万円超部分 20.42%）、消費税の端数処理（税率ごとに 1 回）
- ローカル前提: onclimb-industries リポジトリ内で実行。取引先設定 `invoice-data/config/billing.json`（初回に対話で作成）

## 実装メモ (Implementation Notes)

- ディレクトリ構成: `.claude/skills/ultron-invoice-builder/`（`SKILL.md` / `bin/calc_invoice.py` / `config/defaults.json` / `templates/invoice.md`, `templates/config.example.json`）

- scripts として切り出す処理: 金額計算・検算のすべて（`calc` / `verify` の 2 サブコマンド。float 不使用、法定率はスクリプト内定数）
- 前段の稼働集計は ultron-timesheet-aggregator の出力（timesheet-data/ の稼働サマリ）を入力にできる
- references は現状なし（税率・記載要件は SKILL.md と雛形コメントに埋め込み。法改正時はそこを更新）
