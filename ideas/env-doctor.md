# env-doctor

> 環境構築やローカル作業でエラーに当たったエンジニアが、原因を体系的に切り分けて自分の環境を直し、
> 同じエラーで次の人がハマらないよう手順書の修正案まで一気に作るためのスキル。

---

## 場面 (When)

- ユーザーが「npm install がコケるので直して」とエラー出力を貼ったとき
- 参画直後・マシン移行直後のセットアップで「手順どおりにやったのにエラーになる」と言ったとき
- 「ローカルでサーバーが起動しない、原因調べて」と起動失敗を相談されたとき
- OS 更新や brew upgrade の後に「昨日まで動いてたのに壊れた」と言ったとき
- 環境エラーを解決した後に「直したついでに README の手順も直しておいて」と言ったとき

### 想定トリガー文 (実際のユーザー発話例)

1. 「npm install がコケるので直して」
2. 「この環境構築エラーを診断して」
3. 「ローカルでサーバーが起動しない、原因調べて」
4. 「セットアップ手順どおりにやったのにエラーになる」
5. 「このエラー、環境のせいか調べて直して」
6. 「直したついでに README の手順も直しておいて」
7. 「新しい Mac でセットアップしたら動かない、見て」

---

## 解決する課題 (Problem)

- 現状: 環境エラーは Stack Overflow の断片的な解を場当たりに試して直すことが多く、
  何が効いたのか記録が残らない。しかも直しても手順書は古いままなので、次に環境を作る人
  （未来の自分・同僚・新規参画者）が同じエラーで同じ時間を溶かす。切り分けの定石
  （バージョン → PATH → 権限 → …）を毎回頭から思い出すのも無駄が大きい。
- 痛みの種類: 時間（場当たりな再試行）/ 手戻り（同じエラーの再発）/
  見落とし（手順書が直らないまま次の人がハマる）
- 影響範囲: 個人作業（自分の環境復旧）〜チーム共有物（README / setup ドキュメント）

---

## 解決策 (Solution)

1. 情報収集 — エラー全文・実行コマンド・環境情報（OS / ランタイムバージョン /
   パッケージマネージャ / 直近の環境変更）を集める。不足分は取得コマンドを実行して自分で集める
2. 原因診断 — references のチェックリスト順に切り分ける。最初に「環境起因」か「コード起因」かを
   判定し、コード起因なら範囲外と伝えて通常のデバッグに切り替える。仮説には必ず根拠を添える
3. 修復 — プロジェクト内に閉じる操作は実行して報告。システム状態を変える操作
   （インストール / 設定ファイル変更 / グローバル設定）は実行前に一言確認する
4. 検証 — 元の失敗コマンドの再実行で、元のエラーが再現しないことを確認する
5. 手順書へ反映 — README / docs / setup スクリプトを探し、手順の不備・古さが原因なら
   修正 diff 案を提示。手順書が無ければトラブルシューティング節の追記候補を提案。
   対象リポジトリへの書き込みはユーザー確認後
6. 対応記録 — 試したこと・効かなかったこと・最終解決策を短く提示（worklog に拾われる形）
7. 完了条件: 元のコマンドの再実行で検証済み、かつ手順書反映の判断（diff 案提示 or 不要の明示）まで済んでいること

### 入力 (Inputs)

| 名前 | 型 | 必須 | 出どころ | 例 |
|---|---|---|---|---|
| error_output | string (エラー全文) | yes | 発話 / ターミナル出力 / ログファイル | `EADDRINUSE: address already in use :::3000` |
| repo_path | string (絶対パス) | yes | ユーザー発話 / cwd | `projects/acme-api/` |
| failed_command | string | yes (出力から特定できれば発話不要) | ユーザー発話 / エラー出力 | `npm run dev` |
| env_info | OS / バージョン / 直近の環境変更 | no (不足分はコマンド実行で収集) | ユーザー発話 + 取得コマンド | `macOS arm64, node 20.11` |

### 出力 (Outputs)

| 名前 | 形式 | どこに置く | 例 |
|---|---|---|---|
| diagnosis | markdown (原因と根拠) | チャット返信 | 「原因: node 実測 18.x vs engines >=20。根拠: `package.json:5`」 |
| fix_result | 実施した修復と検証結果 | チャット返信 | 「mise で 20.11 に切替 → `npm run dev` 再実行で起動確認」 |
| doc_diff | 手順書の修正 diff 案 | チャット返信（適用はユーザー確認後に対象リポジトリへ） | README の前提バージョン更新 diff |
| record | 対応記録 (試行 / 効かなかった手 / 解決策) | チャット返信 | worklog に拾われる短いまとめ |

## 想定ユーザー (Who)

- 主: SES / フリーランスで参画中のエンジニア本人。参画直後のセットアップや日々のローカル作業で環境エラーに当たる
- 副: 同じリポジトリを次にセットアップする人（修正された手順書の受益者）

ユーザーが持っている知識: 何をしようとして失敗したか（実行したコマンドと目的）。
持っていない知識: エラーの原因カテゴリと切り分けの定石、手順書のどこが古いか。

## 既存ツール・スキルとの差別化 (Differentiation)

| 代替 | 限界 | このスキルが上回る点 |
|---|---|---|
| Stack Overflow / Web 検索の場当たり対応 | 自分の環境に合うかは運。何が効いたか残らない | チェックリスト順の切り分けと、根拠付き診断・対応記録 |
| Claude に素でエラーを貼って聞く | 直して終わり。検証と手順書反映が抜ける | 元コマンド再実行での検証と、手順書 diff 案までをフローに固定 |
| `arc-reactor-code-review` | 対象はコードの品質で、環境・セットアップは見ない | 環境起因の問題に特化（コード起因と判明したら手を引く棲み分け） |
| `arc-reactor-codebase-onboarding` | コードの読み方の資料化で、環境の故障対応はしない | エラーという事象起点で診断・修復・ドキュメント修正まで行う |

## 成功条件 (Success Criteria)

- [ ] 修復後、元の失敗コマンドの再実行による検証が 100% 実施されている（「直ったはず」で終わる回が 0）
- [ ] 診断結果に必ず根拠（実測値・エラー原文の該当行・ファイルパス）が付いている
- [ ] 手順の不備が原因だった回では、手順書の修正 diff 案が毎回提示されている
- [ ] システム状態を変える操作が無確認で実行された回が 0
- [ ] 同一リポジトリで同じ原因カテゴリのエラーに 2 度目に当たる頻度が下がる（手順書反映の効果）

## 失敗モード予測 (Failure Modes)

| 失敗 | 起きる条件 | 回避策 |
|---|---|---|
| 「ついで」の環境変更でユーザーの他プロジェクトを壊す | グローバルなバージョン切替・brew 一括アップグレードを気軽に実行したとき | システム状態を変える操作は実行前に必ず確認。影響範囲（他プロジェクトに及ぶか）を添えて聞く |
| 同じ失敗コマンドの無限再試行で時間を溶かす | 仮説なしに再実行を繰り返すとき | 再試行は環境を変えた後だけ。3 回切り分けて進展がなければ状況を整理してユーザーに相談 |
| コードのバグを環境問題と誤診して環境をいじり回す | スタックトレースを読まずに環境から疑ったとき | フロー冒頭で環境起因/コード起因のトリアージを必須化。コード起因なら手を引いて通常デバッグに切替 |
| `.env` の中身を診断出力に転記して機密が漏れる | 環境変数欠落の診断で実ファイルを表示したとき | 値は出力せず「変数 X が未設定/設定済み」の形でのみ報告する |
| 未検証の手順を手順書に書いて別 OS の人を新たにハマらせる | macOS で直した内容を Linux 手順にも断定形で書いたとき | 手順書修正は自分の環境で確認できた事実の範囲。他 OS への言及は「(推測)」と明記 |

## 必要なリソース (Resources)

- 外部 API / CLI: `git`, `rg`, `which`, `lsof`, `uname`, 各ランタイム・パッケージマネージャの
  バージョン確認コマンド。ネットワークアクセスは基本不要（パッケージ再インストール時を除く）
- 権限: 対象リポジトリの読み取り + 修復コマンドの実行。手順書への書き込みはユーザー確認後のみ
- 参照すべき公開知識: 各エコシステムの典型エラーと対処（node-gyp / pyenv / Docker for Mac 等）。
  スキル内では `references/common-causes.md` に整理して持つ
- ローカル前提: エラーが再現するローカル環境で実行すること（再現しない環境からの遠隔診断は精度が落ちる）

## 実装メモ (Implementation Notes)

- ディレクトリ構成: `.claude/skills/arc-reactor-env-doctor/`
  - `SKILL.md` … 大原則（検証なくして完了なし）・6 段階フロー・操作区分（プロジェクト内 / システム変更）・品質規約
  - `references/common-causes.md` … 頻出原因 8 カテゴリ（トリアージ + バージョン不整合 / PATH / 権限 /
    ポート衝突 / ネイティブ依存 / 環境変数 / OS 差異 / 古いキャッシュ）の症状・確認コマンド・修復・手順書反映観点
- scripts は作らない: 診断は環境ごとに違うコマンドの組み合わせで、固定スクリプト化に向かない。
  確認コマンドの定石は references に知識として持つ
- 既存スキルとの依存関係: なし。対応記録は jarvis-worklog に拾われる形でチャットに出す（直接連携はしない）
- persona: `personas/arc-reactor.md` に従う（応答は日本語、コード・識別子は英語、推測は ASSUMPTION 明記）
