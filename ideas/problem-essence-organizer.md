# problem-essence-organizer

> **手段に飛び付かずに課題の本質を整理するためのスキル**

### 2026-05-19 — kickoff

- テーマ: フリーランスエンジニアがプロジェクトに参画した時に「必ず」使える思考整理補助スキル
- 「選択肢を提示して誘導しない」「フラットに考える」方針で進める
- 場面 → 課題 → 解決策 の順で言語化していく

### 2026-05-19 — round 2: 起点となる問題意識

ユーザーから出た要件:

- **クライアント側の状況**: 業務委託の現場では、商談時点で「やりたいこと」は語られるが、**現状の何が課題なのかは明確化されていない** ことが多い → 課題の明確化フェーズが先に要る
- **自分側の状況**: AI 導入で思考・開発プロセスが圧縮された結果、
  - 課題を複雑に考えすぎる
  - 目的を見失って具体の手法 (How) に目が入ってしまう
- **欲しいサポート**: 次の 3 つを **分離して** 考えられるようにすること
  1. 今、何が課題か (Problem)
  2. どうなったら課題を解決したと言えるか (Done / Goal state)
  3. それを叶える手段として何があるか (How / Options)
- **副次目的**: 上記を整理したものを **資料化してクライアントに提案** できる形にしたい

### 関連しそうな公共知識 (確定前メモ)

`refining` 以降で照合する候補:
- Issue tree / イシューからはじめよ (安宅) — 課題分解
- Job-to-be-Done — 顧客が達成したい状態を起点に置く
- OKR / Outcome over Output — Done の言語化
- マッキンゼー流 SCQA / 空・雨・傘 — 事実 / 解釈 / 行動の分離

### 2026-05-19 — round 3: 実体験 2 シーン

**シーン 1: 商談中 (クライアント主導の手段先行)**
- 状況: 現場の最初のクライアント面談。RAG を使った業界ドキュメント網羅型のチャットボットを作りたい、という要望
- ユーザーの感覚: 「目的が手段になっているな」とふんわり感じた
- 取れた行動: 従来 RAG → Agentic RAG など **代替手段** の提案はできた
- 取れなかった行動: 「そもそもそれで解決したい課題は何か」を遡る議論
- 失った機会: 信頼度を獲得するチャンス

**シーン 2: 参画後の検証 (自分の手段先行)**
- 状況: AI Agentic Search の精度 / レスポンス速度などで UX 改善の手段を検討
- 起きたこと: 手法・個別課題の最適化に没頭し、本来解決すべき課題を見失った
- 結果: 怒られた、悔しかった (普段から「手段の目的化」を嫌っていたのに)
- 自己分析: 検証開始時点で「現状 / 課題 / どうなったら解決と言えるか」を定義していなかった

**共通パターン (重要)**
- 両シーンとも **「課題定義を飛ばして手段の議論に入っている」** こと自体は同じ
- 立場が違うだけ: シーン 1 はクライアント発、シーン 2 は自分発
- ユーザーの背景: 元アーキテクト / バックエンド API 開発 → AI Agent 検索 (データサイエンス領域) への越境中
- **得意領域から外れたタスクで、手元の手段に飛びつく傾向が出やすい**

### 2026-05-19 — round 4: 2 シーンは「同じ流れの違うフェーズ」

ユーザーによる再整理 (重要な転換):

> シーン 1 とシーン 2 は別の状況ではなく、**「課題を見つけ、整理し、解決の定義をして、手段を考える」という一連の流れの別フェーズを切り取っただけ**。

**フェーズ分解 (このスキルの中核モデル)**

1. **課題発見 (Discovery)**
   - 場面: 商談中、関係者会議、課題の形すら決まっていないとき
   - アウトプット: **課題が明確になっていること**
2. **課題整理 (Structuring)**
   - 場面: 課題が複数 / 重なって混在しているとき
   - アウトプット: 構造化された課題リスト or 1 つに絞られた中心課題
3. **解決定義 (Done Definition)**
   - 場面: 課題は明確だがゴール状態が言語化されていないとき (シーン 2 の本来あるべき着手前)
   - アウトプット: **どうなったら解決と言えるかの定義**
4. **手段検討 (Solution)**
   - 場面: Done が定義された後にようやく入る領域
   - アウトプット: 手段の候補と比較

**逆流のルール (シーン 2 から派生)**
- 4 フェーズは一方向ではない。手段検討中に「Done が決まってなかった」と気付いたら、フェーズ 3 へ戻る必要がある
- 課題が手段ベースで語られているなら、フェーズ 1 へ戻る
- **「現在地のフェーズ」と「戻りの検知」がスキルの中核責務になる可能性が高い**

### 2026-05-19 — round 5: 第一手の言葉 / 3 つの使い方

ユーザーから出た、スキルに **返してほしい言葉**:

**シーン 1 (商談中) で投げてほしい問い**
- 誰を対象に
- 何を叶えるため
- どうやったら達成と言えるか
- RAG じゃなくてもいいのか (= 手段への問い直し)

**シーン 2 (検証着手前) で投げてほしい問い**
- どんな成果が出たらゴールか
- 何を解消したら良いか
- 制約は何か

**「ハッとする」短いフレーズ (キラーフレーズ)**
- 「何を達成したらゴール?」
- 「なんのためにやるの?」

**追加で出た要件 (使い方の広がり)**
- 議事録などからこれらの情報を整理する処理
- 会議の中で「目的が見失われてた」などの **振り返り** 機能

→ **3 つの使い方** が見える:
- (a) リアルタイム介入: 会話中にハッとするフレーズを返す
- (b) 事後整理: 議事録などをインプットに、4 フェーズ情報を抽出 / 構造化
- (c) 振り返り: 会議や作業を振り返って、目的の喪失や手段先行を診断

### 2026-05-19 — round 6: 最終アウトプットは 2 種類 (md / HTML)

ユーザーから出た決定:

**自分用: Markdown ファイル**
- 目的: 自分自身の思考整理 (考え抜くためのフォーマット)
- 内容ラベリング: **「課題 / 派生した手段 / コア / 外枠」** が判別できる構造
- 粒度: 100〜1000 行
- 構成: **4 フェーズそれぞれで章立て**
- 分割ルール: **課題が複数混在していたら、それぞれ別ファイルとして出力**

**クライアント向け: HTML ファイル**
- 目的: クライアントへの提案
- 内容: 「何が課題か / 何が解決できたらいいか / どんな情報が足りていないから何が決められないか / 提案」
- 粒度: 3〜10 ページ
- 構成 (ピラミッドストラクチャ的 / Minto 流): **結論 → 根拠 → 提案 → 何をしてほしいか**

**新出概念**
- **不足情報の可視化** (「何が決められないか」を明示するクライアント向け機能)

**関連公共知識 (追加候補)**
- Minto Pyramid Principle — クライアント向け資料の構成
- SCQA (Situation/Complication/Question/Answer) — 結論先行の組み立て
- Signal vs Noise — 注意配分の分離

### 2026-05-19 — round 7: 「ラベル」ではなく「重要 / 付随」の二項分離

round 6 の表現を訂正:

- 「課題 / 派生した手段 / コア / 外枠」を **4 種のラベル** と読み込んだが、それは早合点
- ユーザーの本意: プロジェクトの現状を管理するにあたって、**重要視するもの** と、**そんなに重要ではないもの (手段など、よく注意を持っていかれるもの)** を **分離** できればよい、という二項
- 「コア / 外枠」「課題 / 派生手段」は具体例として挙げたペアで、本質は「中心 vs 周辺」「注目 vs 脇に置く」の 1 軸

**md フォーマットへの含意**
- 4 フェーズ章立ての中で、各章を「重要 (注目セクション)」と「付随 (脇に置くセクション)」に視覚的に分ければよい
- 手段の話は基本「付随」側に書き出す (引きずられないため)
- 「これは今は脇に置く」を明示する場が md に常にある状態にする

### 2026-05-19 — round 8: 差別化・自己進化・想定ユーザーの確定

ユーザーから出た決定:

**差別化の核 (= スキルの対話スタイル)**
1. **自己進化する** — 使うほど判断軸が研ぎ澄まされる
2. **全肯定しない** — むしろ批判的視点で **ビシバシ突っ込む**
3. **選択式の提案を出さない** — 視野を狭めるため。**常にオープンな質問** にする

**自己進化スコープ (進化対象)**
- どんな成果物を出すか (出力フォーマット)
- 何が課題で何が手段なのかの **判断軸**

**想定ユーザー**
- ユーザー本人のみ。**それ以外は考えない**。汎用化しない

**ディレクトリ構成**
- 任せる (`self-evolving-skill-creator` の scaffolder に従う)

## 場面 (When)

### 想定トリガー文 (実際のユーザー発話例)

1. 「思考を整理したい」
2. 「考えをまとめたい」
3. 「一旦落ち着きたい」 ← 個別最適に没頭して焦っている自己制御サイン (シーン 2 型)
4. 「目的ってなんだっけ?」 ← 目的喪失の自覚 (F1 / F3 への戻りサイン)
5. 「一旦まとめよう」

特徴: 「整理」「まとめ」系だけでなく、「**落ち着きたい**」「**目的ってなんだっけ**」のような **メタ自己制御発話** もトリガーとして拾う必要がある。これは LLM の標準的な「整理して」型ヘルプより一段深い起動条件。

### 起動タイミング

(round 4 — 4 フェーズモデルで再整理)

スキルは「課題発見 → 課題整理 → 解決定義 → 手段検討」のどのフェーズにいても呼べる。具体的な起動タイミング:

- **F1 課題発見**: クライアント / 関係者と話していて、課題の形すら決まっていないとき
- **F1 へ戻り**: クライアントが「○○を使って○○を作りたい」と **手段ベースで要望** してきたとき
- **F2 課題整理**: 課題候補が複数あって絞れないとき
- **F3 解決定義**: 課題は明確だが、検証 / 実装に着手する前に Done 基準を言語化したいとき
- **F3 へ戻り**: 手段検討 / 個別最適化に没頭していて「Done を決めてなかった」と気付いたとき
- **F4 手段検討**: Done が定義された後で、手段候補を比較したいとき
- **資料化**: 上記いずれかの整理結果を、クライアントに渡せる形にしたいとき

## 解決する課題 (Problem)

(仮置き)

- **現状**: 「課題 / 解決状態 / 手段」の 3 層が混ざったまま会話と思考が進んでしまうため、
  - 商談では的外れな提案や、合意点の見えない議論になりがち
  - 自分の作業では目的を見失って手戻りが起きる
  - クライアントへの説明も論点が散らかる
- **痛みの種類**: 認知負荷 + 手戻り + 目的の喪失 + 提案の説得力低下
- **影響範囲**: 商談の質、参画後の生産性、クライアントへのアウトプットの説得力

## 解決策 (Solution)

(round 4 で骨格を初版化、ステップの詳細は次サイクル以降)

### 中核モデル: 4 フェーズの進行ガイド

```
[F1 課題発見] → [F2 課題整理] → [F3 解決定義] → [F4 手段検討]
        ↑              ↑              ↑              │
        └──────────────┴──────────────┴──────────────┘
                       逆流 (検知して戻す)
```

### スキルの 3 つの使い方

| モード | 使う瞬間 | スキルの動き |
|---|---|---|
| (a) リアルタイム介入 | 会話 / 思考の最中に呼ぶ | フェーズに対応する短い「ハッとフレーズ」or 問いを返す |
| (b) 事後整理 | 議事録・作業メモを渡す | 4 フェーズの情報を抽出して構造化、不足項目を可視化 |
| (c) 振り返り | 会議や作業を終えて振り返る | 「目的を見失っていなかったか」「手段に飛びついていなかったか」を診断 |

### スキルの責務 (仮)

1. **現在地の宣言** — 起動時に「今どのフェーズか」をユーザーに確認 or 推定
2. **フェーズ違反の検知** — 例: F1 なのに手段が語られていたら指摘して F1 に留まらせる
3. **フェーズごとの問いかけ** — 各フェーズで定型的に聞くべき問いを持つ
4. **アウトプット生成 (自分用 md)** — 4 フェーズ章立て、各章で「重要 / 付随」を分離 (手段は基本「付随」側)、複数課題なら分割
5. **資料化 (クライアント向け HTML)** — 結論 → 根拠 → 提案 → 依頼 の構成、3〜10 ページ
6. **議事録の解析** — 自然言語の議事録から 4 フェーズ情報を抽出
7. **振り返り診断** — 会議 / 作業のログを後追いで分析し、目的喪失や手段先行を指摘
8. **不足情報の可視化** — 「何が決められないか」と「それを決めるのに必要な情報」をクライアントに明示

| F1 完了 | 明確化された課題ステートメント |
| F2 完了 | 構造化された課題リスト or 中心課題 (コア / 外枠の区別あり) |

| (b) 事後整理 | 4 フェーズ別の抽出表 + 不足項目リスト |
| (c) 振り返り | 目的喪失 / 手段先行の有無の診断 |

## 想定ユーザー (Who)

- **主**: ユーザー本人 (フリーランスエンジニア、元アーキテクト / バックエンド API 開発、現在 AI Agent 検索サービス開発で DS 領域に越境中)
- **副**: なし。汎用化しない。本人専用に振り切る

## 既存ツール・スキルとの差別化 (Differentiation)

| 代替 | 限界 | このスキルが上回る点 |
|---|---|---|
| 素の ChatGPT / Claude に「課題を整理して」 | 全肯定的に応じてしまう / ユーザーの言葉をなぞる / 選択式提案で視野を狭める | **批判的に突っ込む** / **オープン質問のみ** |
| Notion テンプレ / マインドマップ | 静的。ユーザーが書く分しか整理されない。判断軸も変わらない | **対話で深掘りする** / **使うほど判断軸が進化する** |
| 既存の課題整理フレームワーク (Issue tree など) | 一般論に留まる。本人特化しない | **ユーザー本人専用** に判断軸を磨き続ける |
| self-evolving-skill-creator のみ | メタスキルで、思考整理のドメイン知識を持たない | 4 フェーズ × 重要/付随 × 批判的対話の **ドメイン特化** |

**1 行サマリー**: 「全肯定せず、選択肢で誘導せず、本人専用に判断軸が進化する、4 フェーズ思考整理スキル」

## 成功条件 (Success Criteria)

(round 4 で初期化、round 8 で対話スタイル系を追加)

- [ ] スキルが起動された時点で「今どのフェーズか」が 1 分以内に確定する
- [ ] F1 にいる間、手段の話が出たらスキルが必ず指摘して F1 に留める
- [ ] F3 を飛ばしたまま F4 に進もうとしたら、スキルが Done 不在を検知して戻す
- [ ] シーン 1 を再現したとき、クライアントに対して「目的を遡る問い」が 1 つ以上ユーザーから出せる
- [ ] シーン 2 を再現したとき、検証着手前に Done 基準が文書化されている状態になる
- [ ] 各フェーズの記録が、Markdown (自分用) + HTML (クライアント用) として出力される
- [ ] スキルからの問いは **常にオープン形式** (「A or B?」型が混ざっていない)
- [ ] スキルから「それで本当にいいのか」「目的とずれていないか」型の **批判的フィードバック** が最低 1 回出る
- [ ] 自己進化レビュー後、判断軸の改善が `references/judgment_axes.md` に反映されている

## 失敗モード予測 (Failure Modes)

(round 3 で初期化、round 8 で対話スタイル系を追加)

| 失敗 | 起きる条件 | 想定回避策 |
|---|---|---|
| クライアントが手段ベースで要望し、それに手段で返してしまう | 商談初期、相手の語彙が「○○を使って○○を作りたい」 | スキル側が「その手段で解決したい現状の課題は何か」を遡る問いを返す |
| 自分が個別手段の最適化に没頭する | 得意外領域 / 焦りがある状態 | 着手前に「現状 / 課題 / Done 基準」を必ず宣言してから手段検討に入らせる |
| 検証開始時に解決基準を決めていない | 新タスクが降ってきた直後 | Done 基準が空のまま手段層に進めないようスキルがブロックする |
| 整理結果が自分の頭の中だけに留まり、クライアントに渡せない | 出力が会話のみ | Markdown など渡せる成果物を必ず生成する |
| スキルが全肯定して気付きを与えない | LLM の標準挙動のまま | SKILL.md に「全肯定禁止 / 批判的に突っ込む」を不変ルールとして明記 |
| 選択肢を提示して視野を狭める | 「A / B / C のどれですか」型の質問 | SKILL.md に「選択肢提示禁止 / オープン質問のみ」を不変ルールとして明記 |
| 自己進化で対話スタイルが「優しい全肯定」に劣化する | ユーザーが直近で否定的反応を示すと迎合してしまう | 対話スタイル 3 原則は進化対象から除外 (固定) |
