# contract-review-assistant

> フリーランスエンジニアが、業務委託契約・SES 契約の締結・更新前に、契約書の要注意条項を
> 見落とさず洗い出し、先方への確認事項を整理するためのスキル。
> **法的助言ではなく「確認観点の整理」**であり、最終判断は本人または弁護士に委ねる。

---

## 場面 (When)

- クライアントから業務委託契約書（PDF / Word / テキスト）が送られてきて、押印前にリスクを確認したいとき
- SES 案件の基本契約書・個別契約書のドラフトを受け取り、単価・精算・解約条件を確認したいとき
- 既存契約の更新時に、前回から変わった条項がないか・見落としていた条項がないかを確認したいとき
- 契約交渉の前に「先方に何を質問・修正依頼すべきか」の確認事項リストを作りたいとき
- フリーランス保護新法（特定受託事業者取引適正化法）の観点で、取引条件の明示や支払期日が守られているか確認したいとき

### 想定トリガー文 (実際のユーザー発話例)

1. 「この契約書をレビューして」
2. 「業務委託契約書の要注意点を洗い出して」
3. 「この契約書、サインして大丈夫か確認観点を整理して」
4. 「契約書のリスクと先方への確認事項をまとめて」
5. 「このSES契約書、支払サイトと解約条件をチェックして」
6. 「フリーランス新法の観点でこの契約に問題ないか見て」
7. 「前の契約書と比べてどこが変わったか、注意点を教えて」

---

## 解決する課題 (Problem)

- 現状: 契約書を自力で読むが、法律の専門知識がないため「どの条項が危ないか」の判断軸がなく、
  知財の帰属・損害賠償の上限・中途解約条件などの頻出リスク条項を読み飛ばしてしまう。
  弁護士レビューはコスト・時間の面で毎回は頼めない。
- 痛みの種類: 見落とし（不利な条項に気づかず署名）/ 認知負荷（毎回ゼロから全文を精読）/
  手戻り（署名後に不利な条件が発覚して交渉し直す）
- 影響範囲: 個人事業の収益・法的リスクに直結（報酬未払い・損害賠償・競業避止による機会損失）

---

## 解決策 (Solution)

1. 契約書（PDF またはテキスト）を受け取り、全文を読み込む
2. 契約類型を判定する（準委任 / 請負 / SES（準委任＋常駐）/ 混合。判定根拠となる条文を引用）
3. `references/checklist.md` のチェックリストと条項を照合する
   （各観点について: 該当条項の**原文引用＋条番号**、リスク評価、確認文例）
4. チェックリストにあるのに契約書に**記載が見当たらない重要条項**を列挙する
5. 免責文を添えたレビューレポート（Markdown）を出力する

### 入力 (Inputs)

| 名前 | 型 | 必須 | 出どころ | 例 |
|---|---|---|---|---|
| contract_file | file path (PDF / .md / .txt / .docx→テキスト化) | yes | ユーザー発話・添付 | `~/Downloads/gyomu-itaku-keiyaku.pdf` |
| contract_type_hint | string (準委任 / 請負 / SES / 不明) | no | ユーザー発話 | 「SES の基本契約」 |
| focus_points | string[] (重点確認したい観点) | no | ユーザー発話 | 「支払サイトと解約条件を重点的に」 |
| previous_contract | file path (比較用の旧契約書) | no | ユーザー発話 | 「前回の契約書はこれ」 |

### 出力 (Outputs)

| 名前 | 形式 | どこに置く | 例 |
|---|---|---|---|
| review_report | markdown | チャット返信（保存希望時は契約書と同じディレクトリに `<契約書名>-review.md`） | 要点サマリ＋要注意条項表＋欠落条項＋免責 |
| 要点サマリ | markdown 節 | レポート冒頭 | 契約類型・期間・報酬・支払サイトの 5 行要約 |
| 要注意条項表 | markdown 表 | レポート本文 | 原文引用 / 条番号 / リスク / 確認文例 |
| 記載が見当たらない重要条項 | markdown リスト | レポート本文 | 「損害賠償の上限に関する定めが見当たらない」等 |
| 免責 | markdown 節 | レポート末尾（毎回必須） | 法的助言ではない旨・弁護士相談の推奨 |

## 想定ユーザー (Who)

- 主: フリーランスエンジニア本人（業務委託契約・SES 契約を「受託者」の立場で結ぶ）
- 副: 同様の立場の個人事業主（デザイナー・ライター等）**(ASSUMPTION: 副ユーザーは当面想定せず、チェックリストはエンジニア案件に最適化する)**

持っている知識: 開発業務・案件内容そのものは深く理解している。
持っていない知識: 契約法務の判断軸（どの文言がどうリスクになるか、法定の保護がどこまで及ぶか）。
→ 説明はリスクの「なぜ危ないか」を素人向けに 1〜2 文で補足する粒度にする。

## 既存ツール・スキルとの差別化 (Differentiation)

| 代替 | 限界 | このスキルが上回る点 |
|---|---|---|
| 自力で全文精読 | 判断軸がなく見落とす。毎回時間がかかる | フリーランス頻出リスクのチェックリストで網羅的・再現的に照合 |
| 弁護士レビュー | 費用・納期がかかり毎回は頼めない | 無料・即時の一次スクリーニング。弁護士に相談すべき点の絞り込みにも使える |
| 汎用 AI チャットに丸投げ | 観点が毎回ぶれる。免責・機密の扱いが不定 | 観点を references に固定して再現性を担保。機密を外部送信しない運用を明文化 |
| `deep-research` / `edith` 系調査スキル | 一般論の調査であり、手元の契約書の条文との照合はしない | 条文の原文引用＋条番号で「この契約書のどこが」を特定する |

## 成功条件 (Success Criteria)

- [ ] チェックリストの全観点（9 分類以上）について「該当条項の引用」または「記載なし」のいずれかが漏れなくレポートに出力される
- [ ] 要注意条項表の各行に、契約書の**原文引用と条番号（所在）**が付いており、ユーザーが原本で 30 秒以内に該当箇所を確認できる
- [ ] レポート末尾に免責（法的助言ではない旨・最終判断は本人/弁護士）が毎回含まれる
- [ ] 各リスク指摘に「先方にそのまま送れる確認文例」が付いており、ユーザーが交渉メールを追加質問なしで書き始められる

## 失敗モード予測 (Failure Modes)

| 失敗 | 起きる条件 | 回避策 |
|---|---|---|
| 法的助言と誤解される | 出力が断定調（「違法です」「拒否すべき」）になる | 免責を description・SKILL.md 冒頭・レポート末尾の 3 箇所に固定。語彙を「要確認」「〜の可能性」「弁護士への相談を推奨」に統制する |
| 条項の見落とし | 契約書を通読だけしてチェックリストを使わない / 別紙・個別契約に条件が分散している | references のチェックリストを 1 観点ずつ照合し、「該当あり / 記載なし」を全観点で必ず出力。別紙・個別契約への委任文言を検出したら追加提出を促す |
| 原文の誤読・引用の捏造 | PDF の OCR 崩れ・レイアウト崩れ、長文契約の要約時に条文を記憶で書く | 引用は必ず原文からコピーし条番号を併記。読み取りが不鮮明な箇所は「判読不能」と明示し推測で補完しない |
| 契約類型の誤判定 | タイトルが「業務委託」でも実態が請負/準委任いずれか不明瞭 | 判定根拠の条文（完成義務・検収・報酬の定め方）を引用して示し、不明瞭な場合は「類型が曖昧」自体をリスクとして報告 |
| 機密の漏洩 | 契約書の内容を WebSearch 等で外部に送る / 取引先実名入りレポートを共有物に置く | 契約書本文・取引先名を外部サービスへ送信しない。レポート保存は契約書と同じローカルディレクトリのみ。実名を残す場合は注意喚起する |
| 法令情報の陳腐化 | フリーランス新法等の運用・改正が進む | 法令に関する記述は「レビュー時点の一般的な理解」と明示し、references のチェックリストに最終更新日を記録して更新可能にする |

## 必要なリソース (Resources)

- 外部 API / CLI: 不要（PDF は Read ツールで読む。**外部送信しないことが要件**）
- 権限: ローカルの契約書ファイル読み取りのみ
- 参照すべき公開知識: 民法（準委任 643・656 条 / 請負 632 条 / 契約不適合責任）、
  フリーランス保護新法（特定受託事業者に係る取引の適正化等に関する法律、2024-11 施行）、
  下請法、著作権法（27 条・28 条の特掲、著作者人格権）
- ローカル前提: 契約書ファイルがローカルにあること（PDF / テキスト）

## 実装メモ (Implementation Notes)

- ディレクトリ構成:
  - `.claude/skills/ultron-contract-review-assistant/SKILL.md` — フロー・出力フォーマット・免責
  - `.claude/skills/ultron-contract-review-assistant/references/checklist.md` — 頻出要注意条項チェックリスト（本体から分離し、単独で更新可能に）
- scripts は作らない: 照合・引用・リスク整理はすべて自然言語処理であり、決定論的に切り出せる計算がない
- persona `ultron`（事務・金融・資産系）に従う: 出典（＝条文の原文引用）の明示、事実と解釈の区別、免責の必須化
- 判定基準（チェックリスト）を明文化して再現可能にする（persona の「判定基準の明文化」原則の適用）
- 台帳は持たない: 契約書は機密のため、リポジトリ内にレビュー履歴を蓄積しない **(ASSUMPTION: レビュー済み契約の台帳管理は不要。必要になったらユーザー指示で設計し直す)**
