# ideas

スキルを作る前の「場面 × 課題 × 解決策」を 1 アイデア 1 ドキュメントで蓄積する場所です。

## なぜこのディレクトリがあるか

スキルは「何ができるか」より「**どの場面で、何を解決するか**」が曖昧なまま作ると、自己進化ループに頼っても実用に至りません。
ここでアイデアを言語化してから [TEMPLATE.md](TEMPLATE.md) に沿って書き出すことで、初版から実用可能な粒度に落とし込みます。

## 運用ルール

1. 1 アイデア = 1 ファイル。ファイル名は `<kebab-case-skill-name>.md`。
2. 必ず [TEMPLATE.md](TEMPLATE.md) を複製して埋める。空欄のままの項目は `TBD` と書き、後で詰める。
3. アイデアが「実装着手 OK」のラインに達したら、frontmatter の `status` を `ready` にしてから [self-evolving-skill-creator/](../self-evolving-skill-creator/) の scaffolder に渡す。
4. 没アイデアは削除せず `status: dropped` にして、判断根拠を `## 没にした理由` セクションに追記する。
5. 似たアイデアが既にあるか必ず確認 (`ls ideas/`) してから新規作成する。重複なら統合 or 派生として書く。
6. まだ詰めていない「作るか未定」の候補は、1 ファイルに起こさず [`skill-candidates.md`](skill-candidates.md) の一覧表に概要だけ足す。温めることにしたら TEMPLATE.md に展開する。
7. **既存スキル**への不満・改善点は新規アイデアではなく [`skill-feedback.md`](skill-feedback.md)（中央インボックス）にためる。レビューで「守備範囲外＝新スキルが必要」と判明したものだけ skill-candidates.md へ転記する。

## status の運用

| status | 意味 |
|---|---|
| `draft` | 思いつき。場面・課題が言語化途中。 |
| `refining` | 主要セクションは埋まり、不足項目を詰めている段階。 |
| `ready` | 実装着手 OK。scaffolder に流せる粒度。 |
| `building` | 実装中。対応スキルディレクトリへのリンクを `## 実装リンク` に記載。 |
| `shipped` | スキルが公開された。リンク必須。 |
| `dropped` | 没。理由必須。 |

## 「最初から実用可能」を担保するチェックリスト

書いたアイデアを `ready` にする前に、以下に Yes と答えられるか確認する。

- [ ] 「**いつ呼ばれるか**」がトリガー文 (5 例以上) で具体化されている
- [ ] 「**何を入力に、何を出力するか**」が型レベルで書かれている
- [ ] 既存スキル / 既存ツールで代替できない理由が 1 行で説明できる
- [ ] 失敗モードを 3 つ以上想定して、その回避策が書かれている
- [ ] 成功条件が「観測可能 (出力ファイル、ログ、ユーザー反応)」になっている

これらが揃わないアイデアは `refining` のままで OK。無理に `ready` にしない。
