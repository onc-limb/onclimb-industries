# high-dividend-stock-screener (日本の高配当株 調査・選出スキル)

> 個人投資家が「次に仕込む高配当株の候補が欲しい」とき、インターネット上の公開情報から
> **配当利回り 4% 以上**の日本株を拾い、**増配傾向・低配当性向・営業黒字継続**などの
> 健全性条件で篩い、**おすすめ銘柄リスト**を作るスキル。一度に全銘柄は調べきれないため、
> **調べた会社を法人番号で記録**して回を分けて積み増し、最後に **Claude のレビュー**を添える。

---

## 0. このアイデアの背景 (なぜ必要か)

高配当株投資では「利回りが高い」だけでは罠が多い。利回りが高く見える銘柄には
**減配リスク(タコ足配当)・業績悪化・一過性の特別配当**が紛れる。本当に欲しいのは
「**無理なく配当を出し続け、むしろ増やしてきた会社**」であり、その判定には
利回りだけでなく**配当推移・配当性向・利益の継続性**を1社ずつ確認する必要がある。

この確認は手作業だと1社あたり数分かかり、東証の上場銘柄は約4000社。全部を一度に
調べるのは非現実的。だから本スキルは「**スクリーニングで母集団を絞り → 1社ずつ健全性を
検証 → 調べた会社は法人番号で記録して二度調べない**」という、回数を分けて積み増せる
インクリメンタルな運用を前提に設計する(worklog / knowledge-base が digest を
積み増すのと同じ思想)。

---

## 1. 場面 (When)

ユーザーが**意図的に起動**する(自動起動しない)。

- 「**高配当株の候補を探して**」「**おすすめの高配当株をリストアップして**」と言ったとき
- 「**利回り4%以上で増配傾向の日本株を調べて**」と条件付きで頼んだとき
- 既存リストに対し「**続き(まだ調べてない会社)を調べて追加して**」と継続を頼んだとき
- 「**このリストの銘柄、健全性をもう一回チェックして**」と再検証を頼んだとき
- 作ったリストに「**Claude のレビューを付けて**」と講評を求めたとき

### 想定トリガー文 (実際のユーザー発話例)

1. 「高配当株のおすすめをリストにして」
2. 「配当利回り4%以上の日本株を screening して」
3. 「増配が続いてて配当性向50%未満の銘柄を探して」
4. 「前回の続きから高配当株を調べて」
5. 「この候補リストにClaudeの評価コメント入れて」
6. 「減配リスクの低い高配当株を選んで」
7. 「日本株の高配当ポートフォリオ候補を出して」

> 起動の明示性: description は「**日本の高配当株を screening して候補リストを作る**」に
> 限定し、worklog / 一般的な投資相談と誤起動しないようにする。
> **投資助言ではなく情報整理**である旨を毎回明示する(§9 免責)。

## 2. 解決する課題 (Problem)

- **現状**:
  - 高配当株を探すには複数サイト(利回りランキング / 業績推移 / 配当推移)を横断し、
    1社ずつ目視で「増配か」「配当性向は健全か」「赤字はないか」を確認する。
  - 上場約4000社を一度に裁けないので、どこまで調べたかを覚えておらず**同じ会社を二度調べる**。
  - 利回りだけで飛びつくと**減配・タコ足配当の地雷**を踏む。
  - 候補を並べても「**で、結局どれが良いの / 何に注意すべきか**」の総括が無い。
- **痛みの種類**: 時間(サイト横断・1社ずつ確認) / 認知負荷(調査済み管理) / 見落とし(減配リスク) / 判断(総括の欠如)
- **影響範囲**: 個人の資産運用判断 → 誤れば実損。ゆえに**条件の透明性**と**免責**が必須。

---

## 3. 解決策 (Solution)

スクリーニング(母集団の絞り込み)と健全性検証(1社ずつ)の2段で、調査済みは法人番号で記録する。

1. **入力を受け取る**: 利回り閾値(既定4%)、除外区分(既定でREIT/投資法人を除く)、
   今回の調査件数(既定 例:20社)、継続か新規か。
2. **スクリーニング(母集団取得)**: Yahoo!ファイナンス等の高配当利回りランキングから
   利回り≧閾値の銘柄(証券コード・社名・利回り)を取得。**REIT・投資法人・インフラファンド等は
   この時点で除外**(名称・市場区分・銘柄種別で判定)。
3. **調査済みの除外**: ローカルの調査済み台帳(`registry/screened.jsonl`)を読み、
   **既に法人番号が記録済みの会社をスキップ**。未調査の上位 N 社を今回の対象にする。
4. **1社ずつ健全性検証**: IR BANK 等から各社の業績・配当データを取得し、下記条件で判定。
   - **配当が右肩上がり**(直近数年が概ね増配 or 維持、減配年が無い/少ない)
   - **配当性向 < 50%**(無理のない範囲で配当している)
   - **営業利益に赤字が無い**(直近数年の営業利益がすべて黒字)
   - (任意の補助指標は §11 に列挙。コア条件は上記3つ)
5. **法人番号の記録**: 調べた会社は**合否に関わらず**証券コードから法人番号を解決し、
   判定結果とともに台帳へ追記(次回以降スキップするため)。
6. **おすすめリスト生成**: 全条件を満たした会社を `lists/<date>.md` に出力(下表のカラム)。
7. **Claude レビュー**: 生成リストに対し、Claude が**講評・注意点・分散の偏り・要再確認点**を
   コメントとして付す(§4 末)。
8. **完了提示**: 「今回 N 社調査 / 合格 M 社 / 台帳 累計 K 社」を要約し、続きの回し方を案内。

### 入力 (Inputs)

| 名前 | 型 | 必須 | 出どころ | 例 |
|---|---|---|---|---|
| yield_min | number(%) | no(既定 4.0) | ユーザー発話 | 4.0 |
| batch_size | number | no(既定 20) | ユーザー発話 | 20 |
| mode | "new" \| "continue" | no(既定 continue) | ユーザー発話 | "continue" |
| exclude | string[] | no(既定 REIT/投資法人/インフラF) | 既定 + 発話 | ["REIT"] |
| extra_filters | object | no | ユーザー発話 | {payout_max: 50} |

### 出力 (Outputs)

| 名前 | 形式 | どこに置く | 例 |
|---|---|---|---|
| おすすめ銘柄リスト | markdown(表) | `<data>/lists/<date>.md` | 合格銘柄 + 指標 |
| 調査済み台帳 | JSONL(1社1行) | `<data>/registry/screened.jsonl` | 法人番号付き |
| Claude レビュー | markdown | リスト末尾に追記 | 講評・注意点 |

#### おすすめリストの列(案)
`証券コード | 社名 | 法人番号 | 利回り% | 配当性向% | 営業利益(直近n期: 黒/赤) | 連続増配/維持年数 | 業種 | 判定根拠 | 取得元URL`

#### Claude レビューに含める観点(案)
- 業種・利回り帯の**偏り**(同一セクター集中、超高利回りに寄っていないか=減配リスク)
- **要再確認**な銘柄(配当性向が閾値ギリギリ、特別配当で利回りが膨らんでいる疑い 等)
- データの**鮮度・出所の注意**(取得日・四半期跨ぎ)
- **免責**(投資助言ではない/最終判断は自己責任/一次情報での再確認を推奨)

## 4. 想定ユーザー (Who)

- **主**: 高配当株での資産形成を考える個人投資家(= 本人)。証券口座を持ち、銘柄選定の
  入口を効率化したい。
- **副(読み手)**: なし(個人利用)。ただしリストを家族・友人と共有する可能性 → 免責を明記。

ユーザーが持っている知識: 利回り・配当性向・営業利益の意味。持っていない/省きたい作業:
4000社を横断して1社ずつ確認すること、どこまで調べたかの管理。

## 5. 既存ツール・スキルとの差別化 (Differentiation)

| 代替 | 限界 | このスキルが上回る点 |
|---|---|---|
| Yahoo!ファイナンスの利回りランキング | 利回り順に並ぶだけ。健全性(増配/性向/黒字)は別途目視 | 利回り+健全性をまとめて判定しリスト化 |
| 証券会社のスクリーナー | 条件設定は便利だが「調査済み管理」「総括レビュー」は無い | 法人番号台帳で二度調べない + Claudeの講評 |
| 手作業で1社ずつIR BANK確認 | 時間・認知負荷が大、抜け漏れ・重複調査 | 自動収集 + 台帳で重複排除 + 定型判定 |
| 汎用LLMに「高配当株教えて」 | 学習データが古く、利回り/配当性向が現在値でない・幻覚 | **公開サイトの現在値を取得**して判定(出所URL付き) |

## 6. 成功条件 (Success Criteria)

- [ ] スクリーニング結果から **REIT・投資法人・インフラファンドが除外**されている。
- [ ] 出力された各銘柄に **利回り・配当性向・営業利益(黒/赤)・増配傾向**の根拠が**出所URL付き**で並ぶ。
- [ ] 同じ会社を**二度調べない**(台帳の法人番号で重複排除が効いている)。
- [ ] 「続きを調べて」で**前回の続きから**未調査銘柄を処理できる(累計が増える)。
- [ ] リスト末尾に **Claude のレビュー(偏り・要再確認・免責)** が必ず付く。
- [ ] 出力に **取得日と免責**(投資助言ではない旨)が明記されている。

## 7. 失敗モード予測 (Failure Modes)

| 失敗 | 起きる条件 | 回避策 |
|---|---|---|
| 古い/幻覚の数値を出す | LLM 記憶から書く | 必ず**サイト取得値**を使い、各値に取得元URLと取得日を付す。取れない値は「未取得」明示 |
| REIT/投資法人が混入 | 名称・区分判定の漏れ | 名称サフィックス("投資法人"等)+ 市場区分 + 銘柄種別の複合判定。疑わしきは除外しログ |
| 同じ会社を再調査 | 台帳を読まない / コード違いで同一社 | 起動時に台帳必読。**法人番号**で名寄せ(証券コードでなく法人番号を一意キーに) |
| 配当性向の分母取り違え | EPS基準 vs 純利益基準の混在 | 出典の定義を固定(例: 1株配当/EPS)。算出元の数値もリストに残す |
| 特別配当で利回りが過大 | 記念配・特別配を普通配と混同 | 配当内訳が分かれば普通配ベースで再評価。怪しい銘柄は「要再確認」へ |
| スクレイピングがブロック/構造変化 | サイト改修・bot制限 | 取得手段を抽象化(WebFetch/ブラウザ自動化)。失敗時は件数を減らし、取れた分で続行+警告 |
| 「おすすめ=投資助言」と誤解 | 免責が無い | 毎回**免責**を出力。表現は「候補」「情報整理」に統一、断定的推奨を避ける |
| 増配判定が1期の上下でブレる | 単年比較 | 直近3〜5期の推移で「減配年の有無」を見る。記念配の段差は除外して評価 |

## 8. 必要なリソース (Resources)

- 外部データ源(公開情報):
  - 利回りスクリーニング: Yahoo!ファイナンス 高配当利回りランキング 等
  - 健全性データ: IR BANK(業績・配当推移・配当性向)等
  - 法人番号(本筋): **EDINET コードリスト(CSV)** で **証券コード→提出者法人番号** を一括変換。
    上場企業は有報提出義務があるためほぼ必ず載る。一度DLすればローカル突合で API 連打不要。
  - 法人番号(フォールバック): 国税庁 法人番号システム Web-API(無料のアプリケーションID登録要、
    CSV/XML/JSON)で**社名照合**。EDINET 未反映の例外時のみ使用(社名照合は表記揺れで曖昧)。
- 取得手段: `WebFetch` / `WebSearch`、またはブラウザ自動化(claude-in-chrome)。
  サイトの利用規約・robots を尊重し、**過度な連続アクセスをしない**(バッチ件数で制御)。
- ローカル前提: git リポジトリ内で実行。台帳・リストをファイルで永続化。
- 参照知識: 配当性向・タコ足配当・連続増配の基礎、東証の銘柄区分(REIT/インフラF/投資法人)。

## 9. 免責とスタンス (重要)

- 本スキルの出力は**公開情報の整理であって投資助言ではない**。最終判断は利用者の自己責任。
- 数値は**取得時点のもの**で将来を保証しない。発注前に**一次情報(IR・有報)で再確認**を推奨。
- 「おすすめ」は条件合致の意味であり、特定銘柄の売買を勧めるものではない。
- スクレイピング対象サイトの**利用規約・robots.txt を尊重**し、商用再配布等はしない。

## 10. 実装メモ (Implementation Notes)

- ディレクトリ構成(案):
  - `.claude/skills/high-dividend-stock-screener/`
    - `SKILL.md`
    - `bin/`(スクリーニング取得・健全性判定・法人番号解決・台帳更新の補助スクリプト)
    - `references/`(銘柄区分の判定ルール、配当性向の定義メモ、サイト構造メモ)
- データ置き場(案): repo 直下 `stock-data/`(`registry/screened.jsonl`, `lists/<date>.md`)。
  環境変数 `STOCK_DATA` で上書き可。worklog の data 解決方式を踏襲。
- 台帳スキーマ(JSONL 1行/社, 案):
  ```json
  {"corp_number":"<13桁>","ticker":"1234","name":"...","screened_at":"2026-06-27",
   "yield":4.3,"payout_ratio":38.0,"op_profit_all_positive":true,
   "dividend_trend":"increasing","passed":true,"sources":["<url>","<url>"]}
  ```
  一意キーは法人番号(13桁)。証券コード→法人番号は **EDINET コードリスト(CSV)の
  「証券コード」「提出者法人番号」カラムを突合して解決**(本筋)。EDINET で引けない例外のみ
  国税庁 Web-API で社名照合し、曖昧な場合は台帳に candidate を残して人手確認に回す。
  // ASSUMPTION: EDINET の証券コードは末尾0付き5桁(例 トヨタ 72030)で配布される慣習があり、
  //            Yahoo 側の4桁(7203)と突合する際は桁の正規化が要る(実ファイルで要確認)。
- 法人番号解決フロー:
  ```
  証券コード ──[EDINET コードリストCSVで突合]──► 法人番号(本筋・確実)
     └─ 引けない場合 ──[国税庁 Web-API で社名照合]──► 法人番号(フォールバック・曖昧時は人手確認)
  ```
- 判定ロジック(コア3条件)はスクリプト側で決定論的に評価し、LLM は
  「取得・名寄せ・レビュー講評」に専念(数値の捏造を構造的に防ぐ)。
- 取得手段はアダプタ化: まず `WebFetch`/`WebSearch`、構造が取れない場合は
  ブラウザ自動化にフォールバック。サイト構造は `references/` にメモして変化に追従。
- 1回の起動は `batch_size` 件で打ち切り、続きは次回(台帳でレジューム)。

## 11. 判定条件の詳細(コア + 補助)

- **コア条件(必須・ユーザー指定)**:
  1. 配当利回り ≧ 4%(スクリーニング段)
  2. 配当が右肩上がり(直近3〜5期に減配年が無い/維持以上)
  3. 配当性向 < 50%
  4. 営業利益が直近数期すべて黒字(赤字無し)
  5. REIT / 投資法人 / インフラファンドを除外
- **補助指標(任意・レビューや同点処理に活用、合否は変えない案)**:
  自己資本比率、営業CFの黒字継続、連続増配年数、EPS推移、配当方針(累進/DOE)の明文化有無。
  → ready 化の前に「補助指標を合否に入れるか/レビュー材料に留めるか」を確定する(§13 論点)。

## 13. 決定事項と残論点

### 決定済み (2026-06-27 / ユーザー指定)
- スクリーニング閾値: **配当利回り 4% 以上**。
- 除外: **REIT・投資法人**(+ インフラファンドも同区分として除外を提案)。
- 健全性コア条件: **配当右肩上がり / 配当性向 < 50% / 営業利益に赤字なし**。
- **調査済みは法人番号で記録**し、回を分けてインクリメンタルに積み増す。
- 生成したおすすめリストに **Claude のレビューを入れる**。
- 法人番号の解決: **EDINET コードリストで証券コード→法人番号を突合(本筋)**、
  国税庁 Web-API の社名照合はフォールバック。(2026-06-27 追記)

### ready にする前に残る論点
- 取得手段の第一候補(WebFetch か ブラウザ自動化か)を実データで検証して確定。
- EDINET コードリストの実ファイルで、証券コードの桁(5桁/末尾0)と法人番号カラム名を確認。
- 「配当が右肩上がり」の厳密定義(評価期間 何期 / 減配を何回まで許容 / 記念配の扱い)。
- 配当性向の算出基準(EPS基準 vs 純利益基準)をどの出典に固定するか。
- 補助指標を合否に入れるか、Claude レビューの材料に留めるか(§11)。
- 1回の `batch_size` 既定値(サイト負荷と1セッションのトークン量のバランス)。
