# todo-management

> ユーザー自身が、1 日のうち約 2 時間おきのチェックインで、その日の todo の進捗・追加・困りごとを整理し、翌日への申し送りまで残すための対話パートナースキル。

---

## 場面 (When)

ユーザーが 1 日の中で複数回 (始業 / 中間 / お昼 / 夕方 / 終業) に「いま何やってる、何が残ってる、詰まってないか」をスキルに棚卸しさせる場面。

- 始業時にその日の todo を立ち上げたいとき (前日繰越 + backlog からのピックを反映)
- 業務の節目で進捗を点検したいとき (約 2 時間おき)
- 想定外の依頼が来て todo に積みたいとき
- なんとなく手が止まっているが、原因をユーザー自身が言語化できていないとき
- 終業時に今日の作業報告と翌日申し送りを残したいとき

### 想定トリガー文 (実際のユーザー発話例) **(ASSUMPTION: 既存の仮案 6 文をそのまま正式採用し、SKILL.md の description に反映した。運用ログで別の言い回しが頻出したら自己進化で追加する)**

1. 「チェックインお願い」
2. 「進捗整理して」
3. 「今日の todo 作って」(始業時)
4. 「今日の作業報告まとめて」(終業時)
5. 「backlog から今日やるやつ拾って」
6. 「いまの状況棚卸し」

---

## 解決する課題 (Problem)

- 現状:
  - 頭の中で todo を持ち歩いていて、午後になると朝立てた計画が曖昧になる
  - 「進んでいない」状態に自分で気づかず、終業直前になって慌てる
  - 計画変更やゴール変更を memo していないので、後から「なぜこうなったか」が追えない
  - backlog (いつかやる) と今日やることが混ざる、または backlog 自体が存在しない
- 痛みの種類: 認知負荷 / 見落とし / 振り返り材料の喪失
- 影響範囲: 個人作業 (現時点では個人のジャーナル)

---

## 解決策 (Solution)

### チェックイン1回あたりの共通フロー (確定)

1. **進捗確認** — 既存の `todo.md` を読み、各タスクの状態をユーザーに問う
2. **追加 Todo の確認** — 新しく積むものがあるか聞く
3. **困りごと (ストッパー) の確認** — スキル側がステータス変化なし / 経過時間が計画見積超過などを検知した場合は直接ユーザーに問い、何もなければ「特になし」を確認
4. **次のアクションの提案** — 直近で取り組む順序や優先度を提案
5. **ファイル更新** — `todo.md` / `memo.md` を更新 (memo はゴール変更など todo を書き換えると消える追加情報の置き場)

### 始業時の追加動作 **(ASSUMPTION: 「スキルが選別してから提示 → ユーザーが確定」方式を採用。全件をそのまま見せる方式は backlog 肥大時に破綻するため)**

- `journal/YYYY-MM-DD/` ディレクトリと `todo.md` / `memo.md` を新規作成 (`scripts/init_today.py`)
- 前日 `report.md` の「翌日への申し送り」セクションと `backlog.md` を読み、**スキル側が「今日やる候補」を選別して理由付きで提示**する (申し送りは原則全件、backlog からは締切・関連性を根拠に数件)
- ユーザーが候補を確定・修正し、確定分だけを今日の `todo.md` に昇格させる (**最終決定権は常にユーザー**)
- その日の締切 / 既知の所要時間をユーザーから受け取り、所要時間が未指定のものはスキル側で見積もって提案 (見積ロジックは下記「見積の初期値」参照)

### 見積の初期値 **(ASSUMPTION: 「過去実績の中央値 → ユーザーに質問 → 30 分単位の粗い提案」の 3 段フォールバックを採用)**

1. 過去の `journal/*/report.md` に同種タスク (同じカテゴリ・類似の作業内容) の実績があれば、**その中央値**を初期見積として提案
2. 過去実績がなければ **ユーザーに所要時間を質問**する
3. ユーザーも答えられなければ、スキルが **30 分単位の粗い見積** (30m / 60m / 90m ...) を根拠付きで提案し、ユーザーの追認を得る

見積と実績の差は report.md に残り、自己進化ループの学習対象になる。

### 終業時の追加動作 **(ASSUMPTION: 未完了の振り分けは「スキルが提案 → ユーザーが確定」方式を採用。完全自動は誤廃棄リスクがあるため)**

- `report.md` の雛形を作成 (`scripts/finalize_day.py`) し、今日完了したもの / 未完了 / 見積と実績の差 / 翌日への申し送りを埋める
- 未完了タスクについて、**スキルが振り分け案 (翌日繰越 / backlog 戻し / 廃棄) を理由付きで提案**し、ユーザーが確定する
- 確定した振り分け結果を `report.md` の「翌日への申し送り」セクションに記録 (backlog 戻し分は `backlog.md` にも反映)

### 入力 (Inputs)

| 名前 | 型 | 必須 | 出どころ | 例 |
|---|---|---|---|---|
| 起動発話 | string | yes | ユーザー発話 | 「チェックインお願い」 |
| 今日の追加 todo | string[] | no | ユーザー発話 | 「PR #42 のレビュー追加」 |
| 締切 | datetime | no | ユーザー発話 | 「夕方まで」 |
| 進捗報告 | string | no | ユーザー発話 | 「タスク A は半分終わった」 |
| 困りごと | string | no | ユーザー発話 | 「テストが落ちる原因が分からない」 |

### 出力 (Outputs)

| 名前 | 形式 | どこに置く | 例 |
|---|---|---|---|
| 現状サマリ | markdown | チャット返信 | 進捗 / 残り / 推奨アクション |
| todo.md | markdown | `journal/YYYY-MM-DD/todo.md` | タスク一覧 |
| memo.md | markdown | `journal/YYYY-MM-DD/memo.md` | 補足情報の時系列追記 |
| report.md | markdown | `journal/YYYY-MM-DD/report.md` | 終業時のみ生成 |
| backlog.md | markdown | `journal/backlog.md` | カテゴリセクション (プロジェクト別など) で構成 |

ファイルの**内部スキーマ (列・項目) は運用しながら自己進化ループで詰める方針** — 初版は下記の最小構造から始める。

### 各ファイルの最小初期スキーマ **(ASSUMPTION: 初版スキーマとして以下を採用。列の昇格・削除は自己進化ループで行う)**

- `todo.md` — チェックボックス 1 行 1 タスク + 見積 / 実績 / 状態
  ```markdown
  # Todo — YYYY-MM-DD
  - [ ] タスク名 (見積: 30m / 実績: - / 状態: 未着手)
  ```
  状態は `未着手` / `進行中` / `完了` / `保留` の 4 値。完了時は `- [x]` に変えて実績を記入。
- `memo.md` — 時刻付きの時系列追記のみ (`scripts/append_memo.py`)
  ```markdown
  # Memo — YYYY-MM-DD
  - HH:MM 内容
  ```
- `report.md` — 固定 4 セクション: `## 完了` / `## 未完了` / `## 見積と実績の差分` / `## 翌日への申し送り`
- `backlog.md` — カテゴリ見出し (`## カテゴリ名`) + 箇条書き項目
  ```markdown
  # Backlog
  ## <カテゴリ名>
  - 項目 (任意で締切・メモ)
  ```

### ストッパー検知ルール **(ASSUMPTION: 検知条件は「前回チェックインから状態が変わっていないタスクがある」or「実績が見積の 1.5 倍を超過」の 2 つを採用。検知時は断定せず質問形式で確認する)**

- 条件 A: 前回チェックイン時点と比べて **状態が変わっていない未完了タスク** がある
- 条件 B: **実績が見積の 1.5 倍を超過** している (進行中タスクの経過時間含む)
- 検知したら「〜で詰まっていませんか?」のような **質問形式** で確認する。「詰まっている」と断定しない。閾値 1.5 倍と聞き方は自己進化ループの調整対象。

## 既存ツール・スキルとの差別化 (Differentiation)

| 代替 | 限界 | このスキルが上回る点 |
|---|---|---|
| Claude Code 標準の TaskCreate / TaskList | セッション内限定、永続化されない、日次の振り返り不能 | journal/ にファイルとして永続化、日をまたぐ運用 |
| 紙 / 単独の todo アプリ (Things, Todoist) | 進捗の促し・ストッパー検知をしない、振り返り資産にならない | 2 時間おきの対話による棚卸し、ストッパー能動検知 |
| Notion 等のデータベース型 | 構造化はできるが対話的な棚卸し相手にならない | チャットで進捗を引き出し、見積精度を学習 |
| 既存の `loop` スキル | 起動の仕掛けはあるが内容を持たない | 起動後のチェックインフローをスキル本体に内包 |

## 成功条件 (Success Criteria)

- [ ] 1 日を通して 4〜5 回のチェックインを継続できる (チェックイン体験が苦にならない)
- [ ] その日の `report.md` を読むだけで、何をして何が残ったか、なぜ残ったかが追跡できる
- [ ] 見積と実績の差が運用 N 週間後に縮小していることがログから確認できる **(N は要決定)**
- [ ] ストッパー検知の的中率がユーザーの体感で許容範囲 (誤検知で鬱陶しがられない)

---

## 失敗モード予測 (Failure Modes)

| 失敗 | 起きる条件 | 回避策 |
|---|---|---|
| ストッパー検知が過剰反応してユーザーに鬱陶しがられる | 経過時間の閾値が厳しすぎる / memo の文脈を読まずに判断 | 自己進化ループで聞き方を調整、検知時は断定でなく質問形式 |
| 見積が大きく外れて提案が無意味 | 運用ログが貯まる前の初期 | 初期は粗い経験則 + ユーザー追認、ログ蓄積で更新 **(初期値の根拠は要相談)** |
| ファイル更新の途中でセッションが切れて整合性が壊れる | チェックイン中に中断 | 各セクションを独立追記方式に、ロック不要 |
| チェックイン頻度が落ちて歯抜けの日が増える | 手動起動の心理障壁 | 起動時に「最後のチェックインから経過時間」を一目で出す |
| backlog が肥大化して読めなくなる | カテゴリが乱立 / 古いものが残り続ける | 終業 or 週次でユーザーに整理を促す (将来) |

## 残論点 (解決済み — 2026-07-02)

1. **起動トリガー文** — **決定: 既存の仮案 6 文をそのまま採用** し、SKILL.md の description に全文を織り込んだ。運用ログで別の言い回しが頻出したら自己進化で追加する。**(ASSUMPTION)**
2. **始業時の前日繰越 / backlog ピック** — **決定: スキルが選別してから提示する方式**。前日申し送りは原則全件、backlog からは締切・関連性を根拠に数件を理由付きで提示し、ユーザーが確定する。**(ASSUMPTION)**
3. **終業時の未完了振り分け** — **決定: スキルが振り分け案 (繰越 / backlog 戻し / 廃棄) を理由付きで提案し、ユーザーが確定する**。完全自動は誤廃棄リスクがあるため採らない。**(ASSUMPTION)**
4. **見積の初期値の根拠** — **決定: 3 段フォールバック**。(1) 同種タスクの過去実績があればその中央値 → (2) なければユーザーに質問 → (3) 答えられなければ 30 分単位の粗い提案 + ユーザー追認。**(ASSUMPTION)**
5. **`self-evolving-skill-creator` パイプライン採用の最終確認** — **決定: 採用**。`karen-self-evolving-skill-creator` 標準 (pipeline.config.json / logs/pipeline.jsonl / scripts/pipeline.py・evolve.py) にそのまま乗せ、auto_apply: true・しきい値 10 の既定値で運用。ログスキーマも規約標準を流用する。**(ASSUMPTION)**
6. **各ファイルの最小初期スキーマ** — **決定: 本文「各ファイルの最小初期スキーマ」の通り**。todo.md はチェックボックス + 見積 / 実績 / 状態 (4 値)、memo.md は時刻付き追記、report.md は完了 / 未完了 / 差分 / 申し送りの固定 4 セクション、backlog.md はカテゴリ見出し + 項目。**(ASSUMPTION)**
7. **ストッパー検知の閾値** — **決定: 「前回チェックインから状態が変わっていない未完了タスクがある」or「実績が見積の 1.5 倍を超過」の 2 条件**。検知時は質問形式で確認し断定しない。閾値と聞き方は自己進化の調整対象。**(ASSUMPTION)**
