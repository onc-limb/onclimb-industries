# codebase-onboarding

> SES 参画直後のエンジニアが、初見のコードベースを最短で把握するために、
> 「全体地図・主要フロー・読み始めポイント」の 3 部構成オンボーディング資料を
> `projects/<project>/onboarding/` に永続化するスキル。

---

## 場面 (When)

- 新しい案件のリポジトリを `projects/<project>/` に clone した直後、「まず全体像を掴みたい」と言ったとき
- ユーザーが「このコードベースのオンボーディング資料を作って」と依頼したとき
- 「どこから読み始めればいい？」「読む順番を教えて」と、読み方そのものを聞かれたとき
- 「ログイン処理の流れを追って」など、特定ユースケースのコードフロー追跡を求められたとき
- 参画から数日後、「前に作ったオンボーディング資料を最新化して」と更新を求められたとき
- チーム内の後続メンバーに渡す「読み始めガイド」を作りたいと言われたとき **(ASSUMPTION: SES では後続参画者への引き継ぎ資料需要があると推測)**

### 想定トリガー文 (実際のユーザー発話例)

1. 「このコードベースのオンボーディング資料を作って」
2. 「新しい案件に参画したので、このリポジトリの全体像を掴みたい」
3. 「このプロジェクト、どこから読み始めればいい？」
4. 「主要な処理の流れを追ってコードの地図にして」
5. 「読む順番つきでコードリーディングのガイドを作って」
6. 「参画初日。projects/foo のコードを把握したい」
7. 「前に作ったオンボーディング資料、最新のコードに合わせて更新して」

---

## 解決する課題 (Problem)

- 現状: 参画直後、README とディレクトリを手当たり次第に開き、断片的なメモが散逸する。
  数日かけて把握した内容も文書化されず、次案件・後続メンバーで同じ苦労を繰り返す。
  既存の `codebase-reader` は単発の構造レポートをチャットに返すのみで、
  「どの順で読むか」「主要ユースケースが実際にどのコードを通るか」までは踏み込まず、成果も揮発する。
- 痛みの種類: 時間（把握に数日）/ 認知負荷（全体像なしで詳細を読む）/ 手戻り（同じ探索の繰り返し）
- 影響範囲: 個人作業（参画立ち上がり速度）＋間接的にチーム共有物（引き継ぎ資料）

---

## 解決策 (Solution)

1. 対象プロジェクトを特定する（`projects/` 配下のディレクトリ名 or ユーザー指定パス）
2. 既存の `projects/<project>/onboarding/` を確認し、あれば更新モード（対象 commit との差分把握）で動く
3. 全体探索を `codebase-reader`（エージェント）に委譲し、構造化レポートを取得する
4. README・ルーティング・エントリポイントから主要ユースケース候補を 2〜5 本挙げ、ユーザーに確認する
5. 各ユースケースについてエントリポイント → 出口をファイル:シンボル単位で追跡する
6. テンプレートに沿って `01-map.md` / `02-flows.md` / `03-reading-guide.md` を生成・保存する
7. 読み始めポイント上位 3 件と生成パスをチャットで要約し、完了とする

### 入力 (Inputs)

| 名前 | 型 | 必須 | 出どころ | 例 |
|---|---|---|---|---|
| project_path | string (絶対パス) | yes | ユーザー発話 or `projects/` 配下の探索 | `projects/foo-api` |
| project_name | string | no (path から導出) | ディレクトリ名 | `foo-api` |
| focus_use_cases | string[] | no | ユーザー発話（指定なければ候補提示→確認） | `["ログイン", "注文確定"]` |
| mode | enum: `new` \| `update` | no (既存 onboarding/ の有無で判定) | ファイルシステム | `update` |

### 出力 (Outputs)

| 名前 | 形式 | どこに置く | 例 |
|---|---|---|---|
| 全体地図 | markdown | `projects/<project>/onboarding/01-map.md` | 技術スタック表・ディレクトリ役割・依存図 |
| 主要フロー | markdown | `projects/<project>/onboarding/02-flows.md` | ユースケース×ステップ表（ファイル:シンボル） |
| 読み始めガイド | markdown | `projects/<project>/onboarding/03-reading-guide.md` | 番号付き読む順番 5 件以上 |
| 完了サマリ | markdown | チャット返信 | 生成パス＋読み始めトップ 3 |

## 想定ユーザー (Who)

- 主: SES 参画直後のフリーランスエンジニア本人（= このリポジトリのユーザー）
- 副: 同案件に後から参画するメンバー **(ASSUMPTION: 資料を共有する場面があると推測)**

持っている知識: 一般的な言語・フレームワークの読み方。持っていない知識: 当該プロジェクト固有の構成・ドメイン用語・暗黙の規約。
→ 説明は「一般論は省き、このリポジトリ固有の事実＋出典パス」に絞る。

---

## 既存ツール・スキルとの差別化 (Differentiation)

| 代替 | 限界 | このスキルが上回る点 |
|---|---|---|
| `codebase-reader`（既存スキル/エージェント） | 単発の構造レポートをチャットに返すだけ。読む順番・ユースケースのフロー追跡・永続化なし | 読む順番の提案、エントリポイント→出口のフロー追跡、`projects/<project>/onboarding/` への永続化と差分更新。探索の下請けとして codebase-reader を再利用する（競合ではなく上位工程） |
| `/init`（CLAUDE.md 生成） | AI 向けの作業規約が主目的。人間の学習順序は扱わない | 人間（参画者）が読む順に最適化した 3 部構成ドキュメント |
| `tree` + README 手読み | 静的な構成しか見えず、処理の流れ・重要度が分からない | ユースケース単位で実際に通るコードを追跡し、重要ファイルを順位付け |
| `deep-research` | Web 情報向け。ローカルコード探索ではない | ローカルリポジトリの一次情報（コード）に基づく |

## 成功条件 (Success Criteria)

- [ ] `projects/<project>/onboarding/` に 3 ファイル（01-map / 02-flows / 03-reading-guide）が生成され、本文中で参照するファイルパスがすべて実在する（リンク切れ 0）
- [ ] 03-reading-guide の「読む順番」に、具体的ファイルパス＋理由付きの項目が 5 件以上、番号付きで並ぶ
- [ ] 02-flows で主要ユースケース 2 本以上が「エントリポイント → 出口」までファイル:シンボル単位で追跡されている
- [ ] 各ドキュメントの frontmatter に生成日と対象 commit hash が記録され、`update` モードで再実行すると差分が反映される
- [ ] 対象プロジェクトの作業ツリーを一切変更しない（対象リポジトリの `git status` が実行前後で不変）

## 失敗モード予測 (Failure Modes)

| 失敗 | 起きる条件 | 回避策 |
|---|---|---|
| 巨大リポジトリで探索がコンテキストを食い潰す | monorepo・数千ファイル規模 | 全体探索は codebase-reader エージェントに委譲して要約のみ受領。フロー追跡はユースケース 2〜5 本に絞り、パッケージ単位で分割 |
| DI・コード生成・リフレクションで静的追跡が切れ、フローが推測混じりになる | Spring / NestJS / codegen 系 **(ASSUMPTION: 参画案件でこの種の FW が多いと推測)** | 追跡が切れた箇所は文書内に **(ASSUMPTION: ...)** と明記。エントリポイントは実行設定（package.json / Dockerfile / ルーティング定義）で裏取りする |
| 保存先を誤り、対象プロジェクトの管理ファイルや当リポジトリの git 管理下を汚す | 出力パスの取り違え | 保存先を `projects/<project>/onboarding/` に固定（CLAUDE.md のプロジェクト固有情報分離ルール準拠）。対象リポジトリのソースには書き込まない |
| 生成資料がコード変更で陳腐化し、誤った地図を信じてしまう | 参画数週間後の再利用 | frontmatter に対象 commit hash を記録。閲覧時に HEAD と異なれば更新を促し、`update` モードで差分反映 |
| `.env` 等から機密情報を書き起こしてしまう | 設定ファイルの網羅的読み取り | `.env` / credentials 類は値を読まない・書かない。キー名の言及に留める |

## 必要なリソース (Resources)

- 外部 API / CLI: 不要（`git`, `ls`, `tree`, grep/Read のみ）
- 権限: 対象リポジトリの読み取り、`projects/<project>/onboarding/` への書き込み
- 参照すべき公開知識: 特になし（対象リポジトリの一次情報のみ）
- ローカル前提: 対象プロジェクトが `projects/<project-name>/` に clone 済み **(ASSUMPTION: CLAUDE.md の projects 運用ルールに従い配置済みと推測)**

## 実装メモ (Implementation Notes)

- ディレクトリ構成: `.claude/skills/arc-reactor-codebase-onboarding/`（SKILL.md + templates/ の 3 雛形。scripts は不要 — 探索は codebase-reader と grep/Read で足りるため）
- 既存スキルとの依存関係: `codebase-reader`（エージェント）を全体探索の下請けとして利用
- templates として `01-map.md` / `02-flows.md` / `03-reading-guide.md` の雛形を保持し、見出し構成を固定する
- 出力は `projects/` 配下（git 管理外）のため、当リポジトリの履歴を汚さない
