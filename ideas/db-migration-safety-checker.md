# db-migration-safety-checker

> 本番 DB を持つプロジェクトの開発者が、マイグレーションを本番適用する直前に、
> ロック事故・データ破壊・新旧コード非互換を静的チェックで潰すためのスキル。

---

## 場面 (When)

- マイグレーションファイルを書き終えて、本番適用の PR を出す直前に「これ流して大丈夫？」と聞くとき
- レビュー対象のブランチに新規マイグレーションが含まれていて、適用時の影響を確認したいとき
- 深夜メンテか通常デプロイかを決めるために「このマイグレーション、ロック掛かる？」と聞くとき
- ゼロダウンタイム運用のサービスで、カラム rename や NOT NULL 追加をどう分割するか相談するとき
- バックフィル（大量 UPDATE）を含むデータマイグレーションの手順を確認したいとき

### 想定トリガー文 (実際のユーザー発話例)

1. 「このマイグレーション、本番に流して大丈夫か見て」
2. 「migration の安全性をチェックして」
3. 「この ALTER TABLE、ロック掛かる？」
4. 「このブランチの新規マイグレーションをレビューして」
5. 「ゼロダウンタイムで適用できるか確認して」
6. 「migration safety check して」
7. 「この NOT NULL 追加、テーブルでかいけど平気？」

---

## 解決する課題 (Problem)

- 現状: DDL のロック挙動はエンジン・バージョン・テーブル規模の 3 変数で変わり、正確に
  暗記している人はほぼいない。「ADD COLUMN に DEFAULT を付けると書き換えが走る」は PG10 までの
  知識で、PG11+ では安全 — この種の知識の賞味期限切れが逆方向の事故（過剰回避・見逃し）を生む。
  ORM がマイグレーションを抽象化するため、生成される実 SQL を見ずに適用してしまうことも多い。
  結果として、CREATE INDEX で書き込みが数分止まる・rename でローリングデプロイ中の旧コードが
  死ぬ、といった事故が「適用して初めて」発覚する。
- 痛みの種類: 本番障害（ロック・ダウンタイム）/ データ破壊（ENUM 削除・checks=0 の FK）/
  手戻り（適用失敗のロールバック）/ 認知負荷（エンジン別・バージョン別の暗記）
- 影響範囲: プロダクション。ロック事故は 1 回でサービス停止に直結し、非対称に痛い

---

## 解決策 (Solution)

1. 対象マイグレーション（ファイル / 差分ブランチ / SQL 断片）と DB エンジン・バージョン・
   テーブル規模感をユーザーと確定する（バージョン不明のまま判定しない）
2. ORM マイグレーションで実 SQL が自明でなければ、ダンプ機能（Django `sqlmigrate`,
   Alembic `--sql`, Laravel `--pretend` 等）で実 SQL を確認してから判定する
3. references/dangerous-operations.md の危険操作カタログとステートメント単位で照合する。
   スキーマ操作に加えて、デプロイ順序（expand-contract）・ロールバック可能性・
   トランザクション制約（CONCURRENTLY と暗黙トランザクションの衝突）も観点に含める
4. 項目ごとに危険度（Critical / Warning / Info）を条件付きで判定する。
   エンジン・バージョンで危険度が変わるものは必ず条件を併記し、規模不明なら
   「行数次第」+ 行数見積もりクエリを添える
5. 安全性レポート（前提 → 指摘事項 → 確認した観点 → 未確認・免責）を提示する。
   危険ゼロでも「確認した観点」を出し、見ていないのか問題ないのかを区別できるようにする
6. 完了条件: 全指摘に file:line・危険になる条件・安全な代替手順（具体 SQL / 分割案）が付いていること

### 入力 (Inputs)

| 名前 | 型 | 必須 | 出どころ | 例 |
|---|---|---|---|---|
| migrations | ファイルパス / ブランチ名 / SQL 文字列 | yes | ユーザー発話 | `db/migrate/20260703_*.rb`, `feature/add-fk` |
| engine + version | string | yes | リポジトリ設定 / ユーザー発話 | `PostgreSQL 14`, `MySQL 8.0.32` |
| table_size | string (行数オーダー) | no (不明なら「行数次第」扱い) | ユーザー発話 | 「users は 500 万行」 |

### 出力 (Outputs)

| 名前 | 形式 | どこに置く | 例 |
|---|---|---|---|
| safety_report | markdown | チャット返信（希望時のみ `<repo>/migration-safety/<YYYY-MM-DD>-<対象>.md` に保存。保存前に確認） | Critical 1 / Warning 2 / 確認済み観点 6 のレポート |
| 指摘項目 | レポート内の節 (危険度 / file:line / 何が起きるか / 条件 / 代替手順) | safety_report 内 | `[Critical] CREATE INDEX (無印) — db/migrate/...:12` |

## 想定ユーザー (Who)

- 主: SES / フリーランスで参画中のエンジニア本人。本番 DB に触るマイグレーションの適用責任を負う
- 副: レビュー相手・依頼者側のエンジニア（レポートを適用可否の判断材料として共有する）

ユーザーが持っている知識: 対象スキーマの構造と、マイグレーションの意図。
持っていない知識: エンジン・バージョン別のロック挙動の正確な条件（暗記に頼ると賞味期限切れを踏む）。

## 既存ツール・スキルとの差別化 (Differentiation)

| 代替 | 限界 | このスキルが上回る点 |
|---|---|---|
| strong_migrations / squawk 等の lint ツール | 特定 ORM・特定エンジン限定。導入されていないプロジェクトでは使えず、デプロイ順序や分割手順までは出さない | ORM 非依存・エンジン横断で、代替手順（具体 SQL・分割案）とデプロイ順序の観点まで出す |
| ステージング環境での試し打ち | 本番と規模が違うのでロック時間が再現しない。デプロイ順序の問題も単発適用では出ない | 規模を変数として条件付きで判定し、規模確認クエリを添える |
| `arc-reactor-code-review` | 汎用レビューで、DDL のロック挙動のバージョン差までは体系的に見ない | エンジン別カタログに基づく専門チェックに特化 |
| `arc-reactor-slow-query-hunter` | クエリの実行性能が対象で、スキーマ変更の適用時安全性は見ない | 「適用した瞬間に何が起きるか」に特化（相互に領分を明記済み） |
| `arc-reactor-db-schema-designer` | 設計時の壁打ちが対象で、出来上がったマイグレーションの適用リスクは見ない | 適用直前の最終関門として機能する |

## 成功条件 (Success Criteria)

- [ ] 全指摘に file:line・危険になる条件・安全な代替手順が付いている（条件なしの Critical がない）
- [ ] エンジン・バージョンで危険度が変わる項目（NOT NULL + DEFAULT 等）が、対象環境の
      バージョンに即した判定になっている（PG14 のプロジェクトで PG10 前提の指摘をしない）
- [ ] 危険ゼロのレポートでも「確認した観点」が列挙され、チェック範囲が読み手に分かる
- [ ] 提示した代替手順（NOT VALID → VALIDATE、CONCURRENTLY、バッチ化）が、そのまま
      マイグレーションとして書ける粒度になっている（ユーザーが調べ直さずに済む）
- [ ] 本スキルを通したマイグレーションで、指摘済み観点に起因する本番ロック事故がゼロ

## 失敗モード予測 (Failure Modes)

| 失敗 | 起きる条件 | 回避策 |
|---|---|---|
| バージョン差の誤判定（PG11+ で安全な操作を Critical と誤報 / 逆に PG10 で見逃し） | エンジンバージョンを確認せずカタログを機械適用したとき | バージョン不明のまま判定しない。不明なら保守側で判定し「保守側に倒した」ことと確認方法を明記 |
| ORM の抽象メソッドから実 SQL を推測で断定して外す | change 系メソッドの生成 SQL が ORM バージョンで変わるとき | ダンプ機能（sqlmigrate / --sql / --pretend）で実 SQL を確認してから判定。確認できなければ ASSUMPTION を明記 |
| 「一瞬で終わる DDL だから安全」と判定してロックキュー詰まりを見逃す | 対象テーブルに長時間クエリが同居する環境 | カタログ §0 を全指摘の共通前提とし、lock_timeout / lock_wait_timeout + リトライを常に添える |
| 代替手順がマイグレーションツールの暗黙トランザクションと衝突して適用失敗 | CREATE INDEX CONCURRENTLY 等を提案するとき | トランザクション無効化（disable_ddl_transaction! 等）までセットで提案する |
| 誤検知の Warning 濫発でレポートが読まれなくなる | 条件付きの項目を無条件で並べたとき | 危険度は必ず条件付き。規模依存は「行数次第」+ 見積もりクエリで、ユーザーが自分で白黒つけられる形にする |

## 必要なリソース (Resources)

- 外部 API / CLI: `git`（差分ブランチの新規マイグレーション抽出）、対象プロジェクトの
  ORM CLI（実 SQL ダンプ。実行はユーザー環境で）。ネットワークアクセス・DB 接続は不要
- 権限: 対象リポジトリの読み取り。レポート保存時のみ書き込み（保存前に確認）
- 参照すべき公開知識: PostgreSQL / MySQL 公式ドキュメントのロック・オンライン DDL の章、
  expand-contract パターン、gh-ost / pt-online-schema-change の適用条件
- ローカル前提: マイグレーションファイルがローカルで読めること（DB 接続は前提にしない）

## 実装メモ (Implementation Notes)

- ディレクトリ構成: `.claude/skills/arc-reactor-db-migration-safety-checker/`
  - `SKILL.md` … フロー（入力確定 → 実 SQL 確定 → カタログ照合 → 危険度判定 → レポート）と判定原則
  - `references/dangerous-operations.md` … エンジン別の危険操作カタログ
    （PostgreSQL / MySQL(InnoDB) 主、SQLite 補足。ロックキュー詰まり・行数見積もりクエリ含む）
- scripts は作らない: SQL / ORM コードの解釈とカタログ照合は Claude の読解が主で、
  パーサを固定スクリプト化すると ORM・方言ごとの追従コストが利益を上回る
- 既存スキルとの依存関係: なし。`arc-reactor-slow-query-hunter`（クエリ性能）・
  `arc-reactor-db-schema-designer`（設計壁打ち）と領分を相互に明記して棲み分ける
- persona: `personas/arc-reactor.md` に従う（応答は日本語、コード・識別子は英語、推測は ASSUMPTION 明記）

## 自己進化を入れるか (Self-Evolution Scope)

- [ ] パイプラインあり
- [x] パイプラインなし — 理由: 本スキルの改善余地はほぼ危険操作カタログの充実
  （新しい危険パターン・エンジンバージョン差の追記）であり、それは
  `references/dangerous-operations.md` への追記で足りる。判定の中核ルール
  （条件なしで断定しない・保守側に倒す）は使用ログで自動書き換えされるべきでない固定規約。
  また対象は毎回他者のリポジトリで、実行ログにプロジェクト固有情報が含まれるため、
  ログ蓄積は CLAUDE.md のプロジェクト情報分離ルールと相性が悪い。
