# tax-prep-organizer (確定申告 経費・売上整理スキル)

> フリーランスエンジニア（青色申告）が、確定申告前に散らばった取引データ（明細 CSV・手元メモ等）を
> **勘定科目別の集計表**と**要確認リスト**に整理するためのスキル。
> **税務助言ではなく情報整理**。科目分類の候補提示は Claude、合算は決定論スクリプトが行い、
> 按分や科目に迷いのある取引は数字に混ぜず「要確認」として本人 / 税理士の確認に回す。

---

## 場面 (When)

ユーザーが**意図的に起動**する（自動起動しない）。

- 確定申告期（1〜3月）や月次の整理タイミングで「経費を科目別にまとめて」と言ったとき
- 銀行 / クレカ / 会計メモの明細 CSV・JSON を指して「これを確定申告用に整理して」と頼んだとき
- 「この出費って何費？」と科目分類の相談から始まり、まとめて整理する流れになったとき
- 既に分類済みのデータに対し「集計だけやり直して」と再集計を頼んだとき
- 「要確認になってる取引の一覧を出して」と保留リストの確認を頼んだとき

### 想定トリガー文 (実際のユーザー発話例)

1. 「確定申告用に経費を整理して」
2. 「この取引データを勘定科目別に集計して」
3. 「今年の売上と経費をまとめて」
4. 「この明細 CSV を仕分けして経費の科目ごとに合計出して」
5. 「按分が必要そうな経費を要確認で分けておいて」
6. 「青色申告の準備、経費の整理から手伝って」
7. 「要確認リストだけ見せて」

> 起動の明示性: description は「確定申告用の取引データ整理・勘定科目別集計」に限定し、
> 請求書作成（invoice-builder 候補）・稼働集計（timesheet-aggregator 候補）・worklog と誤起動しないようにする。
> **税務助言ではなく情報整理**である旨を毎回明示する（§免責とスタンス）。

---

## 解決する課題 (Problem)

- 現状:
  - 確定申告前に、銀行明細・クレカ明細・レシートのメモが複数ファイルに散らばっており、
    1 件ずつ「これは何費か」を考えながら手作業で表計算に転記・合算している。
  - 家事按分（家賃・通信費・電気代）や科目の迷い（クラウド利用料・10 万円超の機材など）が
    途中で出るたびに手が止まり、**迷った取引と確定した取引が混ざったまま**集計されてしまう。
  - 手集計のため合算ミス・二重計上に気づきにくく、申告直前にやり直しが発生する。
- 痛みの種類: 時間（転記・合算）/ 認知負荷（科目判断と集計の同時進行）/ 見落とし（二重計上・計上漏れ）/ 手戻り（申告直前の再集計）
- 影響範囲: 個人の税務申告 → 誤ると修正申告・追徴のリスク。ゆえに**確定分と要確認分の分離**と**免責**が必須。

---

## 解決策 (Solution)

「分類（Claude が候補提示・ユーザーが確定）」と「集計（スクリプトが決定論的に合算）」を分離する。

1. **入力を受け取る**: 取引データ（CSV / JSON、または会話で列挙されたリスト）と対象期間・前提（青色申告等）を確認する。
2. **科目分類（候補提示）**: Claude が `references/account_mapping.md` の対応表に基づき各取引へ勘定科目の**候補**を付ける。
   確証のない取引（按分対象・科目迷い・摘要不明瞭）は科目を確定せず `needs_review: true` を付けて分離する。
3. **ユーザー確認**: 分類結果（特に要確認候補）をユーザーに提示し、確定できるものは確定してもらう。按分率も本人が決めた値のみ採用する。
4. **決定論集計**: 分類済みデータを `scripts/aggregate.py` に渡し、勘定科目別合計・按分適用後金額・要確認リスト・重複疑いをスクリプトが機械的に算出する（Claude は合算しない）。
5. **出力**: 勘定科目別サマリ（Markdown 表）+ 要確認リスト + 免責を提示し、集計結果ファイルをデータ置き場に保存する。
6. **完了条件の判定**: 「集計対象 N 件 / 要確認 M 件 / 収入合計・経費合計」が提示され、要確認 M 件の理由がすべて明記されていること。

### 入力 (Inputs)

| 名前 | 型 | 必須 | 出どころ | 例 |
|---|---|---|---|---|
| transactions | CSV / JSON（1 行 = 1 取引） | yes | ユーザー提供ファイル or 会話から整形 | `tax-data/2026/transactions.csv` |
| transactions.date | string (YYYY-MM-DD) | yes | 明細 | "2026-04-10" |
| transactions.type | "income" \| "expense"（収入/経費 も可） | yes | 明細 + Claude 整形 | "expense" |
| transactions.description | string | yes | 明細の摘要 | "AWS 利用料 3月分" |
| transactions.amount | number（円、正の数） | yes | 明細 | 12800 |
| transactions.account | string（勘定科目 or "要確認"） | yes | Claude 候補 → ユーザー確定 | "通信費" |
| transactions.business_ratio | number（0–100、既定 100） | no | ユーザー確定値のみ | 30 |
| transactions.needs_review | boolean（既定 false） | no | Claude / ユーザー | true |
| transactions.review_reason | string | no | Claude | "家事按分の率が未確定" |
| transactions.source | string | no | 明細の出どころ | "credit-card-2026-03.csv" |
| period | string | no | ユーザー発話 | "2026年分" |

### 出力 (Outputs)

| 名前 | 形式 | どこに置く | 例 |
|---|---|---|---|
| 勘定科目別サマリ | markdown（表 + 免責） | チャット返信 + `tax-data/<year>/summary-<date>.md` | 通信費 152,300 円（12 件）… |
| 集計結果（機械可読） | JSON | `tax-data/<year>/summary-<date>.json` | 科目別合計・要確認・警告 |
| 要確認リスト | markdown 表（サマリ内の一節） | サマリに含める | 取引 + 理由（按分未確定 等） |

**(ASSUMPTION: データ置き場はリポジトリ直下 `tax-data/<year>/` とし、環境変数 `TAX_DATA` で上書き可。
stock-data / worklog-data と同じ「repo 直下 + gitignore」方式を踏襲する想定。)**

## 想定ユーザー (Who)

- 主: 青色申告するフリーランスエンジニア（= 本人）。会計ソフトに入れる前段の「仕分けの下ごしらえ」と全体把握をしたい。
- 副: 顧問税理士（要確認リストの相談相手として。出力を見せる可能性があるため免責と根拠明記が必須）。

ユーザーが持っている知識: 主要な勘定科目の名前と青色申告の概要。
持っていない / 省きたい作業: 1 件ずつの転記・合算、迷い取引と確定取引の区分け管理。

## 既存ツール・スキルとの差別化 (Differentiation)

| 代替 | 限界 | このスキルが上回る点 |
|---|---|---|
| 会計ソフト（freee / マネーフォワード等） | 自動仕訳の確度が低い明細は結局 1 件ずつ手直し。迷い取引の「保留管理」が弱い | 会話で分類の根拠を確認しながら候補付けし、迷いは要確認として構造的に分離 |
| 表計算での手集計 | 転記・合算ミス、二重計上に気づきにくい | 合算は決定論スクリプト + 重複疑いの機械検出 |
| 汎用 LLM に「経費まとめて」 | LLM が合計を暗算して**数値を誤る / 捏造する** | 合算をスクリプトに限定し、Claude は分類候補とレビューのみ |
| invoice-builder（候補） | 売上（請求）側の作成が主目的 | 本スキルは申告用の経費 + 売上の**科目別整理**が主目的 |

## 成功条件 (Success Criteria)

- [ ] 出力サマリの**科目別合計・総計がスクリプト出力の JSON と一致**する（Claude が数値を書き換えていない）。
- [ ] 按分未確定・科目迷い・重複疑いの取引が**集計から除外され、要確認リストに理由付き**で全件載っている。
- [ ] 同一（日付・金額・摘要）の重複取引が入力に含まれるとき、**2 件目以降が「重複疑い」として検出**される。
- [ ] すべての出力（チャット返信・保存ファイル）の冒頭または末尾に**免責（税務助言ではない旨）が明記**されている。
- [ ] 「要確認 M 件」の件数が入力の `needs_review` + スクリプト検出分と一致し、取りこぼしがない。

## 失敗モード予測 (Failure Modes)

| 失敗 | 起きる条件 | 回避策 |
|---|---|---|
| 科目の誤分類 | 摘要が曖昧（例: "Amazon" が消耗品か新聞図書か） / 対応表にない取引 | Claude は**候補**として提示しユーザー確定を挟む。確証がなければ `needs_review` で分離し集計に混ぜない。対応表 `references/account_mapping.md` に判断根拠を明文化 |
| 二重計上 | 同じ取引が銀行明細とクレカ明細の両方に載る / 入力ファイルの重複結合 | スクリプトが（日付・金額・摘要）一致の 2 件目以降を**重複疑い**として要確認へ回し、合計から除外。解消はユーザー判断 |
| 税務助言と誤解される | 免責が無い / 「この科目にすべき」と断定する出力 | 全出力に免責を固定文で添付（スクリプト出力にも埋め込み）。語彙は「候補」「整理」「要確認」に統一し、節税提案・税額計算はしない |
| 合計金額の誤り | LLM が暗算で合算・按分計算する | 合算・按分適用は `aggregate.py` のみが行う（Decimal・円単位・四捨五入規則を固定）。Claude はスクリプト出力を転記するだけ |
| 按分率の勝手な決定 | 「家賃は 3 割が相場」等の一般論で Claude が率を埋める | 按分率は**ユーザーが確定した値のみ**入力に載せる。未確定は `needs_review` へ。率の妥当性判断は本人 / 税理士に委ねる |
| 計上漏れ（要確認の放置） | 要確認リストが埋もれて申告に反映されない | サマリに「要確認 M 件・保留合計 ○円」を必ず並記し、ゼロでない限り完了と見なさない旨を出力に明記 |
| 機密情報の混入 | 明細に口座番号・カード番号が含まれる | 出力・保存前に摘要から口座 / カード番号らしき列を持ち込まない（必要列のみ整形）。`tax-data/` は gitignore で追跡外 |

## 必要なリソース (Resources)

- 外部 API / CLI: なし（オフライン完結）。`python3`（標準ライブラリのみ）。
- 権限: ローカルファイルの読み書きのみ。
- 参照すべき公開知識: 国税庁の記帳・帳簿等の一般的な勘定科目区分（対応表の根拠として references に要約。**税法の解釈はしない**）。
- ローカル前提: git リポジトリ内で実行。取引データと集計結果は `tax-data/` に置き、git 追跡外とする。

## 実装メモ (Implementation Notes)

- ディレクトリ構成:
  - `.claude/skills/ultron-tax-prep-organizer/`
    - `SKILL.md` … 免責・役割分担・標準フロー・出力フォーマット
    - `scripts/aggregate.py` … 決定論集計（科目別合計 / 按分適用 / 要確認分離 / 重複疑い検出）
    - `references/account_mapping.md` … フリーランスエンジニア向け勘定科目対応表 + 要確認になりやすい取引
- 役割分担は high-dividend-stock-screener の「判定はスクリプト、取得とレビューは Claude」を踏襲:
  **分類候補の提示・対話 = Claude / 合算・按分計算・重複検出 = スクリプト**。
- 入力スキーマ（CSV 列 = JSON キー）: `date,type,description,amount,account,business_ratio,needs_review,review_reason,source`。
- 要確認への分離ルール（スクリプト側・決定論）:
  `needs_review` 真 / `account` 空 or "要確認" / 金額不正 / 按分率が (0,100] 外 / type 不明 / 重複疑い（2 件目以降）。
- **(ASSUMPTION: 金額は税込の円・整数を基本とする。消費税の経理方式（税込 / 税抜）の判断はしない。)**
- **(ASSUMPTION: 按分適用額の端数は円未満四捨五入（ROUND_HALF_UP）。会計ソフト側と丸めが異なる場合は要確認扱いにできるよう raw 値も残す。)**

## 自己進化を入れるか (Self-Evolution Scope)

- [ ] パイプラインあり
- [x] パイプラインなし — 理由:
  - 集計ロジックは決定論スクリプトで完結しており、実行ログから学習して改善する余地が薄い。
  - 進化させたくなる唯一の対象は勘定科目対応表だが、これは**税務判断に直結**するため、
    自動改訂は誤分類の固定化リスクがある。更新は本人 / 税理士の確認を経た**手動追記**に限定する。
  - 利用頻度が年数回（申告期 + 月次）でログが閾値まで溜まりにくく、パイプラインの維持コストが見合わない。
