# release-readiness-checker

> アプリケーションをリリースしようとしている開発者が、公開・納品の前に
> 「あと何が足りないか」を根拠付きで棚卸しするためのスキル。
> 感覚頼みのリリース判断を、チェックリスト駆動の網羅的な確認に置き換える。

---

## 場面 (When)

- 個人開発・受託のアプリが「だいたい動く」段階まで来て、公開に踏み切る前の最終確認をしたいとき
- 依頼者から「いつリリースできる？」と聞かれ、残作業を根拠付きで見積もる材料が欲しいとき
- 社内ツールを一般公開に切り替える前に、公開向けに増える要件（セキュリティ・法務表記）を洗いたいとき
- 課金機能を載せる前に、決済まわり以外も含めて何が足りないか一覧にしたいとき
- リリース直前レビューを依頼され、コード品質ではなく「出せる状態か」を横断的に見たいとき

### 想定トリガー文 (実際のユーザー発話例)

1. 「このアプリ、リリースできる状態か確認して」
2. 「リリースまでに何が足りないか洗い出して」
3. 「本番公開前のチェックをして」
4. 「release readiness をチェックして」
5. 「一般公開して大丈夫か見て」
6. 「リリース前の棚卸しをして」
7. 「セキュリティと法務まわりだけリリース観点で見て」

---

## 解決する課題 (Problem)

- 現状: リリース可否の判断が「主要機能は動いたから出せるだろう」という感覚に頼っている。
  機能以外の観点（バックアップ、シークレット管理、法務表記、ロールバック手順）は
  出してから気づくことが多く、公開後の火消しになる。観点リストを毎回頭の中で組み立てるのは
  抜けが出るし、Web のチェックリスト記事は自分の文脈（社内か公開か、課金の有無）に合っていない。
- 痛みの種類: 見落とし（機能以外の観点が抜ける）/ 手戻り（公開後の火消し）/
  認知負荷（毎回観点を思い出す）/ リスク（シークレット漏えい・法務表記漏れは事後対応が高くつく）
- 影響範囲: 個人作業（リリース判断）〜プロダクション（公開後の障害・信用問題）

---

## 解決策 (Solution)

1. リリース文脈をヒアリングして確定する（何を・誰に・いつ・どこへ、規模感、
   社内 or 一般公開、課金の有無、個人情報の有無）
2. references/readiness-checklist.md の 9 カテゴリ（機能完成度 / テスト / エラー処理・可観測性 /
   セキュリティ / パフォーマンス・容量 / データ / 運用 / ドキュメント / 法務・コンプライアンス表記）を
   `rg`・ファイル存在確認・設定読解で照合する
3. 各項目を OK（根拠 file:line）/ Blocker / Should / Nice / 要確認 に分類し、
   ステップ 1 の文脈で重み付けする。コードから判定できないこと（本番インフラ・監視・契約）は
   「無い」と断定せず要確認の質問リストに回す
4. 固定構成のレポート（前提 → サマリ → Blocker → Should → Nice → OK → 要確認 → 免責）を生成し、
   ユーザー確認のうえ `<repo>/release-readiness/<YYYY-MM-DD>.md` に保存する
5. 完了条件: Blocker / Should の全指摘に根拠が付き、コードから判定できない項目が
   要確認の質問リストとして分離されていること

### 入力 (Inputs)

| 名前 | 型 | 必須 | 出どころ | 例 |
|---|---|---|---|---|
| repo_path | string (絶対パス) | yes | ユーザー発話 / cwd | `projects/acme-app/` |
| release_context | ヒアリング結果（何を/誰に/いつ/どこへ・規模感・社内 or 公開・課金・個人情報） | yes | 初回の対話で確定 | 「一般公開 / 月額課金あり / メールアドレスを保持」 |
| focus_categories | enum[] (カテゴリ名) | no (既定: 全 9 カテゴリ) | ユーザー発話 | `["security", "legal"]` |

### 出力 (Outputs)

| 名前 | 形式 | どこに置く | 例 |
|---|---|---|---|
| readiness_report | markdown | チャット要約 + `<repo_path>/release-readiness/<YYYY-MM-DD>.md`（保存前に確認） | Blocker 3 / Should 7 / 要確認 5 のレポート |
| open_questions | レポート内の要確認表（質問文 + 確認先の例） | readiness_report 内 | 「本番 DB のバックアップとリストア確認は済んでいますか」 |

---

## 想定ユーザー (Who)

- 主: 個人開発・受託でアプリを公開/納品しようとしているエンジニア本人
- 副: リリース判断を共有される依頼者・チームメンバー（レポートを残作業合意の材料にする）

ユーザーが持っている知識: 対象アプリの機能と実装。
持っていない知識: 機能以外のリリース要件の網羅リストと、自分の文脈（公開範囲・課金・個人情報）での各要件の重さ。

---

## 既存ツール・スキルとの差別化 (Differentiation)

| 代替 | 限界 | このスキルが上回る点 |
|---|---|---|
| `arc-reactor-code-review` | 対象はコード品質（correctness / security 等の指摘）で、監視・バックアップ・法務表記など「コードの外の不足」は見ない | リリースという締切に対する不足を、コード外の観点込みで横断チェックする |
| `arc-reactor-tech-debt-auditor` | 蓄積した負債のストックを棚卸しするもので、「リリースに必要か」の軸を持たない | 全項目をリリース可否の軸（Blocker/Should/Nice）で判定し、文脈で重み付けする |
| Web のリリースチェックリスト記事 | 汎用リストで自分のリポジトリと突き合わせてくれない。文脈（社内/公開・課金）の重み付けもない | 実リポジトリを走査して根拠 file:line 付きで判定し、ヒアリングした文脈で重みを変える |
| `/security-review` | セキュリティ 1 カテゴリのみ | 9 カテゴリを横断する（深掘りが要る箇所は既存スキルへ回す設計） |

## 成功条件 (Success Criteria)

- [ ] Blocker / Should の全指摘に根拠（file:line、または探索パスと不在の事実）が付いている
- [ ] コードから判定できない項目が「無い」と断定されず、すべて要確認の質問リストに分離されている
- [ ] 同じ不足でもリリース文脈（社内/公開・課金・個人情報）で判定が変わり、その根拠が 1 行付いている
- [ ] レポートを見たユーザーが「これは違う」と却下する Blocker が 2 割以下
- [ ] レポートの Blocker / Should がそのまま ToDo・issue 化に使える（再整形不要）

## 失敗モード予測 (Failure Modes)

| 失敗 | 起きる条件 | 回避策 |
|---|---|---|
| コードに痕跡が無いだけの項目（監視・バックアップ・TLS 終端）を「無い」と断定し、誤指摘でレポートの信頼を失う | インフラ・SaaS 側で満たされている項目をコード走査だけで判定したとき | リポジトリ外で満たされうる項目を references に明示し、**必ず要確認（質問形式）に回す**規約を SKILL.md の大原則に置く |
| 法務項目で「特商法表記が必要です」等と法的要否を断定してしまう | 法域・事業形態を確認せずに一般論を当てはめたとき | 法務カテゴリは要否を断定せず確認事項として出す + 「法的助言ではない」免責をレポートに毎回明記 |
| 社内ツールに一般公開向けの重い要求（法務表記・レートリミット）を突きつけ、ノイズ化する | リリース文脈を聞かずにチェックを始めたとき | ヒアリングをフローの第 1 ステップに固定し、文脈をレポート冒頭の「前提」に記載。重み付けの根拠を文脈に紐づける |
| `rg` パターンの機械ヒットをそのまま指摘にしてしまう（テストコードの TODO、`.env.example` のダミー値等） | シグナル収集と判定を区別しなかったとき | references に「機械的な確認はシグナル収集であり、ヒット箇所を読んでから判定する」を共通ルールとして明記 |
| 発見したシークレットの値をレポートに転記して二次漏えいする | キーのコミット混入を検出したとき | 値は転記せず場所のみ指摘する規約（tech-debt-auditor と同じ）を SKILL.md に明記。対応は鍵ローテーション優先と references に記載 |

## 必要なリソース (Resources)

- 外部 API / CLI: `git`, `rg`, ファイル存在確認。ネットワークアクセスは不要
  **(ASSUMPTION: 依存脆弱性の照合は Claude の知識の範囲で行い、`npm audit` 等の
  ネットワークを伴う確認はユーザーに実行を依頼する形にした。tech-debt-auditor と同じ制約)**
- 権限: 対象リポジトリの読み取り。レポート保存時のみ書き込み（保存前に確認）
- 参照すべき公開知識: 一般的なリリースチェックリストの観点（production readiness review の類型）、
  OWASP 的なセキュリティ基本観点、日本の EC 関連表記（特商法等）の存在（要否判断はしない）
- ローカル前提: 対象リポジトリがローカルにあること（`projects/<name>/` 配下が典型）

## 実装メモ (Implementation Notes)

- ディレクトリ構成: `.claude/skills/arc-reactor-release-readiness-checker/`
  - `SKILL.md` … ヒアリング → チェック → 判定 → レポートのフロー、判定 5 分類の規約、出力フォーマット
  - `references/readiness-checklist.md` … 9 カテゴリ × 項目別の「何を確認するか / 機械的な確認方法 / 不足時の典型対応」（SKILL.md の肥大化を避けて分離。改善はここへの追記で行う）
- scripts は作らない: 確認は `rg` / ファイル存在確認の組み合わせで足り、判定は文脈依存の
  読解が主のため（tech-debt-auditor と同じ判断）
- 既存スキルとの依存関係: 深掘りが要る箇所を `arc-reactor-code-review`（品質・セキュリティの精査）、
  `arc-reactor-test-scaffolder`（テスト雛形）、`jarvis-issue-planner`（Blocker の issue 化）へ
  回す出口を持つ。必須依存はなし
- persona: `personas/arc-reactor.md` に従う（応答は日本語、コード・識別子は英語、推測は ASSUMPTION 明記）
