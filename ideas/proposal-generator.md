# proposal-generator

> フリーランスエンジニアが、クライアントから相談・要件を受けた直後に、
> 見積根拠が透明な提案書ドラフトを固定構成で数分で作るためのスキル。

---

## 場面 (When)

- クライアントから「こういうことできますか？いくらくらい？」と相談を受け、要件メモを手元に持っているとき
- 商談メモ・議事録（friday-giziroku の出力など）を元に、次回打ち合わせまでに提案書を出す必要があるとき
- 既存案件の追加開発について、追加見積を求められたとき
- 要件は固まっているが、タスク分解と工数の見積もりを整理して金額に落とし込みたいとき
- 過去に口頭で伝えた概算を、正式な提案書・見積書ドラフトとして文書化するとき

### 想定トリガー文 (実際のユーザー発話例)

1. 「この要件で提案書を作って」
2. 「見積もり出したいからドラフト作って」
3. 「この案件の提案書ドラフトお願い」
4. 「追加開発の見積書を作って」
5. 「このメモから提案書と見積内訳をまとめて」
6. 「工数見積もって提案書にして」

---

## 解決する課題 (Problem)

- 現状: 提案書・見積を毎回手作業で Word / Markdown に書いており、構成が案件ごとにバラバラ。
  見積根拠（どのタスクに何時間かかるか）を書き忘れ、クライアントから「なぜこの金額？」と
  問い合わせが来て往復が発生する。スコープ外作業の明記漏れで、無償対応に巻き込まれることもある。
- 痛みの種類: 時間（1通あたり1〜2時間 **(ASSUMPTION: ユーザーの実測値ではなく一般的な作成時間からの推測)**）、
  見落とし（前提条件・スコープ外の明記漏れ）、手戻り（金額根拠の説明往復）。
- 影響範囲: 個人作業（作成コスト）＋クライアントとの信頼関係（根拠の不透明さは受注率・単価交渉に直結）。

---

## 解決策 (Solution)

1. 要件（テキスト・メモ・議事録ファイル）を受け取り、不足情報（納期・予算感・単価など）を対話でヒアリングする。
2. 要件をタスクに分解し、タスクごとの工数見積案を提示して**ユーザーの確認・修正を必ず受ける**。
3. 単価（時間 or 人日）をユーザーに確認し、タスク分解 × 工数 × 単価の**見積内訳表**を組む。
4. 固定構成（背景と課題 / 提案内容 / スコープ / スケジュール / 見積内訳表 / 前提条件 / 体制・進め方）で
   提案書ドラフト（Markdown）を生成する。
5. 金額・スコープ外項目・前提条件をユーザーに最終確認してもらい、承認を得て完了。

### 入力 (Inputs)

| 名前 | 型 | 必須 | 出どころ | 例 |
|---|---|---|---|---|
| requirements | string \| file path (Markdown/text) | yes | ユーザー発話 or 添付ファイル | 「ECサイトに会員機能を追加したい。ログイン、マイページ、購入履歴」 |
| unit_price | number + unit (円/時 or 円/人日) | yes（対話で確認） | ユーザー発話 | 「8,000円/時」 |
| deadline | date \| string | no（対話で確認） | ユーザー発話 | 「8月末リリース希望」 |
| client_name | string | no | ユーザー発話 | 「株式会社〇〇御中」 |
| effort_overrides | list of {task, hours} | no | 対話（工数確認ステップ） | 「ログイン実装は16hじゃなくて24hで」 |

### 出力 (Outputs)

| 名前 | 形式 | どこに置く | 例 |
|---|---|---|---|
| proposal draft | Markdown（固定構成7セクション） | `<repo>/proposals/<YYYY-MM-DD>-<client-or-project>.md` **(ASSUMPTION: 出力先ディレクトリは既存運用が無いため新設。既存の慣習があれば SKILL.md 側で変更)** | 背景〜体制まで揃った提案書ドラフト |
| estimate table | Markdown 表（タスク / 工数 / 単価 / 小計 / 合計） | 提案書内の「見積内訳」セクション | 「会員登録画面実装 \| 16h \| 8,000円 \| 128,000円」 |
| チャット要約 | text | チャット返信 | 合計金額・タスク数・確認してほしい前提の提示 |

---

## 想定ユーザー (Who)

- 主: フリーランスエンジニア本人（受注前の提案・見積を自分で書く立場）
- 副: 提案書の読み手であるクライアント（非エンジニアの意思決定者を想定）

ユーザーは技術と工数感覚を持っているが、提案書の構成整備・スコープ明記の網羅は毎回抜けがち。
読み手（クライアント）は技術詳細を知らないため、本文は専門用語を抑えた表現にする。

---

## 既存ツール・スキルとの差別化 (Differentiation)

| 代替 | 限界 | このスキルが上回る点 |
|---|---|---|
| 手書き（Word / Markdown 直書き） | 構成が毎回バラつく。前提条件・スコープ外の明記漏れ | 固定構成＋含まない作業の明記を強制 |
| 汎用 Claude への都度依頼 | 工数・金額を勝手に創作するリスク。構成が安定しない | 工数はユーザー確認必須・金額承認必須のフローを固定 |
| friday-daily-report / giziroku | 報告・議事録用で、見積・提案の構成を持たない | タスク分解×工数×単価の内訳表と提案構成に特化 |
| 見積 SaaS（Misoca 等） | 金額の帳票は作れるが、提案文・スコープ・根拠の文書化はできない | 提案書と見積根拠を1本のドキュメントに統合 |

## 成功条件 (Success Criteria)

- [ ] 生成された提案書に、固定構成7セクション（背景と課題 / 提案内容 / スコープ / スケジュール / 見積内訳 / 前提条件 / 体制・進め方）が欠けなく含まれている
- [ ] 見積内訳表の「小計の総和 = 合計金額」が機械的に検算して一致している
- [ ] スコープに「含まない作業」が最低1項目以上、前提条件が最低2項目以上明記されている
- [ ] 工数と単価が、ユーザーが対話で確認・承認した値のみで構成されている（承認前の値がドラフトに残らない）
- [ ] 要件ヒアリング開始からドラフト提示までが対話3〜5往復以内に収まる **(ASSUMPTION: 実運用での往復数は未計測。使用感で調整)**

## 失敗モード予測 (Failure Modes)

| 失敗 | 起きる条件 | 回避策 |
|---|---|---|
| 工数の過小見積 | AI が楽観的な見積を出し、そのまま提案書になる | 工数はあくまで「たたき台」として提示し、**ユーザーの明示的な確認・修正なしに本文へ流し込まない**。テスト・環境構築・打ち合わせ等の間接タスクをチェックリストで必ず提示する |
| 要件の創作 | 入力メモが曖昧で、書いていない機能や仕様を勝手に補完する | persona 原則「入力にない情報を創作しない」を適用。不明点は推測で埋めず対話で確認。どうしても仮置きが必要な箇所は「前提条件」セクションに明記して仮置きだと分かるようにする |
| スコープ外の明記漏れ | 「含まない作業」を書き忘れ、無償対応の温床になる | テンプレートに「含まない作業」を必須セクションとして固定し、空のままなら生成を完了させない |
| 金額の計算ミス | 表の小計・合計を LLM が暗算して桁を間違える | 内訳表の各行（工数×単価）と合計を Bash 等で機械的に検算してから提示する |
| 依頼者に伝わらない表現 | タスク名が技術用語のまま（例: 「API エンドポイント実装」） | 内訳表のタスク名は依頼者語彙（例: 「データ連携機能の開発」）に言い換え、必要なら技術補足を括弧書きにする |

## 必要なリソース (Resources)

- 外部 API / CLI: なし（ローカル完結）。検算に `python3` または `awk` を使用
- 権限: リポジトリ内 `proposals/` への書き込みのみ
- 参照すべき公開知識: 特になし（見積の相場観はユーザーが持つ。スキルは相場を提示しない）
- ローカル前提: onclimb-industries リポジトリ内で実行。要件ファイルを渡す場合はローカルパス

## 実装メモ (Implementation Notes)

- ディレクトリ構成: `.claude/skills/friday-proposal-generator/`
  - `SKILL.md` … 役割・トリガー・標準フロー・固定構成・品質原則
  - `templates/proposal-template.md` … 提案書の固定構成テンプレート（プレースホルダ入り）
- scripts は置かない（生成は対話＋テンプレート差し込みで完結し、決定論的レンダラーが必要な複雑さではない）。
  検算は Bash ワンライナーで足りる。
- 既存スキルとの関係: friday-giziroku の議事録や karen-problem-essence-organizer の整理結果を
  要件入力として受け取れる（依存はしない。単体でも動く）。
- persona は `personas/friday.md`（読み手が他者＝クライアントの共有ドキュメント）。

## 自己進化を入れるか (Self-Evolution Scope)

- [ ] パイプラインあり
- [x] パイプラインなし — 理由: 提案書は案件ごとに1回完結のタスクで、出力の良し悪しは
  「クライアントに通ったか」という**スキルの外側でしか観測できない**ため、実行ログからの
  自動学習が機能しにくい。構成・見積根拠の透明性はテンプレートで固定して守るべき不変条件であり、
  進化で崩れると事故（スコープ外明記の消失など）につながる。改善は手動でテンプレートと
  SKILL.md を直す運用とする。
