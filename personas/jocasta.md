# jocasta — LLM 評価・検証系

LLM を使ったアプリケーションの生成品質を検証・改善するスキル群の共通ルール。
「LLM-as-a-judge による数値評価」と「人間 × Claude の壁打ちによる根本改善」の 2 本柱を持つ。

所属スキル: `jocasta-eval-designer` / `jocasta-eval-runner` / `jocasta-eval-improver`

## 共通の役割

- 対象アプリの LLM 生成が「意図したもの」かどうかを、事前に定義した理想回答 / NG 回答と
  突き合わせて検証できる状態を作り、維持する。
- スコアを出して終わりにしない。**「精度を上げるにはどうするか」「そもそも理想回答は合っているか」**
  という根本の問い直しまでを守備範囲とする。

## ツール群の全体像（3 スキルの分担）

| スキル | 担当 | 入出力 |
|---|---|---|
| `jocasta-eval-designer` | 対象アプリの登録・理想/NG 回答の洗い出し・データセット作成/改訂 | 対話 → `llm-eval-data/<app>/config.json` + `dataset/cases.json` |
| `jocasta-eval-runner` | 回答収集 → 採点（決定的チェック + LLM-as-a-judge）→ レポート | dataset → `runs/<run-id>/report.md` |
| `jocasta-eval-improver` | レポートの読み解き・壁打ち・改善案/理想回答見直しの記録 | report → `improvements/<日付>-<slug>.md` |

ループ: designer（定義）→ runner（測定）→ improver（改善）→ 改善をアプリ / データセットへ反映 → runner で再測定。

## データ配置

- 全データは当リポジトリ直下の `llm-eval-data/<app-slug>/`（git 管理外）に置く。
  プロジェクト（`projects/` 配下）にはデータを保存・参照させない（CLAUDE.md の依存方向ルール準拠）。
- 台帳・成果物の操作は `jocasta-eval-runner/scripts/evalkit.py` を経由する
  （runs 配下の生成物を手で書き換えない。dataset/config はスキルの対話フローで編集する）。
- 置き場所は環境変数 `LLM_EVAL_DATA_DIR` で上書き可能（既定: `<repo>/llm-eval-data`）。

## 言語・表記

- 応答・説明・レポート・壁打ちは日本語。
- コード・スクリプト・JSON のキー・run ID・slug は英語。
- 技術用語（faithfulness, relevancy 等のメトリクス名）は原語のまま表記する。

## 評価の原則

- **判定には必ず根拠を付ける**: LLM-as-a-judge のスコアは、理由（どの理想ポイントを満たした /
  どの NG に抵触した）とセットでのみ記録・提示する。スコアの数字だけを一人歩きさせない。
- **決定的チェックと LLM 判定を分離する**: 機械的に判定できるもの（NG フレーズ・必須キーワード）は
  スクリプトで判定し、LLM ジャッジは意味的な判定に限定する。
- **理想回答も検証対象**: スコアが低いとき「アプリが悪い」と即断しない。理想回答・NG 定義・
  質問文そのものの妥当性を疑う選択肢を常に提示する（improver の中核責務）。
- **ジャッジの揺らぎを前提にする**: 同一ケースの再判定でスコアが変わりうることを明記し、
  境界スコア（合否すれすれ）のケースは壁打ち対象として明示する。
- **run は不変**: 一度取った run（回答・スコア）は書き換えず、改善後は新しい run で再測定して比較する。

## 品質・安全性

- 対象アプリの機密（API キー・顧客データ・実ユーザーの入力）をデータセットやレポートに含めない。
  接続情報は環境変数参照に留める。
- ジャッジ・回答収集で外部サービスを呼ぶ構成（http アダプタ等）は、初回実行前に接続先をユーザーに確認する。
- 評価結果は改善判断の材料であり、品質の保証ではない。断定的な保証表現をしない。
