# jocasta — LLM 評価・検証系

LLM を使ったアプリケーションの生成品質を検証・改善するスキル群の共通ルール。
「LLM-as-a-judge による数値評価」と「人間 × Claude の壁打ちによる根本改善」の 2 本柱を持つ。

所属スキル: `jocasta-eval-designer` / `jocasta-eval-runner` / `jocasta-eval-improver` / `jocasta-eval-autopilot`

## 事実ベースの原則（全スキル必須・他のすべてに優先する）

jocasta 系のすべての出力（意見・判断・改善案・評価コメント）はこの原則に完全に従う。

1. **推測だけでものを言わない**: 意見・判断・改善案は、必ず参照可能な根拠とセットで述べる。
   根拠として使えるのは次のいずれかのみ。
   - run の実データ（`responses.jsonl` / `judgments.jsonl` / `scores.jsonl` / `report.md`）
   - dataset・config の定義（ケース ID・フィールドを特定して引用）
   - 対象アプリの実コード・実ドキュメント（パスを特定して引用）
   - 出典付きの Web 情報（下記 3 の条件を満たすもの）
2. **根拠の書式**: データ由来は「ファイル + ケース ID + 該当箇所の引用」、Web 由来は
   「URL + 取得日 + 一次/二次の別」を必ず添える。根拠を示せない主張はそもそもしない。
3. **Web からは信用できるエビデンスだけ取得する**: 公式ドキュメント・公式ルール・一次情報のみを
   根拠に採用する。個人ブログ・まとめ記事等の二次情報は根拠にせず、参考として使う場合も
   その旨を明記する。確認できなかった事項は「未確認」と書く（推測で埋めない）。
4. **推測は明記し、多用しない**: どうしても推測・仮説・意見を述べる場合は、文頭に
   **「（推測）」** を付けて事実と明確に区別する。1 回の出力に（推測）を多用しない
   （目安: 2 件以内。超えるなら根拠集めに戻る）。基本は事実ベースで話す。
5. **推測でデータを書き換えない**: （推測）タグ付きの内容を根拠に dataset・config・改善記録を
   変更しない。interactive ではユーザーの明示的な合意を根拠に昇格でき、auto では適用せず
   保留リストに記録する。
6. **分からないことは分からないと言う**: データに無いことは「現データからは判断できない」と述べ、
   判断に必要な追加データ（ケース追加・再測定・Web での裏取り）を具体的に提案する。

## 実行モード（interactive / auto）と共通サイクル

全スキルの各ステップは、次の 3 段サイクルで動く。

1. **ドラフト提示**: 生成物（データセット案・評価結果・改善案）を根拠付きで提示する。
2. **壁打ちブラッシュアップ**: ユーザーの指摘・追加情報を受けて修正する（往復回数に制限なし。
   ユーザーの追加情報は「ユーザー提供の事実」として根拠に使える）。
3. **確定適用**: 合意した対応案だけをデータ・記録に適用する。

- **interactive（既定）**: 3 の適用は必ずユーザー合意後に行う。
- **auto**: 2 を省略し、根拠の揃っている変更のみ自動適用する（（推測）由来の変更は禁止・保留行き）。
  全変更を記録に残し、終了時にレビュー用サマリを提示する。auto の使用はユーザーが明示的に
  指示したときのみ（`jocasta-eval-autopilot` 経由が標準）。

## ステップの独立性

「理想回答・評価基準の生成」（designer）/「検証＝回答収集」（runner: collect）/
「評価＝採点・レポート」（runner: judge/score/report）/「改善」（improver）は、
それぞれ独立に単体実行できる。autopilot はこれらを連結するだけで、専用の隠れた処理を持たない。

## 共通の役割

- 対象アプリの LLM 生成が「意図したもの」かどうかを、事前に定義した理想回答 / NG 回答と
  突き合わせて検証できる状態を作り、維持する。
- スコアを出して終わりにしない。**「精度を上げるにはどうするか」「そもそも理想回答は合っているか」**
  という根本の問い直しまでを守備範囲とする。

## ツール群の全体像（3 スキルの分担）

| スキル | 担当 | 入出力 |
|---|---|---|
| `jocasta-eval-designer` | 対象アプリの登録・理想/NG 回答の洗い出し・データセット作成/改訂 | 対話 → `llm-eval-data/<app>/config.json` + `dataset/cases.json` |
| `jocasta-eval-runner` | 回答収集 → 採点（決定的チェック + LLM-as-a-judge）→ レポート | dataset → `runs/<run-id>/report.md` |
| `jocasta-eval-improver` | レポートの読み解き・壁打ち・改善案/理想回答見直しの記録 | report → `improvements/<日付>-<slug>.md` |
| `jocasta-eval-autopilot` | 上記 3 ステップを auto モードで連結し、理想回答の改訂まで含めて自動でループを回す | 目標/上限 → `autopilot/<日時>-summary.md` |

ループ: designer（定義）→ runner（測定）→ improver（改善）→ 改善をアプリ / データセットへ反映 → runner で再測定。
autopilot はこのループの自動運転版（各ステップの実体は同じ）。

## データ配置

- 全データは当リポジトリ直下の `llm-eval-data/<app-slug>/`（git 管理外）に置く。
  プロジェクト（`projects/` 配下）にはデータを保存・参照させない（CLAUDE.md の依存方向ルール準拠）。
- 台帳・成果物の操作は `jocasta-eval-runner/scripts/evalkit.py` を経由する
  （runs 配下の生成物を手で書き換えない。dataset/config はスキルの対話フローで編集する）。
- 置き場所は環境変数 `LLM_EVAL_DATA_DIR` で上書き可能（既定: `<repo>/llm-eval-data`）。

## 言語・表記

- 応答・説明・レポート・壁打ちは日本語。
- コード・スクリプト・JSON のキー・run ID・slug は英語。
- 技術用語（faithfulness, relevancy 等のメトリクス名）は原語のまま表記する。

## 評価の原則

- **判定には必ず根拠を付ける**: LLM-as-a-judge のスコアは、理由（どの理想ポイントを満たした /
  どの NG に抵触した）とセットでのみ記録・提示する。スコアの数字だけを一人歩きさせない。
- **決定的チェックと LLM 判定を分離する**: 機械的に判定できるもの（NG フレーズ・必須キーワード）は
  スクリプトで判定し、LLM ジャッジは意味的な判定に限定する。
- **理想回答も検証対象**: スコアが低いとき「アプリが悪い」と即断しない。理想回答・NG 定義・
  質問文そのものの妥当性を疑う選択肢を常に提示する（improver の中核責務）。
- **ジャッジの揺らぎを前提にする**: 同一ケースの再判定でスコアが変わりうることを明記し、
  境界スコア（合否すれすれ）のケースは壁打ち対象として明示する。
- **run は不変**: 一度取った run（回答・スコア）は書き換えず、改善後は新しい run で再測定して比較する。

## 品質・安全性

- 対象アプリの機密（API キー・顧客データ・実ユーザーの入力）をデータセットやレポートに含めない。
  接続情報は環境変数参照に留める。
- ジャッジ・回答収集で外部サービスを呼ぶ構成（http アダプタ等）は、初回実行前に接続先をユーザーに確認する。
- 評価結果は改善判断の材料であり、品質の保証ではない。断定的な保証表現をしない。
