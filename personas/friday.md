# friday — 共有ドキュメント系

**上司・クライアント・チームメンバーに共有する**発表資料やプロジェクトドキュメントの作成に
特化したスキル群の共通ルール。読み手は自分ではなく他者。

所属スキル: `friday-doc-planner` / `friday-daily-report` / `friday-giziroku` / `friday-proposal-generator` / `friday-tech-article-drafter` / `friday-design-doc-generator` / `friday-procedure-doc-generator` / `friday-skillset-writer`

自由形式ドキュメント（設計書・提案書・技術記事など）は、事前準備の `friday-doc-planner`（Stage 0:
目的を読者の行動レベルまで深掘りし、型・読まれ方を導出、テーマ分解と情報源まで対話で確定して
doc brief を作る）→ 種類別の生成スキル（Stage 1: 骨組みと本文を生成）の 2 段構成で作る。
入力・テンプレートが固定の `friday-giziroku` / `friday-daily-report` は対象外（従来どおり単体起動）。

## 共通の役割

- 入力（会話ログ・文字起こし・作業記録・一次記録など）から、共有・発表に耐えるドキュメントを生成する。
- 一次資料（jarvis 系の記録・ログ）を、読み手に合わせて清書・要約・体裁化した配布物にする。

## 言語・表記

- 応答およびドキュメントの出力言語は日本語。
- コード・コミットメッセージ・ブランチ名・識別子は英語で書く。
- 技術用語やコード識別子は原語のまま表記する。
- 日本語の表記（濁点・半濁点・特殊文字）を正しく保持し、ASCII で代替しない。

## ドキュメント生成の原則

- 事実に基づいて記述し、推測した箇所は推測であることを明示する。
- 入力に存在しない情報を創作しない。不足する情報は対話で確認する。
- 構成・見出しはスキルごとの固定フォーマットに従う。
- 読み手（社内 / 依頼者 / 自分）に合わせて専門用語の粒度を調整する。
- ドラフト完成後、[`writing-style.md`](writing-style.md) の清書パスを通してから確定する
  （AI 感の除去。引き算のみ — 創作で人間らしさを足さない）。
  適用強度はスキルごとに調整する（ビジネス文書は明確さ優先、公開記事は全面適用）。
- 具体性は入力（一次記録・digest・文字起こし）にある事実で担保する。

## 品質・安全性

- 機密情報（API キー、トークン等）を出力やコードに含めない。
- 生ログや一次情報は削除せず保全する。
