# テスト戦略（テスティングピラミッド）

## テスティングピラミッドとは

Mike Cohn が提唱したテスト戦略モデル。
ピラミッドの形状が示すとおり、**ユニットテストを最も多く、上位に行くほど少なくする**のが特徴。

### なぜピラミッド型か

バックエンドでは「個々の関数・モジュールが正しい」ことがシステム全体の信頼性に直結する。ビジネスロジック、データ変換、バリデーションなど純粋なロジックが多く、これらはユニットテストで高速かつ安定的に検証できる。上位のテストほどコスト（実行時間・メンテナンス・不安定さ）が増すため、**高速で安定したユニットテストを土台にし、上位層では下位層で検証できないことだけを確認する**のが最も効率的。

### ピラミッドの構造（下から上へ）

```
        ╱‾‾╲
       │ E2E │             ← 最少・最高コスト
      ╱────────╲
     │Integration│         ← 中程度・境界の検証
    ╱──────────────╲
   │     Unit        │     ← 最多・最速・最安定
   └─────────────────┘
```

## テストの種類と観点

### Unit テスト（最も厚くする）

**ビジネスロジック・データ変換・バリデーション**を中心に、関数・メソッド単位で検証する。テスティングピラミッドの土台。

| 観点 | 内容 |
|------|------|
| 目的 | 個々のロジックが仕様どおりに動くことを保証する |
| 対象 | ドメインロジック、バリデーション、データ変換、ユーティリティ、エラーハンドリング |
| 粒度 | 入力パターンごとに1ケース（正常系・異常系・境界値） |
| 原則 | 外部依存（DB, HTTP, ファイルシステム）はモックする。1テストの実行は数ミリ秒以内 |

**ユニットテストを厚くする理由:**
- 実行が高速で、開発ループに組み込みやすい
- 失敗時に原因箇所が即座に特定できる
- リファクタリング時の安全網として最もコストが低い

### Integration テスト

**モジュール間の連携やインフラとの境界**を検証する。「つなぎ目」に絞ることで、ユニットテストでは検出できないバグを捕捉する。

| 観点 | 内容 |
|------|------|
| 目的 | モジュール間の結合やインフラ接続が正しく動くことを保証する |
| 対象 | DB操作（リポジトリ層）、外部API連携、ミドルウェアチェーン、認証・認可フロー |
| 粒度 | ユースケース単位で1ケース |
| 原則 | DB はテスト用の実インスタンスを使う。外部 API はモックしてよい。テストごとにデータをリセットする |

### API テスト

**HTTPリクエスト → レスポンス**の一連の流れをエンドポイント単位で検証する。

| 観点 | 内容 |
|------|------|
| 目的 | API の契約（リクエスト形式、レスポンス形式、ステータスコード）が仕様どおりであることを保証する |
| 対象 | 各エンドポイントの正常系・異常系・認証エラー・バリデーションエラー |
| 粒度 | エンドポイント × メソッド × 主要な入力パターン |
| 原則 | アプリケーションサーバーを起動し、実際にHTTPリクエストを送る。DBは実インスタンスを使い、外部サービスはモック可 |

### E2E テスト

**システム全体を通した主要シナリオ**を検証する。本番に近い構成で実行し、デプロイ可能な品質を最終確認する。

| 観点 | 内容 |
|------|------|
| 目的 | システム全体が連携して期待どおりに動くことを最終確認する |
| 対象 | 収益や信頼に直結するクリティカルパス（ユーザー登録→認証→主要操作→結果確認 など） |
| 粒度 | 主要シナリオごとに1ケース。網羅性よりクリティカル度で選定する |
| 原則 | モックを使わず本番に近い環境で実行する。不安定なテスト（flaky test）は即座に修正または削除する |

## テスト実行タイミング

| タイミング | 実行対象 | 目的 |
|-----------|---------|------|
| ファイル保存時 | 関連する Unit | 即時フィードバック |
| PR 作成時 | Unit + Integration + API | 変更の安全性を担保 |
| main マージ後 | 全テスト（E2E 含む） | リリース品質の確認 |

## テスト設計の原則

<!-- ASSUMPTION: プロジェクトの品質要件に応じて取捨選択する -->

1. **テスト層の責務を分離する** — 各層で検証する対象を明確にし、同じことを複数の層で重複して検証しない
2. **外部依存の扱いを層ごとに決める** — Unit ではモック、Integration では実DB、E2E ではモックなしと明確に分ける
3. **テストデータは各テストで独立させる** — テスト間の依存を排除し、実行順序に関係なく成功する状態を保つ
4. **失敗が原因を示すテストを書く** — 失敗時に「何が壊れたか」が即座にわかる粒度と命名にする
5. **カバレッジは指標であり目標ではない** — 数値を追うのではなく、ビジネス上重要なパスがカバーされているかを重視する
