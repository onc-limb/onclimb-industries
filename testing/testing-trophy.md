# テスト戦略（テスティングトロフィー）

## テスティングトロフィーとは

Kent C. Dodds が提唱したフロントエンド向けテスト戦略モデル。
トロフィー（trophy）の形状が示すとおり、**インテグレーションテストを最も厚くする**のが特徴。

### なぜトロフィー型か

フロントエンドでは「コンポーネント単体が正しい」ことより「組み合わせた結果が正しい」ことにユーザー価値がある。ユニットテストを厚くしても、コンポーネント間の結合やデータフローの不整合は検出できない。インテグレーションテストは**コストと信頼性のバランスが最も良い層**であり、ここを厚くすることでリファクタリング耐性と実ユーザー体験への信頼度を同時に高められる。

### トロフィーの構造（下から上へ）

```
        ╱‾‾‾‾‾╲
       │  E2E   │        ← 少数・高コスト・高信頼
      ╱──────────╲
     │ Integration │      ← 最も厚く・費用対効果が最大
    ╱────────────────╲
   │   Component      │   ← UIの見た目と振る舞いの確認
  ╱────────────────────╲
 │     Unit              │ ← 純粋ロジックに限定
 ├───────────────────────┤
 │   Static Analysis     │ ← TypeScript・Linter・Formatter
 └───────────────────────┘
```

## テストの種類と観点

### Static Analysis（静的解析）

テストコードを書く前の土台。型チェックと Lint で防げるバグはここで排除する。

| 観点 | 内容 |
|------|------|
| 目的 | 型エラー、未使用変数、フォーマット崩れの自動検出 |
| 対象 | 全ソースコード |
| 原則 | CI で必ず実行し、警告を放置しない |

### Unit テスト

**純粋関数・ユーティリティ・ビジネスロジック**に限定する。UIコンポーネントのユニットテストは原則書かない。

| 観点 | 内容 |
|------|------|
| 目的 | 入出力が明確なロジックの正しさを保証する |
| 対象 | 状態計算、バリデーション、データ変換、フォーマッタなど |
| 粒度 | 入力パターンごとに1ケース（正常系・異常系・境界値） |
| 原則 | DOM や外部依存を持たないこと。モックは最小限にする |

### Component テスト

単一コンポーネントを**ユーザー視点で**レンダリングし、見た目と操作を検証する。

| 観点 | 内容 |
|------|------|
| 目的 | コンポーネントが Props に応じて正しく描画・反応することを保証する |
| 対象 | 再利用可能なUIコンポーネント、フォーム部品、表示切替コンポーネント |
| 粒度 | Props のバリエーション × ユーザー操作の組み合わせ |
| 原則 | 実装詳細（state, ref, 内部メソッド）に依存しない。ユーザーが見る・触れるもの（テキスト、ロール、ラベル）をクエリする |

### Integration テスト（最も厚くする）

複数コンポーネントを組み合わせた**機能単位**をテストする。テスティングトロフィーの核。

| 観点 | 内容 |
|------|------|
| 目的 | 機能がユーザー操作の流れに沿って正しく動くことを保証する |
| 対象 | ページ・フィーチャー単位（フォーム送信→バリデーション→API呼び出し→結果表示 など） |
| 粒度 | ユースケース単位で1ケース |
| 原則 | API はモックしてよいが、コンポーネント間の結合はモックしない。ユーザーの操作順序に沿って記述する |

**インテグレーションテストを厚くする理由:**
- コンポーネント単体では検出できない結合部分のバグを捕捉する
- E2E より高速で安定しているため、開発ループに組み込みやすい
- リファクタリング時に「機能が壊れていない」という信頼を最も効率的に得られる

### E2E テスト

実際のブラウザ上で、**ユーザーの主要シナリオ**を通しで検証する。

| 観点 | 内容 |
|------|------|
| 目的 | デプロイ可能な品質であることを最終確認する |
| 対象 | 収益や信頼に直結するクリティカルパス（認証、購入、データ登録など） |
| 粒度 | 主要シナリオごとに1ケース。網羅性よりクリティカル度で選定する |
| 原則 | モックを使わず本番に近い環境で実行する。不安定なテスト（flaky test）は即座に修正または削除する |

## テスト実行タイミング

| タイミング | 実行対象 | 目的 |
|-----------|---------|------|
| ファイル保存時 | Static Analysis + 関連 Unit | 即時フィードバック |
| PR 作成時 | Unit + Component + Integration | 変更の安全性を担保 |
| main マージ後 | 全テスト（E2E 含む） | リリース品質の確認 |

## テスト設計の原則

<!-- ASSUMPTION: プロジェクトの品質要件に応じて取捨選択する -->

1. **ユーザーの使い方に沿ってテストする** — 実装詳細ではなく、ユーザーが見るもの・操作するものを検証する
2. **モックは境界だけ** — 外部API・タイマーなどシステム境界のみモックし、内部モジュール間のモックは避ける
3. **テストがリファクタリングを妨げたら設計を見直す** — 実装に密結合したテストは負債になる
4. **1テスト1アサーションに固執しない** — 1つのユーザーフローを1テストでまとめて検証してよい
5. **カバレッジは指標であり目標ではない** — 数値を追うのではなく、ユーザーにとって重要なパスがカバーされているかを重視する
