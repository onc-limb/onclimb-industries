# コードレビュー観点チェックリスト

<!-- ASSUMPTION: レビュー対象はWebアプリケーション（バックエンド中心）を想定。プロジェクト固有の規約が不明なため、汎用的な観点を採用。 -->

## 正確性

- [ ] 要件・仕様を満たしているか
- [ ] ロジックに誤りがないか（境界値、off-by-one、null処理）
- [ ] エッジケースが考慮されているか

## セキュリティ

- [ ] ユーザー入力のバリデーションがあるか
- [ ] SQLインジェクション・XSS・CSRF対策がされているか
- [ ] 認証・認可チェックが適切か
- [ ] 機密情報（APIキー、トークン等）がハードコードされていないか

## テスト

- [ ] 変更に対するテストが追加・更新されているか
- [ ] 正常系・異常系の両方がカバーされているか
- [ ] テストが独立して実行できるか（他テストへの依存がない）

## エラーハンドリング

- [ ] エラーが適切にキャッチ・伝播されているか
- [ ] エラーメッセージが意味のある内容か
- [ ] リソースリーク（未クローズのコネクション等）がないか

## 可読性・保守性

- [ ] 命名が意図を表しているか
- [ ] 関数・クラスの責務が明確か（単一責任）
- [ ] 不要なコード（デッドコード、デバッグ用コード）が残っていないか
- [ ] 複雑なロジックにコメントがあるか

<!-- ASSUMPTION: コーディング規約の詳細が不明なため、一般的な可読性基準を記載。 -->

## パフォーマンス

- [ ] N+1クエリが発生していないか
- [ ] 不要なループ・再計算がないか
- [ ] 適切なインデックス・キャッシュが使われているか

## 設計

- [ ] 既存のアーキテクチャ・パターンに従っているか
- [ ] 変更の影響範囲が最小限か
- [ ] 将来の変更に対して壊れにくい構造か

<!-- ASSUMPTION: プロジェクト固有のアーキテクチャパターンが不明なため、汎用的な設計観点を記載。 -->

## Git・差分

- [ ] コミット粒度が適切か（機能変更とリファクタリングが分離されているか）
- [ ] 不要な差分（フォーマット変更のみ等）が混入していないか
