# friday-design-doc-generator — コード → 設計書 / README / ADR 草案

対象コードベース（と直前の会話）を入力に、**他者に共有できるドキュメント草案**を 1 本生成する。

中心原則は **創作禁止**。草案に書いてよいのは次の 3 種だけ。

1. **事実** — コードから確認できたこと。技術的記述には根拠（ファイルパス）を添える。
2. **ユーザーの明言** — 対話・直前の会話でユーザーが明示した情報。
3. **明示的な推測** — 「(推測)」タグを付けたもの。

それ以外（コードに無く、ユーザーも明言していないこと）は **「TBD」** と書き、ヒアリングに回す。
空欄をそれらしい文章で勝手に埋めることは、このスキルにおける最大の失敗である。

## 役割

- 読み手は自分ではなく**他者**（チームメンバー・引き継ぎ先エンジニア・依頼者）。
  friday 系（共有ドキュメント系）として、共有・納品に耐える体裁と事実性を担保する。
- 「コードから読める事実の棚卸し」はこのスキルが担い、「コードに書かれていない背景・意図」は
  ユーザーから対話で引き出す。この分担を崩さない。
- 応答・ドキュメントは日本語。コード・識別子・技術用語は英語（原語）のまま表記する。

## トリガー

| ユーザー発話の例 | 動作 |
|---|---|
| 「このプロジェクトの README 作って」 | README 草案を生成 |
| 「今のコードベースから設計書の草案を書いて」 | 設計書草案を生成 |
| 「引き継ぎ用に設計ドキュメントまとめて」 | 読み手=引き継ぎ先として設計書を生成 |
| 「DB を PostgreSQL にした経緯、ADR にして」 | ADR を生成（決定内容を対話で確定） |
| 「README が古いからコードに合わせて書き直して」 | 既存 README を事実ベースで更新 |
| 「今の話、設計書に反映して」 | 直前の会話をユーザーの明言として取り込み更新 |

自動起動はしない。作業ログの整理（jarvis 系）や報告資料（friday-daily-report）の依頼と混同しない。

## 標準フロー

### 1. 対象と読み手の確認

- 対象リポジトリ / モジュールのパスを確定する（未指定なら聞く）。
- **読み手**を確認する: チーム内エンジニア / 引き継ぎ先エンジニア / 依頼者（非エンジニア寄り）。
  読み手によって専門用語の粒度と説明量を調整する。
- 範囲（リポジトリ全体か、特定モジュールか）を確認する。巨大リポジトリなら scope を絞るか
  モジュール単位の分割生成を提案する。

### 2. コード調査（事実収集）

- ディレクトリ構成・エントリポイント・依存関係（package.json / pyproject.toml / go.mod 等の
  manifest）・設定ファイル・既存ドキュメント・テスト・CI 設定を読む。
  調査には codebase-reader（エージェント / スキル）を下請けに使ってよい。
- 収集結果は**出典（ファイルパス）付きの事実リスト**としてまとめる。出典のない項目は事実として扱わない。
- 除外: `dist/` `build/` `node_modules/` `vendor/` 等の生成物・外部コードは事実の根拠にしない。
- 対象が git 管理下なら、生成時点の commit hash を取得して草案冒頭に記録する
  （草案はスナップショットであることを明示するため）。

### 3. ドキュメント種別の選択

目的に応じて README / 設計書 / ADR を提案し、ユーザーと確定する（複数生成も可）。

| 種別 | 向いている目的 | テンプレート |
|---|---|---|
| README | プロジェクトの入口。何であるか・動かし方・構成の概観 | `templates/readme.md` |
| 設計書 | 引き継ぎ・レビュー・納品。アーキテクチャと設計判断の全体像 | `templates/design-doc.md` |
| ADR | 個別の技術的決定の記録（Status / Context / Decision / Consequences） | `templates/adr.md` |

### 4. 草案生成

- 種別のテンプレートに事実リストを流し込む。**見出し構成は固定**。埋められないセクションは
  削除せず `TBD` を入れる（欠落だと「調査漏れ」か「情報不足」か読み手に伝わらないため）。
- 技術的記述（機能・構成・依存・エンドポイント・設定など）には根拠パスを添える。
- コードから読み取れるが確証が持てない記述には「(推測)」タグを付ける。
- 生成後の**自己チェック**: 草案内の各技術的記述について、(a) 出典パスがある / (b) ユーザーの
  明言に基づく / (c) 「(推測)」タグ付き、のいずれかであることを照合する。どれにも該当しない
  記述は削除するか TBD 化する。
- 続けて [`personas/writing-style.md`](../../../personas/writing-style.md) の清書パスを地の文に通す。
  対象は **構造宣言（「この章では〜を説明します」）・コピュラ回避（「〜としての役割を果たす」→「〜である」）・
  三点セット癖** に絞る。事実・出典パス・TBD・「(推測)」タグは変えない。

### 5. 不明点の TBD 明示とヒアリング

- 草案内の TBD を末尾「TBD 一覧」に集約し、**チャットで質問として提示**する。
  典型的にコードから読めない項目: 背景・目的、非機能要件、運用・デプロイの実態、採用理由、今後の予定。
- ユーザーが回答したら該当箇所に反映する（回答は「ユーザーの明言」として扱う）。
- 回答がない項目は TBD のまま残す。**勝手に埋めない**。
- 保存先を確定して書き込み、パスと TBD の残数を提示して完了する。

## 出力の置き場所

CLAUDE.md の「プロジェクト固有情報の分離ルール」に従う。

- 出力は**必ず対象プロジェクトのディレクトリ配下**に置く:
  `projects/<project-name>/docs/` を既定とし、対象リポジトリに既存のドキュメント置き場
  （`docs/` 等）があればその慣習に従う。
- onclimb-industries リポジトリ側（ideas/ や skills/ など）にプロジェクト固有の設計書を置かない。
- 書き込み前に出力パスが対象プロジェクト配下であることを確認し、外れていれば保存先をユーザーに確認する。

## ドキュメント種別ごとの固定構成

テンプレート本体は `templates/` を参照。見出しの追加は可（読み手・対象に応じて）だが、
既定の見出しの削除・順序変更はしない。

- **README** (`templates/readme.md`):
  概要 / 主な機能 / 技術スタック / ディレクトリ構成 / セットアップ / 使い方 / 設定 / 開発（テスト・lint）/ TBD 一覧 / 参照したソース
- **設計書** (`templates/design-doc.md`):
  メタ情報（生成日・対象 commit）/ 目的と背景 / スコープ / 全体アーキテクチャ / コンポーネント詳細 / データ設計 / 外部インターフェース / 非機能・運用 / 設計判断と経緯 / TBD 一覧 / 参照したソース
- **ADR** (`templates/adr.md`): Nygard 形式ベース。
  Status / Context（背景・制約）/ Decision（決定内容）/ Consequences（帰結）/ 検討した代替案 / TBD 一覧 / 参照したソース

## 事実と推測の区別ルール

| 区分 | 書き方 | 例 |
|---|---|---|
| 事実（コード由来） | そのまま記述し、根拠パスを添える | 「HTTP サーバは Express で実装されている（`src/app.ts`）」 |
| 事実（ユーザー明言） | そのまま記述する。必要なら「（ヒアリングによる）」と添える | 「DB は本番で PostgreSQL 15 を使用（ヒアリングによる）」 |
| 推測 | 文末に **「(推測)」** タグ | 「`retry` パラメータは外部 API の不安定さ対策とみられる (推測)」 |
| 不明 | **「TBD」** と書き、TBD 一覧に集約 | 「デプロイ手順: TBD」 |

- 「(推測)」はコード上の手がかりがある場合のみ使う。手がかりすらないものは TBD。
- 既存ドキュメント（古い README 等）の記述は、コードと突き合わせて確認できたものだけ事実として
  引き継ぐ。矛盾があればコード側を優先し、矛盾点をユーザーに報告する。

## 出力フォーマット

- 1 ドキュメント = 1 Markdown ファイル。ファイル名は `README.md` / `design-doc.md`（対象が
  モジュールなら `design-doc-<module>.md`）/ ADR は `adr/NNNN-<kebab-case-title>.md`（連番）。
- 草案冒頭に生成日・対象パス・対象 commit（取得できた場合）を記載する。
- 末尾に「TBD 一覧」「参照したソース」（調査で根拠にしたファイルパスの一覧）を必ず置く。
- 完了時はチャットで、保存パス / 生成した種別 / TBD の残数と質問一覧 を提示する。

## 品質・安全性（persona: friday 準拠）

- 入力（コード・会話）に存在しない情報を創作しない。不足は TBD + 対話で確認する。
- 推測箇所は「(推測)」で必ず明示する。
- 機密情報（API キー・トークン・パスワード・実クレデンシャル）は草案に転記しない。
  設定の説明は「キー名と用途」のみ記載し、値は書かない。実名・顧客名は読み手の範囲を確認して扱う。
- 既存の README 等を書き直す場合も、元ファイルは上書き前に内容を確認し、消える情報
  （コードから確認できず引き継げなかった記述）があればユーザーに報告する。
- 日本語の表記（濁点・半濁点）を正しく保持し、ASCII で代替しない。
