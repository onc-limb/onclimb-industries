# meeting-prep-briefer — 会議事前ブリーフ生成スキル

打合せの**事前**に、議題・関連資料から論点を整理した事前ブリーフを 1 本生成する。
設計の出典: [`ideas/meeting-prep-briefer.md`](../../../ideas/meeting-prep-briefer.md)。

> **giziroku との対の位置づけ**: [friday-giziroku](../friday-giziroku/SKILL.md) が
> 会議の**事後**（文字起こし → 議事録）を担うのに対し、本スキルは会議の**事前**
> （議題・資料 → ブリーフ）を担う。giziroku が蓄積した過去議事録（`giziroku/minutes/`）と
> 参加者名簿（`giziroku/config/roster.yaml`）を**読み取り専用**の下調べ材料として使い、
> 「前回決まったこと・持ち越したこと」を次の会議の準備に還流させる。

> **起動条件（重要）**: ユーザーが会議準備を明示的に依頼したときだけ起動する。
> 会議後の議事録化は friday-giziroku、作業ログの整理は jarvis-worklog / jarvis-record が担う。

## 場所（コードとデータは分離）

- ツール・雛形: このスキルディレクトリ `.claude/skills/karen-meeting-prep-briefer/`（`templates/`）
- **出力（ブリーフ）**: 既定は**チャット返信**。ユーザーがファイル保存を求めた場合のみ
  `giziroku/briefs/YYYY-MM-DD_<会議タイトル>_ブリーフ.md` に保存する
  **(ASSUMPTION: 保存先は giziroku（事後）と対の運用になるよう `giziroku/briefs/` とした。
  `giziroku/` は git 管理外のため、クライアント名等の機密を含んでも安全)**。
  日付は会議開催日を使う。
- **下調べの入力（すべて読み取り専用。書き込み・移動はしない）**:
  - 過去議事録: `giziroku/minutes/`
  - 参加者名簿: `giziroku/config/roster.yaml`（あれば参加者の役割推定に使う）
  - jarvis 系記録: `report-record/`（案件×日付の一次記録）、`knowledge-base/`（技術ノート）
    **(ASSUMPTION: `worklog-data/` は生ログで量・ノイズが多いため既定の検索対象から除外し、
    ユーザーが明示指定したときのみ読む)**
  - ユーザー指定資料: 任意のパス・貼り付けテキスト

## トリガーと非トリガー

| 起動する発話の例 | 起動しない（誤起動を避ける） |
|---|---|
| 「明日の打合せのブリーフ作って」 | 「議事録にして」→ friday-giziroku（事後） |
| 「この議題で事前準備したい。論点整理して」 | 「今日の作業まとめて」→ jarvis-worklog |
| 「次の定例の前に、前回の持ち越しを洗い出して」 | 「思考を整理したい」→ karen-problem-essence-organizer |
| 「〇〇さんとの MTG、想定質問と回答案を用意して」 | 「提案書を作って」→ friday-proposal-generator |
| 「来週の商談の下調べして。決めたいことも整理したい」 | （会議と無関係な一般的な調べもの） |
| 「会議前の頭の整理を手伝って。アジェンダはこれ」 | |

## 標準フロー（4 フェーズ。この順序を崩さない）

「**ヒアリングで前提を固める → 資料を集める → 論点に組み立てる → ブリーフを出す**」。
最初のヒアリングを必須にして、論点の的外れを入口で潰す。

### Phase 1. ヒアリング（会議の目的・参加者・議題）

対話で以下を確定する。ユーザーの発話・貼り付けから推定できるものは**推定案を提示して補正してもらう**
（ゼロから全部質問しない。確認は要点に絞る）。

1. **会議の目的**: この会議は何のためにあるか（1 文）。
2. **決めたいこと**: 会議で決定・合意したい事項。最低 1 件は言語化する。
   「報告だけ」の会議なら「決めることは無い」と明示的に確定する。
3. **参加者と役割**: 誰が出るか。`giziroku/config/roster.yaml` があれば候補として提示する。
4. **議題・開催日時**: アジェンダ（無ければ目的から議題案を起こして確認）と開催日。

### Phase 2. 関連資料の収集（読み取り専用）

1. 議題・案件名・参加者名・技術語など**複数のキーワード**で、以下を検索する
   （表記ゆれを考慮し、キーワードは言い換えを含めて 3 つ以上使う）:
   - `giziroku/minutes/` — 同じ会議体・同じ案件の過去議事録（特に直近回の
     「保留・継続検討」「TODO」セクション）
   - `report-record/` — 当該案件の直近の作業記録（現状・障害・相談事項）
   - `knowledge-base/` — 議題に関わる技術ノート
2. ヒットした候補を**一覧でユーザーに提示し、使う資料を取捨選択してもらう**。
   0 件のときは「見つからなかった」と明示し、パスの直接指定や貼り付けを促す
   （黙って資料なしで進めない — 関連資料の見落とし対策）。
3. ユーザー指定資料（提案書ドラフト・先方からのメール等）があれば読み込む。
4. 採用した資料は**ファイルパスと日付**を控え、ブリーフ末尾の「参照資料」に列挙する。

### Phase 3. 論点整理

1. 議題を「決めたいこと」単位の**論点**に分解する。論点は**優先度順に最大 5 件**
   （準備過剰の防止）。決定に直結しない議題は「その他の議題」として 1 行に圧縮する。
2. 各論点について、資料から以下を組み立てる:
   - **現状**: 資料由来の事実。出典（ファイル名・日付）を紐づける。
   - **選択肢**: 取り得る案。資料に無い案を足す場合は `(推測)` を付ける。
   - **自分のスタンス案**: どの案を推すか＋理由。判断材料が足りない場合は
     「情報不足で決められない — 会議で〇〇を確認してから」と明示する（無理に断定しない）。
3. 資料間で情報が矛盾する場合は**日付の新しい方を採用**し、矛盾があったことを注記する。
4. 前回議事録の「保留・継続検討」「TODO」は**漏れなく**「前回からの持ち越し」に転記し、
   自分が担当の宿題は現時点のステータス（完了/未完了/一部）を付ける。

### Phase 4. ブリーフ生成

1. [`templates/brief.md`](templates/brief.md) の固定フォーマットで組み立てる。
   セクション構成（固定。省略しない — 該当なしのセクションは「なし」と書く）:
   - **会議の目的**
   - **決めたいこと**
   - **論点**（論点ごとに: 現状 / 選択肢 / 自分のスタンス案）
   - **想定質問と回答案**（相手から聞かれそうなこと＋自分の回答案。根拠の出典付き）
   - **前回からの持ち越し**（前回議事録の保留・TODO 由来）
   - **確認したいこと**（自分から相手に確認すべきこと。情報不足の論点はここに落とす）
2. 全体を **150 行以内**に収める（直前に読み切れる分量を守る）。
3. 既定は**チャット返信**。ユーザーが保存を求めたら
   `giziroku/briefs/YYYY-MM-DD_<会議タイトル>_ブリーフ.md` に **1 ファイル 1 会議**で保存する
   **(ASSUMPTION: 会議資料としてファイル保存も可、という運用を想定)**。
   同名ファイルが既にあれば上書きせず連番を付ける。
4. **完了判定**: 「決めたいこと」が 1 件以上言語化され、全論点にスタンス案
   （または「情報不足」の明示）が付いていれば完了。満たせない場合は何が不足かを伝えて
   追加ヒアリングする。

## 品質・安全性（karen persona 準拠）

- **事実と推測を区別する**: 資料由来の記述には出典（ファイル名・日付）を紐づけ、
  推測には必ず `(推測)` を付ける。区別のつかない記述をブリーフに残さない。
- **入力に無い情報を創作しない**: 「先方の意向」「決定済みの方針」等を、資料・ユーザー発話に
  根拠なく書かない。書けないことは「確認したいこと」に落とす。
- **機密の扱い**: ブリーフはクライアント名・案件情報を含みうるため、ファイル保存は
  git 管理外の `giziroku/briefs/` のみに行う。API キー・トークン等を出力に含めない。
- **読み取り専用の徹底**: `giziroku/` や `report-record/` 等の既存データを書き換え・移動しない。
  本スキルが書き込むのは `giziroku/briefs/` 配下だけ。
- 応答・出力は日本語。技術用語・コード識別子は原語のまま表記する。

## やってはいけないこと

- 自動起動しない。明示依頼が無ければ走らない。
- Phase 1（目的・決めたいことの確定）を飛ばして論点を立てない。
- 資料検索が 0 件のとき、黙って資料なしのブリーフを出さない（必ずユーザーに知らせる）。
- 論点を 5 件超に増やさない・ブリーフを 150 行超に膨らませない（準備過剰の防止）。
- 出典の無い「事実」を書かない。推測に `(推測)` を付け忘れない。
- 議事録そのものを生成・更新しない（事後は friday-giziroku の領分）。
