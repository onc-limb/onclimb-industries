# friday-proposal-generator — クライアント向け提案書・見積ドラフト

要件・工数を入力に、**見積根拠が透明な**提案書ドラフト（Markdown）を固定構成で生成する。
毎回手作業だった提案書・見積の作成を、「ヒアリング → タスク分解と工数確認 → 固定構成で生成」の
3ステップに定型化する。読み手は非エンジニアのクライアント。

## 役割

- 要件（ユーザー発話・メモ・議事録ファイルなど）を、クライアントに提出できる提案書ドラフトへ整形する。
- 見積の根拠を **タスク分解 × 工数 × 単価の内訳表** として必ず示し、
  **前提条件** と **含まない作業（スコープ外）** を明記して、金額の透明性と後日のトラブル防止を担保する。
- 工数・単価・金額の確定は行わない。**確定するのは常にユーザー**であり、スキルはたたき台と構成を提供する。

## トリガー

| ユーザー発話の例 | 動作 |
|---|---|
| 「この要件で提案書を作って」 | 発話・直前の会話内容を要件として開始 |
| 「見積もり出したいからドラフト作って」 | 要件ヒアリングから開始 |
| 「この案件の提案書ドラフトお願い」 | 指定された案件情報を要件として開始 |
| 「追加開発の見積書を作って」 | 追加開発分に限定した提案書として開始 |
| 「このメモから提案書と見積内訳をまとめて」 | 指定ファイルを Read して要件として開始 |
| 「工数見積もって提案書にして」 | タスク分解・工数見積ステップを重点に開始 |
| 「この brief で提案書を作って」「doc-briefs/〜.md から見積にして」 | doc brief を Read し、Stage 1 として開始（下記「doc brief 連携」参照） |

自動起動はしない。ユーザーが明示的に依頼したときだけ動く。

## doc brief 連携（Stage 1 として動く場合）

前段の [friday-doc-planner](../friday-doc-planner/SKILL.md)（Stage 0）が作成した doc brief が
入力として渡された場合:

- brief は `doc-briefs/<YYYY-MM-DD>-<slug>.md`（固定見出し: 目的（読者の行動レベル）/ 対象読者 /
  前段の文脈 / 分類と導出根拠 / 最も主張すべきこと / テーマ / テーマ分解 / 混在検査 / TBD）。
- brief で確定済みの項目（目的・読み手・前段の文脈・テーマ分解など）は「ユーザーの明言」として
  扱い、同じ内容の再ヒアリング（標準フロー 1 の該当部分）を省略する。brief の
  「最も主張すべきこと」「前段の文脈」（何が反対され・何と比較することになったか）は
  提案の骨格として使う。
- **工数・単価・金額の確認と承認フロー（標準フロー 2 以降）は brief の有無に関わらず省略しない。**
- brief 無しの単体起動は従来どおり（標準フローを最初から行う）。

## 標準フロー

### 1. 要件ヒアリング

1. 要件の入力を受け取る（発話テキスト / Markdown・テキストファイルのパス / 直前の会話内容）。
2. 提案書に必要な情報のうち不足しているものを対話で確認する。**推測で埋めない。**
   - 依頼者名・案件名（宛先表記に使う）
   - 実現したいこと・背景の課題（「なぜやるか」が書けるだけの情報）
   - 希望納期・マイルストーン
   - 単価（円/時 または 円/人日。**毎回必ずユーザーに確認する**）
   - 予算感・制約（あれば）
3. 要件のうち曖昧な点は、勝手に仕様を補完せず質問する。ユーザーが「仮でいい」と答えた項目は
   仮置きとして扱い、後の「前提条件」セクションに必ず記載する。

### 2. タスク分解と工数見積（ユーザー確認必須）

1. 要件をタスクに分解し、タスクごとの工数のたたき台を **チャット上の表** で提示する。
2. 分解時は、開発本体だけでなく以下の間接タスクの要否を必ず確認する（過小見積の防止）:
   - 要件定義・仕様すり合わせの打ち合わせ
   - 環境構築・デプロイ準備
   - テスト（動作確認・受け入れ確認）
   - ドキュメント・引き継ぎ資料
   - リリース後の初期サポート（含めるなら期間を明記、含めないなら「含まない作業」へ）
3. ユーザーがタスク・工数を確認し、修正指示を反映する。
   **ユーザーの明示的な承認を得るまで、次のステップに進まない。**
4. 承認された工数 × 単価で内訳表を組み、各行の小計と合計を **Bash（python3 等）で機械的に検算** する。
   暗算で表を作らない。

### 3. 固定構成でドラフト生成

1. `templates/proposal-template.md` を雛形に、下記の固定構成でドラフトを生成する。
2. 出力先: `<repo>/proposals/<YYYY-MM-DD>-<client-or-project>.md`
   （`proposals/` が無ければ作成する。ファイル名の日付は作成日、識別子は案件名の英語 kebab-case）
3. 生成後、[`personas/writing-style.md`](../../../personas/writing-style.md) の清書パスを通す。
   ビジネス文書なので明確さ優先で、対象は次の癖に絞る:
   - 「重要です」「最適な」「豊富な」等の空語・宣伝調を、スコープ・数字・作業内容で言い換える
   - 提案・推奨は言い切り、根拠と前提条件を添える（両論併記で終えない）
   - タスク・提案項目の数を癖で 3 に丸めない（分解の結果が項目数を決める）
   - 見積・契約条件に関わる説明は硬い表現のままでよい（信頼性優先）
4. 生成後、合計金額・タスク数・仮置きした前提条件をチャットで提示し、
   **金額とスコープ外項目についてユーザーの最終承認を求める。** 修正があれば反映して再提示する。

## 提案書の固定構成（順序・見出しを崩さない）

1. **背景と課題** — 依頼者が困っていること・実現したいことを、依頼者の言葉で1〜3段落
2. **提案内容** — 何を作る・何をするか。技術詳細ではなく「できるようになること」を中心に
3. **スコープ** — 「含む作業」と「**含まない作業**」を必ず両方箇条書きで。含まない作業が空のまま完成にしない
4. **スケジュール** — フェーズ / 期間 / 完了の目安の表。納期未確定なら「ご発注後 n 週間」の相対表記
5. **見積内訳** — タスク / 工数 / 単価 / 小計 の表 ＋ 合計行（税表記を明記）。★このスキルの核。省略・丸め統合をしない
6. **前提条件** — 見積が成り立つ前提（資料提供、環境アカウント貸与、仮置きした仕様、有効期限など）を最低2項目以上
7. **体制・進め方** — 担当体制、連絡手段・頻度、進捗報告の方法、変更が出た場合の扱い

## 出力フォーマット

- Markdown 1ファイル。依頼者（非エンジニア）が読む前提で、**専門用語を抑える**。
  - タスク名・本文は依頼者語彙にする（例: 「API エンドポイント実装」→「データ連携機能の開発」）。
  - 技術名の記載が必要な場合は括弧書きの補足に留める。
- 見積内訳表の形式:

```markdown
| No. | 作業項目 | 内容 | 工数 | 単価 | 小計 |
|---|---|---|---|---|---|
| 1 | 会員登録機能の開発 | 登録画面・確認メール送信 | 24h | 8,000円/h | 192,000円 |
| ... | | | | | |
| | **合計（税別）** | | **XXh** | | **XXX,000円** |
```

- 金額は3桁区切り。単位（円/h・円/人日、税別/税込）を表内に明記する。
- 日本語の表記（濁点・半濁点・特殊文字）を正しく保持し、ASCII で代替しない。

## 品質・安全性（personas/friday.md 準拠）

- **入力にない情報を創作しない。** 要件・仕様・工数・金額のいずれも、ユーザーが与えた・承認した
  情報だけで構成する。不足は対話で確認し、仮置きは「前提条件」に明示する。
- **金額に関わる値（工数・単価・合計）は、ユーザーの承認なしにドラフトへ確定させない。**
  たたき台の提示 → 承認 → 反映の順を必ず守る。
- 内訳表の計算は機械的に検算する（小計の総和 = 合計を確認してから提示する）。
- 機密情報（API キー、他クライアントの情報・実名）をドラフトに含めない。生成前に最終チェックする。
- 過去の提案書を参照する場合も、**他案件の固有情報を新しい提案書へ持ち込まない**（構成・言い回しのみ参考にする）。
- このスキルは提案書の「ドラフト」を作るものであり、法的な契約文書の作成・レビューは行わない。
  契約条件（支払条件・検収条件など）はユーザー自身の確認を促す。

## 補足

- 雛形: [`templates/proposal-template.md`](templates/proposal-template.md)。
  構成の改善はテンプレートを直接編集する（自己進化パイプラインは持たない）。
- 議事録（friday-giziroku の出力）や思考整理（karen-problem-essence-organizer の出力）を
  要件入力として渡してもよい。その場合も不足情報のヒアリングは省略しない。
