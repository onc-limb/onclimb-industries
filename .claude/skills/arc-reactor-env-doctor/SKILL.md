# env-doctor — 環境エラーの診断・修復・手順書反映 (arc-reactor)

環境構築やローカル作業でエラーが出たとき、原因を体系的に診断し、自分の環境で解消し、
さらに**同じエラーで他の人がハマらないよう手順書（README / setup ドキュメント）の修正案まで作る**スキル。
「自分の環境で直して終わり」にしない。直した知見を手順書に還元するところまでが 1 回の実行。

> 大原則: **検証なくして完了なし、修復だけで終わらせない。**
> 「直ったはず」ではなく元のコマンドの再実行で解消を確認し、
> 手順の不備が原因なら手順書の修正 diff 案まで出して初めて完了とする。

persona: [`personas/arc-reactor.md`](../../../personas/arc-reactor.md) に従う
（応答は日本語 / コード・識別子は英語 / 推測は ASSUMPTION 明記）。

## トリガー

| ユーザー発話の例 | 動作 |
|---|---|
| 「npm install がコケるので直して」 | 診断 → 修復 → 検証 → 手順書反映の全フロー |
| 「この環境構築エラーを診断して」 | 全フロー |
| 「ローカルでサーバーが起動しない、原因調べて」 | 全フロー（起動系はポート衝突・環境変数から切り分け） |
| 「セットアップ手順どおりにやったのにエラーになる」 | 全フロー（手順書の不備を疑う優先度を上げる） |
| 「このエラー、環境のせいか調べて直して」 | 環境起因かの判定から開始 |
| 「直したついでに README の手順も直しておいて」 | 解決済みエラーの手順書反映のみ実施 |

ユーザーが明示的に依頼したときだけ起動する（エラー出力を見かけたついでに自動起動しない）。

## 標準フロー

### 1. 情報収集

診断に必要な情報を揃える。不足していれば取得コマンドを実行して自分で集める（読み取りのみなので確認不要）。

- **エラー全文**: 要約された発話だけで診断しない。ターミナル出力・ログの原文を確認する
  （切れている場合は再実行やログファイルの Read で全文を取る）。
- **実行コマンドと実行場所**: 何を・どのディレクトリで・どう実行したか。
- **環境情報**: OS / アーキテクチャ（`uname -m`。macOS は arm64 / x86_64 の別が重要）、
  ランタイムバージョン（`node -v`, `python --version`, `go version` 等）、
  パッケージマネージャとそのバージョン、バージョンマネージャの有無（asdf / nvm / pyenv / mise 等）。
- **リポジトリが要求する環境**: `.tool-versions`, `.nvmrc`, `package.json` の `engines`,
  `go.mod`, `Dockerfile`, CI 設定など、期待バージョンが書かれたファイルを探して現物と突き合わせる。
- **直近の環境変更**: OS アップデート、brew upgrade、バージョン切り替え、新規マシン移行など。
  「昨日まで動いていたか」で切り分けが大きく変わるため必ず聞く。

### 2. 原因診断

[references/common-causes.md](references/common-causes.md) のチェックリスト順に切り分ける。診断前に必ず読むこと。

- **最初に「環境起因」か「コード起因」かを判定する。** コード起因（ロジックのバグ、テストの失敗、
  型エラー等）なら本スキルの範囲外と伝え、通常のデバッグ作業に切り替える。
  判定の目安は references 冒頭のトリアージ節にある。
- 仮説には必ず根拠を添える（エラーメッセージの該当行、バージョンの実測値と期待値の差、
  `which` の結果など）。根拠なしに「たぶん〜が原因」で修復に進まない。
- 複数の仮説があるときは、検証コストが低い順・可能性が高い順に 1 つずつ潰す。
  一度に複数の変更を入れると、何が効いたのか分からなくなる。

### 3. 修復

診断で特定した原因に対する修正コマンドを実行する。操作の種類で扱いを分ける。

- **プロジェクト内に閉じる操作**（`node_modules` の再構築、ローカルキャッシュ削除、
  lock ファイルからの再インストール等）: 実行して結果を報告する。
- **システム状態を変える操作**（ランタイムのインストール / アンインストール、
  グローバル設定・シェル設定ファイルの変更、brew での導入等）: **実行前に一言確認する。**
  何を・なぜ変えるのか、影響範囲（このプロジェクトだけか、他のプロジェクトにも及ぶか）を添えて聞く。
- 修復は原因 1 つにつき 1 手ずつ。効かなかった手はその場で記録しておく（フロー 6 で使う）。

### 4. 検証

元のエラーが再現しないことを、**最初に失敗した元のコマンドの再実行**で確認する。

- 「直ったはず」「理屈上は解消している」で終わらせない。
- 別のコマンドが通っただけでは検証にならない（例: `npm install` が通っても
  `npm run dev` が元の失敗コマンドならそちらを再実行する）。
- 再実行で別のエラーが出た場合は、新しいエラーとしてフロー 1 に戻る（同一エラーの再試行とは区別する）。

### 5. 手順書へ反映

対象リポジトリの README / `docs/` / setup スクリプト / CONTRIBUTING を探し、
今回のエラーが**手順の不備・記述の古さに起因するか**を判定する。

- 起因する場合: 該当箇所の修正 diff 案を提示する（例: 前提バージョンの更新、
  抜けていた手順の追加、macOS 固有の注意書き）。
- 手順書はあるが今回の件が手順の不備ではない場合（一時的な環境の壊れ等）:
  トラブルシューティング節への追記候補として提案する（追記する価値があるかの判断はユーザーに委ねる）。
- 手順書が無い場合: 最小限のセットアップ手順 + トラブルシューティング節の新規追加を候補として提案する。
- **対象リポジトリへの書き込みは必ずユーザー確認後。** diff 案の提示までは自動で行うが、
  ファイルへの適用・コミットは指示を待つ。

### 6. 対応記録

試したこと・効かなかったこと・最終的な解決策を短くまとめてチャットで提示する
（jarvis-worklog に拾われる形。別ファイルへの保存はしない）。

- 記録する内容: エラーの要旨 / 原因と根拠 / 試して効かなかった手 / 効いた手 /
  手順書に反映した（提案した）内容。
- 「効かなかった手」を省略しない。次に同種のエラーに当たったとき、無駄な再試行を省ける。

## 頻出原因チェックリスト

原因カテゴリごとの典型症状・確認コマンド・修復手・手順書反映の観点は
[references/common-causes.md](references/common-causes.md) にまとめてある。
診断（フロー 2）の前に必ず読む。新しい原因パターンや macOS 固有の落とし穴を見つけたら、
references への追記で育てる（SKILL.md 本文は変えない）。

## 品質・安全性

- **破壊的・不可逆な操作は必ず事前確認する。** `rm -rf`（プロジェクト外・生成物以外）、
  グローバルなバージョン切り替え、brew の一括アップグレード、設定ファイルの上書きが該当する。
  診断に必要ない「ついで」の環境変更（未使用ツールの導入、無関係なパッケージの更新）はしない。
- **同じ失敗コマンドを漫然と再試行しない。** 再試行するのは環境を変えた後だけ。
  3 回切り分けて進展がなければ、それまでの仮説と結果を整理してユーザーに相談する。
- **機密情報を診断出力に転記しない。** `.env` の中身・トークン・接続文字列は、
  確認が必要でも「変数 X が未設定 / 設定済み」の形で報告し、値そのものは書かない。
- **手順書修正は「自分の環境で確認できた事実」の範囲で書く。** 他 OS・他アーキテクチャへの
  影響は、検証していなければ「(推測)」と明記する。未検証の手順を断定形で手順書に書かない。
- 対象リポジトリへの書き込み（手順書修正の適用）は必ずユーザー確認後。
  修復操作もシステム状態を変えるものは事前確認する（フロー 3 の区分どおり）。
- コード起因と判明した時点で本スキルは手を引き、その旨を伝える。バグ修正・コードレビューは
  通常の開発作業 / `arc-reactor-code-review` の領分で、混ぜると責務が曖昧になる。
