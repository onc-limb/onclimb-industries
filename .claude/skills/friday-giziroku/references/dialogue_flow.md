# Phase B 対話フロー（確認 4 点）

このスキルの核。匿名話者・誤変換・会議種別の不確実性を入口で潰す。
**質問は 4 点に絞る**（質問過多での離脱を防ぐ）。各質問は推定値を初期選択にして、
ユーザーが「そのままでよければ即決、違えば直す」だけで済むようにする。

すべて `AskUserQuestion` で聞く。一度に複数質問をまとめて出してよい。

## 対話の作法（[[problem-essence-organizer]] のスタイルを踏襲）

- 全肯定しない。推定の根拠（発言量・抜粋・glossary/roster 一致）を 1 行添える。
- ユーザーが持っている知識（誰が誰か・正しい固有語）を引き出し、機械が清書を担う分担にする。
- 確定した内容は復唱せず、そのまま Phase C に進める（往復を増やさない）。

## 確認 1: 会議種別

出力テンプレを決める。Phase A の話者数・タイムスパン・口調から推定して初期選択にする。

- `meeting`（会議）: 複数話者・議題あり・決定/TODO が出る想定。
- `informal_decision`（口頭決定）: 会議体ではない口頭のやり取り（仕事上の雑談・立ち話含む）で
  何かが決まったもの。議題は無いが決定/TODO は出る。「経緯 → 決まったこと → TODO」の軽量構成。
- `agent_call`（AI エージェントとの電話）: 本人 + エージェントの 2 者、Q&A 構造。
- `chat`（雑談）: 議題が無く脱線主体。決定が薄い前提。

`informal_decision` と `chat` の境界: 決定/TODO を残すことが主目的なら `informal_decision`、
話の記録が主目的で決定が薄いなら `chat`。迷ったら初期選択を `informal_decision` にして確認する。

## 確認 2: 参加者マッピング ★最重要

話者ごとに **発言量（回数・文字数）＋特徴的な抜粋** を提示し、誰なのか・役割を確定する。

- 選択肢は `roster.yaml` の登録者から推定した候補 ＋「自由入力」＋「匿名のまま」。
- 役割（PM / 実装 / 広報窓口 / 報告先 等）も一緒に取る。後で「誰が何を引き受けたか」を辿るため。
- **話者分離が `estimated` の場合**は、暫定の切り分け（`推定話者A/B…`）が妥当かを必ず確認する。
  「この発言群は同一人物では？」という統合や、「ここで話者交代では？」という分割を許可する。
- マッピングが確定したら `roster.yaml` への追記候補にする（横断共有資産が育つ）。

提示フォーマット例:
```
Speaker 1: 発言30回 / 4130字 / 00:00〜19:06
  「執行役員向け資料の様式を合わせたい」「水曜にドラフトを見せる」→ 進行役/PM らしい
Speaker 2: 発言5回 / 458字 / 00:00〜02:04
  「BM 脆弱性診断の指摘に対応」「CloudFront の設定を更新」→ 実装担当らしい
```

## 確認 3: 公開範囲（マスキング要否）

- `internal`（社内限定）: マスキング最小（明確な秘匿情報のみ）。
- `external`（社外共有可）: 実名・顧客名・固有名を `<REDACTED:種別>` 化（[[masking]]）。
- `unknown`（未確認）: **安全側で external 扱い**にしてマスキングする。

## 確認 4: 会議の目的

Phase A で文字起こしから推定した目的案を提示し、補正してもらう。
推定案は「全体要約の前段」になるので、ここがズレると議事録全体がズレる。1 行で確定する。

## 確認後に内部で決めること（質問しない）

- 会議タイトル: 目的・主要議題から命名（ファイル名に使う）。違和感があれば最後に一度だけ確認。
- 日時: 文字起こしのファイル名・先頭メタ・タイムスタンプから採る。
  **日付が採れない場合のみ**、Phase B の確認バッチに「会議日はいつか」を 1 問追加して確定する
  （当日日付で埋めない。録音日と議事録作成日は一致しないことがある）。
