# tech-debt-auditor — 技術負債の監査 (arc-reactor)

対象リポジトリを負債カテゴリ別にスキャンし、**根拠付き・優先度付きの負債リスト**を
Markdown レポートとして生成するスキル。lint 警告の羅列ではなく、
「誤検知が少なく、そのままリファクタ提案や issue 起票に使えるリスト」を作ることが目的。

> 大原則: **証拠なくして断定なし。** 独立した証拠が 2 つ以上ある項目だけを「確定」とし、
> 確証がないものはすべて「要確認」に分類する。全項目に `ファイル:行` の根拠を必ず添える。

persona: [`personas/arc-reactor.md`](../../../personas/arc-reactor.md) に従う
（応答は日本語 / コード・識別子は英語 / 推測は ASSUMPTION 明記）。

## トリガー

| ユーザー発話の例 | 動作 |
|---|---|
| 「このリポジトリの技術負債を洗い出して」 | リポジトリ全体を全カテゴリで監査 |
| 「技術負債を監査して優先度を付けて」 | 監査 + 優先度マトリクス |
| 「負債の棚卸しをして」 | リポジトリ全体を監査 |
| 「`src/legacy` 配下の負債候補を調べて」 | スコープを限定して監査 |
| 「依存の老朽化だけ見て」 | カテゴリを限定して監査 |
| 「tech debt audit して」 | リポジトリ全体を監査 |

ユーザーが明示的に依頼したときだけ起動する（コードを読んだついでに自動起動しない）。

## 標準フロー

### 1. スコープの確定

- 対象リポジトリの絶対パスを確定する（発話 / cwd。`projects/<name>/` 配下が典型）。
- git リポジトリか確認する（`git -C <repo> rev-parse`）。非 git なら履歴系の証拠なしで縮退動作する旨を伝える。
- 規模を概算する（`git ls-files | wc -l`、総行数）。大規模（目安 10 万行超）なら
  ディレクトリ単位への分割監査を提案する。
- **除外パスを確定する**: `node_modules/`, `vendor/`, `dist/`, `build/`, `.git/`,
  自動生成物（`*_pb.*`, `*.gen.*`, lock ファイル本体等）は既定で除外。
  除外一覧はレポート末尾に明示する。判断に迷うパスはユーザーに確認する。

### 2. カテゴリ別スキャン

[references/debt-categories.md](references/debt-categories.md) の判定基準に従い、
カテゴリごとに `rg` / `git log` / ロックファイル読解でシグナルを収集する。カテゴリ:

1. debt-marker（TODO / FIXME / HACK / XXX 等の負債マーカー）
2. oversized（肥大ファイル・肥大関数）
3. duplication（コピペ・重複ロジック）
4. dependency（依存の老朽化・EOL・非推奨パッケージ）
5. test-gap（テスト欠落）
6. deprecated-api（非推奨 API・レガシーイディオムの使用）
7. dead-code（デッドコード疑い）

ユーザーがカテゴリを限定した場合はそれに従う。

### 3. 証拠収集と確定 / 要確認の分類（誤検知抑制の中核）

候補 1 件ごとに、**独立した証拠**（シグナルの種類が異なるもの）を集める。証拠の例:

- 静的シグナル: マーカー本文、行数・ネスト、重複箇所、非推奨 API の使用行
- 履歴シグナル: `git log --follow` による churn（変更頻度）、最終変更からの経過期間、
  fix/revert コミットの集中
- 構造シグナル: 参照ゼロ（デッドコード）、テストファイルの不在、依存グラフ上の位置
- メタデータシグナル: ロックファイル内のバージョンの古さ、既知の EOL / deprecation 情報

分類ルール（**厳守**）:

- **確定 (Confirmed)**: 独立した証拠が **2 つ以上**揃った場合のみ。
- **要確認 (Needs Review)**: 証拠が 1 つ以下、または反証の可能性が残る場合。
  断定表現を使わず「〜の可能性がある」と書き、確認方法を 1 行添える。
- **確定・要確認を問わず、全項目に根拠 `ファイル:行`（依存は `ロックファイル:行` または依存名+バージョン）を必ず添える。**
  根拠を提示できない項目はレポートに載せない。
- dead-code / duplication は誤検知リスクが特に高いため、
  references の定めるとおり**単独証拠では絶対に確定にしない**。
- 意図的な設計の痕跡（`NOTE:` コメント、ADR、テストで意図が固定されている等）を見つけたら、
  それは**反証**として扱い、確定から要確認へ降格または除外する。

### 4. 優先度付け（影響 × 修正コスト）

**確定項目のみ**に優先度を付ける（要確認項目は事実確認が先なので優先度を付けない）。

- **影響 (Impact)**: High = 障害・セキュリティ・開発速度に直結（高 churn 箇所、コアパス）/
  Mid = 特定機能の保守性を下げる / Low = 局所的・美観
- **修正コスト (Cost)**: S = 1 ファイル・半日以内 / M = 複数ファイル・数日 / L = 設計変更を伴う

| | Cost S | Cost M | Cost L |
|---|---|---|---|
| **Impact High** | **P1** | **P1** | **P2** |
| **Impact Mid** | **P2** | **P3** | **P3** |
| **Impact Low** | **P3** | **P4** | **P4** |

各項目に影響・コストの判定根拠を 1 行ずつ添える（例: 「直近 90 日で 14 commit の高 churn」）。

### 5. レポート生成

以下の固定構成で Markdown レポートを生成し、チャットで要約を提示する。
保存する場合は対象プロジェクト配下 `<repo>/tech-debt-audit/<YYYY-MM-DD>.md` に置く
（対象リポジトリの git を汚す可能性があるため、**保存前に必ずユーザーに確認**する）。

```markdown
# Tech Debt Audit — <repo-name> (<YYYY-MM-DD>)

## サマリ
- スコープ / 除外パス / 対象カテゴリ / 規模
- 確定 N 件（P1: n, P2: n, P3: n, P4: n）/ 要確認 M 件

## 確定 (Confirmed)
| # | 優先度 | カテゴリ | 場所 (file:line) | 内容 | 証拠 (2 つ以上) | 影響 | コスト |
|---|---|---|---|---|---|---|---|

## 要確認 (Needs Review)
| # | カテゴリ | 場所 (file:line) | 内容(推定) | 証拠 (1 つ以下) | 確認方法 |
|---|---|---|---|---|---|

## 除外・免責
- 除外したパス一覧
- 本監査は静的シグナルベースであり、負債の全量ではない（設計の歪み・暗黙知は対象外）
- 依存の鮮度判定はローカル情報と既知情報の範囲（ネット照会なし）
```

### 6. 完了提示

- 確定 / 要確認の件数と P1 項目の要点をチャットで要約する。
- 次アクション（P1 の issue 起票、要確認項目のヒアリング、`friday` 系スキルでの提案資料化など）を提案する。

## 負債カテゴリと判定基準

カテゴリごとの詳細な判定基準・証拠の種類・確定条件・除外条件は
[references/debt-categories.md](references/debt-categories.md) を参照する。
スキャン実行前に必ず読むこと。判定基準の改善（新カテゴリ・反証パターンの追加）は
references への追記で行う。

## 品質・安全性

- **誤検知の抑制を最優先とする**。「量より確度」— 疑わしい 50 件より確かな 10 件。
  リストの信頼が失われると監査自体が読まれなくなるため。
- 確証がないものは必ず「要確認」に分類し、断定しない。証拠 1 つで「確定」に置かない。
- 根拠 `ファイル:行` のない項目は出力しない（ユーザーが即座に検証できることが信頼の条件）。
- 対象リポジトリへの操作は**読み取りのみ**。コードの修正・コミットは本スキルの範囲外
  （修正はユーザーの指示を受けて別途行う）。レポート保存時のみ書き込み、保存前に確認する。
- レポートはプロジェクト固有情報のため、対象プロジェクトのディレクトリ配下に閉じる
  （当リポジトリ側や他プロジェクトに持ち出さない。CLAUDE.md の分離ルールに従う）。
- 機密情報（コード中の API キー等）を発見しても、レポートに**値を転記しない**。
  場所の指摘のみ行う（これは負債ではなくセキュリティ問題として最優先で報告する）。
- ネットワークアクセスは行わない。依存の鮮度は ロックファイル + 既知の EOL 情報の範囲で判定し、
  その制約をレポートの免責に明記する。
