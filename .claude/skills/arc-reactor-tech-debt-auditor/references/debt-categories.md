# 負債カテゴリと判定基準

tech-debt-auditor がスキャンする負債カテゴリの定義。カテゴリごとに
**シグナル（何を検出するか）/ 証拠（何を根拠にするか）/ 確定条件 / 反証・除外条件** を定める。

共通ルール:

- 「確定」は**独立した証拠 2 つ以上**が揃った場合のみ。証拠 1 つ以下は「要確認」。
- 独立性の判定: シグナルの**種類**が異なること（静的 / 履歴 / 構造 / メタデータ）。
  同種のシグナルを何個集めても証拠 1 つと数える（例: TODO が 3 個あっても静的シグナル 1 つ）。
- 全項目に `ファイル:行` を添える。行を特定できない項目（依存等）は `ロックファイル:行` または
  依存名 + バージョンを添える。
- 反証（意図的である痕跡）を見つけたら、確定から要確認へ降格するか、リストから除外する。

## 1. debt-marker — 負債マーカー

- **シグナル**: `rg -n "TODO|FIXME|HACK|XXX|WORKAROUND|kludge" --type-add` 等でのマーカー検出。
- **証拠の例**:
  - 静的: マーカー本文が問題を具体的に述べている（「後で直す」だけでなく何が問題か書いてある）
  - 履歴: マーカー追加コミットから長期間経過（`git log -S` で追加時期を特定。目安 6 か月超）
  - 履歴: マーカー周辺行に fix コミットが繰り返し当たっている
- **確定条件**: 具体的な問題記述 + 長期滞留（または高 churn）の 2 つ以上。
- **反証・除外**: issue 番号付きで追跡されているマーカー（`TODO(#123)`）は管理済みとして除外可。
  仕様メモとしての `NOTE:` は負債ではない。

## 2. oversized — 肥大ファイル・肥大関数

- **シグナル**: 行数の外れ値（そのリポジトリ内の分布で判定。絶対値の目安: ファイル 800 行超、
  関数 80 行超だが、言語・リポジトリの平均に合わせて調整する）。
- **証拠の例**:
  - 静的: 行数・ネスト深さ・責務の混在（複数ドメインの import が同居）
  - 履歴: 高 churn（直近 90 日で多数のコミットが集中。`git log --since=90.days --name-only` を集計）
  - 履歴: 複数人のコミットが衝突気味に入っている（コンフリクトの温床）
- **確定条件**: サイズの外れ値 + 高 churn（または責務混在）の 2 つ以上。
  **サイズ単独では確定にしない**（正当に長い定義ファイル・テーブル類があるため）。
- **反証・除外**: 自動生成ファイル、定数テーブル・辞書データ、変更がほぼない安定ファイル
  （サイズが大きくても触らないなら負債の優先度は低い）。

## 3. duplication — 重複ロジック

- **シグナル**: 類似の関数名・同一処理のコピペ（同名関数の複数定義、特徴的な文字列 / 定数の重複を
  `rg` で突き合わせ）。
- **証拠の例**:
  - 静的: 実質同一のコードブロックが 2 箇所以上（差分が変数名程度）
  - 履歴: 片方だけ修正されて他方が置き去りになった形跡（同じ fix が別時期に別箇所へ入っている）
- **確定条件**: 実質同一ブロックの提示（両方の `file:line`）+ 置き去り修正の形跡、の 2 つ。
  **⚠ 誤検知高リスク: 類似コードの単独発見では絶対に確定にしない。**
  意図的な重複（レイヤー分離のための DTO、独立性を優先したテストコード）が多いため。
- **反証・除外**: テストコード同士の類似、レイヤー境界を守るための意図的な二重定義、
  生成コード。

## 4. dependency — 依存の老朽化

- **シグナル**: ロックファイル / マニフェスト（`package-lock.json`, `go.sum`, `requirements.txt`,
  `Gemfile.lock` 等）内のバージョンを読み、メジャーバージョンの遅れ・既知の EOL・
  アーカイブ済み / 非推奨宣言済みパッケージを特定する。
- **証拠の例**:
  - メタデータ: 既知の EOL / deprecation（例: サポート終了が公表済みのランタイム・パッケージ）
  - 静的: コード側でそのパッケージの非推奨 API を使用している（`file:line` を添える）
  - 履歴: 依存更新コミットが長期間ない（lock ファイルの当該行が長期未変更）
- **確定条件**: 既知の EOL / 非推奨 + コード側の使用箇所提示、など 2 つ以上。
- **制約**: **ネットワーク照会は行わない**。最新バージョンとの差は Claude の知識の範囲で判定し、
  知識が古い可能性をレポートの免責に明記する。知識で判断できないものは「要確認」+
  「`npm outdated` 等での確認方法」を添える。
- **反証・除外**: 意図的なバージョン固定（コメントや ADR でピン留め理由が説明されている場合）。

## 5. test-gap — テスト欠落

- **シグナル**: 実装ファイルに対応するテストファイルの不在（命名規約 `*_test.*`, `*.spec.*`,
  `test_*.py` 等から対応を推定）。
- **証拠の例**:
  - 構造: コアロジック（ドメイン層・金額計算・認証等）にテストファイルが対応しない
  - 履歴: 当該ファイルに fix / revert コミットが繰り返し入っている（テスト不在 × 高バグ密度）
  - 静的: 分岐が多い・例外処理が多いのに検証がない
- **確定条件**: テスト不在 + （高 churn / fix 集中 / 分岐複雑）のいずれか、計 2 つ以上。
  **テスト不在単独では確定にしない**（薄いグルーコード・設定ファイルはテスト不要なことが多い）。
- **反証・除外**: E2E / 統合テストで実質カバーされている場合（テストディレクトリ全体を確認してから
  判定する）、宣言的な設定・DI 配線のみのファイル。

## 6. deprecated-api — 非推奨 API・レガシーイディオム

- **シグナル**: 言語 / FW の公式に非推奨とされた API・パターンの使用を `rg` で検出
  （例: 各言語の deprecated 標準 API、FW の旧世代 API、`@Deprecated` / `@deprecated` 注釈付き
  自作 API の呼び出し残り）。
- **証拠の例**:
  - 静的: 使用箇所の `file:line`
  - メタデータ: 公式の非推奨宣言（Claude の知識で確認できるもの。バージョン・出典を書く）
  - 構造: 同一リポジトリ内に新 API への移行が始まっており、旧 API が混在している
- **確定条件**: 使用箇所 + 公式非推奨の根拠、の 2 つ。非推奨か記憶が曖昧なものは「要確認」とし、
  公式ドキュメントでの確認を促す。
- **反証・除外**: 互換レイヤーとして意図的に残されている箇所（コメントで明示されている場合）。

## 7. dead-code — デッドコード疑い

- **シグナル**: リポジトリ内で参照が見つからない関数・クラス・ファイル、
  常に false の feature flag、コメントアウトされた大きなブロック。
- **証拠の例**:
  - 構造: `rg` でシンボル参照ゼロ（定義箇所以外にヒットなし）
  - 履歴: 長期間（目安 1 年超）変更なし + 呼び出し元が過去のコミットで削除されている
  - 静的: 「使わなくなった」旨のコメント、コメントアウトの塊
- **確定条件**: 参照ゼロ + 呼び出し元削除の履歴、など 2 つ以上。
  **⚠ 誤検知最高リスク: 単独証拠では絶対に確定にしない。常に「要確認」始まり。**
  動的参照（リフレクション、文字列ベースのルーティング、DI 設定、外部からのエントリポイント、
  公開ライブラリの API）は静的検索に映らないため。
- **反証・除外**: エクスポートされた公開 API（外部利用者がいる可能性）、
  文字列で参照されうるシンボル（`rg "\"symbol_name\""` で文字列参照も必ず確認する）、
  マイグレーション・スクリプト類。
- **要確認時に添える確認方法の例**: 「動的参照の有無をオーナーに確認」「カバレッジ計測で実行実績を確認」。

## カテゴリの追加・基準の改善

- 監査中にユーザーから「これは負債ではない」と却下された項目が出たら、
  該当カテゴリの**反証・除外条件に却下理由を追記**して基準を育てる（手動運用。自動書き換えはしない）。
- 新カテゴリ（例: 設定と実装の乖離、ドキュメント腐敗）を足す場合も本ファイルに
  同じ構成（シグナル / 証拠 / 確定条件 / 反証）で追記する。
