# sequence-diagram-generator — 処理フローのシーケンス図生成 (arc-reactor)

指定された起点からコードを追跡し、処理フローの Mermaid シーケンス図を 1 枚生成するスキル。
「この処理どう流れてる？」という問いに、口頭説明・引き継ぎ・調査メモに
そのまま貼れる図で答えることが目的。

> 大原則: **実際に読んだコードだけを図にする。** もっともらしい「一般的なフロー」を
> 創作しない。追えなかった部分は追えなかったと書き、図中で `(要確認)` として
> 確認済みの経路と区別する。

persona: [`personas/arc-reactor.md`](../../../personas/arc-reactor.md) に従う
（応答は日本語 / コード・識別子は英語 / 推測は ASSUMPTION 明記）。

## トリガー

| ユーザー発話の例 | 動作 |
|---|---|
| 「この処理の流れをシーケンス図にして」 | 起点を確認して追跡・図生成 |
| 「POST /orders がどう処理されるか図にして」 | HTTP ルートを起点に追跡 |
| 「このバッチの処理フローを Mermaid で描いて」 | CLI / ジョブのエントリポイントを起点に追跡 |
| 「ログインから DB 更新までの流れを追って図にして」 | 画面操作名から起点を解決して追跡 |
| 「この関数から先の呼び出しをシーケンス図で見せて」 | 関数・メソッドを起点に追跡 |
| 「処理の流れを引き継ぎ用に図解して」 | 追跡 + 図生成（保存を提案） |
| 「さっきの図、リトライ処理まで詳細化して」 | 既出の図を指定粒度で再生成 |

ユーザーが明示的に依頼したときだけ起動する（コードを読んだついでに自動起動しない）。

入力: 起点の指定（HTTP ルート / URL、関数・メソッド名、CLI コマンド、イベントハンドラ、
画面の操作名のいずれか）＋対象リポジトリ＋任意で深さ・粒度の指定。

## 標準フロー

### 1. 起点の解決

- 発話からエントリポイントのコード位置を特定する。ルーティング定義・ハンドラ登録・
  CLI のコマンド定義・イベント購読の登録箇所を `rg` で検索し、`file:line` まで確定する。
- 画面の操作名で指定された場合は、フロントエンドのハンドラ → 呼び出す API →
  サーバ側ルートの順に対応付ける。
- 候補が複数ある・曖昧な場合は、候補を `file:symbol` 付きで提示してユーザーに確認する。
  推測で 1 つに決めて追跡を始めない。

### 2. 追跡

- 起点から呼び出しを**実コードで**追う（`rg` と読解。フレームワークのドキュメント知識で
  経路を補完しない）。各ホップで「どのファイルのどのシンボルが、次に何を呼ぶか」を記録する。
- **登場する外部 I/O（DB クエリ・外部 API 呼び出し・キュー・キャッシュ）は必ず図中で
  強調する**。SES 参画者がまず知りたいのはシステムの境界であり、外部 I/O の見落としは
  図の価値を大きく損なうため。強調の記法は
  [references/mermaid-notation.md](references/mermaid-notation.md) に従う。
- 分岐条件・エラーパス・トランザクション境界（begin / commit / rollback の範囲）は
  追跡中にメモし、図の `alt` / `opt` ブロックと補足の箇条書きに反映する。

### 3. 粒度の調整

- 既定の粒度は「層・コンポーネント単位」。参加者 (participant) は
  層（Controller / Service / Repository 等）・コンポーネント・外部システムとし、
  1 関数 1 矢印の羅列で図を潰さない。
- ユーザーが「もっと詳しく」「この部分を関数単位で」と指定したら、その範囲だけ詳細化する。
- 深さの指定（「2 ホップ先まで」等）があればそれに従い、打ち切った先は
  `...` と省略した旨を図中に示す。

### 4. 図の生成と検証

- 出力は 2 点セット:
  - (a) Mermaid `sequenceDiagram`。各矢印のメッセージに対応する実装を示すため、
    図とは別に `file:symbol` の対応表（ステップ番号 → 根拠 `file:line`）を添付する。
  - (b) 補足の箇条書き: 分岐条件・エラーパス・トランザクション境界。
- 分岐が主役の処理（状態機械・条件分岐が本体のバリデーション等）は、
  references の使い分け基準に従い `flowchart` を選んでよい。選んだ理由を 1 行添える。
- **Mermaid 構文が壊れていないかを確認してから渡す**（レンダリング不能な図を渡さない）。
  `mmdc`（mermaid-cli）が使える環境ならパースを実行し、無ければ
  references の構文チェックリストで目視検証する。

### 5. 限界の明示

- 動的ディスパッチ・DI コンテナ・リフレクション・イベント購読など、静的に追えなかった
  箇所は、図中に `(要確認)` の注釈を付けて省略・推測と区別する。
- 推測でつないだ経路がある場合は、矢印のメッセージに `(要確認)` を付けたうえで、
  補足に「何を確認すれば確定するか」（例: DI の設定ファイル、実行時ログ）を 1 行書く。

### 6. 提示と保存

- 図・対応表・補足をチャットで提示する。チャット提示だけで完結してよい。
- ユーザーが保存を望む場合は、対象プロジェクト配下 `<repo>/docs/flows/` 等に
  Markdown で保存する（対象リポジトリの作業ツリーに書き込むため、
  **保存前に必ずユーザーに確認**する）。

## Mermaid 記法の使い分け

sequenceDiagram / flowchart の使い分け基準、外部 I/O の強調記法、
壊れやすい構文のチェックリストは
[references/mermaid-notation.md](references/mermaid-notation.md) を参照する。
図の生成前に必ず読むこと。記法の改善（新しい強調パターン・構文の落とし穴の追加）は
references への追記で行う。

## 品質・安全性

- 対象リポジトリへの操作は**読み取りのみ**。図・レポートの保存時のみ書き込み、保存前に確認する。
- 図とレポートはプロジェクト固有情報のため、対象プロジェクトのディレクトリ配下に閉じる
  （当リポジトリ側や他プロジェクトに持ち出さない。CLAUDE.md の分離ルールに従う）。
- 巨大なフロー（追跡が 3 ホップ以上の分岐爆発を起こす等）になったら、全部を 1 枚に
  詰め込まず、フェーズ単位・分岐単位での分割を提案する。読めない図は渡さない。
- 根拠 `file:line` を示せない矢印は図に入れない（`(要確認)` 注釈付きで残す場合を除く）。
- コード中に機密情報（API キー・接続文字列等）を見つけても、図・対応表・補足に
  **値を転記しない**。参加者名・環境変数名の参照に留める。

## 既存スキルとの棲み分け

- リポジトリ全体の地図とオンボーディング資料一式は `arc-reactor-codebase-onboarding`
  （あちらはユースケースのフロー追跡を含むが、資料一式の永続化が目的。
  本スキルは単一フローの図 1 枚を素早く出すのが目的）。
- 外部アクセスの台帳化は `arc-reactor-external-access-mapper`。図中で強調した外部 I/O を
  台帳として横断整理したい場合に併用できる。
