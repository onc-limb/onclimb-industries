# personal-budget-manager — 個人の資産（現金）管理スキル

レシート・領収書・領収書メール・固定費 config から、**個人の家計の月次収支（収入 − 固定費 − 変動費）**を整理する。
対象はフリーランスエンジニア本人の**私的な家計**であり、事業の帳簿でも家族の共同財布でもない。

> **スタンス（毎回明示）**: 本スキルの出力は**家計情報の機械的な整理であって、金融助言・投資助言・節約指導ではない**。
> レシートや本文から読み取った金額は誤読の可能性があるため、**必ずユーザー確認を挟んでから**記録する。
> 支出の良し悪しの評価はせず、事実（いくら・何に・いつ）の整理に徹する。

## 棲み分け（重要）

- **`ultron-tax-prep-organizer`**: 事業経費の確定申告整理（勘定科目・家事按分・青色申告）。**事業側**。
- **`ultron-family-budget-manager`**: 家族のお金管理（夫婦の共同支出のレシート記録と月次集計。割り勘精算用）。**家族側**。
  共同支出のレシートはそちらのみに入れ、本スキルのエントリと二重にしない。
- **本スキル**: 個人の資産（現金）管理（食費・日用品などの私的支出 + 生活収支）。**個人側**。
- レシートに事業経費らしき支出（開発用書籍・サーバ代等）を見つけたら、その旨を伝えて
  tax-prep-organizer での整理を**案内する**だけにとどめ、本スキルのエントリ・集計には混ぜない
  **(ASSUMPTION: 事業/私用の判断はユーザーに委ね、Claude は「事業経費っぽい」の指摘まで)**。
- 請求書作成（`ultron-invoice-builder`）・稼働集計（`ultron-timesheet-aggregator`）は別系統。

## 場所（コードとデータは分離されている）

- ツール本体・参照資料: このスキルディレクトリ `.claude/skills/ultron-personal-budget-manager/`
  （`scripts/` `templates/` `references/`）
- **データ**: スキルが属する git リポジトリ直下の `budget-data/`（**git 管理外**。`.gitignore` 済み）。
  環境変数 `BUDGET_DATA` で場所を上書き可。
  ```
  budget-data/
  ├── config/
  │   ├── fixed-costs.json     # 固定費（家賃・光熱費・通信費・サブスク等）
  │   └── income.json          # 収入（既定の月収 + 月別上書き）
  ├── entries/
  │   └── YYYY-MM.json         # 変動費の月次エントリ（レシート・メール由来の明細）
  ├── inbox/                   # 未処理のレシート・領収書（画像 / PDF）を置く場所
  │   └── processed/           # 取り込み済みファイルの退避先
  └── reports/
      └── YYYY-MM.md           # 月次レポート
  ```
- config の雛形とスキーマは `references/data-schema.md`、レポート雛形は `templates/monthly-report.md`。

## 役割分担（重要）

集計の誤り・捏造を構造的に防ぐため、**読み取り・分類は Claude、金額の合算・収支計算はスクリプト**に分ける。

- **Claude（あなた）**: レシート画像 / PDF の読み取りと明細 JSON 化、`references/categories.md` に基づく
  カテゴリ分類、Gmail 本文からの明細抽出、ユーザーへの確認対話、スクリプト出力の転記と講評。
- **スクリプト（決定論）**: `scripts/aggregate.py` が カテゴリ別合計・固定費/変動費/収入の合算・
  収支（収入 − 支出）・重複疑い検出・前月比 をすべて計算する。

→ **Claude は合計を暗算しない**。レポート・チャット提示の数値はすべて `aggregate.py` の出力を転記する。
→ 読み取った金額・店名・日付は**ユーザー確認を経てから**エントリに追記する（確認前に書き込まない）。

## トリガーと対応フロー

| ユーザー発話の例 | 実行するフロー |
|---|---|
| 「このレシート読み込んで」「inbox のレシート全部処理して」 | フロー① レシート取り込み |
| 「Gmail から領収書メールを拾って」「領収書メール取り込んで」 | フロー② Gmail 取り込み（MCP 接続時のみ） |
| 「今月の家計をまとめて」「今月の収支どうなってる？」 | フロー③ 月次収支（未処理 inbox があれば①を先に提案） |
| 「先月と比べて食費どうなってる？」 | フロー③（前月比の該当箇所を提示） |
| 「固定費を更新して」「サブスク解約したから消して」 | config 更新（fixed-costs.json を対話で編集 → 変更内容を復唱確認） |
| 「収入変わった」「今月は臨時収入があった」 | config 更新（income.json の default / overrides を編集） |

いずれも**ユーザーが明示的に依頼したときだけ**起動する（自動起動しない）。

## フロー① レシート取り込み

```bash
SKILL=/Users/satoshi-onga/Documents/onclimb-industries/.claude/skills/ultron-personal-budget-manager
DATA=${BUDGET_DATA:-<repo>/budget-data}   # 無ければディレクトリ一式を作成
```

1. `"$DATA/inbox/"` の未処理ファイル（画像 / PDF。`processed/` は除く）を列挙する。
2. 各ファイルを **Read ツールで読み取り**、明細 JSON に起こす（スキーマは下記・詳細は `references/data-schema.md`）:
   ```json
   {"id": "e-20260701-001", "date": "2026-07-01", "store": "スーパー○○",
    "amount": 2345, "category": "食費", "payment_method": "credit",
    "source": "receipt:IMG_0123.jpg", "gmail_message_id": null, "memo": ""}
   ```
   - カテゴリは `references/categories.md` の判定基準で付ける。確証がなければ `その他` + memo に理由。
   - 1 レシート内で明確にカテゴリが分かれる場合（例: 食品と日用品）は分割してよいが、
     分割合計がレシート合計と一致することを確認する。迷ったらレシート単位 1 エントリに倒す。
   - `payment_method` は `cash` / `credit` / `e-money` / `bank_transfer` / `unknown`
     **(ASSUMPTION: レシートから判別できなければ unknown)**。
3. **ユーザー確認（必須）**: 読み取った明細の一覧（日付・店舗・金額・カテゴリ）を表で提示し、
   誤読・カテゴリ修正を反映してから確定する。合計印字と明細の不整合があれば明示して確認する。
4. 確定した明細を `"$DATA/entries/YYYY-MM.json"`（エントリの `date` の月）へ追記する。
   既存エントリと同一「日付+金額+店舗」があれば追記前に警告し、ユーザーに二重計上でないか確認する。
5. 取り込み済みファイルを `"$DATA/inbox/processed/"` へ `mv` する（再処理防止）。
6. 「取り込み N 件 / スキップ M 件（理由付き）」を報告する。

## フロー② Gmail 取り込み（MCP 接続時のみ）

**前提（制約）**: 利用可能な Gmail MCP ツールは search_threads / get_thread / list_labels / create_label /
label_message / label_thread / unlabel_message / unlabel_thread / create_draft / list_drafts のみで、
**添付ファイルをダウンロードするツールは存在しない**。詳細な運用は `references/gmail-integration.md`。

0. Gmail MCP ツールが使えないセッション（ヘッドレス等）では、このフローを**スキップ**し
   「Gmail 連携は未接続のためスキップ。レシートは inbox 経由で取り込める」と案内する（縮退動作）。
1. `list_labels` でラベル `家計/取込済` `家計/DL待ち` `家計/要確認` の存在を確認し、無ければ `create_label` で作成する
   **(ASSUMPTION: ラベル名はこの 3 種。`references/gmail-integration.md` に定義)**。
2. `search_threads` で領収書系メールを検索する（クエリは `references/gmail-integration.md` に定義。
   対象期間は原則ユーザー指定の月、省略時は直近 35 日 **(ASSUMPTION)**）。
   既に `家計/取込済` `家計/DL待ち` `家計/要確認` が付いたスレッドは除外する。
3. 各スレッドを `get_thread` で読み、3 タイプに仕分ける:
   - **本文型**（本文に金額・明細がテキストで書かれている。EC の購入確認等）
     → 本文から明細を抽出してフロー①の手順 3〜4 と同様にユーザー確認 → エントリ追記。
       エントリの `gmail_message_id` に message id、`source` に `gmail:<件名要約>` を記録。
       取り込み後 `家計/取込済` ラベルを付ける。
   - **添付型**（PDF 等の添付に領収書がある） → `家計/DL待ち` ラベルを付け、
     「手動でダウンロードして `budget-data/inbox/` に置いてください」と依頼する（金額はまだ記録しない）。
   - **外部サイト型**（リンク先でのダウンロードが必要） → `家計/要確認` ラベルを付ける（金額は記録しない）。
4. **結果報告（必須）**: 「本文から取り込み N 件 / DL待ち M 件 / 要確認 K 件」と、
   DL待ち・要確認の**件名・差出人・日付の一覧**をチャットに提示する。
5. 二重取り込み防止: 取り込み前に対象月エントリの `gmail_message_id` 一覧と突合し、既取り込み ID はスキップする。
   DL待ちの添付をユーザーが inbox に置いてフロー①で取り込んだら、該当スレッドのラベルを
   `家計/DL待ち` → `家計/取込済` に付け替える（`unlabel_thread` + `label_thread`）。

Gmail の操作は検索・読み取り・ラベル付けのみ。**メールの送信・自動返信はしない**（draft 作成もユーザーが明示的に依頼した場合のみ）。

## フロー③ 月次収支

1. 対象月（YYYY-MM。省略時は当月）を確定する。`"$DATA/inbox/"` に未処理ファイルがあれば
   「先に取り込むか」を確認する（未処理分は収支に含まれないため）。
2. config を確認する。`fixed-costs.json` / `income.json` が無ければ
   `references/data-schema.md` の雛形をもとに対話で作成する。
3. **決定論集計**（Claude は合算しない）:
   ```bash
   python3 "$SKILL/scripts/aggregate.py" --month YYYY-MM --data-dir "$DATA" \
     --out "$DATA/reports/YYYY-MM.summary.json"
   python3 "$SKILL/scripts/aggregate.py" --month YYYY-MM --data-dir "$DATA" \
     --markdown --out "$DATA/reports/YYYY-MM.md"
   ```
   スクリプトが計算するもの: 収入合計（月別上書き対応）/ 固定費合計（`active_from`/`active_until` で対象月フィルタ）/
   変動費のカテゴリ別合計 / 収支（収入 − 固定費 − 変動費）/ 重複疑い（同一 日付+金額+店舗、同一 gmail_message_id）/
   前月比（前月のエントリファイルがあれば）/ 警告（未知カテゴリ・不正金額・対象月外の日付）。
4. 生成された `reports/YYYY-MM.md` に、Gmail 連携の残タスク（DL待ち・要確認の一覧）があれば
   `templates/monthly-report.md` の「要確認リスト」節の形式で**追記**する（数値には触らない）。
5. チャットにレポート要点を提示する。数値は**スクリプト出力の転記のみ**。
   重複疑い・DL待ち・要確認が残っている場合は「**収支は未確定**（取り漏れ・二重計上の可能性あり）」と明示する。

## 二重計上防止（まとめ）

- `aggregate.py` が同一「日付+金額+店舗」の 2 件目以降を**重複疑い**として警告する
  （合計には含めたまま警告する。除外判断はユーザー。本当に 2 回買った場合は memo で区別して再実行）。
- Gmail 取り込みは `gmail_message_id` をエントリに記録し、既取り込み ID をスキップ。同一 ID の重複も警告対象。
- 処理済みレシートは `inbox/processed/` へ移動し、同じファイルの再取り込みを防ぐ。
- レシートと Gmail 本文の両方から同じ買い物が入るケースは「日付+金額+店舗」の重複疑いで検出される。

## 出力フォーマット（月次レポート）

`templates/monthly-report.md` に従う。構成:

1. 免責（金融助言ではない旨 — スクリプトが自動で埋め込む）
2. サマリ表: 収入 / 固定費 / 変動費 / 支出合計 / **収支**
3. 変動費のカテゴリ別内訳（金額・件数・前月比）
4. 固定費の内訳 / 収入の内訳
5. 前月比（収入・固定費・変動費・収支の増減）
6. 重複疑い一覧 / 警告
7. 要確認リスト（Gmail DL待ち・要確認 — Claude が追記する節）

## 品質・安全性（persona: ultron）

- **個人の金銭情報は git 管理外**: `budget-data/` は `.gitignore` 済み。店名・金額・購入品目を含むデータを
  スキルディレクトリ（git 管理内）やコミットに持ち込まない。
- **口座番号・カード番号を残さない**: レシートやメールにカード番号（下 4 桁含む）・口座番号・会員番号が
  印字されていても、エントリ・レポート・チャット出力に**転記しない**。`payment_method` の区分のみ記録する。
- **金融助言ではない**: すべての出力（チャット・保存レポート）に免責を明記する。
  「使いすぎ」「〜すべき」といった評価・指導はせず、事実の整理と提示に徹する。
- **読み取った金額は必ずユーザー確認を挟む**: OCR 誤読は前提とし、確認前のエントリ書き込みを禁止する。
- **数値の正確性・再現性**: 合算はすべて `aggregate.py`。同じ入力から同じ結果が出る。
  重複疑い・要確認が残る間は「収支は未確定」と明示し続ける。
- **Gmail は読み取りとラベル付けまで**: 実際のメール操作はユーザーが依頼したフローの範囲でのみ行い、
  本文の全文をレポートに転載しない（明細に必要な項目だけ抽出する）。

## 注意

- カテゴリ一覧・判定基準は `references/categories.md`。カテゴリの追加・変更は本人確認のうえ
  **手動で** references とスクリプトの既知カテゴリ定数を揃えて更新する（自己進化パイプラインは持たない）。
- 現金支出などレシートが無いものは、ユーザーの口頭申告からエントリを起こしてよい
  （`source: "manual"`）。この場合も確認・復唱を挟む。
- クレジットカードの引き落とし明細は取り込まない **(ASSUMPTION: レシート/メール由来の購入時点計上と
  二重になるため。カード明細ベースの運用に切り替えたい場合は本人と設計を見直す)**。
- 立替や返金（マイナス収支）は `amount` を負にせず、返金は収入側 overrides で扱うか memo 付きで
  ユーザーと相談して決める **(ASSUMPTION: 初版では返金の頻度が低い想定で仕様を簡素に保つ)**。
