# Gmail 連携の運用（検索クエリ・ラベル・仕分けルール)

Gmail MCP 接続時のみ実行するフロー②（SKILL.md 参照）の詳細定義。

## 前提となる MCP の制約

利用可能な Gmail MCP ツールは以下のみ:

`search_threads` / `get_thread` / `list_labels` / `create_label` / `label_message` / `label_thread` /
`unlabel_message` / `unlabel_thread` / `create_draft` / `list_drafts`

- **添付ファイルをダウンロードするツールは存在しない**。したがって PDF 添付の領収書は
  MCP 経由では取り込めず、ユーザーの手動ダウンロードに委ねる（下記「添付型」）。
- メールの送信はできない（draft 作成のみ。ユーザーが明示的に依頼した場合に限る）。
- MCP 未接続のセッション（ヘッドレス等）では Gmail 連携を**丸ごとスキップ**し、
  レシート取り込み（フロー①）と月次収支（フロー③）のみで動作する（縮退動作）。

## ラベル定義 **(ASSUMPTION: ラベル名は以下の 3 種で運用開始。変更する場合はここを更新)**

| ラベル | 意味 | 付けるタイミング |
|---|---|---|
| `家計/取込済` | 本文から明細を抽出してエントリに取り込み済み | 取り込み完了直後 |
| `家計/DL待ち` | PDF 等の添付に領収書があり、手動ダウンロード待ち | 添付型と判定した時点 |
| `家計/要確認` | リンク先（外部サイト）でのダウンロード・確認が必要 | 外部サイト型と判定した時点 |

- 初回実行時に `list_labels` で存在確認し、無ければ `create_label` で作成する。
- DL待ちの添付がユーザーによって `budget-data/inbox/` に置かれ、フロー①で取り込んだら、
  該当スレッドを `unlabel_thread`（家計/DL待ち）→ `label_thread`（家計/取込済）に付け替える。

## 検索クエリ **(ASSUMPTION: 初版のクエリ。取り漏れ・ノイズに応じて手動で調整)**

基本形（対象期間はユーザー指定の月。省略時は `newer_than:35d`）:

```
subject:(領収書 OR レシート OR receipt OR 請求 OR ご購入 OR ご注文 OR 購入完了 OR お買い上げ) newer_than:35d
```

補助クエリ（基本形で拾えない添付型の取りこぼし対策。必要に応じて追加実行):

```
has:attachment filename:pdf (領収書 OR 請求書 OR receipt OR invoice) newer_than:35d
```

- 処理済み除外: 検索結果のうち、既に `家計/取込済` / `家計/DL待ち` / `家計/要確認` の
  いずれかが付いたスレッドはスキップする（クエリに `-label:` を足すより、結果側での除外を優先。
  ラベル名に日本語・スラッシュを含むためクエリ埋め込みの挙動が不安定な場合がある **(ASSUMPTION)**）。
- 明らかに家計と無関係なもの（メルマガ・事業側の請求書・invoice-builder が発行した請求関連）は取り込まない。
  事業側の領収書らしきものは `ultron-tax-prep-organizer` の領分として案内だけする。

## 3 タイプの仕分け基準

`get_thread` で本文を読み、以下の順で判定する:

1. **本文型**: 本文に購入日・金額（合計）がテキストで明記されている（EC の購入確認・注文確認・
   デジタルサービスの決済通知等）。→ 本文から `{date, store, amount, category, payment_method}` を抽出し、
   ユーザー確認のうえエントリへ追記。`gmail_message_id` に message id を記録し、`家計/取込済` を付ける。
   - `store` は販売者名（例: Amazon の場合はマーケットプレイス出品者ではなくサービス名 **(ASSUMPTION)**）。
   - `payment_method` は本文から判別できなければ `unknown`。
   - 本文に明細と合計の両方があり不一致の場合は取り込まず、ユーザーに提示して確認する。
2. **添付型**: 本文に金額の記載がなく（または不十分で）、PDF 等の添付に領収書本体がある。
   → 取り込まず `家計/DL待ち` を付け、件名・差出人・日付を一覧報告して手動 DL を依頼する。
   - 本文に金額があり、かつ添付にも同じ領収書 PDF が付いている場合は**本文型として取り込む**
     （添付はダウンロード不要）**(ASSUMPTION)**。
3. **外部サイト型**: 「領収書はこちら」等のリンクで別サイトに誘導され、リンク先での
   ダウンロード・ログインが必要。→ 取り込まず `家計/要確認` を付け、一覧報告する。

判定に迷うもの（本文の金額が税抜/送料抜きの可能性等）は安全側に倒し、**外部サイト型（要確認）**として扱う。

## 二重取り込み防止

- 取り込み前に対象月の `entries/YYYY-MM.json` から `gmail_message_id` の一覧を作り、既出 ID はスキップする。
- スレッド内に複数メール（注文確認 + 発送通知 + 領収書）がある場合、金額の根拠となる 1 通だけを取り込む
  **(ASSUMPTION: 領収書・決済確定メールを優先。注文確認のみなら注文確認を使う)**。
- 同じ買い物が紙レシートと Gmail の両方から入るケースは、`aggregate.py` の
  「同一 日付+金額+店舗」重複疑い検出で捕捉される（月次レポートで確認）。

## 報告フォーマット（フロー②の完了時）

```
Gmail 取り込み結果（対象: YYYY-MM / クエリ実行 N スレッド）
- 本文から取り込み: N 件（エントリ ID: ...）
- DL待ち（家計/DL待ち）: M 件
  | 日付 | 差出人 | 件名 |
- 要確認（家計/要確認）: K 件
  | 日付 | 差出人 | 件名 |
- スキップ: J 件（既取り込み / 家計と無関係）
```

DL待ち・要確認の一覧は月次レポート（`reports/YYYY-MM.md`）の「要確認リスト」節にも転記する
（`templates/monthly-report.md` の該当節）。
