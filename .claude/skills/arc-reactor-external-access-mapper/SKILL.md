# external-access-mapper — 外部アクセス台帳と利用場面マップ (arc-reactor)

対象リポジトリを静的に走査して**外部にアクセスしている箇所**を洗い出し、
「何に・どこから・どういう認証で繋がっているか」の台帳と、
「そのアクセスはどのユースケースから使われるか」の利用場面マップを 1 本の Markdown に
まとめるスキル。SES 参画直後の外部依存の把握、障害の影響範囲調査、
セキュリティレビューの下敷きを作ることが目的。

> 大原則: **根拠なくして記載なし、値は転記しない。** 全項目に `ファイル:行` の根拠を添え、
> 根拠を提示できない項目は台帳に載せない。接続文字列・API キー等の値は
> 発見してもレポートに書かず、環境変数名・シークレット名の参照だけを書く。

persona: [`personas/arc-reactor.md`](../../../personas/arc-reactor.md) に従う
（応答は日本語 / コード・識別子は英語 / 推測は ASSUMPTION 明記）。

## トリガー

| ユーザー発話の例 | 動作 |
|---|---|
| 「このリポジトリの外部アクセスを洗い出して」 | 全カテゴリで台帳 + 利用場面マップを生成 |
| 「外部 API と DB の接続箇所を台帳にして」 | カテゴリを限定して台帳化 |
| 「このサービスがどの外部サービスに繋がってるかマップにして」 | 全カテゴリで生成 |
| 「障害調査用に DB アクセス箇所と呼び出し元を整理して」 | DB カテゴリ + 利用場面マップに集中 |
| 「Stripe を呼んでる場所と経路を全部出して」 | 特定サービスに絞って台帳 + 経路追跡 |
| 「external access map を作って」 | 全カテゴリで生成 |

ユーザーが明示的に依頼したときだけ起動する（コードを読んだついでに自動起動しない）。

## 標準フロー

### 1. スコープの確定

- 対象リポジトリの絶対パスを確定する（発話 / cwd。`projects/<name>/` 配下が典型）。
- スコープ指定（「DB だけ」「認証系だけ」「Stripe だけ」等）があればカテゴリ・サービスを絞る。
  無ければ全 7 カテゴリを対象にする。
- **除外パスを確定する**: `node_modules/`, `vendor/`, `dist/`, `build/`, `.git/`,
  自動生成物、テストのモック・フィクスチャ（実接続しないスタブ）は既定で除外。
  除外一覧はレポート末尾に明示する。テストコード内の実接続（統合テスト等）は
  「テスト経由」と注記して載せる。
- 規模を概算し、大規模（目安 10 万行超）ならカテゴリ単位・ディレクトリ単位の分割実行を提案する。

### 2. 外部アクセス箇所の検出

[references/access-patterns.md](references/access-patterns.md) の検出パターンに従い、
カテゴリごとに `rg` と設定ファイル読解でアクセス箇所を収集する。**スキャン実行前に必ず読むこと。**
カテゴリ:

1. database（ORM 設定・接続文字列・DB クライアント）
2. auth-saas（認証・認可 SaaS）
3. external-api（外部 API・SaaS の HTTP 呼び出し）
4. cloud-sdk（AWS / GCP / Azure の SDK 利用）
5. queue-messaging（キュー・メッセージング）
6. storage（オブジェクトストレージ・外部ファイルシステム）
7. network-io（gRPC, WebSocket, SMTP 等その他ネットワーク I/O）

検出の入口は 3 系統を併用する: 依存マニフェスト（`package.json`, `go.mod`, `requirements.txt` 等の
外部クライアントライブラリ）→ import / 初期化箇所へ、設定・環境変数（`.env.example`,
`config/`, IaC, `docker-compose.yml`）→ 参照箇所へ、コード直接検索（HTTP クライアント・
既知 SDK のシンボル）。同一の接続先に複数の呼び出し箇所がある場合は、接続先 1 件に
呼び出し箇所を列挙する形でまとめる。

### 3. 台帳化 — 接続先と認証方式の特定

検出した箇所ごとに以下を確定し、台帳の 1 行にする。

- **種類**: 上記カテゴリ。
- **接続先**: サービス名・ホスト。値がハードコードされていなければ環境変数名で判る範囲を書く
  （例: `DATABASE_URL` → PostgreSQL と推定、`ASSUMPTION` 注記付き）。
- **認証方式**: API キー / IAM ロール / OAuth / 接続文字列内クレデンシャル / mTLS 等。
  コードと設定から判定できた範囲のみ。判定できなければ「不明」と書く。
- **該当コード**: `file:line`。クライアント初期化箇所と代表的な呼び出し箇所。
- **利用場面**: フロー 4 で埋める呼び出し元ユースケースの要約。

推定で埋めた欄には「(推測)」を付け、断定しない。

### 4. 利用場面マップ — 呼び出し経路の追跡

台帳の各行について、**エントリポイントから当該アクセス箇所までの呼び出し経路**を追跡する。

- まずエントリポイントを列挙する: HTTP ルート定義、ジョブ・スケジューラ登録、CLI コマンド、
  イベント・メッセージハンドラ。フレームワークのルーティング規約は references の
  エントリポイント検出パターンを参照する。
- 経路は `file:symbol` の連鎖で示す。例:
  `routes/orders.ts:POST /orders` → `services/order.ts:createOrder` → `lib/stripe.ts:charge`
- 動的ディスパッチ・DI コンテナ・リフレクション等で静的に追跡しきれない場合は、
  その旨を「**経路不明・要確認**」と明記し、確認方法（実行時ログ・grep の起点等）を 1 行添える。
  推測で経路を埋めない。
- 1 つのアクセス箇所に複数経路がある場合は主要な経路を列挙し、多すぎる場合は
  エントリポイント数を記して代表例に絞る（絞ったことを明記する）。

### 5. レポート生成

以下の固定構成で Markdown レポートを生成し、チャットで要約を提示する。
保存先は対象プロジェクト配下 `<repo>/external-access-map/<YYYY-MM-DD>.md`
（対象リポジトリの git を汚す可能性があるため、**保存前に必ずユーザーに確認**する）。

```markdown
# External Access Map — <repo-name> (<YYYY-MM-DD>)

## サマリ
- スコープ / 除外パス / 対象カテゴリ / 規模
- 外部アクセス N 件（カテゴリ別内訳）/ 経路不明 M 件 / セキュリティ上の要報告 K 件

## 外部アクセス台帳
| # | 種類 | 接続先 | 認証方式 | 該当コード (file:line) | 利用場面 |
|---|---|---|---|---|---|

## 利用場面マップ
### <接続先ごと>
- エントリポイント: <HTTP ルート / ジョブ / CLI / イベント>
- 経路: file:symbol → file:symbol → file:line（アクセス箇所）
- 経路不明の項目は「経路不明・要確認」+ 確認方法

## セキュリティ上の要報告（該当時のみ・値は伏せる）
- ハードコードされた機密の場所（file:line のみ。値は書かない）

## 除外・免責
- 除外したパス一覧
- 静的調査の限界: 動的な URL 組み立て・リフレクション・実行時設定による接続は
  検出漏れがあり得る（本台帳は全量保証ではない）
- 接続先・認証方式の推定箇所は (推測) 表記の範囲
```

### 6. 完了提示

- 件数のカテゴリ別内訳と、経路不明・要確認の項目、セキュリティ上の要報告をチャットで要約する。
- 次アクション（要確認経路のヒアリング、セキュリティ問題の対処、
  `arc-reactor-codebase-onboarding` での全体地図化など）を提案する。

## 検出パターン

カテゴリごとの検出パターン（ライブラリ名・シンボル・設定ファイル・環境変数の慣習）と
エントリポイント検出パターンは [references/access-patterns.md](references/access-patterns.md) にある。
検出パターンの改善（新しいライブラリ・見落としたパターンの追加）は references への追記で行う。

## 既存スキルとの棲み分け

- `arc-reactor-codebase-onboarding` は全体地図（技術スタック・読み順・主要フロー）が目的。
  本スキルは外部境界に特化した台帳が目的。
- 処理フロー追跡（エントリポイント → 対象箇所の経路）は、将来
  `arc-reactor-sequence-diagram-generator` を作る場合に共通化の余地がある。

## 品質・安全性

- **全項目に根拠 `file:line` 必須。** 根拠のない項目は台帳に載せない
  （ユーザーが即座に検証できることが信頼の条件）。
- 接続文字列・API キー・トークン等の**値はレポートに転記しない**。環境変数名・
  シークレット名の参照のみ書く。ハードコードされた機密を発見した場合は、
  値を伏せて場所（`file:line`）だけを**最優先で報告**する（台帳より先に伝える）。
- 対象リポジトリへの操作は**読み取りのみ**。レポート保存時のみ書き込み、保存前に確認する。
- **ネットワークアクセスは行わない。** 接続先への疎通確認・API 照会はせず、
  コードの静的調査だけで判定する。判定できないものは「不明」「(推測)」と書く。
- レポートはプロジェクト固有情報のため、対象プロジェクトのディレクトリ配下に閉じる
  （当リポジトリ側や他プロジェクトに持ち出さない。CLAUDE.md の分離ルールに従う）。
- 検出漏れがあり得る旨を免責に明記する: 動的な URL 組み立て・リフレクション・
  実行時に注入される設定・プラグイン機構経由の接続は静的調査では拾いきれない。
- 経路が追跡しきれない場合は「経路不明・要確認」とし、推測で埋めない。
