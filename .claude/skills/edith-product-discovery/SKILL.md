# product-discovery — Issue 前のアイデア出し（機能案・課題仮説）

プロジェクトを調査し、**「何を作るか」がまだ決まっていない段階**のアイデア出しを担う。
成果物は機能案・課題仮説のバックログで、開発ループの上流に位置する:

```
edith-product-discovery（アイデア出し） → jarvis-issue-planner（Issue 化） → 開発ループ（実装）
```

persona: [`personas/edith.md`](../../../personas/edith.md)（複数の切り口・事実と解釈の区別・網羅範囲の明示）。

> **起動条件（重要）**: ユーザーが明示的にアイデア出し・壁打ちを依頼したときだけ走る。
> 自動起動しない。アイデアは**提案であって決定ではない** — 採否は必ずユーザーが決める。

## 隣接スキルとの境界

| やりたいこと | 使うスキル |
|---|---|
| 何を作るか決まっていない。ネタ出し・壁打ちしたい | **本スキル** |
| やることは決まった。Issue（実装計画）にしたい | jarvis-issue-planner |
| 技術負債を証拠付きで棚卸ししたい | arc-reactor-tech-debt-auditor |
| 競合・市場を調べたい | edith-competitor-market-scan |
| 初見のコードベースを把握したい | arc-reactor-codebase-onboarding |
| （仕事以外も含む）自分の思考を整理したい | karen-problem-essence-organizer |

本スキルは tech-debt-auditor・market-scan・onboarding の**成果物があれば入力として読む**
（無くても動く。あるものを使い、重複調査をしない）。

## データ配置

- アイデアバックログ: `projects/<project>/discovery/YYYY-MM-DD_<theme>.md`（theme は英語 kebab-case。
  全方位なら `general`）。
- 対象が `projects/` 配下に無い場合は保存先をユーザーに確認してから書く。
  対象がこのリポジトリ自身（スキル群など）の場合は `ideas/` 配下に置く。
- `discovery/` は対象リポジトリの git 管理外の作業ファイルとして扱う（ユーザーが求めない限り
  対象リポジトリにコミットしない）。
- プロジェクト固有情報の分離ルール（ルート CLAUDE.md）に従い、他プロジェクトの固有情報を
  持ち込まない・混ぜない。

## 標準フロー

### 1. 対象とテーマの確定

`AskUserQuestion` は最小限（1 ラウンド目安）。確認するのは:

- **対象プロジェクト**: `ls projects/` で提示して確定（発話で特定できていれば聞かない）。
- **発想の焦点**: 全方位 / 特定レンズだけ（例: 運用まわりだけ）/ 特定の機能領域だけ。
- **アイデアの深さ**: ざっと広く（各レンズ 2〜3 件）/ 特定テーマを深く。

既存材料を確認する（あれば読む・無ければ飛ばす。**この段階で新規調査はしない**）:
`projects/<project>/onboarding/`・`discovery/`（過去分）・`issues/`（既存 Issue と重複させない）・
tech-debt 監査レポート・`market-scan/`。

### 2. プロジェクト理解（調査）

対象の「現在できること」を掴む。全部は読まない:

1. README・規約文書・設定ファイルで、プロダクトの目的と技術スタックを掴む。
2. エントリポイントと主要ユースケースの導線を確認し、**既存機能の棚卸し**（機能マップ）を作る。
   広い探索は codebase-reader / Explore エージェントに委ね、結論だけ受け取る。
3. ユーザー接点（CLI / API / UI）、運用接点（デプロイ・監視・手作業の有無）、
   データの流れ（入力 → 保存 → 出力）を押さえる。

ここで押さえた事実が後段のアイデアの「根拠」になる。**確認していないことを根拠にしない**。

### 3. 発想（多角レンズ）

[`references/lenses.md`](references/lenses.md) の 5 レンズを順に当て、アイデアを発想する。
1 アイデアは必ず次を持つ:

- **種別**: `機能案`（あったほうがいい機能）/ `課題仮説`（問題になりそうな箇所の指摘）
- **根拠**: 調査で確認した事実（`file:line` または資料の出典）。無根拠のアイデアは出さない。
- **想定価値**: 誰の何がどう良くなるか 1〜2 行。
- **規模感**: S（〜半日）/ M（〜数日）/ L（それ以上）の粗い当て。`(推測)` 扱い。
- **確信度**: 高（事実に直結）/ 中（妥当な解釈）/ 低（面白いが飛躍あり）。
  **低も捨てない**（アイデア出しの段階では発散を優先し、選別は次段でユーザーが行う）。

発想の規律:

- 既存機能を「無い」と言わない（機能マップと突き合わせてから出す）。
- 既存 Issue・過去の discovery と重複するものは「既出」と明示して再掲する（黙って落とさない）。
- 課題仮説は tech-debt-auditor の代替ではない — 証拠 2 点の裏取りはせず「仮説」として出し、
  深掘りが要るものは監査スキルへの委託を提案する。

### 4. 壁打ち・選別（対話）

アイデア一覧（番号付き・レンズ別）をチャットに提示し、`AskUserQuestion` で選別する:

- 各アイデアの反応を取る: **深掘りしたい / Issue 化したい / 寝かせる / 却下**。
- 「深掘りしたい」が付いたものはその場で対話を続け、課題の輪郭・スコープ感を具体化する
  （ここが「壁打ち」の本体。全肯定せず、根拠の弱い点・先行例の有無を自分から指摘する）。
- 却下されたものも理由と一緒にバックログへ残す（同じアイデアを再提案しないため）。

### 5. 出力と引き渡し

1. [`templates/discovery.md`](templates/discovery.md) の構成でバックログを書き出す。
   全アイデアを状態付き（採用 / 寝かせ / 却下）で残し、調査でカバーした範囲と
   **見ていない範囲**（当てなかったレンズ・読まなかった領域）を明示する。
2. 出力後、チャットに要約（アイデア総数・採用数・出力パス）を返す。
3. **Issue 化の引き渡し**: 「Issue 化したい」が付いたアイデアは、jarvis-issue-planner への
   起動を提案する（そのまま続けて実行してよい。discovery ドキュメントのパスを引き継ぐ）。
4. **ToDo 台帳への収穫**: 採用・深掘り継続のアイデアを候補として提示し、確認のうえ
   `todo.py add --status inbox --source-type research --source-ref <バックログのパス>` で
   台帳に入れる（追記のたびに一言通知）。

## 自己進化

固定する芯は崩さない（起動条件・5 レンズの枠組み・根拠必須・採否はユーザー）。育てるのは:

- レンズごとの「問い」（[`references/lenses.md`](references/lenses.md)）: 壁打ちで筋の良い
  アイデアを生んだ問い・観点が見つかったら追記する。
- 却下理由のパターン: 同じ理由の却下が繰り返されたら、発想の規律（本ファイル）に
  「出さない条件」として追記する。

## やってはいけないこと

- 自動起動しない。明示依頼が無ければ走らない。
- 無根拠のアイデアを出さない（すべてのアイデアに調査で確認した根拠を付ける）。
- アイデアを勝手に採用・Issue 化・実装しない（採否は必ずユーザー）。
- 既存 Issue・過去の discovery を確認せずに出さない（重複は「既出」と明示）。
- 規模感・価値の見立てを断定調で書かない（`(推測)` を付ける）。
- 他プロジェクトの固有情報を発想の材料・出力に混ぜない。
- 機密情報（API キー・顧客名等）をバックログに書かない。
