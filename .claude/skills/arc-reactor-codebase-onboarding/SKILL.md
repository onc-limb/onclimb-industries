# codebase-onboarding — 参画直後のコードベース把握 (arc-reactor)

初見のリポジトリを対象に、**人間（参画者）が読む順に最適化した**オンボーディング資料 3 部作を生成し、
`projects/<project>/onboarding/` に永続化するスキル。一般論は書かず、
**このリポジトリ固有の事実＋出典パス**だけを書く。

## 役割

- 参画立ち上がりの数日を短縮する「全体地図・主要フロー・読み始めポイント」を作る。
- 断片的な探索メモを揮発させず、案件ディレクトリ配下に差分更新可能な形で残す。
- 後続参画者への引き継ぎ資料としてそのまま渡せる品質にする。

## codebase-reader との棲み分け

| 観点 | codebase-reader | 本スキル |
|---|---|---|
| 成果物 | 単発の構造化レポート（チャット返信） | 3 部構成 Markdown の永続ドキュメント |
| 視点 | 構造の網羅（何があるか） | 学習順序（どの順で・なぜ読むか） |
| フロー | 扱わない | ユースケース単位でエントリポイント→出口を追跡 |
| 再利用 | 揮発する | commit hash 付きで保存、`update` モードで差分更新 |
| 関係 | 本スキルが**全体探索の下請け**として呼ぶ | codebase-reader の上位工程 |

「構造をさっと知りたいだけ」なら codebase-reader を直接使う。
「参画した／読む順番が欲しい／資料として残したい」なら本スキル。

## トリガー

| ユーザー発話の例 | 動作 |
|---|---|
| 「このコードベースのオンボーディング資料を作って」 | 新規生成（フルフロー） |
| 「新しい案件に参画したので全体像を掴みたい」 | 新規生成 |
| 「どこから読み始めればいい？」「読む順番を教えて」 | 03-reading-guide 中心に生成・提示 |
| 「<ユースケース>の流れを追ってコードの地図にして」 | 指定ユースケースを 02-flows に追加 |
| 「オンボーディング資料を最新化して」 | `update` モード（commit 差分で更新） |

## 標準フロー

1. **対象プロジェクトの特定**: 発話 or カレントの文脈から `projects/<project-name>/` を特定する。
   `projects/` に見つからなければパスをヒアリングする。対象の HEAD commit を控える
   （`git -C <project_path> rev-parse --short HEAD`）。
2. **モード判定**: `projects/<project>/onboarding/` が既にあれば `update` モード。
   既存ドキュメントの frontmatter `commit:` と HEAD を比較し、差分ファイル
   （`git diff --stat <old>..HEAD`）を確認して影響セクションだけ更新する。無ければ `new` モード。
3. **全体探索の委譲**: 全体構造の把握は **codebase-reader（エージェント）に委譲**し、
   構造化レポート（構成・技術スタック・エントリポイント候補）を受け取る。
   自分でリポジトリ全ファイルを読み込まない（コンテキスト節約）。
4. **主要ユースケースの選定**: README・ルーティング定義・エントリポイントから候補を 2〜5 本挙げ、
   `AskUserQuestion` でユーザーに確認する。ユーザー指定があればそれを優先する。
5. **フロー追跡**: 各ユースケースについて、エントリポイント → 出口（レスポンス / 永続化 / 外部呼び出し）
   をファイル:シンボル単位で追跡する。DI・コード生成などで静的追跡が切れた箇所は、
   実行設定（package.json / Dockerfile / ルーティング定義等）で裏取りし、
   それでも推測が残る箇所は本文に **(ASSUMPTION: ...)** と明記する。
6. **ドキュメント生成**: `templates/` の 3 雛形の見出し構成に従い、
   `projects/<project>/onboarding/` へ Write する。frontmatter に `generated:`（日付）と
   `commit:`（対象 HEAD）を必ず記録する。本文中のファイル参照はすべて実在パスにする。
7. **完了提示**: 生成パス一覧と「読み始めポイント」上位 3 件をチャットで要約する。

## 出力ドキュメント構成

`templates/` の雛形を見出しの正とする（見出しの増減・順序変更をしない）。

| ファイル | 内容 |
|---|---|
| `01-map.md` | 全体地図: プロジェクト概要（3 行）/ 技術スタック（根拠ファイル付き）/ ディレクトリ構成と役割 / モジュール間依存 / 外部依存・連携 / 実行・ビルドの入口 |
| `02-flows.md` | 主要フロー: ユースケース一覧表 + 各フローの「エントリポイント→出口」ステップ表（ファイル:シンボル / 役割）と読むときの注意 |
| `03-reading-guide.md` | 読み始めポイント: 理由付き・番号付きの読む順番（5 件以上）/ 最初の 1 時間でやること / 変更影響の大きい要注意箇所 / まだ分かっていないこと（要ヒアリング） |

## 保存先規約

- 出力先は **`projects/<project-name>/onboarding/` 固定**（CLAUDE.md の
  「プロジェクト固有情報の分離ルール」準拠。プロジェクト固有資料は当該プロジェクト配下に閉じる）。
- **対象リポジトリのソースツリーには一切書き込まない**。実行前後で対象の `git status` を変えない
  （`onboarding/` が対象リポジトリの追跡対象に入る場合は、書き込み前にユーザーへ確認する）。
- 当リポジトリ（onclimb-industries）側には保存しない（`projects/` は git 管理外）。
- 複数プロジェクトに共通する知見が出たら、固有情報を除いた形で ideas/ や personas/ 側へ
  切り出す（本スキルの出力には混ぜない）。

## 品質・安全性

- 事実は必ず**出典パス付き**で書く。コードに無いことを創作しない。推測は **(ASSUMPTION: ...)** と明記する。
- `.env` / credentials 類は**値を読まない・書かない**。キー名の言及に留める。
- 本スキルは読み取り + `onboarding/` への書き込みのみ。対象コードの変更・整形・コミットは行わない。
- 巨大リポジトリでは全体探索を codebase-reader へ委譲し、フロー追跡はユースケース 2〜5 本に絞る。
- 応答・ドキュメントは日本語、コード識別子・パスは原語のまま表記する（personas/arc-reactor.md 準拠）。
