# 機密マスキングルール & 公開前チェックリスト

friday-tech-article-drafter のステップ 4（機密マスキング）・ステップ 5（公開前チェック）の実体。
**このファイルの項目を全通過するまで、draft.md の `status` を `ready-to-publish` にしてはならない。**

## 前提（多層防御）

入力の tech digest / vault ノートは jarvis-worklog / jarvis-knowledge-base の段階で
マスキング済み（`<REDACTED:種別>` 等）だが、**上流のマスキングを信頼しない**。
公開は蓄積より一段厳しい基準で、ここで独立にマスキングと検査を行う。

## 1. マスキングルール表

| カテゴリ | 例（この列の内容は説明用の架空値） | 置換方針 |
|---|---|---|
| 顧客名・取引先名・案件名/案件コード | 「株式会社〇〇」「PJ-1234」 | 「ある受託案件」「参画中の SES 案件」など役割ベースの一般表現へ。複数案件が登場する場合は「案件A/B」 |
| 個人名・アカウント名（自分以外） | 同僚・顧客担当者の氏名、Slack/GitHub の ID | 「チームメンバー」「先方の担当者」等の役割表現へ |
| 社内 URL・内部ホスト名・IP | `https://wiki.example.internal/...`、`10.x.x.x`、`*.local` | URL ごと削除するか「社内 Wiki」「内部環境」等へ。パス構造も残さない |
| リポジトリ名・プロジェクト識別子（非公開のもの） | private リポジトリ名、digest の project id | 「対象リポジトリ」「本プロジェクト」へ。公開 OSS のリポジトリ名はそのままでよい |
| 認証情報 | API キー、トークン、パスワード、SA 鍵、接続文字列 | 値は一切載せない。設定例では `<YOUR_API_KEY>` 等のプレースホルダに |
| メールアドレス・電話番号 | `taro@client.co.jp` | 削除、または `user@example.com` へ |
| 未公開の事業・契約情報 | 単価、契約条件、未リリース機能、顧客の業務データ | 記事に載せない（言い換えでなく削除） |
| digest 由来のプレースホルダ | `<REDACTED:credential>` 等 | 公開文に残さない。文意が通る一般表現へ書き換える |

置換したら `masking-log.md` に **「カテゴリ + 件数 + 置換後表現」のみ**記録する。
置換前の実値は masking-log.md に書かない（ログを機密の対応表にしない）。

## 2. 機械チェック（grep）

draft.md に対して以下を実行し、**すべてヒット 0 件**であることを確認する。

```bash
DRAFT=/path/to/articles/<dir>/draft.md
REPO=/Users/satoshi-onga/Documents/onclimb-industries

# 認証情報・シークレットの典型パターン
grep -nE 'AKIA[0-9A-Z]{16}|ghp_[A-Za-z0-9]{36}|github_pat_|sk-[A-Za-z0-9_-]{20,}|xox[baprs]-|AIza[0-9A-Za-z_-]{35}|-----BEGIN [A-Z ]*PRIVATE KEY|eyJ[A-Za-z0-9_-]{20,}\.' "$DRAFT"

# 認証系キーワードに実値が続いていないか（プレースホルダ <...> は目視で除外判断）
grep -niE '(api[_-]?key|secret|token|password|passwd|credential)[[:space:]"'"'"':=]+[A-Za-z0-9_/+-]{8,}' "$DRAFT"

# 内部 URL・内部ホスト・プライベート IP
grep -nE 'https?://[^ )]*\.(internal|local|corp|lan)\b|\b10\.[0-9]+\.[0-9]+\.[0-9]+\b|\b192\.168\.[0-9]+\.[0-9]+\b|\b172\.(1[6-9]|2[0-9]|3[01])\.[0-9]+\.[0-9]+\b' "$DRAFT"

# メールアドレス（example.com 系プレースホルダ以外）
grep -nE '[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}' "$DRAFT" | grep -v 'example\.\(com\|org\|net\)'

# 上流マスキングのプレースホルダ残り
grep -n 'REDACTED' "$DRAFT"

# ユーザー管理の NG ワード辞書（顧客名・案件コード等の実名。gitignore 配下なので実値を書いてよい）
[ -f "$REPO/articles/_ng-words.txt" ] && grep -nFf "$REPO/articles/_ng-words.txt" "$DRAFT"
```

- ヒットした場合はステップ 4（マスキング）へ戻り、解消してから再実行する。
- `_ng-words.txt` が無い場合は、今回の記事に登場した顧客名・案件名を追加する形で作成を促す。

## 3. 公開前チェックリスト（人間確認）

ユーザーと 1 項目ずつ確認し、全項目にチェックが付いたときのみ `status: ready-to-publish` へ更新する。

- [ ] **機械チェック全パス**: 上記 grep がすべて 0 件（NG ワード辞書含む）
- [ ] **顧客・案件の直接特定なし**: 顧客名・案件名・担当者名・非公開リポジトリ名が本文・コード例・図に残っていない
- [ ] **組合せ特定なし**: 時期 × 業種 × 特徴的な技術構成/業務ドメインの組合せから案件・顧客を推定できない
      （必要なら時期・規模をぼかす、複数案件の一般化として書く）
- [ ] **認証情報・内部インフラなし**: キー/トークン/内部 URL/ホスト名/IP/バケット名等の実値がゼロ。設定例はすべてプレースホルダ
- [ ] **契約・事業情報なし**: 単価・契約条件・未公開機能・顧客の業務データに触れていない
- [ ] **契約上の発信可否**: 対象案件の NDA・契約が技術発信を禁じていないか確認した（不明なら公開しない）
- [ ] **事実性**: 本文の主張・数値・手順がすべて frontmatter `sources:` の出典と対応し、創作・誇張がない。
      一般論による補足は実体験と区別できる書き方になっている
- [ ] **内部メモ削除**: テンプレート末尾の内部メモ欄・HTML コメントを削除した
- [ ] **ユーザーの公開判断**: 上記すべてを踏まえ、ユーザー本人が「公開してよい」と明言した

1 項目でも No / 未確認なら `status: draft` のままとし、未通過項目を提示して終了する。
