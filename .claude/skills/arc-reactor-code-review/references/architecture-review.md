# アーキテクチャ・原則レビューの手法

観点4（アーキテクチャ・原則）を掘り下げるための手引き。この観点は本スキルで**最重要**に扱う。
数値の目安はすべて**初期値**であり、実運用の結果に応じて人が調整する。

## なぜ今この観点が最重要か

コードは AI 支援で大量かつ高速に生まれるようになった。「動くコードを書く」コストが下がった一方、
**それを人が読んで理解し、安全に変え続ける**コストは下がっていない。むしろ、もっともらしく動くが
意図の読み取れないコード・依存が絡まったコードが増えると、誰も全体を把握できず変更が怖い状態に陥る。

だからこのレビューの主眼は「賢い設計かどうか」ではなく、次の2点に置く。

1. **意図の一目瞭然性** — 人の目で見て、このコードが何をしたいのかが一目で分かるか。
2. **依存関係の健全さ** — 依存が複雑に絡まっていないか。1つ変えたときの波及が読めるか。

賢い抽象より、読める・追える・変えられることを上位に置く。

---

## 1. 意図の一目瞭然性

対象（関数 / ファイル / モジュール / 機能）を**初見の読み手**の目で読み、「何をしたいのか」が
実装を深追いせずに掴めるかを判定する。

### 判定尺度（出力に必ずこの4段階＋根拠で書く）

| 評価 | 基準 |
|---|---|
| ◎ | 名前・シグネチャ・構成（入口）だけで目的が分かる。実装を読む必要がない |
| ○ | 入口のファイル/関数を1つ読めば目的と主要フローが分かる |
| △ | 目的の把握に複数ファイルの読解や実装の追跡が必要。名前が実態を語っていない箇所がある |
| × | 追跡しても目的が判然としない。名前が実態と食い違う／フローが分岐と副作用で追えない |

△以下と判定したら、**何がそう判定させたか**（下記の問いのどれで引っかかったか）を根拠として添える。

### 判定に使う問い（機械的な代理指標つき）

- **名前で分かるか**: 関数・型・モジュール名とシグネチャだけで役割・入出力・単位・真偽の向きが推測できるか。
  名前が実態と食い違っている（嘘をつく名前）箇所は特に重い指摘。
- **主要フローが追えるか**: 対象の「幹」の処理が入口から素直に追えるか。
  代理指標（目安）: 1 関数 **80 行超**・ネスト **4 段以上**・幹の処理に飛び先（間接呼び出し）が
  **5 箇所超**あれば「追いにくい」兆候として扱う。
- **全体像に何ファイル要るか**: この対象の目的を理解するのに開くファイル数。
  目安: 1 機能で **5 ファイル超**を行き来しないと目的が掴めないなら意図が拡散している兆候。
- **ノイズが無いか**: 自明な what を説明するだけのコメント・使われないコード・無関係な差分が、
  本筋の読み取りを邪魔していないか。逆に「なぜ」が必要な箇所（非自明な判断・回避策）に説明があるか。

指摘の書き方の例:
- `[imo] src/order/service.py:40 — process() が検証・在庫引当・課金・通知を1関数で行っており、初見で「注文確定処理」という目的が読み取れない（120行・ネスト5段）。処理段階ごとに私関数へ分け、process() は段階の呼び出しだけにすると意図が一目で分かる。`
- `[ask] src/util/handle.ts:12 — 関数名 handle() と引数 data から役割が推測できない。何を handle する意図か（命名の再検討含め）確認したい。`

## 2. 依存関係の複雑さ

「この対象が何に依存し、何から依存されているか」を把握し、複雑さの兆候を探す。

### 依存を把握する軽量な方法（スコープに応じて）

- import/require 文を拾う: `git grep -nE "^\s*(import|from|require|use )" -- <対象パス>` 等。
- ディレクトリ構成からレイヤ（例: handler → service → repository → model）を**推定**する。
  推定はあくまで仮説なので、**推定レイヤに基づく違反指摘には `(ASSUMPTION: レイヤ構成を◯◯と推定)` を
  必ず付け、確信が持てなければ [ask] に落とす**（実は意図されたレイヤかもしれない）。
  規約文書・README にレイヤ定義があればそれを正とし、ASSUMPTION は不要。
- ある機能の呼び出し元/依存先: `git grep -n "<関数名/型名>"` で参照箇所を辿る。
- 対象リポジトリに導入済みのツールがあれば活用する（例: JS の `madge --circular`、Go の
  `go list`/`go mod graph`、Python の `pydeps`/`import-linter`）。
  **対象リポジトリに新たなツールをインストールしない**（`npx`/`uvx` 等の一時実行のみ可。
  それも使えなければ grep + Read で十分）。

### 複雑さの兆候（見つけたら指摘）

- **循環依存**: モジュール A↔B が相互参照している。
  重要度の付け方: 初期化順・ビルドに実害がある、または変更が常に両側へ波及する密結合なら `[must]`。
  型の共有だけの緩い循環など実害が読み切れないものは `[imo]`（解消案付き）か `[ask]`。
- **レイヤ違反**: UI/ハンドラが DB を直接叩く、下位層がフレームワークやビジネスルールを知っている等、
  層の飛び越え（推定レイヤ由来なら上記のとおり ASSUMPTION/[ask]）。
- **強すぎる結合**: 特定モジュールの内部実装（private・具体型・データ構造）に多数の箇所が依存している。
- **ファンイン/ファンアウト過多**: 目安は**相対評価** — 対象リポジトリ内の同種モジュールと比べ、
  参照される数/依存する数が突出している（おおよそ中央値の 3 倍超、または最大級）モジュールを
  「一点集中」「責務過多の疑い」として挙げる。絶対数の閾値は使わない（規模依存のため）。
- **依存の向きの不健全さ**: 安定して変わりにくいものではなく、変わりやすい具体実装に上位が依存している
  （依存性逆転の原則 DIP の違反）。
- **変更波及の広さ**: 「この機能を1つ変えると何ファイル触るか」を見積もる。
  目安: 単一機能の典型的な変更で **5 ファイル超**に修正が及ぶ構造なら、波及が広い兆候として指摘する。

指摘の書き方の例:
- `[must] src/billing/ ↔ src/order/（代表: billing/charge.py:12, order/complete.py:88）— 相互 import の循環依存。注文確定の変更が常に課金側の再テストを要求する密結合になっている。共有する型を別モジュールへ切り出すか、片方向の依存に整理したい。`
- `[imo] src/api/user_handler.go:30 — ハンドラが DB クライアントを直接呼んでおり、他のハンドラが通す repository 層を飛ばしている (ASSUMPTION: handler→service→repository のレイヤ構成と推定)。層をそろえるとデータアクセスの変更が1箇所で済む。`

## スコープ別の適用

### プロジェクト全体

1. ディレクトリ構成と主要モジュールを把握し、**依存マップ**（どの層/モジュールがどこに依存するか）を素描する。
2. 循環・レイヤ違反・一点集中（god モジュール）を探す。
3. **代表ユースケースを 2〜3 本**選び、エントリポイント→出口を辿って意図の一目瞭然性を判定する。
   選定基準（優先順）: (1) README・ドキュメントが主要機能として挙げているもの、
   (2) エントリポイント（main / ルーティング定義 / CLI サブコマンド）から見て中核のフロー、
   (3) `git log --since="90 days ago" --name-only` で変更頻度が高いパスを通るフロー
   （よく変わる場所ほど読みやすさの価値が高い）。何を選んだかと選定理由を出力に明記する。
4. 全ファイルは追えないので、判定に使った代表フローと、見ていない範囲を明示する。

### 機能（feature）

1. その機能を構成するファイル群が**1つのまとまり**として凝集しているか（役割が散らばっていないか）。
2. その機能の**依存フットプリント**: 何に依存し、何から依存されているか（原則1ホップ）。機能の境界が明確か。
3. 初見の読み手がこの機能の入口から出口までを1度読んで理解できるか（判定尺度◎〜×で評価）。

### PR / 差分

1. この変更が意図の一目瞭然性を**良くしたか・悪くしたか**（可読性の増減）。
2. **新たな結合・循環・レイヤ違反**を持ち込んでいないか。
3. 差分そのものから変更の目的が読み取れるか（PR 説明と整合し、無関係な変更が混ざっていないか）。

---

## 指摘のトーン

- 「賢いか」ではなく「読めるか・追えるか・変えられるか」で評価する。過剰設計は簡素化を提案する。
- 大規模な作り替えを軽々に求めない。まず**意図が伝わる最小の変更**（分割・改名・依存の一方向化）を提案する。
- 根拠を必ず添える（どのファイルのどの依存が、なぜ読み手/変更の負担になるか）。断定できないものは [ask]。
