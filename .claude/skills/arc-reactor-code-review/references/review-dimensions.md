# レビュー観点チェックリスト（3観点）

arc-reactor-code-review が適用する固定チェックリスト。**機能 / セキュリティ /
パフォーマンス**の3観点を、既定では全て適用する（ユーザーが観点を指定したらその観点だけ）。
観点の改善・追加は本ファイルを人が意図的に更新する（自動学習はしない）。
本ファイルの数値の目安はすべて**初期値**であり、実運用の結果に応じて人が調整する。

アーキテクチャ・設計観点（コードスメル・依存関係・設計原則・意図の読みやすさ）は
本スキルの対象外で、`arc-reactor-architecture-review` が担う。

## スコープによる重み付け

- **PR / 差分**（変更駆動）: 3観点すべてを差分の行単位で見る。変更が既存の性能・安全性を
  悪化させていないかを重視する。
- **機能 / プロジェクト全体**（現状駆動）: 全行を追えないので、高リスク箇所を
  サンプリングして3観点を見る。
  - **高リスク箇所の優先順位**: (1) 外部入力を受ける境界（HTTP ハンドラ・CLI 引数・ファイル/メッセージ取込）、
    (2) 認証・認可・課金・データ削除などの実害が大きい処理、(3) 外部 I/O（DB・API 呼び出し）を含む処理、
    (4) ホットパス（下記定義）、(5) 複雑な分岐（下記定義）。
  - **サンプリング量の目安（初期値）**: 優先順位の高い順に**5〜10 ファイル**。機能スコープで対象が
    10 ファイル以下なら全ファイル見る。どのファイルを見たかは「未レビュー範囲」で開示する。
- **用語の定義**:
  - **ホットパス** = 実測せず次で判定する: リクエスト/イベント処理のたびに必ず通る経路、
    ループ・バッチの内側で呼ばれる処理、ミドルウェア/デコレータ等の共通経路。
  - **複雑な分岐** = 次のいずれか（目安）: ネスト 4 段以上 / 1 関数 80 行超 /
    1 つの条件式に 3 項以上の論理結合、を含む箇所。

## 重要度ラベル

| ラベル | 意味 | 使いどころ |
|---|---|---|
| `[must]` | 修正必須 | 下記「must の基準」を満たすもの |
| `[imo]` | 提案。自分ならこうする（対応は任意） | 設計・可読性・性能の改善案で、確度は高いが必須ではないもの |
| `[nit]` | 些細な指摘。好みに近い | typo、軽微な様式ゆれ |
| `[ask]` | 質問。意図の確認 | コンテキスト不足で断定できない懸念、設計意図の確認 |

- **must の基準**（スコープで読み替える）:
  - 変更駆動（PR / 差分）: このままマージすると壊れる・危険になるもの
    （バグ、データ破壊、機密混入、脆弱性、明確な規約違反）。
  - 現状駆動（機能 / プロジェクト全体）: **放置すると本番で実害**（データ破壊・停止・脆弱性・
    誤課金・情報漏えい）**に至る蓋然性が高い欠陥**。設計の綺麗さは本スキルの対象外
    （arc-reactor-architecture-review の領分）。
- 断定できる根拠が無い指摘は `[must]`/`[imo]` にせず `[ask]` に落とす（誤指摘で信頼を毀損しない）。

## 観点1: 機能（correctness）

「意図どおり正しく動くか」。

- [ ] 境界条件: 空 / null / 0 / 負数 / 上限超過 / 空コレクションで壊れないか
- [ ] エラーハンドリング: 例外・エラー戻り値の握りつぶし、失敗パスでのリソースリークが無いか
- [ ] 状態と並行性: 競合状態、非冪等な再実行、トランザクション境界の破れが無いか
- [ ] ロジック: 条件式の反転・off-by-one・早期 return の抜け道が無いか
- [ ] 互換性: 呼び出し元が壊れる signature / スキーマ / API レスポンスの変更が無いか
- [ ] データ: マイグレーション・永続データの扱いが不可逆に壊れないか
- [ ] テスト: 変更ロジックに対応するテストがあるか、写経（トートロジー）でなく失敗しうる検証か、
      既存テストの削除・skip に正当な理由があるか
      （テストが存在しないリポジトリでは、規約でテストが求められていない限り [must] にせず
      [imo]/[ask] でリスクの高い箇所への追加を提案する）

## 観点2: セキュリティ（security）

「悪意ある入力・想定外の使われ方に耐えるか」。入力処理・認証認可・外部連携・依存に触れる変更で特に重視。

- [ ] 機密情報（API キー・トークン・パスワード・接続文字列）が混入していないか（発見時は値を伏せて `[must]`）。
      **シークレット様文字列の判定手がかり**（例。これに限らない）:
      `AKIA`/`ASIA`（AWS）、`ghp_`/`gho_`/`github_pat_`（GitHub）、`sk-`（OpenAI/Stripe）、
      `xoxb-`/`xoxp-`（Slack）、`AIza`（Google）、`-----BEGIN ... PRIVATE KEY-----`、`eyJ` 始まりの JWT、
      `scheme://user:password@host` 形式の接続文字列、設定・コード内の 32 文字超の高エントロピー英数記号列。
      プレースホルダ（`your-api-key`、`xxx`、明らかなダミー値）は除く。
- [ ] インジェクション: SQL / OS コマンド / パス / テンプレート / LDAP などに未検証の外部入力が渡っていないか
- [ ] 出力エスケープ: XSS 等、外部データを出力する箇所で適切にエスケープ/サニタイズしているか
- [ ] 認証・認可: 認可チェックの欠落（ID 直指定でのアクセス、権限昇格）、認証の迂回経路が無いか
- [ ] 入力検証: サイズ・型・範囲・形式の検証があるか、信頼境界を越えるデータを無検証で信頼していないか
- [ ] 機微データの扱い: ログ・エラー・レスポンスに個人情報や機密を漏らしていないか、保存時の扱いは適切か
- [ ] 依存関係: 出所不明・メンテ停止・既知脆弱性のあるパッケージを持ち込んでいないか、バージョン固定は妥当か
- [ ] 安全でない既定: 危険な既定値（認証オフ・全許可 CORS・検証スキップ）を導入していないか

## 観点3: パフォーマンス（performance）

「無駄なコスト・スケールしない構造が無いか」。**静的レビューでは実測できない**ため、
推測で微最適化を指摘せず、コードから読める根拠（計算量・ホットパス該当・データ量の想定）を添える。
データ量・頻度が読み取れず断定できないものは [ask]（「このテーブルは何件規模ですか？」）に落とす。

- [ ] 計算量: ループ内ループ・不要な O(n²)・ホットパスでの高コスト処理が無いか
- [ ] N+1 / クエリ: ループ内での DB/API 呼び出し、まとめられるクエリが無いか
      （インデックス不足の指摘はスキーマ定義が見えるときのみ。見えなければ [ask]）
- [ ] 無駄な再計算・確保: 同じ結果の再計算、不要なコピー/中間コレクション生成、巨大データの全読み込みが無いか
- [ ] I/O とブロッキング: 非同期文脈での同期ブロッキング、直列化できる待ち、コネクションの都度生成が無いか
- [ ] 大量データ: ページング/ストリーミングが無いまま全件をメモリに載せていないか
- [ ] キャッシュ: キャッシュの有無・失効・不整合（古い値を返す）に問題が無いか
- [ ] リソースリーク: ファイル/コネクション/ゴルーチン/タイマーの解放漏れが無いか
- [ ] スケール限界: 入力・利用者・データ量の増加でこの実装が破綻する境界はどこか

## 適用時の共通ルール

- 行単位の指摘は必ず `file:line` を伴い、可能なら具体的な修正案を1つ添える。
  行が一意に定まらない**複数ファイルにまたがる指摘**（モジュール間で重複した検証ロジック・
  横断する N+1 等）は「モジュール対/パス + 代表参照行1つ」の書式で書く
  （例: `src/billing/ ↔ src/order/`（代表: `billing/charge.py:12`））。
- 全量を見た観点は該当なしなら「指摘なし」、サンプリングした観点は「サンプル範囲内で指摘なし」と書き分ける。
- 既存レビューコメント・過去の報告書で既出の論点は再指摘しない（必要なら「既出」と注記して同意のみ示す）。
- 良い点（きれいな分割・丁寧なテスト・素直な制御フロー等）を最低1つ拾い、総評に含める。
