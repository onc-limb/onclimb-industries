# arc-reactor-code-review — スコープ別・観点別コードレビュー

コードを入力に、**スコープ（何を見るか）× 観点（どの目で見るか）** を組み合わせた
観点別レビューを生成するスキル。レビューの初動を自動化し、観点の属人化を references の
固定チェックリストで防ぐ。GitHub PR だけでなくローカルのコードにも使える。

## 役割

- レビュー対象のスコープを特定し、そのスコープに合った粒度で材料を集める。
- [references/review-dimensions.md](references/review-dimensions.md) の4観点で評価し、
  指摘を「観点ラベル + 重要度ラベル + `file:line`」の統一書式で提示する。
- アーキテクチャ観点では [references/architecture-review.md](references/architecture-review.md) の
  手法で「意図の一目瞭然性」と「依存関係の複雑さ」を人の目線で診断する。
- 対象リポジトリのプロジェクト規約（CLAUDE.md / linter 設定 / CONTRIBUTING.md）を根拠として引用する。
- PR スコープでは、ユーザーが承認した指摘のみを `gh` で PR 上に投稿する。

## /code-review との棲み分け

| | `/code-review`（組み込み） | 本スキル |
|---|---|---|
| 対象 | ローカルの作業中 diff（自分の変更） | PR / ローカル差分 / 特定機能 / プロジェクト全体 |
| 目的 | コミット前の素早い検品 | 観点別レビュー成果物の作成（構造化・スコープ選択可） |
| 観点 | 汎用（正しさ・簡素化・効率） | 4観点で固定・分離（機能 / セキュリティ / パフォーマンス / アーキテクチャ） |
| 規約参照 | なし | 対象リポジトリの規約ファイルを読み取り根拠にする |

**振り分け規則**（機械的に判定する）: 次のいずれかに該当すれば本スキル —
(a) スコープの指定がある（PR 番号/URL、機能名・ディレクトリ、「プロジェクト全体」）、
(b) 観点の指定がある（「セキュリティ観点で」「アーキテクチャを見て」等）、
(c) 観点別の報告書・レビューコメントという成果物を求めている。
いずれにも該当しない「コミット前に見て」「この変更ちょっと確認して」は `/code-review` を案内する。

## スコープ（何を見るか）

発話からスコープを1つ特定する。曖昧なら候補を挙げて確認する。

| スコープ | 対象 | 材料の集め方 | レビュー粒度 |
|---|---|---|---|
| **PR** | GitHub 上の PR | `gh pr view` / `gh pr diff`（後述） | 差分の行単位 + 変更が全体に与える影響 |
| **差分 (diff)** | ローカルの未コミット変更 / ブランチ差分 | `git diff` / `git diff <base>...HEAD` | 差分の行単位 |
| **機能 (feature)** | 特定の機能・モジュール・ディレクトリ | 対象ファイル群を Read + 呼び出し元/依存を追う | ファイル単位 + 機能としてのまとまり |
| **プロジェクト全体 (project)** | コードベース全体 | ディレクトリ構成・依存マップ + 代表フローの抽出 | 構造単位（行単位は主要箇所のみ） |

- **差分スコープの既定解釈**: 単に「差分を/この変更をレビューして」と言われたら
  `git diff HEAD`（staged + unstaged の未コミット変更）を対象にする。ブランチ名や「main との差分」等の
  言及があるときだけブランチ差分にし、base 未指定なら `origin/main`（無ければ `main`、それも無ければ
  リポジトリの既定ブランチ）と `git diff <base>...HEAD`（merge-base 起点）で取る。採用した base は出力に明記する。
- **PR / 差分**は「この変更が正しいか・全体を悪化させていないか」を見る（変更駆動）。
  差分外への影響確認は、変更された関数・型の**直接の呼び出し元（1ホップ）**までを見る。
- **機能 / プロジェクト全体**は「今あるコードが健全か」を見る（現状駆動）。全行は追えないので、
  アーキテクチャ観点を主軸に、機能/セキュリティ/パフォーマンスは高リスク箇所をサンプリングする
  （選び方と件数の目安は review-dimensions.md「スコープによる重み付け」）。
- プロジェクト全体で全行レビューはしない。見た範囲と見ていない範囲を必ず明示する。

## 観点（どの目で見るか）

既定は**4観点すべて**を適用する。ユーザーが観点を指定した場合はその観点に絞る。
各観点の定義・チェック項目・重要度ラベルの使い分けは
[references/review-dimensions.md](references/review-dimensions.md) を正とする。

1. **機能 (correctness)** — 意図どおり動くか。境界条件・エラー処理・並行性・互換性・データ整合・テストの妥当性。
2. **セキュリティ (security)** — 機密混入・インジェクション・認可欠落・入力未検証・依存リスク。
3. **パフォーマンス (performance)** — 計算量・N+1・無駄な再計算/確保・ブロッキング I/O・リソースリーク・スケール限界。
4. **アーキテクチャ・原則 (architecture)** — **意図が一目で分かるか / 依存が複雑でないか**を軸に、
   責務分離・抽象の妥当性・命名/可読性・規約整合を見る。詳細は
   [references/architecture-review.md](references/architecture-review.md)。

## 標準フロー

1. **スコープと観点の確定**: 発話から上記のスコープを1つ、観点（既定は全部）を特定する。
   スコープが判別できないときだけ候補を挙げて確認する（例:「この変更だけ？ 機能全体？ プロジェクト全体？」）。
2. **材料の取得**（読み取りのみ）: スコープ別に材料を集める。
   - PR:
     ```bash
     gh pr view <num> --repo <owner/repo> --json title,body,url,baseRefName,headRefName,headRefOid,files,additions,deletions
     gh pr diff <num> --repo <owner/repo>
     gh pr view <num> --repo <owner/repo> --comments   # 既存コメント（重複回避）
     ```
     repo が不明なら cwd の `git remote` から解決し、無ければ `owner/repo` を確認する。
   - 差分: 上記「差分スコープの既定解釈」に従い `git diff HEAD` / `git diff <base>...HEAD`。
   - 機能: 対象ファイル群を Read し、`grep`/`git grep` で呼び出し元・依存先を辿る。
     追跡は**原則1ホップ**（直接の呼び出し元と直接の依存先）。[must] 級の疑いを確定させるのに
     必要な場合のみ、その指摘に限りもう1ホップ辿ってよい。
   - プロジェクト全体: ディレクトリ構成を把握し、依存マップと代表フローを抽出（architecture-review.md 参照）。
3. **プロジェクト規約の読み込み**: 対象リポジトリの CLAUDE.md / linter・formatter 設定
   （`.eslintrc*`, `ruff.toml`, `.editorconfig` 等）/ CONTRIBUTING.md を確認する。ローカルに無ければ
   `gh api repos/<owner>/<repo>/contents/<path>` で取得。規約が無ければ規約系の指摘は ASSUMPTION と明記。
4. **下ごしらえ**: 自動生成物（lock ファイル・ビルド成果物・生成コード）はレビュー対象から除外。
   次の目安（初期値。実運用で調整）を超える対象はファイル/モジュール単位に分割してレビューする —
   差分/PR: **3,000 行超**、機能: **30 ファイル超**（プロジェクト全体は常に構造単位）。
   分割しても見きれない場合は重点を選び、「未レビュー範囲」を必ず明示する。
5. **観点別レビュー**: 適用順は駆動で変える。
   - 変更駆動（PR / 差分）: 機能 → セキュリティ → パフォーマンス → アーキテクチャの順。
   - 現状駆動（機能 / プロジェクト全体）: **先にアーキテクチャ観点**で全体構造（依存マップ・代表フロー）を
     掴み、そこで得た高リスク箇所に対して機能 → セキュリティ → パフォーマンスをサンプリング適用する。
   既存レビューコメント・過去の報告書と重複する指摘は除外 or「既出」と注記。
6. **提示**: 後述の出力フォーマットでチャットに提示する。**この時点では PR に一切書き込まない。**
   報告書として残す場合の保存先とファイル名:
   - 対象が `projects/<name>/` 配下 → `projects/<name>/reviews/YYYY-MM-DD_<scope>_<対象slug>.md`
   - それ以外（任意のリポジトリ）→ 対象リポジトリ内には勝手に置かず、保存先をユーザーに確認する。
   - 同じ対象の過去報告書が保存先にあれば先に読み、解消済み/継続中/新規を区別して重複指摘を避ける。
7. **投稿（PR スコープ・明示承認後のみ）**: ユーザーが投稿を承認したら実行する。
   ```bash
   # 総評（PR 全体へのコメント）
   gh pr review <num> --repo <owner/repo> --comment --body "<総評>"
   # インラインコメント（headRefOid を commit_id に使う）
   gh api repos/<owner>/<repo>/pulls/<num>/comments \
     -f body="<指摘本文>" -f commit_id="<headRefOid>" -f path="<file>" -F line=<line> -f side=RIGHT
   ```
   承認 (`--approve`) / 変更要求 (`--request-changes`) は、ユーザーがそう明示した場合のみ使う。
8. **完了報告**: 投稿時はコメント URL、下書き/報告書のみの場合はサマリ表（観点×重要度）と未レビュー範囲を提示する。

## 出力フォーマット

```markdown
## コードレビュー: <対象> — スコープ: <PR/差分/機能/プロジェクト全体>

### サマリ
- 対象概要: <1〜2 文（PR タイトル / 機能名 / 対象ディレクトリ など。差分なら採用した base も）>
- 指摘件数（観点×重要度）:
  | 観点 | must | imo | nit | ask |
  |---|---|---|---|---|
  | 機能 | n | n | n | n |
  | セキュリティ | n | n | n | n |
  | パフォーマンス | n | n | n | n |
  | アーキテクチャ | n | n | n | n |
- 適用観点: <実施した観点。絞った場合はその理由（ユーザー指定）>
- 未レビュー範囲: <なし | 除外・分割・サンプリングした範囲と理由>

### 機能
- [must] `src/app.py:42` — <指摘>。<根拠>。提案: <修正案>

### セキュリティ
- [must] `src/api.ts:88` — <指摘>...

### パフォーマンス
- [imo] `src/repo.go:120` — <指摘（例: N+1 クエリ）>...

### アーキテクチャ・原則
- 意図の一目瞭然性: <評価（◎/○/△/× と根拠。architecture-review.md の尺度を使う）>
- 依存関係: <過剰結合・循環・レイヤ違反・変更波及の広さの所感（可能なら依存の要約）>
- [imo] `src/service.py:10` — <責務・抽象・命名・規約に関する指摘>...
- [must] `src/billing/ ↔ src/order/`（代表: `billing/charge.py:12`） — <構造レベルの指摘はモジュール対 + 代表参照行で書く>

### 総評（PR に投稿する場合の本文）
<ねぎらい + 全体所感 + must の要約>
```

- 行単位の指摘は「観点見出し配下 + 重要度ラベル + `file:line`」を揃える。行が一意に定まらない
  構造レベルの指摘（循環依存・レイヤ違反・構成）は「モジュール対/パス + 代表参照行1つ」で書く。
- 該当なしの観点は「指摘なし」、サンプリングした観点は「サンプル範囲内で指摘なし」と書き分ける。
- アーキテクチャ観点は個別の [ラベル] 指摘に加え、**意図の一目瞭然性・依存関係の所感**を必ず文章で述べる。
- 良い点をレビューの信頼のために総評で1つ以上言及する。

## 品質・安全性

- **無承認投稿の禁止**: PR への書き込み（`gh pr review` / `gh api` の POST）は、下書き提示後に
  ユーザーが明示承認した場合のみ実行する。デフォルトは常に下書き。ローカルスコープでは投稿は発生しない。
- **読み取り専用**: レビューは対象コードを変更しない。依存解析ツールを対象リポジトリに**インストールしない**
  （`npx`/`uvx` 等の一時実行のみ可。使えなければ grep ベースで代替する）。
  修正の適用を求められたら別作業として切り出す。
- **ASSUMPTION の明記**: 対象外のコンテキストや規約を推測で補った指摘には `(ASSUMPTION: ...)` を明記する。
  確信の持てない指摘は [must]/[imo] にせず [ask] に落として質問形式にする。
- **機密を含めない**: トークン・鍵などのシークレット様文字列を見つけたら、値は伏せたまま
  「機密の混入疑い」として [must] で指摘する。レビュー出力に機密値を転記しない。
- **創作しない**: 対象コードと取得した規約・コメントに無い事実を根拠にしない。根拠は引用できる形で添える。
- **既存スタイル尊重**: 規約観点の指摘は対象リポジトリの既存規約に基づくものに限り、好みを持ち込まない。
