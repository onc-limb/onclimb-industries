# 比較軸の定義（固定 7 軸）

技術選定の比較表で使う軸の定義。**軸の増減・改名はしない**（過去の選定記録と横並び比較する
ため）。テーマ固有の観点は「要件適合」の内訳として扱う。

## 評価記号

| 記号 | 意味 |
|---|---|
| ◎ | 強み。選定理由になり得る |
| ○ | 十分。問題なし |
| △ | 懸念あり。条件付きで許容 |
| × | 不適合。見送り理由になり得る |
| 未確認 | Web で根拠を確認できなかった（推測で埋めない） |

各セルは `記号 + 根拠一言 [出典番号]` の形式で書く。例: `○ 月次でリリース継続 [3]`

## 各軸の定義と確認する一次情報

### 1. 成熟度 (Maturity)

本番採用に足る実績・安定性があるか。

- 確認するもの: 安定版（1.0+）の有無、初リリースからの年数、著名な採用実績、
  破壊的変更の頻度（リリースノート / CHANGELOG）
- ◎ の目安: 安定版が複数年運用され、採用実績が公表されている
- × の目安: alpha/beta のみ、または「production-ready ではない」と公式が明言

### 2. メンテ状況 (Maintenance)

**現在も**活発に保守されているか。鮮度が命の軸。必ず現物を確認する。

- 確認するもの: 最終リリース日・最終コミット日（リポジトリを WebFetch で直接確認）、
  Issue/PR への反応、メンテナ体制（個人 / 組織）、deprecated・アーカイブ宣言の有無
- ◎ の目安: 直近 3 か月以内にリリースまたは活発なコミットがある
- × の目安: 1 年以上更新なし、アーカイブ済み、公式が後継への移行を案内している

### 3. ライセンス (License)

制約なく使えるライセンスか。**必ず LICENSE ファイルまたは公式表記を現物確認**する
（近年ライセンス変更（例: BSL 化）の事例が多く、古い記事の記載は信用しない）。

- 確認するもの: ライセンス種別、デュアルライセンス / 商用条項の有無、直近のライセンス変更歴
- ◎ の目安: MIT / Apache-2.0 / BSD 等の permissive
- △ の目安: copyleft（GPL 系）で利用形態によっては要検討
- × の目安: ユーザーの制約（ヒアリングで確定）に抵触する

### 4. 学習コスト (Learning Cost)

チームが実用レベルに達するまでの負荷。

- 確認するもの: 公式ドキュメント・チュートリアルの充実度、API の規模、
  既存スタック・既知パラダイムとの近さ（ヒアリング結果と突き合わせる）
- ◎ の目安: 既存スキルの延長で使え、公式チュートリアルで主要ユースケースを覆える
- × の目安: 独自パラダイムでドキュメントも薄い

### 5. エコシステム (Ecosystem)

周辺ライブラリ・情報・コミュニティの厚み。

- 確認するもの: プラグイン / 連携ライブラリの数、コミュニティ規模の指標
  （スター数・ダウンロード数は取得したページの数値のみ記載）、Q&A・記事の見つかりやすさ
- ◎ の目安: 主要ユースケースに公式・準公式の連携がそろっている
- × の目安: 周辺が乏しく、詰まったとき自力でソースを読むしかない

### 6. 性能 (Performance)

要件に対して十分な性能か。絶対性能ではなく**要件との相対**で評価する。

- 確認するもの: 公式ベンチマーク・第三者ベンチマーク（出典と測定条件を必ず併記）、
  バンドルサイズ / メモリフットプリント等、要件に効く指標
- 注意: ベンチマークは条件依存。数値を引くときは測定条件と出典を併記し、
  比較可能でない数値同士を並べない
- 該当要件がないテーマでは「○（要件上ボトルネックにならない）」で足りる

### 7. 要件適合 (Requirement Fit)

ヒアリングで確定した must / nice-to-have をどれだけ満たすか。**最重要軸**。

- 確認するもの: must ごとの充足（公式ドキュメントで機能の有無を確認）、nice の充足数
- ルール: **must 未達が 1 つでもあれば、他の軸が全て ◎ でも見送り**とする
- セルには「must n/n, nice m/m」の形式で内訳を書く

## 推奨・次点・見送りの決め方

1. must 未達の候補を見送りに落とす（理由 1 行）
2. 残った候補を「要件適合 → メンテ状況 → ライセンス」の優先順で比較し、推奨 1 件を決める
3. 次点には「どの条件が変われば次点が逆転するか」を 1 行添える
4. 迷ったら断定せず、両論を残して判断材料の差分を明示する（最終判断はユーザー）
