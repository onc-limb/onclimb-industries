# edith-tech-selection-research — 技術選定リサーチ

ライブラリ / フレームワーク / ミドルウェア等の**技術選定に特化した調査スキル**。
要件をヒアリングで確定し、Web 調査で候補を固定 7 軸で評価し、
**出典付き比較表 + 推奨・次点・見送り理由**を固定フォーマットで返し、選定記録を永続化する。

## 役割

- 「毎回ゼロからの選定調査」をなくす。比較の観点・出力の形・記録の置き場を固定し、
  調査本体（候補の洗い出しと評価の根拠集め）だけに集中できるようにする。
- 選定の**判断根拠を追跡可能な形で残す**。後日「なぜこれを選んだか」を記録ファイル 1 枚で説明できる。

## deep-research との棲み分け

| | 汎用 `deep-research` | 本スキル |
|---|---|---|
| 対象 | 任意のテーマ | 技術選定（ライブラリ / FW / ツール比較）のみ |
| 比較観点 | 毎回テーマに合わせて設計 | 固定 7 軸テンプレート（[references/comparison-axes.md](references/comparison-axes.md)） |
| 事前確認 | 質問が曖昧なら 2〜3 問 | 要件ヒアリングが必須ステップ（must / nice / 制約を確定） |
| 出力 | 自由形式の引用付きレポート | 比較表 + 推奨 1 / 次点 1 / 見送り理由 の固定フォーマット |
| 永続化 | なし（チャット返信） | `tech-selection/` に選定記録を必ず保存 |

技術選定**以外**の調査（市場動向・一般的な事実調査など）を頼まれたら、本スキルではなく
deep-research を使う。逆に「どれを採用すべきか」という結論が必要な技術比較は本スキルが担当する。

## トリガー

| ユーザー発話の例 | 動作 |
|---|---|
| 「Next.js と Remix を比較して技術選定して」 | 指名候補あり。候補を補完しつつ比較 |
| 「この要件に合う ORM を選定調査して」 | カテゴリ指定。候補洗い出しから実施 |
| 「moment.js の代替を比較表にして推奨を出して」 | 移行元あり。alternatives 調査 |
| 「状態管理ライブラリのおすすめを出典付きで比較して」 | カテゴリ指定 |
| 「前に選定した○○の記録を見せて」 | 過去記録の参照（`tech-selection/` を検索して提示） |

## データ配置

- 選定記録: `<repo>/tech-selection/<YYYY-MM-DD>-<slug>.md`（slug はテーマの英語 kebab-case）
- 雛形: [templates/decision-record.md](templates/decision-record.md)

## 標準フロー

1. **過去記録の確認**: `tech-selection/` に同カテゴリの記録があれば提示し、
   再調査か記録の更新かをユーザーに確認する。
2. **要件ヒアリング**: `AskUserQuestion` で以下を確定してから調査に入る（省略しない）。
   - 用途・解決したい課題（何に使うか）
   - 技術的制約（言語 / ランタイム / 既存スタックとの整合）
   - ライセンス制約・商用利用の有無
   - チームのスキル・学習に割ける時間
   - **must（未達なら見送り）と nice-to-have の区別**
3. **候補の洗い出し**: WebSearch を複数クエリで実施する
   （テーマ直接検索 / 「<テーマ> alternatives」 / 「<候補名> vs」 / 指名候補の周辺）。
   本表に載せる候補は**最大 5 件**。must 未達・明らかなメンテ停止はこの時点で見送りリストへ。
4. **固定 7 軸で評価**: [references/comparison-axes.md](references/comparison-axes.md) の定義に従い、
   候補ごとに一次情報（公式ドキュメント / リポジトリ / リリースノート / LICENSE）を WebFetch で確認する。
   - 各評価セルには出典番号を付け、出典一覧に **URL と取得日** を記録する
   - Web で確認できなかった項目は「未確認」と書く（推測で埋めない）
5. **比較表 + 推奨の生成**: [templates/decision-record.md](templates/decision-record.md) の構成で
   選定記録を `tech-selection/` に Write し、同じ内容の要約（比較表 + 推奨・次点・見送り）を
   チャットで提示する。
6. **完了提示**: 記録ファイルのパスと、調査でカバーできなかった範囲を明示して終了。

## 比較軸の定義

固定 7 軸。定義・確認する一次情報・評価記号は [references/comparison-axes.md](references/comparison-axes.md) を正とする。

| 軸 | 一言で |
|---|---|
| 成熟度 | 本番採用に足る実績・安定版があるか |
| メンテ状況 | 現在も活発に保守されているか |
| ライセンス | 制約なく使えるライセンスか |
| 学習コスト | チームが習得するまでの負荷 |
| エコシステム | 周辺ライブラリ・情報・コミュニティの厚み |
| 性能 | 要件に対して十分な性能か |
| 要件適合 | ヒアリングで確定した must / nice をどれだけ満たすか |

軸の増減はしない（過去の選定記録と横並び比較できなくなるため）。テーマ固有の観点は
「要件適合」の内訳、または記録の備考欄で扱う。

## 出力フォーマット

チャット返信・記録ファイルとも、以下を必ず含める。

1. **確定した要件**（must / nice / 制約）
2. **比較表**: 行 = 候補（最大 5）、列 = 固定 7 軸。各セルは `評価記号 + 根拠一言 [出典番号]`
3. **推奨（1 件）**: 選定理由を要件と紐付けて 3 行以内で
4. **次点（1 件）**: 「どういう条件なら次点が逆転するか」を 1 行添える
5. **見送り**: 候補ごとに理由 1 行（must 未達 / メンテ停止 / ライセンス不適合 など）
6. **出典一覧**: `[n] タイトル — URL（取得日: YYYY-MM-DD、一次 / 二次）`。**URL と取得日は必須**
7. **カバーできなかった範囲**: 調べきれなかった候補・確認できなかった項目を明示

## 品質・安全性（persona: edith 準拠）

- **出典明示・一次情報優先**: 評価の根拠は公式ドキュメント / 公式リポジトリ / リリースノートを最優先。
  二次情報（ブログ・まとめ記事）は補助に限り、公開日が古いもの（目安 2 年超）は根拠にしない。
- **鮮度の担保**: 全出典に取得日を付ける。メンテ状況は「最終リリース日 / 最終コミット」を必ず現物で確認する。
- **ハルシネーション防止**: バージョン・スター数・ベンチマーク値などの数値は、WebFetch で取得した
  ページの記載のみ採用する。確認できなかったセルは「未確認」とし、記憶や推測で埋めない。
- **不確実性の明記**: 確証のない評価は「〜と思われる（要検証）」と書き、事実と解釈を区別する。
- **カバー範囲の明示**: 探索したクエリ・見た範囲と、カバーできなかった範囲を記録に残す
  （暗黙の打ち切りをしない）。
- 本スキルの出力は**選定判断の材料**であり、最終判断はユーザーが行う。断定的な保証表現はしない。
- 機密情報（案件名・API キー等）を記録ファイルに書かない。要件は一般化した表現で記録する。
