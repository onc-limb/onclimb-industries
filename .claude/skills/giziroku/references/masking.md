# マスキング規約（議事録向け）

worklog の規約（`.claude/skills/worklog/config/redaction.yaml`）を**定義として流用**し、
運用は giziroku 側で独立させる。出典: [[knowledge-base-idea]] §2 / [[work-log-system-instructions]]。

> 原則: 「private だから機密を入れてよい」ではない。`giziroku/` は git 管理外だが、
> それは二重の安全網にすぎない。公開範囲 external / unknown では本文に機密を素で残さない。

## いつ適用するか

- 公開範囲 `external`（社外共有可）／ `unknown`（未確認 → 安全側で external 扱い）→ **適用**。
- 公開範囲 `internal`（社内限定）→ 明確な秘匿情報（鍵・トークン・接続情報・個人情報）のみ最小適用。

## 置換形式

`<REDACTED:種別>` に置き換える。種別は読み手が文脈を失わない粒度にする。

| 種別 | 対象の例 |
|---|---|
| `顧客名` | 参画先・取引先の企業名 |
| `氏名` | 個人名（参加者で匿名指定された人、本文に出る第三者）|
| `システム名` | 社内固有のサービス・プロダクト名 |
| `APIキー` / `トークン` | `sk-…` / `ghp_…` / `xox…` / JWT など |
| `接続文字列` | `scheme://user:password@host` |
| `メール` | メールアドレス |
| `認証情報` | `password=` / `secret=` / `api_key=` の値 |

## 議事録ならではの注意

- **参加者欄**: external のとき、確定した実名はそのまま出さず役割主体にする
  （例: `本人(PM) / <REDACTED:氏名>(実装)`）。匿名指定者は「他N名(匿名)」。
- **誤変換補正メモとの衝突**: 補正メモに固有名を出すと伏字の意味が無くなる。
  外部固有名は補正メモにも `<REDACTED:…> → <REDACTED:…>` か、一般語に置換して載せる。
- **一貫性**: 同一対象は同一の伏字種別で統一する（同じ顧客名が `顧客名` と `システム名` に割れない）。

## roster / glossary の扱い

- `config/roster.yaml`（過去参加者名簿）は固有名を含むため、**共有ファイル自体を公開対象から外す**。
- `config/glossary.yaml`（誤変換辞書）は一般語が主だが、顧客固有語が混じりうるので同様に扱う。
