# jocasta-eval-autopilot — 評価改善ループの自動運転

designer / runner / improver を **auto モード**で連結するオーケストレータ。
persona は [`personas/jocasta.md`](../../../personas/jocasta.md) に従う
（特に「事実ベースの原則」と「実行モード」。auto でも原則は一切緩めない）。

## 位置づけ

- 各ステップの実体は既存 3 スキルと evalkit.py そのもの。**autopilot 専用の隠れた処理を持たない**。
  成果物も通常と同じ場所（`runs/` / `improvements/` / ケースの `revision_notes`）に残るため、
  途中からいつでも interactive（単体スキルでの壁打ち）に切り替えられる。
- ユーザーが自動実行を明示的に指示したときのみ起動する。

## 開始時の合意（1 回だけユーザーに確認する）

auto に入る前に、以下を確認・合意してから走り出す（走り出した後は停止条件まで質問しない）。

1. **対象アプリ**と、dataset・config が現状の意図と合っているか
2. **停止条件**: pass 率の目標（既定: 1.0）と最大イテレーション数（既定: 3）
3. **アプリ側改修の許可範囲**: 自動で書き換えてよいファイルを明示的に列挙してもらう
   （例: `llm-eval-data/<app>/app/system-prompt.md` のみ）。**列挙されなかったファイルは
   絶対に変更せず、改善案を「提案」として記録するに留める**。対象プロジェクト
   （`projects/` 配下等）のコードは、ユーザーがパスを明示した場合のみ対象にできる。

## ループ（1 イテレーション）

1. **測定**: `evalkit.py collect --note "autopilot iter <k>: <直前に適用した変更>"` → `judge` →
   `score` → `report`（初回は現状のベースライン取得。直前 run が現状と一致していればスキップ可）。
2. **評価・切り分け**（improver の auto モード）: ワーストケース・fail ケースを
   ジャッジ→データセット→アプリの順で切り分け、根拠付きの改善案リストと保留リストを
   `improvements/<日付>-autopilot-iter<k>.md` に記録する。
   - 事実関係が論点のケース（ルール・数値など）は WebSearch / WebFetch で一次情報の裏取りを行う。
     裏取りできなければその論点の改善は保留行き。
3. **適用**: 確定改善案から **1 種類だけ**選んで適用する（compare の解釈可能性を守るため、
   データセット改訂とアプリ改修を同一イテレーションで混ぜない）。
   - 優先順位: ①ジャッジ起因（基準の明確化） ②データセット起因（designer auto で改訂 +
     revision_notes 記録） ③アプリ起因（許可範囲のファイルのみ改修）
   - データセットを改訂した場合、次の測定はその変更後のベースラインとして扱い、
     改訂前の run と数値を直接比較しない（レポートにその旨を書く）。
4. **効果確認**: 新 run を取り `compare` する。スコアが動いたケースは reason を読み、
   **意図した理由で動いたか**を確認して記録する（ジャッジの揺らぎによる見かけの改善を
   成果と数えない。改善条件: 対象ケースの reason が適用した変更と対応していること）。

## 停止条件（いずれかで停止）

- pass 率が目標に到達した
- 根拠の揃った改善案が尽きた（保留リストだけが残った）
- 同じ課題への適用を 2 イテレーション続けても改善が確認できなかった（堂々巡り防止）
- 最大イテレーション数に到達した

## 終了時のレビュー用サマリ

`llm-eval-data/<app>/autopilot/<YYYYMMDD-HHMMSS>-summary.md` に以下を書き、チャットでも要約提示する。

1. **スコア推移表**: イテレーションごとの overall / pass 率 / メトリクス平均（run ID 付き。
   データセット改訂を挟んだ箇所は「基準変更」と明記して比較を分断する）
2. **適用した変更の全リスト**: 変更内容・根拠（引用 / URL + 取得日）・対応する improvement record
3. **保留リスト（要ユーザー判断）**: （推測）由来・未確認・許可範囲外で適用しなかった案
4. **未解決の課題**と、次に interactive で壁打ちすべき論点の提案

## 品質・安全性（persona: jocasta 準拠）

- （推測）を根拠にした変更・裏取りできない事実を含む変更は、自動適用しない（保留へ）。
- 許可範囲外のファイルは読み取り・引用はしても変更しない。
- 各イテレーションの成果物はすべて通常の場所に残し、run は不変・改訂は revision_notes 付きで行う。
  autopilot が何をしたかは、サマリを読まなくても個々の記録から追跡できる状態を保つ。
- 外部エンドポイント（http アダプタ）・対象プロジェクトのコードに触れる構成では、
  開始時の合意でその旨を明示してから実行する。
