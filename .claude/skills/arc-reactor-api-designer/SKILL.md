# api-designer — API 契約の設計と一貫性チェック (arc-reactor)

REST / GraphQL API の設計をユーザーと壁打ちしながら作り、
スキーマ（OpenAPI 3.x YAML / GraphQL SDL）と設計記録を出力するスキル。
「エンドポイントごとに流儀がバラバラ」を防ぐため、**既存 API との一貫性チェック**を
設計の入口（規約の抽出）と出口（セルフレビュー）の両方に置く。

> 大原則: **既存規約が正。** 新しい設計は既存 API の流儀に揃える。
> 既存規約を勝手に「改善」せず、逸脱するときは必ず理由を付けて提案し、ユーザーが決める。

persona: [`personas/arc-reactor.md`](../../../personas/arc-reactor.md) に従う
（応答は日本語 / コード・識別子は英語 / 推測は ASSUMPTION 明記）。

## トリガー

| ユーザー発話の例 | 動作 |
|---|---|
| 「この機能の API を設計して」 | フルフロー（規約検出 → ヒアリング → 設計 → スキーマ → セルフレビュー） |
| 「新しいエンドポイントを既存 API と揃えて設計して」 | フルフロー。一貫性チェックを重点に |
| 「OpenAPI のスキーマを書いて」 | 設計が固まっていればスキーマ生成から。曖昧ならヒアリングに戻る |
| 「REST API 設計の壁打ちに付き合って」 | ヒアリングと設計案の往復を厚めに |
| 「この API 設計が既存と一貫してるかチェックして」 | 規約検出 + セルフレビュー（フロー 1 と 5）のみ |
| 「GraphQL のスキーマを設計して」 | GraphQL SDL で同じフロー |

ユーザーが明示的に依頼したときだけ起動する（コードを読んだついでに自動起動しない）。

## 標準フロー

### 1. 既存規約の検出

対象リポジトリの既存 API を読み、現行の流儀を**規約表**として抽出する。読む場所:
ルーティング定義、既存の OpenAPI / GraphQL スキーマファイル、コントローラ / ハンドラ、
エラーハンドリングのミドルウェア。

規約表の観点（各行に根拠 `ファイル:行` を添える）:

| 観点 | 例 |
|---|---|
| 命名 | リソースの複数形 / 単数形、パスとフィールドのケース（snake / camel / kebab） |
| エラーレスポンス形式 | エラーオブジェクトの構造、エラーコードの体系 |
| ページネーション方式 | offset / cursor / page、パラメータ名 |
| 認証方式 | Bearer / API キー / セッション、ヘッダ名 |
| バージョニング | パス（/v1/）/ ヘッダ / なし |
| 日時形式 | ISO 8601 / Unix time、タイムゾーンの扱い |

- 規約表はユーザーに提示して認識合わせをする。コードから読み取れず推測した箇所は
  ASSUMPTION と明記する。
- 既存 API 内で流儀が割れている場合は、その事実を規約表に併記し、どちらに揃えるかを
  ユーザーに確認する（勝手に多数派へ寄せない）。
- **新規プロジェクトで既存 API が無ければ**、
  [references/api-design-checklist.md](references/api-design-checklist.md) の標準案を
  規約表の形で提示し、合意を取ってからそれを本セッションの規約とする。

### 2. 要件ヒアリング

対話で 2〜3 問ずつ聞く（一度に全部並べない）。確定させる項目:

- リソースと操作（何を、どう CRUD / 検索 / 一括操作するか）
- 利用者（自社フロントエンド / 外部公開 / モバイル — エラー詳細度や互換性方針に効く）
- 認証・認可の粒度（エンドポイント単位か、リソースオーナー判定が要るか）
- 互換性制約（既存クライアントの有無。有るなら破壊的変更の許容範囲）

発話から自明な項目は聞き直さず、確認だけで済ませる。

### 3. 設計案の提示

スキーマを書く前に、エンドポイント一覧表で設計の骨格を合意する。

| メソッド | パス | 目的 | 認可 | 主なステータスコード |
|---|---|---|---|---|

- GraphQL の場合は Query / Mutation / 主要型の一覧表に読み替える。
- 判断が分かれる点（ページネーション方式の変更、バルク操作の表現、既存規約からの逸脱等）は、
  選択肢とトレードオフを添えてユーザーに確認する。決めるのはユーザー。
- 既存エンドポイントの変更（破壊的変更）を含む場合は、この段階で互換性影響
  （壊れるクライアント操作、移行パス）を明記する。

### 4. スキーマ生成

合意した設計を OpenAPI 3.x YAML（または GraphQL SDL）に落とす。

- リクエスト / レスポンスの例、エラーレスポンスの例を含める（`examples` を使う）。
- 既存のスキーマファイルがあれば、新規ファイルを作らずそれに**追記する形式**で出す
  （既存の `components` / 共通スキーマを再利用し、重複定義しない）。
- スキーマ内の識別子・description の言語は既存スキーマの流儀に合わせる。

### 5. 一貫性セルフレビュー

生成した設計をフロー 1 の規約表と観点ごとに突き合わせ、逸脱箇所を表にする。

| # | 観点 | 既存規約 | 今回の設計 | 分類 | 理由 / 修正方針 |
|---|---|---|---|---|---|

- 分類は 2 値: **意図的（理由付き）** — フロー 3 でユーザーが選んだ逸脱。理由を記載 /
  **要修正** — 意図しない逸脱。スキーマを修正してから完了する。
- 逸脱ゼロでも「確認した観点」を列挙した表を添付する（チェックした事実を残す）。
- レビュー観点の詳細は [references/api-design-checklist.md](references/api-design-checklist.md) を使う。

### 6. 保存

- スキーマ: プロジェクトに既存の置き場（`openapi/`, `api/`, `schema/` 等）があればそこへ。
  無ければ `<repo>/docs/api/` を提案する。
- 設計記録（規約表・エンドポイント一覧・判断とトレードオフ・セルフレビュー表・ASSUMPTION 一覧）:
  対象プロジェクト配下 `<repo>/docs/api/design-<topic>-<YYYY-MM-DD>.md` 等に置く。
- 対象リポジトリへの書き込みになるため、**保存・書き込み前に必ずユーザーに確認**する。
- 完了時はチャットでエンドポイント数・逸脱の件数と内訳・未確定の ASSUMPTION を要約する。

## 品質・安全性

- **実装コードの生成は範囲外。** スキーマと設計記録まで。ハンドラやクライアントの実装は
  通常の開発作業として別途行う。
- **既存規約を勝手に「改善」しない。** 現行の流儀に問題を見つけても、黙って直した設計を
  出さない。逸脱には必ず理由とトレードオフを付けて提案し、ユーザーが採否を決める。
- 推測（既存コードから読み取れなかった規約、要件の補完）は ASSUMPTION として
  規約表・設計記録に明記する。
- 破壊的変更（既存エンドポイントのパス・パラメータ・レスポンス構造の変更、削除）を含む場合は、
  互換性影響と移行パスを設計記録に必ず明記する。
- 対象リポジトリへの操作は原則読み取りのみ。書き込みはスキーマ・設計記録の保存に限り、
  保存前に確認する。
- スキーマ・設計記録はプロジェクト固有情報のため、対象プロジェクトのディレクトリ配下に閉じる
  （当リポジトリや他プロジェクトに持ち出さない。CLAUDE.md の分離ルールに従う）。
- スキーマの例に実データ・機密情報（実在のトークン、個人情報）を書かない。ダミー値を使う。
- 設計観点の改善（新しい観点・標準案の見直し）は references への追記で行う。
