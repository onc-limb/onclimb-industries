# api-design-checklist — 設計観点と標準案

arc-reactor-api-designer が使う設計観点のカタログ。用途は 2 つ。

1. **規約検出の観点表**（フロー 1）: 既存 API から各観点の現行の流儀を読み取るときの見出し。
2. **標準案**（既存が無い新規プロジェクト向け）: 各観点の推奨デフォルト。
   そのまま押し付けず、規約表の形で提示して合意を取ってから使う。

一貫性セルフレビュー（フロー 5）でも、この観点一覧を「確認した観点」の母集合として使う。
観点を追加・修正したくなったら、SKILL.md ではなくこのファイルに追記する。

## 1. リソース命名

- 検出ポイント: パス上のリソース名が複数形か単数形か、ケース（kebab / snake / camel）、
  ネストの深さ（`/users/{id}/orders` を使うか、フラットに `/orders?user_id=` か）。
- 標準案: リソースは複数形の名詞（`/users`, `/orders`）。パスは kebab-case、
  クエリパラメータと JSON フィールドは snake_case か camelCase のどちらかに統一
  （クライアント言語が JS 中心なら camelCase が摩擦が少ない）。
  ネストは 1 段まで。動詞をパスに入れない（`/users/create` ではなく `POST /users`）。
  CRUD に収まらない操作はサブリソースかアクションエンドポイント（`POST /orders/{id}/cancel`）で
  表し、どちらを使うかをプロジェクト内で統一する。

## 2. HTTP メソッドとステータスコードの使い分け

- 検出ポイント: 更新に PUT / PATCH のどちらを使っているか、作成の成功が 200 か 201 か、
  削除の成功が 204 か 200 か、バリデーションエラーが 400 か 422 か。
- 標準案:
  - GET = 取得（副作用なし）/ POST = 作成・非冪等な操作 / PUT = 全置換 /
    PATCH = 部分更新 / DELETE = 削除。
  - 201 Created（作成。`Location` ヘッダ付き）/ 204 No Content（削除・返す本文が無い更新）/
    400（構文・型のエラー）/ 401（未認証）/ 403（認可拒否）/ 404（不在。認可拒否を 404 で
    隠すかは方針を決めて統一）/ 409（競合）/ 422（意味的バリデーションエラー）/
    429（レートリミット）。
  - 400 と 422 の使い分けはプロジェクトで割れやすい。既存に合わせ、無ければ
    「形式エラーも業務エラーも 422 に寄せない、まとめるなら 400」のどちらかを最初に決める。

## 3. エラーレスポンス形式の統一

- 検出ポイント: エラーボディの構造（`error` / `errors` / `message` のキー名）、
  機械可読なエラーコードの有無と体系、フィールド単位のバリデーションエラーの表現。
- 標準案: RFC 9457 (Problem Details) 準拠か、それに近い固定構造を全エンドポイントで共通にする。

  ```json
  {
    "type": "https://example.com/errors/validation-failed",
    "title": "Validation Failed",
    "status": 422,
    "detail": "email is not a valid address",
    "errors": [{ "field": "email", "code": "invalid_format" }]
  }
  ```

  最低限、機械可読コード（`code` / `type`）と人間向けメッセージを分ける。
  スタックトレース・内部実装の詳細は外部公開 API のエラーに含めない。

## 4. ページネーション方式

- 検出ポイント: offset/limit か cursor か page/per_page か、パラメータ名、
  総件数・次ページ情報の返し方（ボディ内メタか `Link` ヘッダか）。
- 標準案: 一覧が増え続けるデータ（フィード・ログ）は cursor ベース
  （`?cursor=...&limit=`、レスポンスに `next_cursor`）。管理画面的な固定データは
  offset/limit でも足りる。**1 つの API 内で方式を混在させない**のが最優先で、
  既存に offset があるなら新エンドポイントだけ cursor にする場合は逸脱として理由を残す。
  limit には上限（例: 100）を決めてスキーマに書く。

## 5. フィルタ・ソート

- 検出ポイント: フィルタのパラメータ形式（`?status=active` / `?filter[status]=` / 検索 DSL）、
  ソートの表現（`?sort=-created_at` / `?order_by=created_at&order=desc`）。
- 標準案: 単純フィルタはフィールド名そのまま（`?status=active&user_id=42`）。
  ソートは `?sort=` にカンマ区切り + 降順は `-` 接頭辞（`?sort=-created_at,name`）。
  ソート・フィルタに使えるフィールドはスキーマの enum で明示し、任意フィールドを受けない
  （インデックスの無い列でのソートを外から誘発させない）。

## 6. バージョニング

- 検出ポイント: パスに `/v1/` があるか、`Accept` ヘッダや独自ヘッダでのバージョン指定か、
  バージョニング無しか。
- 標準案: 外部公開 API はパスバージョニング（`/v1/`）が運用しやすい。
  自社フロント専用でクライアントとサーバを同時デプロイできるならバージョニング無し + 後方互換維持
  でも成立する。どちらでも「v2 を切る条件は破壊的変更のときだけ」と決めておく。
  既存がバージョニング無しの API に新規だけ `/v1/` を導入するのは逸脱。理由付きで提案する。

## 7. 冪等性（冪等キー）

- 検出ポイント: POST 系に `Idempotency-Key` ヘッダの受け口があるか、
  リトライ時の二重実行をどう防いでいるか（ユニーク制約・トークン等）。
- 標準案: 課金・注文・送信など二重実行が実害になる POST には `Idempotency-Key` ヘッダを
  受け付け、同一キーの再送には初回と同じレスポンスを返す仕様をスキーマに明記する。
  PUT / DELETE は本来冪等なので、そうでない実装になっていたら設計の警告対象。

## 8. 認証・認可の表現

- 検出ポイント: 認証方式（Bearer JWT / API キー / セッション Cookie）、ヘッダ名、
  OpenAPI の `securitySchemes` の定義有無、エンドポイントごとの認可要件の書き方。
- 標準案: OpenAPI では `components.securitySchemes` に方式を定義し、各操作の `security` で
  適用を明示する（「認証必須がデフォルト、公開エンドポイントだけ明示的に外す」構成が事故が少ない）。
  認可の粒度（ロール・リソースオーナー）はスキーマに書き切れないため、
  エンドポイント一覧表の「認可」列と設計記録に文章で残す。

## 9. 後方互換性

- 検出ポイント: 過去の変更でフィールド削除・リネームをどう扱ってきたか
  （deprecated 期間を置くか、即削除か）、クライアントが未知フィールドを無視する前提か。
- 標準案: 互換を壊さない変更 = フィールド追加・任意パラメータ追加・enum 値の追加（クライアントが
  未知値を許容する場合のみ）。壊す変更 = フィールドの削除・リネーム・型変更・必須化、
  ステータスコードやエラー構造の変更、URL 変更。壊す変更はバージョンを分けるか、
  deprecated 告知 → 移行期間 → 削除の手順を設計記録に書く。
  スキーマ上は `deprecated: true` を使い、削除予定を description に記す。

## 10. レートリミットの表明

- 検出ポイント: 429 の定義があるか、`RateLimit-*` / `X-RateLimit-*` / `Retry-After` ヘッダを
  返しているか。
- 標準案: 外部公開 API は 429 レスポンスと `Retry-After` ヘッダをスキーマに定義し、
  制限値（例: 60 req/min per API key）を API ドキュメントに明記する。
  制限値の実装・変更はインフラ側の作業なので、本スキルでは「契約としての表明」だけ扱う。

## GraphQL での読み替え

REST 前提の観点を GraphQL SDL に適用するときの対応:

| REST の観点 | GraphQL での対応物 |
|---|---|
| リソース命名 | 型名（PascalCase）・フィールド名（camelCase）・Mutation 命名（動詞 + 名詞: `createOrder`） |
| ステータスコード | トップレベル `errors` と、ドメインエラーを結果 union / `userErrors` フィールドで返すかの方針 |
| エラー形式 | `errors[].extensions.code` の体系統一 |
| ページネーション | Relay 流 Connection（`edges` / `pageInfo`）か単純な `nodes` + cursor か |
| バージョニング | 原則バージョン無し。`@deprecated` ディレクティブで段階的移行 |
| 認可 | resolver 層の責務。スキーマにはディレクティブ（`@auth` 等）があれば流儀に従う |
