# friday-procedure-doc-generator — 開発者向け手順書（runbook / セットアップガイド）

対象コードベース（と直前の会話）を入力に、**実行者がそのままなぞって作業を完了できる手順書**を
1 本生成する。対象は 2 種: 運用手順書（デプロイ・ロールバック・リリース・障害対応・メンテナンス）と、
開発環境構築ガイド（セットアップ手順）。

中心原則は 2 つ。

1. **創作禁止** — 手順書に書いてよいコマンド・操作は、コードベースから確認できた実在のもの
   （根拠パス付き）か、ユーザーが明言したものだけ。「それらしいコマンド」を書いた手順書は、
   実行者が本番環境で叩くため、設計書の創作よりも実害が大きい。確認できないステップは
   **TBD** と書いてヒアリングに回す。
2. **ステップは「操作 → 期待結果」のペア** — 実行者が各ステップで「成功したか」を判断できない
   手順書は、失敗に気づかないまま次へ進ませる。操作だけ書いて期待結果が不明なステップは、
   期待結果側を TBD にして残す（操作ごと消さない）。

## 役割

- 読み手は**他のエンジニア**（チームメンバー・引き継ぎ先・新規参画者・オンコール担当）。
  friday 系（共有ドキュメント系）として、共有・引き継ぎに耐える体裁と事実性を担保する。
- 分類は doc-planner のマトリクスで **説明型 × 辞書的**（読者は作業のたびに手順を引きに来る）。
  読み物的な背景説明は最小限にし、手順本体をすぐ引ける構成にする。
- 「コードから読める操作の棚卸し」は本スキルが担い、「コードに書かれていない運用の実態
  （承認フロー・実行タイミング・手動操作・過去の事故）」はユーザーから対話で引き出す。
- 応答・ドキュメントは日本語。コード・コマンド・識別子は英語（原語）のまま表記する。

### 隣接スキルとの棲み分け

- **friday-design-doc-generator**: プロジェクト概観の README / 設計書 / ADR。README の
  「セットアップ」節が肥大化してきたら、本スキルでセットアップガイドとして別ファイルに
  切り出し、README からリンクする構成を提案する。
- **arc-reactor-env-doctor**: 環境エラーの診断・修復が起点で、副産物として既存手順書の修正
  diff を出す。本スキルはドキュメント生成が起点。env-doctor の対応記録は本スキルの
  「よくあるエラー」節の情報源として使える。
- **arc-reactor-release-readiness-checker**: リリース可否の判定。本スキルは判定ではなく
  「リリース作業のやり方」を文書化する。

## トリガー

| ユーザー発話の例 | 動作 |
|---|---|
| 「デプロイ手順書を作って」 | 運用手順書（deploy）を生成 |
| 「ロールバック手順をまとめて」 | 運用手順書（rollback）を生成 |
| 「開発環境構築の README を書いて」「セットアップガイド作って」 | セットアップガイドを生成 |
| 「この運用作業を手順書に落として」 | 対話で作業内容を確定して運用手順書を生成 |
| 「この手順書が古いからコードに合わせて直して」 | 既存手順書を事実ベースで更新 |
| 「この brief で手順書を作って」「doc-briefs/〜.md から runbook にして」 | doc brief を Read し、Stage 1 として生成 |

自動起動はしない。プロジェクト概観の README（friday-design-doc-generator）や
環境エラーの修復（arc-reactor-env-doctor）の依頼と混同しない。

## doc brief 連携（Stage 1 として動く場合）

前段の [friday-doc-planner](../friday-doc-planner/SKILL.md)（Stage 0）が作成した doc brief が
入力として渡された場合:

- brief で確定済みの項目（目的・読み手・テーマ分解・情報源）は「ユーザーの明言」として扱い、
  標準フロー 1 の同じ内容の再ヒアリングを省略する。
- **骨組み（見出し構成）以降が本スキルの担当**。brief の「テーマ分解」をテンプレートの
  見出し構成に配置し、「情報源」に従って事実収集する。情報源が「ユーザーの頭の中」の項目は
  ヒアリング対象（TBD 一覧行き）として扱う。
- brief に無い・曖昧な項目は従来どおり対話で確認する。創作禁止・TBD のルールは brief の
  有無に関わらず適用する。
- brief 無しの単体起動は従来どおり可（標準フローを最初から行う）。

## 標準フロー

### 1. 種別・対象・実行者の確認

- **種別**を確定する: 運用手順書（`templates/runbook.md`）か、セットアップガイド
  （`templates/setup-guide.md`）か。依頼文から明らかなら確認だけにする。
- **対象**を確定する: 対象リポジトリ / 環境（本番・ステージング・ローカル）/
  対象作業（deploy・rollback・環境構築 等）。
- **実行者**を確認する: 誰が・どんな権限とアクセス（クラウドアカウント・SSH・CI の実行権限）を
  持った状態で実行するか。実行者の前提が手順書の「前提条件」節を決める。
- 運用手順書の場合は**実行の重さ**も確認する: 対象は本番か / 失敗時の影響は何か /
  作業の所要時間とメンテナンスウィンドウの有無。

### 2. 操作の採取（事実収集）

手順の素材になる操作・コマンドを、根拠パス付きで収集する。調査には codebase-reader
（エージェント / スキル）を下請けに使ってよい。見る場所の典型:

- **CI/CD 設定**: `.github/workflows/` / `.gitlab-ci.yml` / `cloudbuild.yaml` 等
  （デプロイの実体が最も正確に書かれている場所）
- **スクリプト類**: `Makefile` / `package.json` の scripts / `scripts/` / `bin/` /
  `Dockerfile` / `docker-compose.yml`
- **IaC**: `terraform/` / CDK / CloudFormation / Helm chart 等
- **manifest とツールバージョン**: `.tool-versions` / `.nvmrc` / `pyproject.toml` /
  `go.mod` 等（セットアップガイドの前提ツール節の根拠）
- **設定ファイル・環境変数**: `.env.example` / 設定読み込みコード（キー名と用途のみ。値は見ない・書かない）
- **既存ドキュメント**: 既存 README・wiki・過去の手順メモ（コードと突き合わせて確認できた
  記述だけ引き継ぐ。矛盾はコード側を優先し、矛盾点をユーザーに報告する）

収集結果は**出典（ファイルパス）付きの操作リスト**としてまとめる。出典のない操作は手順に載せない。
対象が git 管理下なら生成時点の commit hash を取得し、草案冒頭に記録する。

### 3. 草案生成

- 種別のテンプレートに操作リストを流し込む。**見出し構成は固定**。埋められないセクションは
  削除せず `TBD` を入れる（欠落が「調査漏れ」か「情報不足」か読み手に伝わるようにするため）。
- 各ステップは後述「手順ステップの書式」に従い、操作・期待結果・根拠パスを揃える。
- **破壊的・不可逆な操作**（本番 DB への書き込み・削除系・`--force` 類・マイグレーション適用・
  リソース削除）には `⚠️` を付け、直前に確認ステップ（何を見てから実行するか）を置く。
- **運用手順書ではロールバック・中断の検査を必ず行う**: 「途中で失敗したらどうするか」
  「作業をやめて戻すにはどうするか」が埋まらない場合、その旨を TBD として明示し、
  ヒアリングの最優先項目にする。ロールバック手順の無いデプロイ手順書は未完成として扱う。
- 生成後の**自己チェック**: 草案内の各操作について、(a) 出典パスがある / (b) ユーザーの明言に
  基づく / (c) 「(推測)」タグ付き、のいずれかであることを照合する。どれにも該当しない操作は
  削除するか TBD 化する。
- 続けて [`personas/writing-style.md`](../../../personas/writing-style.md) の清書パスを
  **地の文（前提・背景・注意書き）にだけ**通す。手順ステップ・コマンド・期待結果・出典パス・
  TBD・「(推測)」タグ・警告は変えない。手順書は明確さが最優先なので、適用強度は弱め
  （構造宣言と空語の除去に絞る）でよい。

### 4. 実在検証

手順に載せたコマンドを、**安全な範囲で**機械的に検証する。

- **在庫確認（必ず行う）**: 参照しているスクリプト・Make ターゲット・npm script・CI ジョブ名が
  実際にリポジトリに存在するかを照合する。存在しないものは手順から外すか TBD 化する。
- **無害な実行検証（可能なら提案する）**: バージョン表示（`node --version` 等）・`--dry-run` /
  `--check` を持つコマンド・ビルドのみ等、状態を変えない範囲で実行して確認できるものは、
  ユーザーに提案のうえ実行する。
- **実行しないもの**: デプロイ・DB 操作・削除・外部サービスへの書き込みは検証目的でも
  実行しない。セットアップガイドの通し実行検証（クリーンな環境での再現確認）は、
  ユーザーが明示的に依頼した場合のみ行う。

### 5. TBD の集約とヒアリング

- 草案内の TBD を末尾「TBD 一覧」に集約し、**チャットで質問として提示**する。
  典型的にコードから読めない項目: 実行タイミングと承認フロー / 手動操作（管理画面での作業等）/
  ロールバックの実態 / 過去に起きた失敗とその対処 / 環境ごとの差分 / 秘密情報の受け渡し方法。
- ユーザーが回答したら該当箇所に反映する（回答は「ユーザーの明言」として扱う）。
- 回答がない項目は TBD のまま残す。**勝手に埋めない**。
- 保存先を確定して書き込み、パスと TBD の残数を提示して完了する。

## 手順ステップの書式

手順本体は番号付きステップで書く。1 ステップ = 1 操作。各ステップは次の形を守る。

```markdown
### 3. アプリケーションをデプロイする

```bash
make deploy-prod
```

- 期待結果: GitHub Actions の `deploy` ジョブが起動し、約 5 分で success になる
- 確認方法: https://github.com/<org>/<repo>/actions で対象ジョブのステータスを見る
- 根拠: `Makefile`, `.github/workflows/deploy.yml`
```

- **操作**はコピーしてそのまま実行できる形で書く（環境ごとに変わる値は `<PLACEHOLDER>` 表記に
  統一し、前提条件の節で値の取得方法を説明する）。
- **期待結果**が不明なら `期待結果: TBD` と書いて残す。操作ごと消さない。
- 分岐（失敗したら / 環境によって違うなら）はステップ内に「失敗した場合 → 手順 X へ / 中断手順へ」
  の形で行き先を明示する。
- 1 ステップに複数コマンドを詰めない。連続実行が前提のまとまりだけ 1 ブロックにまとめてよい。

## 種別ごとの固定構成

テンプレート本体は `templates/` を参照。見出しの追加は可（対象・読み手に応じて）だが、
既定の見出しの削除・順序変更はしない。

- **運用手順書** (`templates/runbook.md`):
  メタ情報（生成日・対象 commit）/ この手順書の目的と適用場面 / 前提条件 / 実行前チェックリスト /
  手順 / 作業後の確認 / 中断・ロールバック手順 / トラブルシューティング / TBD 一覧 / 参照したソース
- **セットアップガイド** (`templates/setup-guide.md`):
  メタ情報 / 前提ツール / リポジトリの取得 / 依存のインストール / 環境変数・シークレットの設定 /
  ローカルサービスの起動（DB 等）/ 起動と動作確認 / よくあるエラー / TBD 一覧 / 参照したソース

## 出力の置き場所

CLAUDE.md の「プロジェクト固有情報の分離ルール」に従う。

- 出力は**必ず対象プロジェクトのディレクトリ配下**に置く。既定:
  - 運用手順書: `docs/runbooks/<作業名>.md`（例: `docs/runbooks/deploy-production.md`）
  - セットアップガイド: `docs/setup.md`。プロジェクトの README がセットアップ手順込みの
    慣習なら、README の該当節として組み込む形でもよい（ユーザーと確認）。
- 対象リポジトリに既存のドキュメント置き場（`docs/` / `doc/` / wiki 運用等）があればその慣習に従う。
- 書き込み前に出力パスが対象プロジェクト配下であることを確認し、外れていれば保存先を
  ユーザーに確認する。既存ファイルを上書きする場合は、消える記述（コードから確認できず
  引き継げなかったもの）をユーザーに報告してから書き込む。

## 事実と推測の区別ルール

| 区分 | 書き方 | 例 |
|---|---|---|
| 事実（コード由来） | そのまま記述し、根拠パスを添える | 「デプロイは `make deploy-prod` で行う（`Makefile`）」 |
| 事実（ユーザー明言） | そのまま記述する。必要なら「（ヒアリングによる）」と添える | 「本番デプロイは火曜午前に行う（ヒアリングによる）」 |
| 推測 | 文末に **「(推測)」** タグ。コード上の手がかりがある場合のみ | 「`ROLLBACK_TAG` は直前リリースの git tag を指すとみられる (推測)」 |
| 不明 | **「TBD」** と書き、TBD 一覧に集約 | 「ロールバック手順: TBD」 |

- 「(推測)」の操作は**そのまま実行させない**: 推測タグ付きのコマンドは手順本文に置かず、
  TBD 一覧に「候補: 〜 (推測)」として載せ、ユーザーの確認を経てから本文に昇格させる。
  地の文の補足説明に限り「(推測)」タグ付きで書いてよい。

## 品質・安全性（persona: friday 準拠）

- 入力（コード・会話）に存在しない操作・コマンドを創作しない。不足は TBD + 対話で確認する。
- 機密情報（API キー・トークン・パスワード・接続文字列の実値）は手順書に転記しない。
  環境変数・シークレットは「キー名・用途・どこで取得するか」のみ記載する。
- 破壊的・不可逆操作の警告（`⚠️`）と直前の確認ステップを省略しない。
- 検証目的でも状態を変えるコマンドは実行しない（「4. 実在検証」の範囲を守る）。
- 日本語の表記（濁点・半濁点・特殊文字）を正しく保持し、ASCII で代替しない。
