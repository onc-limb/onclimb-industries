# 評価ケース設計ガイド

`dataset/cases.json` のスキーマと、良いケースを書くための指針。

## スキーマ

```json
{
  "cases": [
    {
      "id": "obs-001",
      "question": "アプリに投げる質問文",
      "context": null,
      "ideal_points": [
        "理想回答が含むべき要点（1 点 1 論点）"
      ],
      "reference_answer": null,
      "ng_points": [
        "してほしくない回答。なぜ NG かも書く"
      ],
      "checks": {
        "must_include": [],
        "must_not_include": [],
        "max_chars": null
      },
      "tags": ["basic"],
      "revision_notes": []
    }
  ]
}
```

| フィールド | 必須 | 説明 |
|---|---|---|
| `id` | ✅ | 一意な英数字 ID（`<領域>-<連番>` 推奨。例: `obs-001`） |
| `question` | ✅ | アプリにそのまま投げる質問文 |
| `context` | - | RAG アプリで、アプリに与えられる想定の検索結果・参照文書。faithfulness 測定に必要 |
| `ideal_points` | 推奨 | 理想回答が含むべき要点のリスト。completeness / correctness の判定基準 |
| `reference_answer` | - | 理想回答の全文。ideal_points だけで足りるなら省略可 |
| `ng_points` | 推奨 | してほしくない回答。ng_compliance の判定基準 |
| `checks` | - | 決定的チェック（LLM を使わない機械判定） |
| `tags` | - | 集計・絞り込み用（例: `safety`, `beginner`, `edge-case`） |
| `revision_notes` | - | 改訂履歴。`{date, change, reason, source}` の配列 |

## ideal_points の書き方

- **1 点 1 論点**: 「AとBに触れている」ではなく「Aに触れている」「Bに触れている」に分ける。
  ジャッジがカバー率を数えられる粒度にする。
- **Yes/No で判定できる表現**: 「わかりやすく説明している」（判定不能）ではなく
  「専門用語を使う場合は説明を添えている」（判定可能）。
- **本当に必須な要点だけ**: あれば嬉しい程度の内容を入れるとスコアが実態より下振れする。
  迷ったら入れず、improver の壁打ちで昇格を検討する。
- 表現の自由度を許す: 特定の言い回しを要求したい場合は ideal_points ではなく
  `checks.must_include` へ（意味の判定と文字列の判定を混ぜない）。

## ng_points の書き方

- **なぜ NG かまで書く**: 「初心者に危険なムーブを無条件で勧める（怪我のリスクを説明しないため）」。
  ジャッジの判定精度が上がり、後で NG 定義自体を見直すときの材料になる。
- 深刻度が違うものを意識する: 安全性に関わる NG（重大）とトーン・スタイルの NG（軽微）が
  混ざる場合は、重大なものを先頭に置き「（重大）」と明記する。
- 確実に禁止したい固有表現（商品名・断定フレーズ等）は `checks.must_not_include` へ。

## 質問の選び方（5〜10 件から始める）

3 方向からバランスを取る:

1. **頻出**: 実ユーザーが最もよく聞く質問。ここの品質がアプリの体感品質。
2. **重要**: 間違えるとユーザーの信頼・安全を損なう質問（tags: `safety` 等を付ける）。
3. **危険（エッジ）**: 曖昧な質問・前提が欠けた質問・スコープ外の質問。
   「答えられないと認める」「聞き返す」が理想回答になるケースを必ず数件入れる。

## アンチパターン

- ideal_points が回答の全文引用になっている（→ reference_answer に置き、要点だけ抽出する）
- 1 ケースに複数の質問を詰め込む（→ ケースを分ける）
- ジャッジへの指示（「厳しく採点して」等）をポイントに書く（→ ポイントは事実の期待だけ）
- 理想回答の完成を待ってから測定を始める（→ 仮説のまま測って壁打ちで育てる）
