# split-criteria — 分割単位の判断基準

pr-splitter が分割計画を立てるときの判断基準。計画立案（標準フロー 2）の前に読む。
基準の改善（新しい単位の型・失敗パターンの追記）はこのファイルへの追記で行う。

## 意味単位の型

変更は次の型に振り分ける。1 PR = 1 型が基本で、型をまたぐ変更を同じ PR に入れるときは
計画にその理由を書く。

| 型 | 内容 | 典型例 |
|---|---|---|
| refactor-prep | 後続の機能追加を可能にする準備リファクタ。挙動を変えない | 関数抽出、インターフェース切り出し、責務の移動 |
| foundation | 複数の機能が乗る基盤・共通変更 | 共通ライブラリ追加、設定スキーマ変更、DB マイグレーション |
| feature | 独立してレビュー・リリースできる機能 1 つ | 機能 A の実装（その機能のテスト含む） |
| test-docs | 既存挙動へのテスト追加・ドキュメント | 回帰テスト追加、README 更新 |
| mechanical | 機械的で差分が大きいが判断が不要な変更 | フォーマッタ適用、一括リネーム、import 整理 |

分け方の指針:

- **mechanical は必ず単独 PR に隔離する。** フォーマット差分が機能変更に混ざると、
  レビューで読むべき行が埋もれる。これが混在 PR の最大の劣化要因。
- feature のテストはその feature と同じ PR に入れる（テストだけ後送りにすると
  「テストなしでマージされた機能」が生まれる）。test-docs 型は既存挙動に対する
  テスト・ドキュメントのみ。
- refactor-prep は「この PR 単体では何も新しくならないが、挙動も変わらない」ことが
  レビュー観点になる。機能変更と混ぜない（persona の「機能変更とリファクタリングは
  コミットを分ける」を PR 単位にも適用する）。

## 依存順序の決め方

- 依存の向きで一列に並べる: refactor-prep → foundation → feature → test-docs が典型。
  mechanical は他とファイルが重ならなければ独立（並列可）、重なるなら列の先頭か最後尾に置く
  （中間に挟むとリベースのたびにコンフリクトする）。
- feature 同士が独立なら並列にし、計画表の依存列に「なし」と書く。
  並列にできる PR を直列に積まない（後続のレビュー・マージが不必要にブロックされる）。
- 循環依存が出たら単位の切り方が誤っている。単位を統合するか、共有部分を
  foundation として切り出して解消する。

## 1 PR のサイズ目安

- 目安は変更 400 行以下・10 ファイル以下（mechanical を除く）。
  ただし行数は補助指標で、**単位の意味的な独立性を行数のために壊さない**。
  600 行でも 1 つの意味単位なら分けない。150 行でも 2 つの関心事なら分ける。
- 逆に、分割後の PR が 20 行未満の断片ばかりになるなら分けすぎ。依存関係の管理コストが
  レビュー負荷の削減を上回る。隣接する単位と統合する。

## 絡み合ったハンクの扱い

1 ファイル内に複数単位の変更が混在している場合の判断:

1. まずハンク境界で分離できるか確認する（ハンク同士が離れていて独立に適用できるか）。
   できるならハンク単位で割り、計画表に「`<file>` (L__-__ のハンク)」と明記する。
2. 同一ハンク内に複数単位が混在（同じ行の変更が 2 つの目的を持つ）する場合、
   パッチの手編集が必要になる。手編集は適用ミスのリスクが高いため、
   **妥協して同じ PR に入れる案を第一候補**として提示し、どうしても分けたい場合のみ
   手編集パッチを使う（検証フローの差分ゼロ確認が守り）。
3. どちらにするかはユーザーが決める。計画段階で「絡み合い箇所一覧」として明示し、
   黙ってどちらかを選ばない。

## 分割しないほうがよいケース

以下に当たるときは、分割せずその旨を報告する（無理に分けるとかえってレビューしにくくなる）。

- 変更全体が 1 つの意味単位（例: 400 行あるが全部が単一機能の実装）。
- 変更のほぼ全ファイルが相互依存で、分けると全 PR が同時レビュー必須になる。
- マージ直前・hotfix 等で、分割の往復コストがレビュー改善に見合わない（ユーザーに確認）。

## 失敗パターン（追記して育てる）

- 生成物（lock ファイル、ビルド成果物）を手動変更と別 PR に分けてしまい、
  各ブランチ単体でビルドが通らない。→ 生成物はそれを生む変更と同じ PR に入れる。
- マイグレーションとそれを前提とするコードを分けたのに依存順を逆にして、
  中間ブランチでアプリが起動しない。→ foundation を必ず先行させ、各ブランチで起動確認する。
