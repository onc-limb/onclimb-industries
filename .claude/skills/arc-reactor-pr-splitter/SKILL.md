# pr-splitter — 肥大 diff の PR 分割 (arc-reactor)

肥大化した working diff / ブランチ / 既存 PR を、レビュー可能な意味単位の PR 列に
分割するスキル。「大きすぎてレビューされない PR」をなくすことが目的で、
成果物は (a) 各 PR の目的・含む変更・依存順序・レビュー観点を付けた分割計画と、
(b) 承認後に機械的に積み上げた分割済みブランチ列（希望があれば `gh` での PR 作成まで）。

> 大原則: **元の状態を必ず無傷で残し、計画はユーザーの承認後にのみ実施する。**
> 分割は変更内容の再配置であり、その過程でコードを 1 文字も「改善」しない。
> 分割後の全ブランチを重ねた最終状態は、元の diff と**機械的に一致**しなければならない。

persona: [`personas/arc-reactor.md`](../../../personas/arc-reactor.md) に従う
（応答は日本語 / コード・コミットメッセージ・ブランチ名は英語 / 推測は ASSUMPTION 明記）。

## トリガー

| ユーザー発話の例 | 動作 |
|---|---|
| 「この diff、大きすぎるから PR 分割して」 | working diff を対象に分割 |
| 「このブランチをレビューしやすい単位に分けて」 | ブランチの一連のコミットを対象に分割 |
| 「PR #123 が大きすぎるので分割計画を立てて」 | 既存 PR を対象に、まず計画のみ提示 |
| 「この変更を複数の PR に割って」 | 対象を確認して分割 |
| 「working diff を意味単位のブランチに分けて」 | working diff を分割（PR 作成なし） |
| 「pr split して」 | 対象を確認して分割 |

ユーザーが明示的に依頼したときだけ起動する（diff が大きいのを見かけても自動起動しない）。

## 標準フロー

### 1. 差分の把握

- 分割対象を確定する: 未コミットの working diff / 1 ブランチの一連のコミット /
  既存の大きな PR（`gh pr diff` / `gh pr view` で取得）のいずれか。
- ベースブランチを確定する（発話 / PR のベース / リポジトリの既定ブランチ）。
- `git diff` / `git log` / `git diff --stat` で変更全体を読み、何がどう変わるかを把握する。
- 変更の**意図**をユーザーに確認する（「このリファクタは機能追加の準備？」
  「この設定変更は機能 A と B のどちらに必要？」等）。意図が分からない変更を
  推測だけで振り分けない。

### 2. 分割計画の提案

[references/split-criteria.md](references/split-criteria.md) の判断基準に従い、
変更を意味単位（例: リファクタ準備 → 基盤変更 → 機能 A → 機能 B → テスト / ドキュメント）に
分け、依存順を付けた計画表を提示する。

```markdown
## 分割計画 — <元ブランチ/PR> → <base>

| # | ブランチ名(案) | 目的 | 含むファイル / ハンク | 依存 | レビュー観点 |
|---|---|---|---|---|---|
| 1 | refactor/extract-auth-helper | 機能追加の準備リファクタ | src/auth.ts (全体), src/utils.ts (L20-45 のハンク) | なし | 挙動が変わっていないこと |
| 2 | feat/two-factor-login | 2FA ログイン | ... | #1 | ... |

- 絡み合ったハンク: src/config.ts に #1 と #2 の変更が混在（後述）
- バックアップ: backup/<元ブランチ>-<YYYYMMDD> を作成してから着手
```

- **1 ファイル内に複数単位の変更が混在（絡み合ったハンク）する場合は、その旨を計画に明示**し、
  ハンク単位で割るか、妥協して同じ PR に入れるかをユーザーに確認する。
  ハンク分割はコンフリクトや適用失敗のリスクが上がるため、黙って選ばない。

### 3. 承認

- 計画はユーザーが承認するまで実施しない。修正要望があれば計画を直して再提示する。
- 着手前に、**元の状態に戻る手順**（バックアップブランチ名、`git stash` の使用有無、
  復元コマンド）を必ず提示する。

### 4. 機械的な分割実施

- 元ブランチ / working tree を**必ず無傷で残す**: 分割前にバックアップブランチを作る
  （working diff の場合は加えて stash またはバックアップブランチへのコミットで退避）。
- ベースブランチから新しいブランチ列を依存順に作り、変更を積み上げる:
  - ファイル単位: `git checkout <source> -- <file>` で対象ファイルの変更を取り込む。
  - ハンク単位: 対象ハンクだけのパッチを切り出して `git apply` する
    （interactive でない `git add -p` 相当。対話プロンプトは使わない）。
- コミットは Conventional Commits 形式（`feat:`, `refactor:` 等）で、各 PR の意味単位に合わせる。
- 各ブランチで可能ならビルド / テストを実行し、**そのブランチ単体で成立する**
  （壊れた中間状態をレビューさせない）ことを確認する。実行できない場合はその旨を報告に残す。

### 5. 検証

- 分割後の全ブランチを依存順に重ねた最終状態と元の diff を `git diff` で機械的に比較し、
  **差分ゼロ**を確認する（例: 最後のブランチと元ブランチ間の `git diff --stat` が空）。
- 一致しない場合は原因（取りこぼしたハンク、適用時のずれ等）を特定して報告する。
  「ほぼ一致」で済ませない。原因が特定できないときも、その事実を報告して指示を仰ぐ。
- 検証結果（比較コマンドと出力）をユーザーに提示する。

### 6. （希望時）PR 作成

- ユーザーが希望した場合のみ、`gh` で依存順に PR を作成する
  （依存先のブランチをベースにしたスタック PR、またはユーザーの指定する形式）。
- PR 本文に「分割元（元ブランチ / 元 PR）」「依存 PR」「レビュー観点」を記載する。
- push は新規ブランチへの通常 push のみ。既存ブランチへの force push はしない（品質・安全性の節を参照）。

## 分割単位の判断基準

意味単位の型・依存順序の決め方・1 PR のサイズ目安・絡み合ったハンクの扱い・
分割しないほうがよいケースは [references/split-criteria.md](references/split-criteria.md) を参照する。
計画立案の前に必ず読むこと。判断基準の改善は references への追記で行う。

## 品質・安全性

- **push 済み履歴の書き換え（force push）は、ユーザーの明示承認なしに絶対に行わない。**
  既存 PR の分割で元ブランチの縮小が必要な場合も、まず新ブランチ列を作って提示し、
  元ブランチの扱いはユーザーに委ねる。
- 元の状態に戻る手順（バックアップブランチ名・復元コマンド）を**作業前に**必ず提示する。
  バックアップ作成が確認できるまで分割を始めない。
- 分割の過程でコードを 1 文字も「改善」しない。変更内容の再配置のみ行う。
  気づいた問題（バグ疑い・タイポ等）は分割に混ぜず、指摘として報告する
  （レビューは `arc-reactor-code-review` の領分）。
- コンフリクトやパッチ適用の失敗で行き詰まったら、中途半端な状態（適用途中のブランチ、
  未解決コンフリクト）を残さず、そこまでの状態と原因を報告して指示を仰ぐ。
- 対象リポジトリの内容（コード・顧客名等）を外部に持ち出さない。作業はローカルの git 操作と、
  ユーザーが希望した場合の `gh` 操作に閉じる。
- 検証（差分ゼロ確認）を通過するまで「分割完了」と報告しない。
