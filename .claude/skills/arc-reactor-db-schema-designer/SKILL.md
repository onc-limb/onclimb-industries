# db-schema-designer — DB スキーマ設計の壁打ち (arc-reactor)

テーブル設計を壁打ちしながら一緒に作り、**ER 図・DDL・設計判断の記録**まで残すスキル。
一人で悩みがちな正規化・インデックス・拡張性の判断に、根拠付きの相談相手と記録を提供する。
「なんとなくこうした」を残さず、採用理由・不採用案・前提をすべて書き残すことが目的。

> 大原則: **アクセスパターンなくしてインデックスなし。** どの画面 / API がどう読み書きするかが
> 確定する前に、インデックス設計・非正規化の判断はしない。
> もう 1 つ: **既存スキーマがあるなら、その規約に合わせる。** 新しい命名規約や
> 主キー戦略を勝手に持ち込まない。

persona: [`personas/arc-reactor.md`](../../../personas/arc-reactor.md) に従う
（応答は日本語 / コード・識別子は英語 / 推測は ASSUMPTION 明記）。

## トリガー

| ユーザー発話の例 | 動作 |
|---|---|
| 「テーブル設計を一緒に考えて」 | 新規 / 既存を確認して標準フローを最初から |
| 「この機能の DB スキーマを設計して」 | 対象機能の要件ヒアリングから設計 |
| 「ER 図と DDL を作って」 | 設計確定後の成果物生成（要件が未確定ならヒアリングから） |
| 「このテーブル設計をレビューして」 | 既存の設計案を references の観点でセルフレビューのみ実行 |
| 「正規化するか迷ってるので壁打ちして」 | 論点のトレードオフ提示に絞った壁打ち |
| 「マイグレーション案を作って」 | 確定済み設計からマイグレーションファイル案を生成 |

ユーザーが明示的に依頼したときだけ起動する（コードを読んだついでに自動起動しない）。

## 標準フロー

### 1. 文脈の把握

- **新規設計か、既存スキーマへの追加か**を最初に確認する。
- 既存への追加なら、現行スキーマを読む: マイグレーションファイル
  （`migrations/`, `db/migrate/`, `prisma/schema.prisma` 等）、ER 図・スキーマドキュメント、
  ORM のモデル定義。
- 読んだ結果から次を検出し、**設計はそれに合わせる**（勝手に新しい規約を導入しない）:
  - 命名規約（単数形 / 複数形、snake_case 等）
  - 主キー戦略（連番 / UUID / ULID、カラム名）
  - 共通カラム（`created_at` / `updated_at` / `deleted_at` 等）とその型
  - DB エンジンと方言（PostgreSQL / MySQL / SQLite 等）、マイグレーションツール
- 既存資産から読み取れず推測した点は、成果物に `ASSUMPTION` として明記する。

### 2. 要件ヒアリング（対話、2〜3 問ずつ）

一度に全部聞かない。2〜3 問ずつ、回答を受けて次を絞る。聞くこと:

- エンティティと関係（何と何が 1:N / N:M か。境界が曖昧なものはここで潰す）
- データ量の見込み（初期 / 1 年後の行数オーダー。増え方が速いテーブルはどれか）
- **主要なアクセスパターン**: どの画面 / API がどう読むか・書くか。
  一覧の絞り込み条件、ソートキー、書き込みの頻度と単位。
  **インデックス設計はここが確定する前に決めない。**
- 整合性要件（外部キー制約を張るか、アプリ層で守るか。一意性制約の単位）
- 履歴・監査の要否（変更履歴を残すか、誰がいつ変えたかが必要か）

### 3. 設計案の提示

- 判断が分かれるポイントは、**選択肢とトレードオフを示してユーザーに確認する**。例:
  - 正規化 vs 意図的な非正規化（結合コストと更新異常のどちらを取るか）
  - ポリモーフィック関連の表現（型 + ID カラム / 中間テーブル分割 / 排他 FK）
  - N:M の中間テーブルに属性を持たせるか
- それ以外（判断が実質一つに決まる点）は案として直接提示し、確認で往復を増やさない。
- 各判断には理由を 1 行添える（後の設計記録にそのまま使う）。

### 4. 成果物の生成

設計が固まったら、次の 4 点を生成する。

- (a) **ER 図**: Mermaid `erDiagram`
- (b) **DDL**: プロジェクトの DB エンジンの方言で（`CREATE TABLE` + 制約 + インデックス）
- (c) **マイグレーションファイル案**: プロジェクトのマイグレーションツール
  （Rails / Prisma / golang-migrate / Alembic 等）の形式に合わせる
- (d) **設計判断の記録**: 採用理由・不採用案とその理由・`ASSUMPTION` の一覧

### 5. セルフレビュー

[references/schema-review-checklist.md](references/schema-review-checklist.md) の観点で
設計を 1 周見直し、懸念を**指摘リスト**として成果物に添付する。
「将来この行数になったらこのクエリのインデックスが足りない」「この NULL は
『未入力』と『対象外』のどちらを意味するのか曖昧」のように、条件と影響まで書く。
問題なしの観点も「確認した」と分かるよう列挙する。

### 6. 保存

- 設計記録（ER 図・DDL・判断記録・指摘リストをまとめた Markdown）は
  対象プロジェクト配下 `<repo>/docs/db-design/<YYYY-MM-DD>-<topic>.md` 等に保存する。
  対象リポジトリの git を汚す可能性があるため、**保存先とあわせて保存前に必ず確認**する。
- マイグレーションファイルの作成はコードへの変更なので、**ユーザー確認後**にのみ行う。
- 既存テーブルの変更（カラム追加・型変更・インデックス追加等）を伴う設計のときは、
  適用時の安全性（ロック・ダウンタイム）を `db-migration-safety-checker` でチェックするよう 1 行提案する。

## 設計レビュー観点

観点ごとの詳細（何を確認し、何が懸念になるか）は
[references/schema-review-checklist.md](references/schema-review-checklist.md) を参照する。
セルフレビュー実行前に必ず読むこと。観点の改善（新しい観点・失敗パターンの追加）は
references への追記で行う。

## 品質・安全性

- **DB への実適用（migrate 実行）は本スキルの範囲外**。提案（ファイル案・DDL）まで。
  実行はユーザーが自分の手順で行う。
- 既存スキーマの規約（命名・主キー戦略・共通カラム）に従い、勝手に新しい規約を導入しない。
  規約と異なる選択を提案したいときは、理由を添えてユーザーに確認する。
- 推測で決めた点は成果物に `ASSUMPTION` として明記する（ヒアリングで確認できなかった
  データ量・アクセスパターンの仮定を含む）。
- 設計記録はプロジェクト固有情報のため、対象プロジェクトのディレクトリ配下に閉じる
  （当リポジトリ側や他プロジェクトに持ち出さない。CLAUDE.md の分離ルールに従う）。
- 接続文字列・認証情報を成果物に転記しない。
- マイグレーションの適用時安全性（ロック・後方互換・ロールバック）は
  `db-migration-safety-checker` の領分。本スキルはスキーマの形を決めるところまでを担う。
