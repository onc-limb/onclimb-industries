# schema-review-checklist — 設計セルフレビューの観点

db-schema-designer のフロー 5（セルフレビュー）で使う観点一覧。
設計案を 1 周見直し、該当する懸念を「条件 + 影響 + 対処案」の形で指摘リストにする。
問題がなかった観点も「確認済み」として列挙する（見ていないのか問題ないのかを区別するため）。

観点を増やしたいとき・実際の設計で見逃した失敗パターンを見つけたときは、
このファイルに追記して育てる（SKILL.md 本文は書き換えない）。

## 1. 正規化と意図的な非正規化

- 更新異常が起きる重複（同じ事実が 2 か所に書かれる）が残っていないか。
- 非正規化している箇所は、それが**意図的**か。理由（読み取り頻度・結合コスト）が
  設計記録に書かれているか。理由の無い非正規化は指摘する。
- 非正規化した値の更新経路は 1 本に絞られているか（誰が同期するのか決まっているか）。

## 2. 主キー戦略（連番 vs UUID）

- 既存スキーマの戦略と一貫しているか（混在は指摘）。
- 連番: 外部に露出する ID か。露出するなら推測可能性（enumeration）が問題にならないか。
- UUID: v4 ランダムならインデックスの断片化・挿入性能への影響を許容できる規模か
  （UUIDv7 / ULID で時系列性を持たせる選択肢の検討有無）。
- 自然キー（メールアドレス等）を主キーにしていないか。変わりうる値は主キーにしない。

## 3. インデックスとアクセスパターン

- ヒアリングで確定した各アクセスパターン（絞り込み・ソート・結合）に対応する
  インデックスがあるか。**アクセスパターンに紐づかないインデックスは根拠を要求する**。
- 複合インデックスのカラム順は「等価条件 → 範囲条件・ソート」の順になっているか。
- 外部キーカラムにインデックスがあるか（MySQL は自動、PostgreSQL は手動）。
- 書き込みが激しいテーブルにインデックスを積みすぎていないか。
- 将来の行数見込み（ヒアリング結果）でフルスキャンになる一覧クエリがないか。

## 4. NULL 許容の意味

- NULL を許容するカラムは、NULL が何を意味するか一意に決まっているか
  （「未入力」「対象外」「まだ発生していない」の混在は指摘）。
- 意味が 2 つ以上あるなら、カラム分割・別テーブル化・デフォルト値を検討したか。
- 一意制約と NULL の組み合わせの挙動（NULL 同士は重複扱いにならない）を踏まえているか。
- ビジネス上必須の値が NULL 許容になっていないか（「とりあえず nullable」は指摘）。

## 5. 論理削除 vs 物理削除

- 削除方式は要件（復元の要否・削除履歴の要否・法的な削除義務）から選ばれているか。
- 論理削除の場合:
  - 全クエリに `deleted_at IS NULL` が必要になる負担を認識しているか（ORM のスコープ等）。
  - 一意制約が論理削除行と衝突しないか（削除済みと同じ値で再登録できるか）。
  - 外部キー参照先が論理削除されたときの扱いが決まっているか。
- 個人情報を含むテーブルで論理削除を選ぶ場合、削除要求（GDPR 等）への対応方法があるか。

## 6. 時刻とタイムゾーン

- 時刻カラムの型は タイムゾーン付き（PostgreSQL `timestamptz` 等）か、
  UTC 固定で保存する規約か。**どちらかに統一されており、混在していないか**。
- 「日付」として扱うべき値（誕生日・締め日）に時刻型を使っていないか（`date` で足りるか）。
- ユーザーのローカルタイムゾーンで意味を持つ値（「その日の集計」等）の変換方針が決まっているか。

## 7. 金額の型

- 金額に浮動小数点型（`float` / `double`）を使っていないか。**使っていたら必ず指摘**。
  `decimal` / `numeric` か、最小通貨単位の整数（円なら int、ドルならセント int）にする。
- 複数通貨を扱う可能性があるなら、通貨コードカラムがあるか。
- 税率・割引率など、金額から導出される値の保存方針（都度計算 vs 記録時点の値を保存）が
  決まっているか。請求・会計系は記録時点の値の保存が原則。

## 8. 履歴・監査要件

- 「誰がいつ変えたか」が必要なテーブルに、その手段（監査カラム / 履歴テーブル /
  イベントソーシング / DB の監査ログ）が用意されているか。
- `updated_at` だけでは「何をどう変えたか」は残らない。要件がそこまで求めるなら
  履歴テーブル方式を検討したか。
- 履歴テーブルの肥大化見込み（元テーブルの更新頻度 × 行数）と保持期間が検討されているか。

## 9. マルチテナント分離

- マルチテナントの場合、分離方式（テナント ID カラム / スキーマ分離 / DB 分離）は
  テナント数・分離要件（契約・コンプライアンス）に対して妥当か。
- テナント ID カラム方式の場合:
  - テナントをまたぐデータ漏えいを防ぐ仕組み( PostgreSQL RLS、ORM のグローバルスコープ等)を
    検討したか。アプリ層の WHERE 句だけに頼る場合はそのリスクを指摘する。
  - 一意制約・インデックスにテナント ID が含まれているか
    （テナント内一意のつもりがグローバル一意になっていないか、逆も）。
- シングルテナント前提の設計なら、マルチテナント化の可能性の有無を ASSUMPTION として残したか。
