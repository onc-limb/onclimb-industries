# capture — 対話インタビューによる技術・経験の記録

ユーザーへのインタビューを通じて、本人の中にある技術レベルと経験を言語化し、
`capture-data/` に一次記録として積み上げる。worklog（ログの機械収集）と並ぶ
**もう一つの第一インプット**であり、後段の jarvis-knowledge-base（技術面の集約）や
friday-skillset-writer（スキルセット紹介文）の原料になる。

> 出自: `projects/omni-archive` の `/capture` スキルを onclimb-industries に統合したもの。
> 移行済みノート（日付なしファイル名）の経緯は `capture-data/README.md` を参照。

## Step 0: モード分岐（必須の入口）

起動時の発話に「技術」「経験」が明示されていればそのルートへ直行する。
明示されていなければ、**最初に必ず AskUserQuestion でどちらを深掘るか選択させる**:

| 選択肢 | 対象 | 出力先 |
|---|---|---|
| **技術キャプチャ** | 使用した技術に対する「何をしたか・どこまで理解しているか・まだ何をしていないか」＝技術レベルの情報 | `capture-data/tech/` |
| **経験キャプチャ** | 技術に限らない経験の内容と、そのときの感情・学び（価値観・信条の言語化も含む） | `capture-data/experience/` |
| **両方** | プロジェクト振り返りなど | 経験 → 出てきた技術の順で直列実行 |

「両方」の場合は先に経験ルートを完了させ、対話中に濃く出てきた技術を候補として
技術ルートに引き継ぐ（同じ話を聞き直さない）。技術と経験は必ずファイルを分ける。

## 技術ルート

### T1. 技術の確定（候補提示）

対象技術が発話で指定されていなければ、以下の 3 ソースから候補を集めて
AskUserQuestion で提示する（自由入力も可）:

1. `capture-data/tech/` の既存ノートの frontmatter（`technologies` / `status` / `captured`）
   → **キャプチャが古い技術**（更新候補。captured が古い順）
2. `knowledge-base/index.md` の技術領域のうち capture ノートが無いもの → **未キャプチャ候補**
3. 直近の worklog digest（`worklog-data/digests/` の新しいもの数件）に出てくる技術
   → **最近触った技術**

候補収集は `ls` と frontmatter の読み取りで行う（一覧化が重くなってきたら
`scripts/` への切り出しを検討する）。

### T2. 前回地点の提示（差分インタビュー）

選ばれた技術の既存ノートを `capture-data/tech/` から探して読む
（`technologies` frontmatter とファイル名で照合。複数あれば最新の日付のもの）。

- **既存ノートあり**: 「前回（日付）時点では status=〜、未踏は〜でした」と現在地を要約提示し、
  質問は**前回からの差分**と**「まだやっていないこと」の解消確認**に絞る。
  変化がない観点は「〜という理解のままで合っていますか」の確認形式で流す。
- **初回**: T3 の 3 点をフルで聞く。

### T3. 深掘り（順序固定）

技術レベルの 3 点をこの順で埋める。各観点、**具体例が 1 つ出たら次へ進む**（網羅より具体性優先）:

1. **何をしたか** — 実際に手を動かした内容（業務/個人、規模、担当範囲）。
   - 「その技術で具体的に何を作りましたか？どの部分を自分で書きましたか？」
2. **どこまで理解しているか** — 理解の深さと軸。
   - 「仕組みをどこまで説明できますか？（使える / 中身を説明できる / 設計判断ができる / 人に教えられる）」
   - **詰まった経験とその解決を必ず 1 つ引き出す**（理解の証跡として最も価値が高い）。
3. **まだ何をしていないか** — 未経験の領域・理解が浅い部分。
   - 「その技術で、やったことがない領域はどこですか？（運用・チューニング・大規模化など）」
   - 「今の理解で不安が残る部分は？」

最後に status（`learning | familiar | proficient`）を自己申告と証跡から提案し、
AskUserQuestion で確認を取る。

### T4. 保存と接続

【まとめ】提示 → 確認 → `capture-data/tech/YYYY-MM-DD_<tech_slug>.md` に保存 →
「ナレッジベースも更新しますか」と一言提案する。

## 経験ルート

### E1. 経験の特定

いつ・どの出来事かを確定する。曖昧なら「直近で一番印象に残っている仕事の場面は？」から入る。
直近の worklog や ToDo から候補を提示してもよい。

### E2. 事実の再構成（短く）

時系列・状況・自分の役割・目標と結果・転換点。ここは感情を掘るための器を作るだけなので
短く済ませる（既知の事実は聞き直さず確認形式にする）。

### E3. 感情の深掘り（技法）

「どう感じましたか」と直接聞いても一般論しか返ってこない。以下の技法で掘る:

- **場面を特定してから聞く**: 「一番しんどかった瞬間はいつですか？そのときまず何をしましたか？」
  → 行動から感情を遡る。うれしかった瞬間も同様。
- **感情語をこちらから与えない**: 「焦りましたか？」ではなく「そのとき頭に浮かんだことは？」。
  ユーザーの言葉が出るまで待ち、出た言葉をそのまま記録する。
- **矛盾や温度差を掘る**: 淡々と語っているのに言葉が強い、逆に大きな出来事なのに感情の言及が
  無い、といったギャップに気づいたら指摘して掘る。
- 感情・意見はファクトチェックしない。平準化もしない（下流の文書品質の原料になる）。

### E4. 意味づけ

- そこから何を学んだか。今のやり方・考え方にどう影響しているか。
- 「同じ状況の人にアドバイスするなら何と言うか」（価値観の言語化に効く）。

### E5. 保存と接続

【まとめ】提示 → 確認 → `capture-data/experience/YYYY-MM-DD_<slug>.md` に保存。
対話中に技術の話が濃く出ていたら「この技術は技術キャプチャにも積みますか」と提案する
（「両方」モードならそのまま技術ルートへ）。

## インタビューの共通ルール

- **選択で答えられる質問は AskUserQuestion、語ってもらう質問は自由対話**と使い分ける
  （入口・技術選択・status 確認は選択肢、深掘りは対話）。
- 自由対話の質問は一度に 1〜2 個。畳みかけない。
- 1 セッション 1 トピック。深掘り質問は目安 5〜7 往復で【まとめ】提示に移る
  （不足があれば追記の往復をする）。
- ユーザーの言っていないことを書かない。要約して返し、確認を取る。
- 既存ノート・worklog にある事実は聞き直さず「〜という理解で合っていますか」の確認形式にする。
- 技術的な事実（仕組み・用語・数値）に不正確さがあれば穏やかに指摘し、
  訂正は本文を書き換えるのではなく「メモ・補足」節に残す。必要なら WebSearch で裏を取る。
  意見・感情・主観的経験はファクトチェックしない。

## 記録フォーマット

書き込み前に必ず【まとめ】を提示し、「この内容で保存してよいですか？」と確認する。

**共通ルール:**

- ファイル名: `YYYY-MM-DD_<英語snake_caseのslug>.md`（日付プレフィックス必須）。
- **過去のノートは編集せず、新しい日付ファイルを積む**（knowledge-base の増分更新が
  新規ファイル単位で取り込むため。同じ技術の再キャプチャも新ファイルにする）。
- 1 トピック 1 ファイル。緩いまとまりで良いが、技術と経験は必ずファイルを分ける。
- Obsidian インラインタグ（技術名は英語 `#NestJS`、概念は日本語 `#振り返り`）を
  セクション見出し直下に 3〜6 個付ける。関連ノートがあれば `[[slug]]` で繋ぐ。

**技術キャプチャ** (`capture-data/tech/YYYY-MM-DD_<tech_slug>.md`):

```markdown
---
tags: [capture, tech]
technologies:
  - <技術名>
status: learning | familiar | proficient
captured: YYYY-MM-DD
---

# <技術名>

<一言サマリ（この技術との関わりの現在地）>

## 何をしたか

## どこまで理解しているか

## まだやっていないこと・理解が浅いこと

## メモ・補足
<ファクトチェックの訂正・裏取り結果。無ければ省略>
```

**経験キャプチャ** (`capture-data/experience/YYYY-MM-DD_<slug>.md`):

```markdown
---
tags: [capture, experience]
period: "YYYY-MM ~ YYYY-MM"   # 分かる場合のみ
skills_gained: []              # 得たもの（技術・知見・態度）
captured: YYYY-MM-DD
---

# <タイトル>

## 何があったか

## 感じたこと

## 学び・教訓
```

## 機密・一般化の扱い

- **技術キャプチャは一般化して書く**（顧客名・接続情報・案件特定情報を持ち込まない）。
  knowledge-base → 公開系（homepage 等）へ流れる前提のため。案件文脈が必要なら
  「業務システム」「SES 先の案件」程度に丸める。
- **経験キャプチャは本人の言葉を優先**する（読み手は自分と AI のみ）。ただし
  API キー・トークン等の機密は含めない。実名は本人が望む場合のみ残す。

## 後段への接続

- 技術キャプチャは jarvis-knowledge-base が worklog の tech digest と併せて取り込む
  （`kb_build.py` が `capture-data/tech/*.md` を自動で読む）。
- 経験キャプチャは friday-skillset-writer の補助入力・friday-tech-article-drafter の
  記事の種になる。記事にできそうな深掘りが出たら一言提案する（勝手に書かない）。
