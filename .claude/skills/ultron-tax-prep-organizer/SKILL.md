# tax-prep-organizer — 確定申告 経費・売上整理スキル

散在する取引データ（明細 CSV / JSON / 手元メモ）を、**勘定科目別の集計表**と**要確認リスト**に整理する。
対象はフリーランスエンジニア本人（青色申告を想定）。

> **スタンス（毎回明示）**: 本スキルの出力は**取引データの機械的な整理であって税務助言ではない**。
> 勘定科目の最終判断・按分率の妥当性・申告内容の確定は、**本人または税理士**が行う。
> 節税提案・税額計算・税法の解釈はしない。判断が分かれる取引は数字に混ぜず「要確認」に分離する。

## 場所（コードとデータは分離されている）

- ツール本体・参照資料: このスキルディレクトリ `.claude/skills/ultron-tax-prep-organizer/`（`scripts/` `references/`）
- 勘定科目対応表: `references/account_mapping.md`（分類候補の根拠。更新は本人 / 税理士確認済みの手動追記のみ）
- **データ**: スキルが属する git リポジトリ直下の `tax-data/<year>/`
  （取引データ `transactions.csv|json`、集計結果 `summary-<date>.md|json`）。
  **(ASSUMPTION: stock-data / worklog-data と同じ「repo 直下 + gitignore で追跡外」方式を踏襲。
  環境変数 `TAX_DATA` で上書き可。初回利用時に `.gitignore` へ `/tax-data/` を追加する。)**

## 役割分担（重要）

合計金額の誤り・捏造を構造的に防ぐため、**分類候補の提示は Claude、合算はスクリプト**に分ける。

- **Claude（あなた）**: 明細の読み取り・整形、`references/account_mapping.md` に基づく勘定科目**候補**の提示、
  ユーザーへの確認対話、スクリプト出力の転記と講評。
- **スクリプト（決定論）**: `scripts/aggregate.py` が科目別合計・按分適用（円未満四捨五入）・
  要確認の分離・重複疑い検出をすべて計算する。
- **確証のない取引は「要確認」**: 科目に迷う・按分率が未確定・私用混在・10 万円以上の資産購入などは
  科目を確定せず `needs_review: true` で分離し、本人 / 税理士の確認に回す（`references/account_mapping.md` §要確認）。

→ Claude は**合計を暗算しない**。サマリの数値はすべて `aggregate.py` の出力を転記する。
→ **按分率を Claude が決めない**。「家賃は 3 割が相場」等の一般論で埋めず、ユーザーが確定した値のみ使う。

## トリガーと対応フロー

| ユーザー発話の例 | 実行する標準フロー |
|---|---|
| 「確定申告用に経費を整理して」「今年の売上と経費をまとめて」 | 標準フロー（データ受領から） |
| 「この取引データを勘定科目別に集計して」「この明細 CSV を仕分けして合計出して」 | 標準フロー（渡されたファイルを入力に） |
| 「按分が必要そうな経費を要確認で分けておいて」 | 標準フロー（分類段階で要確認分離を重点実施） |
| 「集計だけやり直して」「この分類済みデータで再集計して」 | 手順 4 以降のみ（スクリプト再実行） |
| 「要確認になってる取引の一覧を出して」 | 既存 `summary-*.json` の `needs_review` を提示 |
| 「この出費って何費？」→ まとめて整理する流れになったとき | 対応表から候補提示 → 合意後に標準フローへ |

ユーザーが明示的に依頼したときだけ起動する（自動起動しない）。
請求書作成（invoice-builder）・稼働時間集計（timesheet-aggregator）・worklog は別系統で、本スキルでは扱わない。

## 標準フロー

```bash
SKILL=/Users/satoshi-onga/Documents/onclimb-industries/.claude/skills/ultron-tax-prep-organizer
DATA=${TAX_DATA:-<repo>/tax-data}   # repo 直下。無ければ作成し .gitignore に追加
```

1. **入力を確定**: 対象期間（例: 2026 年分）と取引データの所在（CSV / JSON / 会話での列挙）を確認する。
   明細に口座番号・カード番号等の機密列があれば、整形時に**必要列だけ**を持ち込む。
2. **科目分類（候補付け）**: 各取引に `references/account_mapping.md` に基づく勘定科目の**候補**を付け、
   下記スキーマの CSV / JSON に整形して `tax-data/<year>/transactions.csv` に保存する。
   確証のない取引（対応表の §要確認 に該当）は `account` を確定せず `needs_review: true` + `review_reason` を付ける。
   ```
   date,type,description,amount,account,business_ratio,needs_review,review_reason,source
   2026-01-31,expense,自宅家賃 1月分,120000,地代家賃,30,,,bank-2026-01.csv
   2026-02-28,expense,自宅電気代 2月分,9800,要確認,,true,家事按分の率が未確定,bank-2026-02.csv
   ```
3. **ユーザー確認**: 分類結果の全体像と**要確認候補の一覧**を提示し、確定できるものは確定してもらう。
   按分率（`business_ratio`）は**ユーザーが決めた値のみ**記入する（未確定なら要確認のまま進めてよい）。
4. **決定論集計**: 分類済みデータをスクリプトに渡す。Claude は合算しない。
   ```bash
   python3 "$SKILL/scripts/aggregate.py" --file "$DATA/<year>/transactions.csv" \
     --out "$DATA/<year>/summary-<date>.json"
   python3 "$SKILL/scripts/aggregate.py" --file "$DATA/<year>/transactions.csv" \
     --markdown --out "$DATA/<year>/summary-<date>.md"
   ```
   スクリプトは以下を機械的に行う（詳細は `scripts/aggregate.py` docstring）:
   - 科目別合計（収入 / 経費、経費は按分適用後金額も算出。円未満は四捨五入）
   - 要確認の分離（`needs_review` 指定 / 科目未確定 / 金額不正 / 按分率が (0,100] 外 / 区分不明）
   - **重複疑いの検出**（同一の日付・区分・金額・摘要の 2 件目以降 → 要確認へ。二重計上対策）
   - 既定リストにない科目名の警告
5. **サマリ提示**: `summary-<date>.md` の内容をチャットに提示する。数値は**スクリプト出力の転記のみ**。
   要確認リストには全件の理由を並記し、「要確認 M 件・保留合計 ○円」を必ず示す。
6. **完了条件**: 「取引 N 件中、集計 K 件 / 要確認 M 件」を要約し、M > 0 の場合は
   **集計が未確定である旨**と、要確認の解消方法（本人確定 → 手順 2 のデータ更新 → 手順 4 の再実行）を案内する。

## 入出力スキーマ

入力（CSV 列 = JSON キー、1 行 = 1 取引）:

| フィールド | 型 | 必須 | 備考 |
|---|---|---|---|
| date | YYYY-MM-DD | yes | |
| type | "income" \| "expense" | yes | 収入 / 経費 等の日本語も可 |
| description | string | yes | 摘要 |
| amount | number（円・正） | yes | カンマ・¥ 許容。**(ASSUMPTION: 税込金額を基本とし、消費税の経理方式の判断はしない)** |
| account | string | yes | 勘定科目 or "要確認" / 空 |
| business_ratio | 0–100 | no | 既定 100。**ユーザー確定値のみ** |
| needs_review | boolean | no | 既定 false |
| review_reason | string | no | 要確認の理由 |
| source | string | no | 明細の出どころ |

出力: `summary-<date>.json`（機械可読。totals / income_by_account / expense_by_account / needs_review / warnings / counts）と
`summary-<date>.md`（免責 → サマリ行 → 収入表 → 経費表（按分前 / 適用後）→ 要確認表 → 警告）。
免責文はスクリプトが両形式に埋め込むため、転記時に落ちない。

## 品質・安全性

- **免責を毎回明記**: すべての出力（チャット返信・保存ファイル）に「税務助言ではなく情報整理である」旨を添える。
  語彙は「候補」「整理」「要確認」に統一し、「この科目にすべき」という断定や節税提案・税額計算はしない。
- **事実と解釈の区別**: 科目候補には対応表上の根拠を示し、運用が分かれる取引はその旨を明示して要確認へ。
- **数値の正確性**: 金額はスクリプト出力の転記のみ。按分前の raw 値も残し、検算可能にする。
- **再現性**: 分離ルール（要確認の条件・丸め規則）は `aggregate.py` に明文化されており、同じ入力から同じ結果が出る。
- **機密情報**: 口座番号・カード番号・取引先の機密を出力やコードに含めない。`tax-data/` は git 追跡外とする。
- **要確認の放置防止**: 要確認が 0 件になるまで「集計は未確定」と明示し続ける（スクリプトの固定文言）。

## 注意

- `aggregate.py` は**要確認に該当した行を合計から除外**する（保留合計として別掲）。要確認を解消したら
  `transactions.csv` を更新して再実行すること（サマリを手で直さない）。
- 重複疑いは「同一の日付・区分・金額・摘要」の機械判定。**本当に 2 回あった取引**（同日同額の 2 決済など）は
  ユーザー確認のうえ摘要を区別できる形（例: "AWS 利用料 12月分 (2回目)"）に直して再実行する。
- 減価償却費の**償却計算そのものは対象外**（会計ソフト / 税理士へ）。10 万円以上の資産購入は要確認に分離するだけ。
- 自己進化パイプラインは持たない。`references/account_mapping.md` の更新は本人 / 税理士確認済みの手動追記のみ。
