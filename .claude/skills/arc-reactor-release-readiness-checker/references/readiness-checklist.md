# リリース準備チェックリスト — 観点別の確認項目

release-readiness-checker がレビューで使うチェックリスト本体。項目ごとに
**何を確認するか / コードから機械的に確認する方法 / 不足時の典型対応** を定める。

共通ルール:

- 「機械的な確認方法」はローカルの読み取り操作（`rg` / ファイル存在確認 / 設定ファイル読解 /
  `git log`）に限る。対象リポジトリを変更するコマンド・ネットワーク照会は使わない
  （`npm audit` 等のネットワークを伴うコマンドは「ユーザーに実行を依頼する」形で要確認に添える）。
- 機械的な確認は**シグナル収集**であって判定そのものではない。ヒットした箇所は必ず読んで
  文脈を確認してから判定する（テストコード内の TODO と本番導線の TODO は重さが違う）。
- コードに痕跡が無いことは「本番にも無い」ことを意味しない（インフラ側で設定されている場合がある）。
  リポジトリ外で満たされうる項目は、不足断定ではなく「要確認」に回す。各項目に
  （**リポジトリ外で満たされうる**）と注記した。
- 重み付け（Blocker / Should / Nice）はリリース文脈で決める。本ファイルの「不足時の典型対応」に
  書いた重さは目安であり、SKILL.md ステップ 1 のヒアリング結果が優先する。

## 1. 機能完成度

### 1.1 TODO / FIXME 残

- **何を確認するか**: リリース対象の主要導線に、未実装・未修正を示すマーカーが残っていないか。
- **機械的な確認方法**: `rg -n "TODO|FIXME|HACK|XXX|WIP|not implemented|NotImplemented|unimplemented!" --hidden -g '!node_modules' -g '!vendor' -g '!dist'`。
  ヒット箇所を読み、(a) 主要導線か周辺か、(b) 未実装なのかメモなのかを分類する。
- **不足時の典型対応**: 主要導線の未実装マーカーは Blocker。周辺機能・改善メモは
  issue 化して Should / Nice に落とす（`TODO(#123)` 形式で追跡済みのものは管理済みとして扱う）。

### 1.2 フィーチャーフラグの状態

- **何を確認するか**: フラグで隠された作りかけ機能が、リリース時にどちらの状態で出るか
  確定しているか。既定値が「作りかけが露出する」側になっていないか。
- **機械的な確認方法**: `rg -n "feature.?flag|feature.?toggle|FLAG_|_ENABLED|is_enabled|LaunchDarkly|unleash" -i` でフラグ定義を列挙し、
  各フラグの既定値と参照箇所を読む。設定ファイル（`config/`、`.env.example`）側の既定値も突き合わせる。
- **不足時の典型対応**: 既定値が未確定・環境依存で挙動が変わるフラグは Blocker
  （リリース時の状態を確定させる）。使われなくなったフラグの掃除は Nice。

### 1.3 ハードコードされた仮実装

- **何を確認するか**: ダミーデータ・固定の戻り値・localhost 向き URL・テスト用アカウントが
  本番導線に残っていないか。
- **機械的な確認方法**: `rg -n "localhost|127\.0\.0\.1|0\.0\.0\.0|example\.com|dummy|stub|fake|sample_|test@|password.?=.?['\"]" -i -g '!*test*' -g '!*spec*' -g '!*.md'`。
  テストコード・設定サンプル（`.env.example` 等）へのヒットは除外し、本番導線のヒットだけ判定する。
- **不足時の典型対応**: 本番導線の仮実装・localhost 向き URL は Blocker
  （環境変数・設定ファイルへ外出しする）。開発補助のダミーは開発環境限定であることを確認して OK。

## 2. テスト

### 2.1 主要導線のテスト有無

- **何を確認するか**: リリースで壊れると一番困る導線（認証、決済、データの書き込み・削除、
  コア機能）に、自動テストが 1 本以上あるか。カバレッジの高低ではなく「主要導線がゼロでないか」を見る。
- **機械的な確認方法**: テストファイルを列挙し（`rg --files -g '*test*' -g '*spec*' -g 'test_*'`、
  `tests/` / `__tests__/` / `spec/` ディレクトリの有無）、主要導線の実装ファイル名・関数名で
  テストコード内を `rg` して対応の有無を見る。ヒアリングで確定した「主要導線」ごとに対応表を作る。
- **不足時の典型対応**: 課金・データ削除など不可逆な導線のテストゼロは Should 以上
  （課金ありの一般公開なら Blocker 寄り）。まず主要導線に 1 本ずつ書く。
  雛形作りは `arc-reactor-test-scaffolder` に回せる。

### 2.2 CI の有無と通過状況

- **何を確認するか**: push / PR でテスト・lint が自動実行される仕組みがあるか。直近で通っているか。
- **機械的な確認方法**: `.github/workflows/`、`.gitlab-ci.yml`、`.circleci/`、`Jenkinsfile` 等の
  存在と中身（テストを実行しているか、lint だけか）を確認する。通過状況はローカルからは
  確定できないことが多いので、`gh run list` が使えれば参照し、使えなければ要確認に回す。
- **不足時の典型対応**: CI 自体が無い場合、一般公開・複数人開発なら Should（最低限
  テスト実行のワークフローを 1 本足す）、単独開発の社内ツールなら Nice。
  CI はあるが落ちたまま放置されている場合は Should 以上（「通らない CI」は無いのと同じ）。

## 3. エラー処理・可観測性

### 3.1 エラーハンドリング

- **何を確認するか**: 主要導線で例外・エラーが握りつぶされていないか。ユーザー向けの
  エラー応答（500 の生スタックトレース露出など）が整っているか。
- **機械的な確認方法**: `rg -n "except:\s*pass|except Exception:\s*pass|catch\s*\(.*\)\s*\{\s*\}|catch\s*\{\s*\}|\.catch\(\(\)\s*=>\s*\{\}\)|panic\(|unwrap\(\)|rescue nil" ` で握りつぶし・強制終了系を列挙。
  Web アプリならグローバルエラーハンドラ（エラーミドルウェア・error boundary）の有無を
  フレームワークの流儀で確認する。デバッグモード設定（`DEBUG=True` 等）の既定値も見る。
- **不足時の典型対応**: 本番でスタックトレースがユーザーに露出する設定は Blocker
  （デバッグモードを環境変数で切り、グローバルハンドラで汎用エラー画面を返す）。
  周辺処理の握りつぶしは Should（最低限ログに残す）。

### 3.2 ログ

- **何を確認するか**: 障害時に追跡できるログが出るか。`print` / `console.log` デバッグ残りでなく
  ロガー経由か。ログに個人情報・シークレットを書いていないか。
- **機械的な確認方法**: ロガーの導入（`rg -n "import logging|winston|pino|zap|slog|logrus|log/slog"`）と、
  デバッグ出力残り（`rg -n "console\.log|print\(|println!|var_dump" -g '!*test*'`）を突き合わせる。
  ログ出力行にパスワード・トークン・メールアドレス等の変数が渡っていないかをヒット箇所の読解で確認。
- **不足時の典型対応**: ロガーが無く print 頼みなら Should（ロガー導入とレベル分け）。
  ログへの個人情報・シークレット出力は Blocker（マスキングまたは出力削除）。

### 3.3 監視・アラート（リポジトリ外で満たされうる）

- **何を確認するか**: 死活監視の受け口（ヘルスチェックエンドポイント）と、
  異常に気づける仕組み（アラート）があるか。
- **機械的な確認方法**: `rg -n "/health|/healthz|/ping|/status|livenessProbe|readinessProbe"` で
  ヘルスチェックの実装・設定を確認。監視 SaaS の設定ファイル（`datadog.yaml` 等）や
  IaC（`terraform/` 配下）に監視リソースがあるかも見る。**監視の実設定はコードから確定できない**
  ため、無くても不足と断定しない。
- **不足時の典型対応**: ヘルスチェックエンドポイントが無ければ Should（数行で足せる）。
  アラートの有無は要確認として「本番の監視・通知はどう設定していますか」を質問に出す。
  一般公開なら初回リリースまでに最低限の死活監視を Should とする。

### 3.4 エラートラッキング

- **何を確認するか**: 本番で起きた例外を集約・通知する仕組み（Sentry / Rollbar /
  Bugsnag / GlitchTip 等）が入っているか。
- **機械的な確認方法**: `rg -n "sentry|rollbar|bugsnag|@sentry/|raven" -i` と、
  依存マニフェスト（`package.json`, `requirements.txt`, `go.mod` 等）内の該当パッケージを確認。
  DSN が環境変数から来ているか（ハードコードされていないか）も見る。
- **不足時の典型対応**: 一般公開なら Should（ユーザーはエラーを報告してくれないため）。
  社内ツールならログ確認の運用が決まっていれば Nice。導入自体は SDK 初期化数行で済むことが多い。

## 4. セキュリティ

### 4.1 認証・認可

- **何を確認するか**: 保護されるべきエンドポイント・画面に認証がかかっているか。
  認証済みユーザーの**権限**チェック（他人のリソースを操作できないか＝IDOR）があるか。
- **機械的な確認方法**: ルーティング定義を列挙し（フレームワークの流儀で
  `rg -n "router\.|@app\.route|@GetMapping|urlpatterns"` 等）、認証ミドルウェア・デコレータ
  （`rg -n "auth|login_required|authenticate|middleware|@PreAuthorize|requireAuth" -i`）が
  かかっていないルートを洗い出す。ID をパラメータに取る更新・削除系ハンドラで、
  所有者チェック（`user_id` の突き合わせ）をしているかをコード読解で確認する。
- **不足時の典型対応**: 保護されるべきルートの認証欠落・所有者チェック欠落は Blocker。
  該当箇所の深掘りが必要なら `arc-reactor-code-review` のセキュリティ観点に回す。

### 4.2 シークレット管理

- **何を確認するか**: API キー・DB パスワード等がコードや git 履歴にコミットされていないか。
  環境変数・シークレットマネージャ経由になっているか。
- **機械的な確認方法**: `rg -n "api[_-]?key|secret|password|token|BEGIN (RSA|EC|OPENSSH) PRIVATE KEY" -i -g '!*.md' -g '!*test*'` でハードコード疑いを列挙し、値が直書きか環境変数参照かを読み分ける。
  `.env` が `.gitignore` に入っているか、`git log --all --diff-filter=A -- '*.env' '*.pem'` で
  過去のコミット混入が無いかも確認する。
- **不足時の典型対応**: 現行コードの直書き・履歴への混入は Blocker。**鍵のローテーション
  （無効化と再発行）が先**で、履歴からの削除はその後（削除だけでは漏えいは解消しない）。
  値はレポートに転記せず、場所だけ指摘する。

### 4.3 依存脆弱性

- **何を確認するか**: 既知の脆弱性を持つ依存パッケージ・EOL ランタイムを使っていないか。
- **機械的な確認方法**: ロックファイル・マニフェストのバージョンを読み、Claude の知識の範囲で
  既知の脆弱性・EOL を照合する（ネットワーク照会はしない）。知識が古い可能性があるため、
  `npm audit` / `pip-audit` / `govulncheck` / Dependabot 有効化のいずれかの実行をユーザーに依頼し、
  結果が出るまでは要確認とする。
- **不足時の典型対応**: 既知の重大脆弱性（RCE・認証バイパス級）が確認できたら Blocker。
  audit 系の定常実行が無いこと自体は Should（CI に 1 ステップ足す。
  GitHub なら `fix-dependabot` スキルの定常運用も選択肢）。

### 4.4 入力検証

- **何を確認するか**: 外部入力（リクエストボディ・クエリ・ファイルアップロード）を
  検証してから使っているか。SQL 文字列連結・生のコマンド実行に入力が流れていないか。
- **機械的な確認方法**: バリデーション層の有無（`rg -n "zod|joi|yup|pydantic|validator|@Valid|validates" -i`）と、
  危険パターン（`rg -n "execute\(.*(f\"|\+|%|format)|query\(.*\+|exec\(|eval\(|dangerouslySetInnerHTML|os\.system|subprocess.*shell=True"`）を突き合わせ、
  ヒット箇所に外部入力が到達するかを読解で確認する。ファイルアップロードがあれば
  サイズ・拡張子・保存先の検証を見る。
- **不足時の典型対応**: 外部入力が届く SQL 連結・コマンド実行・eval は Blocker
  （プレースホルダ・引数配列形式へ書き換え）。バリデーション層が無い場合、一般公開なら
  Should 以上（主要エンドポイントから導入する）。

### 4.5 HTTPS / セキュリティヘッダ（リポジトリ外で満たされうる）

- **何を確認するか**: 通信が HTTPS 前提になっているか。基本的なセキュリティヘッダ
  （HSTS, X-Content-Type-Options, CSP 等）と、Cookie の Secure / HttpOnly / SameSite 属性。
  CORS が `*` 開放になっていないか。
- **機械的な確認方法**: `rg -n "http://" -g '!*test*' -g '!*.md'` で平文 URL の残りを確認
  （localhost 向けは 1.3 と重複してよい）。`rg -n "helmet|secure_headers|HSTS|Content-Security-Policy|X-Frame-Options|SameSite|CORS|Access-Control-Allow-Origin"` で
  ヘッダ設定の有無と CORS 設定値を確認する。TLS 終端はロードバランサ・CDN 側のことが多いため、
  コードに無くても不足と断定しない。
- **不足時の典型対応**: 認証ありサービスの Cookie 属性欠落・CORS `*` 開放（credentials 併用）は
  Blocker 級。ヘッダ類はミドルウェア（helmet 等）導入で Should。TLS 終端の場所は要確認として
  「本番の TLS はどこで終端していますか」を質問に出す。

## 5. パフォーマンス・容量

### 5.1 N+1・無制限クエリ

- **何を確認するか**: 一覧系の処理でループ内クエリ（N+1）や、LIMIT なし・ページネーションなしの
  全件取得をしていないか。
- **機械的な確認方法**: ORM の遅延ロードをループ内で触るパターン、
  `rg -n "findAll\(|\.all\(\)|SELECT \* FROM" -i` で全件取得系を列挙し、呼び出し元に
  LIMIT / ページネーションがあるかを読解で確認する。for ループの中に
  クエリ発行（`await .*find|\.query\(|\.get\(` 等）が入っていないかを主要一覧処理で見る。
- **不足時の典型対応**: データが増える表への無制限クエリは Should（一般公開・データ増加が
  速い想定なら Blocker 寄り）。ページネーションと eager loading（JOIN / prefetch）へ書き換える。
  データ量が構造的に少ない社内ツールなら Nice。

### 5.2 レートリミット（リポジトリ外で満たされうる）

- **何を確認するか**: 認証・送信系エンドポイント（ログイン、メール送信、高コスト API）に
  回数制限があるか。従量課金の外部 API 呼び出しが無制限に叩かれうる構造でないか。
- **機械的な確認方法**: `rg -n "rate.?limit|throttle|slowapi|express-rate-limit|Bucket4j" -i` で
  実装の有無を確認。無い場合、ログイン・メール送信・外部 API 呼び出しのエンドポイントを列挙して
  「制限なしで叩けるもの」を特定する。API ゲートウェイ・WAF 側で制限している場合があるため、
  コードに無くても不足と断定しない。
- **不足時の典型対応**: 一般公開のログイン（ブルートフォース）・メール送信（踏み台化）・
  従量課金 API（コスト爆発）の無制限は Should 以上。ミドルウェア導入かインフラ側設定で対応し、
  インフラ側にあるかは要確認として質問に出す。

### 5.3 想定負荷への言及

- **何を確認するか**: 想定ユーザー数・トラフィックに対して、現在の構成で足りるかの検討が
  どこかに残っているか（負荷試験の記録、キャパシティの見積もりメモ、コネクションプール設定）。
- **機械的な確認方法**: ドキュメント内を `rg -n "負荷|load test|capacity|performance|k6|locust|jmeter|ab -n" -i` で確認。DB コネクションプール・ワーカー数の設定値
  （`pool_size`, `max_connections`, `workers` 等）が既定値のままかも見る。
- **不足時の典型対応**: これはコードの不足ではなく検討の不足として扱う。規模感ヒアリングの
  結果と突き合わせ、一般公開・初速のアクセスが読めないケースでは「主要導線 1 本の簡易負荷試験」を
  Should として提案する。社内ツール・利用者数が確定しているなら OK / Nice。

## 6. データ

### 6.1 バックアップ（リポジトリ外で満たされうる）

- **何を確認するか**: 消えると困るデータ（DB・アップロードファイル）のバックアップが
  取られる仕組みと、戻せることの確認（リストア手順）があるか。
- **機械的な確認方法**: IaC・cron 定義・スクリプトを `rg -n "backup|snapshot|pg_dump|mysqldump|BackupRetention" -i` で確認。マネージド DB（RDS 等）の IaC 設定に
  バックアップ保持期間があるかも見る。**実運用のバックアップはコードから確定できない**ため、
  痕跡が無くても不足と断定しない。
- **不足時の典型対応**: 要確認の筆頭。「バックアップの取得と、リストアを一度でも試したか」を
  質問に出す。ユーザーデータを預かる一般公開サービスでバックアップ無しが確定したら Blocker。

### 6.2 マイグレーション手順

- **何を確認するか**: スキーマ変更がマイグレーションツールで管理されているか。
  リリース時に流すマイグレーションと、その実行方法（自動 / 手動）が決まっているか。
- **機械的な確認方法**: `migrations/` / `db/migrate/` / `alembic/` ディレクトリ、
  `rg -n "alembic|flyway|golang-migrate|prisma migrate|knex|typeorm migration" -i` で
  ツールの有無を確認。未適用マイグレーションの有無・実行タイミング（デプロイスクリプトに
  組み込まれているか）を CI / デプロイ設定と突き合わせる。
- **不足時の典型対応**: スキーマを手作業 SQL で変えている場合は Should
  （マイグレーションツール導入。既存スキーマの初期マイグレーション化から始める）。
  ツールはあるが実行タイミングが未定なら Should（デプロイ手順に明記する）。

### 6.3 ロールバック可否

- **何を確認するか**: リリースに失敗したとき前の版に戻せるか。特に、マイグレーションが
  後方互換を壊す変更（カラム削除・型変更）を含む場合に、旧バージョンのアプリが動くか。
- **機械的な確認方法**: マイグレーションの down 定義の有無、直近のマイグレーション内容に
  破壊的変更（`rg -n "drop_column|DropColumn|removeColumn|DROP TABLE|ALTER .* DROP" -i`）が
  含まれるかを確認。デプロイ方式（コンテナタグ・リビジョン管理）が戻す操作に対応しているかは
  デプロイ設定から読み取る。
- **不足時の典型対応**: 破壊的マイグレーションと同時リリースは Should 以上
  （expand-contract パターンに分割: 先に追加だけ入れ、削除は次のリリースへ）。
  「戻す手順」が文書に無ければ運用（7.3 Runbook）側の不足として扱う。

## 7. 運用

### 7.1 デプロイ手順の再現性

- **何を確認するか**: デプロイが再現可能な形（スクリプト・CI/CD・IaC）になっているか。
  「本人の手元でしか成功しない手順」になっていないか。
- **機械的な確認方法**: デプロイ関連ファイル（`.github/workflows/` のデプロイジョブ、
  `Dockerfile`, `docker-compose.yml`, `Makefile`, `deploy.sh`, `terraform/`, `Procfile` 等）の
  有無と中身を確認。手順書（README・docs）の記述とスクリプトの実体が一致しているかも見る。
- **不足時の典型対応**: 手順が頭の中にしか無い場合は Should（最低限、コマンド列を
  スクリプトか README に固定する）。一般公開・継続リリース前提なら CI/CD 化まで Should。

### 7.2 環境変数の文書化

- **何を確認するか**: 本番稼働に必要な環境変数の一覧（名前・用途・必須か任意か）が
  文書化されているか。`.env.example` とコードが参照する変数が一致しているか。
- **機械的な確認方法**: コードが参照する変数を列挙し
  （`rg -no "process\.env\.[A-Z_]+|os\.environ\[.[A-Z_]+|os\.Getenv\(.[A-Z_]+|ENV\[.[A-Z_]+" | sort -u`）、
  `.env.example` / README の一覧と突き合わせて差分を出す。
- **不足時の典型対応**: 差分（コードは参照するが文書に無い変数）を列挙して Should
  （`.env.example` に追記。起動時の必須変数チェックを足すとなお良い）。
  一覧自体が無ければ上記コマンドの結果を下敷きに作る。

### 7.3 Runbook（リポジトリ外で満たされうる）

- **何を確認するか**: 障害・異常時の初動手順（再起動方法、ログの見場所、ロールバック手順、
  よくある障害と対処）が書かれた文書があるか。
- **機械的な確認方法**: `rg --files -g '*runbook*' -g '*RUNBOOK*' -g '*operations*' -g '*troubleshoot*' -i` と、docs/ 配下・README の運用セクションの有無を確認。
  社内 wiki 等リポジトリ外にある場合が多いため、無くても不足と断定しない。
- **不足時の典型対応**: 要確認として「障害時の初動手順はどこにありますか」を質問に出す。
  どこにも無いことが確定したら Should（最低限: 再起動・ログ確認・ロールバックの 3 項目を
  README に書く）。運用者が本人だけの社内ツールなら Nice。

### 7.4 障害時連絡（リポジトリ外で満たされる）

- **何を確認するか**: 障害に気づいたユーザー・関係者が誰にどう連絡するか、
  対応する側の連絡経路（オンコール・通知先チャンネル）が決まっているか。
- **機械的な確認方法**: コードからはほぼ判定できない。README・docs に問い合わせ先の記載が
  あるか、アラート通知先の設定（3.3 と重複）があるかを見る程度に留める。
- **不足時の典型対応**: 原則すべて要確認。「障害時、利用者からの連絡はどこに来て、
  誰が受けますか」を質問に出す。一般公開なら問い合わせ窓口（メール・フォーム）の明記を Should とする。

## 8. ドキュメント

### 8.1 README

- **何を確認するか**: 開発者向けに、プロジェクトの概要・セットアップ手順・起動方法・
  テスト実行方法が書かれているか。記述が現状のコードと食い違っていないか。
- **機械的な確認方法**: README の有無と構成を確認し、記載されたコマンド・バージョン・
  ディレクトリ名が実体（`package.json` の scripts、実在するパス、ランタイムバージョン指定
  ファイル）と一致するかを突き合わせる。
- **不足時の典型対応**: README が無い・古い場合は Should（引き継ぎ・復帰コストに直結）。
  作成・書き直しは `friday-design-doc-generator` に回せる。単独開発でも 3 か月後の自分は
  他人なので、社内ツールでも Nice 止まりにはしない。

### 8.2 利用者向け説明

- **何を確認するか**: エンドユーザーが使い方に迷わない材料（画面内の説明、ヘルプページ、
  利用ガイド、公開 API ならその仕様書）があるか。
- **機械的な確認方法**: `docs/` 配下の利用者向け文書、ヘルプ画面のルート・コンポーネント
  （`rg -n "help|guide|faq|usage" -i -g '*.tsx' -g '*.vue' -g 'templates/**'`）、
  API なら OpenAPI 定義（`openapi.yaml`, `swagger`）の有無を確認する。
- **不足時の典型対応**: 画面を見れば分かる規模の社内ツールなら Nice。一般公開・操作が
  自明でないサービスは Should（最低限、初回導線とよくある質問）。公開 API の仕様書欠落は Should 以上。

## 9. 法務・コンプライアンス表記

> **このカテゴリの扱い（厳守）**: 該当法域・適用法令の要否を**断定しない**。
> 「一般公開＋課金ありなら特商法表記の要否確認が必要」のように、確認事項の形で出す。
> レポートには「情報整理であって法的助言ではない。最終判断は本人または専門家が行う」旨の
> 免責を毎回明記する。社内ツールならこのカテゴリは大きく軽くなる（ライセンス表記以外は
> ほぼ対象外）。一般公開・課金あり・個人情報ありの組み合わせでは最も重くなる。

### 9.1 利用規約

- **何を確認するか**: 一般公開サービスとして、利用条件・禁止事項・免責を定めた規約が
  用意され、ユーザーが到達できる場所に置かれているか。
- **機械的な確認方法**: `rg --files -g '*terms*' -g '*tos*' -i` とルーティング・フッターの
  リンク（`rg -n "利用規約|terms of service|/terms" -i`）を確認する。
- **不足時の典型対応**: 一般公開なら「公開前に規約を用意するか」の要確認を出す
  （課金あり・ユーザー投稿ありなら重要度が上がる、と文脈を添える）。内容の妥当性は
  本スキルの範囲外で、専門家確認を促す。

### 9.2 プライバシーポリシー

- **何を確認するか**: 個人情報（アカウント情報・アクセス解析含む）を扱う場合に、
  取得項目・利用目的・第三者提供を示すポリシーがあり、到達できる場所に置かれているか。
- **機械的な確認方法**: `rg --files -g '*privacy*' -i` とフッターリンクを確認。
  実際に何を集めているかの裏取りとして、解析タグ（`rg -n "gtag|google-analytics|GA_|mixpanel|amplitude" -i`）や
  個人情報らしきカラム（email, name, address, phone）の存在を突き合わせ、
  「集めているのにポリシーが無い / ポリシーに書いていない」の不一致を指摘する。
- **不足時の典型対応**: 個人情報ありの一般公開でポリシー不在なら Blocker 級の要確認として
  最上部に出す（「無いままの公開は避け、公開前に整備するか専門家に確認を」と添える。
  要否の断定はしない）。

### 9.3 特定商取引法表記（課金あり）

- **何を確認するか**: 課金がある場合に、事業者情報・価格・支払方法・返金条件等の表記
  （日本向けなら特定商取引法に基づく表記）が用意されているか。
- **機械的な確認方法**: `rg --files -g '*tokushoho*' -g '*legal*' -g '*commerce*' -i`、
  `rg -n "特定商取引|特商法" ` を確認。課金の実体は決済 SDK（`rg -n "stripe|paypal|pay\.jp|square" -i`）の
  有無で裏取りする。
- **不足時の典型対応**: 課金あり・日本向け公開なら「特商法表記の要否と記載内容を公開前に
  確認」を要確認の上位に出す。海外向け・プラットフォーム経由課金（App Store 等）は要件が
  変わるため、その旨も確認事項に含める。要否の断定はしない。

### 9.4 ライセンス表記

- **何を確認するか**: (a) 自プロダクトのライセンス方針が決まっているか（公開リポジトリなら
  LICENSE ファイル）、(b) 依存 OSS のライセンス義務（表記義務・コピーレフト）に反していないか。
- **機械的な確認方法**: LICENSE ファイルの有無。依存のライセンスは
  `license-checker` / `pip-licenses` / `go-licenses` の実行をユーザーに依頼するのが確実
  （ローカルでは `node_modules/*/package.json` の license フィールド読解等で概観できる）。
  GPL / AGPL 系の依存が商用クローズドソース配布と衝突しないかは特に確認する。
- **不足時の典型対応**: コピーレフト依存と配布形態の衝突疑いは要確認の上位（該当依存名と
  ライセンス名を根拠に添える）。サードパーティライセンス一覧画面・NOTICE の不在は
  一般公開なら Should。判断が割れるものは専門家確認を促す。
