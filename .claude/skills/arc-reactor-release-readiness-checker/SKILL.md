# release-readiness-checker — リリース準備状況のレビュー (arc-reactor)

「このアプリケーション、リリースまでにあと何が要るか」を観点別チェックリストで棚卸しし、
足りていないところを**根拠付き・重要度付き**のレポートにするスキル。
感覚頼みのリリース判断を、チェックリスト駆動の網羅的な確認に置き換えることが目的。

> 大原則: **コードから確認できないことを「無い」と断定しない。**
> コードで確認できた事実には根拠 `file:line` を添え、確認できないこと
> （本番インフラ・監視設定・契約・法域の要件）はすべて「要確認」としてユーザーへの質問に回す。

persona: [`personas/arc-reactor.md`](../../../personas/arc-reactor.md) に従う
（応答は日本語 / コード・識別子は英語 / 推測は ASSUMPTION 明記）。
文章表現は [`personas/writing-style.md`](../../../personas/writing-style.md) に従う。

## トリガー

| ユーザー発話の例 | 動作 |
|---|---|
| 「このアプリ、リリースできる状態か確認して」 | 全カテゴリでレビュー |
| 「リリースまでに何が足りないか洗い出して」 | 全カテゴリでレビュー |
| 「本番公開前のチェックをして」 | 全カテゴリでレビュー |
| 「release readiness をチェックして」 | 全カテゴリでレビュー |
| 「一般公開して大丈夫か見て」 | 一般公開文脈を前提にレビュー（ヒアリングで確定） |
| 「セキュリティと法務まわりだけリリース観点で見て」 | カテゴリを限定してレビュー |

ユーザーが明示的に依頼したときだけ起動する（コードを読んだついでに自動起動しない）。

## 標準フロー

### 1. リリース文脈のヒアリング（初回に確定）

判定の重み付けを決めるため、レビュー開始前に次を確定する。発話や既存資料
（README・過去のレポート）から読み取れるものは確認だけにし、足りない分をまとめて聞く。

- 対象リポジトリの絶対パス（`projects/<name>/` 配下が典型）
- **何を・誰に・いつ・どこへ**リリースするか（プロダクトの内容 / 利用者 / 予定日 / デプロイ先）
- 規模感（想定ユーザー数・トラフィックの桁）
- **社内ツールか一般公開か**
- **課金の有無**
- **個人情報の有無**（扱うならどの種類か）

このヒアリング結果はレポート冒頭に「前提」として記載する。文脈が変われば判定も変わるため。

### 2. カテゴリ別チェック

[references/readiness-checklist.md](references/readiness-checklist.md) の観点別チェックリストに従い、
カテゴリごとに `rg` / ファイル存在確認 / 設定ファイル読解でシグナルを収集する。
**チェック実行前に必ず references を読む**こと。カテゴリ:

1. 機能完成度（TODO/FIXME 残、フィーチャーフラグの状態、ハードコードされた仮実装）
2. テスト（主要導線のテスト有無、CI の有無と通過状況）
3. エラー処理・可観測性（エラーハンドリング、ログ、監視・アラート、エラートラッキング）
4. セキュリティ（認証・認可、シークレット管理、依存脆弱性、入力検証、HTTPS/セキュリティヘッダ）
5. パフォーマンス・容量（N+1・無制限クエリ、レートリミット、想定負荷への言及）
6. データ（バックアップ、マイグレーション手順、ロールバック可否）
7. 運用（デプロイ手順の再現性、環境変数の文書化、Runbook、障害時連絡）
8. ドキュメント（README、利用者向け説明）
9. 法務・コンプライアンス表記（利用規約・プライバシーポリシー・特商法表記・ライセンス表記）

ユーザーがカテゴリを限定した場合はそれに従う。

### 3. 判定の付け方（厳守）

各チェック項目を次の 5 分類のいずれかに置く。

- **OK**: 満たしている。根拠（`file:line` または確認できた事実）を必ず添える。
- **Blocker**: これが無いとリリース不可。
- **Should**: 初回リリースまでに強く推奨。
- **Nice**: リリース後の対応でも成立する。
- **要確認**: コードから判定できない。ユーザーへの質問文の形で出す。

判定ルール:

- **リリース文脈で重み付けする。** 同じ不足でも重さは文脈で変わる
  （例: 法務表記は社内ツールなら Nice 〜対象外、一般公開・課金ありなら Blocker 級。
  レートリミットは利用者 10 人の社内ツールなら Nice、一般公開 API なら Should 以上）。
  重み付けの根拠はステップ 1 のヒアリング結果に紐づけて 1 行書く。
- **コードから確認できないこと（本番インフラの構成・監視の実設定・バックアップの実運用・契約）は
  勝手に「無い」と断定しない。** 必ず「要確認」に回し、何を確認すればよいかを添える。
- Blocker / Should の指摘には、不足の根拠（探した場所・見つからなかった事実、
  または問題箇所の `file:line`）を必ず添える。根拠を示せない指摘はしない。
- **法務・コンプライアンス項目は該当法域の要否を断定しない。** 「一般公開＋課金ありなら
  特商法表記の要否を確認」のように、確認事項として出す。

### 4. レポート生成

以下の固定構成で Markdown レポートを生成し、チャットで要約を提示する。
保存先は対象プロジェクト配下 `<repo>/release-readiness/<YYYY-MM-DD>.md`
（対象リポジトリの git を汚す可能性があるため、**保存前に必ずユーザーに確認**する）。

```markdown
# Release Readiness Report — <repo-name> (<YYYY-MM-DD>)

## 前提（リリース文脈）
- 何を / 誰に / いつ / どこへ・規模感・社内 or 一般公開・課金・個人情報

## サマリ
- Blocker N 件 / Should N 件 / Nice N 件 / 要確認 N 件
- 結論 1〜2 行（例: 「Blocker 3 件の解消が先。特にシークレットのコミット混入」）

## Blocker — これが無いとリリース不可
| # | カテゴリ | 項目 | 根拠 (file:line / 確認事実) | 対応の方向 |
|---|---|---|---|---|

## Should — 初回リリースまでに強く推奨
（同上の表）

## Nice — リリース後でも可
（同上の表）

## OK — 確認済み
| # | カテゴリ | 項目 | 根拠 (file:line / 確認事実) |
|---|---|---|---|

## 要確認 — コードから判定できないこと（質問リスト）
| # | カテゴリ | 質問 | 確認先の例 |
|---|---|---|---|

## 免責
- 本レポートはコードと同梱資料から確認できた範囲の整理であり、リリース可否の保証ではない
- 法務・コンプライアンス項目は情報整理であって法的助言ではない。要否の最終判断は
  本人または専門家（弁護士・税理士等）が行う
- チェックしなかったカテゴリ・除外したパスの一覧
```

### 5. 完了提示

- Blocker の件数と要点、要確認リストの存在をチャットで要約する。
- 次アクションを提案する（Blocker の ToDo 化、`arc-reactor-code-review` での該当箇所の深掘り、
  `jarvis-issue-planner` での issue 化など）。

## チェックリストの詳細

項目ごとの「何を確認するか / コードから機械的に確認する方法 / 不足時の典型対応」は
[references/readiness-checklist.md](references/readiness-checklist.md) に定める。
チェックリストの改善（項目追加・確認コマンドの改良・誤検知パターンの追記）は
references への追記で行う（SKILL.md 本文は変えない）。

## 品質・安全性

- 対象リポジトリへの操作は**読み取りのみ**。不足の修正・コミットは本スキルの範囲外
  （修正はユーザーの指示を受けて別途行う）。レポート保存時のみ書き込み、保存前に確認する。
- 根拠のない指摘はしない。OK にも不足にも、検証可能な根拠（`file:line`、
  探索したパスと見つからなかった事実）を添える。
- 法務・コンプライアンス項目は情報整理であって法的助言ではない。レポートの免責に毎回明記し、
  要否の断定をしない。
- レポートはプロジェクト固有情報のため、対象プロジェクトのディレクトリ配下に閉じる
  （当リポジトリ側や他プロジェクトに持ち出さない。CLAUDE.md の分離ルールに従う）。
- チェック中に機密情報（コミットされた API キー等）を発見しても、レポートに**値を転記しない**。
  場所の指摘のみ行い、最優先の Blocker として報告する。
- 既存スキルとの棲み分け: コード品質そのものの指摘は `arc-reactor-code-review`、
  蓄積負債の棚卸しは `arc-reactor-tech-debt-auditor`。それらの領分に踏み込む深掘りが
  必要になったら、本スキルで抱え込まず該当スキルの利用を提案する。
