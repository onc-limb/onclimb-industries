# friday-doc-planner — ドキュメント事前準備（Stage 0）

ドキュメントを書き始める**前**に、対話で前提を固めるスキル。成果物はドキュメント本文ではなく、
**ドキュメント企画書（doc brief）**。brief を後段の生成スキル（Stage 1）へ引き渡すことで、
「何のために・誰に向けて・何を情報源に書くか」が曖昧なまま生成が始まることを防ぐ。

設計の出典: [`ideas/doc-plan-memo.md`](../../../ideas/doc-plan-memo.md)
（「ドキュメント作成とは何か」の壁打ち記録。分類・工程・ルールの根拠はすべてここにある。
参考書籍: ソシム『エンジニアが一生困らない ドキュメント作成の基本』）。

## 2 段構成の全体像

```
Stage 0: friday-doc-planner ── 対話で確定 ──→ doc brief（doc-briefs/ に 1 ファイル）
   目的×読者 → 型・読まれ方の導出 → テーマ設定・分解 → 情報源
                                                  │
Stage 1: 種類に応じた生成スキル ←── brief を入力に、骨組み（見出し構成）と本文を生成
         - friday-design-doc-generator（設計書 / README / ADR）
         - friday-proposal-generator（提案書・見積）
         - friday-tech-article-drafter（技術記事・登壇）
         - friday-procedure-doc-generator（運用手順書 / 開発環境構築ガイド）
         - （該当スキルが無い種類は「Stage 1 スキルが無い種類の扱い」参照）
```

- **対象外**: `friday-giziroku`（入力=文字起こし）と `friday-daily-report`（入力=jarvis-record の記録）は
  入力・テンプレートが固定のパイプライン型のため、本スキルを通さず従来どおり単体起動する。
- brief 無しでの Stage 1 スキル単体起動も従来どおり可能（後方互換）。
- **準備と執筆の境界**: 書籍の 5 工程「目的 → 読み手 → テーマ分解 → 骨組み → 文章」のうち、
  本スキルは**テーマ分解まで**。骨組み（見出し構成）と文章は Stage 1 の領分。

## トリガー

| ユーザー発話の例 | 動作 |
|---|---|
| 「ドキュメントを作りたいので整理から手伝って」 | ヒアリングから開始 |
| 「どんなドキュメントにするか壁打ちして」 | ヒアリングから開始 |
| 「何をどこに書くか整理してから書き始めたい」 | ヒアリングから開始 |
| 「この brief で続きを作って」「brief を更新したい」 | 既存 brief を Read して再開・改訂 |

自動起動はしない。作りたいドキュメントの種類が明確な依頼
（「README 作って」「提案書作って」等）は、従来どおり各 Stage 1 スキルが直接起動してよい。

## 分類の基礎知識（導出に使う 2 軸）

### 軸 1: 型（= 目的の種類）。型が決まると「最も主張すべきこと」の形が決まる

| 型 | 読後に読者に起きてほしいこと |
|---|---|
| **説明型** | 理解し、**それに従って正しく動ける**（行動の中身はドキュメントが与える。例: 手順書・設計書） |
| **説得型** | **書き手の望む特定の判断**をする（結論を書き手が持っている。例: 提案書） |
| **報告型** | 現状を理解し、**自分の判断を下せる材料**を得る（結論は読者に委ねる。例: 進捗・障害報告） |

### 軸 2: 読まれ方（= 情報取得の主体）。決まると流れの作り方が切り替わる

| 読まれ方 | 情報取得の主体 | 流れの作り方 |
|---|---|---|
| **辞書的** | 読み手主導（pull）。「こういう情報があるだろう」と推測して引きに来る | 読者が持ってくる**疑問の列挙**が構成の元になる |
| **読み物的** | 書き手主導（push）。読み手に前提がない | 読者の**出発点とゴール**を決め、間を運ぶストーリーを作る |

### マトリクスは実質 5 マス

**説得型 × 辞書的は存在しない**（説得は書き手主導の行為なので、読み手主導の pull と両立しない）。

| | 辞書的 | 読み物的 |
|---|---|---|
| **説明型** | 設計書 / README / 手順書 | 技術記事・チュートリアル |
| **説得型** | —（存在しない） | 提案書・見積 |
| **報告型** | 議事録（決定事項・TODO を引く） | 進捗報告・障害報告 |

### 導出ルール

分類は **6 択で直接聞かない**。目的と読者のヒアリングから導出し、根拠付きでユーザーに確認する。

- 目的（読後に読者に起きてほしいこと）→ **型**が決まる
- 読者の状況（疑問を持って引きに来るか、こちらから届けて読ませるか）→ **読まれ方**が決まる

## 標準フロー

### 1. 目的 × 対象読者のヒアリング

- **目的は「読者の行動レベル」まで深掘りする**。「共有したい」「まとめたい」で止めない。
  読後に読者が具体的に何をする / できるようになるかまで対話で落とし込む
  （例: 「進捗を伝えたい」→「上層部への連絡とスケジュール判断の材料にしてもらう」）。
- **対象読者**: 誰が・どんな前提知識で・どういう状況で読みに来るか。
- **前段の文脈**: このドキュメントの前に何があったか（例: 報告に添えた案が反対された /
  比較を指示された → 説得型が生まれる典型パターン）。無ければ無しでよい。
- 会話の文脈から既に読み取れている項目は、聞き直さず「この理解で合っているか」の確認だけにする。

### 2. 型と読まれ方の導出・確認

- 導出ルールに従って型と読まれ方を判定し、**導出根拠付きで**ユーザーに確認する
  （例: 「読後にセットアップを完了してほしい → 説明型。困ったとき手順を引きに来る → 辞書的」）。
- ユーザーが違うと言えばヒアリングに戻る（分類が合わないのは目的か読者の理解がずれている兆候）。

### 3. テーマの設定

- **目的を達成するために「何を重点的に伝えるか」**を決める。目的（読者に起きること）と
  テーマ（伝えることの焦点）は別物（例: 日次報告 — 目的は毎回同じでも、テーマ
  「今日は何を重点的に伝えるか」は毎回変わる）。

### 4. テーマの分解

- テーマを **主張の選定 → 主張を支えるのに必要な情報 → 必要な流れ** の順に深掘る。
- **「なぜ・何を・どうやって」の 3 観点**で充足を検査する（なぜ = 理由・背景 / 何を = 中身 /
  どうやって = 手段・手順。3 つが揃って初めて読者が動ける）。欠けている観点は対話で埋めるか、
  意図的に省く理由を brief に書く。
- **型の混在検査**をここで行う（下記「ルール」参照）。

### 5. 情報源の対話埋め

- 分解された**テーマごと**に「どこから情報を持ってくるか」を対話で埋める:
  コード（リポジトリ・パス）/ worklog・digest / 議事録 / 既存ドキュメント / ユーザーの頭の中
  （＝執筆時にヒアリングが必要な項目）等。
- 情報源が「ユーザーの頭の中」の項目は、Stage 1 でのヒアリング対象として brief に明示する。

### 6. brief 化と引き渡し

- ここまでの確定内容を `templates/doc-brief.md` の書式で `doc-briefs/<YYYY-MM-DD>-<slug>.md` に
  書き出す（slug は英語 kebab-case）。未確定項目は TBD として残す（勝手に埋めない）。
- 種類に対応する Stage 1 スキルを提案し、ユーザーの指示で起動する。
  - 同一セッション: そのまま該当スキルに続きを渡す。
  - 別セッション: 「`doc-briefs/<file>` を入力に <Stage 1 スキル> を起動」と案内する。
- brief 完成時に、保存パス・分類・テーマ分解の要約・TBD の残数をチャットで提示する。

## ルール（壁打ちで確定した原則）

- **1 ドキュメント 1 型**。型（説明/説得/報告）は目的そのものなので 1 ドキュメント内で混ぜない。
  混ざっていたら**分割を提案**する。ただし機械的に判定せず**主従で見る**:
  主目的の付随（例: 報告の結果としての次アクション案）は分割不要。
  判定基準は**「分割したときにストーリーがおかしくなるか、ならないか」**。
- **読まれ方の混在は章分けで可、ただしサンドイッチにしない**（辞書的な章と読み物的な章を
  交互に挟まない）。読み物的な部分は先頭か末尾に寄せる。分割した方が良いと判断したら
  別ドキュメントへの分割も考慮する。
- **説得型ドキュメントの生まれ方に注意**: 報告に添えた案への反対・比較指示から生まれることが
  多い。前段の文脈を聞くのはこのため。
- 骨組み（見出し構成）と本文は書かない。Stage 1 の固定テンプレート・生成フローを尊重する。

## doc brief の書式

雛形: [`templates/doc-brief.md`](templates/doc-brief.md)。フロントマター + 固定見出し。

- フロントマター: `type`（説明/説得/報告）/ `reading-mode`（辞書的/読み物的/混在）/
  `target-skill`（引き渡し先。無ければ `none`）/ `status`（draft/fixed）/ `project`（任意）
- 固定見出し: 目的（読者の行動レベル）/ 対象読者 / 前段の文脈 / 分類と導出根拠 /
  最も主張すべきこと / テーマ / テーマ分解 / 混在検査 / 未確定事項（TBD）
- 見出しの削除・順序変更はしない（埋まらない場合は TBD を入れる）。

## データ配置

- 出力: リポジトリ直下 `doc-briefs/<YYYY-MM-DD>-<slug>.md`（**git 管理外**。
  クライアント名等の機密を含みうるため。`.gitignore` に理由コメント付きで登録済み）。
- brief は納品物ではなく準備メモ。対象プロジェクトの成果物ディレクトリには置かない
  （最終ドキュメントの置き場所は Stage 1 スキルの流儀に従う）。

## 改訂フロー

- 既存ドキュメントの改訂・brief の作り直しは、既存 brief を Read して差分を対話で確定し、
  **同じファイルを更新**して再引き渡しする（`status` を一旦 `draft` に戻し、確定で `fixed`）。
- 対応する brief が無い既存ドキュメントの改訂は、新規作成と同じフローで brief を起こしてよい
  （既存ドキュメント自体を情報源の一つとして扱う）。

## Stage 1 スキルが無い種類の扱い

- 仕様書・FAQ など対応スキルが無い種類も、brief の確定までは本スキルが行う
  （`target-skill: none`）。手順書（運用手順書・環境構築ガイド）は
  `friday-procedure-doc-generator` が Stage 1 を担うため、この扱いの対象外。
- 本文は同セッションの通常作業として書いてよい。その場合も friday 共通ルール
  （[`personas/friday.md`](../../../personas/friday.md)・[`personas/writing-style.md`](../../../personas/writing-style.md) の清書パス）を適用する。
- **同じ種類が繰り返し出たら、Stage 1 スキル化を提案**する（ideas/skill-feedback.md の運用に乗せる）。

## 品質・安全性（persona: friday 準拠）

- 入力に存在しない情報を創作しない。ユーザーが確定していない項目は brief に TBD と明示し、
  勝手に埋めない（分類の導出は必ずユーザー確認を挟む）。
- 応答・brief は日本語。コード・識別子・技術用語は英語（原語）のまま表記する。
- 機密情報（API キー・トークン等の実クレデンシャル）を brief に含めない。
  クライアント名・案件名は brief（git 管理外）には書いてよいが、Stage 1 の出力での扱いは
  各スキルの機密ルールに従う。
- 日本語の表記（濁点・半濁点・特殊文字）を正しく保持し、ASCII で代替しない。
