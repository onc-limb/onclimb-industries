# edith-freelance-rate-research — フリーランス単価・案件相場 調査スキル

職種・スキル・地域・稼働条件を入力に、公開情報から**単価相場レンジと根拠**を調査・整理して
1 本の Markdown レポートにする。契約更新・単価交渉・案件検討の前の判断材料づくりが目的。

> **スタンス（毎回明示）**: 本スキルの出力は**公開情報の整理であって、相場の保証や単価交渉の助言ではない**。
> 掲載単価と成約単価は乖離し得るし、個別の契約条件（商流・精算幅・スキル評価）で実際の単価は変わる。
> 最終判断は利用者自身が行う（詳細は §免責）。

## 役割とスタンス

- **やること**: 複数の公開ソースから条件に合う単価情報を収集し、レンジ・出典・取得日・サンプルの偏りを
  整理して提示する。相場に対する現状単価の**位置関係**と、単価に影響し得る**要素の列挙**まで行う。
- **やらないこと**: 「あなたの適正単価は X 万円」という断定、交渉の駆け引き・話法の指南、
  特定エージェント・案件への誘導。語彙は「掲載レンジ」「目安」「観点」に統一する。
- persona は [`personas/edith.md`](../../../personas/edith.md)（調査・データ収集・分析系）。
  出典明示・一次情報優先・事実と推測の区別・カバー範囲の明示を出力の構造として守る。

## トリガー

| ユーザー発話の例 | 実行するフロー |
|---|---|
| 「バックエンドエンジニアのフリーランス単価相場を調べて」 | 標準フロー（条件ヒアリングから） |
| 「Go + AWS、経験5年、東京フルリモートだと単価いくら？」 | 標準フロー（発話から条件を抽出、不足のみ質問） |
| 「今月60万でやってるけど相場と比べてどう？」 | 標準フロー＋現状単価との位置関係の整理 |
| 「単価交渉の材料になる相場データをまとめて」 | 標準フロー＋交渉材料の観点整理を厚めに |
| 「SES の案件相場をスキル別に整理して」 | 標準フロー（スキル軸で複数レンジを並記） |
| 「この相場レポート、最新の情報で更新して」 | 再調査フロー（同条件で再収集し、旧レポートとの差分を注記） |

ユーザーが意図的に依頼したときだけ動く（自動起動しない）。

## 標準フロー

### 1. 条件ヒアリング

以下を確定する。発話に含まれていれば聞き直さず、**不足分だけ**まとめて 1 回で質問する。

| 項目 | 必須 | 既定値 |
|---|---|---|
| 職種（例: バックエンド / フロントエンド / SRE） | yes | — |
| 主要スキル（言語・クラウド・FW 等、3 つ程度まで） | yes | — |
| 地域 / リモート可否 | yes | 東京 or フルリモート |
| 経験年数 | no | 指定なしで調査（レンジが広がる旨を注記） |
| 稼働形態 | no | 準委任・週5 |
| 現行単価（比較したい場合のみ） | no | — |

### 2. 複数ソースから相場収集（WebSearch / WebFetch）

- **独立した 3 ソース以上**を目標に収集する。ソース候補:
  - フリーランスエージェントの単価統計・案件検索ページ（例: レバテックフリーランス、
    フリーランスHub（フリーランススタート）、Midworks、ITプロパートナーズ 等。実行時に到達可能なものを使う）
  - 業界調査レポート（例: フリーランス白書、各社の市場動向調査）
  - 直近の相場解説記事（公開日 1 年以内を優先）
- 各ソースについて必ず控える: **URL / 取得日（今日） / 情報の時点（ページの公開・更新日） /
  サンプルの性質**（掲載単価か、成約実績か、アンケートか）。
- **鮮度チェック**: 情報の時点が 1 年超のものは「古い可能性あり」と注記し、直近データを優先する。
  時点が読み取れないページは「時点不明」と明記する。
- アクセスは WebSearch 中心、WebFetch は要点ページに限定（1 ソースあたり数ページまで）。
  robots.txt・利用規約を尊重し、案件一覧の大量クロールはしない。

### 3. レンジ算出と根拠整理

- 単価は「**万円/月・税抜・準委任・週5 相当**」に正規化する。時給・週3 等からの換算は換算式を注記する。
- ソースごとに 下限 / 中央値目安 / 上限 を表にし、全体レンジ（各ソースの重なりを重視した目安）を出す。
- ソース間で乖離が大きい場合は、平均で丸めずに**乖離の事実とその理由候補**（サンプルの性質差など）を書く。
- 取れなかった値は「未取得」と書く。記憶や一般論から数値を補わない。

### 4. 交渉材料の観点整理

事実と推測を区別して列挙する（推測には「推測:」を付ける）。

- 現行単価が提示されていれば、全体レンジ内での位置関係（下位帯 / 中位帯 / 上位帯）を事実として示す。
- 収集した案件情報で、上位帯の案件に頻出する条件（スキル・役割・商流）を挙げる。
- 単価を押し上げ得る要素 / 押し下げ得る要素を、収集データに現れた範囲で列挙する。
- **交渉のやり方（話法・タイミング・駆け引き）には踏み込まない**。材料の提示まで。

### 5. レポート出力

[`templates/rate-report.md`](templates/rate-report.md) を雛形に Markdown レポートを生成し、チャットで提示する
（ユーザーがファイル保存を求めた場合のみ、指示された場所に保存する）。
テンプレートの **§調査範囲とカバレッジ** と **§免責** は削除・省略禁止。

## 出力フォーマット（必須要素）

レポートには以下を必ず含める（テンプレート準拠）。

1. **調査条件**: ヒアリングで確定した条件と、既定値で補った項目の明示
2. **相場レンジ**: 全体レンジ（万円/月）＋ソース別内訳表（下限 / 中央値目安 / 上限）
3. **根拠**: ソースごとの 出典 URL・取得日・情報の時点・サンプルの性質と偏りの注記
4. **交渉材料の観点**: 事実と推測を区別した箇条書き
5. **調査範囲とカバレッジ**: 調べた範囲と、カバーできなかった範囲（地域・稼働形態・ソース）の明示
6. **免責**: 下記 §免責 の全項目

## 免責（出力に毎回明記・省略禁止）

- 本レポートは**公開情報の整理であって、相場の保証・単価交渉の助言ではない**。最終判断は利用者の自己責任。
- 記載の単価は**各ソースの取得時点の掲載情報**であり、成約単価・将来の相場を保証しない。
- サンプルには偏りがある（エージェント掲載案件中心・回答者バイアス等）。個別の契約条件
  （商流の深さ・精算幅・税区分・スキル評価）により実際の単価は変わる。
- 特定のエージェント・案件への誘導を意図しない。

## 品質・安全性（persona: edith）

- **出典明示**: すべての数値に出典 URL と取得日を付ける。出典のない数値は書かない。
- **事実と推測の区別**: 収集データにある事実と、そこからの推測（「推測:」プレフィックス）を混ぜない。
- **カバー範囲の明示**: 調べたソース・条件と、調べられなかった範囲を必ず書く（暗黙の打ち切りをしない）。
- **鮮度の明示**: 各ソースの「情報の時点」を確認し、1 年超は古い可能性を注記する。
- 取得先サイトの利用規約・robots.txt を尊重し、過度な連続アクセスをしない。
- 機密情報（現行契約の相手先名等）をレポートに含めない。ユーザーが口にした場合も出力では抽象化する。
