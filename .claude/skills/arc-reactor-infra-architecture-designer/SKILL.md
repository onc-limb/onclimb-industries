# infra-architecture-designer — インフラ構成の壁打ち設計 (arc-reactor)

AWS / Google Cloud 等のインフラ構成を、ユーザーと対話しながら一緒に設計するスキル。
成果物は Mermaid の構成図 + コンポーネント表と、決定事項・不採用案・前提を残した設計記録。
「図を 1 枚出して終わり」ではなく、**なぜその構成にしたか**を後から追える形で残すことが目的。

> 大原則: **設計したら必ずセキュリティセルフレビューを 1 周回す。**
> 指摘が出なくても「確認した観点」を列挙する。確認漏れと確認済みの区別がつかないレビューは、
> やっていないのと同じだから。

persona: [`personas/arc-reactor.md`](../../../personas/arc-reactor.md) に従う
（応答は日本語 / コード・識別子は英語 / 推測は ASSUMPTION 明記）。

## トリガー

| ユーザー発話の例 | 動作 |
|---|---|
| 「インフラ構成を一緒に設計して」 | ヒアリングから標準フロー全体 |
| 「この要件で AWS の構成図を描いて」 | 要件を起点にヒアリング → 構成図生成 |
| 「GCP でサーバレス構成を検討したい」 | 方向性を固定してヒアリング → 候補提示 |
| 「構成案を 2〜3 案比較して」 | 候補構成の比較提示に重心を置く |
| 「この構成図のセキュリティレビューをして」 | 既存の構成図に対しフロー 4 のみ実行 |
| 「インフラ設計の壁打ちに付き合って」 | ヒアリングから標準フロー全体 |

ユーザーが明示的に依頼したときだけ起動する（コードやインフラの話が出たついでに自動起動しない）。

## 標準フロー

### 1. 要件ヒアリング（対話）

一度に全部聞かず、**2〜3 問ずつ**対話で確定する。聞く項目:

- ワークロードの種類（Web / API / バッチ / データ処理 等）
- 想定トラフィック / データ量
- 可用性要件（許容ダウンタイム、マルチ AZ / マルチリージョンの要否）
- 予算感
- 運用体制（一人か、24/365 の監視があるか）
- 既存資産（既存 VPC・アカウント・利用中のサービス）
- コンプライアンス制約（個人情報、業界規制 等）
- IaC の有無（Terraform / CloudFormation / なし）

回答が得られない項目は推測で進めてよいが、設計書に必ず ASSUMPTION として明記する。

### 2. 候補構成の提示（2〜3 案 → 1 案に収束）

特性の異なる 2〜3 案（例: フルマネージド寄り / コンテナ / サーバレス）を提示する。
各案にトレードオフ表を付ける:

| 案 | 月額コスト概算 | 運用負荷 | スケール性 | 向いている条件 |
|---|---|---|---|---|

- コスト概算は「概算」と明記し、単価の記憶が曖昧なものは幅で示す
  （正確な見積は AWS Pricing Calculator / Google Cloud 料金計算ツールを案内する）。
- 対話で 1 案に収束させる。ユーザーの選定理由と、不採用案を落とした理由を控えておく
  （フロー 5 の設計記録に使う）。

### 3. 構成図の生成

収束した 1 案を Mermaid（`graph TB` / architecture 記法）のコンポーネント図にする。
図と併せてコンポーネント表を出す:

| コンポーネント | リソース種別 | 役割 | 冗長化 | ネットワーク配置 |
|---|---|---|---|---|

図はコンポーネント間の通信経路（インターネット → LB → アプリ → DB 等）と
ネットワーク境界（VPC / サブネットの public・private）が読み取れる粒度にする。

### 4. セキュリティセルフレビュー（必須・スキップ不可）

[references/security-review-checklist.md](references/security-review-checklist.md) を読み、
チェックリストの全観点で構成図を 1 周レビューする。出力:

- **指摘**: 観点 / 該当コンポーネント / 重大度（High・Mid・Low）/ 対策案 の表
- **確認した観点の列挙**: 問題が見つからなかった観点も「確認済み」として必ず列挙する。
  確認漏れと確認済みを読み手が区別できるようにするため
- **免責**: このレビューは観点の自己点検であり、専門のセキュリティ監査の代替ではない旨を毎回明記する

### 5. 設計記録の保存

以下を含む設計ドキュメントを Markdown で生成する。

```markdown
# Infra Design — <topic> (<YYYY-MM-DD>)

## 要件サマリ
（ヒアリング結果。未回答項目は ASSUMPTION 表記）

## 検討した構成案
（2〜3 案の比較表と、不採用案を落とした理由）

## 採用構成
（Mermaid 構成図 + コンポーネント表）

## セキュリティセルフレビュー
（指摘 + 確認した観点の列挙 + 免責）

## 前提 (ASSUMPTIONS)
（推測で置いた前提の一覧）
```

保存先は対象プロジェクト配下 `<repo または projects/<name>>/infra-design/<YYYY-MM-DD>-<topic>.md`。
対象プロジェクトの git を汚す可能性があるため、**保存前に必ずユーザーに確認**する。

## 既存スキルとの棲み分け

技術選定の調査比較（出典付きの候補比較・推奨）は `edith-tech-selection-research` の領分。
本スキルは構成設計に集中し、ミドルウェア選定（DB 製品・キュー製品の比較等）で深い比較が
必要になったらそちらへ誘導する（比較結果は本スキルの入力として使える）。

## 品質・安全性

- コスト概算は「概算」と明記し、単価の記憶が曖昧なものは幅で示す。正確な見積が必要な場合は
  公式の料金計算ツール（AWS Pricing Calculator / Google Cloud 料金計算ツール）を案内する。
- セキュリティレビューの出力には、専門のセキュリティ監査の代替ではない旨を毎回明記する。
- ユーザーの回答が得られない項目は推測で進めてよいが、設計書に ASSUMPTION として必ず明記する。
- クラウドアカウントへの実操作・リソース作成は本スキルの範囲外
  （構成の Terraform 化は将来の diagram-to-terraform スキルの領分）。
- 設計記録はプロジェクト固有情報のため、対象プロジェクトのディレクトリ配下に閉じる
  （当リポジトリ側や他プロジェクトに持ち出さない。CLAUDE.md の分離ルールに従う）。
- レビュー観点の改善（新しい観点・典型リソースの追加）は
  `references/security-review-checklist.md` への追記で行う。
