# test-scaffolder — テスト雛形＋主要ケース案の生成

対象コードから、**そのプロジェクトの流儀に合った**テスト雛形と主要ケース案を生成するスキル。
「テストを書く工数が惜しく後回しになる」問題を、流儀検出 → 雛形生成 → ケース列挙 → 実行確認
の固定フローで解消する。アイデアの経緯は [`ideas/test-scaffolder.md`](../../../ideas/test-scaffolder.md) を参照。

## 役割

- テスト作成の 4 つの腰の重さ（FW の流儀想起 / 配置・命名合わせ / ケース設計 / ボイラープレート）を肩代わりする。
- **事実（実装と既存テスト）に基づいて**雛形を作る。仕様の創作はしない。期待値に確信が持てない
  ケースは skip/todo で残し、判断はユーザーに委ねる。
- 対象は単体テスト（+ 軽量な統合テスト）。E2E は範囲外。

## トリガーと対応フロー

| ユーザー発話の例 | 動作 |
|---|---|
| 「このファイルのテスト書いて」 | ファイル全体を対象に 全ステップ実行 |
| 「この関数のテスト雛形作って」 | 関数/クラス単位に対象を絞って全ステップ実行 |
| 「この変更にテスト足して」 | `git diff` / 直近の編集から対象を特定して全ステップ実行 |
| 「テストケース案を出して」 | ステップ 3（ケース列挙）のみ。ファイルは書かない |
| 「ユニットテストのたたき台作って」 | 全ステップ実行 |
| 「jest じゃなくて vitest で」等の FW 指定 | 流儀検出より**ユーザー指定を優先** |

## 標準フロー

```bash
SKILL=/Users/satoshi-onga/Documents/onclimb-industries/.claude/skills/arc-reactor-test-scaffolder
python3 "$SKILL/scripts/detect_test_style.py" <project_root> [target_file]   # 流儀検出の入力を JSON で取得
```

### 1. 既存テストの流儀検出

1. `detect_test_style.py` を実行し、言語 / 設定ファイル由来の FW hint /
   既存テストファイル一覧（対象に近い順）/ 推奨 reference を JSON で得る。
2. `test_files` があれば、**対象ファイルに近い順に最大 3 件を Read** し、次を抽出する:
   - 配置（実装と併置 or `tests/` 分離）とファイル・テスト関数の命名
   - アサーションスタイル（`assert` / `expect` / assertion ライブラリ）
   - モック・スタブの手法（`unittest.mock`, `vi.mock`, インタフェース差し替え等）
   - fixture / setup の書き方、パラメタライズ（`parametrize`, `it.each`, table-driven）
3. 既存テストが無い、または対象言語のテストが無い場合は
   `references/<language>.md` のデフォルト流儀を使う（Python / JavaScript・TypeScript / Go / Java）。
   references にも無い言語は、その言語の公式標準テスト FW を推測して使い、
   雛形内に `// ASSUMPTION:` コメントで推測である旨を明記する。
4. **優先順位**: ユーザー指定 > 既存テストの実態 > 設定ファイルの hint > references のデフォルト。
   既存テストと設定ファイルが矛盾する場合（FW 移行中等）は設定ファイル側を現行とみなすが、
   必ずユーザーに確認する。
5. 検出結果（採用 FW / 配置 / 参照した既存テスト）を**雛形を書く前に**ユーザーへ提示する。

### 2. 雛形生成

- 検出した流儀どおりの配置・命名で新規テストファイルを Write する。
  出力先はプロジェクトの既存テスト配置に従い、既存テストが無ければ言語標準の配置を提案して確認する。
- **対象のテストファイルが既に存在する場合は上書きせず、既存構造に沿った追記**を Edit で行う。
- import・fixture・setup を実際のコードに合わせて解決し、そのまま実行可能な状態にする。
- コード・識別子・テスト名は英語。テストの意図を説明するコメントは既存テストの言語慣習に合わせる
  （既存が日本語コメントなら日本語）。

### 3. 主要ケース列挙

- 対象コードの公開 API ごとに、**正常系 / 境界値 / 異常系** の 3 分類でケースを設計し、
  下記「出力フォーマット」の表でチャットに提示する。3 分類のいずれかが本当に存在しない場合のみ
  「該当なし（理由）」と書く。
- 実装から期待値を確定できるケースは実装済みテストとして書く。確信が持てないケースは
  skip/todo マーカー付きで雛形に置き、`// ASSUMPTION:` で推測箇所を明記する。
- 対象が巨大（多責務クラス等）な場合は、公開 API の重要度順に分割し、
  今回どこまで書くかを先に合意する。

### 4. 実行確認

- 生成・編集したテストファイル**だけ**をテストランナーで実行する
  （例: `pytest path/to/test_x.py`, `npx vitest run path/to/x.test.ts`, `go test -run TestX ./pkg/...`）。
  フルスイートは回さない。
- 失敗した場合は「生成物の欠陥（修正して再実行）」と「環境要因（依存未導入・外部サービス）」を
  切り分けて報告する。依存パッケージを勝手にインストールしない。
- 実行不能な環境なら、その旨と実行コマンドを明記して確認をユーザーに委ねる。
- 可能ならプロジェクトのリンター/フォーマッタも生成ファイルに対して実行する。

## 出力フォーマット（チャット返信）

```markdown
## テスト雛形を生成しました

### 検出した流儀
- FW: vitest（package.json devDependencies / 既存テスト 3 件から検出）
- 配置・命名: 実装と併置 / `<name>.test.ts`
- 参照した既存テスト: `src/lib/date.test.ts` ほか 2 件

### 生成ファイル
- `src/service/user.test.ts`（新規）

### ケース一覧
| 分類 | ケース | 入力 | 期待 | 状態 |
|---|---|---|---|---|
| 正常系 | 有効な ID でユーザー取得 | `"u1"` | User オブジェクト | 実装済み |
| 境界値 | 空文字 ID | `""` | ValidationError | 実装済み |
| 異常系 | 存在しない ID | `"nope"` | NotFoundError | skip（期待値要確認） |

### 実行結果
`2 passed, 1 skipped`

### 残 TODO
- skip の 1 件は仕様確認後に期待値を確定してください
```

## 品質・安全性（arc-reactor persona 共通ルール）

- 応答・説明は日本語。コード・テスト名・識別子は英語。
- **既存コードのスタイルに従う**。新しいテスト規約・アサーションライブラリを勝手に導入しない。
- 推測で補った箇所（期待値・FW 選定・配置）は必ず `// ASSUMPTION:` コメントで雛形内に明記する。
- 生成後は必ずテスト実行（+ 可能ならリンター）で確認する。落ちるテストを skip なしで残さない。
- テストデータに機密情報（実在の API キー・トークン・顧客名・個人情報）を含めない。
  fixture はダミー値で作る。
- 既存ファイルの上書き・削除など不可逆な操作は事前に確認する。
- コミットは指示されたときのみ。その場合は `test:` プレフィックスで、実装変更とは分ける。
- 参画先プロジェクトで得た流儀・コードの内容を本リポジトリ側に書き残さない
  （汎化できる知見のみ、固有情報を除いて `references/` へ手動追記する）。
