# レビュー観点チェックリスト

arc-reactor-pr-reviewer が PR diff に適用する固定チェックリスト。
主観点（バグ / 可読性 / 規約）は常に適用し、補助観点（テスト / セキュリティ）は該当する変更があるときに適用する。
観点の改善・追加は本ファイルを人が意図的に更新する（自動学習はしない）。

## 重要度ラベル

| ラベル | 意味 | 使いどころ |
|---|---|---|
| `[must]` | 修正必須。マージ前に対応が必要 | バグ、データ破壊、機密混入、明確な規約違反 |
| `[imo]` | 提案。自分ならこうする（対応は任意） | 設計・可読性の改善案で、確度は高いが必須ではないもの |
| `[nit]` | 些細な指摘。好みに近い | typo、軽微な様式ゆれ |
| `[ask]` | 質問。意図の確認 | diff 外のコンテキスト不足で断定できない懸念、設計意図の確認 |

- 断定できる根拠が無い指摘は `[must]`/`[imo]` にせず `[ask]` に落とす（誤指摘でレビューの信頼を毀損しない）。

## 主観点 1: バグ（correctness）

- [ ] 境界条件: 空 / null / 0 / 負数 / 上限超過 / 空コレクションで壊れないか
- [ ] エラーハンドリング: 例外・エラー戻り値の握りつぶし、失敗パスでのリソースリークが無いか
- [ ] 状態と並行性: 競合状態、非冪等な再実行、トランザクション境界の破れが無いか
- [ ] ロジック: 条件式の反転・off-by-one・早期 return の抜け道が無いか
- [ ] 互換性: 呼び出し元が壊れる signature / スキーマ / API レスポンスの変更が無いか
- [ ] データ: マイグレーション・永続データの扱いが不可逆に壊れないか

## 主観点 2: 可読性（readability）

- [ ] 命名: 名前が実態（役割・単位・真偽の向き）を正しく表しているか
- [ ] 責務: 1 関数 / 1 クラスに複数の責務が混ざっていないか、深すぎるネストが無いか
- [ ] 重複: 既存のユーティリティ・共通処理の再実装になっていないか
- [ ] コメント: 「なぜ」が必要な箇所に説明があるか。コードと矛盾した古いコメントが無いか
- [ ] マジックナンバー / ハードコード: 意図が読める形（定数・設定）になっているか
- [ ] diff の意図: PR 説明と無関係な変更（無断リファクタ・整形差分）が混入していないか

## 主観点 3: 規約（convention）

- [ ] 対象リポジトリの規約文書（CLAUDE.md / CONTRIBUTING.md 等）に反していないか
- [ ] linter / formatter 設定（`.eslintrc*`, `ruff.toml`, `.editorconfig` 等）から読み取れるルールに沿っているか
- [ ] 周辺コードの既存スタイル（命名規則、ディレクトリ構成、エラー処理の流儀）と一貫しているか
- [ ] コミットメッセージ / PR タイトルがプロジェクトの形式（例: Conventional Commits）に沿っているか
- 規約の根拠が見つからない様式指摘はしない。根拠を引用できるものだけ指摘する。

## 補助観点 A: テスト（変更にロジックが含まれるとき）

- [ ] 変更されたロジックに対応するテストが追加・更新されているか
- [ ] テストが実装の写経（トートロジー）になっておらず、失敗しうる検証か
- [ ] 既存テストの削除・skip に正当な理由が説明されているか

## 補助観点 B: セキュリティ（入力処理・認証・依存関係に触れるとき）

- [ ] 機密情報（API キー・トークン・パスワード）が diff に混入していないか（発見時は値を伏せて `[must]`）
- [ ] 外部入力の未検証使用（SQL / コマンド / パスのインジェクション、XSS）が無いか
- [ ] 認可チェックの欠落（ID 直指定でのアクセス等）が無いか
- [ ] 依存追加: 出所不明・メンテ停止のパッケージを持ち込んでいないか

## 適用時の共通ルール

- 指摘は必ず `file:line` を伴い、可能なら具体的な修正案を 1 つ添える。
- 該当なしの観点は「指摘なし」と明示する（見ていないのか、見て問題なしなのかを区別する）。
- 既存レビューコメントで既出の論点は再指摘しない（必要なら「既出」と注記して同意のみ示す）。
- 良い点（きれいな分割、丁寧なテスト等）を最低 1 つ拾い、総評に含める。
