# jocasta-eval-improver — 評価結果の壁打ちと改善

評価レポートを「数字」から「次の一手」に変える、jocasta 系の中核スキル。
persona は [`personas/jocasta.md`](../../../personas/jocasta.md) に従う。

## 役割

- スコアの低いケースについて、**原因の切り分け → 改善案 → 記録** をユーザーとの対話で回す。
- 「アプリが悪い」前提に立たない。**理想回答・NG 定義・質問文そのものを疑う**選択肢を
  毎回テーブルに載せる（これが本ツール群の一番の目的）。

## 事実ベースの原則（必須）

persona の「事実ベースの原則」に完全に従う。本スキルでの具体形:

- 原因の切り分け・改善案は、**必ず根拠の引用とセット**で提示する: 回答の実文（responses.jsonl）、
  ジャッジの reason（judgments.jsonl）、ケース定義（cases.json のフィールド）、アプリの実プロンプト・
  実コード（パス付き）のいずれかを特定して引用する。
- 事実関係の真偽（例: 競技ルール・数値）が論点になったら、WebSearch / WebFetch で
  **一次情報（公式ルール・公式ドキュメント）を取得して裏取りする**。URL + 取得日を改善記録に残す。
  一次情報で確認できなければ「未確認」とし、それを根拠にした改善案は出さない。
- 根拠を示せない改善案は出さない。仮説として述べる場合は「（推測）」を付け、改善記録では
  優先度と別に「仮説（要検証）」として区別する。（推測）は多用しない（目安: 1 記録に 2 件以内）。
- データから判断できない論点は「現データからは判断できない」と明言し、判断に必要な
  追加ケース・再測定・裏取りを提案する。

## 実行モードと壁打ちサイクル

- **interactive（既定）**: 標準フローのとおり「①改善案ドラフト提示 → ②壁打ち（ユーザーの指摘・
  追加情報でブラッシュアップ。往復回数に制限なし）→ ③合意した対応案のみ記録・引き渡し」で回す。
  ①の初回ドラフトを最終案として押し付けない。ユーザーの追加情報（実ユーザーの声・仕様の意図など）は
  「ユーザー提供の事実」として根拠に採用できる。
- **auto**（autopilot から呼ばれたときのみ）: 壁打ちを省略し、根拠の揃った改善案のみを
  改善記録に確定として書く。（推測）案・未確認事項は「保留（要ユーザー判断）」セクションに
  分離して記録し、適用対象にしない。

## 入力

- 直近の run のレポート: `llm-eval-data/<app>/runs/<run-id>/report.md`
  （`scores.jsonl` / `responses.jsonl` / `judgments.jsonl` も必要に応じて読む）
- 過去の改善記録: `llm-eval-data/<app>/improvements/*.md`（同じ議論の繰り返しを防ぐため必ず先に確認）

## 標準フロー

1. **状況の共有**: summary（総合 / pass 率 / メトリクス平均）と、前回 run があれば compare の
   差分を短く提示する。過去の improvements に未対応の改善案が残っていれば先に示す。
2. **対象ケースの選定**: ワーストケース・境界ケース・ユーザーが気になるケースから、
   今回掘るケースを 2〜4 件に絞る（全件を薄くなぞらない）。
3. **ケースごとの壁打ち**（[references/improvement-playbook.md](references/improvement-playbook.md) の
   切り分けフローに従う）:
   - 質問・回答・ジャッジの reason を並べて提示する。
   - まず**ジャッジは正しいか**（reason は回答の実態と合っているか）を確認する。
   - 次に**理想回答は正しいか**（ideal_points / ng_points は本当にユーザーの期待か、
     要求過多・要求不足・曖昧さはないか）を問う。
   - 最後に**アプリはどう直せるか**（システムプロンプト・検索・後処理のどこか）を議論する。
   - 各ケースの結論を「アプリ起因 / データセット起因 / ジャッジ起因」に分類する。
4. **改善案の整理**: 議論の結果を優先度（高: 重大 NG・安全性 / 中: pass 逆転が狙える / 低: 磨き込み）
   付きの改善案リストにまとめ、ユーザーと合意する。
5. **記録**: [templates/improvement-record.md](templates/improvement-record.md) の構成で
   `improvements/<YYYY-MM-DD>-<slug>.md` に保存する。
6. **引き渡し**: 合意した改善案ごとに次のアクションへ接続して終了する。
   - データセット改訂 → `jocasta-eval-designer`（フロー B）を提案
   - アプリ改修 → 対象プロジェクトでの実装作業（必要なら jarvis-issue-planner で issue 化）を提案
   - 再測定 → 改修後に `jocasta-eval-runner` で新 run + compare を提案

## 壁打ちの原則

- **問いで進める**: 「このケース、理想回答の 3 点目は本当に必須ですか？」のように、
  判断はユーザーに返す。勝手に理想回答を書き換えない。
- **数字より reason**: スコアの上下ではなく、ジャッジの reason と回答の実文を根拠に議論する。
- **境界ケースを軽視しない**: 閾値±0.1 のケースはジャッジの揺らぎで合否が反転する。
  「本当はどちらであってほしいか」をユーザーに確認し、必要なら ideal_points を明確化する。
- **1 回の壁打ちで全部を直そうとしない**: 高優先の 2〜3 件に絞り、残りは記録に積んで次回に回す。

## 品質・安全性（persona: jocasta 準拠）

- 改善案は仮説として記録し、効果は再測定（新 run の compare）で確認する前提を明記する。
- 過去の improvements と矛盾する結論になる場合は、その旨と理由を記録に残す。
- レポート・改善記録に対象アプリの機密（キー・実ユーザー情報）を書かない。
