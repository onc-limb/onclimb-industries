# 改善壁打ちプレイブック

低スコアケースを議論するときの切り分けフローと、典型パターン別の改善案。

## 切り分けフロー（この順で疑う）

低スコアの原因は 3 系統。**必ず 1 → 2 → 3 の順で確認する**
（アプリを直す前に、ものさし側の妥当性を先に検証する）。

### 1. ジャッジ起因 — 採点は正しいか

- ジャッジの reason は回答の実文と一致しているか（読み違い・見落としはないか）。
- 同じ回答を再judge したらスコアが大きく動きそうか（境界ケースは特に）。
- **兆候**: reason が具体性を欠く / 回答に書いてあることを「無い」と言っている。
- **改善先**: ideal_points / ng_points の表現を判定しやすい粒度に直す（designer）、
  メトリクスの weights・threshold を見直す（config.json）。

### 2. データセット起因 — 理想回答は合っているか

- ideal_points は本当に「必須」か。あれば嬉しい程度の要点が混ざってスコアを下振れさせていないか。
- 逆に、ユーザーが実際には期待している要点が ideal_points に書かれていないことはないか。
- ng_points は過剰に広くないか（正当な回答まで NG に引っかけていないか）。
- 質問文自体が曖昧・不自然で、実ユーザーの聞き方と乖離していないか。
- **兆候**: 回答を読むと悪くないのにスコアが低い / ユーザーの感覚とスコアが逆転している。
- **改善先**: designer のフロー B（改訂理由を revision_notes に残す）。

### 3. アプリ起因 — どう直すか

ものさしが正しいと確認できてから、アプリ側を議論する。

| パターン（兆候） | 主に下がるメトリクス | 典型的な改善案 |
|---|---|---|
| 要点の抜け（説明が浅い・観点漏れ） | completeness | システムプロンプトに回答へ含めるべき観点リスト・構成を指示する |
| 誤情報・断定 | correctness | 不確実な事項は断定しない指示、根拠の確認を促す指示、（RAG なら）検索結果の充実 |
| NG 抵触（危険な推奨・スコープ外に回答） | ng_compliance | ガードレール指示の明文化（安全第一・範囲外は断る）、後処理フィルタ |
| 質問とずれる・冗長 | relevancy | 「まず質問に直接答える」「長さ上限」の指示、出力フォーマット指定 |
| context に無い創作 | faithfulness | 「与えられた情報のみで答える」「無ければ無いと言う」指示、検索品質の改善 |
| 曖昧な質問に即答してしまう | 複数 | 「情報不足なら聞き返す」指示、聞き返しの条件の明文化 |

## 改善案の優先度付け

- **高**: 安全性・信頼性に関わる NG 抵触（ng_points の「重大」）。pass/fail に関係なく最優先。
- **中**: 改善すれば pass に転じる見込みのあるケース（境界ケース含む）。
- **低**: すでに pass しているケースの磨き込み。着手は高・中が尽きてから。

## 改善ループの回し方

1 回の壁打ち → 改善記録 → 適用 → 新 run で compare → 次の壁打ち、を小さく回す。

- 一度に複数の変更を入れると、compare でどれが効いたのか分からなくなる。
  **アプリ改修とデータセット改訂を同じ run にまとめない**のが原則
  （データセットを変えたら、まず変更後のベースライン run を取り直す）。
- スコアが上がっても、変動したケースの reason を見て「意図した理由で上がったか」を確認する
  （ジャッジの揺らぎによる見かけの改善を成果と誤認しない）。
