# competitor-market-scan — 競合・市場スキャン

対象ドメイン（例:「オンライン議事録サービス」）を入力に、競合と市場トレンドを
**出典・鮮度付き**で 1 本の Markdown レポートにまとめる調査スキル。
案件参画前・提案準備・自社サービス企画の「周辺市場が見えない」を解消する。

> persona: [`personas/edith.md`](../../../personas/edith.md) — 出典明示・一次情報優先・
> 複数切り口での網羅・未カバー範囲の明示を原則とする。

## データ配置

- レポート: `<repo>/market-scan/<domain-slug>/<YYYY-MM-DD>.md`（ドメインごとにディレクトリ → 調査日ごとにファイル）
- 同じドメインを再調査したら新しい日付でファイルを増やす（過去レポートは差分確認の入力になる）

## トリガー

| ユーザー発話の例 | 動作 |
|---|---|
| 「〇〇の競合を調べて」 | ドメイン指定でフルスキャン |
| 「この案件の周辺市場をスキャンして」 | 会話文脈からドメインを確認してフルスキャン |
| 「〇〇の類似サービスと料金を比較して」 | 切り口を競合列挙 + 価格に絞る |
| 「最近の〇〇界隈のトレンドをまとめて」 | 切り口をトレンド + ニュースに絞る |
| 「前回の競合調査を最新化して」 | 過去レポートを読み、差分（新規参入・価格改定・撤退）を更新 |

明示的な依頼でのみ起動する。他スキルの作業中に市場の話題が出ても自動起動しない。

## 標準フロー

### 1. ヒアリング（探索前に確定する）

`AskUserQuestion` 等で以下を確定してから探索を始める。ドメインの解釈がずれたまま
探索すると無関係な競合が並ぶため、**ここを省略しない**。

- 対象ドメインの定義（何を「競合」とみなすかの包含基準。例: SaaS のみか、OSS も含むか）
- 調査目的（提案準備 / 技術選定 / 企画 など。要約の力点が変わる）
- 重視する切り口（省略時は全部）: 競合列挙 / 機能比較 / 価格 / トレンド / ニュース
- 地域（省略時: 日本。グローバルも見るか）
- 更新モードか（過去レポートがあれば Read し、差分に着目する）

### 2. 複数切り口での探索

WebSearch / WebFetch で、**切り口ごとにクエリを変えて**探索する。単一クエリで
打ち切らない。目安として各切り口 2〜3 クエリ、競合は 5 社以上の列挙を目指す。

| 切り口 | クエリ例 |
|---|---|
| 競合列挙 | 「<domain> 比較」「<domain> おすすめ」「<product> alternatives」 |
| 機能比較 | 「<competitor> 機能」各社公式サイトの機能ページ |
| 価格 | 「<competitor> 料金」**必ず公式料金ページを WebFetch で確認** |
| トレンド | 「<domain> 市場 動向 2026」「<domain> market trends」 |
| ニュース | 「<domain> 資金調達」「<competitor> 発表」直近のプレスリリース |

- 比較記事・まとめ記事は「候補の発見」に使い、**事実（機能・価格）は公式サイト
  （一次情報）で再確認**する。
- 探索中、各情報について「出典 URL」「取得日（今日）」「情報の日付（記事日付・
  発表日。判明する範囲で）」を必ず控える。

### 3. 検証

- **出典 URL の無い事実主張はレポートに書かない**（モデルの内部知識で表の空欄を
  埋めない）。載せる価値がある場合のみ「未確認（出典なし）」と明記して事実と区別する。
- 情報の日付が **12 か月超**、または日付不明のものに「古い可能性あり」「日付不明」を注記する。
- 価格は公式料金ページを出典とし、公式で確認できなければその旨を書く。
- 事実と解釈・推測を区別する（推測は「〜と推測される」と書き、根拠を添える）。

### 4. レポート生成

`templates/scan-report.md` の見出し構成で `market-scan/<domain-slug>/<YYYY-MM-DD>.md` に
Write する。見出しは固定（増減しない）:

1. **調査概要** — 対象ドメインの定義（包含基準）・目的・地域・取得日
2. **競合一覧表** — 名前 / 提供元 / 概要 / 主要機能 / 価格 / 出典 / 情報の日付・鮮度注記
3. **トレンド要約** — 市場動向・直近ニュース。各項目に出典番号を付す
4. **出典リスト** — 番号付きで URL・取得日・情報の日付
5. **カバーできなかった範囲** — 探索しなかった切り口・確認できなかった項目・
   打ち切り理由。**空にしない**（暗黙の打ち切りをしない）
6. 更新モード時は **前回からの差分**（新規参入・価格改定・撤退）を冒頭に付す

### 5. 完了提示

チャットにレポートパスと要点（競合数・主要トレンド・特に鮮度に注意が要る項目・
未カバー範囲）を 10 行以内で提示する。

## 品質・安全性

- 出典を明示し、一次情報（公式サイト・公式発表）を優先する。孫引きを事実として扱わない。
- 確証のない主張には不確実性を明記する。全項目に取得日・情報の日付を残し、追跡可能にする。
- 単一の探索角度に頼らず、切り口別・複数クエリで網羅性を確保する。
- カバーできなかった範囲を必ず明示する（網羅した風を装わない）。
- レポートは調査対象ごとに独立させ、他案件・他ドメインの調査で得た固有情報を持ち込まない。
- 機密情報（クライアントの非公開情報・API キー等）をクエリやレポートに含めない。
- 本スキルの出力は情報整理であり、事業判断そのものはユーザーが行う。
