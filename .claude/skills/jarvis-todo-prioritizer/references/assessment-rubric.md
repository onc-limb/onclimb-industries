# 影響度・緊急度の評価尺度（rubric）

評価のブレを抑えるための固定尺度。壁打ちでユーザーがこの尺度と違う感覚を示したら、
ユーザーの感覚を優先して値を確定し、rationale にその理由を残す。

## 影響度 impact（プロジェクトへの影響。1-5）

「やった場合に何が良くなるか」ではなく「**やらなかった場合に何が起きるか**」で測る。

| 値 | 定義 | 例 |
|---|---|---|
| 5 | プロジェクトの成否・収入・信用に直結。やらないと壊れる/失う | 本番障害の恒久対応、契約・請求に関わる作業、データ消失リスクの解消 |
| 4 | 主要な成果物・意思決定をブロックする。やらないと複数の作業が止まる | 依存タスクの前提整備、レビュー待ちの解消、必須データの移行 |
| 3 | 品質・効率に目に見える改善。やらないと非効率が続く | 自動化、頻用スキルの改善、たまり続ける手作業の解消 |
| 2 | あると良い改善。影響は局所的 | 表示の改善、まれにしか使わない機能の整備 |
| 1 | なくても困らない。思いつき・実験 | 試してみたい系、審美的な変更 |

## 緊急度 urgency（時間圧。1-5）

「いつまでにやらないと**価値が下がる/コストが上がる**か」で測る。

| 値 | 定義 | 例 |
|---|---|---|
| 5 | 期限超過または即時対応。今日〜数日 | due 超過、進行中の障害、締切直前の提出物 |
| 4 | 明確な期限が 1〜2 週間以内。逃すと機会損失やペナルティ | 申込締切、税・制度の期日、キャンペーン期限 |
| 3 | 期限は 1〜2 か月以内、または遅らせるほど作業コストが膨らむ | データがたまり続ける移行作業、EOL 対応の準備 |
| 2 | 期限なし。ただし放置で緩やかに劣化する | 古くなっていくドキュメント、腐りかけのナレッジ |
| 1 | 期限なし・劣化もしない。いつでもよい | 純粋な改善アイデア |

- due が設定されているタスクは、due までの残り日数を urgency の下限の根拠にする
  （due 3 日前で urgency 2 は矛盾。矛盾を見つけたら壁打ちで確認）。

## 象限 → 着手順の目安（提案用。決定はユーザー）

スコア（impact × urgency）は並べ替え用。着手順の**提案**は象限で言い換える:

| 象限 | 目安 |
|---|---|
| impact ≥ 4 かつ urgency ≥ 4 | 今すぐやる（他を止めてでも） |
| impact ≥ 4 かつ urgency ≤ 3 | 計画に組み込む（時間を確保して腰を据えて。ここが痩せると後で 5×5 になる） |
| impact ≤ 3 かつ urgency ≥ 4 | さっと片付けるか、縮小・簡略化できないか先に検討する |
| impact ≤ 3 かつ urgency ≤ 3 | まとめて隙間時間に。impact 1 かつ urgency 1 が続くタスクは drop 候補として提案 |

## 外部調査（世間の情報）をするかの判断

WebSearch は**外部要因が評価を左右するタスクだけ**。目安:

- **調べる**: 期限が外部で決まるもの（税・制度・申込・キャンペーン）、技術の EOL・非推奨化・
  セキュリティ勧告が絡むもの、価格・料金改定の影響を受けるもの、外部イベント（発表会・締切）連動。
- **調べない**: 純内部のリファクタ・整理・個人の習慣づけなど、外部情報で緊急度が変わらないもの。
- 調べた情報は 出典 URL + 取得日 + 情報自体の日付 をセットで扱う。12 か月超は「古い可能性あり」を注記。
- 検索で確証が得られなかったら「不明（外部確認できず）」として壁打ちに回す。それらしい根拠を作らない。

## 壁打ちの進め方

- 仮評価は 2〜3 件ずつ。1 件ごとに「事実 → 解釈 → 提案値」の順で短く示す。
  例: 「due が 7/10（事実）。逃すと来月まで待ち（解釈）。urgency 4 を提案します」
- ユーザーが違う値を出したら理由を一言聞いて、そのまま確定する（説得しない。理由は rationale へ）。
- 迷いが残るものは記録せず保留にする。「とりあえず 3」で埋めない。

## 見直し（stale）基準

- `assessed_at` から 30 日超。
- due の変更、親/依存タスクの完了、参照した外部情報の期限到来・変化。
- 見直しは差分だけ: 「前回は◯◯という理由で I4/U2。△△が変わったが、どうしますか」
