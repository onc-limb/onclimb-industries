# db-migration-safety-checker — DB マイグレーションの適用時安全性チェック (arc-reactor)

マイグレーションファイルを本番適用する**前**に、ロック・ダウンタイム・データ破壊・
新旧コード非互換を引き起こす危険操作を静的に検出し、安全な代替手順を提示するスキル。
ロック事故は一度起きるとサービス停止に直結する。事故を 1 回防げば元が取れる守りのスキルであり、
「問題なし」と言うときも何を確認したかを示す。

> 大原則: **エンジンとバージョンで危険度が変わるものは、必ず条件を付けて報告する。**
> 「PG11+ なら NOT NULL + DEFAULT の同時追加は安全」のように、無条件の Critical と
> 条件付きの Warning を混ぜない。テーブル規模が不明なら「行数次第」と明記し、
> 確認用の行数見積もりクエリを添える。

persona: [`personas/arc-reactor.md`](../../../personas/arc-reactor.md) に従う
（応答は日本語 / コード・識別子は英語 / 推測は ASSUMPTION 明記）。

## トリガー

| ユーザー発話の例 | 動作 |
|---|---|
| 「このマイグレーション、本番に流して大丈夫か見て」 | 指定ファイルを全観点でチェック |
| 「migration の安全性をチェックして」 | 対象を確認してチェック |
| 「この ALTER TABLE、ロック掛かる？」 | 単文 SQL をカタログ照合して判定 |
| 「このブランチの新規マイグレーションをレビューして」 | base ブランチとの差分から新規マイグレーションを抽出してチェック |
| 「ゼロダウンタイムで適用できるか確認して」 | ロック・デプロイ順序の観点を重点チェック |
| 「migration safety check して」 | 対象を確認してチェック |

ユーザーが明示的に依頼したときだけ起動する（マイグレーションファイルを読んだついでに自動起動しない）。

## 標準フロー

### 1. 入力の確定

- **対象マイグレーション**を確定する。指定方法は 3 通り:
  - ファイル / ディレクトリの直接指定（ORM マイグレーション or 生 SQL）
  - 差分ブランチ指定: `git diff <base>...HEAD --name-only` でマイグレーションディレクトリ配下の
    新規ファイルだけを対象にする
  - チャットに貼られた SQL 断片
- **DB エンジンとバージョン**を確定する。リポジトリ内の設定
  （`docker-compose.yml`, CI 設定, `database.yml`, `.env.example` 等）から読み取り、
  確定できなければユーザーに聞く。**バージョン不明のまま判定しない**
  （バージョンで危険度が反転する項目があるため）。どうしても不明なら
  「最も保守的なバージョン前提で判定した」と明記する。
- **対象テーブルの規模感**を聞く（行数のオーダーで十分）。不明なら不明のまま進め、
  該当項目に「行数次第」+ 見積もりクエリを付ける。

### 2. 実 SQL の確定（推測で断定しない）

生 SQL ならそのまま次へ。ORM マイグレーションで生成される SQL が自明でない場合
（複合的な change メソッド、ORM 独自の抽象化、コールバック付き等）は、
実 SQL を出力する機能でダンプしてから判定する。例:

- Django: `python manage.py sqlmigrate <app> <番号>`
- Alembic: `alembic upgrade <rev> --sql`（オフラインモード）
- Laravel: `php artisan migrate --pretend`
- Prisma: `prisma migrate diff --script`
- Rails: 生成 SQL のダンプ機能が無いため、メソッドとカタログの対応で判定し、
  自明でないものは `ASSUMPTION` を明記する

ダンプ実行にユーザーの環境操作が要る場合はコマンドを提示して依頼する。
実 SQL が確認できないまま判定した項目には、その旨を必ず書く。

### 3. カタログ照合

[references/dangerous-operations.md](references/dangerous-operations.md) を**必ず先に読み**、
対象のステートメント 1 本ずつをカタログと照合する。カタログ外の操作でも、
テーブル全体の書き換え・長時間ロック・ロックキュー詰まりの兆候があれば独自に指摘してよい
（その場合は判定根拠を書く）。

照合時の観点はスキーマ操作だけではない:

- **デプロイ順序**: マイグレーションと新コードのどちらが先に出るかで壊れないか
  （rename / drop / NOT NULL 追加はここで事故る。カタログの expand-contract 節を参照）
- **ロールバック可能性**: down マイグレーションが定義されているか、down がデータを失わないか
- **トランザクション制約**: `CREATE INDEX CONCURRENTLY` 等、トランザクション内で
  実行できない操作がマイグレーションツールの暗黙トランザクションと衝突しないか

### 4. 危険度判定

各指摘に危険度を付ける:

- **Critical**: 本番で障害級。該当条件下でロック長期化・データ破壊・新旧コード非互換が確実に起きる
- **Warning**: 条件次第で問題（テーブル規模・バージョン・デプロイ手順に依存）。条件を必ず併記する
- **Info**: 留意点（実害は小さいが知っておくべき挙動、ベターな書き方がある等）

判定ルール（**厳守**）:

- エンジン・バージョンで危険度が変わるものは条件を付ける。
  条件付きのものを無条件に Critical と書かない。
- テーブル規模に依存するものは、規模が既知ならその前提で確定し、
  不明なら「行数次第」+ 行数見積もりクエリ（カタログ末尾）を添える。
- 迷ったら Critical 側に倒す（ロック事故は非対称に痛い）。ただしその場合も
  「保守側に倒した判定」であることと、Critical を外せる確認方法を書く。
- 全指摘に該当箇所 `file:line` を付ける。付けられない指摘は出さない。

### 5. レポート生成

チャットに以下の固定構成で提示する。ファイル保存を求められた場合は対象プロジェクト配下
`<repo>/migration-safety/<YYYY-MM-DD>-<対象名>.md` に置く（対象リポジトリを汚す可能性が
あるため、**保存前に必ずユーザーに確認**する）。

```markdown
# Migration Safety Report — <対象> (<YYYY-MM-DD>)

## 前提
- エンジン / バージョン: <例: PostgreSQL 14>（不明時はその旨と判定前提）
- 対象ファイル: <一覧>
- テーブル規模: <既知の値 / 不明（該当項目に見積もりクエリ添付）>
- 実 SQL の確認方法: <ダンプ済み / メソッド対応で判定（ASSUMPTION あり）>

## 指摘事項

### [Critical] <操作の要約> — <file:line>
- 何が起きるか: <ロックの種類・持続時間・レプリケーション影響>
- 危険になる条件: <テーブル規模・バージョン・デプロイ手順>
- 安全な代替手順: <具体的な SQL / マイグレーション分割案>

### [Warning] ... / ### [Info] ...

## 確認した観点（問題なし）
- <観点>: <対象ステートメント> — 問題なしの理由（例: PG14 のため ADD COLUMN + DEFAULT はメタデータ変更のみ）

## 未確認・免責
- 実 SQL 未確認の項目 / 規模不明のまま判定した項目
- 本チェックは静的解析であり、実環境のロック競合・同時実行トラフィックまでは検証していない
```

危険が 1 件も無い場合も「確認した観点」を必ず出す。
**見ていないのか、見て問題なかったのかを読み手が区別できること**がこのレポートの価値。

### 6. 完了提示

- Critical / Warning の件数と、最優先で直すべき項目をチャットで要約する。
- 次アクション（マイグレーション分割の実装、規模確認クエリの実行、
  デプロイ手順書への反映など）を提案する。修正の実装はユーザーの指示を受けて別途行う。

## 危険操作カタログ

エンジン別（PostgreSQL / MySQL(InnoDB) 主、SQLite 補足）の危険操作・ロック挙動・
安全な代替手順は [references/dangerous-operations.md](references/dangerous-operations.md) を参照する。
**照合前に必ず読むこと。** カタログの改善（新しい危険パターン・バージョン差の追記）は
references への追記で行う。

## 品質・安全性

- **DB への接続は行わない**。静的解析のみ。行数確認・ロック状況確認のクエリを提案することは
  あるが、実行はユーザーの判断に委ねる。
- 対象リポジトリへの操作は**読み取りのみ**（実 SQL ダンプの提案は行うが、
  マイグレーションの修正・適用は本スキルの範囲外）。レポート保存時のみ書き込み、保存前に確認する。
- 誤検知を恐れず Critical は保守側に倒す。ただし、条件付きのものを無条件の Critical と
  書かない（条件・バージョン・規模の但し書きが本スキルの信頼の条件）。
- 接続文字列・パスワード等をマイグレーションや設定内で見つけても、レポートに**値を転記しない**。
- レポートはプロジェクト固有情報のため、保存する場合は対象プロジェクトのディレクトリ配下に
  閉じる（CLAUDE.md の分離ルールに従う）。
- クエリの実行性能（インデックス設計・N+1 等）の指摘に踏み込まない。
  それは `arc-reactor-slow-query-hunter` の領分で、本スキルは「適用した瞬間に何が起きるか」に特化する。
