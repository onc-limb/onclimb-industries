# jarvis-record — 依頼者向け報告の一次記録 (Stage 1)

worklog の「プロジェクト整理情報(project digest)」を、**案件×対応日の作業記録(Markdown)**へ
落とし込むスキル。後段の [friday-daily-report](../friday-daily-report/) がこの記録を入力に、非エンジニア向けの
日次報告スライドへ清書する。本スキルは**事実の正確な一次記録**を担い、用語の言い換えはしない。

> 位置づけ: この工程は digest の「転記」ではなく、**複数ソースのマージ点**。
> ログ(digest)は入力の一つにすぎず、「ログに写っていない情報(背景・手動操作・どれが重要か)」を
> 対話で補完・確定して初めて記録が完成する。ここを素通りすると下流の報告が
> 「全部を平らに並べただけ」になる(2026-07-03 改善案の主眼)。

## データ配置

- 記録: `<repo>/report-record/<project>/<YYYY-MM-DD>.md`(案件ごとにディレクトリ → 日付ごとにファイル)
- 草案: `<repo>/report-record/<project>/<YYYY-MM-DD>.DRAFT.md`(確認サイクル未完了。Stage2 の入力にしない)
- 入力の worklog digest: `<repo>/worklog-data/digests/project/<project>_<date>.md`
- 入力の手動メモ: `<repo>/worklog-data/manual/<date>.md`(あれば。jarvis-worklog の「手動メモ」参照)
- 上書き: `REPORT_RECORD_DIR` / `WORKLOG_DATA` 環境変数

## トリガー

| ユーザー発話の例 | 動作 |
|---|---|
| 「今日の作業を案件ごとに記録して」 | 当日の全案件を記録 |
| 「worklog の整理を報告用に落とし込んで」 | digest を記録へ変換 |
| 「<案件>の今日の分を記録して」 | 案件を限定して記録 |
| 「6/25 の分を記録して」 | 日付を限定 |
| 「6/23〜6/27 を記録して」 | 日付ごとに `locate.py` → 生成をループ(1日1ファイルを維持) |
| 「DRAFT を確定して」「昨日の草案の続き」 | DRAFT の確認サイクルを再開して確定 |

## 標準フロー

```bash
SKILL=/Users/satoshi-onga/Documents/onclimb-industries/.claude/skills/jarvis-record
TODAY=$(date +%F)
python3 "$SKILL/bin/locate.py" "$TODAY"              # 入力(digest/前日記録/手動メモ/出力先/DRAFT)を JSON で取得
python3 "$SKILL/bin/extract_gaps.py" <digest path>   # 欠損箇所・band 構成を機械抽出(質問候補の原料)
```

1. **入力をロケート**: `locate.py <date> [project]` を実行。対象案件ごとに、worklog digest・
   前日までの記録・出力先パス・`is_unclassified` / `record_exists` / `draft_exists` フラグ、
   および手動メモ(`manual_notes`)のパスが返る。
   - `draft_exists: true` なら前回の確認サイクルが未完了。DRAFT を Read し、手順4の続きから再開する。
2. **入力を読む**: 各案件について digest(主入力)・`prev_record`(あれば前日の記録)・
   `manual_notes`(あれば手動メモ)を Read。前日の「この先やること」と当日 digest の作業内容を
   突き合わせ、継続タスクの文脈をつなぐ。
3. **下書き生成(草案)**: `templates/record.md` の見出し構成で草案を組み立てる。
   - digest から埋まらない・確定できない箇所は空欄にせず `[要確認]` を付けて明示する。
   - 「やったこと」各項目の重要度 [主]/[従] はこの時点では**仮置き**(digest の成果サマリ由来)とし、
     確認サイクルで確定するまで記録を確定しない。
4. **確認サイクル(必須 — 省略しない)**: 下書きを要約提示し、次の確認リストを対話で 1 つずつ潰す。
   質問は `AskUserQuestion` で集約する(1回最大4問。多い日は優先度順に原則2ラウンド以内、
   残りは低優先として [要確認] のまま残すか続行可否を確認)。
   - **(a) ログ欠損の補完**: `extract_gaps.py` の `gaps`(「記録なし/未確認」箇所)から、報告に
     影響する順に質問を組む。「ここはログに無い。背景/結果を教えてほしい」。型タグ
     (`セッション中断` / `ユーザー手動・ログ外` / `背景の記載なし`)があれば聞き方を変える
     (手動系は「何をどこまでやったか」、背景系は「なぜこの作業をしたか」)。
   - **(b) band 間矛盾の解消**: digest が時間帯分割されている場合(`bands` が複数)、digest 冒頭の
     「時間帯間の重複・矛盾」セクション(worklog が自動生成。無ければ band 見出し単位で同一対象の
     記述を突き合わせる)から食い違いを拾い、「band4 は未対応・band5 は削除済み。どちらが正しい?」と
     確認する。確定結果は meta の `resolved` に残す。
   - **(c) 重要度トリアージ**: 「やったこと」の各項目に [主]/[従] を付け、**1日の主成果を
     1〜3 件に絞る合意**を取る。機械が勝手に決めない(主/従の線引きはユーザー判断)。
   - **(d) 手動操作の確定**: 「これはあなたが手でやった / Claude がやった」を確定し、
     手動分は「手動で実施した操作」欄と「やったこと」(末尾に `(手動)`)へ記す。
   - 従来のヒアリング項目も併せて確認する: `_unclassified` を報告に含めるか / 依頼者からの
     口頭フィードバック / 「判断待ち: なし」は本当か / 不採用案の理由が薄いとき。
   - **digest・回答に無い事実を創作しない。** 埋まらない欄は「なし」/「記録なし」と書く。
5. **無応答時の分岐(重要)**: 確認サイクルに応答が無い(離席・タイムアウト)場合、**自動生成に
   倒さない**。草案を `<date>.DRAFT.md` に保存(H1 末尾に `【DRAFT】`、meta `status: draft`)して
   停止し、「確認サイクルが未完了。次回『DRAFT を確定して』で再開できる」と提示する。
   **沈黙を確定と解釈しない。** DRAFT のまま friday-daily-report へ渡さない。
6. **確定 → 記録書き出し**: 対話の結果を反映し、`record_out` のパスに Write する
   (DRAFT があれば反映後に削除)。
   - `record_exists: true` の案件は既存記録を Read し、meta の `heard` を引き継いで更新する。
     引き継げるか判断がつかない場合は `AskUserQuestion` で上書き可否を確認する。
   - 見出しは固定(増減しない): 取り組んだ課題（なぜこの作業をしたか）/ やったこと /
     今どうなっているか（結果・現状）/ 検討したが採用しなかった案（と見送った理由）/
     障害・つまづき / ログに無い補足（背景・口頭情報）/ 手動で実施した操作 /
     この先やること / 依頼者への相談・判断待ち
   - 「やったこと」の各項目に `[主]`/`[従]` + `[完了]`/`[進行中]`/`[着手のみ]`/`[中断]` を付ける。
     `[中断]` は digest からは来ない。放棄・保留の疑いはヒアリングで確定してから付ける。
   - 末尾 meta コメントに `status`(confirmed/draft)・`source`・`prev`・`manual`・
     `heard`(ヒアリングで補った項目)・`resolved`(矛盾の確定内容)を残す
   - **この段階では専門用語を残してよい**(清書は friday-daily-report の責務)
   - **原料保全**(personas/jarvis.md): 「検討したが採用しなかった案」「障害・つまづき」は
     下流の報告・技術記事の品質を決める原料。digest に痕跡があるのに空欄になりそうなら
     ヒアリングで拾ってから「なし」と確定する。エラーメッセージ・数値は要約で潰さず原文のまま残す。
7. **完了提示**: 生成パスと、friday-daily-report へ渡せる状態か(重要度・手動・欠損が埋まったか、
   障害・相談事項の有無)を要約して提示し、清書に進んでよいか確認する。

## テンプレート

`templates/record.md` を見出しの正とする。見出しの順番・文言は変更しない(friday-daily-report が
この構造に依存する)。digest の各セクションとの対応:

| 記録の見出し | 出どころ |
|---|---|
| 取り組んだ課題 | digest「目的・背景」+ 前日記録の「この先やること」+ ヒアリング(背景系欠損の補完) |
| やったこと | digest「作業内容(進捗ステータス付き)」+ 重要度トリアージ(対話) |
| 今どうなっているか | digest「最終的にどうしたか」 |
| 検討したが採用しなかった案 | 作業内容内の不採用案 / 判断根拠 |
| 障害・つまづき | digest「つまづき・問題(Problem)」 |
| ログに無い補足（背景・口頭情報） | **ヒアリングのみ**(digest の欠損箇所への回答・口頭情報) |
| 手動で実施した操作 | 手動メモ(worklog-data/manual/) + ヒアリング |
| この先やること | digest「次の予定・次への改善(Next/Try)」 |
| 依頼者への相談・判断待ち | digest「課題・判断待ち」 |

## 注意

- worklog 未実行で当日 digest が無ければ、先に worklog の「まとめて」を促す。
- 機密はマスキング済み digest を前提とする。記録に実名・顧客名を新たに書き起こさない。
- 妥協ライン: 確認サイクルのうち **(b) 矛盾の解消と (c) 重要度トリアージは必ず確定させる**。
  (a) 欠損補完・(d) 手動確定は件数が多ければ報告に影響する分だけに絞ってよい(残りは [要確認])。
- 学習: ヒアリングで繰り返し問う項目が出てきたら、本ファイルの確認サイクルの項目に追記して育てる。
