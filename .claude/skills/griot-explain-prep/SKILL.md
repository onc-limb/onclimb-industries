# griot-explain-prep — 説明練習の準備（言語化ノート + 説明資料）

毎日の報告・説明練習の **Step 1（準備）**。壁打ちで学んだことを言語化してノートにまとめ、
説明時に見る 1 枚 HTML 資料を自動生成する。persona は [`personas/griot.md`](../../../personas/griot.md)。

## パイプライン全体像

```
Step 1: griot-explain-prep    ── 壁打ちで言語化 → note.md + deck.html
Step 2: (ユーザーが deck を見ながら口頭で説明し、録音・文字起こしする)
Step 3: griot-explain-coach   ── 文字起こしをペルソナ視点で添削 → review.md + 指摘台帳
Step 4: griot-explain-english ── 英語資料 + 発話スクリプト(英語練習用)
```

## データ配置

- 出力: `<repo>/explain-practice-data/sessions/<YYYY-MM-DD>-<slug>/`（git 管理外）
  - `note.md` — 言語化ノート（テンプレ: [`templates/note.md`](templates/note.md)）
  - `deck.html` — 説明資料（テンプレ: [`templates/deck.html`](templates/deck.html)）
- slug は英語 kebab-case（例: `2026-07-07-go-context`）。同日に複数テーマ可（テーマごとに 1 セッション）。

## トリガー

| ユーザー発話の例 | 動作 |
|---|---|
| 「今日の説明練習の準備」「学んだことを言語化したい」 | フロー A（壁打ちから開始） |
| 「〜について学んだのでノートにまとめたい」 | フロー A（テーマ確定済みで開始） |
| 「このノートから資料だけ作り直して」 | フロー B（資料再生成） |

## フロー A: 壁打ち → ノート → 資料

### 1. テーマと想定聞き手の確定

- **テーマ**: 今日説明する内容を一言で（1 セッション 1 テーマ。複数あればセッションを分ける）。
- **想定聞き手（ペルソナ）**: 誰に説明するつもりか＋その人の前提知識
  （例: 非エンジニアの上司 / 隣のチームのエンジニア / 初学者の後輩）。
  ここで確定したペルソナは note.md の `audience` に記録し、Step 3 の添削の判定基準になる。

### 2. 壁打ち（★このスキルの核）

ユーザーに学んだことを自由に話してもらい、テンプレートの必須セクションに対する
**充足検査**を行う。守ること:

- **Claude が内容を埋めない・代筆しない**。言語化はユーザーの練習そのもの。
  不足・曖昧を指摘して質問し、ユーザーの言葉で埋めてもらう。
  ただし技術的に誤った理解が含まれる場合は、その場で率直に指摘してよい
  （誤った説明を練習させない）。
- 充足検査の観点（[`templates/note.md`](templates/note.md) のセクションに対応）:
  - **結論**: 「一言で言うと」が 1〜2 文で言い切れているか。
  - **なぜ・何を・どうやって**: 3 観点が揃っているか（なぜ = 必要な理由・解決する問題 /
    何を = 仕組み・概念 / どうやって = 使い方・手順）。
  - **具体例**: 実際に動かしたこと・コード・数字があるか。
  - **聞き手適合**: ペルソナの前提知識で通じない用語・飛躍がないか。
- **曖昧語を検出したら具体化を促す**: 「いい感じに」「うまく動く」「〜など」「基本的に」
  「ちゃんと」等が出たら、「具体的には？」と掘る。
- 質問は 1 回の応答で **3 つまで**（尋問にしない）。分からないことは「まだ曖昧な点」
  セクションに落として先に進んでよい（完璧を待たない — persona 原則）。

### 3. ノート生成

- 情報が揃ったら `templates/note.md` の書式で `note.md` を書き出す。
  見出しの削除・順序変更はしない（埋まらないセクションは「なし」または TBD）。
- **ユーザーの表現・言い回しをできるだけ活かす**（Claude の文体に清書しすぎない。
  ユーザーが言っていないことを足さない）。
- ユーザーに提示して確認。修正があれば反映し、確定で frontmatter `status: fixed` にする。

### 4. 説明資料の自動生成

- 確定した note.md から `templates/deck.html` に従って `deck.html` を生成する（自動。壁打ち不要）。
- **資料は「見ながら話す」ための手がかり**であって読み上げ原稿ではない:
  - 全文を載せない。キーワード・短い箇条書き（1 項目 20 字以内目安）。
  - 話す中身はユーザーの頭とノートにある。資料が説明を代弁したら練習にならない。
- 単一の自己完結 HTML（外部 CDN・フレームワーク・Web フォント依存なし）。
  デザインはテンプレート固定（寒色系ミニマル）。崩さない。

### 5. 完了提示

- 生成パス 2 件を提示し、次のステップを一言案内:
  「deck.html を見ながら説明を録音 → 文字起こしを griot-explain-coach に渡すと添削します」。

## フロー B: 資料再生成

既存セッションの `note.md` を Read し、手順 4 だけを実行して `deck.html` を上書き生成する
（ノートをユーザーが手で直した後の作り直し用）。

## 注意

- ノートは Step 3・4 の入力になる。`audience`（ペルソナ）と「まだ曖昧な点」は
  空にしない（曖昧な点が本当に無い日は「なし」と明記）。
- 過去セッションの note.md は書き換えない（訂正・上達は新しいセッションで示す — persona 原則）。
- 練習題材に業務情報が含まれてよい（データは git 管理外）。ただし出力・資料に
  API キー等の機密情報を含めない。
- 応答・ノート・資料は日本語。コード・識別子・技術用語は原語のまま表記する。
