# modeling-domain-challenge — 今月のモデリングドメイン課題

月 1 本、大きめのサービスを題材にしたモデリング課題を生成するスキル。
persona: [`personas/griot.md`](../../../personas/griot.md)（本人の力を引き出す・完成品を代わりに作らない）。

## ゴール

**このスキルのゴールは、ユーザーが 1 か月かけてイベントストーミング〜モデリングに取り組める
「具体的で緊張関係のある題材」を毎月 1 本供給すること。**

- モデリングの正解を教えることではない。**課題に正解は無い**。出すのは仕様・ユーザー・制約だけ。
- 蓄積されたノート（[jarvis-modeling-notes](../jarvis-modeling-notes/SKILL.md) の
  `modeling-notes/notes/`）を読み、**過去の気づきの類似場面・教訓同士が矛盾しそうな場面**を
  題材に仕込むことで、トレーニングを「過去の自分の結論との対話」にする。

## 場所

- 出力: リポジトリ直下 `modeling-notes/challenges/monthly/<YYYY-MM>-<slug>.md`
  （slug はサービス形態からの英小文字ケバブケース。例: `2026-08-equipment-rental`）。
- 入力: `modeling-notes/notes/*.md`（気づき・教訓・未決の論点）、
  `modeling-notes/challenges/monthly/*.md`（過去の出題。サービス形態の重複回避）、
  `modeling-notes/challenges/daily/*.md`（あれば。小課題で触れた論点の把握）。
- `modeling-notes/` は git 管理外（`.gitignore` 済み）。索引ファイルは持たない（都度走査）。
- **取り組みの記録（session-log）**: ユーザーが課題に取り組んだやり取りの記録は、課題ファイルの
  隣に `<slug>-session.md`（frontmatter に `type: session-log`）で置いてよい。スキルはこれを
  **出題として扱わない**（当月ガード・過去出題の走査から除外する）。気づき・教訓の正本への
  反映は jarvis-modeling-notes 経由で行う。

## トリガー

| 発話の例 | 動作 |
|---|---|
| 「今月のモデリング課題出して」「今月の題材ちょうだい」 | 当月分が無ければ生成。あれば再掲（作り直さない） |
| 「今月の課題作り直して」「別のサービス形態で」 | 当月分を作り直す（既存ファイルは `-v2` 等で残し、上書きしない） |
| 「来月の課題を先に作って」 | 対象月を指定して生成 |
| 「過去の課題一覧」 | `challenges/monthly/` を走査して月・サービス形態・狙いを表で出す |

自動起動しない。毎月 1 日の自動実行を仕込む場合はスケジュール機構側で
「今月のモデリング課題出して」を発火させる（本スキルは冪等: 当月分があれば再掲するだけ）。

## 生成フロー

1. **当月ガード** — `challenges/monthly/<YYYY-MM>-*.md`（`type: session-log` は除く）が
   既にあれば、その概要を再掲して終了
   （「作り直して」と言われたときだけ再生成。既存ファイルは残す）。
2. **ノートの読み込み** — `modeling-notes/notes/*.md` を走査し、次を抽出する:
   - 教訓（「教訓」セクション）と、その適用条件が未検証のもの
   - 未決の論点・不明点（壁打ちエントリの「未決」、疑問形の気になり）
   - 直近 2〜3 か月でよく触れているテーマ（責務配置・依存の向き 等）
   ノートが空・薄い場合はこのステップを飛ばし、一般的な良題材として成立する課題を出す。
3. **サービス形態の選定** — 世にある一般的なサービス形態から 1 つ選ぶ
   （例: EC / 宿泊・座席予約 / サブスク / マッチング / 物流・配送 / レンタル / チケット販売 /
   フリマ / 教育・受講管理 / 会員制ジム / 医療予約 / 中古買取）。選定基準:
   - **ノートの論点と接触面が大きいこと**（類似場面・矛盾場面を自然に仕込める）
   - 過去の月次課題とサービス形態が重複しないこと（`challenges/monthly/` を走査）
   - 実在の特定企業のサービスを写さない（一般的な形態として定義する）
4. **課題の生成** — 次の構成で 1 ファイルにまとめる:
   ```markdown
   ---
   month: YYYY-MM
   service_form: <サービス形態（英語 kebab-case）>
   seeds: [<狙いに使ったノートの slug>, ...]  # ノート未参照なら空
   ---

   # <YYYY-MM> モデリングドメイン課題: <サービス名（架空）>

   ## サービス概要
   <何を提供するサービスか。3〜8 行>

   ## 想定ユーザー
   <利用者の種類と、それぞれの目的・行動。管理側・運営側も含める>

   ## 主要ユースケース
   <番号付きで 5〜10 個。イベントストーミングの入口になる粒度>

   ## 業務ルール・制約
   <料金・期限・キャンセル・在庫・権限などの具体ルール。数値を入れる。
    あえて緊張関係（素直にモデリングすると衝突する要求）を 2〜3 個含める>

   ## スコープ外
   <今回考えなくてよいこと（認証基盤・決済代行の内部 等）。課題の際限ない拡大を防ぐ>

   ## 考えどころ
   <open な問いのみ 3〜7 個。「〜はどう表現するか」「〜の責務はどこか」。
    答え・誘導・推奨は書かない>

   ---

   ## 出題メモ（解いた後に読む）
   <どのノートのどの気づき・教訓の「類似」や「矛盾」を狙ったか。[[note-slug]] で参照。
    ノート未参照の出題ならその旨>
   ```
5. **通知** — 保存パスと「サービス概要・考えどころ」を提示する。出題メモの中身は
   本文で繰り返さない（先入観を避けるため「末尾にあります」とだけ伝える）。
   気づき・不明点は jarvis-modeling-notes で随時記録するよう一言添える。

## 出題の質（類似と矛盾の仕込み方）

- **類似**: ノートの教訓が別ドメインでも成り立つかを試す場面を作る
  （例: 「区分オブジェクトが業務基準値を所有する」という教訓があるなら、
  区分が動的に増減する・基準が地域や時期で変わる、など前提を 1 つずらした場面を置く）。
- **矛盾**: 教訓同士・教訓と新しい制約が衝突する場面を作る
  （例: 「依存は特定→汎用の一方向」と「変更頻度の高い側に知識を寄せる」が
  素直に両立しない業務ルールを置く）。
- 仕込みは業務ルール・制約の中に**自然に**埋める。「ここが引っかけです」と本文で示さない
  （狙いは出題メモにだけ書く）。
- ノートの記述を課題本文にコピーしない（ユーザーの言葉は記録側に属する。課題は新しい場面）。

## 品質・安全性（persona: griot 準拠）

- **正解・模範解答・推奨設計・クラス図・コード例を出さない**。設計の相談を受けたら
  jarvis-modeling-notes の壁打ち（モード B）へ誘導する（そこでも結論は押し付けない）。
- 実在の特定サービス・企業の仕様を写さない。案件・顧客の情報を題材に混ぜない。
- ノート（`notes/`）は読み取り専用。書き込みは `challenges/monthly/` 配下のみ。
- 過去の課題ファイルは書き換えない（作り直しは別ファイルで残す）。
- 出力は日本語。識別子・slug は英語。

## 自己進化

固定する芯（正解を出さない・題材提供に徹する・月 1 本・ノート由来の類似/矛盾の仕込み）は崩さない。
育てるのは:

- サービス形態カタログ: 出題が偏る・ネタ切れになったら、本ファイルの形態リストに追記する。
- 課題の分量・具体度: 「1 か月では大きすぎた / 物足りない」という指摘があれば、
  ユースケース数・制約数の目安を調整する。
- 緊張関係のパターン: 効いた仕込み・空振りだった仕込みの傾向を「出題の質」節に追記する。

## やってはいけないこと

- 正解・模範解答・ヒントの誘導を出さない（「考えどころ」は open な問いに留める）。
- ノートに書かれたユーザーの気づき・教訓を課題本文に転載しない（狙いは出題メモで参照のみ）。
- 課題の消化・進捗を催促しない（未着手の月があっても指摘しない）。
- 当月分があるのに黙って作り直さない・既存課題ファイルを上書きしない。
- `notes/` への書き込み・`projects/` 配下への保存をしない。
