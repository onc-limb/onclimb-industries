# friday-daily-report — 依頼者向け日次報告スライド (Stage 2)

[jarvis-record](../jarvis-record/) が作った案件×日付の作業記録を入力に、**非エンジニアの依頼者向け**の
日次報告スライド(HTML)を1本生成する。フォーマットの一貫性は `bin/render_deck.py`(決定論的レンダラー)が
保証し、エージェントは**専門用語の言い換え**と**要点の抽出**に専念する。

## データ配置

- 出力: `<repo>/report-deck/<YYYY-MM>/<YYYY-MM-DD>.html`(月ごとにディレクトリ → 日付ごとにファイル)
- 入力: `<repo>/report-record/<project>/<date>.md`
- 上書き: `REPORT_DECK_DIR` / `REPORT_RECORD_DIR` 環境変数
- 言い換え辞書: `config/glossary.yaml`(脱専門用語の対訳。運用で育てる)

## トリガー

| ユーザー発話の例 | 動作 |
|---|---|
| 「今日の報告スライド作って」「日報スライドにして」 | 当日分を生成 |
| 「6/27 を報告資料に」 | 日付を限定 |
| 「今週分を報告スライドに」 | 日付範囲(各日 or まとめ)を確認して生成 |

## 標準フロー

1. **対象日の確定**: 日付(既定=当日)を受け取る。範囲指定なら各日 or 1枚集約かを確認。
   1枚集約なら `date` に範囲末日、`date_label` に表示用の範囲(例: `6/23〜6/27`)を入れる。
2. **記録を集約**: `<repo>/report-record/<project>/<date>.md` を全案件分 Read。
   - 記録が1件も無ければ「先に jarvis-record で記録を作るか」を確認(導線)。
   - `_unclassified` の記録があれば、**含めるか都度ユーザーに確認**(jarvis-record で除外済みなら不要)。
3. **依頼者言語へ変換**(★このスキルの核):
   - `config/glossary.yaml` を参照し、専門用語・固有技術名・略語を**一般語へ言い換える**。
   - 辞書に無い用語が出たら言い換えを決め、`glossary.yaml` に追記して次回以降に活かす。
     追記前に既存の `term` / `aliases` と重複していないか確認し(重複なら既存エントリの `aliases` に統合)、
     `terms` の並びは term の五十音順(英字は先頭・アルファベット順)を維持する。
   - 各案件の「やったこと/今どうなっているか/この先/相談」を、依頼者が読む粒度(各3〜5行)に要約。
     行数は進捗の大小で濃淡をつける(全案件を機械的に同じ行数にしない。3〜5行は上限の目安)。
   - 言い換えは [`personas/writing-style.md`](../../../personas/writing-style.md) の NG パターンを
     避ける(明確さ優先で語彙のみ対象)。「調整を進めました」「対応中です」型の抽象表現でなく、
     記録にある数字・固有の作業内容で書く。記録に無い進捗表現を足さない。
   - 全案件横断の「今日の要点」を立てる(サマリ用。多くて3点。1点しか無い日は1点でよい)。
4. **payload を組む**: 下記スキーマの JSON を作る。`did` のステータスは
   `done`(完了)/`wip`(進行中)/`todo`(着手のみ)/`pause`(中断)。
5. **レンダリング**: payload を `render_deck.py` に渡して HTML を生成する。
6. **自己チェック**: 生成 HTML に技術固有名詞・略語が残っていないか、逃げ文句・空語
   (「〜と言えるでしょう」「重要です」等)が残っていないか確認。残っていれば payload を直して再生成。
7. **完了提示**: 生成パス・スライド枚数・案件数を提示。ブラウザで開けば即配布できる旨を伝える。

```bash
SKILL=/Users/satoshi-onga/Documents/onclimb-industries/.claude/skills/friday-daily-report
python3 "$SKILL/bin/render_deck.py" --in /path/to/payload.json     # → 出力パスを表示
# 確認用に書き込まず標準出力へ: --stdout
```

## payload スキーマ

```json
{
  "date": "2026-06-27",
  "date_label": "6/23〜6/27",
  "reporter": "担当者名",
  "highlights": ["全案件横断の要点を3点以内", "..."],
  "projects": [
    {
      "name": "案件名(依頼者に伝わる呼称)",
      "did":    [{"status": "done", "text": "..."}, {"status": "wip", "text": "..."}],
      "status": ["今どうなっているか(1〜2行。空配列→『なし』表示)"],
      "next":   ["この先やること(空配列→『なし』表示)"],
      "ask":    ["ご相談・ご判断いただきたいこと(無ければ空配列→『なし』表示)"]
    }
  ]
}
```

- `date` は `YYYY-MM-DD` 形式必須(出力パスの決定に使う。レンダラーが形式を検証する)。
- `date_label` は任意。表紙・タイトルの表示に使う(省略時は `date` を表示)。
  範囲を1枚に集約するときは `date` = 範囲末日、`date_label` = `"6/23〜6/27"` のように指定する。
- `projects` は1件以上必須(空だとレンダラーがエラー終了する)。
- `status` / `next` / `ask` のセクションは常に出力される(空配列は『なし』表示)。

## デザイン方針(固定 — 崩さない)

- 専門用語を使わない / 毎回同じフォーマット / AI 臭・装飾過多を出さない。
- 単一の自己完結 HTML(外部 CDN・フレームワーク依存なし)。素の JS で ←→ 送り、印刷で1枚1ページ。
- 寒色系ミニマル(淡い寒色グラデ背景・濃紺/ティールの2アクセント・項目ごとに枠)。
- ステータスは色アイコンでなく語(「完了」「進行中」)で示す。
- レイアウト変更が必要なときは `templates/deck.html` の CSS とマーカーを直す
  (`<!-- HL_START/HL_END -->`、`<!-- PROJECT_SLIDES ここから/ここまで -->` の間をレンダラーが差し替える)。

## 注意

- 入力の記録が無い案件は載せない(jarvis-record を先に回す)。
- glossary に無い用語の機械的な直訳で意味が壊れそうなら、ユーザーに言い換え方を確認する。
- 機密(実名・顧客名)が記録に残っていないか、生成前に最終チェックする。
