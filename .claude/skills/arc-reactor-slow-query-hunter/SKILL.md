# slow-query-hunter — スロークエリの発見と改善提案 (arc-reactor)

ORM・生 SQL のどのクエリが遅いかを見つけ、**実測 / EXPLAIN の根拠付き**で改善案を出すスキル。
スロークエリログがある環境（ログモード）でも、無い環境（コードモード）でも動く。
「なんとなく遅そう」の列挙ではなく、そのままインデックス追加やコード修正に使えるレポートを作ることが目的。

> 大原則 1: **実測なくして断定なし。** LLM がクエリの速い遅いを見た目で断定しない。
> 実測または EXPLAIN の証拠が取れないクエリは「未計測（優先調査候補）」として分け、遅いとは書かない。
>
> 大原則 2: **DB への操作は読み取りのみ。** 実行するのは SELECT / EXPLAIN のみ。
> EXPLAIN ANALYZE は書き込み系クエリ（INSERT / UPDATE / DELETE）に対しては実行しない
> （どうしても必要ならトランザクション + ROLLBACK で保護し、その場合も実行前にユーザーへ確認する）。
>
> 大原則 3: **本番 DB への接続はユーザーの明示承認が必須。** 既定はローカル / 開発環境。
> 接続先がどこか（ホスト・DB 名）を実行前に必ず表示して確認する。

persona: [`personas/arc-reactor.md`](../../../personas/arc-reactor.md) に従う
（応答は日本語 / コード・識別子は英語 / 推測は ASSUMPTION 明記）。

## トリガー

| ユーザー発話の例 | 動作 |
|---|---|
| 「スロークエリを洗い出して」 | ログの有無を確認してモードを決め、全体を調査 |
| 「このスロークエリログを解析して改善案を出して」 | ログモード（指定ログを集計 → 発行元特定 → EXPLAIN → 改善案） |
| 「どのクエリが遅いか実測して」 | コードモード（抽出 → SQL 組み立て → 実測 / EXPLAIN） |
| 「N+1 になってる箇所を探して」 | コードモードの静的検出パート（N+1 兆候の抽出）を中心に実行 |
| 「このクエリの EXPLAIN を見て直し方を教えて」 | 単一クエリの EXPLAIN 解析 + 改善案 |
| 「クエリのレイテンシ一覧を作って」 | コードモードでレイテンシ一覧 + スロークエリランキング |

ユーザーが明示的に依頼したときだけ起動する（コードを読んだついでに自動起動しない）。
N+1 の静的検出だけ欲しい場合も本スキルで対応する（コードモードの一部）。
マイグレーションファイルの危険操作検出は別スキル（db-migration-safety-checker）の領分で、本スキルは扱わない。

## 標準フロー

### 1. 対象と DB / ORM の特定

- 対象リポジトリの絶対パスを確定する（発話 / cwd。`projects/<name>/` 配下が典型）。
- 設定ファイル（`database.yml`, `settings.py`, `prisma/schema.prisma`, `.env.example`,
  接続初期化コード等）から DBMS（PostgreSQL / MySQL 等）と ORM / クエリビルダを特定する。
  特定できない場合はユーザーに確認する。
- 入力モードを決める: スロークエリログ・APM のクエリ統計をユーザーが持っていれば**ログモード**、
  無ければ**コードモード**。両方使える場合はログモードを主、コードモードを補完に使う。

### 2. 計測環境の確認（実行前ゲート）

- 接続先候補（ローカル / 開発 / 本番）を設定から列挙し、**どこに接続するかを表示して確認**する。
  既定はローカル / 開発。本番はユーザーの明示承認が無い限り接続しない。
- 接続先 DB の**データ規模**（主要テーブルの行数）を `SELECT count(*)` 等で把握する。
  行数が本番と乖離している場合、その計測は「参考値」となる旨をこの時点でユーザーに伝える
  （行数が少ないとインデックス問題は再現しない。オプティマイザが seq scan を選ぶのはむしろ正しい挙動になる）。
- 接続できる DB が無い場合はその旨を伝え、静的解析（クエリ抽出 + N+1 兆候 + スキーマとの突き合わせ）
  のみの縮退動作になることを明示する。この場合、全クエリが「未計測」扱いになる。

### 3. 遅いクエリ候補の特定

**ログモード**:

- スロークエリログ / APM 統計をパースし、正規化したクエリ単位で集計する
  （合計時間・平均時間・実行回数。リテラルはプレースホルダに正規化して同一クエリをまとめる）。
- 合計時間の降順でランキングを作る（平均は速くても回数が多いクエリを見落とさないため）。
- 上位クエリごとに、発行元コードを `rg` で特定する（テーブル名・カラム名・ORM のメソッド名・
  SQL 断片で検索し、`file:line` を確定する）。特定できない場合は「発行元不明」と明記する。

**コードモード**:

- リポジトリからクエリ発行箇所を抽出する: ORM 呼び出し（find / where / include / preload 等）、
  生 SQL（`SELECT` 文字列、`execute` / `query` 呼び出し）、クエリビルダのチェーン。
- ループ内クエリ発行・遅延ロードの繰り返しなど **N+1 の兆候**を静的に検出する
  （判定基準は [references/explain-analysis.md](references/explain-analysis.md)）。
- 抽出箇所ごとに、実際に発行される SQL を組み立てる。パラメータはスキーマと既存データから
  代表値を選ぶ（存在する id、典型的な範囲条件）。組み立てに推測が入る箇所は
  `-- ASSUMPTION:` コメントを SQL に付ける。

### 4. 実測と EXPLAIN 解析

- 各クエリを接続確認済みの DB で実行する。**SELECT / EXPLAIN のみ**。
  - SELECT: 実行時間を計測する（キャッシュの影響を均すため 2 回以上実行し、2 回目以降を採用）。
  - EXPLAIN（PostgreSQL は `EXPLAIN (ANALYZE, BUFFERS)`、MySQL は `EXPLAIN` / `EXPLAIN ANALYZE`）
    で実行計画を取得する。書き込み系クエリは大原則 2 に従い EXPLAIN（ANALYZE なし）に留める。
- 実行計画を [references/explain-analysis.md](references/explain-analysis.md) の判定基準で読む:
  seq scan / full scan、行数見積もりの乖離、filesort、インデックス未使用、N+1。
  **解析実行前に必ず references を読むこと。**
- 実行エラー・タイムアウト・組み立て不能で証拠が取れなかったクエリは、
  推測で埋めず「未計測（優先調査候補）」に回す。

### 5. 問題分類と改善案

計測 / EXPLAIN 済みの各クエリを問題別に分類し、改善案を付ける:

- 分類: missing index / N+1 / full scan / 不要な列・行の取得 / ロック待ち
- 改善案: インデックス追加（DDL 案）、複合インデックスの列順、クエリ書き換え、
  eager load へのコード修正案（`file:line` 付き）。判定基準と定石は references に従う。
- **改善 DDL（CREATE INDEX 等）は提案のみで、実行しない。**
- 期待効果: 改善後の実測で確認できた場合のみ数値で書く（例: 1,200ms → 45ms）。
  実測していない改善案は定性表現に留める（「full scan が index scan になる見込み」）。
  ここでも数値を推測で作らない。

### 6. レポート生成と保存

以下の固定構成でスロークエリレポートを生成し、チャットで要約を提示する。
保存先は対象プロジェクト配下 `<repo>/slow-query-report/<YYYY-MM-DD>.md`
（対象リポジトリの git を汚す可能性があるため、**保存前に必ずユーザーに確認**する）。

```markdown
# Slow Query Report — <repo-name> (<YYYY-MM-DD>)

## サマリ
- モード（ログ / コード / 併用）/ 対象 DB・ORM / 接続先（ローカル・開発等）と主要テーブルの行数
- 計測済み N 件（うち要改善 n 件）/ 未計測 M 件
- データ規模が本番と乖離している場合は「本計測は参考値」の注記

## スロークエリランキング（計測済み）
| # | 発行元 (file:line) | SQL（正規化） | 実測 / EXPLAIN 要点 | 分類 | 改善案 | 期待効果 |
|---|---|---|---|---|---|---|

## 改善案の詳細
（クエリごとに: EXPLAIN の抜粋と読み、改善 DDL / コード修正案、適用時の注意）

## 未計測（優先調査候補）
| # | 発行元 (file:line) | SQL / 抽出内容 | 計測できなかった理由 | 次の一手 |
|---|---|---|---|---|

## 免責・制約
- 計測環境（接続先・データ規模）と、本番との乖離に関する注記
- 発行元を特定できなかったログ上のクエリ一覧
```

### 7. 完了提示

- 要改善クエリの件数と上位の要点をチャットで要約する。
- 次アクション（改善 DDL の適用計画、マイグレーション化するなら db-migration-safety-checker での
  事前チェック、issue 起票）を提案する。

## EXPLAIN の読み方と改善パターン

seq scan / full scan の扱い、行数見積もりの乖離、filesort、N+1 兆候の判定基準と、
インデックス追加・複合インデックスの列順・書き換え・eager load の使い分けは
[references/explain-analysis.md](references/explain-analysis.md) を参照する。
解析知識の改善（新しい兆候・DBMS 固有の読み方の追加）は references への追記で行う。

## 品質・安全性

- **実測 / EXPLAIN の証拠が無いクエリを「遅い」とレポートに書かない。**
  未計測は未計測として分ける。証拠の無い断定はレポート全体の信頼を壊す。
- DB への操作は読み取りのみ（SELECT / EXPLAIN）。改善 DDL・コード修正は提案に留め、実行しない。
  レポート保存時のみファイル書き込みを行い、保存前に確認する。
- 本番 DB への接続はユーザーの明示承認が必須。接続先は実行前に必ず表示する。
- データ量が本番と乖離した環境での計測は「参考値」であることをレポートに明記する。
- 機密情報を扱わない: 接続文字列の値（パスワード・ホスト等）をレポートに転記しない。
  計測時に読んだ実データ（個人情報を含みうる行）もレポートに貼らない。
  SQL 例のパラメータは正規化またはダミー値にする。
- レポートはプロジェクト固有情報のため、対象プロジェクトのディレクトリ配下に閉じる
  （当リポジトリ側や他プロジェクトに持ち出さない。CLAUDE.md の分離ルールに従う）。
