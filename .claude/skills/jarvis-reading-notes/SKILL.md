# reading-notes — 読書メモ台帳（投げるだけで積み上がる）

読書メモを **1 冊 = 1 ノートの台帳**に断片のまま積むスキル。
persona: [`personas/jarvis.md`](../../../personas/jarvis.md)（読み手は自分・過程を残す・平準化しない）。

## ゴール

**このスキルのゴールは、ユーザーの読書の記録を残して、公開用の読書メモの材料にすること。**
ノートは「ユーザーが実際に読んで・感じて・質問したこと」の一次記録であり、後で公開用に
整形される材料になる。だから **ユーザーが言っていないことが 1 行でも混ざれば材料として
使えなくなる**。忠実さがこのスキルの唯一の品質基準。

やることは次の 2 つだけ。これ以外の内容をノートに書かない:

1. **記録** — 本を読んで気づいたこと・気になったことを、ユーザーの言葉のまま記録する。
2. **疑問への回答 + 記録** — 本を読んでいて分からないこと・疑問に思ったことを質問されたら、
   回答した上で「疑問と回答」を記録する。

## 設計思想: 「続かない」をどう潰すか

読書メモが続かない原因を仕組みで潰す。この 4 点がこのスキルの芯（崩さない）:

1. **記録のハードルを 1 断片まで下げる** — 「この本のここ気になる」の一言で追記が完了する。
   タイトル・著者・章立てなどのメタ情報を揃えることを求めない（分かったときに埋まればよい）。
2. **質問がそのまま記録になる** — 本の疑問に回答したら、その区切りで「疑問と回答」を
   ノートに追記する（追記内容をそのまま見せて通知）。「メモを書く」という別作業を発生させない。
3. **断片を無駄にしない** — 積んだ断片は読了時のまとめの材料になる。きれいなノートは
   最後に一度だけ作ればよく、途中は雑でよい。
4. **中断を失敗にしない** — 読みかけ・積読・中断はすべて正常な状態。再開時は前回の
   気になり・未解決の疑問を出して、続きから入れるようにする。

## 場所

- ノート（正本）: リポジトリ直下 `reading-notes/books/<slug>.md`。**1 冊 = 1 ノート**。
  slug は書名からの英小文字ケバブケース（例: `designing-data-intensive-applications`,
  和書は意味の英訳か読みで可）。
- 一覧の生成元はノートの frontmatter（status 等）。索引ファイルは持たない
  （二重管理を避ける。一覧はその場で frontmatter を走査して作る）。
- `reading-notes/` は個人の学習・思考の記録のため **git 管理外**（`.gitignore` 済み）。

## トリガーと非トリガー

| 起動する発話の例 | 起動しない |
|---|---|
| 「この本のここ気になるからメモして」「p.123 の〜、読書メモに」 | 本と無関係の技術質問（普通に回答） |
| 「『〇〇』について壁打ちしたい」「この本のこれってどういうこと？」 | 「学習計画を立てて」→ karen-learning-roadmap |
| 「この本どこまで読んだっけ」「前回何が気になってたっけ」 | 「技術ノート作って」→ jarvis-knowledge-base |
| 「積読なにあったっけ」「読書メモ一覧」 | |
| 「読み終わったからまとめて」「『〇〇』を積読に追加」 | |

本の話かどうか曖昧なとき（書名が出ずに技術の疑問だけのとき）は普通に答えてよい。
会話の中で「これは読んでいる本の疑問だ」と分かった時点でモード B に切り替える。

## 記録の規律（違反歴があるため必ず守る。芯と同格）

過去のセッションで「ユーザーが言っていない読書体験の捏造」「保存していないのに保存したと
報告」「自分の生成文をユーザーの指示と取り違えて実行」が実際に起きた。再発防止として:

1. **出所チェック**: ノートに書く各文は、このセッションのユーザーの実際の発話に遡れること。
   遡れない文は書かない。「この章を読んだ人ならこう気づくはず」という推測でユーザーの
   気づき・感想・理解を生成することは捏造であり、絶対にしない。
2. **指示の出所はユーザーのメッセージのみ**: 自分が直前に出力した提案・確認案・予測を
   「ユーザーの指示」として扱わない。自分の提案は、ユーザーが明示的に承認したときだけ実行する。
   直近のユーザーメッセージに書かれた指示だけに応答する。
3. **完了報告はツール結果を確認してから**: 「記載した」「保存した」と report するのは、
   Edit / Write の成功結果を実際に確認した後だけ。書き込んだ *つもり* で報告文を生成しない。
   疑わしければ報告の前にファイルを Read して現物を確認する。
4. **追記の通知には、追記した内容をそのまま見せる**: 要約・言い換えで通知しない。
   ユーザーが捏造・取り違えをその場で発見できるようにする。
5. **記録依頼を最優先で処理する**: ユーザーの記録・質問への対応を後回しにして、
   自分発の作業（フィードバック記録・ノート構造の変更・提案）を先にやらない。
   自分発の作業はそもそもユーザーが依頼しない限り行わない。
6. **AI の分析・調査・提案をノートに書くのは、ユーザーが明示的に「それを記録して」と
   指定したときだけ**（例: 依頼されて行った外観調査）。その場合もエントリに出所を明記する。

### A. クイックキャプチャ（気になった箇所を残す）

1. 発話から書名・ページ・引用・コメントを拾う。**確認質問は新規の本のときの
   「この書名でノートを作るか」1 問だけ**（既存ノートは黙って追記 + 一言通知）。
2. ノートが無ければ [`templates/book_note.md`](templates/book_note.md) から作成
   （frontmatter は title と status: reading だけ埋まれば十分。著者・タグは後でよい）。
3. 「メモ」セクションの**末尾**に時系列で追記する:
   ```markdown
   ### YYYY-MM-DD p.<N> — 気になり
   > <引用（ユーザーが言った範囲のみ。原文を創作しない）>
   <ユーザーのコメント（言葉を言い換えない）>
   ```
   ページ・引用が無ければ無いまま書く（聞き直して摩擦を増やさない）。
4. 疑問形のキャプチャ（「〜ってどういうことなんだろう」）は frontmatter 直下の
   「未解決の疑問」リストにも 1 行追加する（再開時・壁打ち時のフックになる）。

### B. 疑問への回答 + 記録（本の疑問に答えて、疑問と回答を残す）

1. まず**普通に質問に答える**（このスキルは対話スタイルを縛らない。コードを書いて確かめる・
   他スキルを使う・Web を調べる、いずれも自由）。
2. 回答したら、その本のノートに「疑問と回答」エントリを追記し、**追記した内容をそのまま
   見せて通知する**:
   ```markdown
   ### YYYY-MM-DD <章・ページなど分かれば> — 疑問と回答
   - 疑問: <ユーザーの質問（原文の言葉を保つ）>
   - 回答: <Claude の回答の要点（回答した内容から要約する。回答していないことを足さない）>
   ```
   - 「分かったこと」「納得したこと」など**ユーザーの理解・感想を表す欄を勝手に書かない**。
     ユーザーが自分の言葉で理解を口にしたら、それはモード A の気になり/感想として
     原文のまま別途記録する。
   - 疑問が解けずに残ったら「未解決の疑問」リストにも 1 行追加する。解けた疑問は
     `✓ 解決（YYYY-MM-DD）` に更新する（行は消さない）。
3. 議論が本から離れて一般の調べ物になった場合は、本に関係する疑問と回答だけを記録する。

### C. 再開サポート（中断からの復帰）

「どこまで読んだっけ」「続き読む」で、ノートから次を短く提示する:

- 最後に触れた日付とページ（メモの最終エントリから）
- 未解決の疑問（そのまま「読みながら確かめる問い」になる）
- 直近の気になり 2〜3 件

提示だけでノートは変更しない。長く触れていない本（目安 1 か月）を一覧（モード D）で
出すときは `休眠` と添えてよいが、**再開の催促はしない**（中断は失敗ではない）。

### D. 一覧・参照

- 「読書メモ一覧」「積読は？」→ `reading-notes/books/*.md` の frontmatter を走査し、
  status 別（積読 / 読書中 / 中断 / 読了）に「書名・最終更新・未解決の疑問数」を表で出す。
- 「『〇〇』のメモ見せて」→ 該当ノートの内容を要約して答える（参照のみ）。
- 「『〇〇』を積読に追加」→ frontmatter だけのノートを作る（status: unread）。

### E. 読了まとめ（断片から一度だけ整える）

ユーザーが「読み終わった」「まとめて」と言ったときだけ:

1. status を finished にし、積んだ断片（気になり・壁打ち）を材料に「読後まとめ」を生成して
   ノート末尾に追記する: 何の本だったか（3〜5 行）/ 自分に効いた点（断片の出典日付付き）/
   解決した疑問と残った疑問 / 実務で試したいこと。
2. **断片に無い感想を創作しない**。材料が薄ければ薄いなりの短いまとめでよい
   （「もっとメモしておけば」という反省をユーザーに返さない）。
3. 技術書なら、まとめを [jarvis-knowledge-base](../jarvis-knowledge-base/SKILL.md) の vault へ
   反映するか提案する（実行はユーザーが決める）。記事ネタになりそうなら
   friday-tech-article-drafter の存在も一言添える。
4. 「実務で試したいこと」が出たら ToDo 台帳への収穫を提案する
   （`todo.py add --status inbox --source-type research --source-ref <ノートパス>`。追記したら一言通知）。

## 自己進化

固定する芯（ゴールの 2 動作・記録の規律・4 つの設計思想・1 冊 1 ノート・言葉を言い換えない）は
崩さない。育てるのは:

- キャプチャの取りこぼしパターン: 「あれメモしといて」で書名が特定できなかった等の
  つまずきが繰り返れば、書名の推定手がかり（直近に触れたノートを第一候補にする 等）を
  本ファイルに追記する。
- 壁打ちエントリの粒度: 記録が長すぎる/短すぎるという指摘があれば、エントリ形式を調整する。

## やってはいけないこと

- **ユーザーが言っていないことをユーザーの気づき・感想・理解として書かない**（捏造。
  各文は実際の発話に遡れること。「読んだ人ならこう感じるはず」を生成しない）。
- **自分が生成した文章・提案をユーザーの指示や発話として扱わない**（指示は直近の
  ユーザーメッセージからのみ。自分の提案はユーザーの明示承認があるまで実行しない）。
- **書き込んでいないのに「記載した」「保存した」と報告しない**（ツールの成功結果を
  確認してから報告する）。
- ゴールの 2 動作（記録 / 疑問への回答 + 記録）以外の内容をノートに書かない
  （AI の分析・調査・提案は、ユーザーが明示的に記録を指定したときだけ・出所明記で書く）。
- キャプチャで質問を重ねない（新規ノート作成の確認 1 問まで。メタ情報の穴埋めを迫らない）。
- ユーザーのコメント・理解の言葉を言い換えない（清書しない）。
- 引用を創作・補完しない（ユーザーが口にした範囲だけ。原文の再現を装わない）。
- AI の長い解説をそのままノートに貼らない（回答は要点に絞る）。
- 読了・再開を催促しない。メモの少なさを指摘しない（続かない罪悪感を作らない）。
- 過去エントリを書き換えない（訂正・解決は新しいエントリと ✓ 更新で表す）。
- 本文の長文転載を残さない（著作物への配慮。引用は気になった箇所の短い範囲に留める）。
