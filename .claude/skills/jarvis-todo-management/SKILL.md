# jarvis-todo-management

> **1 日のうち約 2 時間おきのチェックインで、todo の進捗・追加・困りごとを整理し、翌日への申し送りまで残す対話パートナースキル。**
> 読み手は自分自身 (jarvis 系の一次記録)。他者向けの清書はしない。

このスキルは **自己進化** を備えています。

- 各サイクルを `logs/pipeline.jsonl` に記録する。
- 完了条件は `success` / `failure` / `unknown` / `error` のいずれかを必ず付与する。
- `pipeline.config.json` の `evolution_threshold` (既定 10) 回サイクルを回したら、`scripts/evolve.py review` を実行し `EVOLUTION.md` に提案を追記、`auto_apply=true` なので自動適用される。

---

## 役割

- 頭の中の todo をファイルに下ろし、日をまたいで追跡できるジャーナルとして残す。
- 「進んでいない」状態にユーザー自身が気づく前に、スキル側が検知して質問する (ストッパー検知)。
- 計画変更・ゴール変更の経緯を memo.md に残し、後から「なぜこうなったか」を追えるようにする。
- 見積と実績の差を report.md に蓄積し、見積精度を自己進化で改善する。

---

## データ配置

運用データはスキルディレクトリではなく、リポジトリ直下の `journal/` (git 管理外) に置く。

```
journal/
  YYYY-MM-DD/
    todo.md      # その日のタスク一覧 (チェックボックス + 見積/実績/状態)
    memo.md      # 時刻付きの時系列追記 (ゴール変更・計画変更などの経緯)
    report.md    # 終業時のみ生成 (完了/未完了/差分/申し送り)
  backlog.md     # いつかやるリスト (カテゴリ見出し + 項目)
```

- ベースディレクトリは環境変数 **`JOURNAL_DIR`** で上書き可能 (未設定時はカレントリポジトリ直下の `journal/`)。
- `journal/` は `.gitignore` 済み。**生ログ・過去日のファイルは削除しない** (追記・新規作成のみ)。

## トリガー

| 種別 | タイミング | 発話例 | 追加動作 |
|---|---|---|---|
| **始業チェックイン** | その日最初の起動 | 「今日の todo 作って」「backlog から今日やるやつ拾って」 | `init_today.py` で当日ディレクトリ生成 + 前日申し送り・backlog からの選別提示 |
| **中間チェックイン** | 約 2 時間おきの節目 | 「チェックインお願い」「進捗整理して」「いまの状況棚卸し」 | 共通フローのみ |
| **終業チェックイン** | その日の締め | 「今日の作業報告まとめて」 | `finalize_day.py` で report.md 雛形生成 + 未完了の振り分け |

当日の `journal/YYYY-MM-DD/` が無い状態でチェックインを求められたら、始業チェックインとして扱う。
起動時には **最後のチェックインからの経過時間** (memo.md の最終時刻から算出) を最初に一言添える。

## 起動時の手順 (毎サイクル必須)

### Step 0. パイプライン開始

```bash
CYCLE_ID=$(python3 scripts/pipeline.py --skill-root <このスキルのルート> log-start \
  --skill-name jarvis-todo-management \
  --instruction "<受け取った指示の要約 (機密マスク済み)>")
```

### Step 1〜5. チェックイン共通フロー

すべてのチェックイン (始業・中間・終業) で、次の 5 ステップを順に行う。

1. **進捗確認** — 当日の `todo.md` を読み、各タスクの状態 (未着手 / 進行中 / 完了 / 保留) をユーザーに問う。
2. **追加 todo の確認** — 新しく積むものがあるか聞く。追加分は見積を付けて `todo.md` に追記する (見積ロジックは後述)。
3. **ストッパー確認** — 後述の検知ルールに該当するタスクがあれば **質問形式で** 確認する。該当がなければ「困りごとは特にないですか?」と一度だけ確認する。
4. **次アクションの提案** — 残りタスク・締切・見積から、直近で取り組む順序・優先度を提案する (決定はユーザー)。
5. **ファイル更新** — `todo.md` を更新し、ゴール変更・計画変更など todo を書き換えると消える文脈は `append_memo.py` で `memo.md` に時刻付きで追記する。最後に現状サマリ (進捗 / 残り / 推奨アクション) をチャットに返す。

### Step 6. パイプライン終了

```bash
python3 scripts/pipeline.py --skill-root <このスキルのルート> log-end \
  --skill-name jarvis-todo-management \
  --cycle-id "$CYCLE_ID" \
  --completion-state success \
  --completion-reason "<根拠>" \
  --output-summary "journal/YYYY-MM-DD/todo.md ほか更新ファイルのパス"
```

**完了条件を出さずにサイクルを閉じてはいけない。** `log-end` の戻り JSON が `"evolution_due": true` なら `python3 scripts/evolve.py review` を実行する。

完了条件の目安:

- **success** — チェックインの 5 ステップを完了し、ファイル更新とサマリ提示まで終えた。
- **failure** — フロー途中でユーザーの意図を満たせず終了した (原因特定済み)。
- **unknown** — 出力はしたが、振り分けや見積の追認待ちで確定していない。
- **error** — スクリプト失敗・journal 書き込み不能など。

## 始業時の追加動作

1. `python3 scripts/init_today.py` を実行し、`journal/YYYY-MM-DD/` と `todo.md` / `memo.md` を生成する (前日ディレクトリの `report.md` パスも表示される)。
2. 前日 `report.md` の「翌日への申し送り」と `backlog.md` を読む。
3. **スキルが「今日やる候補」を選別して理由付きで提示する** — 申し送りは原則全件、backlog からは締切・関連性を根拠に数件。全件をそのまま見せない。
4. ユーザーが候補を確定・修正し、確定分だけを `todo.md` に昇格させる。**最終決定権は常にユーザー**。
5. 各タスクに見積を付ける (下記「見積の初期値」)。

### 見積の初期値 (3 段フォールバック)

1. 過去の `journal/*/report.md` に **同種タスクの実績** があれば、その **中央値** を初期見積として提案する。
2. 過去実績がなければ **ユーザーに所要時間を質問** する。
3. ユーザーも答えられなければ、**30 分単位の粗い見積** (30m / 60m / 90m ...) を根拠付きで提案し、追認を得る。

<!-- ASSUMPTION: 「同種タスク」の判定は当面、タスク名・カテゴリの類似で行い、厳密な分類器は持たない。自己進化で精緻化する。 -->

---

## 終業時の追加動作

1. `python3 scripts/finalize_day.py` を実行し、`report.md` の雛形 (完了 / 未完了 / 見積と実績の差分 / 翌日への申し送り) を生成する。
2. `todo.md` の内容から各セクションを埋める。見積と実績の差分は必ず数値で残す (自己進化の学習材料)。
3. 未完了タスクについて、**スキルが振り分け案 (翌日繰越 / backlog 戻し / 廃棄) を理由付きで提案し、ユーザーが確定する**。自動確定しない。
4. 確定結果を「翌日への申し送り」に記録し、backlog 戻し分は `backlog.md` の該当カテゴリにも追記する。

## 各ファイルの最小スキーマ

初版スキーマ。列の昇格・削除は自己進化レビューで行う。

### todo.md — 1 行 1 タスク

```markdown
# Todo — YYYY-MM-DD

- [ ] タスク名 (見積: 30m / 実績: - / 状態: 未着手)
- [x] タスク名 (見積: 60m / 実績: 45m / 状態: 完了)
```

- 状態は `未着手` / `進行中` / `完了` / `保留` の 4 値。
- 完了時は `- [x]` に変え、実績を記入する。

### memo.md — 時刻付き追記のみ

```markdown
# Memo — YYYY-MM-DD

- 10:30 タスク A のゴールを X から Y に変更 (理由: ...)
```

追記は `scripts/append_memo.py` を使う。既存行の書き換え・削除はしない。

### report.md — 固定 4 セクション

```markdown
# Report — YYYY-MM-DD

## 完了
## 未完了
## 見積と実績の差分
## 翌日への申し送り
```

### backlog.md — カテゴリ見出し + 項目

```markdown
# Backlog

## <カテゴリ名>
- 項目 (任意で締切・メモ)
```

## ストッパー検知ルール

以下のいずれかに該当するタスクを検知したら、ストッパー確認を行う。

- **条件 A**: 前回チェックイン時点と比べて **状態が変わっていない未完了タスク** がある。
- **条件 B**: **実績 (進行中タスクの経過時間含む) が見積の 1.5 倍を超過** している。

検知時の振る舞い:

- 必ず **質問形式** で確認する。例: 「タスク A、前回から動いていないようですが、何かに詰まっていませんか?」
- 「詰まっている」「サボっている」と **断定しない**。memo.md の文脈 (会議・割り込み等) を先に確認してから聞く。
- ユーザーが「問題ない」と答えたら深追いしない。閾値 (1.5 倍) と聞き方は自己進化の調整対象。

## 自己進化の範囲

**進化対象 (使用ログから書き換えられる)**

- 見積精度 — タスク種別ごとの所要時間の傾向、初期値の提案ロジック
- ファイルスキーマ — todo/memo/report/backlog に必要な列・セクションの昇格・削除
- 聞き方 — チェックインでの問いの量・トーン、ストッパー確認の言い回し
- backlog 推薦 — 「今日やる候補」の選出ロジック

**進化対象外 (固定)**

- チェックイン共通フローの 5 ステップ構造
- 「スキルは提案、確定はユーザー」の原則 (todo 昇格・未完了振り分け・見積確定)
- `scripts/pipeline.py` (自己破壊防止)
- ユーザー個別の好みは `references/user_preferences.md` に分離し、SKILL.md 本文に混ぜない

## 品質・安全性 (jarvis persona 準拠)

- **生ログ保全** — journal/ 配下の過去ファイル・memo の既存行は削除・書き換えしない。修正は追記で表現する。
- **機密マスキング** — API キー・トークン・顧客名等は todo/memo/report にもパイプラインログにも書かない (書く前にマスクする)。
- **推測の明示** — 事実に基づいて記述し、推測した箇所 (見積の根拠、ストッパーの原因仮説など) は推測であることを明示する。コード内は `# ASSUMPTION:`、サイクルログは `--assumption-note` で残す。
- 応答・出力は日本語、コード・識別子は英語。技術用語は原語のまま。

## スクリプト

すべて python3 標準ライブラリのみ。`JOURNAL_DIR` 環境変数でデータ置き場を上書きできる。

| スクリプト | 役割 |
|---|---|
| `scripts/init_today.py` | `journal/YYYY-MM-DD/` と `todo.md` / `memo.md` を生成。前日ディレクトリの `report.md` パスも表示 |
| `scripts/append_memo.py` | `memo.md` に時刻付きで 1 行追記 |
| `scripts/finalize_day.py` | `report.md` の固定 4 セクション雛形を生成 |
| `scripts/pipeline.py` | 自己進化パイプラインのロガー (規約標準・改変禁止) |
| `scripts/evolve.py` | 進化レビュー (規約標準) |

---

## やってはいけないこと

- ユーザーの確定なしに todo の昇格・未完了の振り分け (特に廃棄) を確定する。
- ストッパー検知を断定口調で伝える。
- journal/ 配下の既存ファイル・既存行を削除する。
- 完了条件を空欄のままサイクルを閉じる。
- `scripts/pipeline.py` を進化レビューで書き換える。
- 他者向けの清書・報告資料をこのスキルで作る (それは friday 系の責務)。
