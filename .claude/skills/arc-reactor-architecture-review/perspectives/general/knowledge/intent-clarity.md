# 意図の一目瞭然性

- タグ: readability
- シグナル: 初見の読み手が「このコード/モジュールが何をしたいのか」を実装を追わないと
  把握できない。名前・シグネチャ・入口の構成が目的を語っていない。
- なぜ問題か: 変更のたびに読解コストがかかり、誤読による退行が起きる。AI 支援で動くコードが
  大量に生まれるほど、「人が読んで理解し、安全に変え続けられる」ことの価値が上がる。
- 検出方法: 対象（関数 / ファイル / モジュール / 機能）を初見の読み手の目で読み、4 段階で判定する。

  | 評価 | 基準 |
  |---|---|
  | ◎ | 名前・シグネチャ・構成（入口）だけで目的が分かる。実装を読む必要がない |
  | ○ | 入口のファイル/関数を 1 つ読めば目的と主要フローが分かる |
  | △ | 目的の把握に複数ファイルの読解や実装の追跡が必要。名前が実態を語っていない箇所がある |
  | × | 追跡しても目的が判然としない。名前が実態と食い違う／フローが分岐と副作用で追えない |

  判定に使う問い:
  - **名前で分かるか**: 関数・型・モジュール名とシグネチャだけで役割・入出力・単位・真偽の向きが
    推測できるか。実態と食い違う「嘘をつく名前」は特に重い。
  - **主要フローが追えるか**: 幹の処理が入口から素直に追えるか（数値の目安は
    [[long-function-deep-nesting]] 参照）。
  - **全体像に何ファイル要るか**: 1 機能で 5 ファイル超を行き来しないと目的が掴めないなら
    意図が拡散している兆候（初期値）。
  - **ノイズが無いか**: 自明な what を説明するだけのコメント・死にコードが本筋の読み取りを
    邪魔していないか。逆に非自明な判断・回避策に「なぜ」の説明があるか。
- 重要度の目安: △以下と判定したら、どの問いで引っかかったかを根拠に imo。
  嘘をつく名前は imo 以上。断定できなければ ask（「何を handle する意図か確認したい」形式）。
- 例外: 生成コード・外部仕様の忠実な写しなど、読み手が意図把握を求められない領域。
- メモ・出典: 旧 arc-reactor-code-review references/architecture-review.md から移植（2026-07-21）。
  数値はすべて初期値。運用で調整する。
