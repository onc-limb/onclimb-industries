# perspectives/ — 観点とインデックスの運用規約

arc-reactor-architecture-review の観点管理。**知識の実体はここには置かない** —
正本は onclimb-industries リポジトリルートのドメイン別知識ディレクトリ
（`architecture/` を主に `frameworks/<fw>/`・`testing/` 等）で、各観点は
「どのファイルを読むか」の INDEX.md だけを持つ。スキル本体（SKILL.md）は
このディレクトリを実行時に列挙するため、観点の追加・分割でスキル本体の改修は不要。

## 現在の観点

| 観点 | 状態 |
|---|---|
| `general/` | 全知識を 1 つの INDEX で参照中。分類が固まったら分割する（SKILL.md フロー D） |

## 構造

```
perspectives/
└── <観点名>/                  # kebab-case 英語名。1 ディレクトリ = 1 観点 = 1 サブエージェント
    ├── PERSPECTIVE.md         # 骨格: 観点の定義・レビュー手順・レポート書式。めったに変えない
    ├── INDEX.md               # この観点が読む知識ファイル/ディレクトリの一覧（リポジトリルート相対）
    └── knowledge-template.md  # 知識ファイルの書式テンプレート（進化対象。SKILL.md フロー C）
```

## 知識の追加（SKILL.md フロー B）

- 知見は `architecture/` 等のドメインディレクトリに 1 テーマ 1 ファイルで追加する
  （書式は `knowledge-template.md`。近接テーマがあれば既存ファイルに追記）。
- 追記先ディレクトリが INDEX.md に載っていれば INDEX の更新は不要。
  新しいドメインディレクトリに置いたときだけ INDEX に 1 行足す。
- 全項目が埋まらなくてもよい。空欄 + `TODO` で残して、後で育てる。

## テンプレートの進化（SKILL.md フロー C）

- `knowledge-template.md` は仮の書式。運用しながら項目を足し引きしてよい。
- テンプレートを変えても既存ファイルは書き換えない（新規分から適用）。
  旧書式のファイルや、書式に従わない原則ドキュメント（clean-architecture.md 等）も
  INDEX に載っていればレビュー時はそのまま有効。

## 観点の分割（SKILL.md フロー D）

- 知識が増えて分類が見えてきたら、各ファイルの「タグ」行を手がかりに分類案を作り、
  ユーザー承認のうえで新しい観点ディレクトリを作成する。
- 知識の実体は動かさず **INDEX の振り分けで観点を分ける**のが基本
  （必要なら知識ディレクトリ側にサブディレクトリを切って `git mv` してもよい）。
- 新観点の PERSPECTIVE.md は general のものをベースに、観点定義の節だけ書き換える。
- 分割したらこの README の観点一覧を更新する。
