# perspectives/ — 観点と知識の運用規約

arc-reactor-architecture-review のレビュー判断基準の正本。スキル本体（SKILL.md）は
ここを実行時に列挙して使うため、観点・知識の追加でスキル本体の改修は不要。

## 現在の観点

| 観点 | 状態 |
|---|---|
| `general/` | 全知識を集約中。分類が固まったら分割する（SKILL.md フロー D） |

## 構造

```
perspectives/
└── <観点名>/               # kebab-case 英語名。1 ディレクトリ = 1 観点 = 1 サブエージェント
    ├── PERSPECTIVE.md      # 骨格: 観点の定義・レビュー手順・レポート書式。めったに変えない
    └── knowledge/          # 中身: レビュー知識。どんどん増やす
        ├── _template.md    # 書式テンプレート（これ自体も進化対象。SKILL.md フロー C）
        └── <テーマ>.md     # 1 テーマ = 1 ファイル。kebab-case 英語名
```

## knowledge の追加（SKILL.md フロー B）

- 「こういうコードは怪しい」「この原則はこう考える」という知見が出たら、
  `_template.md` の書式で 1 ファイル追加する（近接テーマがあれば既存ファイルに追記）。
- 全項目が埋まらなくてもよい。空欄 + `TODO` で残して、後で育てる。
- 週次レビューのレポートに出る「knowledge 追加候補」も、採用したらこの手順で追加する。

## テンプレートの進化（SKILL.md フロー C）

- `_template.md` は仮の書式。運用しながら項目を足し引きしてよい。
- テンプレートを変えても既存ファイルは書き換えない（新規分から適用）。
  古い書式のファイルもレビュー時はそのまま有効。

## 観点の分割（SKILL.md フロー D）

- knowledge が増えて分類が見えてきたら、各ファイルの「タグ」行を手がかりに分類案を作り、
  ユーザー承認のうえで新しい観点ディレクトリへ `git mv` する。
- 新観点の PERSPECTIVE.md は general のものをベースに、観点定義の節だけ書き換える。
- 分割したらこの README の観点一覧を更新する。
