# arc-reactor-architecture-review — 週次アーキテクチャレビュー

コードベースを「ソフトウェアをソフトたらしめる」目線で診る週次レビュースキル。
ベースブランチに前回レビュー以降加わった変更を起点に、コードスメル・結合/依存・
設計原則・意図の読みやすさといった**全体構造の健全性**を評価する。

レビューの判断基準は本ファイルにハードコードせず、`perspectives/` 配下の
**観点ディレクトリ（PERSPECTIVE.md + knowledge/）** に置く。ユーザーが知見を
どんどん追記して育てることが前提の設計。

## 役割

- 週次で、対象リポジトリのベースブランチに加わった変更とその周辺構造をレビューする。
- 観点ごとにサブエージェントを並列起動し、観点別レポート + 横断サマリをドキュメントとして残す。
- ユーザーの知見（怪しいコードのシグナル・原則の考え方）を `knowledge/` に蓄積し、
  レビュー精度を運用で上げていく。

## arc-reactor-code-review との棲み分け

| | arc-reactor-code-review | 本スキル |
|---|---|---|
| タイミング | PR 作成ごと・随時 | 週一（長期実行でよい） |
| 対象 | 変更単体（PR / 差分）や特定機能 | ベースブランチの週次変更 + コードベース全体構造 |
| 観点 | 機能 / セキュリティ / パフォーマンス | アーキテクチャ・設計（perspectives/ で管理） |
| 問い | この変更は正しく・安全に・速く動くか | このコードベースは読めて・追えて・変え続けられるか |

「依存関係が複雑になってないか」「このコード何をしたいか一目で分かる？」「設計観点で見て」
「コードスメルを探して」は本スキル。変更の正しさ・脆弱性・性能は arc-reactor-code-review へ。

## 観点と知識の構造

```
perspectives/
├── README.md              # 観点・knowledge・テンプレートの運用規約
└── general/               # 現在は全知識をここに集約（分類が固まったら分割する）
    ├── PERSPECTIVE.md     # 観点の定義・レビュー手順・レポート書式（骨格。めったに変えない）
    └── knowledge/         # ユーザーが育てる知識。1 テーマ = 1 ファイル
        ├── _template.md   # 書式テンプレート（これ自体も進化対象）
        └── *.md
```

- **観点はハードコードしない**。フロー A では実行時に `perspectives/*/` を列挙して
  観点ごとにサブエージェントを立てる。観点を分割・追加してもこの SKILL.md の改修は不要。
- 現在は観点が `general` 1 つなのでサブエージェントも 1 つだが、フローとプロンプトは
  観点数に依存しない（分割した時点で自然に並列になる）。

## フロー A: 週次レビュー（メインフロー）

長期実行してよい。深さを時間で妥協しない。

1. **対象と範囲の確定**:
   - 対象リポジトリ（`projects/<name>/` 配下を想定）とベースブランチ（既定 `main`。
     リポジトリの既定ブランチに従う）を発話から特定する。曖昧なら確認する。
   - レビュー範囲: `projects/<name>/architecture-review/` の**最新日付ディレクトリの
     `summary.md` frontmatter `reviewed_commit`** から現在のベースブランチ HEAD まで。
     過去レポートが無ければ初回として `git log --since="7 days ago"` の範囲 + 全体構造を対象にする。
   - 範囲内に変更が無ければ、その旨を報告してレビューをスキップする（空レポートは作らない）。
2. **共通材料の準備**（オーケストレータが 1 回だけ。scratchpad に `materials.md` を作る）:
   - 期間内のコミット一覧（`git log --oneline <from>..<to>`）
   - 変更ファイル一覧と規模（`git diff --stat <from>..<to>`）。自動生成物（lock ファイル・
     ビルド成果物・生成コード）は除外して明記する
   - 変更が集中したモジュール/ディレクトリ（今回の重点領域）
   - ディレクトリ構成の概観（`ls` / `tree -L 2` 相当）
   - 前回レポートの指摘一覧（あれば。解消済み/継続中の判定材料）
3. **観点別サブエージェントの並列起動**: `perspectives/*/` を列挙し（`_template.md` や
   README.md は観点ではない。ディレクトリのみ）、観点ごとに 1 エージェント
   （general-purpose）を**単一メッセージで並列に**起動する。プロンプトの雛形:

   ```
   あなたは arc-reactor-architecture-review の観点別レビュアーです。以下を順に実行してください。
   1. 観点ディレクトリ <perspectives/<観点>/ の絶対パス> の PERSPECTIVE.md と、
      knowledge/ 配下の全ファイル（_template.md を除く）を読む。
   2. 共通材料 <materials.md の絶対パス> を読み、対象リポジトリ <絶対パス> を
      この観点だけでレビューする。重点は材料に書かれた「変更が集中した領域」だが、
      そこから辿れる依存・呼び出し関係は全体を見てよい。
   3. 前回の同観点レポート <パス>（あれば）を読み、指摘を 解消済み / 継続中 / 新規 に区別する。
   4. 結果を PERSPECTIVE.md の「レポート書式」に従って <出力先絶対パス> に Write する。
   5. 最終応答には「ラベル別の指摘件数と要点 3 行以内」だけを返す（本文はファイルに書く）。
   制約: 対象リポジトリのコードは一切変更しない。knowledge に無い根拠での創作をしない
   （knowledge 外の指摘は可だが必ず「knowledge 外」と明記する）。機密値をレポートに転記しない。
   ```

4. **サマリ生成**: 全エージェント完了後、各観点レポートを読んで `summary.md` を書く。
   内容: 観点×ラベルの件数表 / 今週の要注意領域（3 つ以内） / 前回からの変化
   （解消済み・継続中・新規の要約） / 各観点の「knowledge 追加候補」の集約 / 未レビュー範囲。
   frontmatter は「レポート規約」のとおり。
5. **ユーザーへの提示**: サマリの要点とレポート群のパスをチャットで提示する。
   knowledge 追加候補があれば「knowledge に採用するか」を**この区切りで 1 回だけ**確認し、
   採用分をフロー B で記録する。

## フロー B: 知識の追記（自己進化ルート）

「アーキテクチャレビューの観点に追加: 〜」「こういうコードは怪しいので覚えて」
「この原則はこう考えるようにして」等で起動。CLAUDE.md の「辞書・資産系はその場で追記」
ルートに該当するため、skill-feedback.md を経由せず直接 knowledge/ に書く。

1. 内容を `_template.md` の書式に整形する。埋まらない項目があれば **1 回だけ**まとめて
   ヒアリングし、それでも不明な項目は空欄 + `TODO` で残す（完璧を追記の障壁にしない）。
2. 既存の knowledge ファイルと重複していないか確認する。重複・近接するテーマなら
   既存ファイルへ追記し、新規テーマなら kebab-case 英語名で新規ファイルを作る。
3. 追記したら一言通知する（どのファイルに何を書いたか）。
4. このフローで SKILL.md / PERSPECTIVE.md は書き換えない（それらの変更はユーザーの明示指示のみ）。

## フロー C: テンプレートの進化

「テンプレートに〜の項目を足して」「この項目は要らない」等で `_template.md` を修正する。

- 修正は `_template.md` のみ。**既存 knowledge ファイルの一括書き換えはしない**
  （新規作成分から新テンプレートを適用する。過去分の一括変換はユーザーが明示した場合のみ）。
- レビュー実行時、テンプレートと書式が違う古い knowledge ファイルもそのまま有効として読む。

## フロー D: 観点の分割

knowledge が増えてきて「分類して観点を分けたい」と言われたら実行する。

1. `knowledge/` 全ファイルの「タグ」行と内容を見て、分類案（観点名・所属ファイル）を提示する。
2. ユーザー承認後: 新しい `perspectives/<観点名>/` を作成し、PERSPECTIVE.md を
   general のものをベースに観点定義だけ書き換えて用意し、該当 knowledge ファイルを `git mv` する。
3. SKILL.md の改修は不要（フロー A は実行時列挙）。`perspectives/README.md` の観点一覧だけ更新する。

## レポート規約

```
projects/<name>/architecture-review/
└── YYYY-MM-DD/
    ├── summary.md        # 横断サマリ。frontmatter が次回実行の状態を兼ねる
    └── <観点名>.md       # 観点別レポート（サブエージェントが直接 Write する）
```

`summary.md` の frontmatter:

```yaml
---
target: projects/<name>
base_branch: main
reviewed_range: <from-sha>..<to-sha>
reviewed_commit: <to-sha>        # 次回はここを起点にする
date: YYYY-MM-DD
perspectives: [general]
---
```

- 状態ファイルは別に持たない。**最新レポートの frontmatter が状態**。
- 対象が `projects/<name>/` 配下ならこのパスにそのまま保存してよい。それ以外のリポジトリは
  保存先をユーザーに確認する。
- 重要度ラベルは arc-reactor-code-review と同じ `[must]/[imo]/[nit]/[ask]` を使う
  （基準は PERSPECTIVE.md に記載。現状駆動: 放置すると実害・変更不能に至る蓋然性が高いものだけ must）。

## 品質・安全性

- **読み取り専用**: レビューは対象リポジトリのコードを一切変更しない。依存解析ツールを
  対象リポジトリにインストールしない（`npx`/`uvx` 等の一時実行のみ可。無理なら grep で代替）。
- **創作しない**: 実際に読んだコードだけを根拠にする。指摘には `file:line`（構造レベルの指摘は
  モジュール対 + 代表参照行）と、根拠にした knowledge ファイル名を添える。
- **ASSUMPTION の明記**: レイヤ構成の推定など、推測に基づく指摘は `(ASSUMPTION: ...)` を付け、
  確信が持てなければ [ask] に落とす。
- **機密を含めない**: シークレット様文字列を見つけても値はレポートに転記しない
  （変更単体の機密検査は arc-reactor-code-review の領分だが、見つけたら伏せて報告はする）。
- **モデル**: 判断・設計系のためピン留めせずセッションモデルを継承する
  （personas/model-selection.md 参照）。
