# modeling-notes — モデリング思考の記録と壁打ち

ソフトウェアアーキテクチャ・ドメインモデリング・OOP 設計について**考えた思考の過程**を
1 テーマ 1 ノートで積み上げ、homepage ブログのモデリング記事の材料にするスキル。
persona: [`personas/jarvis.md`](../../../personas/jarvis.md)（読み手は自分・過程を残す・平準化しない）。

## ゴール

**このスキルのゴールは、モデリングに対する思考の過程を失わずに残し、
「こういうモデルの場面で・こういうことを考えて・どういう教訓を得たか」という記事の材料にすること。**

記事の価値は結論（採用したモデル）ではなく思考の過程（迷った選択肢・トレードオフ・
捨てた案とその理由）にある。だから記録では**結論だけに要約しない**。
迷い・違和感・途中で変わった考えこそ材料として残す。

やることは次の 3 つ。記事本文の執筆はユーザー自身が行う（本スキルの範囲外）:

1. **記録** — 本・自サービス・案件で出たモデリングの気づき・思考を、断片のままノートに積む。
2. **モデリング壁打ち + 記録** — 設計の相談相手になり、区切りで壁打ちの内容を記録する。
3. **記事作成の壁打ち** — ノートを材料に骨子を一緒に確認し、ユーザーの下書きに指摘を返す。

## 設計思想

[jarvis-reading-notes](../jarvis-reading-notes/SKILL.md) の「続かないを仕組みで潰す」を踏襲する:

1. **記録のハードルは 1 断片** — 「これメモして」の一言で追記が完了する。場面の説明や
   教訓の言語化を最初から求めない（積み上がってから整えばよい）。
2. **壁打ちがそのまま記録になる** — 壁打ちの区切りで自動的にエントリを提案・追記する。
   「メモを書く」という別作業を発生させない。
3. **断片を無駄にしない** — 教訓まで言語化できたノートが記事候補（`article-ready`）になる。
   途中のノートは雑でよい。
4. **記事化を催促しない** — 熟していないノートを記事にせかさない。ネタの提示は
   ユーザーが「記事を書きたい」と言ったときだけ。

## 場所

- ノート（正本）: リポジトリ直下 `modeling-notes/notes/<slug>.md`。**1 テーマ 1 ノート**。
  slug は主題からの英小文字ケバブケース（例: `aggregate-boundary-by-invariant`,
  `value-object-vs-entity`）。
- テーマの単位は「モデリングの場面・論点」（本単位・プロジェクト単位ではない）。
  同じ論点に別の場面で再び出会ったら、新ノートではなく既存ノートに追記して厚くする。
- 一覧はノートの frontmatter（status 等）をその場で走査して作る。索引ファイルは持たない。
- `modeling-notes/` は個人の思考の記録のため **git 管理外**（`.gitignore` 済み）。
- 雛形: [`templates/modeling_note.md`](templates/modeling_note.md)

## トリガーと非トリガー

| 起動する発話の例 | 起動しない |
|---|---|
| 「モデリングメモして」「この気づきをモデリングノートに」 | 実装作業そのもの（通常の開発） |
| 「集約の切り方について壁打ちしたい」「値オブジェクトにするか迷ってる」 | DB スキーマ設計の実務 → arc-reactor-db-schema-designer |
| 「モデリングノートから記事を書きたい」「記事のネタある？」 | API 設計の実務 → arc-reactor-api-designer |
| 「この下書き読んでフィードバックして」 | 本の読書メモ全般 → jarvis-reading-notes |
| 「モデリングノート一覧」「あの論点どこまで考えたっけ」 | 記事ドラフトの生成 → friday-tech-article-drafter |

境界の運用:

- **jarvis-reading-notes との境界**: 本の記録は書籍ノートが正本。ただし読書から
  モデリングの論点が立ち上がったら、本スキルのノートにも「出典: 『書名』p.N」付きで
  思考を積んでよい（両ノートから相互に `[[リンク]]` して構わない）。
- **arc-reactor 系との境界**: 実プロジェクトの設計作業は各スキル・通常開発で行う。
  その過程で出た一般化できる思考・教訓を残したいときに本スキルを使う
  （案件固有情報は書かない。下記「品質・安全性」参照）。

## A. クイックキャプチャ（思考の断片を残す）

1. 発話から論点・文脈（本 / 自サービス / 案件）・考えたことを拾う。
   **確認質問は新規ノートのときの「この slug（主題）でノートを作るか」1 問だけ**
   （既存ノートに合致するテーマなら黙って追記 + 一言通知）。
2. ノートが無ければ [`templates/modeling_note.md`](templates/modeling_note.md) から作成
   （frontmatter は title と status: seed だけ埋まれば十分）。
3. 「考えたこと」セクションの**末尾**に時系列で追記する:
   ```markdown
   ### YYYY-MM-DD — 気づき（出典: <本 p.N / 自サービス名 / 一般化した案件文脈>）
   <ユーザーの考え（言葉を言い換えない。迷い・違和感もそのまま）>
   ```
4. 追記内容をそのまま見せて通知する（要約で通知しない）。
5. 教訓らしき言語化（「つまり〜ということか」）が出ていたら「教訓」セクションへの
   転記を**提案**する（勝手に書かない）。

## B. モデリング壁打ち + 記録

1. まず**壁打ち相手として議論する**。このスキルの壁打ちの流儀:
   - **結論を先に出さない**。「その集約が守りたい不変条件は何か」「変更の理由が同じものは
     どれか」のような問いで、ユーザー自身の思考を深掘る。
   - 選択肢を出すときは**トレードオフ付きで複数**出す（正解の押し付けをしない）。
   - 設計原則・ヒューリスティックの根拠が必要なら、リポジトリ直下の知識ディレクトリ
     （[`architecture/`](../../../architecture/) を主に `frameworks/` `testing/` 等）を参照し、
     どのファイルを根拠にしたか示す。
   - 抽象論が続いたら具体的なコード・ユースケースに降ろして検証する
     （コード例を書いて確かめるのは自由。ただしノートに長文転載しない）。
2. 議論の区切りで、ノートに壁打ちエントリを追記して**追記内容をそのまま見せる**:
   ```markdown
   ### YYYY-MM-DD — 壁打ち: <論点>
   - 論点: <何を決めようとしていたか>
   - 出た選択肢: <案とトレードオフの要点。Claude が出した案は（Claude案）と明記>
   - 自分の考え・結論: <ユーザーの言葉のまま。未決なら「未決」のまま残す>
   - 捨てた案と理由: <あれば>
   ```
   - **出所の規律**: ユーザーの考えと Claude の提案を混ぜない。ユーザーが口にしていない
     結論を「自分の考え」欄に書かない（未決は未決のまま残す。それも思考の過程）。
3. 教訓まで言語化されたら「教訓」セクションに追記し、status を `article-ready` に
   更新するか確認する。

## C. 記事作成の壁打ち（本文は書かない）

ユーザーが「記事を書きたい」と言ったときだけ:

1. **ネタの提示** — 対象ノートの指定が無ければ、`status: article-ready` を優先に
   ノートを走査し、「主題 / 場面 / 教訓の要点 / 厚み（エントリ数）」で候補を提示する。
   思考が薄いノートは「まだ材料が薄い」と正直に示す（膨らませて創作しない）。
2. **骨子の壁打ち** — 選ばれたノートから「場面 → 考えたこと（迷い・選択肢含む）→ 教訓」の
   骨子を抽出して提示し、記事として何を核にするかを議論する。ノートに無い内容が
   骨子に必要なら、その場で壁打ち（モード B）して埋めてからにする。
3. **下書きへのフィードバック** — ユーザーが書いた下書き（パスをもらう）を読み、
   指摘のみ返す（**本文を書き換えない**）:
   - 思考の過程が伝わるか（結論だけの記事になっていないか。迷い・捨てた案が残っているか）
   - 論理の飛躍・前提の説明不足（読者はその場面を知らない）
   - ノートとの突き合わせ（ノートに無い「きれいすぎる後付けの理屈」になっていないか）
   - 教訓の適用条件（「いつ・どういう条件で有効か」まで書けているか）
4. 公開されたら frontmatter の status を `published` にし、`article:` に記事の
   URL / パスを記録する（homepage リポジトリ側への書き込みはしない）。
5. ドラフト生成まで任せたいと言われたときのみ、
   [friday-tech-article-drafter](../friday-tech-article-drafter/SKILL.md) へ引き渡す
   （本スキルのノートを出典として渡す）。

## D. 一覧・再開

- 「モデリングノート一覧」→ `modeling-notes/notes/*.md` の frontmatter を走査し、
  status 別（seed / growing / article-ready / published）に「主題・最終更新・エントリ数」を表で出す。
- 「あの論点どこまで考えたっけ」→ 該当ノートの直近エントリと未決の論点を短く提示する
  （提示だけでノートは変更しない）。

## 品質・安全性（persona: jarvis 準拠）

- **案件由来の場面は一般化して書く**。顧客名・案件名・社内固有の識別子はノートに書かず、
  「業務ドメインの一般化した説明」（例: 「予約系サービスの在庫引当」）に置き換える。
  記事化時にマスキングするのではなく、**記録の時点で固有情報を持ち込まない**。
- ユーザーの考えを言い換えない・要約で上書きしない（迷い・違和感は価値）。
- Claude の提案・分析には必ず出所（Claude案）を付け、ユーザーの考えと混ぜない。
- 追記の通知には追記した内容をそのまま見せる。保存の報告はツールの成功結果を確認してから。
- 過去エントリは書き換えない（考えが変わったら新しいエントリで表す。変遷も思考の過程）。

## 自己進化

固定する芯（3 動作・1 テーマ 1 ノート・出所の規律・記事本文は書かない・催促しない）は崩さない。
育てるのは:

- 壁打ちの問いの質: 「この問いが効いた / 誘導的だった」という指摘があれば、
  モード B の問いの流儀を本ファイルに追記する。
- テーマ分割の粒度: ノートが肥大化・断片化する傾向が出たら、slug の切り方の
  ヒューリスティックを「場所」節に追記する。
- 知識ディレクトリとの往復: 壁打ちで繰り返し参照する原則が `architecture/` に無ければ、
  1 テーマ 1 ファイルでの追記を提案する（実行はユーザーが決める）。

## やってはいけないこと

- ユーザーが言っていないことをユーザーの考え・結論として書かない（未決は未決のまま残す）。
- 壁打ちで結論を押し付けない・問いを飛ばして正解だけ返さない（思考の過程がこのスキルの成果物）。
- 記事本文を書かない・下書きを書き換えない（指摘のみ。ドラフト生成は明示依頼時に friday へ）。
- 記事化・ノートの熟成を催促しない。ノートの少なさ・薄さを指摘しない
  （ネタ提示時の「材料が薄い」の事実提示は除く）。
- 顧客名・案件名・機密をノートに持ち込まない（記録時点で一般化する）。
- `projects/` 配下（homepage 含む）にスキルのデータを保存しない（参照は URL / パスの記録のみ）。
