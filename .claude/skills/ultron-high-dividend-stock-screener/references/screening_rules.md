# スクリーニング・判定ルール（参照メモ）

このスキルの判定軸の定義を集約する。**コア4条件のしきい値は固定**（ユーザー定義。勝手に変えない）。
除外パターンや配当推移の細則は運用で磨く（自己進化の対象 → §進化メモ）。

## コア条件（必須・すべて満たして「合格」）

| # | 条件 | 既定しきい値 | 判定主体 | 備考 |
|---|---|---|---|---|
| 1 | 配当利回り ≧ 閾値 | 4.0% | `judge.py` | スクリーニング段でも一次足切り |
| 2 | 配当が右肩上がり | 減配年 0 回 / 直近 3〜5 期 | `judge.py` | 維持(同額)は減配にしない |
| 3 | 配当性向 < 上限 | 50% | `judge.py` | `< 50%`。境界ギリは「要再確認」 |
| 4 | 営業利益に赤字なし | 直近 3〜5 期すべて黒字 | `judge.py` | 1 期でも赤字/ゼロなら不合格 |
| 5 | REIT/投資法人/インフラF を除外 | 名称・区分判定 | `judge.py`＋取得段 | 疑わしきは除外しログに残す |

→ しきい値は `config/screener.yaml`。合否は **`judge.py` が決定論的に決める**（LLM は数値を捏造しない）。

## 各指標の定義（出典の取り違えを防ぐ）

- **配当利回り(%)**: `1株あたり年間配当 ÷ 株価 × 100`。スクリーニング元(Yahoo 等)の表示値を採用し、取得日を残す。
  記念配・特別配で膨らんだ利回りは過大評価の罠 → 内訳が分かれば **普通配ベース**で再評価し、怪しければ「要再確認」。
- **配当性向(%)**: `1株あたり配当 ÷ EPS × 100`（= 純利益基準と一致）。
  出典で基準が混在しうるため、**EPS 基準に固定**。算出元の配当額・EPS もリストに残し追跡可能にする。
  // ASSUMPTION: IR BANK の「配当性向」表示は EPS 基準（連結）を想定。実データで定義を要確認。
- **配当推移**: 直近 3〜5 期の **1株あたり年間配当（普通配）**を古い→新しいの順で並べる。
  「減配年が無い/少ない（既定 0 回）」を満たすか。記念配の段差は除外して評価する。
  **株式分割・併合**をまたぐ期間は、見かけの 1 株配当が段差になる（例: 1→2 分割で 50 円 → 25 円は実質維持）。
  必ず**分割調整後の 1 株配当に換算**してから比較する。調整前後の値が混在して換算できない場合は
  減配と断定せず「要再確認」で保留する（`judge.py` は換算済みの値を前提とする）。
- **営業利益**: 直近 3〜5 期の **営業利益（本業の利益）**。すべて黒字(> 0)であること。
  純利益でなく営業利益で見る（特別損益・税の影響を除いた本業の継続性を見るため）。

## 除外区分の判定（REIT / 投資法人 / インフラファンド 等）

一次判定は **社名サフィックス/部分一致**（`config/screener.yaml: exclude_name_patterns`）。
取得段で**市場区分・銘柄種別**が取れる場合は併用する。判定の優先度:

1. 名称に「投資法人」「リート」「REIT」「インフラ投資」等を含む → 除外
   （「リート」のみ例外: 部分一致では一般事業会社を誤除外するため、**名称末尾が「リート」**または
   **「リート投資法人」を含む**場合のみ一致。`hdss_lib.exclusion_reason` 実装。→ §進化メモ）
2. 市場区分が「REIT」「インフラファンド」「ETF/ETN」 → 除外
3. 1・2 のいずれにも当たらないが疑わしい → **除外側に倒して**ログに理由を残す

## 補助指標（任意・合否は変えない。Claude レビューの材料）

自己資本比率 / 営業CF の黒字継続 / 連続増配年数 / EPS 推移 / 配当方針（累進・DOE）の明文化有無。
→ 合否には入れず、レビューでの「要再確認」「安心材料」の根拠に使う（§13 で合否に入れない方針に決定）。

## 進化メモ（運用で追記する欄）

- 2026-07-02: 「リート」の部分一致で **「日本コンクリート工業」が誤除外**される事例（実測）。
  「リート」パターンだけは「末尾がリート」or「リート投資法人を含む」場合のみ一致するよう
  `hdss_lib.exclusion_reason` を修正（「ジャパンリート」「〇〇リート投資法人」は引き続き除外）。
- 誤混入した REIT 等の社名パターン → `exclude_name_patterns` に追記して再発防止。
- 「右肩上がり」判定でブレた事例（記念配の扱い等）→ ここに知見を残し、必要なら `allow_dividend_cuts` をユーザー合意で調整。
- 配当性向の基準が出典で食い違った事例 → 採用基準と理由を残す。
