# データ取得元メモ（サイト構造・取得手段）

公開情報を取得する際の入口と注意点。サイト改修で構造が変わるため、**取得に失敗/成功した知見をここに追記**して
変化に追従する（自己進化の対象）。利用規約・robots.txt を尊重し、**過度な連続アクセスをしない**（`batch_size` で制御）。

## 取得手段の優先順位（アダプタ化）

1. `WebFetch` / `WebSearch`（まずこれ。軽量）
2. 構造が取れない / JS レンダリング依存で取れない場合 → ブラウザ自動化（claude-in-chrome 等）にフォールバック
3. 取れない値は**捏造せず「未取得」**として残す（`judge.py` は未取得を insufficient として扱う）

## 1) スクリーニング（高配当利回りの母集団取得）

- 例: Yahoo!ファイナンス 高配当利回りランキング 等。
- 取りたい: **証券コード / 社名 / 配当利回り(%)**（＋可能なら市場区分・銘柄種別）。
- この段で **REIT・投資法人・インフラファンド・ETF/ETN を除外**（名称・区分で判定。`screening_rules.md` 参照）。
- 利回り ≧ 閾値の銘柄を上から拾い、証券コード列を `registry.py filter` に渡して**未調査だけ**に絞る。
- // ASSUMPTION: ランキングページはページネーションあり。1 回の起動では batch_size 分が埋まるまで上位から取得。

## 2) 健全性データ（1 社ずつ）

- 例: IR BANK 等の個別銘柄ページ（業績・配当の推移）。
- 取りたい（古い→新しいの時系列、直近 3〜5 期）:
  - **1株あたり配当（普通配）**の推移 → `dividend_history`
  - **営業利益**の推移 → `op_profit_history`
  - **配当性向(%)**（EPS 基準）→ `payout_ratio`
  - **配当利回り(%)**（スクリーニング段の値を流用可）→ `yield`
- 取得元 URL を必ず `sources` に残す（リストに出所として載せる）。
- 記念配・特別配が混ざる年は内訳を確認し、普通配ベースに直す（不能なら「要再確認」）。

## 3) 法人番号（証券コード → 法人番号 13桁）

- 本筋: **EDINET コードリスト**（`bin/resolve_corp.py` が自動 DL・突合）。
  - ソース: `https://disclosure2dl.edinet-fsa.go.jp/searchdocument/codelist/Edinetcode.zip`
  - 中身: `EdinetcodeDlInfo.csv`（Shift-JIS）。カラムに「証券コード」「提出者法人番号」「提出者名」「上場区分」。
  - 証券コードは末尾 0 付き 5 桁（例 トヨタ `72030`）で配布 → 4 桁へ正規化して Yahoo の `7203` と突合（`hdss_lib.normalize_ticker`）。
  - キャッシュ: `<STOCK_DATA>/edinet/EdinetcodeDlInfo.csv`。`--refresh` で再取得。
- フォールバック: EDINET で引けない例外のみ **国税庁 法人番号システム Web-API**（無料のアプリケーションID登録要）で
  **社名照合**。表記揺れで曖昧になりやすいため、確定できなければ台帳に candidate として残し**人手確認**に回す。

## 取得失敗時の方針

- サイトがブロック/構造変化 → 件数を減らして取れた分で続行し、**警告を添える**（全滅させない）。
- 値が一部だけ取れない → その値を null（未取得）にし、`judge.py` の insufficient で「データ不足」として不合格扱い。
- 取得に難航したサイト/回避できた方法は、この節に追記して次回の効率を上げる。
