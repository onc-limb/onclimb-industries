# jarvis-issue-planner — 開発 Issue（実装計画）の作成

ユーザーの「やりたいこと」を、**自己進化開発ループ（別途運用）の起点にできる開発 Issue** へ
落とし込むスキル。読み手は開発ループの AI と自分自身。粒度は**大きめだが、ゴールが明確で
検証可能**であることを最優先する。実装の詳細設計は Issue に書かず、ループ側に委ねる。

> 位置づけ: このスキルの成果物 = 「何を達成すれば完了か」が機械的に判定できる一次資料。
> 設計判断・実装手順の分解は開発ループの仕事であり、ここでは拘束しない。

## データ配置

- Issue ドキュメント: `projects/<project>/issues/YYYY-MM-DD_<slug>.md`（slug は英語 kebab-case）
- 対象が `projects/` 配下に無い場合は、保存先をユーザーに確認してから書く。
- `issues/` は対象リポジトリの git 管理外の作業ファイルとして扱う（ユーザーが求めない限り
  対象リポジトリにコミットしない）。

## トリガー

| ユーザー発話の例 | 動作 |
|---|---|
| 「<project> で〜をやりたい。Issue を作って」 | フルフロー（調査 → 対話 → 生成） |
| 「実装計画を立てて」 | 同上 |
| 「開発ループに投げるタスクを作って」 | 同上 |
| 「この Issue を GitHub に登録して」 | 生成済み md を GitHub Issue へ登録のみ |

## 標準フロー

### 1. 対象の特定

- 対象プロジェクトを確定する: `projects/<name>/` にあるか `ls projects/` で確認。
- GitHub リポジトリ指定でローカルに無い場合は、`gh repo clone <owner/repo> projects/<name>`
  での取得を提案する（勝手に clone しない）。
- GitHub Issue 登録の可能性があるため、`git remote get-url origin` で `owner/repo` を控えておく。

### 2. やりたいことのヒアリング

ユーザーの言葉で「何をやりたいか」を受け取る。この時点では言い換え・具体化をせず、
原文のまま控える（後で Issue の「背景・目的」の原料になる）。

### 3. 既存コードの調査

やりたいことに関連する現状を対象リポジトリから把握する。全部は読まない。目安は次の順:

1. `README` / 規約文書（CLAUDE.md 等）で全体像と制約を掴む。
2. やりたいことに関連するモジュールを `Glob` / `Grep` で特定し、主要ファイルを Read する。
3. 類似の既存実装（同じパターンで作られた先行機能）があれば探す — ゴール設定の選択肢と
   「実装方針の手がかり」の原料になる。
4. 広い探索が必要なときは Explore エージェント（またはコードベース探索エージェント）に委ね、
   結論だけ受け取る。

調査で押さえる点: 関連コードのパスと現状の構造 / 触ることになりそうな範囲 /
既存の制約（互換性・依存・規約）/ テスト・検証の仕組みの有無。

### 4. 対話でゴールを確定する

調査結果を踏まえ、`AskUserQuestion` でゴールを確定する。**コードで見つけた事実を選択肢に
織り込む**こと（例: 「認証は既存の session 方式に乗せる / 新規に token 方式を導入する」）。
一般論の質問だけで済ませない。

確認する項目（質問は集約し、原則 1〜2 ラウンド以内）:

- **完了条件**: 何ができたら・何が確認できたらこの Issue は完了か。
- **スコープ境界**: 今回やること / やらないこと（調査で見えた「ついでにやれそうなこと」は
  明示的に in/out を確認する）。
- **制約・前提**: 技術選定の指定、互換性維持の要否、既存規約への追従。
- **検証方法**: 完了をどう確認するか（テスト・コマンド・動作確認手順）。

このタイミングで**粒度チェック**を行う: [references/granularity.md](references/granularity.md)
の基準で大きすぎ/小さすぎを判定し、分割・統合・Issue 化不要を対話の中で提案する。
分割した場合は 1 Issue = 1 ファイルで複数生成する。

### 5. Issue ドキュメントの生成

[templates/issue.md](templates/issue.md) の見出し構成で生成し、ドラフトとして提示する。
ユーザーの修正意見を反映してから確定とする。

- 受け入れ条件は必ず**確認方法とセット**で書く（開発ループの AI が自力で判定できる形）。
- 「現状」セクションには調査で読んだ関連コードを `file:line` で残す（ループ側の探索コストを削る）。
- 「実装方針の手がかり」は拘束しないヒントに留める。手順書にしない。
- 事実と推測を区別する。調査で確認していない記述には `(推測)` を付ける。

### 6. GitHub Issue への登録（任意）

ユーザーが希望した場合のみ。**登録は外部公開の操作なので、実行前に本文の最終確認を取る。**

```bash
gh issue create --repo <owner/repo> --title "<md の h1>" --body-file <本文（meta コメント除く）>
```

- 登録後、返ってきた URL を md 末尾の meta コメント `github_issue:` に追記する。
- タイトル・本文は md と同一内容を保つ（GitHub 側だけ編集しない）。

## 注意

- 機密情報（API キー・顧客名等）を Issue に書かない。GitHub 登録時は特に注意する。
- 他プロジェクトの固有情報を Issue に持ち込まない（プロジェクト固有情報の分離ルール）。
- 過去の Issue（`projects/<project>/issues/` 配下）があれば重複・依存関係を確認し、
  関連 Issue は meta コメントの `related:` で相互参照する。
- 学習: 対話で繰り返し確認する項目が出てきたら、本ファイルの「確認する項目」に追記して育てる。
