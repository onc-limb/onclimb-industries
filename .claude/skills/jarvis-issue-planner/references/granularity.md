# Issue 粒度の判断基準

jarvis-issue-planner が「粒度大きめだが明確にゴールのある Issue」を判定するための基準。
数値の目安はすべて**初期値**であり、実運用の結果に応じて人が調整する。

## ちょうどよい粒度（目指す状態）

- **1 Issue = 1 つの検証可能なゴール**。完了が主観でなく、テスト・コマンド・動作確認で
  客観的に判定できる。
- 作業量の目安（初期値）: **半日〜数日、PR 1〜3 本相当**。開発ループが1つの計画として
  分解・遂行できるまとまり。
- 実装の詳細（クラス設計・手順の分解）は**書かれていない**。「何を達成するか」と
  「何を触りそうか」までで止め、どう作るかはループ側の裁量に残す。
- スコープ外が明記されている（ループが脱線しないためのガードレール）。

## 大きすぎる兆候 → 分割を提案

次のいずれかに当てはまったら、対話の中で分割案を提示する。

- ゴールが「A と B」のように**独立した価値の複合**になっている（片方だけ完了しても
  意味が成立する）。→ 価値ごとに分割。
- 受け入れ条件が**列挙しきれない**、または「〜など」で逃げないと書けない。
- 「〜の改善」「〜の整理」など**終わりが定義できない**表現でしかゴールを書けない。
  → 検証可能な部分ゴールを切り出す。
- 触る範囲の見積もりが対象リポジトリの**複数のサブシステムにまたがる**うえ、
  それぞれ独立にリリース・検証できる。

分割するときは、Issue 間の依存（先行/後続）を各 Issue の meta `related:` に残す。

## 小さすぎる兆候 → Issue 化不要 or 統合を提案

- 単一ファイルの軽微な修正・typo・設定値変更レベルで、完了条件が自明。
  → 「Issue にせずこの場で直す or 直接依頼した方が早い」と提案する。
- 単独では価値が完結せず、常に別の作業とセットでしか意味を持たない。
  → 親になる Issue への統合を提案。

## 「明確なゴール」の要件

- 受け入れ条件は**確認方法とセット**で書く（例: `（確認方法: pytest tests/test_report.py が通る）`
  `（確認方法: CLI で `app report --date 2026-07-01` を実行し md が生成される）`）。
- 確認方法を書けない条件は、ゴールが曖昧なサイン。対話に戻して具体化するか、
  スコープ外へ移す。
- 対象リポジトリにテスト基盤が無い場合は、動作確認手順（コマンド＋期待する出力）を
  確認方法として書く。「テストを書くこと」自体を受け入れ条件に含めるかは対話で確認する。
