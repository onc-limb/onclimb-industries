# 参画初期2日間チェックリスト

> 使い方：1日目のヒアリング後、自宅でチェックを埋める。
> 埋まらなかった項目 = 2日目に確認すべき項目。
> 2日目の終わりに「状況理解メモ」を書いて依頼者に認識合わせをするのがゴール。

---

## 0. 参画前 / 初日の朝（準備）

- [ ] 商談時に言われた「依頼内容」を一文で書き出しておく（後で実態と比較するため）
- [ ] 契約書・SOWに書かれている業務範囲を確認した
- [ ] 会社のプロダクト・サービスを一度ユーザーとして触った
- [ ] 依頼者（契約を決めた人）と、日々一緒に働く現場メンバーが誰なのかを把握した
- [ ] 自分の受け入れ担当（オンボーディング窓口）が誰かを確認した

---

## 1. 依頼者（発注者・契約を決めた人）へのヒアリング

### ゴール・期待値

- [ ] 「このプロジェクトが半年後にどうなっていたら成功か？」を聞いた
- [ ] 「私（自分）が3ヶ月後にどういう状態なら発注してよかったと思えるか？」を聞いた
- [ ] 商談時の依頼内容と、今聞いた期待にズレがないか確認した
- [ ] 成果の測り方（KPI・納期・デモ・リリースなど）があるか聞いた

### 課題感

- [ ] 「今、一番困っていること・止まっていることは何か？」を聞いた
- [ ] なぜ社内メンバーではなく外部に依頼したのかを聞いた
- [ ] 「本当はやりたいが手が回っていないこと」があるか聞いた

### 体制・意思決定

- [ ] 最終的な意思決定者（誰がOKと言えば進むのか）を特定した
- [ ] 依頼者自身が決裁者なのか、上に別の決裁者がいるのかを確認した
- [ ] 自分の作業の報告先・頻度・形式（週次？チャット？）を確認した
- [ ] 判断に迷ったとき、まず誰に相談すべきかを確認した

### 過去の経緯（地雷の把握）

- [ ] 自分の前に同じ立場の人（前任者・別の委託先）がいたか聞いた
- [ ] いた場合、なぜ抜けたのか／何がうまくいかなかったのかを聞いた
- [ ] 過去に頓挫した施策・プロジェクトがあるか聞いた

---

## 2. 現場メンバー（一緒に働くエンジニア・PM等）へのヒアリング

### ゴール・期待値

- [ ] 現場メンバーにも「半年後どうなっていたら成功か？」を聞いた
- [ ] **依頼者の答えと現場の答えのズレの有無をメモした**（ここが最重要）
- [ ] 現場が自分（外部人材）に期待していることを聞いた
- [ ] 逆に「これはやってほしくない・触ってほしくない」領域があるか聞いた

### 課題感・日々の痛み

- [ ] 現場が日々一番時間を取られていること・面倒に感じていることを聞いた
- [ ] 「本当はこうしたいのにできていないこと」を聞いた
- [ ] 技術的負債・触るのが怖いコード・属人化している領域があるか聞いた

### 開発の実態

- [ ] 開発フロー（ブランチ運用・レビュー・デプロイ手順）を聞いた
- [ ] レビューは誰がしているか、回っているかを聞いた
- [ ] タスクはどこで管理され、誰がアサインを決めているかを聞いた
- [ ] 暗黙のルール・ドキュメントにない慣習があるか聞いた
- [ ] キーパーソン（この人に聞けば大体わかる、という人）が誰かを聞いた

---

## 3. 観察・自分で調べる（聞かずにわかること）

### ドキュメント・チケット

- [ ] READMEやオンボーディング資料を読んだ（古さ・整備度も課題のヒント）
- [ ] チケット管理ツールで「長期間止まっているチケット」を確認した
- [ ] チャットの過去ログで、議論が紛糾した／流れたスレッドを確認した

### リポジトリ

- [ ] 直近数ヶ月のコミット履歴を見た（活発さ・偏り）
- [ ] コミットが特定の個人に集中していないか確認した
- [ ] PRのレビューが実際に機能しているか（形骸化していないか）確認した
- [ ] CI/CD・テストの有無と実際に通っているかを確認した

### 会議・空気

- [ ] 定例やスタンドアップに同席し、誰が発言し誰が黙っているかを観察した
- [ ] 実質的に物事を決めているのが誰かを観察した
- [ ] 依頼者と現場の間の空気（信頼関係・温度差）をメモした

---

## 4. 1日目の夜（自宅での棚卸し）

- [ ] このチェックリストを埋め、**未チェック項目を2日目の確認リストに転記した**
- [ ] 依頼者の期待と現場の期待のギャップを一文で書けるか試した
- [ ] 「本当の課題は〇〇ではないか」という仮説を1〜3個書いた
- [ ] 仮説を検証するために2日目に誰に何を聞くか決めた
- [ ] 初週に出せそうな小さな成果（ドキュメント整備・小バグ修正など）の候補を1つ決めた

---

## 5. 2日目

- [ ] 1日目の未チェック項目を確認した
- [ ] 立てた仮説を、断定ではなく質問の形で現場にぶつけて反応を見た
- [ ] 開発環境を構築し、詰まった箇所をメモした（ドキュメント改善ネタになる）
- [ ] 小さな成果に着手した（完了していなくてもよい）

### 2日目の終わり：認識合わせ（最重要アウトプット）

- [ ] 以下を含む半ページ程度の「状況理解メモ」を書いた
  - [ ] 2日間で理解した現状
  - [ ] 本質的な課題だと感じたこと（仮説として）
  - [ ] 商談時の依頼内容との差分（あれば）
  - [ ] 自分が優先すべきと考える動き方の提案
- [ ] メモを依頼者に共有し、「この理解で合っていますか？」と確認した
- [ ] フィードバックを受けて、初週〜初月の自分の動き方を合意した
