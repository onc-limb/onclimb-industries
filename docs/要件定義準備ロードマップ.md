# 要件定義 参画準備ロードマップ(2ヶ月)v2
## 案件想定:メガバンク従業員向け 資産管理アプリ

## 方針

- 「知識のインプット → 型の言語化 → AIスキル化 → 模擬演習で検証」のサイクルを毎週回す
- **要件定義スキル**と**金融ドメイン知識**の2トラックを並行して進める
- AIスキル(テンプレート・チェックリスト・プロンプト)は**作る過程が勉強**。インプットした内容を自分の言葉で構造化してスキルに落とす
- AIに任せるのは「型・下書き・レビュー」。ヒアリングでの引き出し、ずれの検知、合意形成は自分の仕事として残る

## 第1〜2週:工程の全体像+ドメイン基礎

### トラックA:要件定義
- IPA「ユーザのための要件定義ガイド」通読(無料PDF)
- IPA「非機能要求グレード」6大項目の把握
- 書籍「はじめよう!要件定義」(羽生章洋)読了
- 企画→要件定義→基本設計の線引き(Whatまでが要件定義)を自分の言葉で整理
- アジャイル経験を要件定義の語彙にマッピングした対応表を作る

### トラックB:資産管理ドメイン
- **金融商品の基礎**:預金・株式・債券・投資信託・保険・年金の仕組み。体系的に学ぶなら**FP3級のシラバス(特にライフプランニング・金融資産運用の分野)**が最短
- **従業員の資産形成制度**:NISA(新NISA)、iDeCo、企業型DC、財形貯蓄(一般・住宅・年金)、従業員持株会 — 「従業員向け」アプリなら中核になる可能性が高い
- **銀行従業員の特殊性**:役職員の有価証券取引規制(インサイダー取引防止、自己取引の届出・承認制)、日本証券業協会・銀行の社内規程の存在を概要レベルで把握
- 用語集(T9)の器を作り、学んだ用語を貯め始める

### AIスキル化
- スキル①「要件定義の壁打ち相手」
- スキル②「金融・資産管理ドメインQA」:用語を聞くと解説+関連規制+アプリ要件への示唆を返す

## 第3〜4週:成果物の型+金融系の非機能

### トラックA:要件定義
- 要件定義書サンプルを2〜3本読む(官公庁の公開調達資料が実物として優秀。金融庁・財務局系の調達仕様書があればなお良い)
- スイムレーン図の記法を習得
- 要件定義書の標準目次を自分で組み立てる
- 「良い要件記述」の条件(検証可能・一意・実現手段を含まない)を整理

### トラックB:金融系システムの非機能・規制
- **FISC安全対策基準**:金融機関システムの事実上の標準。全読は不要、構成(統制・実務・設備・監査)と存在意義を把握
- **個人情報・データ保護**:個人情報保護法の基礎、金融分野ガイドラインの存在、マイナンバー(特定個人情報)の扱いの厳しさ
- **セキュリティ要件の相場観**:多要素認証、アクセスログ・監査証跡、データ暗号化、社内ネットワーク前提か否か
- 銀行系開発文化の理解:ドキュメント品質・レビュー・監査対応が重い、変更管理が厳格、という心構え

### テンプレート作成
- T1 要件定義書 / T2 機能一覧表 / T3 非機能要件一覧(FISC観点を追加した金融版) / T4 業務フロー図雛形

## 第7〜8週:模擬演習(通し稽古)と最終調整

### 演習のお題
**「メガバンク従業員向け資産管理アプリの新規開発」そのもの**で通し稽古する(本番と同じ題材で練習できるのがこの案件の利点)

### 通しの流れ
1. AIに顧客役(銀行の人事部・コンプライアンス部・システム部門の担当者)をやってもらいヒアリング(スキル⑦)
2. 議事録作成(スキル⑧ → T5)
3. As-Is/To-Be業務フロー図(T4)
4. 機能一覧(スキル⑤ → T2)・非機能要件(スキル③ → T3)
5. 要件定義書ドラフト(スキル④ → T1)→ セルフレビュー(スキル⑥)
6. レビュー会資料(スキル⑩ → T8)を1本作成

### 演習後
- 手が止まった箇所・スキルの出力が使えなかった箇所を洗い出してスキル改修
- 弱かった領域を該当週の資料に戻って補強

## 注意事項

- **秘密保持とAI利用**:参画先の情報(顧客名・業務内容・資料)をAIに入力してよいかは、参画先のAI利用ポリシーとNDA次第。準備期間中は公開情報と架空設定だけでスキルを作り込み、実データを使わない形にしておく
- 現場ごとに要件定義の線引き・フォーマットは異なる。自作テンプレは叩き台とし、初日に現場標準を確認して合わせる
- 金融規制の細部(金商法・銀行法の条文レベル)まで暗記する必要はない。「そういう規制領域がある」と気づけて、正しい確認先(コンプラ部門・法務)に質問を出せることが要件定義担当としての合格ライン
