# 生成ドキュメントの「AI感」低減 — 調査と適用設計

作成日: 2026-07-02
対象: friday 系・jarvis 系を中心としたドキュメント生成スキル群
ステータス: Phase 1〜4 実装済み（2026-07-02）。Phase 5（運用でカタログを育てる）は継続運用

## 1. 背景と目的

friday 系（日報スライド・議事録・提案書・技術記事・設計書）や jarvis 系（一次記録・ナレッジ）が
生成するドキュメントには、読み手に「AI が書いた」と感じさせる癖（以下「AI 感」）が乗りやすい。
読み手が他者であるドキュメントほど、AI 感は信頼性・説得力を下げる。
特に公開記事（friday-tech-article-drafter）は「AI 生成記事」と見なされること自体が読者離れの要因になる。

本ドキュメントは、インターネット上の知見（Wikipedia の AI 文章検出ガイド、日本語圏の実践記事、
技術記事コミュニティでの議論）を収集・整理し、このリポジトリのスキル群にどう適用するかを設計する。

### 目的ではないこと（設計上の制約）

- **AI 検出ツールの回避や「AI 利用の隠蔽」は目的にしない**。目的は読みやすさと信頼性の向上。
- **人間らしさの捏造はしない**。体験談・感情・エピソードを作り話で足すのは、
  persona 共通原則「入力に存在しない情報を創作しない」に反する。
  AI 感の除去は「余計な癖を削る（引き算）」と「入力にある一次情報を活かす（原料の活用）」で行い、
  「人間っぽさを創作で足す（足し算）」では行わない。これが本設計の最重要判断。

## 2. 調査サマリ — AI 感の正体

複数ソース（§6）を横断すると、AI 感は次の 5 カテゴリに整理できる。

### 2.1 語彙・言い回しの癖

| パターン | 例 | 出典傾向 |
|---|---|---|
| 逃げ文句・保険表現 | 「場合によります」「一般的に」「〜と言われています」「状況によって異なります」 | 日本語圏記事・Zenn で共通指摘 |
| 抽象語の空回り | 「本質」「最適」「価値」「効率化」だけで具体像が浮かばない | ai-advisors.jp |
| 道徳の教科書調 | 「〜が重要です」「〜を心がけましょう」「バランスが大切です」 | Zenn（AI 記事の見分け方） |
| 定型比喩 | 「羅針盤」「地図」「土台」「〜への扉」 | ai-advisors.jp / Wikipedia |
| AI 頻出語彙（英語圏由来） | 「極めて重要」「多面的」「包括的」「〜を深掘り」「〜の証」 | Wikipedia (delve / pivotal / crucial 等の直訳) |
| 根拠のない出典ぼかし | 「専門家によれば」「多くの人が〜と感じています」 | Wikipedia / Zenn |
| コピュラ回避 | 「〜である」で済む所を「〜としての役割を果たす」「〜を体現する」 | Wikipedia |

### 2.2 構造・リズムの癖

| パターン | 例 |
|---|---|
| 構造宣言・前置き | 「以下で解説します」「この章では〜を見ていきます」— 言う前に予告する |
| 三点セット（rule of three） | 形容詞や項目をなぜか常に 3 つ並べる |
| 否定並列 | 「単なる X ではなく Y」「X だけでなく Y も」の多用 |
| 均一すぎる段落・節 | 全段落が同じ長さ、全節が同じ丁寧さ、話の脱線や重心の偏りがない |
| テンプレ完全準拠 | 「見出し→小見出し→まとめ」が崩れず、興味の濃淡が文章に出ない |
| 過剰な誘導フレーズ | 「まとめると」「要するに」「ここで気をつけたいのは」の多発 |
| 節末の要約癖 | 各節の最後に毎回まとめ段落を置く |

### 2.3 書式の癖

| パターン | 例 |
|---|---|
| 太字の乱用 | キーワードを機械的に全部太字にする |
| 記号・装飾過多 | 箇条書き記号の乱立、絵文字、`—`（em dash）の多用 |
| 「太字見出し + コロン」リスト | **項目名**: 説明 … の縦列を本文の代わりに使う |
| 散文で書くべき所の表・箇条書き化 | 2〜3 項目しかないのに表にする |

### 2.4 トーン・内容の癖

| パターン | 例 |
|---|---|
| 過剰な丁寧さ・宣伝調 | press release 調、「画期的」「充実の」「豊富な」 |
| 感情・主観の欠落 | 常に冷静で、困惑・驚き・悔しさの揺らぎがない |
| 一次情報の欠如 | 実体験・独自の観察がなく一般論で埋まる |
| 完璧すぎる | 全コード例が一発で動き、エラーや試行錯誤の跡がない。全問題に解決策があり、未解決で終わらない（uhyo 氏の実験で「完璧さ自体が AI の痕跡」と指摘） |
| 両論併記癖 | 何でも「メリットもあるがデメリットもある」で終える |

### 2.5 日本語特有の癖

| パターン | 対処 |
|---|---|
| 「〜です」「〜ます」の単調な連続 | 体言止め・「〜でした」「〜になる」等で文末を散らす（ただしビジネス文書では節度を保つ） |
| 「しかしながら」「したがいまして」等の硬い接続詞 | 「ただ」「なので」等、話し言葉寄りに（文書種別による） |
| 「〜と言えるでしょう」「〜ではないでしょうか」 | 断定できるなら断定する。できないなら根拠を書く |
| 「いかがでしたか」型の締め | 使わない |
| 翻訳調の丁寧すぎる距離感 | 読み手との実際の関係（依頼者・同僚・読者）に合わせる |

### 2.6 ビジネス文書での現れ方（Qiita 圏の分析より）

日本語記事 180 本の定量分析（kenimo49 氏）によると、AI 感は「語彙・構造・リズム」の
3 軸の均質化に集約され、いずれもビジネス文書にそのまま現れる。

- **語彙**: 「〜と言えるでしょう」「重要です」「不可欠です」が 1 文書あたり平均 3 回超。
  提案書・報告書では「重要な課題として認識されます」「今後の戦略に不可欠となります」型が典型。
- **構造**: 毎回「背景 → 課題 → 解決案 3 つ → まとめ」の金太郎飴構成。内容を無理やり 3 分割する癖。
  人間は論点の重さに応じて構成を変える。
- **リズム**: 文長の変動係数が狭い範囲に収まる（＝全文がほぼ同じ長さ）。人間は重要な論点で
  短く言い切り、補足で長くなる、という揺れを持つ。**報告書全体で文長が均一なら機械生成に見える**。

また Claude Code の Skill として AI 感除去を実装した先行例（yukurash 氏『16-minds』）は、
設計原則を「**省略する**（読者への信頼を前提に親切な定義・接続詞を削る）/
**偏る**（両論併記を禁止し、一つの立場から言い切る）/
**具体で殴る**（固有名詞と数字で語る）」の 3 つに集約しており、
SKILL.md のプロンプトに原則を焼き付ける本設計と同じアプローチで効果を確認している。

### 2.7 対策の類型（ソース横断の合意点）

1. **禁止事項の先制指定** — 「AI っぽさの 9 割は禁止すべき癖の復活が原因」（ai-advisors.jp）。
   生成前にプロンプトで NG パターンを縛るのが最も効きやすい。
2. **二段階生成（清書パス）** — 下書き生成後、「意味と事実を変えずに癖だけ削る」書き直しパスを通す。
3. **文体サンプル few-shot** — 本人の過去の文章を例示して文体を寄せる。
4. **具体性の強制** — 抽象語を「何がどうなるか」の動詞中心表現へ。数字・固有名詞・実測値を使う
   （ただし本リポジトリでは**入力に実在するものだけ**）。
5. **一次情報の注入** — 実体験・エラー・試行錯誤・採用しなかった案を書く。AI 記事と人間の記事の
   最大の差は「オリジナルの知見と感情の有無」（Zenn 圏の共通見解）。
6. **スタンスの明示** — 両論併記で終えず、推奨・判断を言い切る（16-minds「偏る」）。
   ビジネス文書では「言い切り + 根拠 + 適用条件」の形にすれば誠実さと両立する。
7. **AI の役割を構成段階に限定する** — 構成・見出しレベルまでを AI に任せ、本文は一次情報で
   埋める運用（kskwtnk 氏）。本リポジトリのスキルは元々「固定フォーマット + 入力事実の流し込み」
   構造なので、この型には既に適合している。
8. **人間の最終チェック** — どのソースも「プロンプトだけでは完結しない。最後は人が読む」で一致。
9. **計画的な不完全さ** — 脱線・未解決・重心の偏りを意図的に残す（uhyo 氏）。
   ※効果はあるが「演出」に近く、業務文書には不適。適用は公開記事に限定し、かつ捏造はしない
   （実際にあった脱線・未解決だけを削らずに残す、という形で実現する）。

## 3. 適用設計

### 3.1 大原則: 読み手で適用強度を変える

このリポジトリは「誰が読むか」でスキルを分類済み（jarvis = 自分/AI、friday = 他者）。
AI 感対策も同じ軸で強度を変える。

| 分類 | 適用強度 | 理由 |
|---|---|---|
| friday 系 | **全面適用** | 読み手が他者。AI 感が信頼性・説得力を直接下げる |
| jarvis 系 | **限定適用 + 原料強化** | 読み手は自分/AI。機械的で正確なことが価値であり、文体の人間化は不要。ただし §3.4 の「原料を残す」責務を追加する |

さらに friday 系の中でも文書種別で調整する。ビジネス文書（提案書・議事録・日報）は
「自然さより明確さを優先しつつ AI 臭だけ落とす」、公開記事は「一次情報と文体まで含めて全面対応」。

**ビジネス文書共通の 4 指針**（§2.6 の 3 軸に対応。技術記事向けの文体演出とは切り分ける）:

1. **語彙** — 逃げ文句と空語（「重要です」「不可欠です」「〜と言えるでしょう」）を禁止し、
   数字・固有の作業内容・具体的な影響で語る。
2. **構造** — 「ポイント 3 つ + まとめ」の金太郎飴を避ける。項目数は内容が決める。
   固定フォーマットの見出しは維持しつつ、**論点の重さに応じて記述量に濃淡をつける**
   （全項目を同じ厚さで書かない）。
3. **リズム** — 文長を揃えない。結論・判断は短く言い切り、根拠・経緯は長くてよい。
4. **スタンス** — 両論併記で終えない。推奨・判断を言い切り、根拠と適用条件を添える。
   ヘッジが必要な箇所は「何が確認できたら確定するか」まで書く。

### 3.2 実装アーキテクチャ: 3 層構成

```
Layer 1  personas/writing-style.md（新規）
         └ 全スキル共通の「AI 感 NG パターンカタログ + 清書パス手順」
Layer 2  personas/friday.md, personas/jarvis.md（追記）
         └ 分類ごとの適用方針（friday: 参照必須 / jarvis: 原料保全の責務）
Layer 3  各 SKILL.md（追記）
         └ スキル固有の適用ポイント（フローのどこで清書パスを通すか、固有の NG、チェックリスト連携）
```

新しい persona 横断ルールを 1 ファイルに集約するのは、既存の「共通知見はスキル・personas 側に
切り出す」運用（ルート CLAUDE.md）に沿う。各 SKILL.md に全文コピーしない（重複メンテを避ける）。

### 3.3 Layer 1: `personas/writing-style.md` の内容設計

次の 4 部構成で作る。

**(a) 原則（冒頭に明記）**

- AI 感の除去は引き算で行う。創作で人間らしさを足さない。
- 事実・数値・意味を変えない。癖だけ削る。
- 明確さ > 自然さ > 個性、の優先順位（文書種別により Layer 3 で調整）。

**(b) NG パターンカタログ** — §2 の 5 カテゴリを「パターン / NG 例 / 書き換え例」の表にする。例:

| NG | 書き換え |
|---|---|
| 「テストの整備は極めて重要です。品質・速度・信頼性の向上が期待できます」 | 「テストが無いので、この関数はリファクタのたびに手で確認している。まずここに 1 本書く」 |
| 「以下で設定手順を解説します」 | （削除して手順から始める） |
| 「〜と言えるでしょう」 | 「〜だ」または「〜かは未確認」 |
| キーワードを全部**太字** | 本当に読み飛ばされると困る 1〜2 箇所だけ |

**(c) 清書パス（de-AI パス）の標準手順** — 二段階生成の共通手順:

1. ドラフト完成後、カタログを NG 観点として全文を読み直す。
2. 意味・事実・数値を変えずに、該当箇所だけ書き換える（この段階で情報を足さない）。
3. 文末の連続（です/ます 3 連超）と段落長の均一さを崩せる箇所があれば崩す。
4. 書き換えで文意が怪しくなった箇所は元に戻す（明確さ優先）。

**(d) セルフチェックリスト** — 出力前の最終確認 5 項目程度:

- [ ] 構造宣言・前置き・「いかがでしたか」型の締めが残っていないか
- [ ] 逃げ文句（一般的に/場合によります/〜と言われています）が残っていないか
- [ ] 抽象語だけの文がないか（何がどうなるか読めるか）
- [ ] 太字・箇条書き・記号が本文の代わりになっていないか
- [ ] 文長・段落の厚さが全体で均一になっていないか（重い論点ほど厚く、結論は短く）
- [ ] 両論併記・項目の 3 分割が癖で入っていないか（判断は言い切り、項目数は内容が決める）
- [ ] 入力に無い体験・感情・数値を「足して」いないか（← 逆方向のチェック）

### 3.4 Layer 2: persona への追記

**personas/friday.md** — 「ドキュメント生成の原則」に追記:

- ドラフト完成後、`personas/writing-style.md` の清書パスを必ず通す。
- 具体性は入力（一次記録・digest・文字起こし）にある事実で担保する。不足なら創作せずヒアリング。

**personas/jarvis.md** — 「共通の役割」に追記（文体ルールではなく**原料保全**の責務）:

- 下流（friday 系）の AI 感除去の原料は一次情報の豊かさである。ログ整理・一次記録の段階で、
  **つまずき・エラー原文・試行錯誤・採用しなかった案・そのとき感じたこと**を「些末」として
  削らない。整理はするが平準化しない。

この非対称が本設計の核: **friday は削る、jarvis は残す**。
AI 感のない文章の最大の特徴は「一次情報がある」ことなので、jarvis が過程の生データを保全して
いなければ、friday がどれだけ癖を削っても「きれいだが中身が一般論」の AI 的文章に戻ってしまう。

### 3.5 Layer 3: スキル別適用ポイント

| スキル | 適用強度 | 固有の適用ポイント |
|---|---|---|
| friday-tech-article-drafter | **最重要・全面** | (1) フローに「ステップ 3.5 清書パス」を挿入（マスキングの前）。(2) 公開前チェックリスト（references/masking-checklist.md とは別観点）に AI 感チェック項目を追加。(3) 記事骨子「試したこと」に、digest にある失敗・回り道を**必ず**含める（一次情報注入）。(4) `articles/` に公開済み・執筆済みの過去記事が溜まったら、文体 few-shot として 1〜2 本をドラフト生成時に参照する |
| friday-daily-report | 中（明確さ優先） | 非エンジニア向けスライド。逃げ文句・抽象語の禁止と「数字・固有の作業内容で語る」ことに絞る。案件ごとの記述量は進捗の大小で濃淡をつける（全案件を同じ厚さで書かない）。レイアウトはレンダラー固定なので書式癖は対象外 |
| friday-proposal-generator | 中（明確さ優先） | 宣伝調・両論併記癖の除去が中心。「豊富な実績」「最適なソリューション」「〜が重要です」型の空語を禁止し、スコープ・数字で語る。提案・推奨は言い切り、根拠と適用条件を添える。タスク数・提案項目数を 3 に丸めない（内容が決める）。見積の説明は硬くてよい（信頼性優先） |
| friday-giziroku | 低 | 発言の事実性が最優先。適用は「構造宣言・節末要約癖・過剰な誘導フレーズの除去」のみ。発言の言い換えで文体を「良く」しない |
| friday-design-doc-generator | 中 | 構造宣言・コピュラ回避・三点セットの除去。TBD/(推測) タグ運用は既存のまま（これ自体が誠実さ＝反 AI 感に寄与している） |
| jarvis-record | 原料強化 | 固定見出し「検討したが採用しなかった案」「障害」を空欄で流さない。digest に痕跡があればヒアリングで拾う。エラーメッセージ・数値は原文で残す |
| jarvis-worklog / knowledge-base | 原料強化 | 整理・分類時に試行錯誤の過程を要約で潰さない（既存原則の再確認レベル。文体対策は不要） |
| jarvis-todo-management | 対象外 | 自分専用の運用データ。適用しない |

### 3.6 段階的導入計画

| Phase | 内容 | 成果物 |
|---|---|---|
| 1 | `personas/writing-style.md` 新設（§3.3 の 4 部構成）。friday.md / jarvis.md へ参照追記。`.claude/skills/README.md` に一言追記 | 共通ガイド 1 本 + persona 2 ファイル修正 |
| 2 | friday-tech-article-drafter へ適用（清書パス挿入・チェックリスト拡張・一次情報注入ルール） | SKILL.md + references 修正 |
| 3 | 残りの friday 系 4 スキルへ Layer 3 の表どおり適用 | SKILL.md 修正 ×4 |
| 4 | jarvis-record / worklog へ「原料保全」追記 | SKILL.md 修正 ×2 |
| 5 | 運用フィードバック: 実際の出力で「AI っぽい」と感じた箇所をカタログの NG 例に追記して育てる（辞書方式。`articles/_ng-words.txt` と同じ流儀） | writing-style.md の継続更新 |

Phase 1〜2 だけでも効果の大半（公開記事の品質）が得られる。3 以降は独立に進められる。

## 4. 設計上のトレードオフと判断

- **「感情・体験談を足せ」系ノウハウの扱い** — Web の定番ノウハウだが、そのまま採用すると
  創作禁止の原則と衝突する。本設計では「足す」のではなく「jarvis 段階で実在の過程・感情を
  保全し、friday 段階でそれを削らず使う」に読み替えた（§3.4）。
- **「計画的な不完全さ」の扱い** — 脱線や未解決を演出として挿入するのは業務文書では不誠実。
  「実際にあった未解決・回り道を、きれいにしすぎて消さない」という消極形でのみ採用する。
- **文末バリエーションの限度** — 体言止めの多用はビジネス文書では逆に軽くなる。
  「3 連続を超える同一文末を避ける」程度の弱いルールに留め、種別ごとの調整は Layer 3 に置く。
- **議事録は聖域** — 発言の記録に文体改善を加えると事実性を損なう。giziroku への適用は最小限。
- **チェックの自動化は見送り** — NG 語の grep 自動チェックも可能だが、AI 感は文脈依存が強く
  （「重要です」が常に NG ではない）、偽陽性が多くなる。まずカタログ + セルフレビューの運用で始め、
  Phase 5 で頻出パターンが固まってから機械チェック化を検討する。

## 5. 本設計で変更しないもの

- 各スキルの固定フォーマット・見出し構成（AI 感対策は文章表現の層で行い、構造は変えない）
- マスキング・機密ルール（既存のまま。清書パスはマスキングの**前**に置き、マスキングを最終工程に保つ）
- データディレクトリ・スクリプト（文書生成の指示レイヤーのみの変更）
