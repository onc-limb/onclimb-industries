# 2026-05-19-meeting__ai-collaboration-process

> 中心課題: AI との検証作業で自分が主導権を握れているか (目的を見失わず / 具体に引きずられていないか) を、最初に定義した完了基準への沿い方で判定する。Claude MD 標準は 10 回中 1 回までのミスリードを許容、5 回以上のミスリードで撤退判断、1 ヶ月改善しても精度が上がらなければ撤退。

## F1. 課題発見 (Discovery)

### 重要 (中心)

- 課題ステートメント: 自分が AI と検証作業をしているとき、AI の出力に引っ張られて『課題把握』フェーズで『変更コード』が出てきたり、『ついで実装』が混ざったりして、検証の単位がブレる
- 現状の観測可能な事象: ①前提ズレが頻発 ②課題把握フェーズに変更コードが混ざる ③『ついでに』提案が来る ④ブランチ / リポジトリが散らかり差分比較が困難 ⑤『机を蹴るレベルでムカつく』という感情的負荷
- 自覚した中心: 『AI の議論に付き合わず、自分のフロー・分岐図に基づき検証を主導する姿勢が必要』

### 付随 (脇に置く)

- 周辺の事実: 6 回くらいラウンドを反復した / ワークツリー多用で干渉検知が困難 / 並行 3 件は破綻
- 同様の困り方をしている人がいるか (= スコープ拡大の問い) は、現時点では脇に置く

## F2. 課題整理 (Structuring)

### 重要 (中心)

- 中心課題: AI 出力の主導権を自分が握れていない (= 自分のフロー / 検証単位を維持できない)
- 中心と直結する派生問題: ①リポジトリ / ブランチが散らかると比較困難 ②並行作業で単位がブレる — どちらも『前提を維持できない』の系統

### 付随 (脇に置く)

- 独立に扱える (今は脇): 雑談 / 画面共有トラブル / Teams 出入りの呼びかけ空振り — 業務に直接影響薄
- 感情負荷 (『苛立ち』『机を蹴る』) は中心課題のシグナルだが、解決対象そのものではない

## F3. 解決定義 (Done Definition)

### 重要 (中心)

- **Done 基準 (主導できた状態)**: 目的を見失わず、具体に引きずられていなければ「主導できた」と言える。具体的には、作業が **目的に沿った内容** で、かつ **最初に作った完了の定義を満たすために動けているか** で判定する。
- **撤退ルール (Claude MD 標準 — 単一セッション基準)**: 1 質問に対して **AI が 5 回以上ミスリード** してきたら撤退判断。
- **ミスリードの定義**: 現在検討している粒度よりも **細かい具体** に話題を持っていった場合をミスリード 1 回としてカウント。
- **許容水準**: 10 回の質問のうち **1 回は具体に行ってしまう** 程度は許容範囲。
- **撤退ルール (継続運用基準)**: Claude MD 標準を 1 ヶ月運用して **精度が上がらなければ撤退** (実際には改善になるケースも含む)。

### 重要 (未確定 — Done を完全に閉じるために残る論点)

- **U1. 「精度」の指標**: 1 ヶ月で見る「精度」とは具体に何の数値か。ミスリード発生回数の推移か / 主導できたと感じたセッションの比率か / 検証 1 サイクルの所要時間か — これが決まらないと「上がった / 上がらない」が判断できない。
- **U2. ミスリード 1 カウントの判定主体と判定タイミング**: 「ハッと気づいた瞬間」を 1 カウントとするのか、セッション後のログ振り返りで数えるのか。判定が自分の主観に依存していると、評価自体がミスリードを起こす可能性。
- **U3. 「目的に沿った内容」「最初の完了の定義」の参照先**: プロダクト側 (ai-qa-system) の Done が U1〜U4 で未確定のまま。プロセス側の判定基盤が、揺れているプロダクト Done に依存している構造を許容するのか、プロダクト Done が固まるまではプロセス側も暫定運用とするのか。
- **U4. ミスリード以外の失敗パターン**: 「ついでに」提案 / 課題把握フェーズに変更コードが出る、はミスリードの一種か別カテゴリか。撤退カウントの対象に含めるかどうか未定。

### 付随 (脇に置く)

- 周辺指標: 検証 1 サイクルの所要時間 / リファクタが混入した PR の本数 — 補助的に見るが Done 判定には使わない

## F4. 手段検討 (Solution)

### 重要 (中心)

- 採用 / 着手中: ①Claude MD 標準の整備と運用 ②Ref フォルダで良例コードを置いて差分比較 ③クリアデスク / リポジトリ整頓
- 撤退条件 (確定): 単一セッションで 5 回以上のミスリード / 1 ヶ月運用で精度向上なし

### 付随 (脇に置く)

- 並行作業ルール (『同時に 3 つやらない』) は宣言のみで運用基準なし — Claude MD 標準の運用が安定したら定式化を検討
- 「机を蹴るレベルでムカつく」感情負荷は撤退トリガーではないが、ミスリード頻発時に自覚しやすいシグナル

## 振り返り

- 前回 (議事録時点) → 今回の差分: 撤退条件 (5 回 / 1 ヶ月) と許容範囲 (10 回中 1 回) が数値化された。Done の体裁が立ち上がった。
- 残る主観依存: U2 (ミスリード 1 カウントの判定が主観) と U3 (土台のプロダクト Done が主観依存)。プロダクト側の主観依存と入れ子で同じ構造を持っており、片方が揺れるともう片方も揺れる。
- 戦略的な観察: プロセス Done をプロダクト Done より先に固める意義 — プロセス側で「目的に沿えているか」を測れる状態が先にあれば、プロダクト側の Done 主観性を検知しやすくなる可能性。
